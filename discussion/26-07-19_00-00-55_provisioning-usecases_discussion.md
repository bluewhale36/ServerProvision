## 0. 이 문서의 목적 (두 갈래)

이 세션에서 사용자가 제기한 두 가지를 다룬다.

1. **물리 서버 의존성 구분 + 인프라 선개발** — 개발 환경에 물리 서버가 상시 있지 않으므로, "물리 서버 없이 개발 · 검증 가능한 것" 과 "물리 서버가 반드시 필요한 것" 을 갈라, **물리 불요 부분(인프라 · 로직 · 자산 준비)을 먼저 완성**하고 물리 서버는 몰아서 테스트한다. → §1

## 1. 물리 서버 의존성 구분 + 인프라 선개발

### 1-1. 검증 Tier 복기 (E 로드맵에서 이미 정의됨)

| Tier | 의미 | 물리 서버 |
|---|---|---|
| **T0** | `@WebMvcTest` 계약 테스트 (상태코드 · 바디) | 불요 |
| **T1** | 모의 게스트 하네스 (curl 시퀀스) | 불요 |
| **T2** | QEMU/KVM PXE 랩 (가상 게스트로 부팅 · 설치 E2E) | 불요 (Rocky 서버 위 KVM / macOS 는 부팅 스모크까지) |
| **T3** | **실기** (flash 집행 · FRU · Redfish 실보드) | **필요** |

핵심: **T0 · T1 은 지금 그대로 하고 있고, T2(QEMU 가상 게스트)까지가 "물리 불요" 의 상한**이다. 인프라(진단 리눅스 이미지 · dhcpd · iPXE · apkovl · 에이전트)를 실제로 만들어 두면 **T2 로 진단~OS 설치까지 가상 게스트로 검증**할 수 있고, 물리 서버(T3)는 flash · Redfish 같은 하드웨어 직접 조작에만 필요하다.

### 1-2. 단계별 물리 의존성 (현행 로드맵 기준 — 초안, 토론으로 확정)

| 단계 | 물리 불요로 어디까지 | 물리 서버(T3) 필요 지점 |
|---|---|---|
| **E1-0a/0b** (엔진 · 프로토콜) | **전부** — 하네스로 완결 (완료) | 없음 |
| **E1-1** (진단 리눅스 부팅 체인) | 체인로드 스크립트 로직 + **진단 리눅스 이미지 실제 제작** + QEMU 로 실부팅 · 에이전트 체크인까지 | dmidecode/ipmitool 실보드 출력 검증(V3 · V8)만 |
| **E1-2** (수집 · 적재) | hardwareSpec 파싱 · 저장 로직 + QEMU 의 `-smbios` 주입값으로 수집 재현 | 실보드의 실제 하드웨어 값 · placeholder 여부 |
| **E1-I** (인프라 관리) | dhcpd/tftp 자산 생성 · 검증 · 서비스 제어 로직 + 진단 이미지 자산 화면 — **대부분 물리 불요** | 격리망 실배포 시 sudoers · 서비스 실동작 |
| **U3** (할당 · 스냅샷) | **전부** — 순수 앱 로직 | 없음 |
| **E2** (펌웨어) | resolve 판정 · 가상 USB 이미지 조립 로직 + QEMU 부팅까지 | **flash 집행 = 실보드 필수** |
| **E3** (BIOS/BMC 설정) | Redfish 클라이언트 로직 + (에뮬레이터 가능 시 일부) | **실 BMC 필수** (에뮬레이터는 E3-R 조사 항목) |
| **E4/E5** (OS 설치 · 후처리) | Kickstart 렌더 · 서빙 로직 + **QEMU 로 무인 설치 E2E 전부** | 실보드 성능 · 드라이버 특이사항만 |

> **초안 관찰**: **"진단 리눅스 이미지 · PXE 인프라 · Kickstart · 가상 USB 조립" 같은 자산 · 서빙 로직은 거의 다 물리 불요(T2 까지)** 로 만들 수 있고, 물리 서버는 **flash(E2) · Redfish(E3) · 실하드웨어 값 확인(E1-2 V3/V8)** 세 곳에 집중된다. 즉 사용자 방향(인프라 먼저)은 로드맵 대부분을 물리 없이 밀어붙일 수 있다는 뜻이다.

### 1-3. 인프라 선개발 = 무엇을 "실제로 만드는가"

로직뿐 아니라 아래 실물 자산을 준비하는 것을 포함한다(E1-R 확정안 기준):

- **진단 리눅스**: Alpine netboot 아티팩트(vmlinuz-lts · initramfs-lts · modloop-lts) + **apkovl.tar.gz**(world 에 dmidecode · ipmitool, firstboot 스크립트) + **부분 apk 저장소**(서명 포함) + **agent.sh**(수집 · 보고 에이전트).
- **PXE 인프라**: dhcpd.conf(iPXE 분기) + tftp 의 ipxe.efi + boot.ipxe(→ 앱 /boot 로 체인).
- **QEMU 랩**: OVMF + iPXE ROM 으로 가상 게스트를 격리망 흉내로 부팅시키는 스크립트 — T2 의 실행 환경.

이것들을 만들어 두면 진단~OS 설치 흐름을 가상 게스트로 반복 검증할 수 있다. **이 준비 작업을 어느 슬라이스(E1-1? E1-I? 별도?)에 담을지가 §5 의 결정 사항.**

그리고 guest server 가 provisioning 을 끝냈는데 다시 진행해야 하는 상황, 회수한 서버를 다시 프로비저닝 되도록 하는 상황을 고려해야 한다.

## 2-R. Claude 응답 — 유스케이스별 현행 대조 · 교정 · 제안 (2026-07-19)

> 기입하신 UC-1~4 와 자유 서술을 종전 토론문서 3편(E-roadmap 1차 · 2차 · decisions)과 현행 엔진 코드에 대조한 결과다. "참고만 해둘 것" 이라 하셨지만 전부 실질적인 설계 입력이었다 — 특히 UC-1 은 **현행 모델로 표현이 불가능한 시나리오**라서 모델 확장 결정이 필요하다.

### 2-R-0. 먼저 바로잡을 현행 동작 2가지 (이후 분석의 전제)

자유 서술에 적어주신 이해와 실제 설계가 두 군데에서 어긋난다. 오해가 아니라 아직 설명된 적이 없는 부분이다.

1. **"게스트가 계속 재부팅하면서 /boot 를 보낸다" → 대기 중 폴링은 재부팅이 아니다.** `/boot` 가 주는 대기 스크립트는 iPXE 안에서 30초 멈췄다가(sleep) 같은 `/boot` 를 다시 부르는(chain) 루프다. 전원이 꺼졌다 켜지는 것이 아니라 화면에 iPXE 카운트만 돌고 있는 상태다. **재부팅이 일어나는 지점은 phase 실행 환경이 바뀔 때뿐이다** — 예를 들어 진단 리눅스가 임무를 끝내고 다음 phase 환경(펌웨어 이미지, OS 설치기)으로 갈아타야 할 때 재부팅 → 다시 PXE → `/boot` → 다음 환경 체인로드. 그래서 걱정하신 "재부팅 루프가 끊기는 경우" 는 대부분 "phase 전환 재부팅에서 network boot 로 복귀하지 못하는 경우"(UC-4)로 좁혀진다.
2. **"/boot 가 다음 스텝을 진행시킨다" → `/boot` 는 상태를 바꾸지 않는다** (읽기 전용 판정 — DEC-2). 커서를 전진시키는 것은 게스트 안의 에이전트가 보내는 **체크인 · step 보고**다. `/boot` 는 "지금 상태에 맞는 부팅 스크립트를 줄 뿐" 이고, 그 스크립트로 부팅된 환경(진단 리눅스 등)이 보고를 보내면 그때 전이가 기록된다. 이 분리 덕에 `/boot` 를 아무리 여러 번 불러도(폴링 · 재부팅 반복) 상태가 오염되지 않는다.

### 2-R-1. UC-1 — 회수 후 재프로비저닝: 현행 모델로 표현 불가, "회차" 도입 필요

**① 시나리오를 현행 구현에 그대로 대입하면 어디서 막히는가.**
1. 완주 시점 — 할당 정의서의 마지막 phase 를 완주하면 completedAt 이 기록되고(DEC-25), 이후 `/boot` 는 "로컬 디스크로 부팅"(iPXE exit)을 준다. 공정 표준상 게스트 NIC 은 DHCP auto 로 전환되어 프로비저닝망을 떠난다(DEC-11 의 사용자 근거).
2. 회수 — 상세 페이지 회수 액션으로 `GuestServer.decommissionedAt` 이 기록되고 배지가 '회수됨' 이 된다. 물리 분리.
3. 요구사항 변경 후 재연결 — PXE 부팅하면 등록이 systemUUID 멱등이라 **기존 서버로 올바르게 인식**되지만, `/boot` 판정 2행이 "decommissioned server. not a provisioning target." 대기 스크립트를 무한 반복한다. **여기서 끝 — 앞으로 갈 길이 없다.**

막히는 이유는 현행 모델의 비가역 3중이다: (a) decommissionedAt 을 해제하는 경로가 없다(회수는 기록 전용 멱등 메서드). (b) ProvisioningProgress 의 상태 변경 메서드 4개(start · advanceTo · markFailed · markCompleted)는 전부 전진 방향 가드라 startedAt · completedAt 을 리셋할 수 없다. (c) 개시 버튼은 미개시(`!isStarted()`)를 요구해 이미 완주한 서버에는 409 로 거절된다. 즉 **UC-1 이 요구하는 것은 새 화면이 아니라 모델 확장이다.**

**② 종전 토론이 이미 준비해 둔 절반.** "요구사항 변경 → 새 정의서" 쪽은 규칙이 있다 — DEC-18: 대기 · 완료 상태 재할당 시 미소비 스냅샷은 대체하고, 실행에 소비된 스냅샷은 이력으로 보존한다(사용자 조건: "이전에 어떤 스냅샷으로 프로비저닝이 수행됐는지의 이력이 삭제되어서는 안 된다"). 없는 절반은 **회수 해제(재투입)** 와 **진행 상태의 2회차 개념**이다.

**③ 설계 방향 후보 3안 비교.**
- **A. 필드 리셋** — 재프로비저닝 시 startedAt · completedAt · failedAt 을 null 로 되돌린다. 1회차의 진행 사실이 통째로 사라져 DEC-18 의 이력 보존 조건과 정면 충돌하고, 전진 전용 가드 계약도 부순다. **탈락 권고.**
- **B. 회차(프로비저닝 라운드) 모델** — ProvisioningProgress 를 "서버당 1행"(현행 guest_server_id UNIQUE)에서 "**프로비저닝 회차당 1행**" 으로 일반화한다. 재프로비저닝 = 새 회차 행을 여는 것이고, 이전 회차 행은 그대로 이력이 된다. 상태 배지 · `/boot` 판정 · 에이전트 게이트는 모두 "현재(최신) 회차" 를 입력으로 삼는다. SetupStep 원장도 어느 회차의 실행인지 구분 축이 필요해진다(현재는 guest_server FK 만). 이력 보존이 구조적으로 자연 충족되고, SetupStep append-only(DEC-3) · 소비 스냅샷 보존(DEC-28)과 같은 철학("과거는 고치지 않고 새 행을 쌓는다")의 연장이다. **권고.**
- **C. 재투입 = 새 GuestServer 등록** — 기존 서버 행은 회수 상태로 박제하고 재연결 시 새 서버 행을 만든다. 이력은 남지만 동일 물리 서버가 목록에 2행으로 나타나고, systemUUID 멱등 등록 가정이 깨지며, 운영자 입력(이름 · 모델 · 시리얼 · 메모)의 연속성이 끊긴다. **탈락 권고.**

B 안의 세부(회차 구분 방식, 원장 · 스냅샷의 회차 연결, DDL)는 해당 슬라이스 CP1 에서 확정하고, 여기서는 방향만 합의한다.

**④ "OS 설치 및 설정 부분을 새롭게 진행" 과 실행 범위의 관계.** 권고 방향은 **항상 진단부터 전체 재실행**이되, 펌웨어(E2)의 resolve 판정에 "게스트의 현재 펌웨어가 이미 target 과 같으면 flash 를 건너뛴다" 규칙을 추가하는 것이다 — 현재 설계의 skip 사유에는 이 축이 없다('enabled 펌웨어 0개' · '파일 부재' 만 열거). 이러면 2회차에서 펌웨어 phase 는 자동으로 스치듯 지나가고 실질 작업은 OS 설치 · 설정만 남는다 — 기대하신 결과("OS 부분만 새로")가 "정의서 변경분 diff 판정" 같은 새 복잡도 없이 자연히 나온다. 진단 재실행은 어차피 필요하다: 회수 · 보관 중 하드웨어가 바뀌었을 수 있고, 위 skip 판정의 입력(현재 펌웨어 버전)을 진단이 수집한다.

**파생 질문**
- **Q-UC1-1.** ④ 의 방향(항상 진단부터 전체 재실행 + E2 는 '이미 최신이면 skip' 판정으로 자연 단축)에 동의하는가? 아니면 재프로비저닝 시에도 무조건 재-flash 하는 공정인가?

> BMC 는 최신이 기준이 아니라 할당된 세팅 정의서 기준으로 이미 동일한 버전 또는 해당되는 조건인 경우 skip 으로 한다.
> BIOS 는 무조건 세팅 정의서 기준으로 재 flash. 커스텀 bios 파일을 사용하므로 버전만으로 판단할 수 없기 때문.

- **Q-UC1-2.** 회수 해제(재투입)는 상세 페이지의 **명시 버튼**으로 두는 것을 권고한다 — 재연결만으로 자동 복귀시키면 "회수했는데 케이블이 남아 있으면 마음대로 되살아나는" 오동작 표면이 생긴다(개시 버튼 DEC-26 과 같은 명시 액션 철학). 동의하는가?

> 동의.

- **Q-UC1-3.** 회수 · 보관 중 NIC 교체 같은 하드웨어 변경이 실제로 일어나는가? MAC 이 바뀐 채 재등장하면 NIC 바인딩(host_mac UNIQUE) 갱신 처리가 필요하다. 디스크에 이전 OS 가 남은 채 재투입되는 경우의 디스크 초기화 정책도 함께 알려달라.

> NIC 교체는 가끔 발생하긴 하나 무시할 수 있는 수준은 아니다.
> OS 재설치의 경우 디스크 초기화 후 진행한다. 이 때 대부분의 경우 RAID card 를 별도로 사용하고 있으므로 진단 리눅스 또는 RAID 생성 단계에서 관련 명령어를 사용하여 RAID configuration 을 Clear 한 후 다시 구성하는 식으로 진행.

### 2-R-2. UC-2 — 프로비저닝 도중 중단: '중단' 운영자 액션 신설 후보

현행에는 중단 수단이 없다. 회수 액션이 있으나 의미가 다르다 — 회수는 "이 서버는 운용에서 뺀다"(창고행)이고, 중단은 "이번 프로비저닝을 그만둔다" 이다. 회수를 중단 용도로 쓰면 의미가 오염되고, 현행 회수는 비가역이라 더 위험하다.

**제안**: 운영자 액션 '중단' 신설 — 진행 중(PROVISIONING) 상태에서만 활성(UI 1차 차단 원칙, disabled + tooltip). 효과: ① 현재 회차를 중단 종결로 박제(실패 · 완주와 구분되는 제3의 종결). ② 다음 `/boot` 접촉부터 "중단됨 · 운영자 처분 대기" 스크립트. ③ 에이전트 보고는 기존 게이트(requireProvisioning)가 자동으로 거절 — 이미 있는 안전망이 그대로 동작한다. ④ 재개는 회차 모델에서 **새 회차 개시** = 재프로비저닝과 동일 경로. "중단 지점부터 이어하기" 는 만들지 않는다 — phase 실행 환경이 이미 날아간 뒤라 이어하기의 실익이 없고 복잡도만 크다.

**한계를 정직하게**: 게스트가 phase 실행 한복판(예: OS 설치기 구동 중)이면 `/boot` 를 부르지 않으므로 중단 표식은 **다음 접촉 시점에야 효력**이 있다. "지금 즉시 강제 정지" 는 BMC 전원 제어(§2-R-6)가 생긴 뒤에나 가능 — T3 예약 목록으로 적립.

- **Q-UC2-1.** 이 방향(중단 = 회차 박제 + 다음 접촉부터 차단, 재개 = 새 회차, 즉시 강제 정지는 E3 이후)에 동의하는가?

> 동의.

### 2-R-3. UC-3 — 개시를 안 눌렀는데 눌렀다고 착각: 게스트는 정상, 문제는 "보이지 않음"

**현행 사실**: 이 상황에서 게스트는 고장이 아니다. `/boot` 판정 5행이 개시 대기 스크립트를 주고, 게스트는 iPXE 폴링 루프(30초 간격, 재부팅 없음 — §2-R-0)로 무한 대기한다. 개시 버튼을 누르는 즉시 다음 폴링(최대 30초 안)에 진단 체인로드로 넘어간다. 즉 흐름 로직은 이 유스케이스를 이미 안전하게 처리하고 있다.

**갭은 운영자 쪽 가시성이다.** 상세 페이지 배지는 '등록됨' 이지만, ① 게스트가 지금 접촉해 오고 있는지(케이블이 꽂혀 폴링 중인지 / 꺼져 있는지) ② "당신이 개시를 눌러야 진행된다" 는 사실 — 둘 다 화면에 없다. 그래서 "눌렀다고 착각하고 서로 기다리는" 상황이 가능해진다.

**제안**: ① **게스트 접촉 시각 기록**(마지막 접촉 시각 — `/boot` 접촉 · 에이전트 보고 시 갱신) + 상세/목록에 "접촉 중(N초 전)" · "무접촉(N분째)" 표시. ② '등록됨 + 접촉 중 + 미개시' 조합일 때 상세 페이지에 개시 유도 안내(개시 버튼 강조 + "게스트가 개시를 기다리는 중"). dispatch 읽기 전용 원칙(DEC-2)과의 관계 — 접촉 기록은 판정 입력이 아니라 관찰 로그라 원칙 훼손이 아니다. 다만 게스트당 30초마다 UPDATE 1건이 생기는 비용은 CP1 에서 계량한다.

- **Q-UC3-1.** 접촉 시각 기록 + 표시 방향에 동의하는가? (E1-1 과 E1-2 중 어느 슬라이스에 배정할지는 plan 에서 정한다)

> 동의.

### 2-R-4. UC-4 — network boot 복귀 실패(BIOS setup 진입 등): 탐지는 접촉 공백, 복구는 사람(당장) → BMC(장래)

**현행 사실**: 게스트가 network boot 를 못 타면 서버 입장에서는 **침묵**만 남는다 — `/boot` 도 에이전트 보고도 오지 않는다. 종전 결정의 대응은 무보고 실패 규약(DEC-4): 경과 표시 + 운영자 수동 FAILED 전환. 상태 데이터는 오염되지 않는다 — 전이는 보고로만 일어나기 때문(§2-R-0-2).

**제안**: 탐지는 UC-3 과 같은 접촉 시각 축("무접촉 N분")을 그대로 쓴다 — 두 유스케이스가 같은 관찰 인프라를 공유한다. 복구는 당장은 현장 조치(재부팅 + 부팅 순서 복원)이고, 장래에는 Redfish 의 BootSourceOverride(다음 1회 부팅을 PXE 로 강제) + 전원 재인가가 정석이다 — 실 BMC 필요라 T3 예약 목록으로. 참고로 "사원이 BIOS 세팅을 바꿔놓는" 문제 자체는 E3(FIRMWARE_SETTING)가 세팅을 다시 밀어 넣는 단계라 회복 경로가 로드맵 안에 이미 있다.

### 2-R-5. UC-5 (자유 서술에서 승격) — 물리 서버 ↔ 게스트 서버 식별(여러 대 동시 연결)

**문제 재정리**: LAN 선이 구분되어 있지 않아 여러 대를 동시에 꽂으면, 목록의 게스트 서버 행이 실물 어느 섀시인지 사람이 알 수 없다 — systemUUID 와 IP 는 실물 겉면에 붙어 있는 정보가 아니기 때문이다. 정확한 지적이고, 데이터센터 운영에서 실재하는 고전 문제다.

**후보 4가지** (병행 가능, 아래로 갈수록 구현 비용 증가):
1. **등록 순서 운영 절차** (도구 불요, 지금 가능): 한 대씩 켜면 등록 시각 순서 = 물리 순서. 다대 동시엔 무력하지만 소수 입고에는 실용적.
2. **진단 리눅스 콘솔 배너** (E1-1 자산으로 흡수 가능, 물리 불요): 게스트 화면(모니터)에 자기 식별 정보(서버명 · systemUUID 끝자리 · 부여된 IP)를 큰 글씨로 표시 — agent.sh 또는 부팅 스크립트가 출력. 실물에 모니터를 꽂아 보면 즉시 매핑된다.
3. **시리얼 넘버 대조** (E1-2 수집으로 자동): dmidecode 가 수집한 섀시 시리얼 ↔ 실물 섀시 스티커 대조. 수집 적재가 되면 목록/상세에서 검색 가능해진다.
4. **식별 램프 점멸** (Redfish IndicatorLED — T3 예약): 상세 페이지 버튼 → 해당 실물의 UID LED 점멸. 데이터센터 표준 기법이지만 실 BMC 필요.

- **Q-UC5-1.** 배분 제안 — 2(콘솔 배너)는 E1-1 자산 요구사항에 포함, 3은 E1-2 기존 계획으로 소화, 4는 T3 체크리스트 적립, 1은 운영 문서에 한 줄. 동의하는가?

> 동의.

### 2-R-6. 사견(Redfish 로 서버 끄기 · 켜기) 응답

방향에 동의한다. 진단(E1-2)이 수집한 BMC 주소로 Redfish 전원 제어(끄기 · 켜기 · 재부팅 · 다음 부팅 PXE 강제)를 확보하면 — UC-2 의 즉시 강제 정지, UC-4 의 원격 복구, phase 전환 재부팅의 신뢰성(게스트 자율 재부팅이 실패하면 서버가 밀어붙임)이 한꺼번에 좋아진다. 다만 실 BMC 없이는 한 줄도 검증할 수 없는 전형적 T3 영역이라, 로드맵 위치는 **E3(Redfish 클라이언트가 처음 생기는 단계) 이후의 강화 확장**으로 두고 Q-C 체크리스트에 적립한다. 중요한 것은 현행 E1~E5 흐름이 전원 제어 없이도 성립한다는 점 — 이것은 전제가 아니라 강화다.

## 3. 유스케이스 → 로직 영향 분석  (2026-07-19 1차 매핑 — §2-R 근거)

- **dispatch 매트릭스** (E1-0b — `/boot` 가 상태별로 주는 스크립트 7행): UC-1 — 판정 입력을 "현재 회차" 기준으로 재정의(이전 회차가 완주였어도 새 회차가 열리면 exit 가 아니라 개시 대기 → 진단 위임으로), 회수 행은 재투입 액션 후 통과. UC-2 — '중단됨' 행 신설(중단 회차 → 처분 대기 스크립트). UC-3 · UC-4 — 변화 없음(현행 응답이 이미 올바름 — 문제는 관찰성).
- **PhaseSequence** (E1-0a — 다음 phase 결정: 보유 필터 · 종단): **변화 없음** — 회차 안에서의 순서 규칙은 모든 UC 에서 동일하다.
- **개시 게이트 / 재개시 / 재프로비저닝** (started_at 모델): UC-1 · UC-2 — "재개시" 를 따로 만들지 않고 **재프로비저닝 = 새 회차의 최초 개시** 로 일원화. 개시 가능 조건은 "회수 아님 + 현재 회차 미개시" 로 재정의. UC-3 — 게이트 자체는 유지, 개시 유도 UI 보강.
- **U3 할당 · 스냅샷** (한 서버에 정의서 재할당 · 부분 적용): UC-1 — DEC-18 그대로(재할당 = 미소비 스냅샷 대체 + 소비분 보존). 추가 결정 1건: 회수 상태에서 재할당 차단(재투입 후에만 — disabled + tooltip).
- **실패 · 재시도 의미론** (failedAt · 운영자 액션): UC-2 — 중단과 실패를 별개 종결로 구분(실패 = 게스트가 보고한 사실 / 중단 = 운영자 의사). UC-4 — 무보고 침묵은 DEC-4(경과 표시 + 수동 FAILED 전환) 유지, 접촉 시각이 경과 판단의 입력이 된다. 재시도(같은 회차 안에서 실패 지점 재개 — E1-2 예정)와 재프로비저닝(새 회차)은 별개 액션.
- **동시성** (여러 게스트 동시 진행): 흐름 로직 충돌 없음 — UC-5 의 본질은 동시성 버그가 아니라 물리 식별 UX 다. §2-R-5 의 4후보로 대응.
- **관찰성 (신설 축)**: UC-3 · UC-4 가 공통으로 요구 — 게스트 접촉 시각 기록 + "접촉 중 / 무접촉 N분" 표시. 기존 6축에 없던 축이라 추가한다.

## 4. 도출된 변경 사항  (확정 — 2026-07-19 사용자 답변 반영, 토론 종결)

> §2-R 파생 질문 전건에 답변이 달려 이 라운드를 종결한다. 후보 C-1~C-7 을 결정 번호 **DEC-29~35** 로 고정한다(E-roadmap decisions 문서의 연번 승계). 상세 설계는 각 담당 슬라이스 CP1 로 넘긴다. 유스케이스가 더 떠오르면 §2 에 계속 추가하면 된다 — 문서는 열려 있다.

- **DEC-29 (←C-1). 회차(프로비저닝 라운드) 모델** — ProvisioningProgress 를 서버당 1행에서 회차당 1행으로 일반화. 재프로비저닝 · 중단 후 재개 · 실패 후 처음부터 다시 — 전부 "새 회차" 하나의 개념으로 수렴. [UC-1 · UC-2]
- **DEC-30 (←C-2). 재투입(회수 해제) 명시 버튼** — 상세 페이지 명시 액션. 재연결만으로 자동 복귀하지 않는다(개시 버튼 DEC-26 과 같은 명시 액션 철학). [UC-1, Q-UC1-2 동의]
- **DEC-31 (←C-3). 중단(abort) 운영자 액션 + 중단 종결 상태** — 진행 중에만 활성(UI 1차 차단), 실패 · 완주와 구분되는 제3의 회차 종결, 효력은 다음 접촉부터, 재개 = 새 회차. 즉시 강제 정지는 범위 밖(DEC-35 로 적립). [UC-2, Q-UC2-1 동의]
- **DEC-32 (←C-4). 게스트 접촉 시각 기록 + 표시** — `/boot` 접촉 · 에이전트 보고 시 갱신, "접촉 중 / 무접촉 N분" 표시 + 미개시 유도 안내. 스코프 배정(E1-1 vs E1-2)은 E1-1 CP1 에서. [UC-3 · UC-4, Q-UC3-1 동의]
- **DEC-33 (←C-5). 진단 리눅스 콘솔 식별 배너** — E1-1 자산 요구사항에 포함(서버명 · systemUUID 끝자리 · 부여 IP 를 게스트 화면에 표시). 시리얼 대조는 E1-2 기존 계획으로, IndicatorLED 는 DEC-35 로, 등록 순서 절차는 운영 문서 한 줄로. [UC-5, Q-UC5-1 동의]
- **DEC-34 (←C-6, 사용자 답변으로 수정 확정). E2 재-flash / skip 판정** — 기준은 "최신" 이 아니라 언제나 **할당 정의서가 resolve 한 target** 이다. 게스트 현재 버전과 대조하는 skip 은 **BMC 펌웨어에만** 적용한다(정의서 기준 동일 버전 또는 해당 조건이면 skip). **BIOS 는 skip 없이 무조건 재-flash** — 커스텀 BIOS 파일을 쓰므로 버전 동일이 내용 동일을 보증하지 않기 때문. 진단(E1-2)이 수집하는 현재 펌웨어 버전은 BMC 대조의 입력이 된다. [UC-1, Q-UC1-1 답변]
- **DEC-35 (←C-7). T3 예약 목록 적립** — Redfish 전원 제어 · BootSourceOverride(다음 부팅 PXE 강제) · IndicatorLED 는 E3 이후 강화 확장으로 `docs/T3-checklist.md` 에 적립. [UC-2 · UC-4 · UC-5]

**미결 — 신규 발견, 후속 조사 (Q-UC1-3 답변에서)**
- **OPEN-1. RAID 초기화 · 재구성 단계** — 실태: OS 재설치 시 디스크 초기화는 **RAID configuration Clear 후 재구성**으로 진행하며 대부분 RAID 카드를 사용한다. 결정할 것: ① 어느 phase/step 에 배치하나(진단 리눅스 내 step vs OS 설치 직전) ② 벤더 RAID CLI(storcli 등 — Alpine 공식 저장소에 없는 독점 바이너리)를 진단 리눅스 자산에 어떻게 동봉하나 ③ QEMU 의 RAID 카드 에뮬레이션(megasas)으로 어디까지 검증 가능한가. **E4(OS 설치)의 선행 요구**이므로 E4 로드맵 확정 전까지 결론 낸다 — E1-1 의 선행 요구는 아니다.
- **OPEN-2. NIC 교체 재등장 처리** — 실태: NIC 교체가 무시 못 할 빈도로 발생. MAC 이 바뀐 채 재등장하면 NIC 바인딩(host_mac UNIQUE) 갱신이 필요하다. 재투입(DEC-30) · 회차(DEC-29) 담당 슬라이스에서 함께 설계한다.
