## T1. 모델 · 시리얼 각인의 분리 (구 §5-1)

**첨언 요지 확인.** 공정 특성상 modelName/serialNumber 를 프로비저닝 초기에 입력할 수 없다. 따라서 IPMI 각인은 E1 의 인라인 단계가 아니라, **프로비저닝 완료 이후 운영자가 독립적으로 실행하는 '모델 · 시리얼 기록' 액션**으로 분리한다. 이때도 진단 리눅스(Alpine)를 메모리에 적재해 수행하는 설계는 유지한다.

**분리 액션의 동작 스케치 (제 해석 — 맞는지 확인 요청).** 운영자가 게스트 서버 상세에서 modelName/serialNumber 를 입력하고 "모델 · 시리얼 기록" 을 실행하면 → 서버에 '각인 임무 대기' 표식이 저장되고 → 대상 서버가 네트워크 부팅으로 재진입하면 `/boot` 가 커서와 무관하게 각인 임무용 진단 리눅스 스크립트를 반환 → 에이전트가 ipmitool 로 각인 → 보고 → 표식 해제, 서버는 원래 상태(로컬 부팅)로 복귀.

이 그림에서 각인 임무는 **커서를 움직이지 않는 '부수 임무'** 로 모델링하는 것이 자연스럽다. 프로비저닝이 끝난(또는 진행 중이 아닌) 서버의 currentPhase 를 되돌리면 1차 문서 §4-D2 의 "phase 역행 금지" 불변식과 충돌하기 때문이다. `SetupStep` 원장에는 IPMI_SETTING 행이 그대로 남으므로(enum 상수 재사용) 이력 가시성은 유지된다.

**파생 질문 4개:**

1. **실행 가능 시점** — '완료 이후'를 어디까지 좁힐지. (권고: "프로비저닝 진행 중이 아니고 FAILED 도 아닌 서버" 전부 — 등록만 된 서버 · 진단만 완주한 서버 · 완료 서버 모두 허용. 공정상 값이 나오는 시점이 유동적이므로 넓게 여는 편이 사용성에 부합.)
2. **재부팅 트리거** — OS 설치가 끝난 서버는 로컬 부팅 상태라 iPXE 폴링이 더는 돌지 않는다. 각인 임무를 받으려면 네트워크 부팅으로 재진입해야 하는데, 그 방법이 (a) 운영자가 현장에서 수동 재부팅+부트 메뉴(F12) (b) BMC 를 통한 원격 재부팅+부트 디바이스 지정(ipmitool chassis bootdev pxe / Redfish) 둘 중 하나다. (b)는 E3(자격증명 · Redfish)에 의존하므로, **각인 액션의 1차 구현은 (a) 수동 재부팅 안내로 시작하고 (b)는 E3 이후 자동화**를 권고한다. 수용 여부?
3. **개시 게이트의 거취** — 각인이 E1 에서 빠지면서 1차 문서 §5-1 의 (a) 명시 개시 버튼의 원동기(입력 미비 차단)가 사라졌다. 새 권고: **별도 개시 버튼을 두지 않는다.** 진단(E1)은 등록 즉시 자동 진입해도 무해하고(정보 수집만 하며 하드웨어를 바꾸지 않음), 하드웨어를 실제로 바꾸는 단계(E2 이후)는 **정의서 할당의 존재 자체가 게이트** 역할을 한다(할당 전엔 어차피 대기 루프). 버튼 하나가 줄어 단순 구현 본 동기에도 부합한다. 동의하는지?
4. **분리 액션의 인벤토리 코드** — 각인 액션(UI + 임무 표식 + 각인 지시 · 보고)은 어느 슬라이스로 둘지. 진단 리눅스 임무 체계를 재사용하므로 **E1 의 마지막 하위 슬라이스(E1-3 을 '각인 액션' 슬라이스로 재정의)** 가 자연스럽다는 것이 제 제안이다 — E1-3 의 원래 내용(완주 판정 · 대기)은 E1-2 에 흡수될 만큼 작아진다. 대안: 운영 액션이므로 U 트랙(U4). 어느 쪽으로?

> 우선 진단 리눅스에서 modelName / serialNumber 를 새기는 작업은 조금 더 고민을 해봐야 하는 상황인 듯하다.
> 공정 표준 세팅 상 모든 네트워크 카드는 마지막에는 DHCP auto 로 나가야 하며, BMC 는 IP 를 고정해야 하기 때문. 즉, 세팅 정의서에 따라 provisioning 이 종료된 후 최후에는 이 네트워크와 BMC 연결을 끊게 되는 것.
> 이를 되돌릴 수 있는 방법이 생각나지 않고, 자동화가 되지 않는 한 의미 없는 것이며, 어차피 provisioning 이후 사람이 마지막으로 guest server 를 직접 점검하는 업무 프로세스가 있으므로 현 시점에서는 진단 리눅스를 통해 modelName / serialNumber 를 새기는 작업의 진행이 어색하다는 것이 그 근거이다.

## T2. "모의 게스트 하네스" 설명 (구 §5-2 후속)

**하네스(harness)란**: 물리 서버 없이 게스트 서버의 HTTP 행동을 흉내내는 **셸 스크립트 묶음**이다. 게스트 pull 구조에서 게스트가 하는 일은 전부 HTTP 요청(부팅 스크립트 요청 → 체크인 → 보고)이므로, curl 만으로 실제 게스트와 구별 불가능한 시퀀스를 재연할 수 있다. 예를 들어 E1 검증용 스크립트는 대략 이런 순서다:

```bash
# 1. 부팅 흉내 — 등록/dispatch 확인
curl "http://localhost:7779/api/pxe/v1/boot?uuid=...&mac=...&ip=...&vendor=...&product=..."
#    → 응답 바디가 '진단 리눅스 체인로드 스크립트'인지 눈으로/grep 으로 확인
# 2. 에이전트 체크인 흉내
curl -X POST .../checkin -H "X-Guest-Token: <부팅 응답에 있던 토큰>"
# 3. 수집 보고 흉내 (준비된 견본 JSON — dmidecode 출력 상당)
curl -X POST .../report -d @sample-hardware-spec.json ...
# 4. 상세 페이지를 열어 step 이력 · 수집 값 반영을 확인
```

즉 **CP5 가 "브라우저 E2E" 대신 "하네스 실행 + 상세 페이지 확인"** 이 된다는 뜻이었다. 스크립트는 게스트 관점의 시나리오(정상 완주 / 실패 보고 / 중복 보고)별로 하나씩 쌓이고, 슬라이스가 진행될수록 E2 · E4 시나리오가 추가된다.

**질문: 저장 위치.** 프로젝트 최상위에 `scripts/mock-guest/` 디렉토리 신설을 권고한다(plan/ · report/ · discussion/ 과 나란한 git 추적 자산). 대안은 `tools/`. **어느 쪽으로 할지, 그리고 이 CP5 예외를 CLAUDE.md 에 명문화하는 문구는 토론 종결 시 마지막 정리 문서에서 확정하는 것으로 좋은지?**

> 저장위치 및 예외 정리 제안 수용.

## T3. PXE 인프라의 앱 내 관리 (구 §5-3)

Notion 'PXE Server 구축' 페이지를 실사했다. 현행 수동 인프라는 격리망 `192.168.1.10` 위의 세 시스템 서비스다: **dhcpd**(IP 부여 + iPXE 분기: 최초 부팅→TFTP `ipxe.efi`, iPXE 환경→`http://192.168.1.10/boot.ipxe`), **tftp**(ipxe.efi 1개 파일 서빙), **httpd**(정적 `boot.ipxe` 메뉴 + OS 이미지 트리 + kickstart + 펌웨어 FAT32 이미지 서빙).

"Provisioning Server 내 관리" 는 정도에 따라 세 단계로 나뉘고, 각각 비용이 다르다:

- **1단계 — 역할 흡수 (이미 로드맵에 포함, 추가 결정 불요).** 정적 `boot.ipxe` · `rocky9.ks` 를 앱 endpoint 가 대체한다. dhcpd 설정의 filename 한 줄을 `http://192.168.1.10:<port>/api/pxe/v1/boot` 로 바꾸면, 이후의 모든 분기 · 메뉴 · kickstart 는 앱이 동적으로 생성한다(E1-0b · E4 의 본체). dhcpd/tftp 는 그대로 두되 역할이 "앱까지 데려다주는 것" 으로 축소된다. OS 이미지 같은 대용량 정적 트리는 httpd 에 남길지 앱이 서빙할지 E4-0 에서 결정.
- **2단계 — 구성 관리 UI (사용자 희망 사항 = 이것으로 이해).** dhcpd.conf · tftp 자산을 앱이 생성 · 배포하고, 서비스 기동 상태 · DHCP 임대 현황을 관리 페이지에서 보고, 설정 변경(대역 · 서버 IP)을 폼으로 한다. 기술 제약: 앱은 비 root 로 돌므로 systemctl restart dhcpd 같은 제어는 **한정된 sudoers 규칙**(특정 명령만 무비밀번호 허용) 또는 polkit 규칙이 필요하다 — 구축 문서에 1회 기재할 항목이지 앱 복잡도는 크지 않다.
- **3단계 — DHCP/TFTP 를 Java 로 내장.** 관리 포인트는 정말 1개가 되지만, 특권 포트(67/69) 바인딩 · 프로토콜 구현 검증 부담 · 앱 장애 = 프로비저닝망 전체 마비라는 결합을 만든다. **탈락 권고** — 이론적 가치가 구현 복잡도를 정당화하지 못한다(본 동기).

**권고: 1단계(로드맵 내장) + 2단계(신규 슬라이스).** 2단계는 자가 점검 · 복구 성격이 Maintenance 와 닮았지만 대상이 자원이 아니라 인프라이므로, **E 트랙의 독립 슬라이스 `E1-I`(Infrastructure) 로 신설**하는 안을 제안한다. 우선순위는 E1 완주 뒤로 미뤄도 된다 — 1단계만으로 E1 착수에 지장이 없다(수동 dhcpd 변경 1회).

**질문: ① 2단계 채택 여부와 코드(`E1-I` vs MK 트랙) ② 우선순위(E1 완주 후 vs U3 앞) ③ 진단 리눅스 이미지(Alpine 빌드 산출물)의 관리 주체** — 첨언대로 MA 자원 도메인은 부적절하다는 데 동의한다. 대안: 마커 · 무결성 검증은 재사용하되 **lifecycle 액션(비활성 · 삭제) 없이 '교체(replace)' 만 가능한 별도 시스템 자산 화면**(E1-I 에 포함). 이 방향이면 충분한지?

> 수용.

## T4. U3 스냅샷 — 설계 세부 질문 (구 §5-4)

스냅샷 채택과 "별도 테이블 필요" 전망에 동의한다. 세부 4개를 여쭌다 (U3 CP1 전 방향만 합의 목적):

1. **스냅샷 시점** — (a) **할당 시 즉시 복사** (권고: 단순하고, "할당했을 때 본 그대로 실행된다" 는 운영자 직관과 일치) vs (b) 첫 소비 phase 진입 시 복사(할당~실행 사이의 정의서 수정이 반영되나, 무엇이 실행될지 할당 시점에 알 수 없어짐).

> a 로 진행.

2. **테이블 구조** — (a) **`setting_process` 행 단위 복사** (process_type + payload JSON 을 할당 FK 아래 복제 — 권고: 실행기가 원본과 동일한 구조를 읽음) vs (b) 정의서 전체를 JSON 한 덩어리로.

> a 로 진행.

3. **재할당 규칙** — 진행 중 재할당은 UI 1차 차단 유지, 대기 · 완료 상태에서 재할당하면 기존 스냅샷 폐기 + 새로 복사. 이견 없는지?

> 기존에 할당된 스냅샷 자체는 없어지고 새로운 스냅샷으로 대체되는 것에 동의하나, 그 guest server 가 이전에 어떤 스냅샷을 통해 provisioning 이 수행되었는지에 대한 이력이 삭제되어서는 안된다. 이러한 이력은 다른 테이블에서 관리되는 것으로 알고 있으므로 프로그래밍 과정에서 주의만 하면 될 것으로 생각한다.

4. **원본 정의서 삭제 정책** — 스냅샷이 자립하므로 기술적으로는 원본 삭제를 허용할 수 있다(스냅샷에 유래 정의서 이름을 문자열로 보존). 다만 "이 서버가 어떤 정의서로 세팅됐는지" 를 원본 상세로 이동해 보는 동선은 끊긴다. (a) **삭제 허용 + 이름만 보존** (권고: 스냅샷 채택의 이점을 온전히 취함) vs (b) 할당 존재 시 삭제 차단 유지.

> 자원에서 soft-delete 를 통해 휴지통에 넣어두고 복구했던 것처럼, 삭제 액션이 purge 를 의미하지는 않도록 진행하는 것이 나을 것으로 보임.
> 어쨌든 어떤 구체적인 이력으로 세팅이 진행되었는지 확인할 수 있는 방법은 존재해야 한다고 생각.
> 다만 이 또한 이력이 다른 테이블에 존재하므로 크게 설계 상 신경 쓸 포인트는 아닐 수도 있다는 생각.
> 삭제 정책으로 넘어와서, 원본 정의서를 삭제할 때 스냅샷에 의해 provisioning 이 진행, 대기 중 또는 중단 상태인 guest server 가 존재할 경우 목록에서 완전히 없애는 것이 아닌, 비활성화 처리가 된 것처럼 단순 조회만 가능하도록 남겨두고, 해당 스냅샷의 guest server 가 모두 provisioning 이 완료된 경우에는 그 정의서가 더 이상 목록에서 보이지 않고 DB 에서도 삭제되도록 하는 것이 좋을 것 같음.

## T5. 게스트 pull 이 전 구간에서 어떻게 도는가 (구 §5-6)

"최초 등록에서만 그림이 그려진다" 는 첨언에 대한 설명이다. 핵심은 **게스트 쪽에 상주 엔진이 없고, 세 가지 형태의 pull 이 반복될 뿐**이라는 것:

- **(가) iPXE 폴링** — OS 가 없는 동안 서버는 매 부팅마다 `/boot` 를 부른다. 할 일이 없으면 응답 스크립트가 `sleep 30 + chain(/boot)` 이므로, 사실상 30초 주기 폴링이 된다. **서버가 명령을 보내는 게 아니라, 게스트가 계속 "다음 할 일?" 하고 묻는다.**
- **(나) 진단 리눅스 에이전트 루프** — 진단 리눅스로 부팅한 동안은 에이전트(셸 스크립트)가 체크인 → 지시 수신 → 실행 → 보고 → 다음 지시 수신을 반복하다가, 서버가 "이 phase 끝, 재부팅" 을 주면 재부팅한다. 재부팅하면 다시 (가)로 돌아간다.
- **(다) kickstart 콜백** — OS 설치 중에는 anaconda 가 주도권을 가지므로, 우리가 끼어들 자리는 kickstart 의 `%post` 스크립트뿐이다. 그 안의 curl 이 설치 완료(와 OS 후처리)를 보고한다.

부팅 단위 타임라인으로 전체를 이으면:

| 부팅 | `/boot` 가 주는 것 | 게스트에서 벌어지는 일 |
|---|---|---|
| #1 | (등록됨) 대기 스크립트 | 30초 폴링 — 운영자가 정의서를 할당할 때까지 |
| #2 | 진단 리눅스 커널+initrd | (나) 루프: 수집 → 보고 → "재부팅" 지시 |
| #3 | 펌웨어 임무 스크립트 (E2-R 결정 방식) | flash → 자동 리셋 |
| #4 | 원장에 미종결 flash 축 → 진단 리눅스 재진입 | 버전 재수집 보고 → 완주 판정 → 재부팅 |
| #5 | 설치 커널 + `inst.ks=<앱 URL>` | anaconda 무인 설치 → `%post` 의 curl 보고 → 재부팅 |
| #6 | "로컬 디스크로 부팅" (exit) | 설치된 OS 기동 — 이후 `/boot` 호출 없음 |

이 표에서 §5-6 의 질문은 **부팅 #5 와 #6 사이** 의 문제다: OS 후처리(OS_SETTING — SELinux · 서비스 · 추가 패키지)를 (A) `%post` 안에서 설치와 함께 끝낼 거냐, (B) 설치된 OS 의 첫 부팅에 실행되는 서비스(first-boot 에이전트)를 심어 거기서 할 거냐.

**수정 권고 — (A) `%post` 병합 + 별도 보고.** 원래 (B)의 장점으로 꼽던 "OS_SETTING 이 원장에 별도 step 으로 남는 가시성" 은, `%post` 스크립트가 OS_SETTING 의 시작/종료를 **각각 curl 로 따로 보고**하면 (A)에서도 그대로 얻어진다. 반면 (B)는 설치된 OS 에 상주물을 심고(제거 시점 관리 필요) 구현 층이 하나 는다. E6(TESTING — first-boot 의 또 다른 용도 후보)이 보류된 지금은 (A)의 우세가 더 커졌다. 남는 (A)의 진짜 약점은 하나: **OS_SETTING 만 실패했을 때 재시도 단위가 'OS 재설치부터'** 가 된다는 것.

**질문: 이 트레이드오프(후처리 단독 재시도 불가 — 재설치로 재시도)를 수용하고 (A)로 확정해도 되는지?**

> 대부분의 경우에서 OS 후처리로 그렇게 많은 작업을 진행하지는 않으므로 애당초 post script 영역에서 실패가 발생할 위험이 적을 것으로 보이나, 일부 상황의 경우 패키지 매니저를 통해 특정 패키지를 설치하거나, local repo 를 구성하거나 하는 등 복잡한 작업이 들어가는 경우가 존재한다.
> 이 부분은 애초에 OS_SETTING 에서 좀 더 구체적으로 작업을 진행할 수 있도록 설계 및 구현을 했어야 하나, 우선은 MVP 제작 목적이었으므로 별도로 언급은 하지 않고 건너 뛴 것이었다. 그러나 생각보다 구현이 잘 되고 있는 상황이어서, 해당 부분은 U 단계를 신설하거나 기존 단계에서 하위 단계를 파생시켜서 OS_SETTING 에 대한 세팅 정의서를 보다 커스텀 가능하도록 만드는(지금 당장 머릿속으로는 커스텀 bash script 를 넣는 방식. 실행 스크립트 자체이므로 위험할 가능성이 충분히 있긴 하다.) 단계와 병행하는 것이 좋을 것으로 보인다.
> A 와 B 안을 하이브리드로 가져가서, 간단한 os 후처리의 경우에는 A 안으로, 복잡한 안의 경우 (즉, 병행되는 U 단계에서 'OS 후처리 커스텀 bash script 사용'을 체크 또는 적용) B 안으로 가져가서 정상적으로 처리되었는지 확인하는 것으로 진행하는 것이 좋아 보임.

## T6. 네트워크 구성 — 사내망 접속과 프로비저닝망의 공존 (구 §5-10)

네트워크 지식이 없다고 하셔서 현행 구성 기준으로 풀어 설명한다. 'PXE Server 구축' 페이지의 구성은 이미 **격리망**이다: PXE 서버가 `192.168.1.10`(게이트웨이 = 자기 자신)이고, 게스트들은 `192.168.1.100~200` 을 받는다. 이 대역은 사내망과 물리적으로 분리된 스위치/케이블 위에 있다.

**권장 구성은 "NIC(랜 포트) 2개"** 다. 배포 서버(Rocky 9.x)에:

- **NIC A — 사내망**: 사내 네트워크 체계(DHCP 또는 네트워크 담당자가 주는 고정 IP)를 따른다. 사내 와이파이에 붙은 노트북이 Provisioning Server 웹 UI 에 접속하는 경로가 이것이다. 와이파이와 유선 사내망은 보통 같은 네트워크이므로 "와이파이로 접속" 은 특별한 설정 없이 된다 — 필요한 것은 이 NIC 의 IP 를 고정해 달라고(또는 DNS 이름을 달아 달라고) 요청하는 것뿐이다.
- **NIC B — 프로비저닝 격리망**: 지금의 `192.168.1.10` 그대로. dhcpd 는 이 NIC 에만 응답하므로(설정의 subnet 선언이 이 대역 한정) **사내망의 DHCP 와 충돌하지 않는다** — 이것이 격리를 유지해야 하는 실질적 이유다. 프로비저닝 대상 서버들은 이 스위치에만 꽂는다.

Spring 앱은 기본값(0.0.0.0 바인딩)으로 두면 두 NIC 모두에서 접근된다 — 운영자는 NIC A 주소로 UI 를, 게스트는 NIC B 주소로 `/boot` 를 부른다. `192.168.0.x 에 배포` 는 불필요하다: 사내망 쪽 주소는 사내 체계를 따르면 되고, 격리망은 지금 대역을 유지하는 편이 안전하다.

**질문: 배포 서버에 NIC 이 실제로 2개 있는지(또는 추가 가능한지) 확인 요청.** 1개뿐이면 구성이 달라져야 해서(VLAN 또는 단일망 운용의 위험 감수) 그때 다시 논의한다.

> LAN 은 두 개 이상이 있을 가능성 농후. LAN 이 부족한 문제는 고려하지 않아도 될 것.

## T7. E6 보류의 파생 — 종단 신호를 어디로 옮기나 (구 §5-9)

1차 문서에서 completedAt 기록(→ GuestServerStatus.PROVISIONED 도달)은 E6 의 산출물이었다. E6 보류로 갈 곳이 없어졌으므로 재배치가 필요하다.

**제안: "할당 정의서가 보유한 마지막 phase 를 완주한 시점에 completedAt 기록" 으로 일반화.** 정의서에 없는 phase 는 어차피 건너뛰는 규칙(1차 §4-D8)과 한 몸이 되는 규칙이고, TESTING 이 없는 정의서는 지금도 이렇게 끝났을 것이므로 E6 을 나중에 구현해도 규칙이 바뀌지 않는다. 구현 위치는 phase 완주 판정을 처음 만드는 **E1 의 엔진 코어(E1-0a)** 가 된다.

## T8. Gigabyte BMC Redfish 선례 조사 결과 (구 §5-8)

지시대로 웹 선례 조사를 수행했다. 전체 보고서(출처 URL·확인/추정/불가 구분·예상 URI 표 20행·실보드 확인 체크리스트 8항목)는 **`26-07-12_11-00-53_E3-R-bmc-redfish-survey_discussion.md`** 로 저장했다 — E3-R 조사 트랙의 선행 산출물이다. 토론에 필요한 결론만 추리면:

1. **URI 조사의 막힌 부분이 상당수 풀렸다.** Giga Computing 공식 「Redfish API Reference Guide for ASPEED AST2600」(2024, 93쪽 PDF)을 확보해 전문 파싱했다 — 4종 보드의 BMC 는 ASPEED AST2600 + AMI MegaRAC SP-X 계열(MS73-HB1 은 datasheet 명기, 나머지 3종은 동세대 유추)이라 이 가이드가 그대로 적용될 가능성이 높다.
2. **BIOS 설정 변경 경로 확정 후보**: `PATCH /redfish/v1/Systems/Self/Bios/SD` 바디 `{"Attributes":{...}}` — pending 객체 이름이 표준 관례 `Settings` 가 아니라 AMI 고유 **`SD`** 다. 1차 문서가 "U2-2 템플릿을 무변환으로 Attributes 에 싣는다" 고 했던 그 경로가 공식 문서로 실재 확인됐다. 적용은 재부팅 시(표준 `@Redfish.Settings` 모델), attribute 키(`PCIS003` 류)는 보드·버전마다 달라 **Attribute Registry 런타임 해석이 필수 요건**으로 확정 후보다.
3. **BMC 계정·네트워크·NTP 경로도 공식 가이드에 실재**: `PATCH /redfish/v1/AccountService/Accounts/{id}` (비밀번호 변경 — §5-5 의 '고정 비밀번호 1회 변경' 이 이 한 호출), `PATCH /redfish/v1/Managers/Self/EthernetInterfaces/{eth0…}`, `PATCH .../NetworkProtocol`. 단 **Syslog 는 Redfish 트리 밖(AMI 비-Redfish `/api/...`)으로 추정** — E3-2 계약 범위를 정할 때 고려할 사실.
4. **기본 자격증명 규약이 공식 확인됐다**: 2020년 이후 Gigabyte 서버 BMC 는 `admin` + **비밀번호 = 메인보드 시리얼번호의 마지막 11자** (unique pre-programmed password, 공지 1762). 첨언하신 "dmidecode 로 시리얼 수집 → 최초 자격증명" 방향이 공식 규약과 정확히 일치한다.
5. **펌웨어 업데이트도 Redfish 로 가능함이 실전 문서로 확인** — `POST .../UpdateService/Actions/SimpleUpdate` (Gigabyte OEM 파라미터 `UpdateComponent`) + `Oem.AMIUpdateService.FlashPercentage` 진행률 폴링. HPE Cray 가 Gigabyte 노드 운영에 실사용 중. **E2-R(펌웨어 실행 메커니즘) 토론에 새 선택지가 강해졌다**: 기존 수동 방식(FAT32 가상 USB + UEFI Shell)보다 Redfish UpdateService 쪽이 자동화 친화적일 수 있다 — 단 E3 의 자격증명 선행이 E2 로 당겨지는 의존 변화가 생긴다.
6. **함정 목록**(클라이언트 설계 요건 후보): PATCH 시 ETag/If-Match 412 (fresh GET 후 PATCH + 1회 재시도 패턴), 세션 수 상한(로그아웃 DELETE 필수), 업데이트 중 WebGUI 접속 금지, MegaRAC Redfish 인증 우회 CVE 이력(자동화 전 BMC 펌웨어 최신화 선행).

**질문: 실보드 URI 수집은 survey 문서 말미의 "확인 체크리스트 8항목" 을 그대로 사용해 진행하면 된다** — Claude Desktop Browser 로 확인하실 때 우선순위는 ② `/redfish/v1` 버전 ④ `Bios/SD` 실재 ⑤ UpdateService 실측 순을 권한다. 이 방식으로 진행해도 좋은지? 그리고 **5번(펌웨어 업데이트를 Redfish 로)** 이 매력적이라면 E2-R 조사 범위에 "Redfish UpdateService vs UEFI Shell 이미지 방식 비교" 를 정식 추가한다.

> 가상 usb 제작 방식에서 불편했던 점은, linux shell 에서 직접 내가 만들었어야 한다는 데에 있다.
> 확인해본 결과, `.RBU` 및 `.ima_enc` 파일이 각각 bios, bmc 업데이트 파일 번들 내에 존재하는 것이 확인되었으므로 이 방식으로 시도해보고 가능하다면 변경해보는 것도 좋을 것으로 보인다.
> 다만, BIOS 파일의 경우 별도 요청한 커스텀 파일을 사용하므로 (GIGABYTE 측에서 공식적으로 받는 이미지이긴 하지만), 커스텀 된 것들이 redfish api 를 통한 업데이트를 통해서도 적용이 되는 지는 별도로 확인이 필요한 부분이다.
> 우선은 가상 USB 제작 로직이 provisioning server 내 로직으로 구현이 가능하다면 가상 USB 로 선진행하고, 이후 redfish api 로 넘어가는 것을 목표로 한다.
