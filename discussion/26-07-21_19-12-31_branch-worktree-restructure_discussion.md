### D6. 워크트리 배치 방식과 부트스트랩

| A. 내장 `claude --worktree <이름>` | 워크트리 생성과 세션 시작이 한 명령이고, gitignore 파일 자동 복사와 권한 자동 공유, tmux 연동을 얻는다. 대신 디렉토리가 저장소 안쪽 `.claude/worktrees/` 에 중첩되고 위치와 이름을 Claude Code 가 정한다 |
| B. 수동 `git worktree add ../ServerProvision-<이름>` | 형제 디렉토리로 나열되고 이름을 마음대로 정한다. 대신 부트스트랩을 직접 짜야 한다 |

권장: A. 워크트리 디렉토리는 GitHub 에 올라가지 않으므로 위치는 포트폴리오와 무관하고, 자동 부트스트랩의 이득이 크다. 사용자만 답할 수 있는 이유는, 터미널과 IDE 에서 어느 배치가 다루기 편한지가 사용자의 작업 환경 취향이기 때문이다.

부트스트랩은 방식과 무관하게 필요하다. `CLAUDE.local.md`(빌드 명령과 DB 환경변수)와 권한 목록이 gitignore 대상이라 새 워크트리에 없다. 해결은 `.worktreeinclude` 파일에 `CLAUDE.local.md` 를 적어 자동 복사시키고, 공유해도 되는 권한은 커밋되는 `.claude/settings.json` 으로 올리는 것이다. 누적 메모리는 저장소 기준으로 묶여 모든 워크트리가 자동 공유하므로 별도 조치가 필요 없다(공식 문서로 확인).

> A 진행.
>
> (2026-07-22 정정) A(`claude --worktree`)를 실제로 쓰려다 우리 규약과 두 군데서 충돌함이 드러났다. ① 내장 방식은 브랜치를 워크트리 이름으로 만드는데 워크트리 이름에 `/` 를 못 써서, 같은 문서에서 확정한 `feat/<코드>-<슬러그>` 규약명이 나오지 않는다. ② base 가 `origin/main` 고정이라 feature 를 dev 에서 떠야 하는 모델과 어긋난다. 이 두 충돌은 D6 결정 시점에 이미(브랜치 명명 규약과 base-ref 조사 결과로) 추론 가능했으나 그때 놓쳤다. **따라서 표준을 B(수동)로 정정한다**: `git worktree add -b <feat/코드-슬러그> <base>/ServerProvision-<브랜치> origin/dev`. 폴더 경로는 저장소 기존 관례대로 **`ServerProvision-<브랜치>`**(브랜치의 `/` 를 경로 구분자로 보존)로 둔다 — 예: 폴더 `ServerProvision-feat/E1-I-boot-infra` ↔ 브랜치 `feat/E1-I-boot-infra`. 기존 `ServerProvision-renew/main`, `ServerProvision-feature/setting` 과 동일 형태이고, 브랜치명이 경로에 그대로 드러나며 타입별로 묶인다(폴더명과 브랜치명이 어긋나 혼란스럽다는 사용자 지적 반영). 편의는 `~/.zshrc` 함수 `spv-wt <브랜치>`(폴더 `ServerProvision-<브랜치>` 자동 생성 + origin/dev 기반 + CLAUDE.local.md 승계 + cd)로 대체한다.

## 3. 판단 없이 진행할 것

명백히 이득이라 결정 항목으로 두지 않는다. 이견이 있으면 말씀해 달라.

- 기동 가능화. 현재 스키마를 떠 `ddl/schema.sql` 1건과 docker-compose.yml 을 추가하고 README 에 기동 절차 4줄을 넣는다. 인수인계 담당자의 첫 행동이 실패하는 문제를 없앤다.
- CI 도입. GitHub Actions 로 JDK 21 셋업과 `./gradlew test` 한 스텝을 돌리고 README 에 배지를 붙인다. DB 컨테이너가 필요 없어 비용이 거의 없고, 테스트 1,280건이 외부 검증을 얻는다.
- 언어 분류 교정. `.gitattributes` 에 plan, report, docs, discussion, diagram 을 문서로 지정한다. 파일 이동 없이 분류만 바꿔 HTML 비율을 사실상 0으로 만든다.
- README 죽은 참조 2건 수정. 18행의 없는 디렉토리 참조와 69행의 추적 0건 경로 참조를 고친다.
- 잔재 정리. `worktree-agent-*` 워크트리 5개와 브랜치 5개 제거(고유 작업 0건 검증 완료), 레거시 feature 워크트리 2개 제거, dev 절대경로를 가리키는 실효 없는 `core.hookspath` 설정 해제.
- 에이전트 정의 정합화. `.claude/agents/*.md` 의 절대경로 하드코딩과 문서를 docx 로 만들라는 낡은 규약을 현행 html 규약으로 고친다. 추적 파일이라 새 워크트리로 그대로 전파되므로 세션을 늘리기 전에 고친다.
- 구 자산 박제. 구 프로토타입은 `v0-prototype` 태그로, 구 dev 라인은 `archive/dev-legacy` 태그로 고정한 뒤 정리한다. 태그로 도달성이 보존되어 이력 손실이 없다.

## 4. 하지 않기로 한 것

포트폴리오를 위해 이력을 꾸미는 조치는 나중에 발각되면 손해가 더 크다. 아래는 의도적으로 하지 않는다.

- 145커밋 재작성. 이메일이 이미 귀속되어 잔디 누락이 없고, 재작성하면 4개월치 기여 그래프 연결을 잃는다.
- AI 검토를 인간 검토로 옮겨 적기. 커밋 본문과 설계 문서에 검토 주체가 별도 Claude 세션이라고 원문으로 남아 있어, 재라벨링하면 반드시 발각된다.
- self-merge 를 author 와 reviewer 가 분리된 병합으로 부르기. 두 번째 계정으로 승인을 만드는 것도 하지 않는다.
- squash 병합으로 전환점 감추기. 캠페인 단위 브랜치는 커밋이 1개에서 2개라 squash 이득이 거의 없고, `--no-ff` 가 캠페인 경계를 그래프에 남긴다.
- 발생하지 않은 충돌에 대비한 규약 선제작. pre-push 훅, main 보호 규칙, 경로 소유 매트릭스는 CLAUDE.md 의 불가침 원칙(갈라지는 시점에 분리)에 어긋난다.
- plan 과 report 를 지금 재배치. 경로가 CLAUDE.md 규약과 에이전트 정의에 박혀 있어 세션 진입 직전에 옮기면 세션마다 다른 경로에 쓰는 드리프트가 난다. 언어 분류 왜곡은 파일 이동 없이 `.gitattributes` 로 해결된다.

## 5. 착수 순서 (되돌릴 수 있는 것부터)

1. 안전망. 구 프로토타입과 구 dev 를 태그로 박제. ref 스냅샷은 이미 확보됨.
2. 트렁크 승격. dev 를 현재 작업으로 맞추고, D1 결정에 따라 main 을 baseline 처리한 뒤 push.
3. GitHub 기본 브랜치 전환. gh 설치와 인증 후 처리. 이 전에는 원격 renew/main 을 지우지 않는다.
4. 잔재 정리. renew/main 과 구 dev 삭제, 워크트리와 브랜치 정리.
5. 문서와 설정 정합화. `chore/CH1-repo-tidy` 브랜치에서 PR 로 처리하고 세션을 늘리기 전에 병합. 이 PR 이 새 브랜치 모델의 첫 사례가 된다.

1번과 2번은 세션 종료와 D1 확정 후 바로 착수할 수 있다. 3번부터 5번은 나머지 결정(D2 부터 D6)을 받은 뒤 진행한다.
