# E 로드맵 토론 — 결정 정리

## 1. 확정 결정 목록

두 차례 토론에서 확정된 것을 결정 번호로 고정한다. (근거 · 대안 · 탈락 사유의 상세는 1 · 2차 문서에 있고, 여기는 결과만 적는다.)

**엔진 골격 (1차 문서 §4 합의 그대로 — 이견 없이 통과)**
- **DEC-1. 게스트 구동 pull + 명시 보고.** 게스트의 HTTP 요청만 상태를 전진시킨다. 서버 push · 작업 큐 탈락.
- **DEC-2. 커서 전이 = 게스트 사실 신호 수신 트랜잭션 내 즉시.** `/boot` 는 읽기 전용 dispatch.
- **DEC-3. SetupStep 원장 = 이벤트 시점 append-only.** PENDING 선적재 · 이벤트 소싱 탈락.
- **DEC-4. 실패 · 종단 = ProvisioningProgress 명시 컬럼**(failedAt/failedStepCode/completedAt). 자동 재시도 없음(운영자 명시 액션), 무보고 실패는 경과 표시 + 수동 FAILED 전환.
- **DEC-5. 통신 채널 이원화** — 부팅 계열 text/plain(PXE 한정 advice 가 예외를 안전 스크립트로 변환) / 에이전트 계열 JSON. 게스트 신원 토큰 + 보고 멱등.
- **DEC-6. phase 실행기 SPI = 최소 폭 단일 인터페이스 + 부분 등록 registry**, 미구현 phase 도달 시 HOLD 대기(silent 통과 금지).

**프로세스 · 검증**
- **DEC-7. E 단계 CP5 예외 승인** — 브라우저 E2E 대신 **모의 게스트 하네스 실행 + 상세 페이지 확인**. 하네스는 **`scripts/mock-guest/`** (git 추적 자산). CLAUDE.md 명문화 문구는 §4 — 확인 후 반영.
- **DEC-8. 조사 트랙 `E*-R` 코드 승인** — Notion 행 신설 완료(E1-R/E2-R/E3-R). 산출물 = discussion 문서.
- **DEC-9. 검증 Tier 선언 유지** — T0 계약 테스트 / T1 하네스 / T2 QEMU(가능성은 E1-R 조사) / T3 실기(시점을 Notion 후속 마일스톤에 강제 기재).

**E1 — 진단 리눅스**
- **DEC-10. 진단 리눅스 = Alpine Linux 커스텀 빌드** (dmidecode · ipmitool 사전 가용). 빌드 파이프라인 · 에이전트 배포 방식은 E1-R 조사.
- **DEC-11. IPMI 각인(modelName/serialNumber 하드웨어 기록) 보류.** 근거(사용자, 2026-07-12): 공정 표준상 프로비저닝 종료 후 게스트의 NIC 은 DHCP auto 로 전환되고 BMC 는 IP 고정 — 즉 프로비저닝망 · BMC 연결이 끊긴 뒤라 재진입 자동화가 성립하지 않고, 어차피 사람이 마지막으로 게스트 서버를 직접 점검하는 업무 프로세스가 존재한다. `ProvisioningPhaseStep.IPMI_SETTING` 상수는 유지하되 **소비자 없음(보류)** 를 Javadoc 에 명기 — 공정 여건이 바뀌면 재개. modelName/serialNumber 는 U1 이 만든 **운영자 입력 필드로만** 관리.
- **DEC-12. E1 하위 재편** — E1-R(조사) / E1-0a(엔진 코어 + 종단 규칙) / E1-0b(게스트 프로토콜 v1) / E1-1(부팅 체인, DIAGNOSTIC_BOOTING) / **E1-2(수집 · 적재 + phase 완주 판정 + 할당 대기 — 구 E1-3 잔여 흡수)**. 각인 슬라이스는 소멸.

**인프라**
- **DEC-13. PXE 인프라 = 1단계(앱이 boot.ipxe · kickstart 역할 흡수 — 로드맵 내장) + 2단계(dhcpd/tftp 구성 관리 UI) 채택.** 2단계 = 신규 슬라이스 **E1-I**, 우선순위는 E1 완주 후. 3단계(Java 내장 DHCP/TFTP) 탈락.
- **DEC-14. 진단 리눅스 이미지 관리 = "교체(replace)만 가능한 시스템 자산" 화면** (E1-I 포함). MA 자원 도메인 편입 탈락 — 삭제 · 신규 생성이 제한되고 접근이 격리되어야 하므로. 마커 · 무결성 검증 엔진은 재사용.
- **DEC-15. 네트워크 = NIC 2개 구성** (사내망 = UI 접속 / 격리망 192.168.1.x = PXE · 게스트 채널). LAN 포트 부족은 고려 대상 아님(사용자 확인).

**U3 — 할당 · 스냅샷**
- **DEC-16. 스냅샷 시점 = 할당 시 즉시 복사.**
- **DEC-17. 스냅샷 구조 = setting_process 행 단위 복사** (process_type + payload JSON, 할당 FK 아래).
- **DEC-18. 재할당 = 미소비 스냅샷 대체 + 소비된 스냅샷 이력 보존.** 사용자 조건: "이전에 어떤 스냅샷으로 프로비저닝이 수행됐는지의 이력이 삭제되어서는 안 된다." 구현 주의(§3-3 에서 해석 확인): SetupStep 원장은 *실행 사실*만 저장하고 *세팅 내용*은 스냅샷이 유일한 보관처이므로, **실행에 소비된 스냅샷 행은 재할당 · 원본 삭제와 무관하게 보존**해야 이 조건이 성립한다.
- **DEC-19. 원본 정의서 삭제 = soft-delete 계열 + 지연 purge.** 삭제 액션은 purge 가 아니다. 스냅샷으로 진행 · 대기 · 중단 상태의 게스트 서버가 존재하면 목록에서 숨기되 단순 조회는 가능하게 남기고, 해당 스냅샷의 게스트 서버가 모두 완료되면 목록에서 사라지고 DB 에서도 삭제된다. 세부 설계(기존 lifecycle/trash 인프라와의 접점)는 U3 CP1.

**E2 — 펌웨어**
- **DEC-20. 펌웨어 집행 = 가상 USB(FAT32 이미지) 방식 선행, Redfish UpdateService 후행 전환 목표.** 사용자 확인: 기존 방식의 불편은 이미지를 리눅스 셸에서 수동 제작해야 했다는 점 — **가상 USB 제작 로직의 앱 내 구현 가능 여부가 E2-R 의 1번 조사 항목**이 된다. `.RBU`(BIOS)/`.ima_enc`(BMC) 파일이 공식 번들에 실재함은 사용자가 확인했으므로 Redfish 경로의 재료는 갖춰짐. 단 **BIOS 는 Gigabyte 공식 발급이되 커스텀 파일** — Redfish 업데이트로 커스텀 이미지가 적용되는지는 실기 확인 항목(T3 유보). E2-R 조사 범위에 "Redfish UpdateService vs 가상 USB 방식 비교" 정식 포함.

**E3 — BMC · Redfish**
- **DEC-21. BMC 자격증명 부트스트랩** = dmidecode 로 수집한 메인보드 시리얼 기반 공장 기본 비밀번호(웹 조사로 공식 규약 확인: `admin` + 시리얼 마지막 11자) → Redfish 로 고정 비밀번호 1회 변경. 운영자 입력 불요. 고정 비밀번호 보관 방식(환경변수/설정 파일 — DB 평문 금지)은 E3-R 잔여.
- **DEC-22. 실보드 URI 수집** = survey 문서(26-07-12_11-00-53) 말미 체크리스트 8항목으로 진행(사용자 + Claude Desktop Browser).

**E5 · E6 — 후처리 · 종단**
- **DEC-23. OS 후처리 = 하이브리드.** 기본(간단한 후처리) = `%post` 병합 + 시작/종료 별도 curl 보고. **커스텀 bash script 옵션 적용 시** = first-boot 에이전트로 실행 · 확인. 이를 위해 **세팅 정의서 OS_SETTING 계약의 커스텀 확장(커스텀 bash script 첨부)을 U 트랙에서 병행** — 실행 스크립트라는 위험성은 사용자도 인지, 안전장치는 해당 slice plan 에서 상술(§3-2 에서 코드 확인).
- **DEC-24. E6(TESTING) 구현 보류** — Windows OS 지원 시점에 재개.
- **DEC-25. 종단 신호 = "할당 정의서가 보유한 마지막 phase 완주 시 completedAt 기록" 으로 일반화.** 구현 위치 E1-0a. E6 을 나중에 구현해도 규칙 불변.
