1. 왜 이 조사인가 (문제 정의)

계층형 자원 — 부모 BoardModel / OSMetadata → 자식 BIOS / BMC / Subprogram / ISO — 은 LifecycleEntity @MappedSuperclass 의 3 boolean(is_enabled / is_deprecated / is_deleted)을 공유한다. 현재 BoardModelService · OSMetadataLifecycleService 의 cascade 는 write-based(eager) — 부모 상태 전이가 자식 엔티티 boolean 을 직접 덮어쓴다.

결함 시나리오 — 자식 수동 선택의 소실

운영자가 자식 A 하나만 수동으로 deprecate 해뒀다고 하자. 이후 부모를 deprecate 했다가 다시 undeprecate 하면, undeprecate() 의 자식 루프가 filter(b -> b.isDeprecated()) 로 deprecated 자식 전부를 되돌려 — 운영자가 명시적으로 deprecate 해둔 자식 A 까지 undeprecate 시킨다. write-cascade 가 inherited(cascade-derived) 와 explicit(수동) 을 구분하지 못하기 때문.

근거 코드 : BoardModelService.deprecate()/undeprecate() 의 children.forEach(BoardBIOS::deprecate) · OSMetadataLifecycleService 대칭.

naive 해법 — 차원마다 is_*_cascaded 플래그 3개 추가 — 는 컬럼 3배 폭증으로 기각(사용자 지적). 또한 부수적으로 차원별 cascade 방향이 불일치한다 : enabled=단방향(disable만), restore=opt-in, deprecate/undeprecate=양방향 항상. 본 조사는 ① 보존(explicit↔inherited) ② 방향성 일관화 두 축을, 컬럼 폭증 없이 푸는 정립 패턴을 권위 출처로 찾는다.

2. 핵심 통찰 — 6 분야가 공유하는 4 원리

6 분야가 한 골격으로 수렴한다 : effective = explicit ?? inherited 를 write 가 아니라 read-time 에 resolve 하고, explicit/inherited 구분은 "값을 저장했는가(provenance)" 단일 비트로만 표현한다.

#원리요지
1explicit 우선 단계함수explicit(declared/directly-applied) 값이 있으면 inheritance 분기는 평가조차 안 됨 → "cascade 가 explicit 을 덮는 사건"이 구조적으로 발생 불가 [1][9]
2inherited 는 값이 아니라 부재 시 fallback자식은 자기 명시값만 저장, 상속분 미저장 → NULL(부재) 자체가 "inherited" 비트를 겸함. 추가 플래그 컬럼 0 [1][20]
3저장이 불가피하면 1비트 provenanceWindows INHERITED_ACE(0x10) 단일 비트가 entry 출처를 표지 → 재전파는 inherited 항목만, explicit 보존. 차원×플래그 곱셈을 entry당 1비트로 압축 [8][25]
4방향성 비대칭restrictive 하향은 자동, permissive 상향(역전파)은 표준에 부재 — SQL ON DELETE CASCADE 는 있어도 "ON UNDELETE"는 없다 [26][27][30]

종합 : 자식 행을 cascade-write 하지 않으면 부모 되돌림 시 건드릴 explicit 이 없어 "수동 선택 보존"이 자동 성립한다. 가장 강력한 해법은 cascade-write 제거(effective 계산형), write 를 유지해야 하면 1비트 provenance 가 차선이다.

3. 산업 사례 6 분야

① CSS Cascade & Inheritance standard-body

explicit(declared)이 있으면 specified value=그 값, 없을 때만 inherited=부모 computed value. inheritance 는 저장 값이 아닌 fallback 규칙 → cascade 가 explicit 을 못 덮는 구조적 보장. revert/unset 이 특정 origin/layer 기여분만 1토큰으로 롤백 → 플래그 0.

— W3C CSS Cascading and Inheritance Level 5 §4 Value Processing, §7 Defaulting [1][4][5]

② Windows ACL 자동 상속 standard-body vendor-official

상속된 ACE 에만 INHERITED_ACE(0x10) 비트 표지, directly-applied(explicit)는 무표지+앞쪽 배치+항상 우선. 부모 inheritable ACE 제거 시 자동 상속이 inherited 복사본만 제거("removes any copies that were inherited")하고 explicit 은 보존. SE_DACL_PROTECTED 로 자식이 상속 자체를 차단.

— [MS-DTYP] ComputeInheritedACLfromParent / Automatic Propagation of Inheritable ACEs [8][9][11]

③ IAM 정책 계층 (GCP / AWS / Azure) vendor-official

정책은 attach 노드에만 1벌 저장; 자식 effective 는 평가 시점에 ancestor chain 합성(GCP union, AWS @@assign/@@append/@@remove, Azure additive+deny override). 부모 변경이 자식 저장정책을 물리적으로 안 건드림 → 수동 선택 자동 보존. attachment point 가 곧 explicit/inherited 구분자 → 자식 플래그 컬럼 0.

— GCP IAM resource-hierarchy / effective policies · AWS Organizations inheritance · Azure RBAC scope [13][14][15][18]

④ 계층형 설정 / feature flag 3-state vendor-official

tri-state nullable(null=inherit / true / false). Kubernetes *bool 은 nil(unset)과 explicit-false 를 반드시 구별(zero value 가 valid choice) → pointer 필수로 규정. 단일 nullable 필드가 값+출처를 동시 인코딩 → 추가 컬럼 0. LaunchDarkly prerequisite 도 평가(read) 시점에 부모 먼저 본다.

— Kubernetes API Conventions (Optional vs Required, *bool) · LaunchDarkly flag hierarchy [20][22]

⑤ DB 모델링 (PostgreSQL) vendor-official

naive 6컬럼(차원당 값+is_cascaded) 대신 ① bit(3) provenance bitmask 단일 컬럼(차원별 1비트 압축) 또는 ② nullable override + VIRTUAL generated column("occupies no storage, computed when read")로 effective 무저장. STORED(write-time) vs VIRTUAL(read-time) trade-off.

— PostgreSQL 18 §5.4 Generated Columns, §8.10 Bit String Types [24][25]

⑥ Cascade 방향성 (SQL 참조동작 / K8s GC / RBAC ceiling) standard-body academic

SQL ON DELETE CASCADE·RESTRICT·SET NULL 은 단방향 파괴적 전파만 — 역방향 restore 동작 없음. K8s propagationPolicy 도 Foreground/Background/Orphan 만, un-deletion 전파 부재. undeprecate 양방향 전파는 "restore cascade 가 표준에 없는 것"과 동형인 비표준 안티패턴. RBAC: effective ≤ granted ceiling, 상향(permit)은 명시 grant 필요 — 비대칭의 보안 근거.

— PostgreSQL §5.5 Referential Actions · K8s Garbage Collection · NIST RBAC [26][27][29][30]

4. 본 앱 적용안 3 (is_cascaded 3컬럼 폭증 회피)

안 A — effective 계산형 (자식 nullable override + read-time resolve, cascade-write 제거)

메커니즘 : 3 boolean → 3 nullable override(NULL=상속, TRUE/FALSE=명시). 부모 전이는 자식 행을 write 하지 않고, effectiveDeprecated() = override != null ? override : parent.effectiveDeprecated() 를 read-time 계산. enabled 는 CSS 비상속 property 대응.

스키마 : NOT NULL boolean → nullable 변경이라 순증 0. effective 는 무저장(또는 VIRTUAL generated). 보존 : 구조적 보장(cascade-write 가 없어 덮어쓰기 불가). 비용 : 모든 read 경로(getter→effective*())·뷰모델·서버가드 전면 개편 + deleted 의 물리 부작용(trash 이동)으로 차원별 모델 혼재 → "단순 구현"과 상충. read-path 전면.

안 B — provenance 압축형 권장

메커니즘 : write-based cascade 유지 + 자식 각 차원에 "cascade-set 인가" 1비트. Windows 자동 전파 차용 — 부모 deprecate→자식이 explicit(origin=0)이면 skip, inherited(=1)면 set+origin=1. 부모 undeprecate→origin=1 자식만 환원, 수동 deprecate(origin=0)한 자식 A 보존.

스키마 : 3 boolean 폭증 대신 단일 bit(3) cascade_origin(또는 nullable Boolean 3 — NULL=inherited, 값+출처 겸용) 1~3 컬럼·순증 최소. enabled 는 단방향이라 항상 explicit 취급(실질 2비트).

read 영향 거의 0 — effective=저장 boolean 이라 기존 getter·LifecycleStage.of() 그대로. write — cascade 루프 filter 에 origin 검사 1줄 추가, 수동 전이 시 origin=EXPLICIT 마킹. 보존 — Windows "removes any copies that were inherited"와 1:1. 비용 — bitmask 가독성(@Convert), origin set/clear 누락 회귀(테스트 방어 필수).

적합성 : 현 cascade 루프 구조를 거의 그대로 두고 filter 만 보강 → R3-1 코드와 마찰 최소. LifecycleEntity 1컬럼 + 4 service 루프 보강 + R2-2 capability 공유로 끝나는 얇은 후속 슬라이스.

안 C — monotonic 무-역전파 cascade (방향성 일관화 + 컬럼 0)

메커니즘 : 역전파(undeprecate/restore cascade)를 아예 제거 — 하향 restrict 만 cascade. 스키마 : 추가 0(출처 구분 불필요). 보존 : 역전파가 없어 덮어쓸 일 원천 부재. 비용 : 부모를 undeprecate 해도 자식이 자동 복구 안 됨 → 운영자가 자식을 일일이 풀어야 함("명확한 사용성" UX 회귀).

5. cascade 방향성 일관 정책

원칙(조사 ⑥) : restrictive 하향 전파(부모가 자식을 막음)는 invariant 보호를 위해 자동/opt-out 이 적절하고, permissive 상향 역전파(부모가 자식을 품)는 표준 부재 → 자동 전파가 안티패턴이다. 자식 ≤ 부모 invariant 를 깨는 방향만 자동 전파하고, 푸는 방향은 명시 동작으로 게이팅.
차원현행권고근거
enableddisable=하향 cascade / enable=역전파 없음✓ 현행이 정답 = reference modelCSS 비상속 property / RBAC ceiling 과 정합. 부모 enable 이 운영자가 개별 비활성한 자식을 임의 부활 안 함.
deprecated양방향 항상 전파 ← 결함 진원deprecate=하향 유지 / undeprecate=역전파 제거 또는 inherited-only 환원undeprecate 양방향 전파가 "restore cascade 가 표준에 없는 것"과 동형인 비표준. 자식 수동 deprecate 의도 소실의 직접 원인.
deleteddelete=하향 cascade / restore=opt-in✓ 현행 유지SQL ON DELETE CASCADE + restore 표준부재(=명시 opt-in)와 정확히 일치. 물리 부작용 때문에 write-cascade 유지가 합리적.
유일한 실질 수정점 = undeprecate 양방향 전파 (말씀하신 그 불일치). 3 차원을 enabled 의 비대칭(하향 restrict 자동 / 상향 permit 비전파-또는-opt-in)으로 정렬하면 끝난다. undeprecate 처리는 채택 옵션에 따라 — (안 C) 자식 루프 삭제 / (안 B) origin=INHERITED 자식만 환원·EXPLICIT 보존(절충 권장) / (안 A) 개념 자체 소멸.

6. 권고 — 2단계

(1) 지금 — R3-1 은 현행 write-cascade 패턴으로 완료

R3-1 의 목표는 "Subprogram 을 형제(BIOS/BMC)와 동일 수준으로". Subprogram 만 undeprecate cascade 를 빼면 형제와 새 비대칭이 생겨 R3-1 취지에 역행한다. 따라서 R3-1 은 전 자식 도메인이 동형인 현 상태로 마치고, undeprecate 방향성 교정은 전 도메인 일괄 후속으로 미룬다. (단 CLAUDE.md §조건분기 legacy 무분별 확장 금지 정합 — 무비판 확장이 아니라 의도된 parity 임을 plan/§에 명기.)

(2) 후속 — 안 B(provenance) + undeprecate 방향성 교정을 얇은 별도 슬라이스(R3-2/R4)

본 프로젝트 동기("명확한 사용성 + 단순 구현 > 이론적 가치, 컬럼폭증 회피, 상태전이 빈도 낮음")에 안 B 가 가장 정합 :

  • 단순 구현 — read-path 무개편(기존 boolean=effective 유지). 안 A 는 모든 getter 호출처 교체라 정면 배치.
  • 컬럼폭증 회피 — is_*_cascaded 3컬럼 → 단일 bit(3)/nullable 1비트.
  • 전이 빈도 낮음 — read-time 재계산(안 A)의 이론적 이점이 실익 적음.
  • 마찰 최소 — 현 cascade 루프 유지 + filter 보강만, R3-1 코드 재작업 0.

안 C(monotonic)는 가장 단순하나 "undeprecate 했는데 자식 안 풀림" UX 회귀가 "명확한 사용성"을 해친다. 도입 시 origin 비트 set/clear 누락 회귀를 Step8 통합 테스트(부모 deprecate→undeprecate 사이클 후 수동 deprecated 자식 보존 검증)로 반드시 방어.

7. Open Decisions (후속 슬라이스 CP1 에서)

  1. undeprecate 역전파 — 안 B(origin=INHERITED 자식만 환원, 자동복구 유지) vs 안 C(역전파 완전 제거, 수동 복구). "명확한 사용성"이면 안 B 절충.
  2. provenance 저장 형식 — bit(3) bitmask(컬럼 1, @Convert 비용) vs nullable Boolean 3(NULL=inherited, 순증 0, 가독성↑). MariaDB JPA 매핑 난이도상 nullable Boolean 3 이 더 단순할 수 있음.
  3. deleted 차원 포함 여부 — restore 가 이미 opt-in 이라 역전파 문제가 deprecated 만큼 첨예하지 않음. deprecated 차원에만 적용하는 최소안도 가능.
  4. R3-1 의 Subprogram undeprecate cascade — (a)양방향 그대로 vs (b)deprecate 하향만 추가하고 undeprecate 보류. 재작업 최소화 관점 (b) 가 안전(단 형제 비대칭 trade-off).
  5. 자식의 영구 cascade 거부(SE_DACL_PROTECTED 대응 isolated 비트) — 현재 요구에 없음 → 과설계 경계, 보류 권장.
  6. effective*() ↔ R2-2 capability 가드 · UI disabled SSOT 공유 — 안 B 면 origin 비트가, 안 A 면 effective*() 가 단일 소스. ChildLifecycleBlockedByParentException 가드와 드리프트 0 유지.
