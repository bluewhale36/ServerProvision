1. 요약 + D1~D5 결정

OS ISO 등록 후 자동 작업 (RHEL 의 comps 추출 등) 을 Strategy 패턴으로 흡수. CLAUDE.md 의 "조건 분기문 무분별 확장 금지" 원칙 준수.

결정채택
D1 트리거 시점A — OSImageService.finalizePreparedIsoRegistration() 의 마지막 (ISO 영속화 직후)
D2 분기 방식C — PostCreationTaskStrategy 인터페이스 + Spring 빈 list 주입
D3 실패 정책B — ISO 등록 성공 + Job 실패 알림 (try-catch 흡수)
D4 범위A — RHEL_BASED 만 자동, 나머지는 Noop strategy
D5 직렬화A — 병렬. ISO_REGISTRATION + COMPS_EXTRACTION 동시 진행
핵심 가치 관리자가 ISO 등록 → 알림 패널에서 두 Job (등록 + comps 추출) 동시 관찰. 사용자 액션 1 단계 (별도 추출 클릭) 제거.
2. 신규 클래스 — 3 개

2.1 PostCreationTaskStrategy (interface)

public interface PostCreationTaskStrategy {
    boolean supports(OSFamily family);
    /**
     * @return 트리거된 Job 의 id. 자동 작업이 없는 경우 null.
     */
    String trigger(OSImage osImage, ISO iso);
}

2.2 RhelPostCreationTaskStrategy

@Component
@RequiredArgsConstructor
public class RhelPostCreationTaskStrategy implements PostCreationTaskStrategy {
    private final CompsExtractionLauncher compsExtractionLauncher;

    public boolean supports(OSFamily family) { return family == OSFamily.RHEL_BASED; }

    public String trigger(OSImage osImage, ISO iso) {
        String jobId = compsExtractionLauncher.startExtraction(osImage.getId(), iso.getId());
        log.info("[S5-7] RHEL ISO 등록 후 comps 추출 자동 트리거. osImageId={}, isoId={}, compsJobId={}",
                osImage.getId(), iso.getId(), jobId);
        return jobId;
    }
}

2.3 NoopPostCreationTaskStrategy

@Component
public class NoopPostCreationTaskStrategy implements PostCreationTaskStrategy {
    public boolean supports(OSFamily family) {
        return family == OSFamily.DEBIAN_BASED || family == OSFamily.WINDOWS_BASED;
    }
    public String trigger(OSImage osImage, ISO iso) {
        return null;  // 의도된 noop
    }
}

// S5-7 — ISO 등록 후 자동 작업 트리거 (OS family 별 strategy 다형성).
// 실패는 ISO 등록 자체 영향 없도록 fail-safe — D3 정책 B (등록 성공 + Job 실패 알림).

return saved.getId();

private void triggerPostCreationTask(OSImage osImage, ISO iso) {
    // 단위 테스트 @InjectMocks 환경에서 List 자동 주입이 null 인 경우 방어.
    if (postCreationTaskStrategies == null || postCreationTaskStrategies.isEmpty()) return;
    try {
        postCreationTaskStrategies.stream()
                .filter(s -> s.supports(osImage.getOsName().getFamily()))
                .findFirst()
                .ifPresent(s -> s.trigger(osImage, iso));
    } catch (RuntimeException e) {
        log.error("[S5-7] PostCreationTask 트리거 실패 (ISO 등록은 영속됨). osImageId={}, isoId={}, family={}",
                osImage.getId(), iso.getId(), osImage.getOsName().getFamily(), e);
    }
}

3.3 fail-safe 정당화

  • strategy 의 RuntimeException 흡수 → ISO 영속화 트랜잭션 보존 (D3 B)
  • 실패는 서버 로그 + BackgroundJob 의 FAILED 상태로 사용자 인지 가능
  • List null 방어 — 단위 테스트의 @InjectMocks 환경에서 List<PostCreationTaskStrategy> 가 null 주입되어도 noop

5. 사용자 가시 변경
  • Before : Rocky/CentOS ISO 등록 → ISO_REGISTRATION Job 만 노출 → 관리자가 별도로 "환경·패키지 추출" 클릭해야 comps 작업 시작
