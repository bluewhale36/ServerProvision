0. 한 줄 요약과 hotfix 의 4 가지 변경

SSR controller 의 form POST 가 brower 에 raw JSON 으로 응답되던 회귀를 advice level 의 우선순위 에서 일괄 차단. 예외별 ad-hoc 매처 추가가 아니라 두 advice 의 책임 경계를 numeric @Order 로 명시. frontend 는 XHR 의 Accept 를 단순화하고 거절 응답을 공통 ErrorModal 로 통합.

영역변경효과
backend (1) WebExceptionHandler @Order(HIGHEST_PRECEDENCE - 100) 로 우선 명시 두 advice 가 동일 우선순위였던 ambiguity 제거. SSR 의 Accept: text/html 호환 매처가 무조건 우선 잡힘
backend (2) ChildLifecycleBlockedByParentException extends ConflictException (이전 hotfix) 도메인 예외가 advice 의 일반 매처 (ConflictException) 에 잡혀 ad-hoc 매처 불요
frontend (3) submitAsync 의 Accept: application/json 명시 (text/html q 제거) XHR 은 무조건 JSON 매처 (ApiExceptionHandler) 로 fall-through
frontend (4) global/error-modal.js + .gem-* CSS + layout include XHR 거절 응답이 alert(msg) 가 아닌 공통 modal 로 표시. 호출자별 window.AsyncSubmitResult override 보존

1. 옛 흐름의 결함 — 두 advice 가 ambiguous

1.1 두 advice 의 책임 표기

// WebExceptionHandler.java (옛)
@ControllerAdvice
@Order(Ordered.HIGHEST_PRECEDENCE)
public class WebExceptionHandler {
    @ExceptionHandler(value = NotFoundException.class, produces = MediaType.TEXT_HTML_VALUE)
    public String handleNotFound(NotFoundException ex, Model model, ...) { ... return "error"; }
    @ExceptionHandler(value = ConflictException.class, produces = MediaType.TEXT_HTML_VALUE)
    public String handleConflict(ConflictException ex, ...) { ... return "error"; }
    @ExceptionHandler(value = DomainException.class, produces = MediaType.TEXT_HTML_VALUE)
    public String handleDomain(DomainException ex, ...) { ... return "error"; }
}

// ApiExceptionHandler.java (옛)
@RestControllerAdvice
@Order(Ordered.HIGHEST_PRECEDENCE)   // ← 동일 우선순위
public class ApiExceptionHandler {
    @ExceptionHandler(value = NotFoundException.class, produces = MediaType.APPLICATION_JSON_VALUE)
    public ResponseEntity<ApiErrorResponse> handleNotFound(NotFoundException ex) { ... }
    @ExceptionHandler(value = ConflictException.class, produces = MediaType.APPLICATION_JSON_VALUE)
    public ResponseEntity<ApiErrorResponse> handleConflict(ConflictException ex) { ... }
    @ExceptionHandler(value = DomainException.class, produces = MediaType.APPLICATION_JSON_VALUE)
    public ResponseEntity<ApiErrorResponse> handleDomain(DomainException ex) { ... }
}

1.2 매처 우선순위가 모호

Spring 의 ExceptionHandlerExceptionResolver 는 advice 들을 @Order 순으로 시도. 두 advice 가 동일 @Order(HIGHEST_PRECEDENCE) 라 어느 쪽이 우선 잡힐지는 등록 순서 (보통 Bean 이름의 알파벳 순) 에 의존. ApiExceptionHandler 가 A 로 시작 → 알파벳 우선 → JSON 매처가 SSR Accept 헤더와도 매칭 (*/* 호환) → raw JSON 응답.

❌ 사용자 보고 화면 — {"message":"부모 자원 'Rocky Linux 9.5' 이(가) 삭제 상태라 자식의 restore 액션이 거절됐어요. 부모부터 복구해주세요."}
브라우저 form POST 가 SSR error 페이지가 아닌 raw JSON 을 그대로 표시. SSR 사용자에게 친화적이지 않음.

3. backend 변경 — 코드 diff + 효과 설명

3.1 WebExceptionHandler.java — @Order numeric 우선

// 옛
@ControllerAdvice
@Order(Ordered.HIGHEST_PRECEDENCE)
public class WebExceptionHandler { ... }

// 새
@ControllerAdvice
@Order(Ordered.HIGHEST_PRECEDENCE - 100)   // ← ApiExceptionHandler 보다 명시적 우선
public class WebExceptionHandler { ... }

Why -100 ?

Spring 의 @Order 는 int 값이 작을수록 우선. HIGHEST_PRECEDENCE = Integer.MIN_VALUE 이지만 산술적 비교라 HIGHEST_PRECEDENCE - 100 으로 정확히 HIGHEST_PRECEDENCE 보다 우선. 다른 advice 가 끼어들 여지를 100 단위로 두어 향후 확장성도 확보.

대안 검토 — annotations=Controller.class 로 적용 범위 제한도 시도했으나, hybrid controller (@Controller 가 @ResponseBody 메서드 보유) 의 JSON 응답이 깨져 28 test fail. @Order 분리가 hybrid 도 지원.

3.2 ApiExceptionHandler.java — annotations 원복, @Order 유지

// hotfix 시도 1 (실패) — annotations = RestController.class 로 제한
@RestControllerAdvice(annotations = RestController.class)   // ← hybrid controller 의 JSON 응답 깨짐
@Order(Ordered.HIGHEST_PRECEDENCE)

// hotfix 시도 2 (실패) — @Order LOWEST 로 낮춤
@RestControllerAdvice
@Order(Ordered.LOWEST_PRECEDENCE)   // ← XHR test 의 매처 우선순위 깨짐

// hotfix 시도 3 (성공) — @Order HIGHEST 유지, Web 이 더 우선
@RestControllerAdvice
@Order(Ordered.HIGHEST_PRECEDENCE)   // ← Web 의 -100 으로 자동 후순위

3.3 ChildLifecycleBlockedByParentException.java — extends 변경 (이전 hotfix)

// 옛
public class ChildLifecycleBlockedByParentException extends DomainException { ... }

// 새
public class ChildLifecycleBlockedByParentException extends ConflictException { ... }

예외 계층의 의도

DomainException 의 fallback 매처는 500 INTERNAL_ERROR 로 잡힘 — "잠재적 버그" 로 간주. 본 예외는 명시적 409 Conflict 인데 DomainException 직접 상속이라 의도와 맞지 않음. ConflictException 으로 상속 변경 → 두 advice 의 handleConflict 매처에 자동 catch.

4. frontend 변경 — 공통 ErrorModal + Accept 명시

4.1 confirm-modal-base.js — Accept 단순화

// 옛
headers: {'X-Requested-With': 'XMLHttpRequest',
          'Accept': 'application/json,text/html;q=0.9,*/*;q=0.5'},

// 새
headers: {'X-Requested-With': 'XMLHttpRequest',
          'Accept': 'application/json'},

Why?

옛 헤더는 text/html (q=0.9) 도 매처 가능 → WebExceptionHandler 의 produces=TEXT_HTML 매처가 잡힐 가능성 → XHR 이 HTML 응답 받음 → resp.text() 의 첫 500 자가 modal 에 보임. 새 헤더는 application/json 만 명시 → WebExceptionHandler 매처 미호환 → ApiExceptionHandler fallback 매처 확정 → JSON 응답 보장.

4.2 global/error-modal.js — 신규 (전역 modal)

// 핵심 구조
window.ErrorModal = { show, hide };

// AsyncSubmitResult 의 onRejected 통합 — 옛 alert(msg) 대체
if (!window.AsyncSubmitResult) {
    window.AsyncSubmitResult = {
        onSuccess: () => window.location.reload(),
        onRejected: (_form, status, payload) => {
            const msg = (payload && payload.message) || '요청이 거절되었어요.';
            show({ message: msg, status: status });
        },
        onNetworkError: () => show({ message: '서버와 통신할 수 없어요. …', status: 0 })
    };
}

5. 매처 매트릭스 — Accept × 예외 × 응답 형식

5.1 옛 동작

Accept SSR endpoint (form POST navigation) XHR endpoint (submitAsync fetch)
NotFoundConflictDomain fallback NotFoundConflictDomain fallback
text/html, *, */* JSON ⛔JSON ⛔JSON ⛔ JSON ⛔JSON ⛔JSON ⛔
application/json, text/html;q=0.9 N/AN/AN/A HTML/JSON 혼재 ⚠혼재 ⚠혼재 ⚠

⛔ = SSR 사용자가 raw JSON 본 케이스. ⚠ = XHR 이 HTML 응답 받을 가능성.

5.2 새 동작

Accept SSR endpoint (form POST navigation) XHR endpoint (submitAsync fetch)
NotFoundConflictDomain fallback NotFoundConflictDomain fallback
text/html, *, */* error.html ✅error.html ✅error.html (500) ✅ N/AN/AN/A
application/json (만) N/AN/AN/A JSON ✅JSON ✅JSON (500) ✅
완전 분리 — Accept 헤더만으로 routing 이 결정. 각 흐름에 친화 응답.

7. 미해결 영역 + 후속

7.1 사용자 보고 케이스 B 의 진짜 원인 — 진단 보류

사용자 흐름 :

  1. OS 버전 삭제 → 자식 ISO 도 cascade trash 이동 (trashed_path 설정, disk 의 ISO 가 trash 위치로 mv)
  2. OS 만 복구 (nudge 의 메타 부분만, cascade=false) → 자식 ISO 의 trashed_path / trash disk 파일은 그대로 있어야 정상
  3. 자식 ISO 단독 복구 시도 → trashService.moveBack 가 trash 파일 → 원래 위치 mv. 트랜잭션 commit. redirect 후 selectKey/includeDeleted 사라짐 → 사용자 view 에서 ISO 안 보임
  4. 사용자가 다시 '삭제된 항목' 체크 + 같은 ISO 클릭 → trashed_path 가 disk 에 부재 → RestoreTrashLostException

본 hotfix 는 단계 3 의 redirect 보존 부재 만 해소. 단계 4 의 "왜 trashed_path 가 disk 에 없는가" 는 trans rollback / 다른 흐름의 silent cleanup 의심. 단독 진단 슬라이스 필요.

의심 영역
  • nudge 의 OS 만 복구 흐름 (OSNudgeService 또는 OSMetadataNudgeController) 이 자식 ISO 의 trashed file 을 silent 정리할 가능성
  • 첫 ISO restore 시도가 부분 성공 (trash mv → entity update) 후 어떤 후속 가드 (markerService.write 등) 실패로 트랜잭션 rollback. trash 파일은 이미 mv 됨 + entity 변경은 rollback. 다음 시도 시 trash 부재.
  • S5-2-3 시점에 자식 ISO 의 trashed_path 가 잘못 설정되어 처음부터 disk 부재인 경우 (Ghost 분류)

7.2 본 hotfix 의 범위 밖

  • 다른 도메인 (BIOS / BMC / Subprogram / Board) 의 lifecycle controller 도 selectKey/includeDeleted 보존 fix 필요 — 본 hotfix 는 OS area 만
  • updateISO endpoint 의 redirect 도 보존 미적용 (간단 옵션 추가 필요)
  • OSNudgeService 의 confirm 흐름의 redirect — 미점검

7.3 후속 슬라이스 후보

  • P0 진단 슬라이스 — case B 의 trashed_path/disk 불일치 원인 추적
  • R1-4-2 에 포함될 영역 — IsoRegistrationService 분리 시 nudge / trash 흐름 정리 동반
  • 다른 도메인의 selectKey 보존 (R2 영역)
