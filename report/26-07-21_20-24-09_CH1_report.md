# CH1 저장소 재편과 정비 수행 보고

작성 2026-07-21 KST. 대상 브랜치 `main`(HEAD 84ff8f6). 이 문서는 2차 개발 진입을 앞두고 수행한 브랜치 구조 재편과 저장소 정비(CH1)의 수행 내역을 기록한다.

## 1. 개요

두 가지를 수행했다. 첫째, 단일 작업 라인 `renew/main` 을 실 배포와 개발을 분리한 세 계층 브랜치 구조로 재편했다. 둘째, 그 후속으로 저장소를 정비해(CH1) 외부인이 클론해서 바로 기동할 수 있게 하고, 기본 브랜치 방문자에게 4개월치 작업이 보이게 했다.

재편의 설계 근거와 사용자 결정 6건은 `discussion/26-07-21_19-12-31_branch-worktree-restructure_discussion.md` 에 있다. 이 보고서는 그 결정을 실제로 어떻게 집행했는지를 다룬다.

## 2. 브랜치 구조 재편

### 이전 상태의 문제

`main` 이 3월 19일에 멈춘 35파일짜리 프로토타입(14fe1ec)이었고 README 가 없었다. 그것이 GitHub 기본 브랜치라, 저장소 링크를 받은 사람은 실제 작업 145커밋을 하나도 보지 못했다. 실작업은 `renew/main`(668a5cd) 단일 라인에 있었다.

### 집행

`main` 이 `renew/main` 의 순수 조상임을 확인해, 이력 재작성이나 force push 없이 처리했다. 커밋 해시, 공동 작성자 표기, 기여 그래프가 모두 보존된다.

| 브랜치 | 이전 | 이후 | 역할 |
|---|---|---|---|
| `main` | 14fe1ec 프로토타입 | 668a5cd 로 fast-forward | 실 배포 소스, GitHub 기본 브랜치 |
| `dev` | 1c9dcc6 구 라인 | 668a5cd 로 이동 | 개발 완료 도달점 |
| `renew/main` | 실작업 라인 | 삭제(로컬, 원격) | |

착수 전 안전망으로 태그 두 개를 만들어 원격에 push 했다. `v0-prototype`(구 프로토타입 14fe1ec)과 `archive/dev-legacy`(구 dev tip 1c9dcc6)로 원본 도달성을 보존한다. 구 `dev` 의 고유 커밋은 병합 커밋 하나뿐이고 그 두 부모가 모두 `renew/main` 의 조상이라, `dev` 를 새 라인으로 옮겨도 내용 손실이 없음을 검증했다.

### 잔재 정리

Claude 에이전트 작업 잔재인 워크트리 5개와 레거시 feature 워크트리 2개를 제거했다. 삭제 대상 브랜치 7개가 모두 `main` 의 조상임을 확인해 커밋 손실이 없다. `dev` 절대경로를 가리키던 실효 없는 `core.hookspath` 설정을 해제했다. 최종 원격은 `main` 과 `dev` 두 브랜치와 태그 두 개만 남는다.

### 안전성 확인

에이전트 워크트리의 미커밋 파일 13건이 모두 병합 이전 버전이고 `renew/main` 이 상위집합임을 파일 단위로 대조했다. 특히 `PathReconciliationService` 와 `MarkerLayout` 의 차이는 손실이 아니라 HF4-5 가 분기를 enum 상수별 메서드로 승격한 결과임을 확인했다.

## 3. 저장소 정비 (CH1)

새 브랜치 모델의 첫 작업 단위로, `chore/CH1-repo-tidy` 브랜치에서 여섯 커밋으로 작업한 뒤 PR #5 로 `dev` 에 병합하고 `main` 으로 릴리스했다.

### 클론 후 기동 가능화

이전에는 `@Entity` 27개에 대해 `CREATE TABLE` 이 9개뿐이었고 `ddl-auto=validate` 라 Hibernate 가 스키마를 만들지 않아, 클론한 사람이 애플리케이션을 기동할 수 없었다.

격리된 Docker MariaDB 11.4 에 애플리케이션을 `ddl-auto=create` 로 한 번 기동해 Hibernate 가 생성한 전체 스키마 25개 테이블을 덤프하고, `ddl/schema.sql` 로 정리했다. 알파벳 순 덤프라 자식 테이블이 부모보다 먼저 나오는 것을 감안해 `FOREIGN_KEY_CHECKS` 래퍼를 넣었고, 새 빈 DB 에 적재해 25개 테이블이 오류 없이 생성됨을 검증했다. `docker-compose.yml` 은 이 스키마를 초기화 스크립트로 자동 적재한다.

### CI

GitHub Actions 워크플로를 추가했다. 테스트에 `@SpringBootTest` 를 쓰는 클래스가 0건이라 `./gradlew test` 가 데이터베이스 없이 통과하므로, CI 는 JDK 21 에서 테스트만 돌리고 DB 서비스 컨테이너가 필요 없다. 최초 작성에서 액션 경로를 `actions/setup-gradle` 로 잘못 적어 CI 가 실패했고(존재하지 않는 액션), PR #6 으로 `gradle/actions/setup-gradle@v4` 로 교정해 `main` CI 초록을 확인했다. 실제 실행 테스트는 1,345건, 실패 0건이다.

### README 재작성

인수인계 담당자가 보는 첫 문서 성격과 외부 평가자가 보는 소개 성격을 섞어 재작성했다. 복수 초안을 만들어 두 독자층 적합성과 사실 정확성으로 비평한 뒤 종합했다.

앞부분에 제품 흐름과 설계에서 눈여겨볼 점(테스트가 DB 없이 통과하는 구조, 예외를 분기문 대신 다형성과 advice 로 라우팅, HMAC 마커 무결성, 활성 상태 own/effective 분리, 원시 타입 회피)을 두고, 이어서 클론 기동 절차, 저장소 구조, 확장 지점, 문서 지도, AI 협업, 브랜치 모델을 배치했다. README 가 언급하는 클래스와 인터페이스 13종이 모두 코드에 실재함을 검증했고, iPXE 부팅 채널이 실패도 200 재시도 스크립트로 변환한다는 서술은 `ExecutionRestController` 의 예외 핸들러로 확인했다.

현 README 의 죽은 참조 두 건을 교정했다. `docs/user-guide/` 는 존재하지 않아 예정으로 표기했고, `redfish_materials/` 는 `.gitignore` 대상이라 저장소에 없으므로 런타임 외부 제공 디렉터리임을 명시했다.

### AI 협업 공개

역할 분담을 명시했다. 설계 검토, 체크포인트 승인, 브라우저 검증은 소유자가 하고 구현 코드 작성은 Claude Code 가 한다. 병렬 에이전트가 구현하고 별도 검수 에이전트가 반려하는 파이프라인도 있는 그대로 적었다. 이미 커밋 99/145 에 공동 작성자 표기가 공개되어 있어 은폐는 선택지가 아니었고, AI 검토를 인간 검토로 옮겨 적지 않았으며 self-merge 를 self-merge 라 불렀다.
