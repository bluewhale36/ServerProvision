② 프로그래밍 측면 — 어떤 방향으로 만들었나

  • 분기 대신 전략 등록 — 해결([적용])과 확인([다시 점검]) 모두 "종류별 전략 bean 을 Map 으로 조립"하는 구조. 새 종류의 해결 추가 = 클래스 1개 등록으로 끝나고 서비스 본문의 if-else 는 늘지 않는다. 종전의 Path.of(null) 잠복 NPE 도 이 치환으로 소멸.
  • 의미는 enum 이 단일 소유 — 해결 등급(DriftResolutionMode 3단), 확인 지원(recheckable), 사용자 문구가 전부 DriftKind 한 곳에 — 서버 가드·버튼 노출·배지 색·무인 자동 자격이 같은 축을 읽어 화면과 서버가 어긋날 수 없음.
  • 새로 만들지 않고 재사용 — 복귀 해결=기존 복원 절차(안전장치 계승), 소실 정리=기존 영구삭제 파이프라인(감사 기록 공짜), 격리 이동=기존 mv 인프라, 자원명 검증=기존 TypedNameGuard. 신규 FS 로직은 재격리 1곳뿐이며 그마저 기존 역보상 패턴을 따름.
  • 불변·감사 존중 — 점검 기록(Drift)은 수정하지 않는다(해소 시 제거만). 위험 액션은 전부 [AUDIT] 로그, 소실 정리는 감사 테이블에 이력.
  • 정리도 일 — 실체 없던 상수 2종, 사용처가 사라진 SPI(인터페이스+6 어댑터+저장소 4곳)를 반쪽 상태 없이 제거. DB 칸 확장 2건은 부여받은 권한으로 직접 적용·검증.
  • 테스트 1078 → 1145 (신규 순증 67) — 모든 신규 분류·해결·거절 경로에 단위+HTTP 시나리오, 의도된 동작 변화는 기존 "한계 문서화 테스트"를 새 계약으로 갱신.
