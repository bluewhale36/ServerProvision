2026-07-19 전수 점검에서 설계 특성으로 관찰됐던 두 항목(O-3 · O-1)에 대해 사용자가 내린 결정 두 건이 구현되어 main 에 통합됐습니다. O-3 은 자원 무결성 점검(reconciliation) 화면의 스캔 이력 건수 배지가 "지금 남은 미해결 건수"로만 표시돼, 드리프트를 해결할 때마다 과거 보고서의 배지까지 함께 줄어들어 "그 스캔에서 몇 건이 탐지됐었는가"라는 역사적 사실이 사라지던 문제입니다 — 탐지 시점 건수를 보고서에 스냅샷(snapshot, 시점 고정 저장)으로 보존하고 "탐지 N · 미해결 M" 으로 병기하도록 했습니다. 항목전수 점검에서의 관찰사용자 결정
O-3 스캔 이력의 건수 배지가 '미해결 잔수'로 동적 감소한다 — 해결하면 과거 보고서도 "0 건" 표기가 되어 탐지 이력이 소실된다. 감사(audit) 관점에서 "탐지 당시 N건 / 미해결 M건" 병기로 바꾼다. 탐지 건수는 보고서에 스냅샷으로 영속 보존.
O-1 PATH_DRIFT(경로 이동) 탐지 범위가 활성 자원 상위 폴더의 합집합 + reconciliation.scan.extra-roots 설정이라는 사실과, 범위 밖 이동이 '자원 소실'로 보고되는 동작·보강법이 화면 어디에도 없다. 점검 화면에 스캔 범위와 extra-roots 보강 경로를 안내한다.

② 유스케이스 전후

O-3 스캔 이력 배지가 해결할수록 줄어 탐지 이력이 사라지던 문제

상황: 운영자가 /maintenance/reconciliation 에서 자원 무결성 점검을 돌려 드리프트 4건을 탐지한다. 이후 각 드리프트를 [적용] 또는 [보고 닫기] 로 하나씩 해결한다.

전: 배지가 표시하던 값은 보고서에 남아 있는 자식 드리프트 행 수(drifts.size())였다. 해결은 자식 행을 orphanRemoval(고아 제거) 로 물리 삭제하므로 배지가 4 → 3 → … → 0 으로 줄고, 상단 요약줄도 같이 줄었다. 전부 해결하면 그 스캔의 보고서가 "0 건"(처리 완료 초록)이 되어, 나중에 이력을 열어봐도 "그때 몇 건이 탐지됐었는지" 알 수 없었다 — 감사 추적이 불가능했다.

후(실측): 보고서가 만들어질 때 탐지 건수를 detected_drift_count 컬럼에 고정 저장한다(해결로 감소하지 않음). 화면은 "탐지 N · 미해결 M" 으로 병기한다 — 4건 탐지 후 전부 해결하면 배지가 "탐지 4 · 미해결 0"(처리 완료 초록) 으로 남아, 미해결이 0 이라는 현재 상태와 탐지 4 라는 역사적 사실이 동시에 보인다. 색 판정은 종전대로 미해결 기준을 유지해, 미해결이 1건이라도 있으면 회색(주의) 배지다.

O-1 스캔 범위와 '자원 소실' 보고 조건이 화면에 안내되지 않던 문제

상황: 운영자가 자원 파일을 스캔 범위 밖(예: 별도 백업 폴더)으로 통째 옮긴 뒤 점검을 돌린다.

전: 스캔은 활성 자원의 상위 폴더 합집합과 reconciliation.scan.extra-roots 설정 폴더만 훑는다. 범위 밖으로 옮긴 자원은 마커를 찾을 수 없어 '경로 이동'이 아니라 '자원 소실' 로 보고된다. 이 동작 자체는 설계 의도이지만, 운영자가 그 사실을 알 수 있는 곳은 서비스 코드의 주석(Javadoc)뿐이었다 — 화면 상단 설명에도, '자원 소실' 카드의 권장 조치에도 extra-roots 언급이 없어, 운영자는 왜 '경로 이동'으로 안 잡히는지, 어떻게 보강하는지 알 길이 없었다.

후(실측): 점검 화면 상단 배너에 ① 스캔 범위 = 활성 자원 폴더 + extra-roots ② 범위 밖 이동은 '자원 소실'로 보고됨 ③ extra-roots 에 경로를 추가해 범위를 넓힐 수 있음, 세 사실을 적었다. 아울러 '자원 소실' 카드의 권장 조치에도 "점검 범위 밖으로 옮긴 경우라면 reconciliation.scan.extra-roots 에 등록하면 다음 점검부터 '경로 이동'으로 탐지된다" 한 문장을 덧붙였다.

③ 설계 결정 (대안 비교 · 검수 통과 논거 포함)

결정채택탈락 대안과 사유
D1. 탐지 건수를 어떻게 보존하나 drift_report.detected_drift_count 스냅샷 컬럼. 보고서 생성(자식 드리프트 추가) 시점에 확정되고 이후 불변. "탐지 건수는 보고서 생성 시점에 확정되는 값"이라는 도메인 사실과 일치한다. 선례 R9-5(drift.display_name 스냅샷 — 자원 영구삭제 후 보고서에서 실명이 사라지던 같은 구조의 문제를 컬럼 하나로 해결) 와 동일 패턴이라 컬럼 1개·ALTER 1회로 끝난다. 자식 행 집계 쿼리로 복원: 불가 — 해결이 자식 행을 orphanRemoval 로 물리 삭제하므로 집계할 원본이 남지 않는다. 삭제를 상태 플래그(soft-flag)로 바꾸면 가능하지만 "드리프트는 불변, 해소 = 행 제거"라는 기존 설계와 resolution/recheck 계층 전반을 갈아엎는 큰 변경(churn) 이라, 표시 수치 하나의 동기로는 과하고 병렬 슬라이스 HF4-5 와도 충돌한다. 스캔 이벤트 로그 테이블 신설: 수치 하나를 위해 신규 테이블·엔티티·보존 정책까지 만드는 과설계(over-engineering) — 감사 추적 요구가 실제로 생기는 시점에 도입해도 늦지 않다.
D2. 스냅샷을 어디서 기록하나 DriftReport.addDrift() 내부에서 detectedDriftCount++, removeDrift() 는 감소시키지 않음. "탐지 수 = 이 보고서에 추가된 적 있는 드리프트 총수"라는 정의가 엔티티 안에서 단일 진실 출처(SSOT — Single Source of Truth)로 성립해, 저장 호출부는 아무것도 기억할 필요가 없다. 강제 적용 임시 보고서(persistAndForcedApply)도 자동으로 탐지 1이 기록된다. JPA 가 DB 에서 엔티티를 다시 읽을 때는 컬렉션 필드를 직접 주입할 뿐 addDrift() 를 타지 않으므로, 재로딩 시 증가 부작용이 없다. 저장 호출부에서 빌더로 .detectedDriftCount(drifts.size()) 명시 전달: 저장 호출부가 두 곳(performScan / persistAndForcedApply)이라 각각 기억해야 하고, 새 저장 경로가 생기면 누락이 침묵으로 지나간다. 서비스 파일 수정 범위도 커져 HF4-5 충돌면이 넓어진다 — 탈락.
D3. 스냅샷 없는 구행(도입 이전 저장 행) 표기 INT NOT NULL DEFAULT 0 — 구행은 0. 표시 규칙은 "스냅샷이 0 이면 미해결 수로 대체". 핵심은 신행에서 스냅샷 0 은 곧 "탐지 0건"이고, 그때 미해결도 반드시 0(자식이 애초에 없고 줄기만 하므로)이라 대체 표기가 항상 참이라는 점이다 — 0 경계의 의미 충돌이 구조적으로 없다. 대체 규칙의 SSOT 는 DriftReport.getDetectedDriftCountForDisplay() 하나이고, 화면 표시 두 곳(C1 배지·상단 요약줄)은 서비스가 내려준 값을 출력만 한다. NULL 허용 컬럼(null = 구행): 표기 결과는 같지만 Integer 박싱과 null 분기를 도처에 만든다. R9-5 의 display_name 은 "null 이면 다른 정보(타입 #id)로 대체"라 null 구분이 필수였지만, 여기는 0 이 자연스러운 경계라 null 이 주는 정보가 없다 — 원시 int 유지가 낫다. 탈락.
D4. O-1 안내 문구의 위치 ① 점검 화면 상단 설명 배너에 안내 줄 추가 ② DriftKind.MISSING.recommendedAction 문자열에 한 문장 추가. ②는 R9-2 가 확립한 "사용자에게 노출하는 드리프트 문구는 DriftKind enum 이 SSOT" 원칙 그대로라, 카드·확인 모달·추후 가이드가 전부 한 곳을 따른다. 이것은 enum 상수 신설이 아니라 기존 상수(MISSING)의 문구 수정이므로, 본 캠페인의 scope 경계(DriftKind 상수 신설은 병렬 HF4-5 소관, 본 슬라이스 금지)와 무관하다. 템플릿 카드에 조건부 하드코딩 문구: MISSING 문구의 두 번째 소유자를 만든다 — R9-2 이전의 "템플릿 하드코딩 ×3곳" 사고 재발 경로. 탈락. 별도 도움말 페이지/모달 신설: 안내 세 문장을 위해 URL·뷰·라우팅을 새로 만드는 것은 과함 — 필요 시점에 확장. 탈락.

구행 backfill비목표. 도입 이전 행의 탐지치는 복원 불가 — 자식 드리프트 행이 apply/dismiss 로 이미 물리 삭제(orphanRemoval)되어 복원할 원본이 없다. default 0 으로 두면 화면·API 매핑이 getDetectedDriftCountForDisplay() 로 미해결 잔수를 대체 표기하고, 보고서 보존 정책(FIFO retention, 기본 100건 유지)에 따라 구행이 밀려나며 자연 해소된다. sql 주석에 명시했다.
적용 주체실행 환경은 ddl-auto=validate 라 이 DDL 적용 전에는 앱이 기동되지 않는다(Schema-validation: missing column [detected_drift_count]). claude_code 계정은 ALTER 권한이 없으므로 ALTER 권한 계정(root 등)으로 수동 적용이 필요하다 — mysql -u root -p server_provision < sql/HF4-4_drift_report_detected_count.sql 후 SHOW CREATE TABLE drift_report; 로 컬럼 존재를 확인한다.

⑦ 잔여 · 후속

  • 실 DB 컬럼 반영은 수동 적용 대기 — 위 sql 을 ALTER 권한 계정으로 적용하기 전까지 실 DB 기동 시 스키마 검증에 걸린다. 샌드박스는 ddl-auto=update 로 자동 생성되어 실측을 마쳤다.
  • 구행 탐지치는 소급 복원하지 않음 — 자식 행이 이미 물리 삭제되어 원본이 없다. 화면 대체 표기 + FIFO retention 으로 자연 해소되며, 데이터 수술은 하지 않는다.
  • HF4-5(DriftKind 상수 신설) merge 대조 — 본 슬라이스는 DriftKind.java 의 MISSING 문구와 list.html 을 만졌다. 병렬 HF4-5 가 같은 두 파일을 상수 추가로 건드릴 수 있어, merge 시 인접 충돌 여부 대조가 필요하다.
  • CP5 브라우저 최종 확인은 사용자 단독 — 본 위임 범위는 CP4(구현·테스트·검증)까지다. 배지 병기·안내 문구·구행 표기의 브라우저 E2E 는 사용자 몫으로 남는다(샌드박스 실측으로 사전 확인 완료).
