§1. 구현 요약

plan 의 4개 결정을 전부 이행했다. 핵심은 두 가지가 한 쌍이라는 것 — 탐지 로직이 한 번도 생성한 적 없는 RESOURCE_RENAMED 계열 상수 2종을 삭제(D1)하고, 그 정리가 남기는 서사 공백("파일명만 바뀐 자원은 어떻게 드러나나")을 4a 본체 존재 검사(D2·D3)와 MISSING 권장 조치 문구 보강(D4)이 메운다. DriftKind 는 12종 → 10종, autoApplicable(자동 적용) 집합은 5종 → 4종이 됐고, 종전 deep 주기(기본 24h)까지 침묵하던 "마커 有·본체 無" 상태가 quick scan(기본 1h)에서 즉시 MISSING 으로 보고된다.

엔티티·DB 스키마·HTTP 계약·템플릿 전부 무변경 — plan 대로 순수 리팩토링 슬라이스(Step 2~7·9 N/A)로 끝났다.

§5. 동작 변화 · CP5 안내
CP5 에서 새 MISSING 이 나타날 수 있다 (정상) — 기존 환경에 "마커만 남고 본체가 없는" 자원이 있었다면, 적용 후 첫 quick scan 부터 MISSING 으로 보고된다. 종전의 침묵이 버그였고 보고가 올바른 동작이다. detail 에 "본체 파일 부재 — 파일명 변경 또는 삭제 가능성"이 표기되고, 권장 조치가 이름 변경 복원까지 안내한다.
  • 4b 경유 MISSING 의 detail 이 layout 명 병기 형식("다른 위치에 SIDECAR 마커는 있으나 본체가 부재…")으로 통합 — 정보 손실 없음.
  • REST JSON·화면 계약 무변화. RESOURCE_RENAMED 는 화면·API 에 노출된 적이 없어(탐지 미생성) 사용자 관점 제거 흔적도 없다.

CP5 확인 시나리오 (사용자)

  • SIDECAR 자원(ISO 등)의 본체 파일만 다른 이름으로 mv → [지금 점검] → MISSING + "본체 파일 부재" detail + 보강된 권장 조치 확인
  • 파일을 원래 이름으로 복원 → 재점검 → drift 소멸
  • 본체+마커를 함께 다른 이름으로 mv → 재점검 → PATH_DRIFT + [적용] 동작(RESOURCE_RENAMED 없이 동등 서사)
