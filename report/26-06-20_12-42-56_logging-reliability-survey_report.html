§1. 한눈 요약

현행 로깅은 Spring Boot 기본 설정 그대로(logback 설정 파일 0, logging.* 프로퍼티 0)이며, 신뢰할 수 있는 로깅의 가장 큰 결손은 요청 상관관계(correlation) ID 가 전무하다는 점이다. 여기에 사용자 액션 실패가 서버 로그에 거의 남지 않고(NotFound·일반 Conflict 핸들러 무로깅, ERROR 14건이 전부 비동기 경로), lifecycle 상태 전이 성공도 무로깅이라, 한 요청에서 무슨 일이 일어났는지·왜 실패했는지를 로그만 보고 복원할 수 없다. 방금 디버깅한 ghost catch-22 와 BIOS 펌웨어 업로드 차단(executable-DENY) 이 그 산 증거다.

핵심 결론 — 통증의 1차 근원은 "무엇을 로깅하느냐" 보다 ① correlation 부재 + ② 실패·전이 경로의 무로깅 + ③ 구조화 컨텍스트 미방출 + ④ 로그 설정 전무 의 복합이다. 다행히 해법은 추가 의존성 0 으로 가능하다 — Spring Boot 4 네이티브 구조적 로깅 + MDC requestId 필터 1개 + @ControllerAdvice 단일 진입점의 예외 로깅 표준화 + lifecycle 전이 로깅.

권고 한 줄

P1 MDC requestId(상관관계) → P2 advice 예외 로깅 표준화(레벨 규율 + 자원 컨텍스트, SSOT) → P3 전이/거절/skip 로깅 → P4 운영 JSON(ECS) + Async + 마스킹 2차. 모두 의존성 추가 없이 logback-spring.xml 신설 + 필터 1개 + advice·서비스 로깅 보강으로 달성.

사례 B — BIOS 펌웨어 업로드 차단 (executable-DENY)

ContentGuard:108 DENY 분기는 throw new ExecutableContentRejectedException(); 만 — 로그 0줄. 41개 파일 중 어느 파일이 어떤 MIME 으로 걸렸는지 서버 로그에 안 남는다. WARN 분기(111)는 file+mime 를 찍는데 DENY 는 안 찍고, (filename, mime) 생성자(17)는 인자를 받고도 super(GENERIC_MESSAGE) 로 버린다. 사용자는 "실행 가능 형식의 파일은 업로드할 수 없습니다" 만 보고 원인 파일을 못 짚음.

패턴 — 두 사례 모두 거절/skip 지점에 자원 컨텍스트(type·id·이유)를 가진 채 로그를 안 남긴다. 개편의 핵심 타겟이 바로 이 "조용한 실패" 다.
§5. 베스트프랙티스 (외부 조사 종합)

① actionable 로그 — 5W 기준

판정 기준 : "로그만으로 who/what/where/when/why 를 답할 수 있는가". 답하면 actionable, 못 하면 노이즈. 에러는 에러 자체 + 직전 이벤트를 함께 남겨 재현 가능하게. 각 로그가 담을 필드 : timestamp · severity · 메시지 · 컨텍스트(request_id/resourceType/resourceId/duration_ms/status) · 스택트레이스(에러 시). 출처: lanre.wtf, betterstack, honeycomb

② 레벨 규율 — ERROR vs WARN 결정 트리 (불가침)

레벨언제
ERROR요청한 작업을 완료할 수 없음(DB 연결 실패, 재시도 소진 외부 타임아웃, 결제 실패) → 고우선 알림
WARN시스템이 복구/우회했거나(재시도 성공, fallback) 예상된 실패(잘못된 입력·인증, deprecated 호출)
INFO의미 있는 비즈니스 이벤트(기동, 인증, 트랜잭션 완료) — "시스템의 서사"
DEBUG변수·페이로드·분기 등 진단용. 프로덕션 OFF, 조사 시 임시 ON 후 원복(Actuator /loggers)
안티패턴 — 입력검증·인증 실패를 ERROR 로 찍으면 alert fatigue → WARN/DEBUG. 본 프로젝트 "예외=진짜 예외, UX 모순은 UI 차단, 서버 가드는 안전망" 원칙과 정합 : 정상 흐름에서 도메인 가드가 발동하면 그건 direct POST/동시성/stale 같은 진짜 비정상 → WARN 이상으로 조사 대상화. 과거 보안예외 silent 흡수 사고(catch(DomainException))는 "삼키기" 그 자체. 출처: oneuptime, betterstack, edgedelta

③ 경계 1회 로깅 + 삼키기 금지

  • Either log OR throw, never both — 스택트레이스는 최상위 경계(@ControllerAdvice)에서 단 1회. 잡아서 로깅+재던짐 → 상위가 또 로깅(중복).
  • 삼키기 금지 — catch 후 "로깅하거나 래핑하거나 재던지거나" 셋 중 하나는 필수.
  • 메시지만 말고 ex 객체 전달 — log.error("...", ex) (SLF4J 가 스택 처리). e.getMessage() 만 = 스택 소실. printStackTrace() 금지.
  • over-catching 금지 — catch(Exception) 광범위 포획은 보안예외까지 흡수 → 가장 구체적 타입을 잡는다. (본 프로젝트 보안 silent 500 사고와 직결.)
정합 — 본 프로젝트의 @ControllerAdvice 단일 진입점이 "경계 1회 로깅" 의 이상적 위치. 컨트롤러별 try/catch 개별 로깅(복붙 회귀 진원)을 advice 의 타입별 1회 로깅 + 레벨 결정으로 끌어올리면 ① 스택 중복 제거 ② 보안예외 흡수 차단 ③ 레벨 규율 SSOT 를 동시 달성. 출처: gaetanopiazzolla java exception anti-patterns

④ 상관관계 · 작업 경계 · 구조화

  • MDC correlationId — thread-local 맵으로 모든 로그 줄에 자동 부착. 서블릿 필터에서 X-Request-Id 수신/없으면 UUID → MDC.put, finally 에서 MDC.clear(누수 방지), 응답 헤더 반영. 비동기는 TaskDecorator 로 MDC 복사. 출처: spring docs tracing, fati.dev, javabyexamples
  • 작업 경계 / canonical log line — 요청·saga·배치 경계마다 시작/종료, 가능하면 종료 시 단일 요약 엔트리(입력·신원·소요시간·결과). 단편 로그로 흩뿌리지 말 것.
  • 구조화(JSON/key-value) — Elasticsearch/Datadog 가 정규식 없이 필드 인덱싱 → 검색·상관·집계 가속. 일관된 키 네이밍(OpenTelemetry semantic conventions), 구조/비구조 혼용 금지. 출처: signoz, baeldung, digma
  • 민감정보 마스킹 — 비밀번호·토큰·PII 평문 금지. "개발자 기억"에 의존 말고 시스템 레벨(필터/Converter)에서 강제. 본 프로젝트 직결: Server.ipmiPassword · BMC 자격증명 · HMAC secret. 출처: skyflow, betterstack sensitive-data
§6. 도구·설정 — Spring Boot 4 / Logback (의존성 추가 0)
핵심 — Spring Boot 3.4+(4.0.3 동일)는 네이티브 구조적 로깅을 내장한다. logging.structured.format.console=ecs|logstash|gelf 로 외부 인코더 의존성 없이 JSON 로그 가능. 본 프로젝트(단일/소수 인스턴스 관리 도구)엔 ECS(Elastic Common Schema) 권장 — logstash-logback-encoder 도입은 over-engineering(필요 시 후일 도입). 출처: spring.io blog 2024-08, docs.spring.io logging

권장 구성 — logback-spring.xml 신설 + <springProfile> 분리

<configuration>
  <springProfile name="dev">
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
      <encoder><pattern>%d{HH:mm:ss.SSS} [%X{requestId}] %-5level %logger{36} - %msg%n</pattern></encoder>
    </appender>
    <root level="DEBUG"><appender-ref ref="CONSOLE"/></root>
  </springProfile>
  <springProfile name="prod">
    <appender name="JSON" class="ch.qos.logback.core.ConsoleAppender">
      <encoder class="org.springframework.boot.logging.logback.StructuredLogEncoder">
        <format>ecs</format>
      </encoder>
    </appender>
    <appender name="ASYNC" class="ch.qos.logback.classic.AsyncAppender">
      <queueSize>1024</queueSize><discardingThreshold>0</discardingThreshold><neverBlock>false</neverBlock>
      <appender-ref ref="JSON"/>
    </appender>
    <root level="INFO"><appender-ref ref="ASYNC"/></root>
  </springProfile>
</configuration>
  • 반드시 logback-spring.xml 명명(logback.xml 은 <springProfile> 미해석).
  • MDC 값은 ECS/structured 출력 시 JSON 에 자동 포함. dev 콘솔은 %X{requestId} 슬롯.
  • AsyncAppender 무손실 구성: queueSize=1024 · discardingThreshold=0 · neverBlock=false. 운영 파일 appender 에만. 출처: logback.qos.ch/performance
  • correlationId: Micrometer Tracing 이 클래스패스에 없어 내장 traceId 미작동 → 직접 MDC 필터(의존성 0) 권장. 단일 인스턴스라 분산추적 불요.
  • 마스킹 2차: 코드 1차(이미 BmcPasswordService:41·BiosRedfishPayloadAssembler:67 가 값 미출력) + logback %replace 정규식 치환(커스텀 클래스 불필요). 출처: baeldung logback-mask
추가 의존성 : 0개. 변경 = logback-spring.xml 신설 + 소수 logging.*(ECS service.name 등) + (선택) MDC 필터 1개. JPA show-sql 은 이미 false(application.properties:17)로 적절.
