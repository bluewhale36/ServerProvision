§1. 후보 B — 미생성 DriftKind 6종: 종류별 전수 판정 (1순위, 결정이 본체)

배경 — 무엇이 "미생성"인가

DriftKind enum 12종 중 6종은 ① MK3 가 javadoc 으로 의미를 확정했고 ② R9-2 가 라벨·설명·권장 조치 문구까지 부여했으며 ③ 그중 3종은 자동 적용 허용 목록에도 들어 있으나, performScan 어디에도 이 6종을 생성하는 분류 코드가 없다(grep 전수 — R9 사전 조사에서 확인). MK3-2 가 "12 DriftKind 3축 매트릭스" 슬라이스로 미룬 문서-구현 괴리다. 아래 표가 6종 각각의 정의 원문, 현재 실제로 일어나는 일(코드 실측), 미감지의 실해악이다 — 이 표가 결정의 본체다.

§2. 후보 C — 재발급 부분 실패 표면화 (2순위, S)

현재 흐름과 남은 구멍

performReissue 는 자원별로 마커 read → 재서명 → write → 엔티티 갱신을 수행하고, 자원 단위 실패를 catch 로 흡수해 ReissueResult.failures("자원 라벨 : 사유" 목록)에 쌓는다. R9-1 이 건수는 완료 토스트로 표면화했지만("성공 41건, 실패 2건"), 어느 자원이 왜 실패했는지는 여전히 서버 로그([AUDIT] WARN)로만 남는다.

공짜 해법 — 기존 드리프트 화면이 실패 목록 역할을 한다

실패 유형의 대부분(마커 read 실패·디스크 write 실패)에서 마커는 구 secret 서명(또는 손상) 상태로 남는다. 따라서 다음 quick 점검의 서명 검증이 이들을 전부 SIGNATURE_INVALID 드리프트로 잡고, R9-2·R9-5 가 만든 화면(라벨·감지 내용·권장 조치·자원 실명)이 실패 목록 + 조치 안내 역할을 그대로 수행한다. 전용 UI 를 만들 이유가 없다:

runReissueAsync 완료 & 실패 > 0
  → (a) 자동으로 quick 점검 1회 트리거   ← 권장
        · 위치: finally 의 running.set(false) 이후 (동시 실행 가드와 순서 충돌 없음)
        · 이미 다른 점검이 돌면 ReconciliationAlreadyRunningException catch 후 무시 (다음 주기 점검이 잡음)
  → (b) 토스트에 "지금 점검을 실행해 실패 자원을 확인하세요" 안내 + 버튼   ← 백엔드 0 이나 사용자가 안 누르면 끝
결과: 실패 자원들이 SIGNATURE_INVALID 로 목록화 — 신규 화면 0, 후보 A-2(단일 재서명)와 자연 결합
공짜 해법의 한계 1건 (정직하게 명기) — 마커 write 까지 성공하고 엔티티 갱신 단계만 실패한 경우, 디스크 마커는 이미 새 secret 으로 유효해 다음 점검(디스크 기준 검증)에 드리프트로 나타나지 않는다. 이 유형만 로그 의존이 잔존한다 — 엔티티 저장 서명과 디스크의 괴리를 별도 무결성 검증 Job 이 잡는지는 착수 시 CP1 에서 확인할 항목.
결정 블록 C
  1. C-1. 방식 — (a) 자동 트리거 권장. 변경량: runReissueAsync 수 줄 + 토스트 문구 1건("실패 자원은 방금 시작된 점검 결과에서 확인하세요") + 테스트 2건(실패>0 시 트리거 verify / 동시 실행 시 무시). (b) 는 백엔드 0 이지만 안내를 놓치면 끝이라 자동화 가치가 낮다.
  2. 규모 S (1 슬라이스), 선행 조건 없음. 착수 승인 시 인벤토리 코드 부여 + Notion 페이지 신설.

액션별 설계 스케치 + 이중 의도 검증 명제 대조

MK3 §13 (이중 의도 검증 명제, 원문): "한 케이스의 drift 감지가 둘 이상의 운영자 의도 변화를 동시에 포함하면 자동 적용 대상에서 제외한다."
슬라이스액션과 동작 흐름 (구체)명제 대조 · 가드규모
A-1 MISSING
"복원 확인 재점검"		 드리프트 카드에 [이 자원만 재점검] — 기존 scanForResource(type, id)(:691, 읽기 전용·영속화 없음) 호출 → 결과 empty(마커 복원 확인)면 해당 drift 를 dismiss 재사용으로 제거 + 토스트 "정상 확인 — 드리프트를 해소했습니다" / 여전히 문제면 현 상태 유지 + 사유 토스트. 전체 스캔 없이 복원 검증 루프가 화면 안에서 닫힌다. 의도 0개(읽기 + 확인) — 명제 무관. 가드 불요. S~M · 첫 슬라이스 권장
A-2 SIGNATURE_INVALID
"단일 재서명"		 [이 자원만 재서명] — 전체 일괄(performReissue)의 단일 자원판. 재서명 전 본체 존재·재검증 절차를 끼워 "손상 복구"와 "변조 합법화"를 구분. 후보 C 와 결합 시 "재발급 실패 자원 개별 재시도" 동선 완성. 의도 1개("이 마커는 정당하다")이나 변조 자원을 사용자가 임의 정당화하는 통로가 될 수 있음 — 검증 선행 + ConfirmModal. 자동 적용 불가(수동 전용). M
A-3 ORPHAN
"정리 / 재등록 진입"		 ① [휴지통으로 정리] — 파일+마커를 .soft-deleted/ 규약으로 이동(즉시 삭제 아님 — 기존 trash 인프라 재사용, 복구 여지 보존) ② [재등록 진입] — 자원 종류별 등록 폼 URL 에 경로 프리필(분기 아님 — ResourceType → URL 매핑). ①은 의도 1개("이 파일은 불필요") — 파괴적이라 ConfirmModal 필수, trash 경유로 완화. ②는 이동뿐. M
A-4 HASH_MISMATCH
"재업로드 / 기준 승인"		 ① [원본 재업로드 진입] — 해당 자원 편집/업로드 폼으로. ② [현재 내용을 새 기준으로 승인] — 재해싱 + 마커 갱신. ②는 의도 2개("내용이 바뀐 걸 안다" + "그 변경이 정당하다") — 명제 위반 소지가 가장 큼. 제공한다면 typed-name 급 가드 전제, 또는 ①만 제공하고 ②는 보류가 안전. M~L · 보류 가능

재사용 자산 (R9 가 만든 것 — 신규 발명 최소)

  • modal 파이프라인: ConfirmModalType 상수 추가(OCP)와 페이지 로컬 바인딩 — R9-3 선례 그대로.
  • 문구 SSOT: 액션이 생기면 해당 kind 의 recommendedAction 을 "버튼 안내형"으로 enum 한 곳만 수정.
  • 완료 피드백: sessionStorage 토스트+reload(R9-1) / 실명 표시(R9-5) / trash 규약·typed-name 가드(S5).
  • 주의: forced-apply saga(SoftDeleteIntentService → persistAndForcedApply)가 apply 를 공유하므로, 신규 액션은 apply 확장이 아니라 별도 엔드포인트로 두는 편이 경계가 깨끗하다 — CP1 에서 확정.
결정 블록 A
  1. A-1(시점) — CP5 + 운영 관찰 후 권장. 어떤 드리프트를 실제로 자주 만나는지가 슬라이스 우선순위의 재료다. 다만 A-1(재점검)은 읽기 전용이라 관찰 전 착수해도 리스크가 없다 — "지금 A-1 만 먼저" 도 유효한 선택지.
  2. A-2(첫 슬라이스) — A-1 권장(최저비용·무가드·기존 메서드 재사용). 이후 C 채택 시 A-2 가 자연스러운 2번째.
  3. A-4 의 "기준 승인" 액션은 이중 의도 명제와 정면 긴장 — 캠페인에 포함할지 자체를 별도 결정 항목으로 남긴다.
§4. 결정 체크리스트 — 이 문서를 읽고 바로 답할 것들
  • B-1 RENAMED 계열 2종 정리 (권장: 승인 — 중복 kind 실측 근거 §1 표) → 승인 시 S 슬라이스 착수
  • B-2 SOFTDEL·TRASH 4종 노선 (권장: 관찰 후 결정 / 대안: TRASH_LOST 만 선행 구현)
  • C-1 재발급 실패 → 자동 점검 트리거 (a) 착수 (권장: 승인 — S)
  • A-1 in-app 액션 착수 시점 (권장: 관찰 후 / 대안: 읽기 전용인 A-1 재점검만 선착수)
  • A-2 첫 슬라이스 = MISSING 재점검 동의 여부
  • 승인 항목의 인벤토리 코드 부여(R10* 또는 MK* — 사용자 결정) + Notion 페이지 신설 지시

본 문서는 착수 전 조사·설계 방향 기록이다 — 착수 시 각 슬라이스 CP1(plan html)에서 대안 비교·적대 검증을 표준 절차대로 다시 수행한다. v2 는 아직 미커밋 상태 — 검토 후 커밋 지시와 함께 처리.
