§1. 한눈 요약

(1) R3-5(Board) — BoardModelLifecycleService 의 중복 restore(Long) override 를 인터페이스 default 로 흡수 + purgeWithTypedNameCheck 의 inline typed-name 비교를 공통 TypedNameVerifier 위임으로 전환. §5. ◑ 순환 참조 분석 (이번 변경의 핵심 설계 이슈)

왜 R1-5 가 순환을 닫았나

R1-5 전에는 IsoLifecycleService 가 TypedNameVerifier 에 의존하지 않았다. R1-5 가 이 의존을 추가하면서, OS 도메인 고유의 스캐너 비대칭과 맞물려 eager 생성자 사이클이 닫혔다.

BEFORE 수정 — R1-5 가 추가한 의존이 닫는 사이클
  TypedNameVerifierImpl  (@Component)
     └─ List<MarkableScanner>   생성자에서 eager — scannerByType 맵 구성 → 모든 스캐너 즉시 인스턴스화
          └─ IsoMarkableScanner
               └─ IsoLifecycleService   ← 직접 final 주입 (이 스캐너만 ObjectProvider 미사용)
                    └─ TypedNameVerifier   ◀── R1-5 가 추가한 의존
                         ╰─────────────────────────╮
  ◀═══════════════════════ 사이클 폐쇄 ═══════════════════════╯
  ⇒ spring.main.allow-circular-references 미설정 → BeanCurrentlyInCreationException 로 bootRun 실패

왜 Board(R3-5)는 같은 변경에도 멀쩡했나 — 스캐너 비대칭

스캐너lifecycle service 주입 방식R1-5/R3-5 후 순환
BoardModelMarkableScannerObjectProvider<BoardModelLifecycleService> R3-3 에서 이미없음 — 지연 주입이 back-edge 차단
OSMetadataMarkableScannerObjectProvider<OSMetadataLifecycleService>없음

수정 — 형제 스캐너와 동형화

BEFORE

@RequiredArgsConstructor
class IsoMarkableScanner ... {
  private final ISORepository isoRepository;
  private final IsoLifecycleService
                    isoLifecycleService;
  ...
  void restoreFromTrash(Long id){
    isoLifecycleService.restore(id);
  }
  void purgeFromTrash(Long id){
    isoLifecycleService.purge(id);
  }
}

AFTER

@RequiredArgsConstructor
class IsoMarkableScanner ... {
  private final ISORepository isoRepository;
  private final ObjectProvider<IsoLifecycleService>
                    isoLifecycleServiceProvider;
  ...
  void restoreFromTrash(Long id){
    isoLifecycleServiceProvider.getObject().restore(id);
  }
  void purgeFromTrash(Long id){
    isoLifecycleServiceProvider.getObject().purge(id);
  }
}
AFTER 수정 — 지연 주입으로 eager 그래프가 DAG
  TypedNameVerifierImpl
     └─ List<MarkableScanner>
          └─ IsoMarkableScanner
               └─ ObjectProvider<IsoLifecycleService>   생성자에서 미해소(프록시만 보유)
                    ┆ getObject() 는 restoreFromTrash / purgeFromTrash 본문 실행 시점에만 해소
  ⇒ eager 의존 그래프에 back-edge 없음 → 사이클 없음 (bootRun Started 실측)
덤으로 확인된 정합 (D1 재확인) — IsoMarkableScanner.restoreFromTrash 는 단일 인자 restore(resourceId) 를 호출한다. ObjectProvider 가 반환하는 프록시의 single-arg restore(= ISO 가 보존한 override)가 @Transactional 진입점이 된다. 이 호출자의 존재가 "ISO 는 restore override 를 제거하면 안 된다"(R3-5 D1)를 코드로 재확인한다 — Board 는 그런 single-arg 호출자가 0 이라 제거 SAFE, ISO 는 존재하라 보존 필수.
왜 테스트로는 안 잡혔나 — 단위 테스트는 plain Mockito(Spring 컨텍스트 없음), @WebMvcTest 는 컨트롤러+모킹 빈만 로드하는 슬라이스라 verifier→scanner→service 전체 그래프를 와이어링하지 않는다. 순환은 full context refresh 에서만 터진다 → 그래서 Claude 의 bootRun 통합 검증이 필수였다.
