1. 요약 — 무엇을 고쳤나

2026-07-19 격리 샌드박스 전수 점검에서 두 항목이 잡혔다. F-6 은 ISOCreateRequest.description 에 길이 상한 검증(@Size)이 없어 1,600자 설명이 입력 검증(Bean Validation)을 통과했고, ISO 등록 background job 의 INSERT 단계에 이르러서야 MariaDB 1406(Data too long — iso.description VARCHAR(1024)) 으로 실패한 결함이다. 실패 자체는 오펀 격리 saga 가 DB_CONSTRAINT 사유로 정확히 회수해 데이터 유실은 없었으나, 사용자는 수 분의 업로드를 다 마친 뒤에야 실패 → 복구 페이지로 흘렀다. O-7 은 미매핑 500 오류의 JSON 응답 바디에 trace 필드로 전체 스택트레이스가 노출된 관찰이다.

HF4-2 는 이 둘을 함께 정비했다. F-6 은 개별 필드 한 곳을 때우는 대신, 관리(Management) 5개 자원 도메인과 프로비저닝 세팅(Provisioning setting) 의 Request 데이터 전송 객체(DTO, Data Transfer Object) 문자열 필드 약 40개를 엔티티 @Column(length) 와 전수 대조해, 불일치 19건을 표준 관례(@Size(max = 엔티티 length) + 한국어 메시지)로 정렬했다. 오류가 실제로 표시되지 않던 서버 사이드 렌더링(SSR, Server-Side Rendering) 재렌더 경로 7개 템플릿에는 오류 표시 슬롯을 보강했다. O-7 은 spring.web.error.include-stacktrace=never 로 응답 바디의 스택 노출을 고정 차단했다(서버 로그의 스택은 유지).

계약 정렬의 성격 — 설계 누락이 아니라 관례 미정렬

이미 bios·bmc·subprogram·biossetting 의 Create·Update 계열 DTO 는 "@Size(max) = 엔티티 length + 한국어 메시지" 관례를 지키고 있었다. 이탈해 있던 것은 그 관례가 확립되기(리팩토링 캠페인) 전에 작성된 OS·ISO·Board 3개 도메인, upload-intent 핸드셰이크 4종, 세팅 정의서 이름이다. HF4-2 는 새 검증 어휘를 도입하지 않고 저장소의 기존 관례에 이들을 맞췄다 — Spring 의 BindingResult 자동 매핑이 분기를 떠맡으므로 컨트롤러·서비스·엔티티는 무변경이다(엔티티 length 가 기준값이라 스키마도 무변경).

2. 길이 계약 전수 대조표 — Request DTO 문자열 필드 vs 엔티티 @Column(length)

아래는 이번에 정렬한 19개 필드 전체다. 상한값의 기준은 언제나 대응 엔티티 컬럼의 length 이며, @Size 는 그 값을 앞단으로 끌어와 제출 즉시 400 필드 오류로 거절하게 만든다. BMC 3종만은 예외적으로 엔티티 컬럼이 이중으로 매핑돼 있어(아래 별도 설명) effective(실효) 제약인 255 를 기준으로 삼았다.

BMC targetDirectory — 왜 255 로 하향했나

BMC 의 targetDirectory 는 BoardBMC.treeRootPath(firmware_path VARCHAR(1024)) 에 저장되는 동시에, legacy 컬럼 file_path VARCHAR(255) NOT NULL 에도 같은 값이 미러링된다(BoardBMC.legacyFilePath). 즉 검증 상한(1024)이 실제 DB 제약(255)보다 컸던 유일한 케이스로, 256~1024자 경로가 검증을 통과했다가 INSERT 에서 실패하는 F-6 과 동형(同型)의 잠복 결함이었다. F-6 을 고치면서 같은 구조의 결함을 알고도 남기는 것은 자기모순이므로, effective 제약인 255 로 상한을 정렬하고 그 이유를 각 DTO 의 주석으로 남겼다.

대조 결과 무변경 — 이미 정렬돼 있던 필드

  • bios·bmc·subprogram 의 Create·Update name(128)·version(64)·description(1024)·entrypointRelativePath(512), biossetting name(128)·description(1024) 는 이미 @Size 가 컬럼과 일치해 손대지 않았다(BMC targetDirectory 제외).
  • 세팅 정의서(setting) 의 Process 하위 문자열(파티션·계정·타임존 등)은 setting_process.payload_json JSON 컬럼에 통째로 저장돼 개별 컬럼 length 계약이 없다 — 길이 대조 대상이 아니며(§7 비목표), 길이 정책이 필요하면 도메인 결정이 따르는 별건이다.
3. 유스케이스 — 적용 전후 비교

이번 정비가 사용자 경험을 어떻게 바꿨는지 세 상황으로 풀어 쓴다.

상황 A — ISO 등록 폼에 1,600자 설명을 넣고 제출 (F-6 당사자)

적용 전 (AS-IS)
  1. 1,600자 설명이 @Size 부재로 입력 검증 통과.
  2. ISO 바이트 전송 + 등록 background job 시작 — 수 분 소요.
  3. INSERT 단계에서 MariaDB 1406 (Data too long, iso.description VARCHAR(1024)) 로 실패.
  4. 오펀 격리 saga 가 DB_CONSTRAINT 로 분류 → 파일 격리행 생성 → 사용자는 복구 페이지로 이동. 데이터 유실은 없으나 업로드를 다 기다린 뒤의 실패라 비용이 크다.
적용 후 (TO-BE)
  1. 제출 즉시 @Size(max=1024) 위반 → 400 필드 오류.
  2. "설명은 1024자 이하로 입력해주세요." 가 description 필드 슬롯에 표시.
  3. 업로드·등록 job 이 아예 기동되지 않음 — 자원 소모 0, 격리행 생성 0.

상황 B — BMC 등록에 256~1024자 대상 디렉토리 경로 입력

적용 전 (AS-IS)
  • @Size(max=1024) 를 통과 → treeRootPath(1024) 에는 저장 가능하나 legacy file_path VARCHAR(255) NOT NULL 미러링에서 INSERT 실패.
  • 검증 상한이 DB 제약보다 큰 F-6 동형 잠복 결함 — 사용자는 원인 필드를 모른 채 실패를 마주.
적용 후 (TO-BE)
  • 256자부터 앞단 400 필드 오류로 즉시 거절.
  • 행동 변경 인지 — 종전엔 통과(했다 실패)하던 256~1024자가 이제 제출 단계에서 막힌다. bios·subprogram(1024)과의 상한 비대칭은 legacy 컬럼 제거 마이그레이션 시 해소 예정(각 DTO 주석에 명기).

상황 C — 수정(edit) 폼에서 길이 초과 제출 시 오류가 무표시로 삼켜지던 UX 버그

수정 폼 계열은 컨트롤러가 검증 실패 시 뷰 이름을 그대로 반환하는 진짜 SSR 재렌더 경로다. 이 경로의 유일한 표시 수단은 템플릿의 <div class="n-error" th:errors> 슬롯인데, description 필드에 그 슬롯이 없었다. 그 결과 bios·bmc·subprogram 의 수정 폼은 이미 @Size 가 있었음에도 1,025자 입력 시 오류가 화면에 표시되지 않은 채 재렌더되는, 재현 가능한 display 버그를 갖고 있었다(같은 폼 안에서 name·version 은 슬롯이 있어 자기모순).

적용 전 (AS-IS)
  • SSR 재렌더 폼 7종(수정 6 + ISO 등록 form POST)의 description 에 n-error 슬롯 부재.
  • 길이 초과 제출 → 뷰는 다시 그려지지만 왜 저장이 안 됐는지 안내 없음.
적용 후 (TO-BE)
  • 7개 템플릿(os/edit·os/iso-edit·os/iso-new·board/edit·bios/bios-edit·bmc/bmc-edit·subprogram/subprogram-edit)의 description 에 인접 필드 마크업을 그대로 복제해 n-error 슬롯 추가(일부는 입력 강조용 is-invalid class 병행). 신규 CSS 0, 인라인 스타일 0.
  • bios·bmc·subprogram 수정 폼은 이번 슬롯 추가로 기존 무표시 버그까지 동시 해소. 나머지 4개(os/edit·iso-edit·iso-new·board/edit)는 신규 @Size 와 슬롯이 함께 활성화.
오류 표시 경로가 둘로 갈리는 이유 — 신규 등록(create) 폼은 XHR JSON 응답(400 + ApiErrorResponse.fieldErrors[] → form-error.js 가 data-error-field 요소에 주입) 이라 description 슬롯이 이미 동작했다. 그래서 템플릿 보강은 뷰 이름을 되돌려주는 SSR 재렌더 7종에만 필요했다. ISO 등록만은 예외적으로 create 이면서 SSR 재렌더(iso-new 뷰 반환)라 이 7종에 포함된다.
4. O-7 — "죽은 키" 발견 경위와 신 키 적용

전수 점검 보고서 O-7 은 개선 방향으로 구 키 server.error.include-stacktrace=never 가 운영 프로파일에서 보장되는지 확인하라고 적었다. 그대로 적용하지 않고, 그 키가 실제로 바인딩되는지부터 jar 레벨 메타데이터로 실측했다 — 결과는 다음과 같다.

발견 1 — 구 키는 Spring Boot 4 에서 바인딩되지 않는 죽은 키

server.error.include-stacktrace 는 Spring Boot 4.0 부터 구성 메타데이터에 deprecation level = error 로 표시된다(spring-boot-web-server 4.0.3 구성 메타데이터 실측: replacement = spring.web.error.include-stacktrace). level=error 는 단순 경고가 아니라 더 이상 바인딩되지 않음을 뜻한다 — 이 키로 넣으면 값이 반영되지 않아 O-7 이 고쳐지지 않는다.

발견 2 — devtools 가 기본값을 ALWAYS 로 끌어올리는 메커니즘

본 프로젝트는 developmentOnly 'spring-boot-devtools' 를 쓴다. 개발/샌드박스 실행에서 spring-boot-autoconfigure-4.0.3.jar!/META-INF/spring-devtools.properties 가 defaults.spring.web.error.include-stacktrace=always 를 선언하고, 이를 DevToolsSettings.getPropertyDefaults() → DevToolsPropertyDefaultsPostProcessor 가 최하순위 property source 로 주입해 기본값 never 를 always 로 상향한다. O-7 에서 관찰된 trace 노출은 운영 코드의 결함이 아니라 이 devtools 동봉 실행의 부수효과였다. 노출되는 것은 응답 바디에 한정되며 서버 로그의 스택은 이 설정과 무관하게 유지된다.

적용 — 신 키로 고정

application.properties 에 아래를 명시했다. application.properties 는 devtools 주입분(최하순위) 보다 항상 우선하므로, 어떤 실행 모드에서도 응답 바디의 스택이 차단된다. 기본값이 이미 never 인데도 명시하는 이유(devtools 상향)와 구 키 함정을 WHY 주석으로 함께 남겼다.

# ─────────────────────────── 오류 응답 노출 (HF4-2 / O-7) ───────────────────────────
# 500 JSON 오류 바디의 "trace"(전체 스택트레이스) 노출을 고정 차단한다. 기본값이 이미 never 인데도
# 명시하는 이유: spring-boot-devtools 동봉 실행에서는 spring-boot-autoconfigure jar 의
# META-INF/spring-devtools.properties 가 선언한 defaults.spring.web.error.include-stacktrace=always 를
# DevToolsSettings.getPropertyDefaults() → DevToolsPropertyDefaultsPostProcessor 가 property source 로
# 주입해 기본값을 ALWAYS 로 상향한다(2026-07-19 O-7 발견의 원인). application.properties 는 devtools
# 주입분(최하순위)보다 우선하므로 여기 명시하면 어떤 실행 모드에서도 응답 바디 스택이 차단된다.
# 서버 로그의 스택트레이스는 본 설정과 무관하게 유지된다(응답 바디 한정).
# 주의: 구 키 server.error.include-stacktrace 는 Boot 4.0 부터 deprecation level=error 로 바인딩되지
# 않는다(spring-boot-web-server 구성 메타데이터, replacement=spring.web.error.include-stacktrace).
spring.web.error.include-stacktrace=never
검수 판정 — O-7 은 지시 원문의 구 키가 죽은 키임을 메타데이터로 실측하고, devtools 상향 메커니즘까지 근거를 확보해 신 키로 적용했다. 검수는 이를 "지시보다 나은 구현" 으로 평가했다. include-message / include-binding-errors 두 키도 devtools 가 always 로 올리지만, 발견은 trace 에 한정되고 message 는 개발 진단에 유용하므로 이번 차단 대상에서 제외했다(§7 비목표).
5. 테스트 · 검증

단위 테스트 — DTO Bean Validation 경계값 (저장소 첫 도입)

jakarta Validator 를 직접 구동해 각 필드의 상한 경계를 고정하는 단위 테스트 2개 파일을 신설했다.

  • OsRequestSizeValidationTest — ① ISOCreateRequest.description 1,025자 → violation 1건 + "설명은 1024자 이하로 입력해주세요." ② 정확히 1,024자 → violation 0(경계) ③ ISOCreateRequest.isoPath 1,025자 → violation ④ OSMetadataCreateRequest.osVersion 65자 → violation + "OS 버전은 64자 이하로 입력해주세요." ⑤ IsoUploadIntentRequest.isoPath 1,025자 → 핸드셰이크 단계 violation.
  • BmcRequestSizeValidationTest — 255 하향의 행동 변경을 경계값으로 고정: ① BmcCreateRequest.targetDirectory 256자 → violation + "대상 디렉토리 경로는 255자 이하로 입력해주세요." ② 정확히 255자 → 통과(= legacy 컬럼 length) ③ BmcUploadIntentRequest.targetDirectory 256자 → violation.

사용자 액션 통합 테스트 — @WebMvcTest (기존 파일 관례에 추가)

  • ISO (OSMetadataControllerUploadFlowTest) — ① 등록 form POST /management/os/1/iso description 1,025자 → view().name("management/os/iso-new") 재렌더 + model().attributeHasFieldErrors("isoForm","description"), 등록 파이프라인(isoRegistrationService.prepare) 미기동 검증(F-6 사고 경로 차단) ② 정확히 1,024자 → 302 redirect(경계) ③ upload-intent isoPath 1,025자 → status().isBadRequest() + message 에 "1024자" 포함(업로드 바이트 전송 전 선차단).

  • SSR form POST (ISO 등록) — 1,025자 설명 제출 → HTTP 200 재렌더 + "설명은 1024자 이하로 입력해주세요." 필드 표시 확인.

7. 잔여 · 후속

이 슬라이스가 남긴 잔여(설계상 의도)

  • 교차 필드 합성 엣지 — intent 의 rawPath(≤1024) + filename(≤255) 합성 결과가 1024 를 넘는 경우는 필드 단위 @Size 로 잡을 수 없다(plan D5). 교차 필드 커스텀 검증기는 핫픽스 범위의 과설계라 도입하지 않았고, 기존 DB_CONSTRAINT 격리 안전망(F-6 사고에서 정상 동작이 실증됨)이 최후방을 유지한다.
  • BMC 상한 비대칭 — BMC targetDirectory 는 255, bios·subprogram 은 1024 로 비대칭이다. 이는 legacy file_path VARCHAR(255) NOT NULL 컬럼이 존재하는 한 불가피하며, 컬럼 제거 마이그레이션 시 1024 로 복원하면 해소된다. BMC 255 하향이 행동 변경(종전 256~1024자는 INSERT 실패였고 이제 앞단 400)이라는 점은 각 DTO 주석에 명기했다.

비목표(out of scope) — 이번에 손대지 않은 것

  • 세팅 정의서 Process payload 문자열(파티션·계정 등)의 길이 정책 — JSON blob 저장이라 컬럼 length 계약이 없고, 상한 정책은 도메인 결정이 필요한 별건.
  • devtools 가 always 로 올리는 다른 두 키(include-message / include-binding-errors) 고정 — 발견은 trace 뿐이고 message 는 개발 진단에 유용. 운영 프로파일 점검 시 별도 후보.
