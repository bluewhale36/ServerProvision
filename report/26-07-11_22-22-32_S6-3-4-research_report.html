조사 보고 — "변조를 정본으로 선언", 다른 시스템들은 어떻게 하나

S6-3-4(내용 변경 수용) 사전 조사 · 2026-07-11 KST · 7개 시스템·표준을 공식 문서로 검증 (근거 링크는 부록)

질문에 대한 직답 — 예, "감지된 변경을 관리자가 새 정본으로 승인"하는 로직은 업계에 실재하며, 파일 무결성 감시(FIM) 제품군의 핵심 기능입니다. 원조 격인 Tripwire 가 30년 가까이 이 워크플로(위반 항목별 체크박스 승인 + 키 암호 서명)를 제공해 왔고, PCI-DSS·NIST 같은 보안 표준은 오히려 "사람의 승인 없이 자동으로 정본을 갱신하지 말 것"을 요구합니다. 즉 S6-3-4 가 만들려는 기능은 특이한 발명이 아니라 업계 표준 관행이며, 관건은 "어떤 안전장치를 두느냐"입니다.

① 업계의 4가지 패턴

패턴대표 사례동작 방식
A. 명시 승인Tripwire(OSS/Enterprise), 백신·EDR 허용목록, PCI-DSS·NIST 요구감지된 변경을 사람이 항목별로 검토·승인해야 정본이 갱신된다. Tripwire OSS 는 리포트의 위반 항목마다 체크박스([x])를 두고, 승인 후 키 암호(passphrase) 입력으로 서명된 DB 에 반영. Enterprise 판은 승인(promote)을 변경관리 티켓(ServiceNow 등)과 대조해 자동화하기도 — "티켓이 있는 변경만 승인". EDR 은 탐지 파일을 해시 기준 허용목록에 등록(권한 있는 역할만 + 감사 로그)
B. 자동 재기준선OSSEC/Wazuh (syscheck)승인 게이트가 없다 — 변경 알림은 정보성 통지일 뿐이고, 저장된 기준 체크섬은 스캔 주기를 따라 자동 갱신. 같은 파일이 3회 연속 바뀌면 알림 자체를 억제(피로 방지). "알림이 곧 통제"라는 가장 느슨한 형태
C. 불변·재발급Docker/OCI(digest), Git(commit hash)내용의 해시가 곧 신원이라 "같은 신원으로 재승인"이 설계상 불가능. 내용이 바뀌면 반드시 새 신원(새 digest)이고, 대응은 항상 "새 버전 발행 + 참조 이동". 재승인 문제를 원천적으로 소거한 모델
D. 재서명 권한Linux IMA/EVM (커널 무결성)변경된 파일의 재유효화는 암호학적 서명 키 보유자만 가능(evmctl 재서명)하고, 그 개인키는 의도적으로 운영 시스템 밖(빌드 서버·HSM)에 보관. 콘솔의 승인 버튼이 아니라 키 소유가 곧 권한
표준·규제의 입장 — PCI-DSS 11.5(변경 탐지)는 FIM 알림을 변경관리 절차와 대조해 "승인된/미승인" 변경을 가리도록 요구하고, NIST SP 800-53 SI-7 은 기준선 갱신이 정식 변경관리를 통해 명시적으로 승인된 경우에만 이뤄져야 한다고 본다. 두 표준 모두 자동 재기준선(패턴 B)을 보안 중요 시스템에 허용하지 않는 방향 — 사람 승인 + 감사 추적이 규범.

② S6-3-4 설계와의 대조 — 우리는 어디에 서 있나

S6-3-4 설계 요소업계 대응물판정
MANUAL 전용 — 무인 자동 수용 금지PCI/NIST 의 "자동 정본화 금지" 규범, Tripwire 의 승인 게이트정합
자원명 입력 확인(typed-name)Tripwire 의 키 암호 입력 — "승인 의식(ceremony)을 무겁게"라는 같은 취지. 우리는 키 대신 자원명 재입력으로 실수 방지 축에 무게정합
[AUDIT] 감사 로그표준 요구(감사 추적) 및 EDR 허용목록의 감사 로그정합
카드(드리프트) 단위 승인Tripwire 의 항목별 체크박스 — 전체 재기준선(AIDE 식)이 아닌 개별 승인이 더 엄격한 쪽정합
마커의 HMAC 서명(서버 secret)Tripwire 의 서명된 DB / IMA 의 파일 서명과 같은 축. 단 저들은 키를 운영 시스템과 분리(site key 별도 보관, IMA 개인키는 시스템 밖) — 우리는 서버가 secret 을 보유하므로 "서버 자체가 뚫린 경우"까지는 못 막는다부분 정합 — 한계 명시

③ 조사가 S6-3-4 설계에 주는 시사점 (세부 plan 반영 제안)

1. 확인 창에 "무엇이 어떻게 바뀌었나"를 보여줄 것 — Tripwire 검토 관행

Tripwire 승인은 위반 리포트(무엇이 언제 어떻게 달라졌나)를 보면서 체크한다. 우리 확인 창도 자원명 입력만 요구하지 말고 등록 시점 지문 vs 현재 지문, 파일 크기·수정 시각의 변화를 함께 표시해 "보고 승인"이 되게 한다. (현재 설계는 빈 확인 창 + 이름 입력만 — 보강 권장)

2. "재업로드" 경로를 확인 창에서 1차 권장으로 안내 — 패턴 C 의 활용

ServerProvision 에는 이미 불변·재발급형 경로(같은 파일을 정식 재업로드 → 새 지문으로 재발급)가 있다. 콘텐츠 주소 시스템들이 보여주듯 재발급이 재승인보다 항상 안전하다(업로드 절차의 검증을 다시 통과하므로). 수용 확인 창에 "정식 재업로드가 더 안전합니다 — 그래도 현재 파일을 그대로 정본화하려면 자원명을 입력하세요"를 명시해, 수용을 2차 수단으로 자리매김한다.

3. 자동 재기준선(패턴 B)은 기각 근거가 더 단단해짐

OSSEC 식 "알림만 하고 자동 수용"은 표준(PCI/NIST) 관점에서 보안 중요 자원에 부적합 — 수용을 AUTO 등급으로 승격하자는 미래 제안이 나오면 이 조사가 기각 근거.

4. 장기 후보(이번 범위 밖) — 서명 키 분리·티켓 대조

IMA/Tripwire 의 키 분리(승인 권한을 서버 밖 키 보유자로 한정), Enterprise 판의 변경관리 티켓 대조(promote by change request)는 인증·조직 프로세스가 갖춰진 뒤의 진화 방향으로 기록만 해 둔다 — 지금 도입하면 over-engineering.

결론 — S6-3-4 의 방향(명시 승인 + 무거운 확인 + 감사 추적)은 업계 표준 패턴 A 와 정확히 일치하며, 규제 관행도 같은 것을 요구한다. 조사로 얻은 보강 2건(확인 창에 변경 내역 표시 / 재업로드 1차 권장 안내)을 세부 plan 에 반영해 구현 착수 시 포함할 것을 제안.
