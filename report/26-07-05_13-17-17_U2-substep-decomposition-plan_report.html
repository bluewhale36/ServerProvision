결론 — 3 세부단계 + 이연 1

U2 는 3 개 세부단계(U2-1 → U2-2 → U2-3)로 순차 진행하고, *Builder/*Resolver 계층(16+ 클래스)은 U2 밖으로 이연한다(§11 포인터만 남김). 자르는 축은 두 개다: ① 사용자 지시인 top-down 레이어 축(화면·계약 먼저 → 영속 나중)이 1차 경계를, ② 프로젝트 불가침 원칙 "갈라지는 시점에 분리"가 2차 경계(flat 단계 vs OS-다형 단계)를 만든다.

U2-1
화면 + 요청/응답 계약 이식
레거시 HTML 4종+fragment, *Request 12종, Controller 스텁(더미 데이터). 엔티티·영속 0.
U2-2
영속 + 조회 + flat 단계 쓰기
ServerSetting 엔티티(JSON 컬럼)·Repository·목록/상세 실동작 + BasicUpdate/BasicSetting 생성·수정.
U2-3
OS-다형 단계 + 검증
OSInstallation/OSSetting + rhel/debian 분기 + OS 전용 검증. 가장 갈라지는 지점을 마지막에.
이연
execution 도메인 resolver/builder
U2 아님. 소비자(실행 엔진)가 생길 때 가공 편입 — U3/Stage 4 국면.
이 보고의 위치

본 보고는 분할 계획까지다. 사용자 확인 후 지시에 따라 U2-1 부터 CP1 plan html (설계 대안 비교 + 인터랙티브 미리보기) 작성으로 진입한다. Notion 페이지 신설/개편도 지시 이후에 진행한다.

01

조사 방법과 근거

"이전까지 어떤 형태로 단계·세부단계를 나눴는가" 를 workflow(5 agent 병렬) 로 조사했다. 네 갈래 증거를 독립 수집한 뒤 종합(synthesis) 하고, Notion DB 는 별도로 직접 질의해 교차 확인했다.

증거원범위핵심 발견
git커밋 100건 (인벤토리 코드 표기 관행)코드는 계층형 하이픈 연결(세그먼트 수 = 중첩 깊이, 최대 4단계 S5-2-3-1). 순수 기계적 복제 캠페인은 여러 코드가 한 커밋으로 붕괴(fb094f8 = R4-5/R5-5/R6-4).
plan/120개 plan html 파일명·내용묶음 plan 은 코드 연결 표기(R4-3_R5-3_R6-3). 코드들이 구현 전 일괄 발행되기도 함(초 단위 연속 timestamp). §8 잔존 비대칭 + §11 후속 지정이 경계 유지 장치.
report/52개 report html세부단계 목록은 동결이 아니다 — 재검증 report 가 이연 자식을 재심사해 DROP/병합(R2 6→2). "소비자 없으면 추출하지 않는다"(R2-4 ProblemDetail DROP) 선례.
CLAUDE.md·memory불가침 원칙 + 이력 메모"갈라지는 시점에 분리" / over-abstraction = 중복과 동일 비용 / MA1-async 보류·R3-6 [삭제] 등 이연·삭제도 1급 결정.
Notion DB상위·하위 항목 self-relation 직접 질의세부단계 파생 = 부모-자식 relation. R2 는 리팩터 국면 축으로 6분할, S5 는 관심사 축으로 12분할(4단계 중첩). R2-1 = "사전조사" 가 첫 세부단계인 선례 — 이번 레거시 조사 report 와 정확히 같은 위상.
발견된 충돌 1건

Notion 에 과거 발행된 U2 스캐폴드(U2-1 데이터 모델링 / U2-2 CRUD 화면)는 모델-우선(bottom-up) 이라, 이번 top-down 지시(화면·계약 우선)와 순서가 반대다. 본 분할안 확정 시 Notion 쪽 개편이 필요하다 — §06 열린 질문 참고.

02

프로젝트의 분할 문법

조사 결과, 이 프로젝트에서 큰 단계가 세부단계를 파생시키는 축은 다섯 가지로 정리된다. 축은 서로 직교하며, 작업 유형이 어느 축을 쓸지 결정해 왔다.

#축언제 쓰였나대표 선례
①관심사/책임단일 도메인 내부의 비대 클래스·기능 분해R4-3 fat *Service 5분할 · R1-1/2/3 CRUD 분할
②엔티티/도메인검증된 레시피의 6 도메인 캠페인 복제R3(Board)/R4(BIOS)/R5(BMC)/R6(Subprogram)
③아키텍처 레이어대형 도메인 리팩터 — controller 먼저, service 나중R3-2 → R3-3
④lifecycle-액션/케이스 매트릭스남은 액션×엔티티 셀을 더 깊은 자식 코드로 마감S5-2-3 → S5-2-3-1
⑤데이터-형태-지금 vs 런타임-행동-나중신규 기능 aggregate — 형태(엔티티+CRUD)는 지금, 그것을 구동할 엔진은 명명된 후속으로U1 (모델 지금, 엔진 → Stage 4)

경계를 지키는 규칙 (전 선례 공통)

  • 갈라지는 시점에 분리 — 로직이 실제로 갈라지는 지점(배포판별 rhel/debian, 액션별 cascade)만이 세부단계 경계가 된다. 선제 분할 금지(MA1-async 보류, speculative @Lazy 제거).
  • reference-first / hardest-last — 병렬성 있는 자식들 중 가장 깨끗한 것을 먼저 완성해 템플릿을 굳히고, 가장 갈라지는 것을 마지막에(R4-3 D5: BIOS 먼저, Subprogram 마지막).
  • 빚을 남기는 경계는 §8 명명 + §11 후속 코드 지정 — 분할이 임의 파편화가 아니라 앞으로 자기-연쇄하도록.
  • 묶음(bundle) — 형제 세부단계들을 부모 코드로 묶고 공유 CP5 에서 함께 닫는다(S5-2). 공유 파일이 있으면 병렬 금지·직렬화.
  • 소비자 없는 추출 금지 — 실제 중복/소비자가 생겨야 추출(IsoMarkerWriter 추출 거절, R2-4 DROP). 코드 번호는 식별자일 뿐 실행 순서가 아니며, 세부단계 목록은 재검증으로 축소·삭제될 수 있다(R2 6→2).
03

U2 의 좌표와 제약

U2 = 정의서 CRUD 전용
Notion 로드맵상 서버↔정의서 연결(척추)·적용·시각화는 U3. U1 이 예약(D5)한 FK 는 U2 에서 건드리지 않는다.
신규 기능 aggregate
리팩터가 아니므로 10단계 전부를 밟는다(엔티티는 Step 7 에서 생성). U1 과 같은 U-계열 신규 슬라이스.
단일 슬라이스론 과중
다형 단계 4종 + 배포판 fragment 4종 + *Request 12종 + 3중 검증 — 한 CP4 테스트 매트릭스로 감당 불가.

사용자 지시가 그은 1차 경계

① top-down: 화면(HTML) + 요청 객체(*Request) 이식이 U2 의 시작이다. ② *Builder/*Resolver 는 이연: 추후 execution 도메인에 가공해 넣는다. 이는 조사에서 확인한 축 ⑤(형태-지금 vs 행동-나중)와 정확히 같은 결이다 — U1 이 "게스트 서버 모델은 지금, 프로비저닝 엔진은 Stage 4" 로 자른 것과 동형. 다만 지시는 여기서 한 발 더 나아가 형태 내부에서도 레이어 순서(계약 → 영속)를 지정했으므로, 이를 1차 축으로 삼는다.

중복 금지 제약 — 단계 모델

레거시 SettingProcessStep(BASIC_UPDATE 1 → OS_SETTING 5)의 정제판인 execution.enums.ProvisioningPhase / ProvisioningPhaseStep 이 현 브랜치에 이미 존재한다. U2 는 레거시 enum 을 부활시키지 않고 이 모델과 정합(재사용 또는 파생)해야 한다 — 어느 세부단계에서 정합하는지가 분할안에 명시돼 있어야 한다(U2-2 에 배치, §04).

04

권고 분할안 상세

U2-1 · 화면 + 요청/응답 계약 이식 — top-down 골격

항목내용
범위레거시 Thymeleaf 이식(list/new/detail/edit + rhel-based/debian-based + detail-rhel/detail-debian fragment) · *Request 전량 이식(AbstractProcessRequest 계층 + BasicUpdate/BasicSetting/OSInstallation/RHEL/Ubuntu/OSSetting/RHELOSSetting + Partition/User/RootPassword/Timezone 하위 요청) · *Response · Controller 골격(스텁 엔드포인트, 더미 데이터)
제외엔티티·Repository·영속 일절 없음(Step 6/7/9 N/A 에 준하는 형태) · Builder/Resolver 없음
의의사용자 지시의 직접 이행. 계약(무엇을 입력받아 무엇을 보여주나)이 먼저 굳어, 이후 영속 설계가 계약을 따라가게 한다. DESIGN.md·기존 CSS 정합화도 이 단계에서 처리.

U2-2 · ServerSetting 영속 + 조회 + flat 단계 생성·수정

항목내용
범위ServerSetting 엔티티(process JSON 단일 컬럼) + 직렬화 Converter + SettingStatus · 단계 모델을 ProvisioningPhase/ProvisioningPhaseStep 과 정합(레거시 enum 부활 금지) · Repository · 목록/상세 실동작 전환 · flat 단계 2종(BasicUpdate/BasicSetting) 생성·수정 — 배포판 분기 없는 reference 경로 먼저
의의reference-first: 다형 분기가 없는 가장 깨끗한 쓰기 경로로 영속·검증(1·2차)·테스트 템플릿을 굳힌다.

U2-3 · OS-다형 단계 생성·수정 + 검증

항목내용
범위OSInstallation/OSSetting 단계 쓰기 경로 · rhel/debian fragment 실동작(폼 동적 전환) · OS 전용 3차 검증(파티션·계정 등) · @JsonTypeInfo 다형 역/직렬화 확정
의의hardest-last: 배포판별 로직이 실제로 갈라지는 지점이라 별도 세부단계 자격이 있다. U2-2 에서 굳은 템플릿 위에 분기만 얹는다. OSSetting 이 OSInstallation 과 충분히 갈라지면 그때 U2-3-1 로 추가 분할(선제 분할 금지 — S5-2-3-1 방식의 escape hatch).

이연 · execution 도메인 resolver/builder 편입 (U2 아님)

2-tier dispatch(SettingProcessResolver → OS별 *Builder, 16+ 클래스)는 U2 어디에도 넣지 않는다. 정의서를 실행 명령으로 해석하는 책임은 실행 엔진의 소비 시점에만 의미가 있고, 지금 옮기면 소비자 없는 죽은 산출물이 된다(R2-4 DROP·IsoMarkerWriter 추출 거절과 동일 논리). 각 U2 plan 의 §8 에 의도된 잔존으로 명명하고 §11 에 후속(execution 편입 슬라이스, U3/Stage 4 국면)을 지정한다.

05

비채택 대안과 탈락 사유

대안 A · 엔티티-우선 (기존 Notion 스캐폴드 + workflow 종합의 원안)
U2-1 을 "ServerSetting aggregate + 읽기 화면" 으로 시작하는 안. U1 선례와는 정합하지만, 사용자의 top-down 지시(화면·계약 먼저)와 정면 충돌한다. 계약이 굳기 전에 영속 형태를 정하면 Request 형태가 엔티티를 따라가는 역방향 설계가 된다. → 기각, 엔티티는 U2-2 로 이동.
대안 B · 단일 슬라이스 (U1 방식 그대로)
U2 전체를 한 슬라이스로. U1 보다 훨씬 무겁다 — 다형 4종·fragment 4종·*Request 12종·3중 검증이 한 CP4 테스트 매트릭스에 몰린다. "CP4 에서 보고 불가능하면 너무 크다" 휴리스틱에 걸림. → 기각.
대안 C · Request 단위 12분할
*Request 클래스별 세부단계. 페이지/기능 그레인보다 잘게 쪼개져 "의미를 잃는 분할 금지" 위반. 형제들이 전부 같은 파일 집합을 공유해 직렬화 비용만 커진다. → 기각.
채택 · 레이어(top-down) 1차 + 갈라지는 지점 2차
1차 경계는 사용자 지시(계약 → 영속), 2차 경계는 flat vs OS-다형(실제 분기 지점). 축 ③(controller-then-service)·⑤(형태 지금/행동 나중)·reference-first/hardest-last 선례와 모두 정합.
06

진행 방식 · 열린 질문

진행 방식 (선례 준용)

  • 묶음 패키징: U2 를 부모로 U2-1/2/3 을 묶고(S5-2 방식), 각 세부단계는 자체 10단계·CP1~CP4 를 돌되 최종 브라우저 E2E 는 공유 CP5 로 함께 닫는 안을 권고.
  • 직렬 진행: 세 세부단계가 ServerSetting 파일 집합을 공유하므로 병렬 불가 — U2-1 → U2-2 → U2-3 순.
  • plan 발행: 사용자 지시 후 U2-1 CP1 plan 부터 착수. 3 개 plan 을 일괄 발행할지(R3~R6 코드 일괄 발행 선례) U2-1 완료 후 순차 발행할지는 지시에 따름.

열린 질문 — plan 방향 지시에 필요한 결정
