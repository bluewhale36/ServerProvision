BIOS 세팅을 정의서로 —
템플릿화 설계와 3분할

기존 무상태 BIOS 셋업 PoC 위에 "이름 붙은 정의서" 를 얹고, guest server 세팅 정의서가 그 템플릿을 선택하게 한다. 3 lens 독립 설계 → 교차 적대 비판 → 종합의 workflow 로 검증했고, 사용자의 "중첩 JSON" 직관은 사실 검증을 거쳐 그 자리를 다시 정했다.
목적U2-2 설계 + 하위 단계 구성
방법인벤토리 조사 + 7-agent 설계 workflow
결론U2-2-1 → -2 → -3 (3 슬라이스)
작성일2026-07-05 (KST)
00

결론 — 확정 설계 4 기둥 + 3분할

flat diff JSON
값은 변경분만 flat 1-depth 저장. 중첩의 자리는 DB 가 아니라 렌더 계층(§03) — Redfish PATCH Attributes 와 동형이라 execution 이 무변환 소비.
REFERENCE 참조
guest 의 BASIC_SETTING 은 biosSettingTemplateId 만 운반(살아있는 참조) — "계약은 의도만, 해석은 execution" 형제 철학과 동형. 3 설계 만장일치.
lifecycle 비적용
파일 없는 순수 데이터 자원 — marker/휴지통/soft-delete 는 지킬 대상이 없어 미적용. 사용중 삭제 차단(409) + confirm-modal hard delete 로 충분.
boardKey 유지
BoardModel FK 병합은 이연(사상 부재 — 가짜 검증 금지). 템플릿-보드 정합의 최종 판정은 execution 단일 지점. AUTO+템플릿은 모순이 아니라 지연된 단언.
U2-2-1
저장 모델 + 생성·목록 + PoC 이관·폐지
BiosSettingTemplate 엔티티 + 편집기 이관(move) + 구 bios-setup 라우트 즉시 삭제(복붙 공존 창 0).
U2-2-2
상세·수정·삭제 — 재조인 렌더
flat 저장값 → registry 재조인 페이지 그룹 렌더 + edit pre-fill overlay. 최난도 UI 집중.
U2-2-3
guest 연동 + 사용중 삭제 차단
BasicSettingRequest 실체화(@NotNull templateId) + 최대 1개 규칙 + SPI 기반 in-use 가드.
채택 골격

기반 = 단순성 렌즈 설계 (프로젝트 본 동기와 "파일 없는 자원" 판정, AUTO+템플릿 허용, 병합 이연이 가장 정확). 여기에 도메인 정합 렌즈에서 BiosAttributeType.templatable() 상수별 다형(PASSWORD 배제의 단일 SSOT)과 BiosSettingValues VO 를, 진화 내성 렌즈에서 "복제 승격이 아닌 이관(move) + 동일 슬라이스 라우트 폐지" 를 이식했고, 교차 비판의 지적 7건(VO 경계 관철·in-use 가드 도입 시점·SPI 단방향 의존·BASIC_SETTING 최대 1개 등)을 수용했다.

02

기존 자산 진단 — 무엇 위에 짓는가

기존 /provisioning/bios-setup 은 완전 무상태 PoC 다: 보드별 리소스 파일을 읽어 화면을 만들고, 저장하면 검증 후 Redfish 적용 계획을 로그로만 출력한다. 영속도, 이름 붙은 정의서도, 목록도 없다.

자산사실
데이터 원천보드별 파일 2+1종 — registry JSON(속성 메타: 4타입 enum ENUMERATION/INTEGER/PASSWORD/BOOLEAN 이 validate()/coerce() 다형 보유) + SetupData XML(페이지 재귀 트리) + 선택적 initial settings(Redfish GET 덤프).
화면 계층menuBar → BiosPage(parentId 트리, 깊이 상한 없음) → BiosControl sealed(leaf 속성 | submenu 링크) + Dependencies(HIDDEN/GRAYOUT 클라 라이브 평가). 계층의 SSOT 는 SetupData XML.
저장값 형태기본값 대비 변경분(diff)만 flat Map<AttributeName, String> → 서버 coerce 후 {이름: 타입보존값} flat. 속성명은 보드/칩셋 세대별 byte-exact(예: BirchStream0064_…).
Redfish 표면PATCH /redfish/v1/Systems/Self/Bios/SD {"Attributes":{flat}} · 비밀번호는 별도 Bios.ChangePassword 액션 · 적용은 ComputerSystem.Reset(ForceRestart). URL 은 코드 상수(BiosRedfishPayloadAssembler). 실호출 없음.
보드 스코프boardKey 문자열(application.properties boards[] 4종). management 의 BoardModel(DB)과 미병합 — 코드 주석에 "향후 병합, 해석은 BiosSetupLoader.load(boardKey) 단일 seam 유지" TODO 기존재.
간극 5정의서 명명 / 영속 / 목록·조회 / 보드 귀속 영속 / guest seam(BasicSettingRequest 필드 0 스텁) — 전부 신규.
03

핵심 판정 — "중첩 JSON 필요" 직관의 검증

지시에 담긴 직관("BIOS 세팅은 내부적으로 JSON 이 중첩되어 들어가야 할 필요가 있어 보임")을 사실로 검증했다. 결론: 저장 모델은 flat 이 옳다 — 다만 직관의 정당한 핵 2가지는 설계에 수용된다.

flat 이 옳은 인과 (4단계)

  1. 최종 적용 표면이 flat 이다 — Redfish PATCH …/Bios/SD 의 Attributes 페이로드는 flat 객체다. 중첩으로 저장하면 적용 직전 flatten 변환이 영구적으로 필요해진다.
  2. 화면의 중첩은 렌더 모델이다 — 사용자가 본 계층(메뉴→페이지 트리→위젯)의 SSOT 는 SetupData XML 파일이다. 저장 JSON 에 계층을 복제하면 SSOT 이중화가 되고, BIOS 버전업으로 페이지 배치가 바뀌는 순간 저장본과 파일이 drift 한다.
  3. 계층은 언제든 복원 가능하다 — 조사로 확정된 사실대로, flat 저장값을 registry/SetupData 와 재조인하면 페이지 경로별 그룹 렌더가 복원된다(기존 PoC 의 저장 계약도 이미 flat diff 다).
  4. 속성 1건에는 중첩할 하위 구조가 없다 — 유일한 구조는 값 타입 혼재(문자열/정수/불리언)인데, 그것은 중첩이 아니라 JSON 스칼라 타입 보존의 문제다.
직관의 정당한 핵 — 수용 2건

(a) "JSON 으로 들어가야 한다" 는 맞다 — 속성 스키마가 보드/세대별로 달라 행 분해(EAV)는 무의미하므로 JSON 문서 컬럼을 채택한다(guest 정의서의 process JSON 방향과도 결이 같다). (b) 사람이 읽는 화면은 반드시 중첩이어야 한다 — 상세/편집 화면은 재조인으로 페이지 경로별 그룹(중첩) 렌더를 한다. 즉 중첩의 자리는 DB 가 아니라 표시 계층이다.

-- bios_setting_template.values_json (flat 1-depth · diff 만 · coerce 후 타입 보존)
{
  "BirchStream0064_PowerPerformanceTuning": "OS Controls EPB",
  "IIO0012_LinkTrainingTimeout": 1000,
  "SETUP012_QuietBoot": true
}
-- 이 형태 그대로 Redfish PATCH {"Attributes": …} 에 투입 가능 (execution 무변환 소비)
04

확정 설계 상세

데이터 모델

항목결정
패키지신규 feature provisioning.biossetting (controller/service/repository/entity/vo/dto/exception). 카탈로그 엔진(domain/parser/loader)은 현 위치 유지 — 소비자가 biossetting 과 Stage 4 execution 둘이 될 예정이라 특정 feature 로의 이동은 오귀속.
엔티티BiosSettingTemplate extends BaseTimeEntity — LifecycleEntity·Markable 비상속. 필드: id / name(전역 unique, 중복 409) / description / boardKey(@Embedded BoardKey record VO, 생성 후 불변 — PUT 계약 미포함) / values(BiosSettingValues VO = Map<BiosAttributeName, BiosAttributeValue> 래핑, 최소 1속성 invariant, @Convert JSON 컬럼). 기존 provisioning.domain.vo 의 VO 2종 재사용.
검증저장 시 BiosSetupLoader.load() 로 registry 를 얻어 기존 validate()/coerce() 다형 루프를 신규 CommandService 로 이동 재사용(별도 공용 컴포넌트 추출 안 함 — PoC save 가 같은 슬라이스에서 폐지돼 소비자 복수 창이 없음). unknown 속성은 서버 항상 400(엄격).
PASSWORD 정책템플릿에서 배제 — 근거: 평문 DB 저장 회피 + Redfish 도 Attributes 가 아닌 별도 액션이라 flat values 에 실을 수 없음. 단일 SSOT 는 BiosAttributeType.templatable() 상수별 다형(PASSWORD 만 false): 편집기 뷰모델이 이 판정으로 위젯을 아예 미출력(구조적 UI 차단), 서버는 같은 판정으로 400(안전망). → §07 Q1 로 사용자 확인.
서비스Query/Command 2분할(U2-1 선례). lifecycle 없으므로 R4~R6 식 5분할은 over-abstraction. 신규 예외 3(NotFound 404 / DuplicateName 409 / InUse 409) + 기존 4 재사용.

템플릿 참조 — REFERENCE (3 설계 만장일치)

guest 의 BASIC_SETTING 은 biosSettingTemplateId 만 운반한다. 수정은 자유(사용 중 정의서 N건에 실행 시점 반영됨을 편집 화면에 고지), 삭제는 사용중 차단 — 판정 SSOT 는 biossetting 이 소유한 1-메서드 SPI BiosSettingTemplateUsageChecker.isInUse(id) 를 setting 이 구현 (패키지 의존을 setting→biossetting 단방향으로 유지, MarkableScanner SPI 선례 동형). 목록/편집의 삭제 버튼 disabled+tooltip 과 DELETE 409 가 같은 메서드를 호출한다. SNAPSHOT 은 SSOT drift·정의서 비대·개선 미반영으로 기각, HYBRID(실행 시점 고정)는 별도 메커니즘이 아니라 execution 이 킥오프 시 resolved 값을 실행 레코드에 기록하면 자연 충족되므로 지금 설계할 것이 없다.

보드 스코프와 자동 감지 정합

채택 — execution 단일 판정
"템플릿-보드 정합의 최종 판정 지점은 execution 단 한 곳: 실행 시점에 확정된 보드의 boardKey ≠ 템플릿의 boardKey 면 해당 단계 적용 거부." 보드 AUTO + 템플릿 선택은 모순이 아니라 지연된 단언 — 사용자의 명시 시나리오(자동 감지→최신 펌웨어→BIOS 세팅)가 정확히 이 조합이다. U2-2 의 UI 책임은 차단이 아닌 인지 보조: 템플릿 select 에 boardKey 뱃지 + "실행 시점 보드와 다르면 적용 거부" 정적 안내문.
기각 — 설계 시점 차단 시도 2종
① "AUTO ⇒ 템플릿 금지" @AssertTrue: BASIC_UPDATE 의 'AUTO⇒LATEST' 와 달리 보드-무관 선택지(LATEST 템플릿)가 성립 불가(속성명 byte-exact)라 금지하면 본 기능이 죽는다. ② SPECIFIED 보드와의 비교 가드: boardModelId(DB)↔boardKey(파일) 사상이 부재 + 폼 보드 선택지가 아직 더미 — 존재하지 않는 매핑 위에 가짜 검증을 만들지 않는다. 병합 슬라이스(Stage 4 준비)에서 실사상과 함께 도입.

추가로 설계 시점에 확정 가능한 모순 1건은 지금 막는다: 정의서당 BASIC_SETTING 최대 1개(@AssertTrue, 요청-로컬) — 복수 템플릿 간 boardKey 불일치와 속성 충돌 규칙(later-wins?)이라는 두 구멍을 추가 개념 0개로 동시에 닫는다. 완화(허용)는 추후 데이터 비파괴로 가능하지만 그 역은 저장 데이터를 깬다.

05

하위 단계 U2-2-1 / -2 / -3
