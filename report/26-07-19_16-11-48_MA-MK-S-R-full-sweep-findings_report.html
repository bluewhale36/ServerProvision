2. 결함 상세 (F-1 ~ F-7)
F-1 · 높음 휴지통 "+30일 연장" — 메타 자원에서 비라우팅 500 + UI 버튼 활성 노출

증상

휴지통에 OS 버전(OS_IMAGE) 행이 있을 때 "+30일 연장" 버튼이 활성 상태로 노출된다. 확정하면 서버가 500 을 반환하고 TTL 은 연장되지 않는다. 사용자에게는 F-3 의 깨진 오류 모달이 보인다.

재현

  • OS 버전 하나를 삭제해 휴지통으로 보낸다 → 휴지통에서 해당 행의 "+30일 연장" 클릭 → (오표기된 F-2 모달) 확정.
  • curl 재현: POST /maintenance/trash/OS_IMAGE/{id}/extend → HTTP 500.

근본 원인

global/marker/MarkableTrashOperator.extendTrashTtl 은 default 메서드가 UnsupportedOperationException("<type> 는 trash TTL 연장을 지원하지 않습니다") 를 던지고, 이 메서드를 재정의한 구현체는 파일 자원 4종(IsoMarkableScanner · BoardBiosMarkableScanner · BoardBmcMarkableScanner · SubprogramMarkableScanner)뿐이다. 메타 자원(OS_IMAGE, BOARD_MODEL) 스캐너에는 구현이 없어 default 가 발동하는데, 이 예외는 advice 어디에도 매핑되어 있지 않아 [http.request.unhandled] 로 빠져 500 이 된다. 파일 자원(OS_ISO 등)의 연장은 302 로 정상 동작함을 양성 대조로 확인했다.

왜 문제인가

프로젝트 불가침 원칙 두 가지에 동시에 걸린다. ① "정상 UX 흐름의 논리 모순은 UI 에서 1차 차단(버튼 disabled + tooltip)" — 연장 미지원 자원이면 버튼이 비활성이어야 한다. ② "예외는 advice 가 HTTP 응답으로 변환" — 지원 여부는 도메인 상태이므로 설령 direct POST 로 들어와도 4xx 로 거절되어야 한다.

권고

  • 1차: 휴지통 뷰모델에 자원 종류별 연장 지원 여부 플래그를 내려 메타 자원 행의 연장 버튼을 disabled + tooltip 처리 (SSOT 는 MarkableTrashOperator 지원 여부 판정 메서드 하나로 공유).
  • 2차: 설계 의도를 정한다 — 메타 자원도 연장을 지원해야 한다면 (cascade 로 자식 파일 자원 TTL 재기준) 구현 추가가 본질 해법. 지원하지 않을 것이면 UnsupportedOperationException 대신 도메인 예외를 던지고 advice 에서 409/400 으로 라우팅.
F-2 · 중간 연장 확인 모달 라벨 오배선 — 제목·버튼이 "Deprecated 표시"

"+30일 연장" 클릭 시 뜨는 confirm-action 모달의 제목이 "Deprecated 표시", 확정 버튼도 "Deprecated 표시"다. 본문은 "Ubuntu 22.04 의 보존기간 을(를) Deprecated 로 표시할까요? +30일 연장돼요."로 두 액션의 문구가 섞여 있다. S5-2 confirm-action 모달 자산 호출부에서 연장 액션의 라벨 파라미터에 다른 액션(Deprecated 표시)의 라벨이 배선된 것으로 보인다. 사용자는 자신이 어떤 액션을 확정하는지 오인할 수 있다 — 특히 F-1 과 겹치면 "Deprecated 를 눌렀더니 서버 오류"로 읽혀 원인 파악이 더 어려워진다.

권고

휴지통 연장 버튼의 모달 호출부 라벨 배선 수정(제목 "보존기간 연장", 버튼 "연장"). 같은 화면의 다른 액션(복원/영구삭제) 라벨은 정상이었으므로 연장 한 곳의 오배선이다.

F-3 · 중간 오류 모달에 HTML 에러 페이지 원문 덤프 — 오류 응답 협상 불일치

연장 확정(F-1) 실패 시 "서버 오류" 모달 본문에 <!DOCTYPE html> <html lang="ko"> <!-- 공통 에러 뷰. WebExceptionHandler 가 … 처럼 공통 에러 뷰의 HTML 소스(주석 포함)가 그대로 출력됐다. 화면 JS(fetch)는 실패 응답 바디를 JSON ApiErrorResponse 로 기대하는데, 서버의 미매핑 예외는 HTML 에러 뷰로 렌더되기 때문이다. IsoUploadController 의 주석("전역 핸들러의 HTML 에러 뷰 경로로 빠지면 클라이언트가 실제 사유 메시지를 파싱하지 못해…")이 이미 이 함정을 경계하고 있는데, 휴지통 연장 XHR 경로는 그 방어가 없다.

권고

  • 클라이언트 공통 fetch 헬퍼에서 응답 Content-Type 이 JSON 이 아니면 바디를 버리고 일반화된 메시지("서버 오류가 발생했습니다 (HTTP 500)")로 대체.
  • 서버측: XHR 전용 엔드포인트는 @ResponseBody + JSON 오류 계약으로 통일하거나, Accept 헤더 기반으로 advice 가 JSON variant 를 선택하도록 확장 (기존 variant=json 라우팅이 이미 있으므로 미매핑 예외의 fallback 만 보강하면 된다).
F-4 · 낮음~중간 ISO 등록: 디렉토리 경로 + 파일 첨부 direct POST → 500

POST /management/os/{id}/iso 에 isoPath 를 디렉토리로 주고 파일을 첨부하면 IsoRegistrationService.storeUploadedFile:433 의 Files.newOutputStream(대상이 디렉토리)이 실패해 ISOFileStorageException → 500 이 된다. 정상 화면 JS 는 "이 디렉토리에 저장" 선택 시 클라이언트가 디렉토리 + 파일명으로 완전한 경로를 조립해 보내므로 노출되지 않는다 — direct POST 한정. 다만 "경로가 디렉토리"는 클라이언트 입력 오류(4xx 계열)이지 INFRA 장애가 아니므로, 500(ISOFileStorageException) 매핑은 과잉이다. 서버 가드 안전망 관점에서 prepare() 단계에 Files.isDirectory(resolved) 검사 → 필드 오류(400) 한 줄이면 충분하다.

F-5 · 낮음 checkbox 마커 없는 direct POST — 바인딩 실패의 오류 무표시 silent 200

같은 엔드포인트에 Spring checkbox 관례 필드 _allowCreateDirectory 없이 POST 하면 record 생성자 바인딩(primitive boolean)이 실패해 bindingResult.hasErrors() 분기로 폼이 재렌더되는데, 화면 어디에도 오류가 표시되지 않고 입력값만 보존된 200 이다 (템플릿에 해당 필드의 오류 슬롯이 없음). 브라우저 폼은 hidden 마커를 항상 포함하므로 정상 UX 노출은 없지만, "silent 흡수"는 프로젝트가 가장 경계해 온 사고 패턴이므로 기록한다. 바인딩 오류 시 최소한 폼 상단 배너(글로벌 오류)로 표면화하는 방어를 권고.

F-6 · 낮음~중간 ISO 설명 길이 미검증 — DB 제약(1406)까지 전파

ISOCreateRequest.description 에 @Size(max=1024) 가 없어 1024자 초과 설명이 검증을 통과하고, 등록 background job 의 INSERT 에서 MariaDB 1406(Data too long) 으로 실패한다. 실패 자체는 orphan 격리 saga 가 DB_CONSTRAINT 로 정확히 분류·회수했으므로(파일 격리 → 복구/폐기 UI) 데이터 유실은 없다 — 오히려 이 실수 덕에 saga 의 실전 동작이 검증됐다. 그러나 사용자 입장에선 "업로드까지 다 끝난 뒤 실패 → 복구 페이지행"이라 비용이 크다. DTO @Size 1 줄로 앞단 400 차단을 권고. 같은 패턴(폼 textarea ↔ DB varchar 길이 계약)이 다른 자원의 설명 필드에도 있는지 일괄 점검 가치가 있다.

F-7 · 낮음 휴지통 리네임 — 디렉토리명의 마지막 점을 확장자로 취급

디렉토리 자원이 휴지통으로 이동할 때 충돌 방지 리네임이 파일 확장자 보존 규칙을 디렉토리에도 적용해 v1.10 → v1_260719-064732-312_26f378f3.10, v2.3 → v2_….3 처럼 버전 표기가 끊긴다. 복원은 원경로 기준이라 기능 영향은 없지만, 휴지통 디렉토리를 직접 열어보는 운영자는 혼란스럽다. 리네임 유틸에서 대상이 디렉토리면 확장자 분리 없이 전체 이름 뒤에 접미사를 붙이는 분기 권고.

  • Spring Devtools 재시작 3회(16:00 대) — 같은 프로젝트 디렉토리에서 외부 Gradle/IDE 빌드 활동으로 classpath 가 3연속 변경(38 삭제 → 51 추가 → 103 수정)되며 자동 재시작. 이후 SpringEL T(IntegrityStatusView) 평가 실패로 bios/bmc/subprogram 목록 3페이지가 템플릿 파싱 오류를 냈다. 신규 JVM 재기동 후 동일 페이지가 완전 렌더됨을 확인 — RestartClassLoader 아티팩트로 판정(운영엔 devtools 없음).

5. 커버리지와 미검증 잔여

검증됨 (요약)

  • CRUD·검증·중복: OS/보드 등록 폼 400 필드 오류·409 중복 배너(HTML/JSON 양쪽)·수정 read-only 정책.
  • 업로드 파이프라인: ISO(form POST + 사이드카 마커), BIOS Zip(전개+in-tree 마커+진입점 자동탐지), BMC 단일 파일, Subprogram 보드연결/공용 — intent 핸드셰이크 → 업로드 → background job → manifest 해시·HMAC 마커·DB 3자 일치 확인.
  • lifecycle: 토글/deprecate/soft-delete/복원 cascade(부모-자식 동반, 공용 자원 독립), UI 1차 차단(disabled+tooltip "부모 OS 버전이 비활성/삭제 상태입니다…") + 서버 가드 409 + forging 404.
  • 휴지통: TTL 표기, 복원 cascade 모달(radio), typed-name 영구삭제(오입력 시 버튼 disabled), 운영 설정 저장 왕복, 감사 로그(사용자 직접/DRIFT_TRASH_LOST 2건).
  • 드리프트 7종 재현: 경로 이동(+적용=DB 경로 갱신) · 자원 소실 2형태(+수동 복구 후 자연 해소) · 내용 변경(정밀 전용, typed-name 수용 → HASH_ACCEPT 잡 → 마커·DB 지문 동기) · 서명 불일치(+적용=재서명) · 휴지통 소실(복원 거절 모달 → TRASH_LOST 탐지 → 적용=기록 정리+감사) · 부팅 자동 스캔.
