§1. 검토 동기 · 결론 요약

R4-6(BIOS/BMC/Subprogram fat-service 5분할)에서 세 도메인이 각자 *MarkerWriter 를 갖게 됐다. file-bearing 도메인(물리 파일 + 마커를 쓰는 자원) 중 ISO 만 별도 writer 클래스가 없고 marker 로직이 IsoRegistrationService 안에 inline 으로 남는다. "다른 도메인 소스를 읽을 때 모호하다"는 우려에서, ISO 에도 IsoMarkerWriter 를 추출하는 방안의 타당성을 신규 단계에 준하는 수준으로 검토한다.

결론 요약 — 추출은 타당하나, BIOS/BMC/Subprogram 과는 동기가 다르다. 저들은 marker 4-step 이 3 경로에 복붙돼 있어 중복 제거가 1차 동기였다. 그러나 ISO 는 이미 단일 helper finalizeMarker 로 응집(2 경로가 공유)돼 있어 중복 제거 이득이 0 이다. 따라서 ISO 추출의 유일한 동기는 cross-domain 네이밍 일관성(탐색 모호 해소)뿐이며, 그 가치는 실재하나 modest 하다. 게다가 finalizeMarker 는 순수 4-step 이 아니라 ISO 고유 책임(SIDECAR · IO 실패 wrapping · 권한 강화)이 얽혀 있어 BIOS/BMC 처럼 깔끔한 thin-adapter 추출이 아니다. → 권고: Option C(순수 4-step 만 thin writer 로, ISO 고유 책임은 registration 에 잔류)로, R4-6 과 별개의 저우선 단독 슬라이스.
§2. ISO 현재 마커 처리 구조

ISO 의 marker 쓰기는 IsoRegistrationService.finalizeMarker(saved, prepared, manifestHash, target)(L358-387) 한 곳에 있다. 2 경로가 이 helper 를 공유한다 — finalize(L220), completePendingFromNudge(L273).

private void finalizeMarker(ISO saved, PreparedIsoRegistration prepared, String manifestHash, Path target) {
    MarkerContent unsigned = new MarkerContent(ResourceType.OS_ISO.name(), saved.getId(),
            Map.of("osMetadataId", ..., "originalFilename", ...), Instant.now(), manifestHash, null);   // (1) 조립
    String signature = markerService.computeSignature(unsigned);   // (2) 서명
    saved.reissueMarker(manifestHash, signature);                  // (3) 엔티티 재발급
    try {
        markerService.write(target, MarkerLayout.SIDECAR, unsigned.withSignature(signature));   // (4) SIDECAR 기록
    } catch (RuntimeException e) {
        throw new IsoMarkerWriteFailedException("ISO 마커(sidecar) 기록 실패. path=" + target, e);   // ★ ISO 고유: IO 실패 wrapping
    }
    fileSystemHardener.applyDefaultPermissionsForFile(                                              // ★ ISO 고유: 권한 강화(S3.1/S3.2)
            markerService.resolveMarkerFile(target, MarkerLayout.SIDECAR));
}

공용 엔진 ProvisionMarkerService(computeSignature/write/resolveMarkerFile)는 BIOS/BMC/Subprogram 과 동일하게 공유한다 — 마커의 진짜 공통 로직은 이미 1개 클래스로 묶여 있다.

§3. 핵심 구분 — ISO 는 복붙이 없다 (de-dup 이득 0)

BIOS/BMC/Subprogram 의 *MarkerWriter 추출이 불가침으로 정당했던 1차 이유는 marker 4-step 의 3중 복붙 제거였다(churn 경고와 묶임). ISO 는 그 전제가 성립하지 않는다.

함의 — ISO 의 finalizeMarker 는 이미 single-source 다(이름도 의도를 드러냄). 추출해도 줄어드는 중복이 없다. 따라서 ISO 추출은 "중복 제거"가 아니라 "4 도메인이 모두 *MarkerWriter 클래스를 가져 탐색이 균일"해지는 가치에만 기댄다. 이는 실재하는 유지보수성 이득이지만(개발자가 os/service/iso/ 에서 기대한 자리에 writer 를 찾음), 동작·구조의 결함을 고치는 것이 아니라 균일성을 높이는 성격이다.
§4. 얽힘 분석 — finalizeMarker 는 순수 4-step 이 아니다

BIOS/BMC 의 writer 는 ProvisionMarkerService 1개만 주입하는 얇은 어댑터다. ISO 의 finalizeMarker 는 4-step 외 ISO 고유 책임 3가지를 더 갖는다 — 그대로 통째 추출하면 writer 가 peer 보다 무거워진다.

권고

Option C 를, R4-6 과 분리된 저우선 단독 슬라이스로. 근거를 풀면: (1) ISO 추출의 유일한 동기는 일관성인데, 그 일관성은 "writer 가 peer 와 같은 모양"일 때 비로소 달성된다 — 통째 추출(A)은 모양을 어긋나게 해 동기를 자가당착시킨다. (2) C 는 IsoMarkerWriter 를 BiosMarkerWriter 와 같은 1-dep thin adapter 로 만들어 4 도메인 균일성을 진짜로 달성하고, ISO 고유의 IO-실패-격리/권한-강화는 그 책임이 속한 registration 계층에 남긴다. (3) 단, de-dup 결함이 없으므로 긴급도는 낮다 — R4-6 캠페인에 끼우지 말고(ISO 는 R1 완료 영역, scope 밖), 별도 소규모 슬라이스로 다루거나 우선순위가 낮으면 보류해도 무방하다. "하면 C 로, 안 해도 결함 아님" 이 정직한 판정이다.
§6. 위험 · 범위 · 후속

위험 / 범위

  • 범위: ISO 는 R1(완료) 영역. R4-6 캠페인(R4-3/R5-3/R6-3, BIOS/BMC/Subprogram) scope 밖 — 별도 슬라이스.
  • 위험: 낮음·기계적. C 는 순수 4-step 만 이동하고 호출처(finalize/completePendingFromNudge → finalizeMarker)는 그대로라 표면이 작다. 동작 불변(SIDECAR 기록·예외 wrapping·권한 강화 순서 보존).
  • 검증: IsoIntegrityService(마커 읽기·검증)와 무관(쓰기 경로만). 단위 테스트는 writer 의 서명/기록 + finalizeMarker 의 IO 실패 wrapping 분리 검증.
  • 순환: writer 는 ProvisionMarkerService 1 dep(leaf) → 순환 무관(R7 토대 보존).

후속 위치

  • 본 검토를 근거로 ISO writer 추출을 별도 슬라이스로 채택할지 사용자 결정
  • 채택 시 Option C 로 — os/service/iso/IsoMarkerWriter(1-dep, SIDECAR), finalizeMarker 는 writer 호출+wrapping+hardening 으로 잔류
  • R3456 캠페인 메모리의 후속 항목에서 본 보고서로 대체(메모리 노트 제거됨)
메모리 처리 — 본 검토는 메모리 노트 iso-markerwriter-followup 를 정식 보고서로 승격한 것이다. 사용자 지시에 따라 해당 메모리 노트는 본 보고서 작성과 함께 제거되었다(아이디어가 단발 메모 → 검토된 산출물로 graduate).
