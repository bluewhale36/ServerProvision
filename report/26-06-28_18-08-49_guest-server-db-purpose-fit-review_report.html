명세 충실도·PXE 계약 일치는 논외로 두고, "물리 서버를 자동으로 프로비저닝한다" 는 프로그램의 목적이 데이터에 요구하는 것을 먼저 도출한 뒤, 현재 스키마가 그것을 표현·수행할 수 있는지를 따진다.

00

판정 — 절반은 맞고 절반은 비어 있다

프로그램의 목적은 두 동작으로 쪼갤 수 있다. ① 식별(identify) — 부팅한 서버가 무엇인지 알아내 그 보드에 맞는 생산 자료(펌웨어·드라이버·OS)를 자동으로 골라낸다. ② 실행(execute) — 고른 자료로 5단계 공정(BIOS/BMC 업데이트 → BIOS 세팅 → BMC 세팅 → OS 설치 → OS 세팅)을 서버마다 끌고 가며 추적한다. 스키마를 이 두 축으로 보면 평가가 선명해진다.

식별 = 합격
서버→보드모델 FK 가 자료 자동 선택의 join 축으로 정확히 박혀 있다. 네트워크 신원(NIC)도 타입화돼 견고.
실행 = 미성립
"무엇으로 프로비저닝할지"(세팅 정의서·OS·BMC 접근)가 스키마에 없고, 진행/단계 테이블은 아무도 쓰지 않으며 구조 결함을 안고 있다.
설계 품질 = 보통
3중 1:1 과잉 정규화, 파생·중복 필드, 진행 커서 이중화, 낙관적 락 위치 역전 등 정합성 리스크가 누적.
한 줄 요약

현재 스키마는 "이 서버가 무엇이고 그 보드에 어떤 펌웨어·드라이버가 있나" 까지는 답할 수 있으나, "이 서버를 무엇으로 만들지, 그 BMC 에 어떻게 접속할지, 각 단계에서 무엇이 일어났는지" 는 아직 답하지 못한다. 선언된 골격 스코프 안에서는 자연스러운 상태지만, 자동 프로비저닝의 척추인 서버 ↔ 세팅 정의서 연결이 비어 있다는 점은 골격 단계에라도 의식적으로 예약·표식해 두어야 한다.

01

02

요구 ↔ 충족 매트릭스

목적에서 도출한 데이터 요구를, 현재 스키마가 충족하는지 1:1 로 대조한다.

목적이 요구하는 능력필요한 데이터상태근거 / 비고
부팅 서버 재식별guest_server.system_uuid (UK)충족재부팅 멱등 매칭 가능.
보드별 자료 자동 선택detail.board_model → BoardModel FK충족BoardModel 1:N {BIOS·BMC·Subprogram} 으로 join. 스키마의 가장 강한 목적 정합점.
호스트 네트워크로 접속host_nic_binding.mac/ip (VO) · is_primary충족1:N + primary. 타입 검증된 주소.

03

식별의 절반 — 목적에 잘 맞는다

"서버마다 하드웨어가 달라 필요한 생산 자료를 자동 식별한다" 는 Notion 의 명시 목적은, 결국 한 서버를 검증된 보드 카탈로그에 묶는 join 키가 있느냐로 환원된다. 이 핵심이 정확히 구현돼 있다. 

// 자동 선택의 join 축
guest_server ──1:1── guest_server_detail ──FK──▶ BoardModel
                                                  │ 1:N
                                  ┌───────────────┼───────────────┐
                              BoardBIOS        BoardBMC        Subprogram(driver)
// 부팅한 서버의 board_model 만 알면 그 보드용 펌웨어·드라이버 후보가 전부 질의된다.

게다가 보고서가 자유 문자열로 두었던 board_model 을 관리 카탈로그로의 FK 로 격상해, 등록 시 미등록 보드를 BoardModelNotFoundException 으로 거절한다. 즉 "카탈로그에 없는 보드는 프로비저닝 대상이 될 수 없다" 는 무결성이 데이터 층에서 강제된다 — 목적(일관된 자동 설치)에 정확히 부합하는 결정이다.

네트워크 신원도 목적에 맞다. 실제 서버는 LAN1/LAN2·본딩으로 다수 NIC 을 가지므로 host_nic_binding 의 1:N + is_primary 는 옳고, MAC/IP 를 VO 로 정규화·검증해 "표기 차이로 같은 NIC 을 다른 것으로 오인" 하는 사고를 차단한다(자동화가 잘못된 주소로 접속하는 것을 막음).

요지

식별 구간은 추가 작업 없이 목적을 지지한다. 이 부분의 설계 판단(보드 FK 격상, NIC 1:N, VO 타입화)은 계승할 가치가 있다.

04

실행의 절반 — 비어 있거나 결함이 있다

G1"무엇으로 프로비저닝할지" 가 어디에도 없다 — 가장 큰 목적 공백

자동 프로비저닝의 정의는 "정해진 절차(세팅 정의서)에 따라 서버를 만든다" 이다. 그런데 게스트 서버에서 세팅 정의서·OS 이미지·ISO 로 향하는 링크가 전무하다. 검색해도 execution 패키지에 OS/ISO/ SettingDefinition 참조가 0건이고, SettingDefinition 도메인 자체가 아직 구현돼 있지 않다.

지금 가능한 질의
이 서버의 보드는? 그 보드용 BIOS/BMC/드라이버 후보는?
목적상 반드시 가능해야 할 질의
이 서버는 어떤 OS · 어떤 세팅 정의서로 프로비저닝되어야 하는가?

구 설계의 Server 엔티티에는 settingDefinition FK(nullable) 가 있었는데, GuestServer 로 재모델링되며 그 연결이 사라졌다. 공정의 OS_INSTALLING 단계는 "어떤 ISO 를 설치할지" 를 알아야 실행되는데, 그 정보를 담을 자리가 스키마에 없다. 세팅 정의서 작성이 7단계로 이연된 점을 감안해도, 서버↔정의서 연결 지점만큼은 골격에서 예약(자리 + 카디널리티)해 두는 편이 옳다 — 이 연결이 전체 도메인의 척추이기 때문이다.

G2펌웨어/BMC 단계를 수행할 데이터가 없다

공정의 처음 세 단계(BIOS/BMC 업데이트·BIOS 세팅·BMC 세팅)는 BMC 에 Redfish/ipmitool 로 접속해야 한다 — 즉 BMC IP + 자격증명이 필수다. 그러나 게스트 서버 스키마에는 BMC 주소도, 자격증명 참조도 없다 (bmc_binding·credential_ref 는 스코프 밖 이연). 한편 ProvisioningPhase enum 에는 FIRMWARE_UPDATING · FIRMWARE_SETTING 과 step 의 BMC_UPDATING · BMC_SETTING 이 이미 들어 있다. 실행 어휘(enum)는 앞서 있는데 그 어휘로 실제 행동할 데이터(접속 정보)는 비어 있는 비대칭이다. 이연 자체는 정당하나, "phase 는 있는데 수행 데이터가 없다" 는 상태를 문서로 명시해야 한다.

참고: provisioning 의 BmcPassword* 기능은 보드/BIOS 세팅용이며 특정 게스트 서버 인스턴스에 묶여 있지 않다.

G3진행/단계 추적 테이블이 아무도 쓰지 않고, 구조도 결함이다

provisioning_progress 와 setup_step 은 "공정을 끌고 가며 추적" 이라는 목적의 절반을 담당해야 한다. 그러나 현재 두 테이블에 행을 쓰는 코드가 단 한 곳도 없다(엔진 미구현이라 등록 서비스는 server/detail/custom/nic 만 만든다). 게다가 막상 엔진이 붙으면 곧장 막힌다:

  • setup_step 이 @OneToOne → guest_server_id 에 UNIQUE 가 걸려 서버당 단 1행만 저장 가능. "12 step × 재시도" 의 실행 이력을 담을 수 없다. 1:N(@ManyToOne)이어야 한다.
  • provisioning_progress 의 감사 리스너 누락 → last_transition_at(@LastModifiedDate, nullable=false) 가 자동 충전되지 않아 최초 INSERT 가 깨진다.
  • @Version 위치 역전 → 거의 불변인 detail 엔 낙관적 락이 있는데, 비동기 엔진이 자주 전이시킬 progress 엔 없다. 동시 전이가 lost update 로 새어 나간다.

즉 실행 추적의 절반은 "비어 있을 뿐 아니라, 채우려는 순간 부서지도록" 모델링돼 있다. 엔진을 올리기 전에 반드시 정비해야 할 토대다.

G4"무엇이 적용됐나" 와 "지금 어떤 상태인가" 가 약하다

자동화 시스템의 가치는 일관성 + 추적성이다. 그런데 (a) step 이 "OS 설치 성공" 만 남기고 어떤 ISO를 설치했는지 구조적으로 링크하지 않으며(status_meta JSON 뿐), (b) 서버의 운영 상태(유휴/진행/완료/실패/폐기)를 담는 필드가 없다. discovery_stage 는 "우리가 얼마나 아는가" 일 뿐 "이 서버가 운영상 어떤 상태인가" 가 아니다. 운영자가 "실패한 서버 / 완료된 서버 / 미할당 서버" 를 질의할 수단이 없다.

05

설계 품질 비판 (목적 정합 관점)

Q1 — 3중 1:1 과잉 정규화: 분리 근거를 구현이 스스로 무너뜨린다

guest_server · guest_server_detail · guest_server_custom 셋 다 한 서버에 1:1 이다. 보고서는 분리 근거로 "채워지는 시점이 다르다 / NULL 희소 / version / JSON" 을 들었다. 그런데 ServerRegistrationService 는 iPXE 한 번에 세 테이블 + NIC 을 모두 즉시 생성한다.

분리의 전제
detail/custom 은 나중에 lazy 하게 채워진다 → 그래서 별 테이블
실제 구현
등록 시점에 셋 다 eager 생성 (custom 은 빈 행) → 항상 함께 존재

전제가 깨지면 남는 건 "이득 없는 join 3개" 다. 특히 guest_server_custom 은 version 도 JSON 도 없는 문자열 2칸짜리라 분리 정당성이 가장 약하다 — detail 또는 guest_server 로 흡수할 후보다. 지금은 쪼개 놓고 한꺼번에 만드는, 두 방향의 단점만 취한 상태다. 셋 중 하나를 택해야 한다: (a) 진짜 lazy 생성으로 분리를 살리거나, (b) 합쳐서 join 을 줄이거나.

Q2 — 파생·중복 식별 필드: 자동 선택이 신뢰해야 할 데이터에 드리프트 씨앗

  • detail.vendor 는 boardModel.vendor 로 항상 파생됨 → 둘이 어긋나면 vendor 기반 분기가 오작동(자동화가 잘못된 벤더 경로 선택).
  • guest_server.model_name 은 어떤 코드도 쓰지 않는 고아인데 custom.product_model_name 과 의미가 겹침.
  • setup_step.phase_code 는 step_code.getPhaseType() 로 파생 가능.

자동 프로비저닝은 식별 데이터를 신뢰해 자료를 고른다. 같은 사실을 두 곳에 두면 그 신뢰가 깨질 자리를 만드는 셈이다(§중복 금지·SSOT 철학과도 충돌).

Q3 — 진행 상태의 이중 커서

"지금 어느 단계인가" 가 provisioning_progress.current_phase 와 "가장 최근 setup_step" 두 곳에서 읽힌다. 단일 진실 출처(SSOT)를 정하지 않으면 둘이 어긋난 채 운영자/엔진이 서로 다른 답을 본다. 어느 쪽을 권위로 삼을지 명문화가 필요하다.

Q4 — 라이프사이클/소유 모델의 부재가 의도인지 확인 필요

관리 자원은 LifecycleService(enabled/deprecated/deleted) 다형을 따르지만, 게스트 서버는 인스턴스라 그 패턴을 적용하지 않은 것은 적절하다(과적용 회피). 다만 서버도 "폐기/회수(decommission)" 라는 종료 상태가 실무에 존재하므로, G4 의 운영 상태(server status)와 묶어 최소한의 수명 종료 표현은 두는 편이 목적에 맞는다.

06

목적에 잘 맞는 설계 (계승)

  • board_model FK = 자동 선택의 축 — 목적의 핵심을 데이터 한 줄로 성립시킨다. 미등록 보드 거절로 무결성까지 확보.
  • 식별 데이터의 타입화 — MAC/IP VO 검증·정규화로 "잘못된 주소로 자동 접속" 류 사고를 원천 차단.
  • NIC 1:N + primary/bond — 실제 서버의 다중 NIC·본딩 현실에 맞음.
  • 공정 단계의 다형 enum — Phase/Step 이 method-per-constant 로 소속 관계를 표현, 새 단계 확장에 열려 있음.
  • 식별/인벤토리 분리 발상 — 방향 자체는 건전(다만 과적용은 Q1 에서 교정).
07

결론 · 권고

목적 적합성으로 보면 현재 스키마는 식별 엔진의 토대로는 합격, 실행 엔진의 토대로는 미완이다. 부팅한 서버를 보드 카탈로그에 정확히 묶어 자료 후보를 좁히는 능력은 갖췄으나, "무엇으로 만들지(세팅/OS)", "어떻게 접속할지(BMC)", "무엇이 일어났는지(진행·산출물·운영상태)" 를 표현하는 능력은 비어 있거나 부서져 있다. 선언된 골격 스코프 안에서는 이해되는 상태지만, 다음 두 가지는 골격 단계에서도 처리해 두길 권한다.

지금 결정해 둘 것 (척추 예약)

  1. 서버 ↔ 세팅 정의서/OS 선택 연결의 자리와 카디널리티를 지금 예약한다(G1). 자동 프로비저닝 전체가 이 연결 위에 선다 — 나중에 끼우면 진행/단계 테이블 설계까지 흔들린다.
  2. 서버 운영 상태(server status) 를 도입한다(G4). 운영자 질의·엔진 분기·종료(폐기) 표현의 공통 기반.

실행 엔진 올리기 전 정비

  1. setup_step 1:N 전환, provisioning_progress 감사/@Version 정비(G3) — 안 고치면 엔진이 첫 INSERT 부터 막힌다.
  2. 진행 커서 SSOT 확정(Q3), step↔적용 산출물 링크 도입(G4).
  3. BMC 접속 데이터(bmc_binding/credential)와 phase 어휘의 비대칭을 문서화하고 도입 시점을 못박는다(G2).

정합성 빚 청산

  1. 3중 1:1 을 lazy-생성 또는 병합으로 일관화(Q1).
  2. 파생·중복·고아 필드 제거(detail.vendor, guest_server.model_name, setup_step.phase_code) (Q2).
최종 판정

"식별" 목적에는 잘 설계되었고, "실행" 목적에는 아직 설계되지 않았다. 가장 시급한 한 가지를 꼽으면 — 새 테이블을 더 만드는 일이 아니라, 서버를 "무엇으로 프로비저닝할지" 에 연결하는 자리를 지금 비워둔 채로 두지 않는 것이다. 그 연결이 곧 이 프로그램이 존재하는 이유이기 때문이다.
