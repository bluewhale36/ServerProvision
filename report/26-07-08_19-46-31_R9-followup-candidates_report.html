§1. 요약 · 우선순위

세 후보 모두 reconciliation(자원 무결성 점검) 도메인의 잔여 과제다 — R9 는 "화면이 사용자에게 말을 걸게" 만들었고, 이 후보들은 그 다음 층위인 "시스템이 조치까지 대신하는 범위"(A), "약속했지만 만들지 않은 감지의 정리"(B), "마지막 남은 로그 전용 정보"(C)를 다룬다.

후보본질규모선행 조건권장 순서
A — 4종 드리프트 in-app 복구 액션탐지-조치 비대칭의 근본 해소: 심각한 드리프트일수록 조치가 "보고 닫기"뿐인 역설 제거캠페인 (3~5 슬라이스)이중 의도 검증 명제 대조 설계 + CP5 후 운영 관찰(어떤 종류가 실제로 자주 뜨는가)3순위
B — 미생성 DriftKind 6종: 구현 vs 정리문서-구현 괴리 해소: enum·라벨·허용 목록까지 있는데 탐지 코드가 없는 6종의 거취 결정결정 + 정리 S / 구현 시 M~L발생 시나리오의 실재성 판단 (운영 경험)1순위
C — 재발급 부분 실패 상세 표면화건수 토스트(R9-1)에서 한 걸음: 어느 자원이 왜 실패했는지 — 단, 공짜 해법 존재(§4)S (1 슬라이스)없음 (후보 A 의 SIGNATURE_INVALID 액션과 자연 결합)2순위

순서 논거: B 는 결정만 내리면 즉시 정리되면서 A 설계의 전제("어떤 kind 가 실재하는가")를 깨끗하게 만든다. C 는 비용 대비 효과가 가장 좋고 별도 UI 없이 기존 드리프트 화면을 재사용하는 해법이 있다. A 는 가치가 가장 크지만 종류별 쓰기 경로 설계가 필요해, CP5 이후 실제 운영에서 자주 만나는 드리프트부터 슬라이스를 고르는 것이 조사·설계 비용을 아낀다.

§2. 후보 A — 4종 드리프트의 in-app 복구 액션 (탐지-조치 비대칭 해소)

문제의 실체

현재 실제로 감지되는 6종 중 화면에서 해결까지 되는 것은 2종뿐이다 — PATH_DRIFT(적용 = DB 경로 갱신)와 GHOST_DB_ROW(적용 = 잔존 행 삭제). 나머지 4종은 더 심각한 불일치인데도 시스템이 제공하는 조치가 "보고 닫기"뿐이다. R9-2 의 '권장 조치' 문구가 무엇을 해야 하는지는 알려주지만, 그 조치 자체(백업 확인·파일 복원·재등록·재검증)는 여전히 화면 밖 수작업이다. R9 조사 §7-6 이 "백엔드 신기능 캠페인 규모"로 판단해 의도적으로 제외했다.

종류별 액션 설계 후보 (성격이 서로 다르다)

종류in-app 액션 후보성격 · 리스크규모

왜 캠페인 규모인가

  • 각 액션이 FS+DB 정합 쓰기 경로 + 예외 계층 + 확인 UX + 4범주 통합 테스트를 갖는 독립 수직 슬라이스(종류당 1개, 3~5 슬라이스).
  • 설계 단계에서 MK3 의 이중 의도 검증 명제("한 drift 의 처리가 둘 이상의 운영자 의도 변화를 동시에 포함하면 자동화 제외")와 액션별 대조가 선행되어야 한다 — 특히 HASH_MISMATCH 승인과 ORPHAN 정리.
  • R9-3 이 만든 modal 파이프라인(ConfirmModalType 확장)과 R9-2 의 recommendedAction 문구가 재사용 자산 — 액션이 생기면 해당 kind 의 권장 조치 문구를 "버튼 안내"로 갱신(enum 한 곳 수정).

§3. 후보 B — 미생성 DriftKind 6종: 구현이냐 정리냐 (결정이 본체)

문제의 실체

enum 12종 중 6종(RESOURCE_RENAMED·RESOURCE_RENAMED_ORPHAN·SOFTDEL_ESCAPE_TO_ORIGINAL·SOFTDEL_ESCAPE_TO_OTHER·TRASH_LOST·TRASH_MARKER_STALE)은 javadoc 의미 정의·R9-2 라벨·(3종은) autoApplicable 허용까지 갖췄는데 탐지 코드가 없다. MK3-2 가 "12종 3축 매트릭스" 슬라이스로 미룬 문서-구현 괴리이며, R9-2 는 javadoc 에 "탐지 미구현(MK3-2 소관)" 을 명기하는 선에서 격리했다.

잠복 결함 1건 (구현 노선 선택 시 함께 해소 필요) — autoApplicable 3종이 만약 생성된다면, 현재 apply() 가 PATH_DRIFT 와 동일하게 applyDriftedPath(newPath) 를 호출한다(코드 주석 스스로 "다른 동작이 필요할 수 있음" 명시). TRASH_MARKER_STALE 은 newPath 가 null 이라 NPE 경로다 — 미생성이라 현재는 도달 불가하지만, 구현을 택하면 kind 별 apply 동작 정의가 필수다.

갈림길과 판단 기준

노선내용선택 조건
① 구현 (MK3-2 노선)파일명 변경(sidecar 1:1 매칭)·휴지통 이탈·잔여 마커 정리 탐지 패스 추가 + kind 별 apply 동작 정의해당 시나리오(운영자가 파일명을 바꾸거나, 휴지통에서 수동으로 꺼내는 등)가 실운영에서 실제로 발생할 때
② 정리 (MK3-X 노선)dead 상수 6종 삭제 + 라벨·화이트리스트·javadoc 정리 — ghost 코드를 "한시적 안전망"으로 재해석한 MK3-1 선례와 같은 결발생 빈도가 무시 가능할 때. CLAUDE.md "갈라지는 시점에 분리"(선구현 금지) 원칙과 정합

이 후보는 코드량이 아니라 결정이 본체다. 판단 재료(어떤 드리프트가 실제로 뜨는가)는 CP5 이후 운영 관찰이 가장 정직한 출처이므로, 지금 정리 노선을 가결하기보다 "관찰 기간 후 결정" 으로 기록해 두는 것도 유효한 선택지다 — 다만 그 경우에도 위 NPE 잠복과 "문서가 약속한 감지가 동작하지 않는다"는 괴리는 결정 시점까지 잔존한다는 사실을 기록해야 한다.

§4. 후보 C — 재발급 부분 실패 상세 표면화 (공짜 해법 존재)

문제의 실체

R9-1 이 성공/실패 건수는 토스트로 표면화했지만("마커 재서명 완료 — 성공 41건, 실패 2건"), 어느 자원이 왜 실패했는지(ReissueResult.failures 목록)는 여전히 서버 로그로만 남는다.

설계 관찰 — 별도 UI 가 필요 없을 수 있다

재서명에 실패한 마커는 구 secret 서명인 채로 남는다. 따라서 다음 quick 점검에서 전부 SIGNATURE_INVALID 드리프트로 자연히 잡히고, R9-2 가 만든 드리프트 화면(라벨·감지 내용·권장 조치·R9-5 실명)이 실패 목록 역할을 그대로 한다. 전용 화면을 만드는 대신:

재발급 완료 & 실패 > 0
  → 자동으로 quick 점검 1회 트리거 (또는 토스트에 "지금 점검을 실행해 실패 자원을 확인하세요" 안내 + 버튼)
  → 실패 자원들이 SIGNATURE_INVALID 로 목록화 — 기존 화면 재사용, 신규 UI 0

이 방식이면 S 사이즈 슬라이스 1개로 끝나고, 후보 A 의 SIGNATURE_INVALID 단일 재서명 액션과 자연 결합된다(실패 자원 카드에서 바로 개별 재시도). 유의점 1가지: 재발급 직후 자동 점검은 running 동시 실행 가드와 순서 조정이 필요하다(재발급 Job 종료 후 트리거 — runReissueAsync 의 finally 이후).

§5. 착수 조건 · 다음
  • 공통 선행: R9 CP5(브라우저 E2E) 통과 — 특히 후보 A·B 는 운영 관찰(어떤 드리프트가 실제로 뜨는가)이 설계 재료다.
  • 착수 지시 시: Notion 에 후보별 페이지 신설(본문 4항목 규약) → 표준 체크포인트(CP1 plan 부터). 인벤토리 코드는 신규 캠페인이면 R10* 또는 기능 성격상 MK* 계열 — 부여는 사용자 결정.
  • 이 보고서의 위치: 세 후보의 조사·설계 방향 기록이 목적 — 착수 전 CP1 을 대체하지 않는다(각 슬라이스 CP1 에서 대안 비교·적대 검증을 다시 수행).
