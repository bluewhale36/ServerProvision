§1. 결과 요약
3개한 도메인의 화면 이름
1 / 4툴팁 있는 헤더 버튼
0개스캔 완료 이벤트 리스너
12종enum 원시명 노출 (생성은 6종)
5개제안 하위 단계 R9-1~5

결론부터: 이 도메인의 결핍은 인프라가 아니라 "접속"이다. 스캔 완료를 화면에 반영할 폴링 인프라(background-jobs.js), 확인/오류 모달(ConfirmModal/ErrorModal), 비동기 폼 제출(data-async-submit)이 전부 이 페이지에 이미 로드되어 있으나 구독·연결이 안 된 상태다. 여기에 ② 내부 구현 용어(enum 원시명, manifestHash, secret 회전)가 번역 층 없이 1차 UI 문자열로 노출되고, ③ 성격이 다른 세 기능(일상 점검 / secret 회전 1회용 관리 도구 / 업로드 실패 복구 saga)이 같은 버튼 줄에 동렬 배치되어 있다. 세 요인이 겹쳐 "무엇을 하는 도메인인지, drift 가 나면 뭘 눌러야 하는지" 감을 잡을 수 없는 현재 상태가 만들어졌다.

개선 방향은 기존 패턴 정합(C안) 골격 + 멘털모델 요소(A안) 이식으로 확정했다(§4). 하위 단계는 5개 — R9-1(스캔·재발급 완료의 화면 자동 반영) → R9-2(DriftKind 사용자 언어화 + 자동적용 SSOT 승격) → R9-3(modal·폼 정합) → R9-4(정보구조 재배치·명칭 단일화) → R9-5(drift 자원 실명 표시, 유일한 엔티티 변경). 종합안을 3렌즈 적대 검증에 통과시켜 高 4건 포함 12건의 결함을 사전 보정했다(§6) — 특히 "완료 건수 metadata 는 현재 BackgroundJob API 로 기록 불가"(global/job 확장 필요), "R9-5 는 ddl-auto=validate 환경에서 수동 DDL 필수" 두 건은 그대로 진행했으면 CP4 에서 무너졌을 결함이다.

이 문서는 CP1(plan) 이 아니다 — R9 묶음의 현황·방향·단계 분해를 확정하는 조사 보고서다. §8 열린 질문에 대한 사용자 결정 후, R9-1 부터 슬라이스별 plan html(CP1)을 별도 산출한다.
§2. 불만 3건의 원인 실측 (코드 근거)

① "스캔 후 화면을 새로고침 할 필요가 있다" — 사실로 확정. 원인은 이벤트 구독 누락

스캔은 POST /maintenance/reconciliation/scan 이 JobStartResponse(jobId) 만 반환하고 실제 스캔은 @Async 스레드에서 돈다(ReconciliationRestController.java:59-67). 프론트 list.js 는 응답 수신 후 버튼을 "스캔 중…"으로 바꾸고 실제 완료와 무관한 고정 3초 setTimeout 후 라벨을 원복한다(list.js:43-51) — Deep 스캔은 스스로 "수 분 소요"라 경고하면서다. 완료를 화면에 전달할 인프라는 이미 돌고 있다: background-jobs.js:311-340 이 job 종료 시 bgjob:completed 이벤트를 전 페이지에 dispatch 하고, os-list.js:379·bios-list.js:98 가 이를 구독해 부분 갱신·토스트를 수행한다. 그러나 PATH_RECONCILIATION/MARKER_REISSUE 타입의 리스너는 코드베이스 전체에 0개 — reconciliation 페이지만 자기 도메인 job 의 완료를 구독하지 않는다. 개발자 주석 스스로 "후속 개선: jobId 폴링 후 COMPLETED 시 자동 reload"(list.js:46-47)로 인지된 상태였다.

② "'격리 복구'는 도대체 어떤 기능인가?" — reconciliation 이 아닌 별개 도메인의 링크

실체는 ISO 업로드 등록의 비동기 후처리가 인프라/일시 오류로 중단됐을 때, 업로드 파일을 격리 디렉토리(.iso-quarantine/<recoveryId>/)에 보존하고 재시도/폐기 결정을 받는 업로드 실패 복구 saga 다(OrphanQuarantineService.java:42-96, 상태 전이 PENDING → RECOVERED | DISCARDED, TTL 7일 후 reaper 자동 폐기). reconciliation 과는 데이터 소스(orphan_quarantine vs DriftReport)·트리거(업로드 실패 vs 스캔)·액션(재시도/폐기 vs 경로 정합)이 전혀 겹치지 않는 별개 개념이고, 격리 파일은 마커가 없어 마커 기반 스캔으로는 탐지 자체가 불가능하다(OrphanQuarantine.java:31). 그런데 진입 링크가 스캔 액션 버튼들과 같은 줄에 동렬 배치되어(reconciliation/list.html:58-60) "스캔이 찾은 격리를 복구하는 버튼"으로 오독된다. 더해서 reconciliation 자체에 DriftKind.ORPHAN(디스크 마커가 DB 와 매칭 안 됨)이 다른 의미의 '오펀'으로 존재해 같은 화면에서 용어가 충돌한다.

③ "'마커 서명 재발급'도 어떤 기능인가?" — secret 회전용 1회성 관리 도구의 상시 노출

PROVISION_MARKER_SECRET(HMAC 서명 키) 회전 직후 1회 실행하는 관리 도구다: 전 활성 자원의 마커를 현재 secret 으로 재서명해, 다음 스캔에서 전 자원이 SIGNATURE_INVALID 로 오탐되는 것을 막는다(PathReconciliationService.java:255-281). manifestHash 는 의도적으로 재계산하지 않아 변조 감지를 보존한다. 문제는 사용 빈도·위험도가 극단적으로 다른 이 도구가 일상 스캔 버튼과 동열로 상시 노출되고, 설명은 native confirm() 안에만 있으며("secret 회전 직후에만 실행하세요"), 부분 실패는 로그 WARN 으로만 소멸해 UI 에 전달되지 않는다(:274-277 — Job 은 그냥 COMPLETED).

④ (암묵) "drift 가 나도 어떤 대처를 해야 할지 감을 못 잡음" — 번역 층 부재 + 조치 비대칭

  • enum 원시명 12종이 그대로 사용자 문자열: PATH_DRIFT·MISSING·… 에 라벨/설명/권장 조치 메타데이터가 전무. IntegrityStatus.displayMessage 라는 선례(IntegrityStatus.java:23-27)가 있는데 DriftKind 엔 없다. 자원 표기도 OS_ISO #42(내부 PK).
  • 12종 중 실제 생성되는 것은 6종뿐 — RESOURCE_RENAMED 등 6종은 javadoc 과 자동적용 화이트리스트에만 존재하고 생성 코드가 없다(문서-구현 괴리, MK3-2 이연분).
  • 조치 비대칭의 역설: 자동 적용 가능한 것은 사실상 PATH_DRIFT(DB 경로만 갱신, FS 무변경)·GHOST_DB_ROW(행 삭제)뿐이고, 더 심각한 MISSING/ORPHAN/SIGNATURE_INVALID/HASH_MISMATCH 는 "무시"(기록 삭제) 외 시스템이 제공하는 대처가 없다.
  • modal 의미 불일치: "무시" 버튼이 soft-delete 용 modal 을 차용해(list.html:241) 제목 "자원 삭제" + "…삭제할까요?" 가 뜬다 — 사용자는 자원이 지워진다고 읽는다.
  • SSOT 위반 현존: 서버 화이트리스트는 5종(isAutoApplicable, :771-777)인데 템플릿은 4종 하드코딩 ×3곳 — GHOST_DB_ROW 는 서버가 허용하는데 UI 는 버튼을 숨기고 "허용되지 않습니다"라고 안내한다.
  • 보조 결함: DriftAutoApplyNotAllowedException 메시지가 두 발생 지점 모두에서 사실과 어긋남(stale) / 진행바는 3단계 중 SCANNING 만 계측 / 스캔 소요가 PT0.45S(ISO-8601 Duration 원문) 노출 / 화면 이름 3개(경로 점검·자원 경로 점검·경로 재조정).
Notion 이력이 확인해 준 것 — 이 문제의식은 새 발견이 아니라 기록된 부채다: MK1 의 scope 는 "감지까지"였고(설명 속성), MK3-2 후속 목록에 "UI 슬라이스: 12 DriftKind 3축 매트릭스"가 명시적으로 이연된 채 미착수였으며, R9 페이지(2026-06-20 생성, 상태 '시작 전', 본문 blank)에 "기능이나 사용법 등이 명확하지 않은 화면을 개선"이라는 문제의식이 이미 등록되어 있다.

골격 = C안. 근거: ① 실측이 확정한 문제의 형태("인프라는 있는데 접속이 없다")와 "기존 자산 접속"을 1원칙으로 세운 C안이 정확히 동형이다. ② C안만이 재발급 부분 실패의 UI 전달과 진행바 1단계 문제를 Job 인프라로 해소한다. ③ 자원 실명화는 A안의 렌더 시 resolve 대신 스캔 시점 스냅샷(스캔이 이미 Markable 인벤토리를 손에 들고 있어 기록 비용 0, purge 후에도 이름 보존)으로 채택.

A안에서 이식: 진단→처방 3단 카드 구성(무슨 일이→시스템이 할 수 있는 일→관리자가 할 일), "무시"→"보고 닫기" 개명 + 전용 문구. B안에서 이식: 12종 전부 라벨링(미생성 6종 포함 — 상수별 유무 분기를 만들지 않음), scan Job 에 reportId metadata 는 넣지 않음(전체 reload 방식에선 소비자 없는 데이터). C안 약점 보정: 주기 스캔(1h)이 열람 중 완료되면 예고 없는 reload 로 화면이 리셋되는 문제 — 자동 reload 는 이 페이지에서 방금 트리거한 jobId 에 한정하고, 그 외 완료는 알림 + 수동 새로고침으로 제한(데모 1 참조).

§5. R9 하위 단계 확정안 (검증 반영판)

§7. 비채택 대안 · 탈락 사유
  1. 렌더 시 자원 이름 실시간 resolve (A안) — purge 후 이름 소실 + 읽기 경로에 MarkableScanner 횡단 조회(N+1, R7 이 정리한 결합 방향과 역행). 스캔 시 스냅샷이 기록 비용 0 으로 동일 가치.
  2. os-list.js 식 fragment 부분 갱신 — 이 페이지는 지킬 폼 상태가 없는 read-only 목록이라 과설계. reload 선례(quarantine)로 충분.
  3. scan Job 의 reportId metadata — 전체 reload 방식에선 소비자 없는 데이터. 부분 갱신으로 승격하는 시점에 추가.
  4. "미해결 항목 1차 뷰" 전면 재편 (A안) — Drift 는 상태 필드 없는 존재=상태 모델이라 보고서 횡단 동일성 추적이 엔티티 대변경 없인 불가. 최신 보고서 요약 배너로 축소 채택.
  5. 미생성 DriftKind 6종의 삭제/생성 구현 — MK3-2 "12종 3축 매트릭스"·MK3-X 정리 슬라이스 소관. R9 은 표시층만 — 단 라벨링 시 javadoc 에 미생성 사실 명기로 dead 상수 포장 위험 완화.
  6. MISSING/ORPHAN 등 4종의 in-app 복구 액션 신설 — 탐지-조치 비대칭의 근본 해소이나 백엔드 신기능 캠페인 규모. R9 은 recommendedAction 텍스트로 한정, 별도 캠페인 후보로 Notion 기록.
  7. 문구의 MessageSource 이전 — 단일 언어 운영에선 enum 필드가 SSOT 로 더 단순(IntegrityStatus 선례와 통일). i18n 요구 발생 시 이전.
  8. 재발급의 별도 페이지 분리 — 1회용 도구에 페이지 신설은 네비 부담. 접이 "관리 도구" 구획 + 상시 설명문으로 충분.
