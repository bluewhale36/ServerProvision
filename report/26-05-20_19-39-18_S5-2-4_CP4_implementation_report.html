1. 개요 — S5-2-4 가 해결하는 문제

1.1 결손 영역

S5-2-2 가 사용자 직접 hard-delete 에 typed-name 검증을 도입한 뒤에도, 사용자 입력이 부재한 시스템 진입 경로 가 결손 상태였다 :

  • TTL 자동 만료 — TrashTtlWorker.purgeExpired() 가 식별 + log 만. 실제 hard-delete 미구현
  • nudge REPLACE — service 의 purge 직접 호출. typed-name 검증 우회

1.2 본 슬라이스의 약속

축달성 내용
typed-name휴지통 페이지 + nudge REPLACE 모두 동일 TypedNameVerifier 통과
TTL 자동 hard-deleteTrashTtlWorker → PurgeExecutor 위임, retry 3회 backoff
감사 영속화단일 purge_log 테이블 — 성공/실패 모두 1행씩 누적
운영 설정trash_settings singleton 테이블 — TTL/cron/알림/retry 일원화
회고자원별 WHERE resource_id=? ORDER BY occurred_at 단일 쿼리로 전체 이력 조회

9. 운영자 회고 시나리오

9.1 자원별 회고 — 어떤 자원의 lifecycle 추적

-- 자원 27 의 전체 hard-delete 이력
SELECT id, origin, outcome, occurred_at, purged_at, details
FROM purge_log
WHERE resource_type = 'OS_ISO' AND resource_id = 27
ORDER BY occurred_at;

9.2 실패 자원 추적 — 마지막 outcome 이 FAILED 인 자원 셋

WITH last_log AS (
  SELECT resource_type, resource_id,
         ROW_NUMBER() OVER (PARTITION BY resource_type, resource_id
                            ORDER BY occurred_at DESC) AS rn,
         outcome
  FROM purge_log
)
SELECT resource_type, resource_id
FROM last_log
WHERE rn = 1 AND outcome = 'FAILED';

9.3 3회 초과 누적 실패 자원 — 운영자 격상

-- details.attemptNumber > 3 인 자원 (JSON 컬럼 조회)
SELECT resource_type, resource_id, MAX(occurred_at) AS last_attempted_at
FROM purge_log
WHERE outcome = 'FAILED'
  AND JSON_EXTRACT(details, '$.attemptNumber') > 3
GROUP BY resource_type, resource_id;
