도메인 로직 없이 DirectoryBrowseService 에 위임만 하던 동형 4 컨트롤러(OS/BIOS/BMC/Subprogram)를 단일 DirectoryBrowseController(GET /management/browse)로 흡수했다. 4곳에 하드코딩돼 있던 includeFiles 는 요청 param(기본 false)으로 이동해 표시 정책 결정권이 호출처(frontend)로 넘어갔고, frontend 6 호출처는 템플릿의 data-browse-url 교체만으로 원자적으로 이행됐다. 사용자 체감 동작(파일 표시 여부·에러 응답)은 6 호출처 전부 보존 — 유일한 동작성 수정은 subprogram-edit.js 의 includeFiles=true 명시 1건이며, 이는 구 컨트롤러의 서버 하드코딩 true 를 클라이언트 명시로 대체한 것이다.

R8 묶음 관점 — R8-1(browse 예외 advice 승급)은 R2-3(ba8dc73)이 이미 흡수 완료(Notion reconcile 됨), 본 R8-2 로 R8 의 실질 작업이 모두 끝났다. CP5(사용자 브라우저 E2E) 통과 시 R8 묶음 종결.

§3. 동작 보존 검증 — includeFiles 전이 매트릭스 (실측)

통합의 유일한 의미 변화는 "서버 하드코딩 → param 기본 false". 보존 논거의 핵심은 클라이언트가 이미 includeFiles 설정으로 렌더를 필터한다는 실측(bundle-upload-bootstrap.js renderEntries 의 entries.filter(type==='DIR'))이다:

호출처통합 전: wire / 서버값 / 표시통합 후: wire / 서버값 / 표시판정
iso-new / iso-editincludeFiles=true 명시 / true / 📁+📄동일 명시 / true / 📁+📄보존
bios-new미전송 / false(하드코딩) / 📁미전송 / false(기본) / 📁보존
bmc-new · subprogram-new미전송 / true(하드코딩) / 📁 (클라 필터가 📄 숨김)미전송 / false(기본) / 📁 (📄 가 wire 에서도 사라져 페이로드 감소)보존
subprogram-edit미전송 / true(하드코딩) / 📁+📄 (필터 없음 — 하드코딩 의존)includeFiles=true 명시(신규 수정) / true / 📁+📄수정 1건으로 보존

에러 응답 경로도 불변 — 호출처 전부 명시 Accept: application/json → ApiExceptionHandler JSON 채널(R2-3=R8-1 라우팅 그대로), 신규 테스트가 4종 status 를 통합 URL 에서 재고정.

§6. CP5 가이드 — 사용자 브라우저 E2E

확인 포인트는 "경로 탐색이 여전히 되는가 + 파일 표시 규칙이 그대로인가" 두 가지다. 개발자도구 Network 에서 요청이 /management/browse 로 나가는지 함께 보면 확실하다.

  1. BIOS / BMC / Subprogram 신규 폼 — "대상 디렉토리" 옆 [탐색] → 패널이 뜨고 디렉토리만 나열, 하위 진입/상위 이동/선택 적용 정상.
  2. OS ISO 신규/편집 폼 — [탐색] → 디렉토리 + 📄 파일(.iso) 도 표시되고 파일 클릭 시 경로 입력 채워짐.
  3. Subprogram 편집 폼 — entrypoint 상대경로 선택 트리에 📄 파일이 표시되는지 (이번 슬라이스의 유일한 동작성 수정 지점 — 여기서 파일이 안 보이면 즉시 보고).
  4. 에러 케이스(선택) — 탐색 패널에서 존재하지 않는 경로 직접 입력 시 패널 status 줄에 오류 메시지(404)가 뜨는지.
CP5 승인 후 commit 은 R8-2 파일만 명시 add — 작업 트리의 report/26-07-05_…_U2-substep…_report.html 은 병행 세션(U2) 산출물이라 제외한다.
