1. 요약 — 무엇을 신설했나

정상 등록된 자원의 디렉토리를 .provision.json 마커째 스캔 범위 안 다른 위치에 복사해 두면, 원본이 살아 있는 동안 정합화 점검이 아무 신호도 내지 않았다. 이 침묵은 그 자체로는 눈에 띄지 않지만, 나중에 원본이 유실되면 방치돼 있던 사본이 "경로 이동됨(PATH_DRIFT)"으로 오인되어 정본 자리를 슬그머니 차지하는 정본 하이재킹의 씨앗이 된다. 이 관찰은 2026-07-19 전수 점검의 O-2 로 기록됐고, 사용자가 표면화·해소 방식을 ①~④로 결정했다. HF4-5 는 그 결정을 신규 드리프트 종류 RESOURCE_DUPLICATED("자원 중복 존재")로 구현한 슬라이스다.

구성무엇을어떻게
탐지 원본이 완전 정상인데 동일 신원(자원 종류·번호)의 마커와 본체가 다른 위치에도 있는 상황 정합화 점검의 분류 단계에 RESOURCE_DUPLICATED 를 추가 — 사본 경로마다 드리프트 1행. 원본이 유실된 뒤에는 기존 PATH_DRIFT/MISSING 분류가 그대로 유효(이중 보고 없음).
택일 해소 운영자가 원본과 복제본 중 정본으로 남길 쪽을 고르고 나머지를 파일시스템에서 삭제 전용 모달의 radio 택일(기본값 = 등록 경로 원본). 복제본을 고르면 원본을 삭제하고 등록 경로(DB)를 복제본 경로로 갱신(PATH_DRIFT 적용 로직 재사용).
승격 가드 복제본을 정본으로 올릴 때, 내용이 변조된 사본이 정본 자리를 빼앗지 못하게 차단 서명·마커 기록 지문·본체·포함 관계에 더해 사본 실제 바이트의 지문을 재계산해 정본 기록과 대조(검수 반려 반영 — §4). 불일치·재계산 불능이면 409.
무인 제외 이 드리프트는 자동 적용 대상에서 구조적으로 제외 택일 입력이 필요하고 파일 삭제가 비가역이라, DriftResolutionMode.NONE + 해결 전략 bean 미등록으로 표준 [적용]·강제 적용·스캔 무인 적용 모두 도달 불가(HASH_MISMATCH 선례).
검수는 1차 반려됐다. 초기 승격 가드는 "사본 마커에 기록된 지문"만 대조했는데, 샌드박스 실측에서 마커는 정상 최신본을 그대로 복사하고 페이로드 바이트만 변조한 사본이 이 대조를 통과해 정본으로 승격되고 원본이 삭제됐다(302). 수정으로 승격 갈래 마지막에 사본 실제 바이트의 지문 재계산 가드를 추가했고, 재검증에서 같은 시도가 409 로 거절되며 파일·DB·드리프트 카드가 모두 무변경임을 확인했다. 상세는 §4.
결과. 워크트리 테스트 1,281건 green(신설 — 탐지 6, 해소 12, 통합 9, 라벨 1). main 통합(병렬 핫픽스 묶음 합산) 후 1,345건 green. 통합 jar 를 샌드박스에 올려 다섯 시나리오(탐지·양 갈래 해소·반려 재현 재검증·정리 회귀)를 실측으로 확인했다(§5). 스키마 변경은 없다("RESOURCE_DUPLICATED" 19자 ≤ 기존 kind 컬럼 VARCHAR(32)).

3. 설계 결정 — 왜 이렇게 지었나

D1. 탐지 우선순위 — 원본이 완전 정상일 때만 중복으로 판정

중복 판정은 정합화 점검의 기존 분류 순서(원본 위치를 먼저 검사) 안에서, 원본이 "완전 정상"으로 확인된 지점에서만 발동한다. "완전 정상"은 마커 존재 + 파싱 성공 + HMAC(Hash-based Message Authentication Code, 해시 기반 메시지 인증 코드) 서명 유효 + 본체 존재의 네 검사를 모두 통과한 상태다. 이 순서가 두 가지를 보장한다.

  • 원본 자체의 이상이 우선. 원본에 서명 불일치나 본체 부재 같은 자체 드리프트가 있으면 그 신호(보안·소실)가 먼저 잡히고, 중복 신호에 묻히지 않는다. 원본이 회복되면 다음 점검이 중복을 보고한다.
  • 원본 소실 시엔 이중 보고 없음. 원본이 사라지면 그 사본은 기존 PATH_DRIFT(또는 본체 없으면 MISSING)로 분류될 뿐, 같은 사본이 두 종류로 동시에 보고되지 않는다. "이동"이라는 서사는 원본이 없어야 성립하므로 두 상황은 원본 생존 여부로 의미가 갈린다.

정밀 점검(deep scan)의 HASH_MISMATCH(내용 변경)와 RESOURCE_DUPLICATED 는 독립 신호라 동시 보고될 수 있다(TRASH_MARKER_STALE 의 독립 신호 선례). 그래서 중복 검사는 본체 존재 확인 직후, deep 해시 검사와 무관한 위치에 둔다.

D2. 동일 자원 복제본이 여럿일 때 — 사본 경로당 드리프트 1행

사본이 셋 이상이어도 사본 경로마다 드리프트 1행을 만든다(oldPath=원본, newPath=그 사본). 모달은 언제나 2지선다(원본 vs 그 사본)다. 드리프트 엔티티가 이미 가진 old_path/new_path 두 컬럼을 재사용하므로 스키마가 늘지 않고, 카드 한 장 = 결정 하나라 UX 가 단순하다. N지선다로 한 행에 사본 목록을 다 담으려면 두 컬럼으로는 부족해 자유 텍스트에 구겨 넣게 되는데(파싱 취약), 이는 해결 파라미터의 신뢰 출처를 무너뜨린다.

한 행에서 사본을 승격하면 DB 경로가 바뀌어 남은 행들의 oldPath 가 현재 경로와 어긋난다 — 이 낡은 행으로 해소를 시도하면 stale 가드가 409 로 거절하고, 다음 점검이 새 기준으로 재분류한다. 드리프트를 불변으로 두는 설계(해결마다 카드 내용을 갱신하지 않음)와도 정합적이다.

D3. 삭제 방식 — 모달 확인 경유 직접 삭제 (휴지통·격리 아님)

사용자 결정 원문이 "파일시스템에서 삭제"다. 삭제 대상은 자기 DB 행이 없는 잉여 사본이라 복원 개념 자체가 없다. 휴지통(.soft-deleted) 경유를 택하지 않은 이유는 휴지통 lifecycle 이 soft-delete 된 DB 행과 결합돼 있어서다 — 대응 행이 없는 사본을 넣으면 휴지통 화면에도 안 보이고 보존기간 자동 정리도 안 되는 "휴지통 안의 새 orphan"이 되어, 정합화 도구가 오히려 부정합을 낳는 모순이 된다. ORPHAN 격리 구역 이동도 부적합하다 — 격리는 "주인 없는 파일"의 비파괴 보존이 목적인데 여기는 주인(정본)이 명확히 살아남는다. 삭제 시 [AUDIT] 로그로 삭제 경로·생존 경로·드리프트 id 를 남긴다.

D4. 무인 자동 적용 제외 — mode=NONE + 해결 bean 미등록

DriftResolutionMode.NONE 을 부여하고 해결 전략(DriftResolution) bean 을 등록하지 않는다(HASH_MISMATCH 선례 그대로). 그러면 표준 [적용]·강제 적용·스캔 무인 적용이 모두 구조적으로 도달 불가하다. 근거는 세 가지가 겹친다 — ① 해결이 택일 입력(survivor)을 요구하는데 무인 실행에는 그 입력이 없고 ② 파일 삭제라는 비가역 파괴 동작이며 ③ 어느 쪽이 정본인지는 운영자 의도라 시스템이 단일화할 수 없다(이중 의도 검증 명제). mode 가 NONE 이면 isAutoApplicable()==false 가 enum 안에서 자동 성립하므로 별도 플래그가 필요 없다. 인터페이스에 파라미터를 추가하는 대안은 기존 8개 해결 bean 전부가 안 쓰는 인자를 받는 churn 을 부르므로 탈락시켰다.

D5. 마커 정합 — 생존측 마커 재발급 불요 (단, 무검증 승격은 아님)

복제본을 승격해도 마커를 재발급하지 않는다. 마커의 서명 대상 필드(resourceType, resourceId, attributes, createdAt, manifestHash)에 절대 경로가 없어 마커가 위치 독립적이기 때문이다(실측 — attributes 는 boardId·상대경로·원본 파일명 등만 담는다). 사본 마커는 원본 마커의 byte-동일 복사라 서명이 그대로 유효하고, manifestHash 는 내용 기반이라 경로와 무관하다. PATH_DRIFT 해결도 파일을 건드리지 않고 엔티티 경로만 갱신하는 확립된 계약이라 선례가 일치한다.

다만 재발급이 불요라고 해서 승격이 무검증인 것은 아니다. 승격 직전 가드 네 겹을 둔다 — ① 사본 마커 서명 유효 ② 사본 마커 기록 지문 == 엔티티 기록 지문(원본 재업로드 후 남은 구버전 사본 차단) ③ 사본 본체 존재 + 원본/사본이 포함 관계 아님(사본이 원본 트리 내부면 원본 삭제가 사본까지 지움 → 거절) ④ 사본 실제 바이트 재계산 지문 == 엔티티 기록 지문(검수 반려로 추가 — §4). 저비용 가드(①②③)를 모두 통과한 맨 마지막에 비용이 가장 큰 ④를 둔다.

D6. 라벨 다형 + 본체 존재 판정 SSOT — 조건분기 대신 다형성

상세 화면의 경로 2필드 라벨은 DriftKind 에 getOldPathLabel()/getNewPathLabel()(기본 "이전 경로"/"새 경로")을 두고 RESOURCE_DUPLICATED 상수만 "원본 경로 (DB 기록)"/"복제본 경로"로 재정의했다(enum method-per-constant). 템플릿은 이 메서드를 참조할 뿐 kind 를 비교하는 분기가 없다 — 사용자 문구의 진실 출처를 enum 에 모으는 R9-2 원칙 그대로다. 또한 "layout(마커 배치)별 본체 존재" 판정을 PathReconciliationService 의 private static 에서 MarkerLayout.resourceBodyExists(Path) 다형 메서드로 끌어올렸다 — 중복 해소 서비스도 같은 판정이 필요해졌기 때문으로, 두 곳 복붙 대신 layout 자신이 판정을 소유하게 해 SSOT 를 지켰다.

4. 검수 반려와 수정 — 마커는 정상, 내용만 변조된 사본의 승격 차단

이 슬라이스의 설계 자체는 1차 검수에서 승인됐다(탐지 우선순위·다중 사본·삭제 방식·auto-apply 제외·라벨 다형). 반려는 설계가 아니라 승격 가드의 실측 구멍 하나에서 나왔다. 아래 네 단계로 정리한다.

① 재현 — 무엇이 통과되면 안 되는데 통과됐나

샌드박스 실측에서 다음 사본을 만들었다: 마커 파일은 정상 최신본을 byte-동일 복사해 서명이 유효하고 마커에 기록된 manifestHash 도 엔티티 기록값과 일치하지만, 페이로드 바이트만 변조한 사본이다. 이 사본으로 [복제본 유지]를 시도하자 승격이 302(성공)로 통과했다 — 변조된 파일이 정본으로 승격되고 원본이 삭제됐다. 기대는 409 거절이었다.

② 원인 — 가드가 "기록값"만 대조했다

초기 승격 가드는 사본 마커 안에 적힌 manifestHash 를 엔티티 기록값과 비교하는 데서 멈췄다. 마커 파일을 정상본에서 그대로 복사하면 그 기록값은 당연히 일치하므로, 페이로드가 아무리 변조돼도 이 대조는 무력했다. 마커의 서명은 마커 자신의 내용만 보증할 뿐, 같은 디렉토리의 페이로드가 그 마커와 실제로 일치하는지는 보증하지 않는다. 승격은 원본 삭제 + DB 경로 갱신이 걸린 비가역 액션이라 이 구멍이 그대로 하이재킹의 우회로가 된다.

③ 가드 설계 — 실제 바이트를 다시 계산해 대조

승격 갈래의 맨 마지막에 가드 ④를 추가했다: 사본 실제 바이트의 지문을 재계산해 엔티티 기록 지문과 대조하고, 불일치이거나 재계산이 불가능하면 DriftResolutionNotAllowedException.duplicateNotPromotable(...)(409)로 거절한다. 설계에서 지킨 원칙 세 가지가 있다.

  • 해시 규칙 중복 구현 0. 재계산은 정밀 점검(deep scan)이 쓰는 MarkableScanner.recomputeManifestHash(Markable) 를 그대로 재사용한다(HashAcceptService·정합화 점검 deep 경로와 동일 SPI — Service Provider Interface, 서비스 제공자 인터페이스). 해시 규칙을 여기서 다시 구현하지 않는다.
  • 6개 scanner 동반 수정 0. 이 SPI 는 getResourcePath() 기준으로 계산하는데, 필요한 것은 "같은 엔티티를 사본 경로로 본" 계산이다. 그래서 경로 하나만 사본으로 바꿔 끼우는 읽기 전용 위임 뷰 DuplicatePathView(record implements Markable) 를 넘겼다 — SPI 시그니처를 넓히지 않아 6개 scanner 구현을 손대지 않는다. 이 뷰의 상태 변이 메서드(reissueMarker·deprecate·softDelete 등)는 전부 즉시 IllegalStateException 으로 실패해, 재계산 경로가 실수로 엔티티를 변이시키는 사고를 시끄럽게 차단한다.
  • 가드 배치와 갈래 구분. ④는 비용이 가장 큰 가드라 저비용 가드(서명·마커 기록 지문·본체·포함 관계)를 모두 통과한 뒤에 둔다. 그리고 원본 유지(keepOriginal) 갈래는 재계산하지 않는다 — 어차피 삭제할 사본의 내용 진위는 결과에 영향이 없어 검증 비용이 무의미하다.
동기 수행을 유지한 이유(D6 갱신). 가드 ④가 대용량 ISO 에서 수십 초 걸릴 수 있음에도 동기로 둔다 — 승격은 운영자 확인을 거친 드문 비가역 액션이라, 비동기의 완료 이벤트 배선·중복 시작 차단 비용이 오히려 과하다. 빈도가 늘면 HASH_MISMATCH 선례의 BackgroundJob 비동기 전환이 후속 후보이며, 이 판단을 코드 주석에 명기했다. 원본 유지 갈래는 재계산이 없어 종전대로 즉시 응답한다.

④ 재검증 — 같은 시도가 이제 409

수정 후 통합 jar 샌드박스에서 반려 시나리오를 재현했다. 변조 페이로드 사본으로 [복제본 유지]를 시도하자 409 가 떨어졌다:

// HTTP 409 — 승격 거절 (파일·DB·드리프트 카드 무변경)
{"message":"복제본을 정본으로 승격할 수 없습니다 — 복제본 실제 내용의 지문이 현재 정본 기록과
 다릅니다 — 마커만 정상이고 내용이 변조된 사본일 수 있습니다. 그 내용을 쓰려면 재업로드가
 올바른 경로입니다. 상태를 확인한 뒤 다시 점검하세요."}

파일시스템·DB·드리프트 카드가 모두 그대로였고, 이어서 [원본 유지] 해소로 사본을 정리하는 정상 회귀도 확인했다. 변조 내용을 정본으로 삼는 올바른 경로는 승격이 아니라 재업로드(검증 재통과) 또는 HASH_MISMATCH 수용 흐름이라는 메시지도 함께 제공한다.

6. 잔여·후속
  • 마커만 복사된 사본(본체 없음)은 원본 정상 시 침묵(의도 — 테스트로 고정). "복제본"의 의미(실체 있는 사본)를 지키고 모달 택일의 전제(양쪽 다 실물)를 깨지 않기 위한 스코프 결정이다. 원본 소실 시엔 기존 분류가 MISSING("마커만 이동 가능성")으로 이미 잡는다. 표면화가 필요해지면 후속 백로그.
  • ORPHAN·ESCAPE 분류는 여전히 key 당 첫 발견만 사용한다. 수집 단계가 List 화되어 발견 자체는 전부 보존되지만, 소비를 바꾸는 것은 이 슬라이스 범위 밖이라 행동 변화를 중복 탐지에만 한정해 회귀 면적을 최소화했다. 다중 orphan 전체 보고는 후속 후보.
  • recheckable=false — RESOURCE_DUPLICATED 에는 [다시 점검] 축을 부여하지 않는다(PATH_DRIFT 도 동일). 사본을 수동으로 지운 운영자는 [보고 닫기] 또는 다음 점검으로 해소한다. 재확인 지원은 후속 후보.
  • 승격 재계산의 비동기 전환 후보 — 대용량 자원 승격 빈도가 늘면 HASH_MISMATCH 선례의 BackgroundJob 비동기 전환을 검토(코드 주석에 명기).
  • 스키마 변경 없음 — 신규 컬럼·테이블 0. enum 문자열 "RESOURCE_DUPLICATED"(19자)는 기존 kind 컬럼 VARCHAR(32)에 수용된다. 수동 DDL 산출물 없음.
  • 병렬 슬라이스 경계 — 보고서 저장·건수 표시 계층(HF4-4)은 접촉하지 않았다. 접점은 분류 로직(PathReconciliationService)과 해소·상세 UI 뿐. execution/·provisioning/ 무접촉, CLAUDE.md 수정 없음.
