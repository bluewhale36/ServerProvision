§1. 요약 (TL;DR)

VERDICT
단일 권장안 : 옵션 C 잔류 → 정책 질문 (isoPath 가변성) 선결 후 R1-4-3 의 후속 슬라이스에서 옵션 A 또는 옵션 L 로 흡수

본 agent 의 최종 판단은 R1-4-3 본 슬라이스에서 updateIso 를 이동하지 않는 것. 그 근거는 5 senior 중 3 명 (#2 · #3 · #5) 이 공통으로 짚은 사실 — 현 OSMetadataService.updateISO 본문이 marker 재발급도 sidecar 이동도 manifestHash 재계산도 하지 않는 단순 2 줄 (검증 + setter)[1] — 이 본질적 결함을 안고 있는 메서드를 어디로 옮기든 silent corruption 위험 은 그대로라는 점이다. 즉 R1-4-3 의 D-1 은 위치 결정 문제 가 아니라 도메인 정책 결정 문제 이며, 정책 결정 (isoPath 가변성 인정 / 부정) 이 선행되어야 위치 결정이 자명해진다.

핵심 trade-off — "지금 이동 (B/A) 하면 R1-4 묶음이 깔끔하게 닫히지만, 본질적 결함을 가진 메서드의 책임 위치가 잘못 표준화된다. 지금 잔류 (C) 시키면 R1-4 묶음 종료가 한 슬라이스 미뤄지지만, 정책 결정 후 자연스러운 위치가 도출된다."

정책 분기 미리보기
① isoPath 가변성 인정 (현 동작 유지 + 무결성 회복 강화) → 옵션 A (IsoIntegrityService 로 이동, finalizeMarker 동형 강화 동반)
② isoPath 가변성 부정 (description-only 환원) → 옵션 L (편집 UI 자체 축소, updateIso 책임 소멸 → 위치 결정 무의미화)
③ 판단 보류 (R2 까지 미루기) → 옵션 C 잔류 (R1-4-3 은 무결성 분리만, ISO 책임 100% 분리는 R1-4-4 또는 R2 로 이연)

§2. 문제 정의

R1-4-3 슬라이스는 OSMetadataService 에서 ISO 무결성 책임 4 메서드를 IsoIntegrityService 로 분리하면서 R1-4 묶음 (R1-4-1 lifecycle / R1-4-2 registration / R1-4-3 integrity) 을 닫는다. 본 슬라이스 종료 시점에 OSMetadataService 에 유일하게 잔류하는 ISO 책임 이 updateISO(osMetadataId, isoId, request) 한 메서드[1].

왜 모호한가

  • 등록 (registration) 도 아님 — 이미 영속화된 자원의 속성 갱신
  • 무결성 (integrity) 도 아님 — verify / verifyAndRecord 와 다른 의미 축
  • lifecycle 상태 전이 도 아님 — toggle / softDelete / restore / deprecate 의 4 boolean 축에 속하지 않음
  • 편집 (mutation) 이라는 별도 축 — 본 프로젝트의 service 분할 명세에 없음

이 모호함이 5 senior 가 서로 다른 위치를 권장하는 원인이다.

현재 본문 (line 242-247)

@Transactional
public void updateISO(Long osMetadataId, Long isoId, ISOUpdateRequest request) {
    // S4.x — isoPath 를 PathPolicyService 로 검증해 ".." 등 traversal 입력이 DB 에 silent 저장되는 사고 차단.
    Path validated = pathPolicyService.assertWritablePath(request.isoPath());
    requireLiveISO(osMetadataId, isoId).update(validated.toString(), request.description());
}

실측 사실 — marker 재발급 호출 없음. sidecar 이동 없음. manifestHash 재계산 없음. ISO.update(isoPath, description) 은 this.isoPath = isoPath; this.description = description; 의 2 줄 setter[2].

#5 — 반박 b · javadoc ≠ 책임 정합 "controller javadoc 의 '편집/toggle/...' 은 URL prefix 일치 일 뿐. URL 같다고 service 책임이 같은 게 아니다. IsoUploadController 가 별도 controller 로 분리된 것처럼, URL prefix 일치는 단순 그루핑 결정이다."
#5 — 반박 c · SRP 위반 "IsoLifecycleService 의 단일 책임은 4 boolean 상태 전이 (active/disabled, live/soft-deleted, current/deprecated, alive/purged). updateIso 는 속성 변경 — 동일 service 에 두면 '상태 전이' 와 '속성 변경' 두 책임이 응집점을 잃는다."
#4 — 옵션 B 재방어 "DDD application service 관점에서 'use-case 응집점' 이 핵심. 사용자가 ISO 의 toggle 도 update 도 동일 controller (= 동일 application service) 에서 호출하는 게 자연스럽다. #5 의 '상태 전이만 lifecycle' 정의는 본 프로젝트가 채택하지 않은 좁은 정의일 뿐."
#5 — 재반박 "본 프로젝트는 LifecycleService interface 를 명시 정의했고 메서드 명단은 toggleEnabled / softDelete / restore / deprecate / undeprecate / purge. 코드가 이미 좁은 정의를 채택 했다. #4 의 '좁은 정의는 본 프로젝트가 채택 안 함' 은 사실 오인."
#2 — 옵션 D (옵션 B 의 정교화) 제안 "#5 의 interface 우려는 옵션 D 로 흡수 가능 — service 에는 두되 interface 에는 미포함. 도메인별 DTO 다양성으로 interface 추상화 자연성 떨어진다는 점은 옵션 B 의 plan 본문도 인정한 약점[4]."
#5 — D 에도 반박 "interface 회피로 우회한다 해도 '같은 service 안에서 상태 전이와 속성 mutation 이 섞이는 문제' 는 해소 안 됨. interface 포함 여부가 아니라 service 내부 응집의 문제."

R2 표준화 권고

  • 표준 A — 자원 경로 가변성이 인정되면 마커 인접 service (Integrity / Lifecycle 의 정의 정렬에 따라 명명) 가 mutation 의 단일 진입점이며 본문은 sidecar 이동 + hash 재계산 + 재서명을 동반. 전제 : isoPath 가변성 정책 답 = A.
  • 표준 B — 자원 경로 불변 정책이 채택되면 description 등 metadata-only mutation 만 허용, 경로 변경은 reconciliation 도메인 (Scanner / FileMover) 으로 표준화. 전제 : isoPath 가변성 정책 답 = B.
  • 경고 — 본 D-1 에서 옵션 B 를 단순 채택하면 R2 의 4 도메인 표준이 "lifecycle service 가 update 까지 담당" 으로 굳어진다. 이는 본 프로젝트의 LifecycleService interface 정의 (4 boolean 상태 전이) 와 충돌하는 책임 확장 — 4 도메인에 동시에 잘못된 표준이 박힘.
R2 확장성 요약 — D-1 의 정책 답이 R2 표준의 갈래를 결정한다. 위치 결정 (A/B/D) 보다 정책 결정 (가변/불변) 이 4 배 큰 결정.
