§3. Q2 — 구현된 예외 로직은 적절한가 (mixed)

현재 라우팅 위상 (실측)

두 advice 모두 @Order(HIGHEST_PRECEDENCE) + produces 매처로 분리: ApiExceptionHandler(@RestControllerAdvice, JSON 16핸들러), WebExceptionHandler(@ControllerAdvice, HTML 5핸들러 = NotFound/ChildLifecycleBlocked/Conflict/OptimisticLock/Domain fallback). 양쪽 handleDomain 이 이미 AnnotationUtils.findAnnotation(@ResponseStatus) 로 status 흡수.

강점 (코드 검증)

  • 보안 예외 다형 단일 핸들러 — handleSecurity(ApiExceptionHandler:252)가 SecurityException.httpStatus() 로 400/403/413/415/500 을 if/switch/Map 0건 매핑. 과거 catch(DomainException) 가 보안 예외 흡수해 silent 500 으로 새던 회귀를 계층 설계로 차단 — CLAUDE.md "분기 대신 다형성"에 정합.
  • ExceptionLogPolicy — is5xxServerError() 다형으로 양 advice 로그레벨 SSOT(4xx WARN/no-stack, 5xx ERROR+stack). 2-advice 복붙 drift 방지.
  • @Order 산술 금지 주석 명문화(v1 underflow / v2 28건 회귀 흉터 기록).

잔여 결함 (대부분 latent — critique 가 다운그레이드)

  • latent FieldBoundBadRequest XHR400/SSR500 비대칭 — WebExceptionHandler 에 전용 핸들러 부재 + FieldBoundBadRequestException 에 @ResponseStatus 없음 → SSR 이 만나면 handleDomain fallback 500. 단 throw site 가 전부 XHR 채널(InvalidBiosValueException=RestController, InvalidReplaceTargetException=nudge)이라 오늘 SSR 500 불가 = latent foot-gun.
  • latent JSON-전용 핸들러 4개 produces 미지정 — handleMethodArgumentNotValid/handleSecurity/handleZipInspectionFailed/handleMaxUploadSize 가 produces 미지정이라 text/html 요청에도 매칭 가능 → raw JSON bleed 잠재. 단 발생 엔드포인트가 XHR 전용이라 SSR 진입 경로 부재.
  • latent */* Accept 등록순서 의존 — 양 advice 동순위라 */* 는 bean 등록 순서 의존(주석이 자인). 단 1st-party JS 는 명시 Accept 송신, */* 는 curl/Accept-less fetch 에서만.
  • live browse/upload 컨트롤러 try/catch — CLAUDE.md 가 반복 사고 진원으로 지목한 컨트롤러 catch 복붙(browse 4-catch)이 advice 로 안 올라간 채 잔존.
판정 — 명시적 Accept 경로는 견고하나, R1-4-1 hotfix 가 표면화시킨 문제군을 "오늘 그런 경로 없음" 한시 invariant 로 가려둔 잔여가 있다. 이는 정확히 R2-3 이 닫으려던 지점이므로 방향성은 유효(다만 우선순위 R2-3 > R2-4/5).

§4. Q3 — 미실시 단계 재설계

R2-3 advice 통합 MODIFY (재초점)

"단일 advice 물리 통합 + @Order 제거" 헤드라인은 제외 — live 결함을 안 닫고(v3 가 라우팅·HTML-bleed 이미 차단) R1-4-1 3차 회귀 계열(HIGHEST_PRECEDENCE 산술 / Accept */* fallthrough) + webflux 공존 blind spot 의 흉터 위험만 진다. 대신 의도(라우팅 드리프트를 구조적으로 닫기)로 재초점:

  1. 상태 매핑 다형화 — NotFoundException@404 / ConflictException@409 / FieldBoundBadRequestException@400 super 에 @ResponseStatus 부착. 양 advice handleDomain 이 이미 findAnnotation 하므로 hierarchy 로 status 흡수 → FieldBoundBadRequest SSR 비대칭 root 소멸 + 일반 handleNotFound/handleConflict(plain body)를 handleDomain 으로 수렴=분기 감소(특수 body 핸들러 FieldBound/Nudge/DeleteReject/ChildLifecycleBlocked 는 유지).
  2. produces 명시 — JSON-전용 4핸들러에 produces=application/json 부착, raw JSON bleed 차단. (critique: 이는 Accept-매처 변이라 Web variant 추가 또는 whitelabel 수용 결정 + 회귀 가드 필수 — §5)
  3. browse/upload 4-catch → advice 승급 — 컨트롤러 try/catch 안티패턴 제거(선행: InvalidBrowsePathException@400 등).
  4. 회귀 가드 — AdviceExceptionMappingTest 에 */*, native-form bleed, browse status 회귀 가드 신설.

R2-4 ProblemDetail DROP

  • 외부 머신 컨슈머 부재 — 1st-party SSR+XHR 앱, PXE 진입점은 kickstart 반환(에러봉투 파싱 안 함).
  • 3종 이질 409 바디(ApiErrorResponse / NudgeRequiredResponse / DeleteRejectResponse)를 RFC 9457 flat {type,title,status,detail} 로 평탄화 불가.
  • 네이티브 ProblemDetail 활성 시 form-error.js fieldErrors 계약 회귀. 52 jsonPath 단언 + 20 JS 파일 blast radius 에 사용자 가치 0.
  • YAGNI — 표준 봉투가 미래에 진짜 필요해지면(외부 컨슈머 출현) 그린필드 재진입. 독립 슬라이스 신설 안 함.

R2-5 errorCode 다형성 MODIFY (축소 → R2-6 병합)

  • DomainException/SecurityException 에 errorCode() default 추가하는 109-클래스 cross-cutting 메커니즘 폐기(over-abstraction + 클래스명→wire silent 결합 + 응답 record 상수와 SSOT 중복).
  • 실 결함은 nudge-modal.js:89 의 body.message.indexOf('만료') fragile string 분기 1건뿐 → NudgeSessionExpiredException 에 안정 머신코드(NUDGE_SESSION_EXPIRED) 부여 + 프론트 1라인 교체. 기존 두 record 코드(NUDGE/SOFTDELETE)는 SSOT 유지. R2-6 에 흡수.

R2-6 alert→ErrorModal MODIFY (helper 중앙화)

  • 단순 alert()→ErrorModal.show() 치환 금지(파싱 N복제 잔존). ErrorModal.fromResponse(resp,opts) async helper 신설 — 바디 shape 단일화(ct.includes('json') 로 problem+json substring 버그 회피, detail||message 동시 수용).
  • 진짜 타깃 라우팅: delete-reject(:68 의 409+SOFTDELETE 사전체크 보존)/bios-list/reconciliation/confirm-modal-base. confirm() 파괴적 확인은 비대상.
§5. critique 보정 (R2-3 plan 에 필수 반영)
(1) produces 미지정 핸들러 목록 정정 — synth 가 handleNudgeRequired:82/handleSoftDeleteRequiresIntent:95 로 오기했으나 이 둘은 이미 produces 보유(no-op). 실제 대상은 4개: handleMethodArgumentNotValid:222 / handleSecurity:252 / handleZipInspectionFailed:264 / handleMaxUploadSize:278. (코드 직접 확인.)
(2) produces 부착은 그 자체가 Accept-매처 변이 — R1-4-1 3차 회귀와 동일 계열이다. 4핸들러에 produces 를 달면 SSR(text/html) 요청은 Web variant 부재로 whitelabel 로 fall-through(현재는 raw JSON bleed). → Web variant 추가 또는 whitelabel 수용을 명시 결정하고, native-form 회귀 가드(AdviceExceptionMappingTest)는 선택이 아니라 필수 전제.
(3) FieldBoundBadRequest 비대칭은 latent — throw site 가 전부 XHR 이라 SSR 500 발생 불가. "live 결함" 표기를 "latent(invariant-gated foot-gun)" 로 재프레이밍. 수리(@ResponseStatus(400))는 여전히 가치 있음(미래 SSR throw site 방어 + 분기 수렴).
(4) advice 물리 미통합 = ExceptionLogPolicy 보존 — advice 를 합치면 ExceptionLogPolicy 의 존재 이유(2-advice drift 방지)가 부분 소멸하는데, "물리 통합 제외" 판정이 이를 역으로 보존한다. R2-6 fromResponse 의 detail|| 수용은 drop 된 R2-4 의 speculative 선반영이나 비용 trivial + DRY 실가치라 무해.
