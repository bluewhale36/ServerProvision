00

v2 변경 요약

v1에서 v2로 넘어오면서 일어난 구조적 변화를 한 줄씩 정리합니다. 가장 큰 변화는 정체성(guest_server)과 인벤토리(guest_server_detail)의 분리입니다.

스키마 변경 내역
변경 guest_server 슬림화 — id / system_uuid / name / model_name / memo만 남김. board_serial · manufacturer · discovery_stage · product_name 등은 detail로 이동.
신설 guest_server_detail 추가 — vendor / board_model / board_serial / hardware_spec(JSON) / software_spec(JSON) / discovery_stage. guest_server와 1:0..1 매핑.
변경 host_nic_binding 1:1 → 1:N — is_primary, bond_group 컬럼 추가. 한 서버에 여러 NIC 표현 가능.
변경 location_assignment 슬림화 — datacenter/rack_id/rack_unit/role 제거, location 문자열과 kvm 한 자리 양수로 단순화.
제거 bmc_binding.vendor 삭제 — vendor가 guest_server_detail로 옮겨가서 중복 제거.
변경 낙관적 락 위치 재배치 — version 컬럼은 잦은 갱신 가능성이 있는 guest_server_detail과 provisioning_progress에만 둠.

01

설계 원칙

원칙 1 — 정체성과 인벤토리의 분리

guest_server는 운영자가 보는 "이 서버는 이런 이름의 이런 모델"이라는 가벼운 신원만 담습니다. CPU·메모리·BIOS 버전 같은 인벤토리는 guest_server_detail에 있습니다. 둘은 같이 만들어지지 않습니다 — 운영자가 UI에서 서버를 먼저 등록할 수도 있고, iPXE 등록이 먼저 일어날 수도 있으며, 진단 리눅스가 떠야만 detail이 채워집니다.

원칙 2 — 자격증명은 DB에 두지 않는다

v1과 동일. bmc_credential_ref는 Vault 경로만 갖고, 실제 비밀번호는 Vault에서 조회합니다.

원칙 3 — 스펙은 JSON, 분기는 다형성

hardware_spec·software_spec은 JSON 컬럼입니다. CPU 종류·메모리 슬롯 구성·디스크 개수가 모델마다 다르고, 시간에 따라 우리가 수집할 수 있는 정보의 형태도 진화합니다. 스키마를 미리 못 박는 대신 어플리케이션의 sealed record로 형태를 강제하고, DB는 JSON으로 자유롭게 둡니다.

원칙 4 — 낙관적 락은 "갱신이 잦은 곳"에만

@Version 컬럼은 비용이 있습니다 — 모든 갱신마다 version을 체크하고 충돌 시 ObjectOptimisticLockingFailureException을 처리해야 합니다. 그래서 거의 불변인 테이블에는 두지 않습니다. v2에서는 guest_server_detail과 provisioning_progress에만 둡니다.

원칙 5 — 모든 본체 테이블에 created_at / updated_at

감사·디버깅에서 필수입니다. "이 행이 언제 만들어졌고, 마지막으로 언제 갱신됐나"는 운영적 가치가 매우 큽니다. v2에서는 모든 본체 테이블에 두 컬럼을 표준으로 둡니다.

03

각 테이블 상세

v2에서 변경된 테이블 다섯 개 (guest_server, guest_server_detail, host_nic_binding, bmc_binding, location_assignment)를 중심으로 살펴봅니다. 변경 없는 테이블은 간략히만 다룹니다. 헤더를 클릭하면 펼쳐집니다.

guest_server 가벼운 신원 v2 변경
▶

v2의 핵심 변경입니다. 이제 이 테이블은 단순합니다 — 시스템 식별자(system_uuid) 하나와 운영자가 보는 표시 정보(name, model_name, memo) 그리고 타임스탬프뿐. 하드웨어·소프트웨어 스펙은 모두 guest_server_detail에 있습니다.

왜 version을 뺐나

v1에서는 모든 본체 테이블에 version을 두는 것을 표준으로 잡았지만, 이 테이블은 생성 후 거의 갱신이 없습니다 (name·memo 정도). 낙관적 락의 비용이 이득보다 큽니다. 동시 갱신 충돌이 운영적으로 보고된다면 그때 추가합니다.

컬럼타입제약설명
idBINARY(16)PKUUID v7. 시간 정렬 가능한 UUID.
system_uuidCHAR(36)UKNNSMBIOS Type 1 UUID. iPXE 등록 시 보고되는 시스템적 식별자.
nameVARCHAR(128)UKNN운영자가 임의로 부여하는 식별 이름. 시스템 내에서 유일해야 함.
model_nameVARCHAR(128)NN사내 서버 모델명. 같은 모델이 여러 대 존재 가능하므로 UNIQUE 아님.
memoTEXT자유 메모. nullable.
created_atTIMESTAMP(3)NN자동 입력.
updated_atTIMESTAMP(3)NN자동 갱신.
 
CREATE TABLE guest_server (
    id           BINARY(16)    PRIMARY KEY,
    system_uuid  CHAR(36)      NOT NULL,
    name         VARCHAR(128)  NOT NULL,
    model_name   VARCHAR(128)  NOT NULL,
    memo         TEXT,
    created_at   TIMESTAMP(3)  NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    updated_at   TIMESTAMP(3)  NOT NULL DEFAULT CURRENT_TIMESTAMP(3)
                    ON UPDATE CURRENT_TIMESTAMP(3),
    CONSTRAINT uk_guest_server_uuid UNIQUE (system_uuid),
    CONSTRAINT uk_guest_server_name UNIQUE (name),
    INDEX idx_guest_server_model    (model_name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci;
guest_server_detail 하드웨어·소프트웨어 인벤토리 v2 신규
▶

guest_server에서 분리되어 나온 인벤토리 테이블입니다. 진단 리눅스가 떴거나 운영자가 수동으로 스펙을 입력해야 행이 생깁니다. 1:0..1 관계입니다 — guest_server가 있어도 detail은 없을 수 있습니다.

왜 1:1이 아닌 1:0..1인가

iPXE 등록 직후엔 우리가 vendor·board_serial을 아직 모릅니다. 이때 NULL이 줄줄이 늘어선 detail 행을 미리 만드는 것보다, 정보가 확보된 뒤에 INSERT하는 게 깔끔합니다. 어플리케이션 레이어에서는 Optional<GuestServerDetail>로 받습니다.

JSON 컬럼의 역할

hardware_spec은 CPU·메모리·디스크·NIC 모델 등의 하드웨어 정보를, software_spec은 최초 BIOS 버전·BMC 버전·펌웨어 인벤토리를 담습니다. 모델별로 다른 스펙 구조를 갖기 때문에 컬럼화하면 NULL 천국이 됩니다. JSON으로 두고 어플리케이션의 sealed record가 구조를 강제합니다. 04 섹션에 JSON 스키마 예시가 있습니다.

왜 version을 뒀나

guest_server와 달리 detail은 진단 리눅스가 부분 갱신, 운영자가 수동 보강, 정기 검증 작업이 덮어쓰기 등 여러 주체가 갱신합니다. 낙관적 락이 충돌 감지에 유용합니다.

컬럼타입제약설명
idBINARY(16)PKUUID v7.
guest_server_idBINARY(16)FKUKNN1:0..1. UNIQUE이므로 guest_server당 최대 하나.
vendorVARCHAR(64)NNv1의 manufacturer 개명. GIGABYTE, Supermicro 등. Redfish 분기 키.
board_modelVARCHAR(128)메인보드 모델 (예: MZ72-HB0). v1의 product_name에 해당.
board_serialVARCHAR(128)UKSMBIOS Type 2 보드 시리얼. nullable + UNIQUE.
hardware_specJSONJSONCPU·메모리·디스크·NIC. 04 섹션 참조.
software_specJSONJSON최초 BIOS·BMC 버전 등 펌웨어 인벤토리. 04 섹션 참조.
discovery_stageVARCHAR(32)NNIPXE_REGISTERED 또는 DIAGNOSTIC_ENRICHED. 정보 수집 깊이.
versionBIGINTNNJPA @Version. 낙관적 락.
created_atTIMESTAMP(3)NN자동 입력.
updated_atTIMESTAMP(3)NN자동 갱신.
 
CREATE TABLE guest_server_detail (
    id              BINARY(16)   PRIMARY KEY,
    guest_server_id BINARY(16)   NOT NULL,
    vendor          VARCHAR(64)  NOT NULL,
    board_model     VARCHAR(128),
    board_serial    VARCHAR(128),
    hardware_spec   JSON,
    software_spec   JSON,
    discovery_stage VARCHAR(32)  NOT NULL,
    version         BIGINT       NOT NULL DEFAULT 0,
    created_at      TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    updated_at      TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3)
                       ON UPDATE CURRENT_TIMESTAMP(3),
    CONSTRAINT uk_detail_server UNIQUE (guest_server_id),
    CONSTRAINT uk_detail_board  UNIQUE (board_serial),
    CONSTRAINT fk_detail_server FOREIGN KEY (guest_server_id)
        REFERENCES guest_server(id) ON DELETE CASCADE,
    CONSTRAINT chk_detail_hw_json CHECK (hardware_spec IS NULL OR JSON_VALID(hardware_spec)),
    CONSTRAINT chk_detail_sw_json CHECK (software_spec IS NULL OR JSON_VALID(software_spec)),
    INDEX idx_detail_vendor (vendor),
    INDEX idx_detail_stage  (discovery_stage)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci;
host_nic_binding 호스트 NIC (1:N) v2 변경
▶

v1의 1:1 제약을 풀어 1:N으로 만들었습니다. 한 서버에 여러 NIC을 가질 수 있고, 그중 하나가 is_primary = TRUE로 표시됩니다. 같은 본딩 그룹에 속한 NIC들은 동일한 bond_group 값을 공유합니다.

DB로 강제 못 하는 제약

"서버 하나에 primary NIC는 하나뿐"이라는 제약은 MariaDB에서 깔끔하게 DDL로 표현이 안 됩니다 — PostgreSQL의 partial index가 없기 때문입니다. 이 제약은 어플리케이션 레이어에서 보장합니다. sealed interface로 NIC role을 표현하고, 도메인 서비스에서 새 primary가 지정될 때 기존 primary를 unset하는 트랜잭션을 돌립니다. 트리거로 강제할 수도 있지만, 비즈니스 로직이 DB에 흩어지는 게 더 큰 비용이라 권장하지 않습니다.

컬럼타입제약설명
idBINARY(16)PKUUID v7.
guest_server_idBINARY(16)FKNN1:N. UNIQUE 제약 제거 (한 서버에 여러 NIC 가능).
host_macCHAR(17)UKNNMAC은 전역 유일.
lan1_ipVARCHAR(45)IPv6까지 표현 가능. OS 부팅 전엔 NULL.
ip_sourceVARCHAR(16)NNDHCP / STATIC / RESERVED.
hostnameVARCHAR(253)FQDN까지 담을 수 있게 253자.
is_primaryBOOLEANNN기본 FALSE. 서버당 정확히 한 NIC이 TRUE가 되도록 어플리케이션이 보장.
bond_groupVARCHAR(64)본딩 그룹 식별자. 같은 그룹의 NIC끼리 묶임. nullable.
bound_atTIMESTAMP(3)NN바인딩 생성 시각.
 
CREATE TABLE host_nic_binding (
    id              BINARY(16)   PRIMARY KEY,
    guest_server_id BINARY(16)   NOT NULL,
    host_mac        CHAR(17)     NOT NULL,
    lan1_ip         VARCHAR(45),
    ip_source       VARCHAR(16)  NOT NULL,
    hostname        VARCHAR(253),
    is_primary      BOOLEAN      NOT NULL DEFAULT FALSE,
    bond_group      VARCHAR(64),
    bound_at        TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    CONSTRAINT uk_nic_mac    UNIQUE (host_mac),
    CONSTRAINT fk_nic_server FOREIGN KEY (guest_server_id)
        REFERENCES guest_server(id) ON DELETE CASCADE,
    INDEX idx_nic_server  (guest_server_id),
    INDEX idx_nic_primary (guest_server_id, is_primary),
    INDEX idx_nic_bond    (guest_server_id, bond_group)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci;
location_assignment 위치 (단순화) v2 변경
▶

v1의 datacenter / rack_id / rack_unit / role을 모두 제거하고, 운영자가 자유롭게 적는 location 문자열과 한 자리 양수 kvm 두 컬럼만 남겼습니다.

kvm 컬럼 — 한 자리 양수의 의미

KVM 스위치 포트 번호로 해석했습니다. TINYINT UNSIGNED는 본래 0~255 범위지만, CHECK 제약으로 1~9 범위를 강제합니다. 만약 "0번 포트"를 허용해야 한다면 CHECK 제약을 kvm BETWEEN 0 AND 9로 바꿔야 합니다 — 알려주시면 수정하겠습니다.

컬럼타입제약설명
idBINARY(16)PKUUID v7.
guest_server_idBINARY(16)FKUKNN1:1. 한 서버에 위치는 하나.
locationVARCHAR(255)NN자유 문자열. 운영자가 표기.
kvmTINYINT UNSIGNEDNNCHKKVM 포트 번호. 1~9 범위 강제.
assigned_atTIMESTAMP(3)NN위치 지정 시각.
 
CREATE TABLE location_assignment (
    id              BINARY(16)       PRIMARY KEY,
    guest_server_id BINARY(16)       NOT NULL,
    location        VARCHAR(255)     NOT NULL,
    kvm             TINYINT UNSIGNED NOT NULL,
    assigned_at     TIMESTAMP(3)     NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    CONSTRAINT uk_location_server UNIQUE (guest_server_id),
    CONSTRAINT fk_location_server FOREIGN KEY (guest_server_id)
        REFERENCES guest_server(id) ON DELETE CASCADE,
    CONSTRAINT chk_kvm_one_digit  CHECK (kvm BETWEEN 1 AND 9)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci;
bmc_binding BMC 신원 v2 변경
▶

v1과 거의 동일하지만 vendor 컬럼이 제거되었습니다. vendor가 guest_server_detail로 옮겨갔으므로 중복을 피합니다. Redfish 호출 시 vendor가 필요하면 detail을 JOIN해서 가져옵니다.

컬럼타입제약설명
idBINARY(16)PKUUID v7.
guest_server_idBINARY(16)FKUKNN1:1.
bmc_macCHAR(17)UKNN호스트 MAC과 다른 별개 NIC.
bmc_ipVARCHAR(45)관리망 IP.
ip_sourceVARCHAR(16)NNDHCP / STATIC 등.
fw_versionVARCHAR(64)BMC 펌웨어 버전. 펌웨어 업데이트 단계 후 갱신.
credential_ref_idBINARY(16)FKbmc_credential_ref 참조. 자격증명 확보 전엔 NULL.
bmc_credential_ref Vault 참조 (v1과 동일)
▶

v1과 동일합니다. DB에 비밀번호 없음, Vault 경로와 사용자명만 보관.

provisioning_progress 큰 단계 진행 상태 (v1과 동일)
▶

v1과 동일. current_phase 코드 + phase_meta JSON + version (낙관적 락) + 타임스탬프.

setup_step 세부 체크포인트 (v1과 동일)
▶

v1과 동일. Phase 안의 개별 작업 단위와 상태 (PENDING/RUNNING/SUCCEEDED/FAILED/SKIPPED).

provisioning_transition_log 전이 이력 (v1과 동일)
▶

v1과 동일. append-only 이력 테이블.

04

JSON 컬럼 설계

guest_server_detail.hardware_spec과 software_spec의 권장 스키마입니다. DB는 JSON_VALID만 강제하지만, 어플리케이션 sealed record가 형태를 보장합니다.

hardware_spec — 하드웨어 인벤토리

CPU·메모리·디스크·NIC의 모델과 수량을 담습니다. 모델 정보는 진단 리눅스의 dmidecode, lspci, lsblk 출력에서 추출합니다.

{ "cpu": { "model": "AMD EPYC 7773X 64-Core", "sockets": 2, "cores_per_socket": 64, "threads_per_core": 2 }, "memory": { "total_gib": 1024, "modules": [ {"slot": "DIMM_A1", "size_gib": 64, "speed_mts": 3200, "vendor": "Samsung"} ] }, "disks": [ {"bay": 0, "model": "Samsung PM9A3", "size_tb": 3.84, "interface": "NVMe"} ], "nics": [ {"slot": "onboard-1", "model": "Intel X710", "speed_gbps": 10} ] }

software_spec — 펌웨어·소프트웨어 인벤토리

최초 등장 시점의 BIOS·BMC·옵션 펌웨어 버전을 기록합니다. Phase 1 (BIOS/BMC 업데이트) 완료 후에는 bmc_binding.fw_version과 guest_server_detail.software_spec.bmc.version이 갱신됩니다.

{ "bios": { "vendor": "American Megatrends", "version": "R12", "release_date": "2024-08-15" }, "bmc": { "vendor": "AMI MegaRAC", "version": "12.84.10", "redfish_version": "1.18.0" }, "option_firmware": [ {"component": "raid-controller", "model": "Broadcom 9560-16i", "version": "5.230.02"} ] }
JSON 검색을 피하라

MariaDB의 JSON_EXTRACT 조건 검색은 가능하지만, 일상 쿼리에서는 피하시길 권합니다. 만약 "CPU가 EPYC인 서버를 다 찾아라" 같은 쿼리가 자주 발생한다면, 그건 컬럼화 신호입니다 — JSON에서 끄집어내 정규 컬럼이나 별도 인덱스 테이블로 옮기는 게 운영적으로 안정적입니다.

05

관계와 카디널리티

v2의 관계 그래프입니다. guest_server가 중심이고, detail이 새로 1:0..1로 붙었으며, host_nic_binding이 1:N으로 풀렸습니다.

왜 detail은 1:0..1인가

guest_server가 만들어졌어도 detail이 아직 없을 수 있습니다. 이는 두 가지 시나리오에서 자연스럽게 발생합니다.

① 운영자가 UI에서 먼저 등록 — 입고 예정인 서버를 미리 시스템에 등록하는 경우. physical 박스가 아직 연결되지 않았으므로 system_uuid는 어떻게 채우냐가 문제가 됩니다. 이 경우엔 임시 placeholder UUID를 발급하거나, system_uuid를 nullable로 풀어야 합니다 — 이 부분은 미확정 사항으로 분류하겠습니다 (07 섹션).

② iPXE 등록만 끝남, 진단 리눅스 미진입 — 가장 흔한 케이스. SMBIOS UUID는 확보됐지만 vendor·board_serial을 아직 모름. detail 행 자체가 없습니다.

06

설계 결정 Q&A

왜 guest_server에는 version을 안 두고 detail에는 두는가?+

version 컬럼은 비용입니다. 모든 UPDATE에 WHERE 조건이 추가되고, 충돌 시 OptimisticLockingException 핸들링이 필요합니다. 그래서 "갱신이 잦고 동시에 일어날 가능성이 있는" 테이블에만 둡니다. guest_server는 name·memo 정도가 가끔 바뀌고 동시 갱신이 거의 없습니다. guest_server_detail은 진단 리눅스가 부분 갱신, 운영자 수동 보강, 검증 작업의 덮어쓰기 등 여러 주체가 갱신하므로 둡니다.

hardware_spec과 software_spec을 별도 컬럼이 아니라 한 JSON에 합치면?+

합칠 수도 있지만 두 정보의 갱신 주기가 다릅니다. hardware_spec은 진단 리눅스 시점에 한 번 채워지고 거의 변하지 않습니다 (디스크 교체 정도). software_spec은 펌웨어 업데이트 단계마다 갱신됩니다. 분리해두면 어플리케이션 레이어에서 부분 갱신 시 의도하지 않은 덮어쓰기를 피할 수 있습니다. JSON 컬럼 전체를 매번 다시 쓰는 게 아니라 둘 중 하나만 갱신 가능합니다.

vendor가 guest_server_detail에 있는데 Redfish 호출 시 매번 JOIN해야 하나?+

네, 또는 어플리케이션 레이어에서 vendor를 Redis에 캐싱하는 두 가지 선택지가 있습니다. vendor는 거의 불변이고 잦게 조회되므로 Redis 캐시가 유리합니다 — 키는 vendor:guest:{guestServerId}, TTL은 길게 (예: 1시간) 잡고 detail 갱신 시 invalidate. JOIN 비용을 매번 내는 것보다 깔끔합니다.

host_nic_binding 1:N에서 "primary NIC 하나"는 어떻게 보장?+

MariaDB는 PostgreSQL의 partial unique index를 지원하지 않아 DB 레벨로 깔끔하게 강제 못 합니다. 세 가지 방법이 있는데, 어플리케이션 레이어 보장을 권합니다.

① 어플리케이션 (권장) — NicAssignmentService.markAsPrimary(serverId, nicId)가 트랜잭션 안에서 기존 primary를 false로, 새 primary를 true로 함께 갱신. sealed interface NicRole로 컴파일 타임 안전성 추가.

② 트리거 — BEFORE INSERT/UPDATE에서 같은 guest_server_id의 다른 primary를 false로 갱신. 비즈니스 로직이 DB에 흩어져 추적이 어려워집니다.

③ 데이터 모델 변경 — host_nic_binding 1:1을 유지하고, 추가 NIC은 별도 테이블로 두기. 본딩 표현이 복잡해집니다.

kvm 컬럼을 왜 JSON이 아닌 정수로?+

"한 자리 양수"라는 명시적 요구를 받았기 때문에 TINYINT UNSIGNED + CHECK 제약으로 범위를 강제합니다. 만약 미래에 "KVM 포트가 두 개 이상이면" 같은 요구가 생기면 JSON이나 별도 테이블로 옮기는 게 맞습니다. 지금은 단일 정수가 가장 정직한 표현입니다.

discovery_stage가 detail로 옮겨가면, 운영자가 등록만 한 서버의 stage는 어떻게 표현?+

detail 행 자체가 없으면 stage는 "아직 시작 안 함"으로 해석됩니다. 어플리케이션 레이어에서 Optional<GuestServerDetail>로 받고, 없으면 DiscoveryStage.NOT_STARTED로 매핑합니다. sealed interface로 표현하면 모든 stage가 컴파일 타임에 망라됩니다.

07

미확정 사항

v2를 정리하면서 새로 떠오른 결정 포인트들입니다.

미결 ①

system_uuid를 NOT NULL로 유지할지 — 운영자가 입고 전에 UI로 서버를 등록하는 시나리오가 있다면 system_uuid가 NULL일 수 있습니다. 만약 그런 시나리오가 없다면 (모든 guest_server는 iPXE 등록을 통해서만 만들어진다면) 현재 NOT NULL 유지가 맞습니다. 알려주시면 정합니다.

미결 ②

kvm 컬럼 범위 — "한 자리 양수"를 1~9로 해석했는데, 0을 포함해야 하면 0~9로 바꿔야 합니다 (TINYINT UNSIGNED + CHECK 제약 수정).

미결 ③

hardware_spec / software_spec의 JSON 스키마 확정 — 04 섹션의 예시는 권장안입니다. 실제 진단 리눅스가 어떤 형태로 수집해서 보고할지에 따라 키 구조가 달라질 수 있습니다. 어플리케이션 sealed record 정의 시점에 함께 확정하는 게 자연스럽습니다.
