§1. 결론 (verdict)

3곳의 ObjectProvider(Board / OS-metadata / OS-iso 스캐너)는 speculative 가 아니라 real — 직접 주입으로 바꾸면 부팅이 깨진다(bootRun 실증). 반면 R4~6(BIOS / BMC / Subprogram) 스캐너에는 ObjectProvider 가 애초에 없으며, 있을 필요도 없다.

사용자 가설 확정 — "R3 를 먼저 끝내면 R4~6 하위단계가 달라질 것"은 정확했다. R3/R1 은 "verify 를 LifecycleService 로 끌어올림 → scanner 에 ObjectProvider 추가" 패턴을 박았고 이게 순환의 원인이다. R4~6 가 이 패턴을 답습하면 동일 순환이 재발해 ObjectProvider 가 또 필요해진다. R4~6 는 inline 유지가 옳다(cycle-free, ObjectProvider 불요).
핵심 구분 — cascade @Lazy(commit fb094f8에서 제거)는 speculative였다(자식 service 가 Board 로 역참조 0). 그러나 스캐너 ObjectProvider 가 끊는 순환은 실재하는 닫힌 생성자 순환이다 — 둘은 다른 사안.
§2. 검증 질문·배경

R4~6 = BIOS / BMC / Subprogram 도메인. 각 도메인의 마지막 하위단계(R4-5 / R5-5 / R6-4)에서 cascade @Lazy 를 일괄 제거했다(fb094f8, "speculative 확정"). 이 시점에 제기된 질문 :

  • 각 단계에 존재하는 ObjectProvider lazy 주입이 여전히 필요한가, 아니면 @Lazy 처럼 speculative 인가?
  • (사용자 통찰) R3 와 R4~6 은 동일 출발이었으나 R3 만 먼저 완주한 이유는 "R3 를 다 구현하면 R4~6 하위단계가 변할 것"이라는 예상 때문. 그 예상이 맞는지?

§3. 정적 분석 — 단일 변수: "LifecycleService 가 TypedNameVerifier 를 주입하는가"

ObjectProvider 의 유무를 가르는 변수는 단 하나다. 해당 도메인의 purgeWithTypedNameCheck 가 typed-name 검증을 TypedNameVerifierImpl 로 위임하는가, 아니면 엔티티 displayName() 으로 inline 비교하는가.

TypedNameVerifierImpl ──eager List<MarkableScanner>──▶ {그}MarkableScanner ▲ │ │ │ ObjectProvider (LAZY) ← 유일한 break └────eager TypedNameVerifier──── {그}LifecycleService ◀┘
  • TypedNameVerifierImpl:35 가 List<MarkableScanner> 를 eager(생성자, @RequiredArgsConstructor) 주입 — expected name 을 scanner 의 displayName() 으로 해석하기 위해.
  • Board/OS/Iso 의 LifecycleService 가 TypedNameVerifier 를 eager 주입 (purge 시 verify() 호출).
  • 그 사이 scanner → LifecycleService 변만 ObjectProvider 로 lazy. 전부 생성자 주입이므로 이 lazy 가 사라지면 닫힌 생성자 순환이 되어 Spring 이 자체 해소 불가(BeanCurrentlyInCreationException).
왜 controller 의 verify() 는 순환을 안 만드나 — BiosNudgeController:50 등도 typedNameVerifier.verify() 를 호출하지만, 컨트롤러는 스캐너가 역주입하지 않으므로 순환이 닫히지 않는다. 순환은 오직 "스캐너가 주입하는 service" 가 verifier 를 주입할 때만 형성된다.
§4. 실증 (bootRun) — @Lazy 제거 때와 동일 잣대

정적 논증(닫힌 생성자 순환)만으로 충분하지만, cascade @Lazy 제거가 "bootRun 실측 순환 0"으로 정당화됐던 것과 동일한 잣대로 실증했다. IsoMarkableScanner 의 ObjectProvider<IsoLifecycleService> 만 직접 주입으로 바꿔 bootRun :

§5. R4~6 가 다른 이유 — inline displayName 이 이미 SSOT

BIOS/BMC/Subprogram 의 purgeWithTypedNameCheck 는 verifier 를 경유하지 않고 엔티티에서 직접 비교한다 :

// BiosService:481-484 (BMC:643 / Subprogram:667 동형)
String expected = bios.displayName();        // ← SSOT 소스 (Markable.displayName())
if (!expected.equals(typedName)) {
    throw new TypedNameMismatchException(expected, typedName);
}
  • service 가 TypedNameVerifier 를 주입하지 않음 → §3 의 순환 변이 닫히지 않음 → scanner 가 service 를 직접(eager) 주입해도 안전 → ObjectProvider 불요.
  • inline 의 expected 가 이미 displayName()(= SSOT 소스)이므로 typed-name SSOT 요구는 verifier 경유 없이도 충족된다.
  • purge 경로는 이미 엔티티를 로딩(requireExisting…)하므로 verifier 의 "active→trash lookup + 부재 응집" 이 중복 — service 에서는 불필요한 책임.
함의 — R4~6 는 R3/R1 보다 더 단순하고 순환이 없는 상태다. R4~6 의 남은 typed-name SSOT 하위단계는 R3 의 "verify→LifecycleService + scanner ObjectProvider" 패턴을 답습하면 안 된다. 답습하면 ObjectProvider 가 또 생긴다.

R7 을 "스캐너에서 ObjectProvider 만 빼기"로 하면 §4 대로 부팅이 깨진다. 진짜 선택지 :
