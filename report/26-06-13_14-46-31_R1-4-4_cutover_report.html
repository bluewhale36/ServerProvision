§1. 한눈 요약

직전 커밋 8ab2bf2 가 ISO 등록의 FS-then-DB 비원자성 결함(파일은 디스크에 써졌으나 후속 DB insert·마커 발급 실패 시 트랜잭션이 파일을 되돌리지 못해 오펀 발생)을 오펀 복구 saga 로 메웠다. 이 saga 는 ISO 도메인에 묶인 채 두 부채를 남겼다 — ① 실패 종류마다 자라는 Runner 의 multi-catch, ② Launcher→Runner→OrphanIsoRecoveryService→Launcher 생성자 순환을 @Lazy 로 임시 차단(프로젝트 4번째 누적). §3. ◑ 의존 그래프 — @Lazy 순환을 책임 분리로 DAG 화

핵심 도식. 한 서비스가 두 소비자(record/retry)의 책임을 모두 가져 순환이 생겼다. record/purge(launcher 불요)와 retry/discard(launcher 필요)를 분리하니 그래프가 DAG 가 되어 @Lazy 가 사라진다.

# BEFORE — 생성자 순환 (빨강 변), @Lazy 로 임시 차단
   ┌────────────────────────── @Lazy (순환 차단) ──────────────────────────┐
   │                                                                       ▲
   ▼                                                                       │
IsoRegistrationLauncher ──▶ IsoRegistrationRunner ──▶ OrphanIsoRecoveryService
                                                              (record + retry 한 덩어리)
                              retry() ──▶ launcher.startRegistration  ⟲ 순환!
# AFTER — 책임 분리 → DAG (순환 변 없음), @Lazy 삭제
[action-side]                                          [write-side · leaf]
OrphanRecoveryService                                  OrphanQuarantineService ◀──┐
   │ Map<ResourceType, OrphanRecoverySpi>                  (record/purge/조회)     │
   ▼                                                                          │ record 위임
IsoOrphanRecoverySpi ──▶ IsoRegistrationLauncher ──▶ IsoRegistrationRunner ──┘
   (launcher 의존이 여기로 이동)

✓ 어느 경로도 OrphanRecoveryService / IsoOrphanRecoverySpi 로 되돌아오지 않음 → 순환 0 → @Lazy 삭제
런타임 증명 — env 충족 후 bootRun 이 Started ServerProvisionApplication in 4.3s 로 전체 컨텍스트 로드 성공. BeanCurrentlyInCreationException/순환 0건. 즉 DAG 가 정적 추적뿐 아니라 Spring 빈 와이어링으로도 확정.
§4. ◑ 실패 → disposition() → dispatch — 자라는 분기의 다형성 흡수

BEFORE — Runner 가 종류별로 분기 (자람)

catch (IsoNudge e)           { nudge }
catch (HashMismatch
     | DuplicateContent
     | UploadIntentConflict
     | DuplicateFilename e)   { fail }   // CONTENT
catch (ISOFileStorage e)     { quar(STORAGE_IO) }
catch (DataIntegrity e)      { quar(DB_CONSTRAINT) }
catch (IsoMarkerWriteFailed) { quar(MARKER_WRITE) }
catch (RuntimeException e)   { quar(UNEXPECTED) }
// ← 새 예외 추가 시 catch 줄이 또 자람 (누락=silent 오분류 사고)

AFTER — 예외가 자기 처분 보유, Runner 단일 dispatch

catch (DataIntegrityViolationException e)   // Spring 어댑터(유일 잔존)
    quarantineAndFail(DB_CONSTRAINT)
catch (RuntimeException e)
    if (e instanceof RegistrationFailure f)
        switch (f.disposition()) {          // sealed → exhaustive
          Cleanup       -> fail (격리 X)
          Nudge(id)     -> fail NUDGE_REQUIRED:{id}
          Quarantine(fc)-> record() → fail ORPHAN_RECOVERY:{recId}
        }
    else quarantineAndFail(UNEXPECTED)
// ← 새 예외는 disposition() 선언을 컴파일이 강제. catch 안 자람.

처분 진리표 (예외 7종 → disposition, 구 multi-catch 와 1:1)

예외disposition()Runner 행동
IsoClientHashMismatch · DuplicateISOContent · DuplicateFilename · IsoUploadIntentConflictCleanup격리 없이 fail (finalize 가 이미 파일 정리)
ISOFileStorageExceptionQuarantine(STORAGE_IO)격리 + ORPHAN_RECOVERY:
IsoMarkerWriteFailedExceptionQuarantine(MARKER_WRITE)격리 + ORPHAN_RECOVERY:
IsoNudgeRequiredExceptionNudge(nudgeId:UUID)파일 보존 + NUDGE_REQUIRED:{uuid}
DataIntegrityViolationException Spring— (구현 불가)어댑터 catch → Quarantine(DB_CONSTRAINT)
미분류 RuntimeException— (instanceof 아님)else → Quarantine(UNEXPECTED)
왜 catch(RegistrationFailure) 가 아니라 instanceof? — RegistrationFailure 는 Throwable 비상속 인터페이스라 catch 절 타입이 될 수 없다(Java 제약). 그래서 catch(RuntimeException) → instanceof 2-갈래로 받되, 이 분기는 새 실패 타입이 늘어도 자라지 않는다 — 분류는 disposition() 다형성이 전담하고 여기는 "자기-분류 실패인가/미분류인가" 고정 판정일 뿐.

§6. ◑ 클래스 관계도 — 계약 / 다형성 / 도메인 어댑터
# 처분 계약 (control flow 다형성)
«interface» RegistrationFailure
   └ FailureDisposition disposition()
        ▲ implements
        ├── IsoClientHashMismatch / DuplicateISOContent / DuplicateFilename / IsoUploadIntentConflict  → CLEANUP
        ├── ISOFileStorageException        → Quarantine(STORAGE_IO)
        ├── IsoMarkerWriteFailedException  → Quarantine(MARKER_WRITE)
        └── IsoNudgeRequiredException      → Nudge(payload().nudgeId())

«sealed» FailureDisposition   permits  ←── switch exhaustive (default 불요)
   ├── record Cleanup()
   ├── record Nudge(UUID nudgeId)
   └── record Quarantine(OrphanFailureClass failureClass)
# 재등록 위임 SPI + 도메인 payload (behavior 분산 · data 다형성)
«interface» OrphanRecoverySpi                  «interface» OrphanRecoveryPayload
   ├ ResourceType supportedType()                    └ @JsonTypeInfo(Id.CLASS)
   └ OrphanRetryResponse relaunch(ctx)                  ▲ implements
        ▲ implements                                    └── IsoRecoveryPayload(description, clientHash)
        └── IsoOrphanRecoverySpi ──▶ IsoRegistrationLauncher

OrphanRecoveryService ──(주입)──▶ Map<ResourceType, OrphanRecoverySpi>  # OS_ISO → IsoOrphanRecoverySpi
분리 원칙 : 처분(control flow)은 sealed 로 exhaustive 보장(같은 패키지 집중). payload(durable data)는 비-sealed + Id.CLASS 로 도메인-로컬 구현 허용(global→management 역의존 0). 재등록 behavior 는 SPI 로 도메인에 분산.
§7. ◑ orphan_quarantine 스키마 — 엔티티 일반화 (Hibernate 생성 DDL)

ddl-auto=update 부팅 시 Hibernate 가 신규 엔티티로부터 생성한 실제 DDL(검증 증거). 구 orphan_iso_quarantine 대비 +resource_type, +payload, os_metadata_id→parent_id, description/client_hash 컬럼은 payload(JSON)로 흡수.

# Hibernate generated DDL (실측)
create table orphan_quarantine (
    id                bigint not null auto_increment,
    created_at        datetime(6) not null,        # BaseTimeEntity
    updated_at        datetime(6) not null,
    recovery_id       varchar(36)  not null,        # unique 키
    resource_type     enum(BIOS_BUNDLE,BMC_FIRMWARE,OS_ISO,SUBPROGRAM,…) not null  # 신규 (SPI 라우팅)
    parent_id         bigint not null,              # 구 os_metadata_id 일반화
    resolved_path     varchar(1024) not null,
    quarantine_path   varchar(1024),                # null = in-place / 격리실패 degraded
    original_filename varchar(512) not null,
    register_existing bit not null,
    payload           varchar(2048)                 # 신규 (도메인 재시도 데이터 JSON, Id.CLASS)
    failure_class     enum(DB_CONSTRAINT,MARKER_WRITE,STORAGE_IO,UNEXPECTED) not null,
    exception_detail  varchar(2048),
    state             enum(DISCARDED,PENDING,RECOVERED) not null,
    retry_count       integer not null,
    job_id            varchar(64),
    primary key (id)
) engine=InnoDB;
create unique index ux_orphan_recovery_id on orphan_quarantine (recovery_id);
create index ix_orphan_state_created on orphan_quarantine (state, created_at);
CP5 주의 : 구 orphan_iso_quarantine 테이블은 dev DB 에 잔존(미사용). PENDING 격리가 없으면 drop 가능(별도 chore 커밋). 신규 테이블은 DDL 권한 보유 계정 부팅 시 자동 생성.
