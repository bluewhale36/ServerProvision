자원 무결성 점검(Reconciliation) — 전체 인프라 안내서

2026-07-11 KST 기준 · 지금 무엇이 만들어져 있고, 어떤 어긋남(드리프트)이 왜 생기며, 시스템이 각각을 어떻게 다루는지 한 문서로 정리

① 등장물 4가지② 점검이 하는 일③ 드리프트 지도(조합표)④ 10종 카탈로그 — 발생 원인⑤ 해결 체계⑥ 관리 도구⑦ 구축 현황

한 문장 요약 — 이 시스템은 서버가 기억하는 것(DB)과 디스크에 실제로 있는 것(파일·마커·휴지통)을 주기적으로 대조해서, 어긋난 곳을 드리프트라는 이름의 카드로 보여주고, 가능한 것은 시스템이 직접 바로잡아 주는 자가 점검·복구 장치입니다.

① 등장물은 딱 4가지

드리프트 10종이 복잡해 보이지만, 전부 이 4가지가 서로 어긋나는 조합일 뿐입니다.

🗄️DB 기록

서버가 기억하는 것 — 자원이 어디에 있고(경로), 살아있는지/삭제됐는지, 내용 지문(hash)이 무엇인지.

📀자원 파일 (본체)

ISO·BIOS·BMC·드라이버의 실제 파일. 디스크 위 실물.

🏷️마커 (신분증)

등록할 때 파일 옆에 발급되는 .provision.json. "이 파일은 몇 번 자원"인지 + 내용 지문 + 위조 방지 서명이 들어 있음. 파일이 어디로 가든 마커만 있으면 신원 확인 가능.

🗑️휴지통

삭제해도 바로 파괴하지 않고 .soft-deleted/ 폴더에 격리 보관. DB 에 "휴지통 어디에 뒀는지" 기록이 남고, 기한 내 복원 가능.

② 점검이 하는 일

버튼을 누르거나(수동) 매시간 자동으로 돌며, 다음 순서로 대조합니다.

1제자리 확인

등록된 모든 자원마다 "기록된 자리에 마커가 있나 → 서명이 진짜인가 → 본체 파일이 있나"를 차례로 확인.

2수색

제자리에 없으면 자원 폴더들을 뒤져 같은 신분(자원 번호)의 마커를 찾는다 — 찾으면 "이사 갔구나", 못 찾으면 "사라졌구나".

3주인 찾기

거꾸로, 디스크에서 발견됐는데 DB 에 주인이 없는 마커를 가려낸다. 삭제된 자원의 것이면 별도 분류.

4휴지통 대조

삭제된 자원마다 "휴지통에 잘 있나 / 밖으로 나왔나 / 사라졌나"를 대조 — 다섯 상태(정상/복귀/이탈/소실/유령)를 한 번에 판정.

③ 드리프트 지도 — "무엇이 있고 무엇이 없나" 조합표

머릿속 정리용 핵심 표입니다. 자원이 사용 중이냐 삭제됨(휴지통)이냐로 크게 나뉘고, 그 안에서 4 등장물의 있고/없음 조합이 곧 드리프트 종류입니다.

한 가지 의도된 예외 — 삭제 자원의 휴지통 파일이 멀쩡히 있는데 원래 자리에 다른 파일이 있는 경우(자리 재사용)는 드리프트로 보고하지 않습니다. 같은 자리에 새 자원을 올리는 건 정상 업무 흐름이라, 그걸 문제라고 알리면 오탐이 되기 때문입니다. 그 파일의 진위는 나중에 복원을 시도하는 순간 검증됩니다.

🎬 이 조합들을 눈으로 확인하려면 — 드리프트 상황 시뮬레이터(report/26-07-11_17-57-19_drift-simulator_guide.html)에서 상황 버튼을 눌러 직접 만들어 보고, 점검·[적용]까지 체험할 수 있습니다.

④ 10종 카탈로그 — 언제, 왜 생기나

각 카드의 "이럴 때 생깁니다"가 발생 원인입니다. 배지 색 = 해결 방식(⑤ 참조).

경로 이동됨지금 동작

이럴 때 생김 :
  • 디스크 정리·폴더 재구성을 하며 자원 파일을 (마커와 함께) 다른 폴더로 옮겼다
  • 스토리지를 갈아타며 파일들을 통째로 이사시켰다
발견 방법 : 제자리에 마커가 없어 수색해 보니, 같은 신분의 마커+본체가 다른 폴더에서 온전히 발견됨.
해결 : [적용] → DB 의 경로만 새 위치로 고쳐 적음. 파일은 안 건드림. 원하면 점검 때 무인 자동 처리도 가능.

자원 소실지금 동작

이럴 때 생김 :
  • 파일만 지워졌다 (마커는 제자리에 남음)
  • 파일 이름이 바뀌었다 — 마커 입장에선 본체가 사라진 것과 같음
  • 마커까지 통째로 지워지거나, 마커만 어딘가로 옮겨졌다
  • 디스크 마운트가 빠지는 등 폴더 전체가 안 보인다 (이때는 "일부 폴더를 읽지 못함" 경고도 함께 표시)
발견 방법 : 제자리에도 없고 수색해도 안 나옴. 마커만 남은 경우는 매시간 점검이 즉시 잡음(S6-1 개선 — 종전엔 하루 뒤 정밀 점검에서야 드러났음).
해결 : 어느 파일이 진짜인지 시스템이 알 수 없어 자동 복구 없음 — 백업·이동·이름 변경 여부를 확인해 원래 자리·이름으로 되돌린 뒤 재점검.

미등록 마커지금 동작

이럴 때 생김 :
  • 백업해 뒀던 자원 파일(마커 포함)을 자원 폴더에 도로 복사했다
  • 다른 서버에서 쓰던 파일을 그대로 가져왔다
  • DB 를 초기화·재구축해서 파일만 남고 기록이 사라졌다
발견 방법 : 디스크에서 발견된 마커의 자원 번호가 DB 에 없음.
해결 : 등록할지 버릴지는 운영자 판단 — 필요하면 관리 화면에서 재등록, 아니면 파일 정리 후 보고 닫기. (파일 정리·재등록을 화면에서 바로 하게 하는 액션은 S6-3 후보)

마커 서명 불일치지금 동작

이럴 때 생김 :
  • 서버의 서명 비밀 키를 교체했다 — 교체 직후엔 모든 자원이 일제히 이렇게 보임 (정상 현상)
  • 누군가 마커 파일을 열어 손댔거나, 파일이 손상됐다
  • 다른 서버(다른 키)에서 만든 마커를 가져왔다
발견 방법 : 마커의 위조 방지 서명이 현재 키로 검증되지 않음.
해결 : 키 교체 직후라면 관리 도구의 [마커 서명 재발급] 한 번으로 전체 해결. 아니라면 변조 가능성 — 파일 경위 확인. 시스템이 자동으로 덮어쓰지 않는 이유: 변조를 정상화해 버리면 안 되기 때문.

내용 변경 감지지금 동작

이럴 때 생김 :
  • 같은 이름으로 다른 파일을 덮어썼다 (실수 재업로드, 악의적 교체)
  • 파일이 부분 손상됐다
발견 방법 : 정밀 점검에서만 — 파일 내용을 다시 읽어 계산한 지문이 등록 때와 다름. (마커 재발급을 해도 내용 지문은 유지되므로, 변조를 은폐할 수 없게 설계되어 있음)
해결 : 원본 재업로드 또는 변경 경위 확인. 어느 쪽이 정본인지 시스템이 판단하지 않음. (변경을 "의도된 것"으로 받아들이는 액션은 S6-3 검토 대상)

유령 DB 기록지금 동작

이럴 때 생김 :
  • 삭제 처리 당시 파일이 이미 없어서 기록만 "삭제됨" 표시된 자원이 그대로 남았다
  • 과거 수동 정리 과정에서 파일·휴지통은 지워지고 DB 한 줄만 남았다
발견 방법 : 삭제 표시인데 휴지통 기록도, 휴지통 파일도, 원위치 파일도 전부 없음.
해결 : [적용] → 남은 기록 한 줄 삭제. 복구할 실물이 애초에 없어 잃을 것이 없는 안전한 정리 — 그래서 자동 후보.

삭제 자원 복귀S6-2-2 예정

이럴 때 생김 :
  • 휴지통 폴더에서 파일을 손으로 꺼내 원래 자리에 되돌려 놓았다 (정식 복원 버튼 대신)
  • 기록만 삭제됐던 자원(위 유령의 사촌)의 파일이 백업 복사 등으로 원위치에 다시 나타났다
발견 방법 : 삭제 표시인데 원래 자리에 파일이 살아 있음.
해결 : 되돌려 놓았다는 건 다시 쓰겠다는 뜻 — [적용] 하면 시스템이 정식 복원 절차(삭제 해제 + 휴지통 기록 정리 + 마커 재발급)를 대신 밟아 줌. 해석이 하나뿐이라 자동 후보.

삭제 자원 위치 이탈S6-2-2 예정

이럴 때 생김 :
  • 백업을 복원하다가 삭제했던 자원의 사본(마커 포함)이 엉뚱한 폴더에 풀렸다
  • 누군가 휴지통에서 꺼내 아무 데나 두었다
발견 방법 : 삭제 자원의 마커가 사용 중 폴더 어딘가에서 발견됨. (종전에는 점검이 이걸 보고도 조용히 넘어갔음 — 완전 침묵 구간)
해결 : 의도를 알 수 없어 자동으로 움직이지 않음 — 내가 [적용]으로 확인해 주면 그때 휴지통으로 도로 회수. 단, 휴지통에 기존 사본이 살아 있으면 회수를 거절하고 안내(그대로 회수하면 기존 사본이 미아가 되므로).

휴지통 자원 소실S6-2-3 예정

이럴 때 생김 :
  • 디스크 청소 스크립트·용량 정리가 휴지통 폴더까지 지웠다
  • 누군가 휴지통 파일을 직접 삭제했다
발견 방법 : 휴지통 기록이 가리키는 자리에 파일이 없고, 다른 어디에서도 발견되지 않음 — 복원이 불가능해진 상태.
해결 : 복구 불능 확정은 되돌릴 수 없는 판단이라 자동 금지 — 내가 [적용]으로 확인하면 남은 기록을 정리하고 감사 기록(언제·무엇이·왜)을 남김.

잔여 마커 정리 필요S6-2-3 예정

이럴 때 생김 :
  • 삭제 처리 때 마커 정리 단계가 디스크 오류 등으로 실패했다 — 시스템이 경고만 남기고 넘어간 찌꺼기
발견 방법 : 휴지통 파일 옆에 마커가 남아 있음.
해결 : [적용] → 마커 파일만 삭제. 자원·기록은 안 건드리는 무해한 청소라 자동 후보.

⑤ 해결 체계 — 세 등급과 안전장치

등급의미해당 종류
주황 — 시스템 해결해석이 하나뿐이라 시스템이 바로잡을 수 있음. [적용] 버튼 + 원하면 점검 때 무인 자동 처리(설정으로 종류별 선택, 기본은 전부 꺼짐)경로 이동 · 유령 기록 · 삭제 자원 복귀 · 잔여 마커
노랑 — 확인 후 해결위험하거나 의도가 갈리는 상황 — 사람이 [적용]으로 확인해 줘야만 시스템이 실행. 무인 자동은 불가위치 이탈(회수) · 휴지통 소실(기록 정리) · 서명 불일치(단일 재서명) · 미등록 마커(격리 회수) · 내용 변경(자원명 확인 후 수용 — 백그라운드+감사 기록)
회색 — 안내만어느 쪽이 정본인지 시스템이 판정할 수 없음 — 원인 후보와 할 일을 안내, 직접 조치 후 재점검자원 소실 1종 (파일을 만들어낼 수는 없으므로 — 단 [다시 점검]으로 복원 확인 가능. 소실·서명·미등록 3종에 확인 버튼 제공)
  • 마스터 스위치 — 설정 하나로 시스템 해결 전체(버튼 포함)를 끌 수 있음(점검·보고만 하는 관찰 모드). 꺼져 있으면 버튼이 비활성화되고 이유가 말풍선으로 표시됨.
  • 확인 절차 — [적용]·[보고 닫기]는 항상 확인 창을 거치고, 결과는 화면 이동 없이 알림으로 반영. 거절되면(이미 처리된 카드를 다시 누른 경우 등) 이유가 안내창으로 표시.
  • 확인 축 — 해결과 별개로, 조치를 마친 뒤 [다시 점검]을 누르면 그 자원 하나만 즉시 재확인해 해소 시 카드가 그 자리에서 사라진다(자원 소실·서명 불일치·미등록 마커). 내용 변경은 [현재 내용을 정본으로 수용](자원명 확인·감사 기록) 제공 — 원본 재업로드가 1차 권장.

⑥ 점검 화면 밖의 관리 도구

마커 서명 재발급서명 키를 교체한 뒤 한 번 실행 — 모든 자원의 마커를 새 키로 다시 서명해 "서명 불일치" 일제 오탐을 막음. 내용 지문은 유지되므로 변조된 파일은 여전히 정밀 점검에 걸림. 실패 건수까지 완료 알림에 표시되고, 실패가 있으면 자동으로 후속 점검이 이어져 실패 자원이 보고서 카드로 드러남(R9-6).
업로드 실패 복구업로드 도중 실패로 어중간하게 남은 파일을 격리해 두고 재시도/정리하는 별도 화면 — 점검과는 독립된 흐름 (navbar "업로드 실패 복구").
점검 보고서 이력매 점검의 결과가 보고서로 저장(최근 100건). 화면 상단에 최신 결과 요약, 카드로 남은 드리프트 목록, 일부 폴더를 못 읽었으면 그 사실도 표시.
