PART A

Restore FS-DB 무결성 + raw 에러 (심각 버그)

A1. 두 버그 정의 — 사용자 재현 시나리오

1. OS 버전 soft-delete → 자식 ISO 도 cascade soft-delete ISO: is_deleted=true, trashed_path=.soft-deleted/OS_ISO/<id>/dvd_….iso, 파일은 trash 로 이동 2. "삭제된 항목 보기" → OS 버전만 복구 (cascade=false) 3. ISO 단독 복구 클릭 → "페이지는 넘어가나 실제 복원 안 됨, 삭제항목 체크 사라짐" 4. 삭제항목 다시 체크 → ISO 재복구 → ★ 브라우저에 raw 에러 + "휴지통에서 복원된 것으로 보임" 5. 사용자 진단: FS 의 .soft-deleted/OS_ISO/<id>/ 는 비어있고 ISO 는 원위치 복구됨. 그러나 DB 는 is_deleted=true, trashed_path=유효경로 (stale). 렌더링은 DB 기준 → 여전히 삭제로 보임.
버그정의
버그 1 FS-DB 무결성partial-restore 로 FS 는 복구됐으나 DB 는 stale → "dangling stale row" 양산
버그 2 raw 에러재시도 시 RestoreTrashLostException 메시지가 가공 없이 화면 노출. R1-4-1 hotfix 가 못 막음

A2. 버그1 근본원인 — FS↔DB 작업 순서 (적대적 검증 완료)

TrashLifecycleService.restoreFromTrash 의 실행 순서 (try/catch 0건 확인):

L137 trashService.moveBack(trashedPath, originalPath) ← FS, 비가역 L147 computeSignature(content) L148 markerService.write(originalPath, SIDECAR, …) ← FS, 가장 유력한 예외원 L149 entity.reissueMarker(hash, sig) ← DB L151 entity.restore() ← DB L152 entity.clearTrashed() ← DB 전체가 caller @Transactional (IsoLifecycleService:144) 안.
고정 비대칭의 메커니즘
moveBack(L137) 성공 후 어느 단계든 RuntimeException (가장 유력 = MarkerWriteFailedException, ProvisionMarkerService:111-112) → FS 는 이미 원위치 복구 (트랜잭션 관할 밖, 롤백 불가) + DB 는 @Transactional 롤백 으로 is_deleted=true + trashed_path=stale 유효경로 그대로. FS-비가역-우선 + DB-나중 순서라 partial-failure 시 항상 FS복구·DB-stale 방향으로 고정.
결정적 비대칭 (두 버그 공통 핵심) — trashed_path==null 분기 (L114-120) 는 이미 "FS 존재 → DB만 정합" self-heal 을 보유. 그러나 trashed_path!=null 분기 (L123-152) 에는 그 사전검사가 없음. 이 비대칭이 버그1·2 의 공통 진원지.
정정 (적대적 검증) — 이 stale row 는 GhostEvaluator.isGhost 의 4조건(trashed_path=null 포함)을 불만족 → 기존 ghost 처리(MK3-1)가 못 잡는 "빈 trash 를 가리키는 dangling stale row". (스카우트 가설 "manifestHash null → NPE" 는 코드상 반박됨 — null 도 정상 HMAC.)

A3. 버그2 근본원인 — raw 에러 전파 (중대 정정 포함)

두 직교 원인의 합:

  1. (a) self-heal 부재 — restoreFromTrash:126-128 이 재시도 시 trash 빈 상태(Files.exists(trashedPath)=false)에서 originalPath 존재 여부를 선검사하지 않고 곧장 RestoreTrashLostException(404) throw.
  2. (b) message 무가공 — error.html:24-28 의 th:text="${message}" 가 도메인 예외 message 를 그대로 출력.
중대 정정 — 사용자가 본 것은 "raw JSON 페이지"가 아니라 native nav 로 렌더된 error.html(스타일된 404 페이지) 본문에 노출된 도메인 메시지다.
OS list 의 ISO restore form 에 data-async-submit 마커 부재(trash list 엔 존재) → form.requestSubmit() native POST(Accept:text/html) → WebExceptionHandler.handleNotFound → error.html 렌더.

R1-4-1 hotfix v3 가 못 막은 이유

v3 변경은 @Order 통일 + 주석뿐 (content-type routing 만 손댐). (a)self-heal 부재 / (b)message 무가공 / (c)OS list restore form 의 async-submit 비대칭 — 셋 다 안 건드림. 또 RestoreTrashLost 만 404 이고 형제 3개(RestorePathOccupied/RestoreTargetUnreachable/GhostRowRestoreNotAllowed)는 모두 409 — 계층 내부 불일치.

A4. 횡단 영향 — 4 도메인 공통 + 별도 결함 1건

두 버그는 4 도메인 공통 — 진원지 TrashLifecycleService.restoreFromTrash/softDeleteToTrash 는 ISO(IsoLifecycleService:163) / BIOS(BiosService:444) / BMC(BmcService:504) / Subprogram(SubprogramService:565) 이 모두 위임 호출하는 공통 인프라. 네 자식 restore 전부 @Transactional → ISO 에서 보고된 버그가 코드 동일성으로 4 도메인 전부 존재.
→ TrashLifecycleService 1곳 + 예외 1줄 + restore form 마커 수정으로 4 도메인 동시 해결.
별도 발견 (직교 결함, partial-failure 계열 아님) — BoardModelService 의 cascade 가 Subprogram 을 전혀 안 다룸 (softDelete/restore/purge 가 BIOS/BMC 만 위임). board-scoped Subprogram 은 부모 Board soft-delete 시 active 잔류 → 부모 deleted / 자식 active 불일치 + orphan Subprogram. → 본 슬라이스 비목표로 분리 권장 (아래 결정 A-3).

A6. ⬢ 결정 사항 — Part A

결정 A-0 · 완료

설계 방향 — FS-first + 역보상 채택, CP1 진입 ✓

이미 동의받음. judge 권장 hybrid (멱등 사전게이트 + FS-first 역보상 + 404→409 + async-submit 마커) 채택. afterCommit 2-phase 무결성은 별도 후속 슬라이스로 분리.

결정 A-1

RestoreTrashLostException 의 status 를 404 → 409 로 변경?

현재 RestoreTrashLost 만 NotFoundException(404), 형제 3개는 ConflictException(409). 계층 일관성 회복 + GhostRow 의미 합치. advice 가 polymorphic 흡수라 신규 핸들러 0.

권장 ✓409 로 변경 — 형제 일관성 회복. 단 기존 테스트의 status assert 동반 갱신 필요(OSMetadataControllerRestoreFlowTest 404→409)
대안404 유지 — 외부 API 계약 의존이 있을 경우만
결정 A-2

멱등 (원O·trashX) self-heal 시 사용자 안내?

FS 는 이미 복구됐고 DB만 정합하는 경우, 사용자에게 어떻게 보일지.

권장 ✓조용히 DB 정합 후 성공(2xx) — 매끄러운 흐름. 사용자는 "복구됨"만 인지
대안"이미 복구되어 있어 동기화했습니다" 토스트 노출 — 인지 강화
결정 A-3

BoardModel→Subprogram cascade 누락 (별도 결함) 을 본 슬라이스에 포함?

partial-failure 계열이 아닌 cascade 범위 누락. 부모 Board 삭제 시 Subprogram orphan.

권장 ✓별도 cascade-asymmetry 슬라이스로 분리 + Notion 본문에 잔존 책임 기재. 본 버그(restore FS-DB)와 결이 다름
대안본 슬라이스에 포함 — scope 비대 위험
