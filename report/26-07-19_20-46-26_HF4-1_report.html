1. 요약 — 무엇을 고쳤나

휴지통에서 자원 하나의 보존기간을 뒤로 미는 "+30일 연장" 버튼은 겉보기에 한 클릭이지만, 2026-07-19 전수 점검에서 이 한 클릭이 세 결함을 연쇄로 노출했다. HF4-1 은 그 세 결함(F-1·F-2·F-3)과, 연장이라는 동작의 의미 자체를 바꾸기로 한 사용자 결정(O-4)을 한 슬라이스로 함께 처리했다. 서로 얽혀 있어 따로 고치면 반쪽짜리가 되기 때문이다 — 예컨대 오류 모달의 원문 덤프(F-3)는 서버가 500 을 내던 원인(F-1)을 없애야 재현 자체가 사라진다.

2. 유스케이스 전후 (AS-IS 결함 흐름 → TO-BE)

운영자가 실제로 겪는 흐름을 두 종류의 자원으로 나눠 본다. 휴지통에는 파일 자원(ISO·BIOS·BMC·Subprogram — 디스크에 실물 파일이 있는 자원)과 메타 자원(OS 버전·보드 모델 — 파일 없이 메타데이터만 있는 자원)이 섞여 나열된다. 연장이 의미를 갖는 것은 파일 자원뿐인데, 고치기 전에는 그 구분이 화면에도 서버에도 없었다.

유스케이스 A 메타 자원 행에서 연장을 눌렀을 때
AS-IS (고치기 전)
  1. OS 버전을 삭제해 휴지통으로 보낸다.
  2. 그 행의 "+30일 연장" 버튼이 활성 상태로 보인다 — 눌린다.
  3. 확인 모달이 뜨는데 제목이 엉뚱하게 "Deprecated 표시"다(F-2).
  4. 확정하면 서버가 500 을 던진다. 원인은 SPI 기본 메서드가 던지는 UnsupportedOperationException 이 예외 라우팅(advice) 어디에도 매핑되지 않아 "unhandled" 로 새기 때문이다(F-1).
  5. 화면에는 F-3 로 HTML 에러 페이지 원문이 그대로 덤프된 모달이 뜬다. 보존기간은 연장되지 않았다.
TO-BE (고친 후)
  1. OS 버전을 삭제해 휴지통으로 보낸다.
  2. 그 행의 연장 버튼은 disabled 이고, 마우스를 올리면 tooltip "이 자원 종류는 보존기간 연장을 지원하지 않아요." 가 뜬다 — 애초에 눌리지 않는다.
  3. 혹시 direct POST 로 우회하면 서버 가드가 TtlExtensionUnsupportedException 을 던지고, 이 예외는 ConflictException 하위라 기존 advice 가 자동으로 409 로 변환한다.
  4. Accept 헤더에 따라 JSON({"message":"OS 버전 자원은 보존기간 연장을 지원하지 않습니다."}) 또는 SSR 에러 뷰로 협상 응답한다. 500 도, 원문 덤프도 없다.

핵심은 화면의 disabled 조건과 서버의 거절 조건이 같은 판정 하나(MarkableTrashOperator.supportsTrashTtlExtension())를 함께 호출한다는 점이다. 두 곳에 조건을 복붙하면 언젠가 어긋나(drift) silent 사고가 재발하므로, 판정을 SPI(Service Provider Interface — 도메인이 구현해 끼우는 확장점)에 하나만 둔다.

유스케이스 B 파일 자원의 보존기간을 연장할 때 (연장의 의미)
AS-IS — "재기준"

연장은 해당 자원의 trashed_at 을 현재 시각으로 다시 맞추는 방식이었다. 만료일은 trashed_at + TTL 로 계산되므로 결과적으로 +TTL 일 미뤄지긴 하지만, 부작용으로 "휴지통 이동 시각" 표시까지 오늘로 바뀌어 언제 버렸는지 정보가 사라진다. 삭제 직후 연장하면 티가 안 나지만, 만료 임박에 연장하면 실제 잔여일과 크게 어긋난다.

TO-BE — "가산"

연장 1회 = 만료일에 +N일(N = 운영 설정 TTL 일수) 누적. 새 컬럼 ttl_extension_days 에 더해 나가고, trashed_at 은 건드리지 않는다. 만료 = trashed_at + TTL + ttl_extension_days일. 이동 시각 표시는 정확하게 유지되고, 두 번 연장하면 +2N 일이 그대로 쌓인다. 복원하거나 영구삭제되면 누적은 0 으로 초기화된다.

TTL 자체를 컬럼에 굳히지 않고 "이동 시각 + 운영 설정 TTL + 누적 연장"으로 매번 계산하므로, 운영자가 전역 TTL 을 30일에서 45일로 바꾸면 이미 휴지통에 있는 행들의 만료도 함께 움직인다(현행 동적 의미 보존). 절대 만료일을 저장했다면 잃었을 성질이다.

3. 설계 결정 (D1~D6 + plan 대비 이탈 5건)

구현은 CP1 plan 의 결정 D1~D6 을 그대로 따랐다. 아래는 각 결정의 채택안과, 구현 중 plan 문구에서 벗어난 5개 지점의 사유다. 이탈 5건은 검수에서 전부 수용됐다.

D1. 연장 지원 여부의 SSOT — 채택: SPI 기본 메서드

MarkableTrashOperator(SPI)에 default boolean supportsTrashTtlExtension() { return false; } 를 두고, 파일 자원 4 스캐너(Iso·BoardBios·BoardBmc·Subprogram)만 true 로 override 했다. 화면 뷰모델의 버튼 disabled 플래그와 서버 가드가 이 판정 하나를 함께 호출한다. 대안이던 "서비스에 지원 타입 Set 상수"는 SPI 밖에 두 번째 진실 원천을 만들어 drift 위험이 있어 탈락, "메타 자원도 연장 구현"은 cascade 의미(자식 파일 자원 TTL 동반 연장?)가 미확정이라 과설계로 탈락(§6 후속 항목).

D2. 가산 저장 방식 — 채택: ttlExtensionDays 누적 컬럼

LifecycleEntity(@MappedSuperclass) 에 int ttlExtensionDays = 0 을 추가하고 만료를 trashedAt + TTL + ttlExtensionDays일 로 계산한다. 이동 시각 불변 + 전역 TTL 변경의 동적 반영을 동시에 만족한다. "재기준(trashed_at 밀기)"은 사용자 결정과 불일치 + 이동 시각 왜곡으로 탈락, "절대 만료일 저장"은 전역 TTL 변경이 기존 행에 반영 안 돼 탈락.

D3. 연장 step 일수 — 채택: 운영 설정 TTL 일수

연장 1회의 가산량과 버튼 라벨("+N일 연장")이 모두 운영 설정 TTL 일수를 쓴다. 고정 30 은 설정을 45일로 바꾸면 라벨·동작이 설정과 어긋나 탈락.

D4. 자동 영구삭제(worker) 만료 판정 — 채택: 넓은 후보 조회 + Java 재검증

worker 는 기존 findTrashedBefore(now - TTL) 로 후보를 넓게 조회한 뒤, entity 만료 SSOT(trashExpiresAt)로 Java 에서 재검증해 연장으로 아직 만료되지 않은 행을 제외한다. "SQL 날짜 연산으로 한 방 쿼리"는 4개 repository 에 방언 의존 date 연산이 중복되고 JPQL 이식성이 떨어져 탈락 — 후보 초과 조회량은 만료 임박 행뿐이라 비용이 무시할 만하다.

D5. 오류 응답 협상 (F-3) — 채택: 서버 예외 계층 흡수 + 클라이언트 비 JSON 방어

서버는 신설 예외가 ConflictException 하위라 기존 이중 advice(WebExceptionHandler/ApiExceptionHandler)가 JSON/HTML 협상까지 자동 처리한다(advice 무수정). 클라이언트는 confirm-modal-base.js 의 submitAsync 가 비 JSON 응답 바디를 message 로 쓰지 않도록 고쳐, 원문은 콘솔에만 남기고 화면엔 일반화 문구로 fallback 한다. "클라이언트만 고치고 서버 500 방치"는 로그에 unhandled ERROR 가 남아 모니터링을 오염시켜 탈락.

D6. 연장 확인 모달 (F-2) — 채택: 전용 모달 타입 신설

ConfirmModalType.EXTEND_TTL 상수(자원 lookup 불요 — SOFT_DELETE 패턴) + confirm-extend.html fragment + confirm-extend.js(마커 data-confirm-extend)를 신설했다. "deprecate 모달에 라벨 파라미터 추가"는 S5-2 모달 자산의 "액션별 책임 분리" 원칙을 깨고 deprecate 라벨의 두 번째 소유자를 만들어 탈락.

5. 스키마 변경 · 수동 DDL 절차

가산 방식(D2)은 라이프사이클 6 테이블에 누적 컬럼 ttl_extension_days 하나를 추가한다. 메타 자원 2 테이블(os_metadata·board_model)은 연장을 지원하지 않아 이 컬럼이 항상 0 이지만, 컬럼을 통일해 두면 향후 메타 자원 연장을 구현하게 될 때 재-ALTER 를 하지 않아도 된다.

산출 파일: sql/HF4-1_ttl_extension_days.sql

-- LifecycleEntity(@MappedSuperclass) 상속 6 테이블 전부에 ttl_extension_days 추가.
--   만료 SSOT = trashed_at + TTL + ttl_extension_days일 (LifecycleEntity.trashExpiresAt)
ALTER TABLE os_metadata ADD COLUMN ttl_extension_days INT NOT NULL DEFAULT 0;
ALTER TABLE iso         ADD COLUMN ttl_extension_days INT NOT NULL DEFAULT 0;
ALTER TABLE board_model ADD COLUMN ttl_extension_days INT NOT NULL DEFAULT 0;
ALTER TABLE board_bios  ADD COLUMN ttl_extension_days INT NOT NULL DEFAULT 0;
ALTER TABLE board_bmc   ADD COLUMN ttl_extension_days INT NOT NULL DEFAULT 0;
ALTER TABLE subprogram  ADD COLUMN ttl_extension_days INT NOT NULL DEFAULT 0;
실 DB 적용 절차(수동). 실 운영 프로파일은 ddl-auto=validate 라, 이 DDL 을 적용하기 전에는 앱이 기동되지 않는다(스키마 검증 단계에서 missing column [ttl_extension_days] 로 중단). 또한 claude_code 계정은 ALTER 권한이 없으므로 ALTER 권한 계정(root 등)으로 실행해야 한다.
# 적용
mysql -u root -p server_provision < sql/HF4-1_ttl_extension_days.sql

# 확인 — 6 테이블 모두 동일 컬럼 존재
SHOW CREATE TABLE iso;   -- ttl_extension_days INT NOT NULL DEFAULT 0

샌드박스 검증은 ddl-auto=update 로 이 6 컬럼을 자동 생성한 상태에서 수행했다(§4-2). 실 DB 반영은 위 수동 절차가 선행돼야 한다.

6. 잔여 · 후속

이번 slice 는 F-1·F-2·F-3·O-4 로 명시된 범위만 처리했다. 관련 주변에서 관찰됐으나 이 slice 밖으로 둔 항목을 그대로 기록한다.

#항목상태 / 다음
잔여-1 TTL 원천 이원화. 화면 표시·연장 step 은 TrashPolicy(trash.ttl.days 프로퍼티)를, worker 의 만료 판정은 TrashSettingsService(trash_settings 테이블)를 각각 참조한다. 이번 slice 는 두 경로 모두 entity 만료 SSOT(trashExpiresAt)를 쓰도록만 통일했고, TTL 값의 원천 이원화 자체는 기존 상태 유지(scope 밖).
잔여-2 임박 알림의 창 조회 기준. notifyUpcomingExpiration 은 여전히 trashed_at 기준으로 알림 창을 조회한다 — 연장된 행에 대해 이르게(실제 만료 전에) 알림이 발송될 수 있다. 후속 백로그. worker 의 purge 판정은 이번에 가산분을 반영했지만(D4), 알림 경로는 별도 조정이 필요.
잔여-3 실 DB DDL 미적용. sql/HF4-1_ttl_extension_days.sql 을 관리자 계정으로 수동 적용하기 전에는 실 DB 기동 불가. 운영 반영 시점에 ALTER 권한 계정으로 §5 절차 실행. (claude_code 는 ALTER 불가)
후속-1 메타 자원 연장 지원 여부. D1-c 로 보류 — 지원한다면 cascade 의미(자식 파일 자원 TTL 동반 연장)를 먼저 정의해야 한다. 필요 판단 시 신규 백로그로 논의 개시.
