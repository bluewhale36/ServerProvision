00

한눈에 요약

이 보고서는 물리적으로 연결된 한 대의 게스트서버가 시스템 안에서 어떤 모양으로 저장되어야 하는지를 정합니다. 게스트서버는 입고 직후의 익명의 박스에서 시작해, iPXE 등록 → 진단 리눅스 보강 → 5단계 프로비저닝을 거쳐 운영 상태에 도달합니다. 각 단계마다 우리가 아는 정보가 늘어나고, 진행 상태가 갱신되며, 자격증명이 발급됩니다.

이 시간 흐름을 단일 테이블에 몰아넣으면 절반의 컬럼이 항상 NULL이고, 자격증명 한 컬럼 때문에 전체 행이 민감 데이터가 됩니다. 대신 정체성·네트워크·위치·자격증명·진행 상태·이력을 8개 테이블로 분리합니다.

핵심 결정 한 줄

guest_server는 변하지 않는 신원의 앵커이고, 그 외 테이블들은 시간이 흐르며 채워지는 맥락이다. 비밀번호는 DB에 저장하지 않고 Vault 참조만 둔다.

01

풀어야 할 문제

한 대의 게스트서버는 시간에 따라 정보가 늘어난다

같은 물리 서버여도, 우리가 그 서버에 대해 아는 정보는 시점마다 다릅니다. 각 시점의 정보 출처도 다릅니다.

정보 출처별 분류

① iPXE 등록 단계 (boot.ipxe) — SMBIOS UUID, 시스템 시리얼, 제조사, 모델명, LAN1 MAC. 박스가 PXE로 부팅되자마자 보고하는 가장 빠른 정보.

② 진단 리눅스 단계 — 보드 시리얼, BMC MAC, BMC IP, BMC 자격증명, NIC 인벤토리. dmidecode, ipmitool 같은 Linux 도구로 수집.

③ 운영자 입력 — 데이터센터, 랙 위치, 서버 역할(role). 사람이 결정해서 시스템에 알려주는 영역.

④ 프로비저닝 엔진 — Phase 0 + 5단계의 현재 상태, 각 step의 성공/실패/스킵, 전이 이력. 시스템 자체가 만들어내는 정보.

그래서 단일 테이블은 답이 아니다

만약 이 모든 정보를 한 테이블 guest_server에 몰아넣으면 다음 문제가 동시에 생깁니다.

희소성 문제: iPXE 등록만 끝난 서버는 boardSerial / bmcMac / bmcIp / rackId 가 모두 NULL인 행이 됩니다. 절반 이상이 비어 있는 테이블은 인덱스 효율과 가독성을 동시에 깎아먹습니다.

보안 응집 문제: BMC 자격증명 한 컬럼 때문에 테이블 전체에 데이터-앳-레스트 암호화·접근 제어를 걸어야 합니다. 정체성만 조회하는 일상 쿼리도 같은 보안 부담을 집니다.

락 경합 문제: 진행 상태는 자주 갱신되는데, 신원 컬럼들과 같은 행을 공유하면 낙관적 락 충돌이 잦아집니다. 진행 상태 갱신 트랜잭션이 신원 갱신 트랜잭션을 차단하는 일이 발생합니다.

02

설계 원칙

원칙 1 — 시점이 다른 정보는 테이블이 다르다

같이 채워지고, 같이 갱신되고, 같이 사라지는 정보만 같은 테이블에 둡니다. iPXE 단계 정보와 진단 리눅스 단계 정보는 채워지는 시점이 다르므로 분리합니다.

원칙 2 — 자격증명은 DB에 두지 않는다

평문은 물론, 암호화한 비밀번호조차 저장하지 않습니다. DB에는 Vault 참조 경로만 두고, 실제 자격증명은 Vault에서 조회합니다. DB 백업이 유출되어도 BMC에는 접근할 수 없게 만드는 게 목표입니다.

원칙 3 — Primitive obsession 회피, 다형성으로 분기

DB는 단순히 코드 문자열을 보관하지만, 어플리케이션 레이어에서는 sealed interface와 record로 타입을 만듭니다. vendor가 GIGABYTE인지 Supermicro인지는 DB에서는 문자열 하나지만, Java 21에서는 BmcVendor sealed interface의 구현체로 변환되어 다형성 디스패치를 통해 분기됩니다. switch-case가 등장하면 그건 분기가 아니라 타입 디스패치입니다.

원칙 4 — 상태 전이는 데이터로, 검증은 타입으로

provisioning_progress.current_phase는 코드 문자열, phase_meta는 JSON. 어플리케이션이 둘을 합쳐 적절한 ProvisioningPhase record로 복원하고, sealed interface가 허용되지 않는 전이를 컴파일 타임에 차단합니다.

원칙 5 — 이력은 append-only, 본체는 가변

provisioning_transition_log는 한 번 쓰면 수정·삭제하지 않습니다. 본체 테이블의 현재 상태와 별개로, 감사·트러블슈팅·SLA 분석용 원장입니다.

03

테이블 한눈에 보기

8개 테이블이 한 게스트서버를 표현합니다. 각 카드를 누르면 해당 테이블의 상세 설명으로 이동합니다.

guest_server
정체성의 앵커
SMBIOS UUID·시스템 시리얼·보드 시리얼·제조사·모델 등 변하지 않는 신원 정보. 다른 모든 테이블이 이 테이블을 참조합니다.
host_nic_binding
호스트 네트워크 신원
LAN1의 MAC과 IP, 호스트네임. OS가 부팅된 뒤의 호스트 네트워크 정보가 들어갑니다.
bmc_binding
BMC 신원
MGMT 포트 쪽 BMC의 MAC·IP·벤더·펌웨어 버전. 호스트와 완전히 분리된 별개의 네트워크 노드.
bmc_credential_ref
Vault 참조
BMC 자격증명은 DB에 없습니다. Vault 경로와 사용자명, 회전 시각만 보관합니다.
location_assignment
물리적 위치
데이터센터·랙·U 위치와 역할. "어디에 꽂혀서 무엇을 한다"를 표현합니다.
provisioning_progress
큰 단계 진행 상태
현재 Phase가 무엇인지 (Bootstrapping, FirmwareUpdating 등). 1:1로 게스트서버에 묶입니다.
setup_step
세부 체크포인트
한 Phase 안의 개별 작업들 (RAID 구성, OS 설치, NTP 설정 등). 성공·실패·스킵을 기록합니다.
provisioning_transition_log
전이 이력 원장
언제 어떤 Phase에서 어떤 Phase로 옮겨갔는지를 append-only로 기록. 감사·디버깅용.
04

각 테이블 상세

각 테이블 헤더를 클릭하면 컬럼 목록과 DDL이 펼쳐집니다. PK / FK / UK / NN 표기는 호버하면 의미가 뜹니다.

guest_server 정체성의 앵커
▶

한 게스트서버를 시스템에 묶어주는 앵커 테이블. 이 테이블의 행 하나가 곧 한 대의 물리 서버입니다. SMBIOS UUID와 보드 시리얼이 핵심 식별자이고, 다른 모든 테이블은 이 테이블의 id를 FK로 참조합니다.

discovery_stage 컬럼이 흥미로운 부분입니다. iPXE만 끝났는지(IPXE_REGISTERED), 진단 리눅스까지 마쳤는지(DIAGNOSTIC_ENRICHED)를 표현합니다. 이는 프로비저닝 단계와는 다른 차원입니다 — "우리가 이 서버에 대해 얼마나 아는가"를 가리킵니다.

컬럼타입제약설명
idBINARY(16)PKUUID v7. 시간 정렬이 가능하고 외부 노출이 안전합니다.
system_uuidCHAR(36)UKNNSMBIOS Type 1 UUID. iPXE 단계에서 무조건 확보되는 가장 빠른 식별자.
system_serialVARCHAR(128)SMBIOS Type 1 시스템 시리얼. iPXE 단계에서 수집.
manufacturerVARCHAR(128)GIGABYTE, Supermicro 등. vendor 분기의 1차 단서.
product_nameVARCHAR(128)메인보드 모델 (예: MZ72-HB0). BIOS 프로파일 매핑의 키.
board_serialVARCHAR(128)UKSMBIOS Type 2. 진단 리눅스 단계에서만 채워집니다. NULL 허용.
discovery_stageVARCHAR(32)NN우리가 이 서버에 대해 얼마나 아는지: IPXE_REGISTERED 또는 DIAGNOSTIC_ENRICHED.
first_seen_atTIMESTAMP(3)NNiPXE에 처음 등장한 시각. 평생 변하지 않음.
last_seen_atTIMESTAMP(3)NN가장 최근 부팅·체크인 시각. 자동 갱신.
versionBIGINTNNJPA 낙관적 락. 동시 갱신 충돌 방지.
 
CREATE TABLE guest_server (
    id              BINARY(16)   PRIMARY KEY,
    system_uuid     CHAR(36)     NOT NULL,
    system_serial   VARCHAR(128),
    manufacturer    VARCHAR(128),
    product_name    VARCHAR(128),
    board_serial    VARCHAR(128),
    discovery_stage VARCHAR(32)  NOT NULL,
    first_seen_at   TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    last_seen_at    TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3)
                       ON UPDATE CURRENT_TIMESTAMP(3),
    version         BIGINT       NOT NULL DEFAULT 0,
    CONSTRAINT uk_guest_server_uuid  UNIQUE (system_uuid),
    CONSTRAINT uk_guest_server_board UNIQUE (board_serial),
    INDEX idx_guest_server_stage     (discovery_stage),
    INDEX idx_guest_server_product   (manufacturer, product_name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci;
host_nic_binding 호스트 네트워크 신원
▶

OS가 부팅된 뒤 LAN1 NIC이 가지는 네트워크 정보. 호스트 측 IP는 BMC IP와 완전히 다른 네트워크에 속합니다. 같은 박스 안의 다른 두 칩이 각자 IP를 받는 구조라, 별도 테이블로 두는 게 자연스럽습니다.

컬럼타입제약설명
idBINARY(16)PKUUID v7.
guest_server_idBINARY(16)FKUKNNguest_server를 1:1로 참조. 한 서버에 호스트 NIC 바인딩은 하나.
host_macCHAR(17)UKNN정규화된 형태 aa:bb:cc:dd:ee:ff. 어플리케이션의 MacAddress value object가 정규화 보장.
lan1_ipVARCHAR(45)IPv6까지 표현 가능. OS 설치 전엔 NULL.
ip_sourceVARCHAR(16)NNDHCP, STATIC, RESERVED 중 하나. sealed interface와 매핑.
hostnameVARCHAR(253)FQDN까지 담을 수 있게 253자 (DNS 최대 길이).
bound_atTIMESTAMP(3)NN바인딩이 최초 생성된 시각.
bmc_binding BMC 신원
▶

MGMT 포트 쪽의 BMC가 가지는 네트워크 정보. BMC는 PSU에 전원이 들어오는 순간 부팅되므로 호스트 OS보다 먼저 IP를 가집니다. vendor 컬럼이 어플리케이션의 BmcVendor sealed interface와 매핑되어, Redfish 경로 차이를 다형성으로 처리합니다.

컬럼타입제약설명
idBINARY(16)PKUUID v7.
guest_server_idBINARY(16)FKUKNNguest_server 1:1.
bmc_macCHAR(17)UKNN호스트 MAC과 다른 별개 NIC. AMI / ASPEED OUI로 시작.
bmc_ipVARCHAR(45)관리망 IP. DHCP 또는 Static.
ip_sourceVARCHAR(16)NN호스트와 같지만 별개로 관리.
vendorVARCHAR(32)NNGIGABYTE, SUPERMICRO, DELL. Redfish 경로 분기의 키.
fw_versionVARCHAR(64)BMC 펌웨어 버전. 펌웨어 업데이트 단계 후 갱신.
credential_ref_idBINARY(16)FKbmc_credential_ref 참조. 자격증명 확보 전엔 NULL.
bmc_credential_ref Vault 참조 (DB에 비밀번호 없음)
▶
보안 핵심

이 테이블에는 비밀번호가 없습니다. 평문도, 암호화한 값도 없습니다. Vault 경로 (vault_path)와 사용자명, 마지막 회전 시각만 보관합니다. DB 백업이 유출되어도 BMC 접근이 불가능하도록 만드는 게 이 테이블의 존재 이유입니다.

컬럼타입제약설명
idBINARY(16)PKUUID v7.
vault_pathVARCHAR(255)UKNN예: secret/bmc/srv01-bmc. 실제 비밀번호는 이 경로로 Vault에 요청.
usernameVARCHAR(64)NNBMC 사용자명 (보통 admin 또는 별도 자동화 계정).
rotated_atTIMESTAMP(3)NN마지막 비밀번호 회전 시각. 회전 주기 관리용.
location_assignment 물리적 위치와 역할
▶

"이 서버가 어느 데이터센터의 어느 랙 어느 U에 꽂혀 있고 어떤 역할을 한다"를 표현합니다. uk_location_slot 제약이 핵심으로, 같은 데이터센터의 같은 랙 같은 U에 두 대가 꽂힐 수 없음을 DB 레벨에서 강제합니다 — 이건 실제 물리 제약이라 어플리케이션이 아니라 DB가 보장하는 게 안전합니다.

컬럼타입제약설명
idBINARY(16)PKUUID v7.
guest_server_idBINARY(16)FKUKNN한 서버에 위치 정보는 하나.
datacenterVARCHAR(64)NN데이터센터 코드 (예: seoul-dc1).
rack_idVARCHAR(64)NN랙 식별자 (예: rack-A-03).
rack_unitVARCHAR(8)U 위치 (예: U21). 미지정 허용.
roleVARCHAR(64)서버 역할 (예: web, db). Kickstart 분기에 사용.
provisioning_progress 큰 단계 진행 상태
▶

현재 게스트서버가 어느 Phase에 있는지를 한 행으로 표현합니다. current_phase는 코드 문자열, phase_meta는 JSON으로 해당 Phase record의 추가 필드를 직렬화합니다. 어플리케이션의 ProvisioningPhase sealed interface가 둘을 합쳐 적절한 record로 복원합니다.

왜 guest_server와 분리했나: 진행 상태는 자주 갱신되지만 신원 정보는 거의 불변입니다. 둘이 같은 행을 공유하면 낙관적 락 충돌이 잦아집니다. 분리하면 진행 상태 갱신이 신원 갱신을 막지 않습니다.

컬럼타입제약설명
idBINARY(16)PKUUID v7.
guest_server_idBINARY(16)FKUKNN1:1 매핑. 한 서버에 진행 상태는 하나.
current_phaseVARCHAR(32)NNPhase 코드. BOOTSTRAPPING, FIRMWARE_UPDATING, OS_INSTALLING 등.
phase_metaJSONJSONPhase record의 추가 필드. 예: OsInstalling이면 {"kickstartProfile":"web-rocky9","raidLevel":"RAID10"}.
last_transition_atTIMESTAMP(3)NN자동 갱신. 가장 최근 전이 시각.
setup_step 세부 체크포인트
▶

한 Phase 안에 여러 step이 있을 수 있습니다. 예를 들어 OS_INSTALLING Phase 안에는 RAID_CONFIGURE와 KICKSTART_INSTALL 두 step이 순서대로 들어갑니다. 각 step의 성공·실패·스킵을 기록하면 "BIOS는 끝났는데 RAID에서 막혀있는 서버" 같은 상세 상태를 표현할 수 있고, 진행률 계산도 자연스럽습니다.

컬럼타입제약설명
idBINARY(16)PKUUID v7.
guest_server_idBINARY(16)FKNNguest_server 1:N.
phase_codeVARCHAR(32)NN소속 Phase 코드.
step_codeVARCHAR(64)NNStep 식별 코드. RAID_CONFIGURE, KICKSTART_INSTALL 등.
definition_orderINTNN같은 Phase 내 실행 순서.
statusVARCHAR(16)NNPENDING, RUNNING, SUCCEEDED, FAILED, SKIPPED.

게스트서버가 거치는 큰 단계는 Phase 0 (Bootstrap)부터 운영 진입까지 총 7개입니다. 각 셀을 클릭하면 해당 Phase의 의미와 이 DB에서 어떻게 표현되는지가 아래 패널에 나타납니다.

discovery_stage와 current_phase는 다른 차원

두 컬럼이 비슷해 보일 수 있지만 의미가 다릅니다.

discovery_stage는 "이 게스트서버에 대해 시스템이 얼마나 알고 있는가"를 나타냅니다. iPXE로 등록만 됐는지, 진단 리눅스까지 떴는지의 차원입니다.

current_phase는 "프로비저닝이 어디까지 진행됐는가"를 나타냅니다. Bootstrap이 끝났는지, OS 설치 중인지, 운영 중인지의 차원입니다.

진단 리눅스를 띄우지 못한 채 등록만 된 서버도 존재할 수 있어야 합니다 — 그래서 두 컬럼을 따로 둡니다. 한 차원을 다른 차원으로 추론할 수 없는 경우가 실제로 생깁니다.

06

관계와 카디널리티

8개 테이블 사이의 관계는 단순합니다. guest_server가 중심이고, 나머지는 모두 guest_server.id를 FK로 참조합니다.

1:1 관계는 별도 테이블로 둘 이유가 있어야 한다

host_nic_binding, bmc_binding, location_assignment, provisioning_progress 네 개가 guest_server와 1:1입니다. "한 행이면 같은 테이블에 합치면 되지 않나"라는 질문이 자연스럽지만, 분리한 이유는 셋입니다.

채워지는 시점이 다르다 — iPXE 직후엔 bmc_binding이 없습니다. NULL 컬럼이 줄줄이 늘어서는 대신, 행 자체를 안 만드는 게 깔끔합니다.

락 단위가 다르다 — provisioning_progress는 자주 갱신되고, guest_server의 신원 정보는 거의 불변입니다. 같은 행에 있으면 둘이 서로의 트랜잭션을 막습니다.

보안 단위가 다르다 — credential_ref 참조를 가진 bmc_binding은 접근 통제 대상이지만, guest_server 자체는 모든 운영자가 읽어야 하는 정보입니다. 분리해야 권한을 다르게 줄 수 있습니다.

07

설계 결정 Q&A

설계 과정에서 자주 나왔던 질문들과 답을 정리합니다. 각 질문을 클릭하면 답이 펼쳐집니다.

왜 BIGINT auto-increment가 아니라 BINARY(16) UUID인가?+

두 가지 이유입니다. 첫째, auto-increment ID는 외부에 노출되면 "총 몇 대 등록됐는지", "어느 순서로 들어왔는지"가 추측 가능해 정보 누출입니다. 둘째, 분산 환경에서 ID를 미리 생성하기 어렵습니다 (DB에 INSERT가 일어나야 ID가 결정됨). UUID v7은 시간 정렬이 가능한 UUID라 인덱스 효율도 BIGINT에 근접합니다.

왜 board_serial이 nullable인데 UNIQUE인가? 모순 아닌가?+

모순 아닙니다. MariaDB의 UNIQUE 제약은 NULL 중복을 허용합니다 — 즉 NULL은 여러 개 존재할 수 있지만, NULL이 아닌 값은 유일해야 합니다. iPXE 단계에서는 board_serial을 못 얻으므로 NULL이지만, 진단 리눅스가 채운 뒤로는 절대 중복되면 안 됩니다. 이 동작이 정확히 우리가 원하는 것입니다.

왜 BMC 비밀번호를 암호화해서라도 DB에 두지 않나?+

암호화 키와 암호화된 데이터가 같은 환경에 있으면 결국 둘이 동시에 노출됩니다. 어플리케이션 인스턴스가 DB도 읽고 키도 갖고 있으면, 어플리케이션 권한이 탈취되는 순간 두 개를 모두 잃습니다. Vault는 별도 신뢰 경계를 가진 시스템이라, 어플리케이션이 매번 인증해서 잠시 자격증명을 받아 쓰는 구조가 됩니다. DB 백업이 통째로 유출되어도 BMC 비밀번호는 안 나옵니다.

setup_step의 phase 정의는 DB에 마스터 테이블로 둬야 하지 않나?+

둘 다 가능합니다. 어플리케이션 sealed interface로 두면 다형성 원칙에 부합하고 컴파일 타임 안전성을 얻습니다. DB 마스터 테이블로 두면 운영자가 step 순서를 런타임에 조정할 수 있습니다. 현재 설계는 전자를 따릅니다 — Phase와 Step의 정의는 사실상 비즈니스 로직이지 데이터가 아닙니다. 사용자께서 운영적 조정 요구가 있다고 알려주시면 마스터 테이블로 분리하는 마이그레이션을 추가할 수 있습니다.

phase_meta·status_meta·transition_meta에 왜 JSON을 쓰나? 정규화해야 하지 않나?+

세 JSON 컬럼이 담는 데이터는 모두 sealed interface record의 가변적 추가 필드입니다. OsInstalling record는 kickstartProfile을, BiosConfiguring record는 profileName을 가지는데, 이걸 모두 컬럼화하면 phase마다 NULL이 줄줄이 늘어섭니다. JSON 컬럼으로 검색을 하지 않고 보관만 한다면, JSON 직렬화가 정규화보다 깔끔합니다. JSON 조건 검색이 필요한 패턴은 의도적으로 피합니다 — 그건 NoSQL이 더 잘 합니다.

진행 상태 이력은 왜 별도 테이블로 두나? provisioning_progress에 컬럼 더 추가하면?+

provisioning_progress는 현재 상태를 표현하고, transition_log는 변경의 역사를 표현합니다. 본체에 컬럼을 더 추가해서 이력을 적으면, 행이 무한히 커지거나 덮어쓰기가 일어납니다. append-only 별도 테이블은 디스크 사용 패턴(시간순 INSERT), 인덱스 패턴 (시간 범위 스캔), 권한 패턴(audit 권한)이 본체와 완전히 다르므로 분리가 맞습니다.

호스트가 LAN1 외에 LAN2까지 본딩으로 쓰면? 1:1 host_nic_binding으로 표현되나?+

현재 설계는 1:1입니다. 본딩이 운영 요구사항으로 들어오면 host_nic_binding을 1:N으로 풀고 is_primary·bond_group_id 같은 컬럼을 추가하는 마이그레이션이 필요합니다. 이 부분은 사용자께 확인이 필요한 미확정 사항입니다.

08

미확정 사항

다음 두 가지는 사용자님의 결정이 필요한 항목입니다. 결정에 따라 마이그레이션 한두 개가 추가되거나 기존 구조가 약간 수정될 수 있습니다.

미결 ①

host_nic_binding을 1:N으로 풀어둘지 — LAN2 본딩이나 추가 NIC을 OS 세팅 단계에서 쓰는 경우가 운영적으로 발생하면 1:N이 필요합니다. 지금은 1:1로 시작하되 향후 확장 여지를 두고 싶다면, is_primary 플래그를 추가한 1:N으로 미리 풀어두는 게 마이그레이션 비용을 줄입니다.

미결 ②

setup_step의 정의를 DB로 끌어올릴지 — 현재 설계는 Phase/Step 정의를 어플리케이션 코드(sealed interface)에 둡니다. 다형성 원칙에 부합하고 컴파일 타임 안전합니다. 하지만 운영자가 step 순서를 런타임에 조정해야 하는 케이스가 있다면 setup_step_definition 마스터 테이블이 추가로 필요합니다.

다음 작업 제안

설계가 확정되면 다음 순서로 진행할 수 있습니다.

① Flyway 마이그레이션 6개 파일 작성 — V1부터 V6까지 위 DDL을 그대로 옮기면 됩니다.
