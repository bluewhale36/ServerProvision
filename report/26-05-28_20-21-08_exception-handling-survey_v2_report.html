1. 왜 이 보고서를 따로 쓰는가

지난 며칠간 예외 처리 한 가지가 R1-4-1 hotfix 단계를 세 번 가로막았다 :

  • v1 @Order(HIGHEST_PRECEDENCE - 100) 산술 underflow → 의도와 정반대 우선순위
  • v2 annotations 속성으로 advice 스코프 좁힘 → hybrid controller 28건 회귀
  • v2' @Order(HIGHEST_PRECEDENCE + 100) 양수 offset → Accept 미명시 MockMvc 28건 회귀
  • v3 양 advice 모두 HIGHEST_PRECEDENCE + ContentNegotiation 위임 → 안정 (현재)

반복된 회귀의 원인은 "두 개의 advice 를 어떻게 협조시킬까" 라는 출발점 자체가 Spring 권장 모델과 어긋났기 때문이다. 그러나 단순 "Spring 권고대로" 가 아니라 본 프로젝트의 CLAUDE.md 철학과 실제 코드 스타일에 자연스럽게 흡수시키는 모양이 필요하다.

본 보고서의 목적 — 두 단계 :
  1. 먼저 현재 코드의 스타일 을 §3 에서 정리 (대부분 옳은 방향).
  2. 그 위에 4 권위 출처의 권고를 스타일을 깨지 않고 흡수 하는 형태로 §8 에 매핑.
v2 보정 v1 산출 직후 java-spring-senior-engineer 에이전트 검토에서 4 항목 보정 도출 :
  1. §3.1, §3.5 — 보안 sub-class 수 (7 → 11) 및 frontend 일관성 평가 실측 재기술
  2. §8 변경 2 — "frontend 무변경" 단정 보정 (content-type 매처 1줄 수정 필요 + 도메인 응답 record 재배치)
  3. §8 변경 3 — default errorCode() 식 SCREAMING_SNAKE 변환 (PascalCase 그대로면 frontend 분기 silent 깨짐)
  4. §10 함정 추가 4건 — BindException / HttpMessageNotReadableException / multipart 비-Size / AsyncSubmitResult defer 협조
모든 보정은 본문 해당 위치에 REV-v2 마커 + 보정 카드로 표시. v1 의 원문은 직선으로 인용되지 않으며, 보정된 사실 기반 진술로 갱신.

2. 예외 처리의 두 축 — 라우팅과 본문 형식

축결정 내용현 시스템 답변
① 라우팅 "어떤 핸들러 메서드가 이 예외를 잡을 것인가?"
→ 예외 타입 + Accept 헤더 (produces) 매칭		 두 개 advice (WebExceptionHandler + ApiExceptionHandler) 분리
② 본문 "JSON / HTML 응답 필드 스키마는 무엇인가?"
→ 응답 DTO + view model		 자체 정의 ApiErrorResponse(message, fieldErrors) + Thymeleaf error.html
비유 호텔 컨시어지로 비유하자.
  • 축 ① = "한국어 손님은 한국인 직원, 영어 손님은 영어 직원" — 라우팅 규칙.
  • 축 ② = "어느 직원이 응대하든 답변 양식은 같은 문서 템플릿" — 응답 표준.
현 코드는 ① 을 위해 두 직원(advice)을 두고 손짓 약속(@Order) 으로 순서를 정하려 했다. Spring 권장은 "한 직원이 두 언어를 다 소화하되 호출 받은 언어에 맞춰 답변" 하는 것이다 (§4).

3. 현 코드의 스타일 진단 REV-v2

본 프로젝트의 CLAUDE.md 는 코딩 철학에 매우 구체적이다. 본 보고서와 관련된 두 불가침 원칙 :

CLAUDE.md 인용 조건 분기문의 legacy 무분별 확장 절대 지양 (불가침). 신규 케이스 / 신규 예외 / 신규 도메인 타입이 추가될 때 분기문에 줄을 늘리는 방식으로 처리하지 않는다. 대신 Java 의 다형성 (interface / abstract class / sealed type / enum / strategy pattern) 과 Spring Framework primitive (@ControllerAdvice / HandlerExceptionResolver / @ResponseStatus / BindingResult 자동 매핑 / AOP / @EventListener / MessageSource / Converter) 를 적극 활용해 framework 측이 분기를 떠맡도록 설계한다.

3.1 보안 예외 다형성 — 가장 성숙한 패턴

SecurityException 은 abstract super-class 이고, sub-class 가 httpStatus() 를 다형적으로 구현한다 : 

public abstract class SecurityException extends RuntimeException {
    public abstract HttpStatus httpStatus();   // 다형성으로 status 결정
}
v2 보정 — 정량 수치 갱신 v1 은 보안 sub-class 를 "7 개" 로 표기했으나 실측 결과 11 개 다 : 
global/security/exception/ 11 files :
  ForbiddenException                   → 403
  PathTraversalException               → 400
  PathOutsideAllowedRootsException     → 400
  EntrypointInvalidException           → 400
  UnsupportedMediaTypeException        → 415
  UploadLimitExceededException         → 413
  ZipBombSuspectedException            → 415
  ZipBombInspectionFailedException     → 500
  MaliciousContentSuspectedException   → status 별도 매핑
  ExecutableContentRejectedException   → status 별도 매핑
  SuspiciousFilenameException          → status 별도 매핑
결론은 오히려 강화 — 단일 핸들러 handleSecurity(SecurityException) 가 11 sub-class 의 분기 없이 폴리모픽 흡수 중. CLAUDE.md "다형성으로 framework 가 분기를 떠맡도록" 원칙의 가장 성숙한 사례.

3.2 @ResponseStatus 어노테이션 자동 흡수 — Spring primitive 적극 활용

도메인 예외 sub-class 에 @ResponseStatus 만 붙이면 advice 의 fallback 핸들러가 AnnotationUtils.findAnnotation 으로 자동 매핑 :

@ResponseStatus(HttpStatus.CONFLICT)
public class ChildLifecycleBlockedByParentException extends ConflictException { ... }

@ResponseStatus(HttpStatus.BAD_REQUEST)
public class TypedNameMismatchException extends DomainException { ... }

@ResponseStatus(HttpStatus.SERVICE_UNAVAILABLE)
public class PurgeIoFailedException extends DomainException { ... }
스타일 평가 Spring primitive 의 모범 활용. 신규 도메인 예외 추가 시 sub-class + 어노테이션 한 줄로 끝. advice 코드 0줄 추가. Spring 공식[1] 권장 그대로.

3.3 도메인 super-class 계층 — 의미 단위로 분류

스타일 평가 AIP-193[4] 의 "표준 에러 코드 적은 수 + 자원 많은 수" 원칙과 일치.

3.4 보안 예외 계층 분리 — 사고 회피의 학습

"컨트롤러의 catch (DomainException) 가 보안 예외를 흡수해 500 으로 새는 사고 (S3.3 silent-500 회귀) 를 차단하기 위함이다." — global/security/exception/SecurityException.java 주석

4. 출처 ① Spring 공식 — 단일 advice + dual produces

Spring Framework 공식 reference 의 "Exceptions"[1] 가 동일 예외 타입의 응답 분기에 명시한 패턴 :

"In addition to exception types, @ExceptionHandler methods can also declare producible media types. This allows to refine error responses depending on the media types requested by HTTP clients, typically in the Accept HTTP request header." — Spring Framework Reference, Media Type Mapping[1] 
@ExceptionHandler(produces = "application/json")
public ResponseEntity<ErrorMessage> handleJson(IllegalArgumentException exc) { ... }

@ExceptionHandler(produces = "text/html")
public String handleHtml(IllegalArgumentException exc, Model model) { ... }

같은 advice 안의 두 메서드. Spring 공식은 "중복 정의가 아니다" 라고 명시.

비교 항목두 개 advice (현재)단일 + dual produces (Spring 권장)
매칭 순위@Order overflow 위험produces vs Accept 호환성 한 가지로 결정
Hybrid controllerannotations 좁힘 시 깨짐스코프 좁힐 필요 자체가 없음
새 예외 추가양쪽 advice 에 핸들러 추가한 advice 의 produces 다른 메서드 두 개

5. 출처 ② RFC 9457 Problem Details

IETF 가 2023년 발행[3]. RFC 7807 후속 + 호환.

"This document defines a 'problem detail' to carry machine-readable details of errors in HTTP response content to avoid the need to define new error response formats for HTTP APIs." — RFC 9457, §1[3]
필드역할
type문제 유형 URI
title사람이 읽는 짧은 제목
statusHTTP 상태 사본
detail이번 발생의 구체 메시지
instance이번 발생 URI (trace 등)
"Problem type definitions MAY extend the problem details object with additional members that are specific to that problem type." — RFC 9457, §3.2[3]
"When serialized in a JSON document, that format is identified with the 'application/problem+json' media type." — RFC 9457, §3[3]
"Truly generic problems are usually better expressed as plain status codes. For example, a 'write access disallowed' problem is probably unnecessary, since a 403 Forbidden status code in response to a PUT request is self-explanatory." — RFC 9457, §4.2[3]

6. 출처 ③ Google AIP-193

Google 사내 API 강제 가이드라인[4].

8. 우리 스타일에 권고 흡수 — 4 변경 REV-v2

즉시 채택

변경 1 — advice 통합 : WebExceptionHandler → ApiExceptionHandler 흡수

현 코드
@ControllerAdvice
@Order(HIGHEST_PRECEDENCE)
class WebExceptionHandler {
    @ExceptionHandler(NotFoundException.class,
                     produces = TEXT_HTML_VALUE)
    String handleNotFound(...)  // view
}

@RestControllerAdvice
@Order(HIGHEST_PRECEDENCE)
class ApiExceptionHandler {
    @ExceptionHandler(NotFoundException.class,
                     produces = APPLICATION_JSON_VALUE)
    ResponseEntity... handleNotFound(...)
}
→
매핑 후
@ControllerAdvice    // @RestControllerAdvice 아님
class GlobalExceptionHandler {

    @ExceptionHandler(value = NotFoundException.class,
                     produces = TEXT_HTML_VALUE)
    String handleNotFoundHtml(...)

    @ExceptionHandler(value = NotFoundException.class,
                     produces = APPLICATION_JSON_VALUE)
    @ResponseBody
    ResponseEntity... handleNotFoundJson(...)
}

스타일 보존 검증

  • §3.1 보안 다형성 — 그대로. handleSecurity(SecurityException) 한 메서드가 11 sub-class 다형 흡수
  • §3.2 @ResponseStatus — 그대로
  • §3.3 도메인 super-class 계층 — 그대로. NotFound / Conflict / FieldBound 각 super 에 produces 짝만 추가
  • §3.4 보안 계층 분리 — 그대로
v2 보정 — 작업량 산정 수정 v1 은 "2~3 시간" 으로 산정. 실측 :
  • 현 @RestControllerAdvice 의 ApiExceptionHandler 안 핸들러 메서드 10+ 개 모두에 명시적 @ResponseBody 추가 필요 (@ControllerAdvice 로 어노테이션 변경 시 자동 @ResponseBody 가 사라짐)
  • 각 핸들러의 응답 시그니처 재검토 (ResponseEntity 반환은 그대로 작동하나 String 반환 추가 시 view name 으로 해석되는지 확인 필요)
  • 매처 우선순위 — 같은 advice 안의 produces 다중 메서드는 메서드 선언 순서 가 silent 우선순위 결정. HTML / JSON 어느 쪽을 먼저 둘지 명시 결정 필요
  • Spring Boot 4.x 의 spring.mvc.problemdetails.enabled autoconfig 가 켜져 있으면 default ProblemDetail 핸들러와 충돌 가능 — 검토 필요
수정 산정 : 반나절~1일 (반나절 + 회귀 안전망 + 매처 순서 결정 + autoconfig 검토)

해소되는 함정

  • @Order 산술 overflow 함정 영구 제거
  • 같은 advice 의 produces 다중은 Spring 이 ambiguity 명시 거부
  • Hybrid controller 의 모든 메서드 동일 advice 매처 적용

RFC 9457 흡수 후
{
  "type": "https://sp.local/errors/iso-conflict",
  "title": "ISO 등록 충돌",
  "status": 409,
  "detail": "동일 경로 ISO 가 있어요.",
  "errorCode": "ISO_PATH_DUPLICATE",
  "message": "동일 경로 ISO 가 있어요.",    // frontend 호환
  "fieldErrors": [...]                          // 호환 보존
}
// Content-Type: application/problem+json
v2 보정 — "frontend 무변경" 단정 수정 v1 은 변경 2 가 "frontend 무변경 점진 도입 가능" 이라 단정. 실측 :

(1) frontend content-type 매처 1줄 수정 필요

현 코드의 fetch 응답 처리는 다음과 같이 substring 매처를 사용 :

// confirm-modal-base.js:164 등 다수
const ct = resp.headers.get('content-type') || '';
if (ct.includes('application/json')) { /* JSON 파싱 */ }

그러나 ProblemDetail 의 Content-Type 은 application/problem+json. includes('application/json') 은 substring 매치 안 됨 (problem+json 이 중간에 들어가지 않음). 다음 중 한 가지 수정 :

// 옵션 A — 두 케이스 명시
if (ct.includes('application/json') || ct.includes('application/problem+json')) { ... }

// 옵션 B — suffix 매처
if (ct.match(/json$/) || ct.includes('+json')) { ... }

옵션 B 권장 — 향후 다른 +json media type (e.g. application/ld+json) 도 자연 수용.

(2) DeleteRejectResponse / NudgeRequiredResponse 스키마 재배치 작업

이 두 record 는 ApiErrorResponse 가 아닌 별도 응답 타입. ProblemDetail 통합 시 두 가지 방안 :

  • 유지 (권장) — ProblemDetail 와 별개로 두 record 유지. 각자의 도메인 응답 의미가 강함 (예: DeleteRejectResponse 의 token + ttlSec + actions[] 는 follow-up action 식별자라 표준화의 의미가 약함). 단 두 record 의 code 필드는 errorCode 와 의미 중복 — 점진 정렬
  • 흡수 — ProblemDetail extension members 로 흡수. 단점 — frontend 의 body.code === 'SOFTDELETE_REQUIRES_INTENT' 매처 (delete-reject-modal.js:68) 가 깨짐. 동반 frontend 수정 필요

도입 슬라이스에서 결정.

수정 산정 : 한 슬라이스 (CP1~CP5) — frontend 매처 수정 1줄 + advice 응답 타입 변경 + record 결정 + 회귀 테스트

Spring 6+ 의 ProblemDetail 활용

@ExceptionHandler(value = ConflictException.class, produces = APPLICATION_PROBLEM_JSON_VALUE)
@ResponseBody
public ProblemDetail handleConflictJson(ConflictException ex) {
    ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.CONFLICT, ex.getMessage());
    pd.setTitle("리소스 충돌");
    pd.setProperty("errorCode", ex.errorCode());   // 변경 3 의 메서드
    pd.setProperty("message", ex.getMessage());    // frontend 호환
    return pd;
}
장기 채택 (점진)

변경 3 — errorCode() default 메서드 REV-v2

AIP-193 의 머신 가독 식별자 권고[4] 를 SecurityException.httpStatus() 패턴 흉내로 도입.

v2 보정 — default 식 SCREAMING_SNAKE 변환 v1 의 default 식 : 
public String errorCode() {
    return getClass().getSimpleName().replaceAll("Exception$", "");
    // "DuplicateISOContentException" → "DuplicateISOContent"  ← PascalCase
}
문제 — frontend 의 기존 매처는 body.code === 'SOFTDELETE_REQUIRES_INTENT' (delete-reject-modal.js:68) 처럼 SCREAMING_SNAKE_CASE 를 기대. DeleteRejectResponse.CODE = "SOFTDELETE_REQUIRES_INTENT" 가 그 값을 frontend 에 전달. v1 의 default 식이 그대로 노출되면 frontend 분기가 silent 깨짐 (PascalCase vs SCREAMING_SNAKE 불일치).

v2 수정 default 식 :

public String errorCode() {
    String simple = getClass().getSimpleName().replaceAll("Exception$", "");
    // PascalCase → SCREAMING_SNAKE_CASE
    return simple
        .replaceAll("([a-z])([A-Z])", "$1_$2")
        .replaceAll("([A-Z]+)([A-Z][a-z])", "$1_$2")
        .toUpperCase();
    // "DuplicateISOContent"  → "DUPLICATE_ISO_CONTENT"
    // "SoftDeleteRequiresIntent" → "SOFT_DELETE_REQUIRES_INTENT" (frontend 기대 SOFTDELETE_REQUIRES_INTENT 와 다름)
}

잔여 위험 — frontend 의 기존 매처 SOFTDELETE_REQUIRES_INTENT (한 단어) 와 default 식 출력 SOFT_DELETE_REQUIRES_INTENT (분리) 가 어긋남. 두 가지 해결 :

  1. 해당 sub-class 가 errorCode() override 해서 명시 — @Override public String errorCode() { return "SOFTDELETE_REQUIRES_INTENT"; }. 권장 — 기존 frontend 호환 + 신규 예외는 default 식으로 자동 도출
  2. frontend 매처를 SOFT_DELETE_REQUIRES_INTENT 로 변경. frontend 동반 수정 → 변경 3 도입 슬라이스가 변경 2 와 묶임

도입 순서 권고 : 변경 2 와 변경 3 을 한 슬라이스로 묶어 ProblemDetail content-type 매처 수정 + errorCode 도입 + frontend 매처 정렬을 한 번에. v1 의 "장기 점진" 분류는 변경 2 와 동기화 로 수정.

왜 enum class 가 아니고 메서드인가

본 프로젝트 도메인 예외 100+ sub-class. 별도 enum 으로 옮기면 새 예외 추가 시 두 곳을 동기화 — "분기문 / 매핑 테이블 확장 회피" 원칙에 어긋난다. 예외 클래스 자체가 errorCode 의 source of truth.

선택적 채택

변경 4 — transient 실패에 Retry-After 헤더 추가 REV-v2

Microsoft 권고[5] 적용 :

v2 보정 — SSR / JSON 양쪽 적용 명시 v1 은 JSON variant 만 예시. 실측 — WebExceptionHandler:72-84 의 SSR variant 도 동일 처리 필요.

두 advice 모두 수정 (변경 1 통합 후엔 한 곳) :

@ExceptionHandler(value = OptimisticLockingFailureException.class, produces = APPLICATION_JSON_VALUE)
@ResponseBody
public ResponseEntity<ApiErrorResponse> handleOptimisticLockJson(OptimisticLockingFailureException ex) {
    return ResponseEntity.status(HttpStatus.CONFLICT)

}

@ExceptionHandler(value = OptimisticLockingFailureException.class, produces = TEXT_HTML_VALUE)
public String handleOptimisticLockHtml(..., HttpServletResponse response, Model model) {
    response.setHeader(HttpHeaders.RETRY_AFTER, "2");   // + SSR variant 도 동일
    response.setStatus(HttpStatus.CONFLICT.value());
    ...
    return "error";
}

10. 함정 — v1 의 5건 + v2 추가 4건 REV-v2

10.1 @Order(HIGHEST_PRECEDENCE ± N) 산술 overflow

Ordered.HIGHEST_PRECEDENCE = Integer.MIN_VALUE. MIN_VALUE - 100 → underflow → 양수 영역. 의도 정반대.

회피 : @Order 값에 산술 자체 금지. 단일 advice 면 @Order 자체 불필요.

10.2 annotations 으로 hybrid controller 가 빠짐

@ControllerAdvice(annotations = RestController.class) 좁힘 → @Controller + @ResponseBody hybrid 가 advice 대상에서 제외.

회피 : annotations 좁히지 말고 단일 advice + produces 매처.

10.3 MockMvc 기본 Accept: */*

mockMvc.perform(post(...)) 만이면 */* → 양 advice 매처 호환 → 등록 순서 의존.

회피 : XHR 시나리오 .accept(MediaType.APPLICATION_JSON) 명시, SSR 시나리오 .accept(MediaType.TEXT_HTML, MediaType.ALL).

10.4 frontend fetch 의 Accept 기본값

fetch 기본 Accept 가 */* → 매처 우선순위 영향.

회피 : XHR 호출은 항상 Accept: application/json 명시.

10.5 "raw JSON 화면 노출" 의 원인 분류

  1. SSR form submit 이 application/json 응답 받음 — advice 매처 분리 실패 (§10.1, §10.2)
  2. XHR fetch 가 페이지로 navigate — form 의 onsubmit 가 preventDefault 누락
  3. error-modal 로딩 실패 — defer 순서 / CSS 미로드
v2 추가

10.6 HttpMessageNotReadableException 미커버

@RequestBody JSON malformed (예: IsoLifecycleController:112 의 @Valid @RequestBody DeleteIntentRequest) 진입에서 발생. 두 advice 모두 핸들러 없음 → Spring 기본 DefaultHandlerExceptionResolver 가 400 fallback 응답. 응답 body 는 Spring autoconfig 의 default 라 frontend 의 payload.message 와 다른 필드.

회피 : advice 에 핸들러 추가

@ExceptionHandler(HttpMessageNotReadableException.class)
public ResponseEntity<ApiErrorResponse> handleUnreadable(HttpMessageNotReadableException ex) {
    return ResponseEntity.status(400).body(new ApiErrorResponse("요청 본문 형식이 올바르지 않습니다."));
}
v2 추가

10.7 BindException + path-variable 타입 변환 미커버

@ModelAttribute @Valid 위반 + BindingResult 매개변수 누락 시 BindException throw. @PathVariable("osId") Long osId 에 알파벳 들어오면 MethodArgumentTypeMismatchException. advice 가 안 잡으므로 Spring default. 현재는 BindingResult 흡수로 우회 중이나, 미래 컨트롤러 추가 시 silent 회귀.

회피 : advice 에 핸들러 추가 — 본 두 예외 모두 400 + ApiErrorResponse.ofValidation 으로 매핑.

v2 추가

10.8 multipart 비-Size 예외 미커버

ApiExceptionHandler:255-261 은 MaxUploadSizeExceededException 만 처리. 그 외 MultipartException 하위 (MissingServletRequestPartException 등) 는 advice 미커버 → Spring default. XHR 흐름이면 응답 형식이 JSON 가 아닐 위험.

회피 : MultipartException super-class 단일 핸들러 (다형성 — §3.1 SecurityException 패턴 그대로)

@ExceptionHandler(MultipartException.class)
public ResponseEntity<ApiErrorResponse> handleMultipart(MultipartException ex) {
    // MaxUploadSize 는 별도 sub-class 핸들러가 우선 매칭, 나머지는 본 fallback 으로 흡수.
    return ResponseEntity.status(400).body(new ApiErrorResponse("multipart 요청 처리 실패 : " + ex.getMessage()));
}
v2 추가

10.9 AsyncSubmitResult override 의 defer 협조 함정

error-modal.js 는 defer 로 commonHead 에 로드되어 DOMContentLoaded 시점에 실행. trash-action.js 는 페이지 끝의 non-defer <script> 라 HTML 파서가 만나는 즉시 실행되어 window.AsyncSubmitResult 를 먼저 설정. 이후 error-modal.js 가 if (!window.AsyncSubmitResult) 가드로 default 미설정. 이 협조는 암묵적.

위험 : 새 페이지에서 override 를 defer 로 두면 error-modal.js 의 default 가 먼저 자리잡아 override 무효.

회피 :

  • override 스크립트는 non-defer + 페이지 끝 으로 일관 배치 (trash-action.js 패턴)
  • 또는 error-modal.js 가 default 를 즉시 설정하지 말고 lazy proxy 로 두기 (onRejected: (...args) => (window.AsyncSubmitOverride ?? defaultHandler)(...args))

두 방안 모두 변경 1 도입 슬라이스에서 동반 검토.

10.10 v2 추가 — 기타 senior 권고 (참고)

  • error-modal.js 의 STATUS_LABEL 누락 상태 — 410 / 413 / 415 / 422 등 advice 가 던지는 코드가 정의 없음. modal 노출 시 라벨 없는 "HTTP 410" 만 표시. UX 사소하나 v2+ 정렬 시 보강 권장.
  • error.html 의 nav 하드코딩 — error.html:11 의 nav include 가 os-metadata activePage 강제. 사용자가 다른 페이지에서 에러를 만나도 nav 가 OS 로 전환. 변경 1 통합 시 동반 보완.
