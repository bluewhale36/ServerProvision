§1. 결과 요약
12종라벨·설명·권장조치 부여

0엔티티·REST 계약 변경

드리프트 상세(C3)가 이제 진단→처방 구조로 읽힌다: 종류(사용자 라벨 + enum 보조 표기) → 무슨 일이(description) → 권장 조치(recommendedAction). 12종 전부에 문구를 부여했고(IntegrityStatus.displayMessage 선례), 이 표가 곧 추후 사용자 가이드의 원문이다. 소비는 템플릿이 ${drift.kind.label} 처럼 enum 접근자를 직접 호출 — DTO 복제 없이 enum 이 유일 SSOT 이고 REST JSON 의 kind 직렬화(enum 이름)는 불변이다.

불가침 위반 2건을 해소했다. ① 자동 적용 허용 종류가 서버 5종 나열 vs 템플릿 4종 하드코딩 ×3곳으로 갈라져 있던 SSOT 위반 — DriftKind.isAutoApplicable() 로 승격해 서버 가드·버튼 노출·배지 색이 한 소스를 읽는다(부수 효과로 GHOST_DB_ROW 의 적용 버튼이 처음으로 노출 — 서버는 종전부터 허용했는데 UI 만 숨기고 "허용되지 않습니다"라 안내하던 모순 해소). ② 전역 reconciliation.auto-apply OFF 시 정상 클릭이 409 예외로 거절되던 것 — 서비스의 isAutoApplyEnabled() 한 메서드를 서버 가드와 뷰모델이 공유하고, OFF 면 버튼이 disabled + tooltip 으로 1차 차단된다(childEnableBlockReason() 선례 패턴). 서버 409 는 direct POST 안전망으로만 남고, 발동 시 문구도 발생 지점별로 정확해졌다(stale "PATH_DRIFT 종류에만 허용" 폐기 → factory 2종).

§6. CP5 가이드 — 사용자 브라우저 E2E
  1. 드리프트 상세(C3)에서 라벨 배지 + "무슨 일이" + "권장 조치" 3단이 보이는지 (enum 원시명은 배지 옆 작은 보조 표기).
