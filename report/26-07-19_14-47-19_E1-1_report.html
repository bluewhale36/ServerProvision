§1. 무엇이 가능해졌나 — 운영자 관점 한살이

E1-1 이전의 게스트는 개시 버튼을 눌러도 "미구현 단계(HOLD)" 안내만 30초마다 받으며 iPXE 화면에 머물렀다. E1-1 이후의 한살이는 이렇다:

  1. 서버를 연결하고 켠다 — PXE 부팅 → 자동 등록(기존 U1) → 개시 대기.
  2. 목록에서 서버를 확인하고 [프로비저닝 개시]를 누른다 — 다음 폴링(최대 30초)에서 게스트가 진단 리눅스 체인로드 스크립트를 받는다.
  3. 게스트가 스스로 진단 리눅스로 부팅한다 — Alpine 커널·모듈·설정 가방·패키지(~250MB)를 서버에서 받아 디스크를 건드리지 않는 RAM 상주 리눅스가 뜬다. 소요 약 30초(QEMU 실측 — 실기는 하드웨어 POST 가 더해진다).
  4. 에이전트가 자동 기동해 보고한다 — 체크인으로 상세 페이지 배지가 '진단 리눅스' 로 바뀌고, 원장에 DIAGNOSTIC_BOOTING → SUCCEEDED 행이 남는다("체인로드를 줬다" 가 아니라 "진짜 부팅됐다" 는 게스트 사실 기록).
  5. 게스트 화면(모니터)에 식별 배너가 뜬다 — 서버명 · UUID 끝자리 · IP. 여러 대를 동시에 꽂아도 실물 섀시와 목록의 행을 즉시 매핑할 수 있다(UC-5 · DEC-33 — 유스케이스 토론에서 확정한 물리 식별 해법의 1차 구현).
  6. 여기서 대기한다 — 하드웨어 수집 지시는 E1-2 몫. 에이전트는 30초 주기 WAIT 폴링이 정상 종착이다.
이 슬라이스의 의미 — E 로드맵에서 처음으로 게스트가 진짜 운영체제로 살아났다. 프로토콜(E1-0b)은 curl 모의였지만, 이제 실제 리눅스가 같은 API 를 밟는다. 그리고 그 검증을 물리 서버 없이 반복 가능한 QEMU 랩(T2)으로 해냈다 — 유스케이스 토론 §1(물리 의존성 구분)의 전략이 실증된 것이다.
🎬 실측 재연 — 부팅 한살이 + 스모크가 잡은 결함 4건 재현

아래는 2026-07-19 T2 스모크의 실측 타임라인 재연이다. 시나리오를 고르고 ▶ 재생을 누르면 게스트 콘솔과 서버 로그가 시간순으로 흐른다. 결함 시나리오를 고르면 개발 중 실제로 부딪힌 그 지점에서 그 증상 그대로 멈춘다 — 왜 그랬는지 설명이 함께 뜬다.

§3. 스모크 여정 — 8번의 부팅이 잡아낸 결함 4건

T0(계약 테스트)·T1(curl 하네스)은 처음부터 전부 통과했다. 그런데 진짜 리눅스를 부팅시키자 네 겹의 결함이 차례로 드러났다 — 전부 문서·조사로는 알 수 없고 실부팅으로만 잡히는 종류다. T2 랩을 정식 자산으로 만든 판단(Q-B)의 근거가 이 표다.

#증상 (실측)원인교정 (조립 파이프라인에 각인)
①모든 유지 스크립트 exit 127 → /sbin/init 부재 → 응급 셸. 응급 셸에서 Exec format error + ld-musl-aarch64.so.1 발견Apple Silicon 의 Docker 가 arm64 alpine 이미지를 선택 → apk fetch 가 aarch64 패키지를 수집. 인덱스의 --rewrite-arch 가 불일치를 은폐--platform linux/amd64 고정 + APKINDEX 아키텍처 검증 가드(이종 혼입 시 조립 자체가 실패)
②world 설치 실패 — openssl (no such package)Alpine init 이 apkovl 을 URL 로 받으면 world 에 openssl 을 스스로 추가한다 — 부분 저장소에 없으면 트랜잭션 전체가 실패하고 busybox 심링크 팜(=/sbin/init)이 안 생긴다fetch 목록에 openssl 동봉 (34개)
③패키지 설치까지 성공했는데 modloop·agent.sh 요청이 영영 안 옴 — OpenRC 단계 침묵apkovl 이 존재하면 Alpine init 은 기본 런레벨 구성을 만들지 않는다(오버레이가 가져온다고 가정) — networking 서비스가 어느 런레벨에도 없어 firstboot 의 need net 이 영구 미충족apkovl 에 표준 런레벨 12링크(sysinit 4 · boot 4 · default 1) + etc/network/interfaces 동봉
④에이전트의 step 보고가 500 — Data truncated for column 'step_code'Java enum 에 DIAGNOSTIC_BOOTING 을 추가했지만 DB 컬럼이 ENUM 타입이라 값 목록 확장 DDL 이 필요했다 — plan 의 "스키마 변경 0" 판단 오류(validate 모드는 타입 호환만 봐서 기동 시엔 안 잡힘)DDL 산출·적용 + plan 정정 주석. 이후 enum 상수 추가 시 ENUM DDL 동반을 관례로

디버깅 수단도 자산이 됐다: 시리얼 콘솔(커널 인자 console=ttyS0 + 랩의 캡처 옵션 — 실서버 SOL 겸용)과 응급 셸 원격 조작(시리얼을 유닉스 소켓으로 물려 게스트 내부를 직접 검사)이 이번 원인 규명의 결정타였다.

부수 발견 (CP5 중) — QEMU 기본 MAC(52:54:00:12:34:56)이 모든 게스트 공통이라 두 번째 게스트부터 host_mac UNIQUE 충돌 → UUID 파생 MAC 으로 교정. 이때 등록 실패가 게스트를 죽이지 않고 200+재시도 스크립트(E1-0b 예외 변환)로 흡수되는 것이 실게스트 경로에서 확인됐다 — 안전망 설계의 실증.
