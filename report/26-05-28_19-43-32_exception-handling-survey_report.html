1. 왜 이 보고서를 따로 쓰는가

지난 며칠간 예외 처리 한 가지가 R1-4-1 hotfix 단계를 세 번 가로막았다 :

  • v1 @Order(HIGHEST_PRECEDENCE - 100) 산술 underflow → 의도와 정반대 우선순위가 됨
  • v2 annotations 속성으로 advice 스코프 좁힘 → hybrid controller 28건 회귀
  • v2' @Order(HIGHEST_PRECEDENCE + 100) 양수 offset → Accept: */* 의 MockMvc 28건 회귀
  • v3 양 advice 모두 HIGHEST_PRECEDENCE + ContentNegotiation 위임 → 안정 (현재 상태)

반복된 회귀의 진짜 원인은 "두 개의 advice 를 어떻게 협조시킬까" 라는 우리의 출발점 자체가 Spring 의 권장 모델과 어긋났기 때문이다. 그러나 단순히 "Spring 권고대로 따르세요" 라는 보고서는 본 프로젝트에는 부족하다. 본 프로젝트는 CLAUDE.md 에 매우 구체적인 코딩 철학이 명시되어 있고, 실제 코드가 그 철학을 이미 상당 부분 따르고 있기 때문이다.

본 보고서의 목적 — 두 단계 :
  1. 먼저 현재 코드가 이미 따르고 있는 스타일 을 §3 에서 정리한다 (대부분 옳은 방향).
  2. 그 위에 4 권위 출처의 권고를 스타일을 깨지 않고 자연스럽게 흡수 하는 형태로 §8 에서 매핑한다.
결과 : "Spring 권장 = 단일 advice" 같은 추상 권고가 아니라 "현 ApiExceptionHandler 안의 SecurityException.httpStatus() 다형성 패턴을 그대로 살리되 produces 매처 한 행을 추가" 같은 구체 매핑이 나온다.

2. 예외 처리의 두 축 — 라우팅과 본문 형식

본문에 들어가기 전 어휘를 통일한다. 예외 처리는 독립된 두 결정 이다.

축결정 내용현 시스템 답변
① 라우팅 "어떤 핸들러 메서드가 이 예외를 잡을 것인가?"
→ 예외 타입 + Accept 헤더 (produces) 매칭		 두 개 advice (WebExceptionHandler + ApiExceptionHandler) 분리
② 본문 "JSON / HTML 응답 필드 스키마는 무엇인가?"
→ 응답 DTO + view model		 자체 정의 ApiErrorResponse(message, fieldErrors) + Thymeleaf error.html
비유 호텔 컨시어지로 비유하자.
  • 축 ① = "한국어 손님은 한국인 직원, 영어 손님은 영어 직원" — 라우팅 규칙.
  • 축 ② = "어느 직원이 응대하든 답변 양식은 같은 문서 템플릿" — 응답 표준.
현 코드는 ① 을 위해 두 직원(advice)을 두고 손짓 약속(@Order) 으로 순서를 정하려 했다. Spring 권장은 "한 직원이 두 언어를 다 소화하되 호출 받은 언어에 맞춰 답변" 하는 것이다 (§4).

3. 현 코드의 스타일 진단 — 이미 무엇이 잘 되어 있나

본 프로젝트의 CLAUDE.md 는 코딩 철학에 매우 구체적이다. 그중 본 보고서와 관련된 두 불가침 원칙 :

CLAUDE.md 인용 조건 분기문의 legacy 무분별 확장 절대 지양 (불가침). 신규 케이스 / 신규 예외 / 신규 도메인 타입이 추가될 때 분기문에 줄을 늘리는 방식으로 처리하지 않는다. 대신 Java 의 다형성 (interface / abstract class / sealed type / enum 의 method-per-constant / strategy pattern) 과 Spring Framework primitive (@ControllerAdvice / @RestControllerAdvice / HandlerExceptionResolver / @ResponseStatus / BindingResult 자동 매핑 / AOP / @EventListener / MessageSource / Converter) 를 적극 활용해 framework 측이 분기를 떠맡도록 설계한다.

이 원칙이 코드 곳곳에 이미 실현되어 있다. 보고서는 권고를 흡수할 때 이 그릇을 깨지 않아야 한다.

3.1 보안 예외 다형성 — 가장 성숙한 패턴

SecurityException 은 abstract super-class 이고, sub-class 가 httpStatus() 를 다형적으로 구현한다 : 

public abstract class SecurityException extends RuntimeException {
    public abstract HttpStatus httpStatus();   // 다형성으로 status 결정
}

public class PathTraversalException extends SecurityException {
    @Override
    public HttpStatus httpStatus() { return HttpStatus.BAD_REQUEST; }
}
// PathOutsideAllowedRootsException → 400
// UnsupportedMediaTypeException → 415
// UploadLimitExceededException → 413
// ZipBombSuspectedException → 415
// ForbiddenException → 403
// EntrypointInvalidException → 400

그리고 단일 핸들러 하나가 if-else / switch 없이 폴리모픽으로 받는다 :

@ExceptionHandler(SecurityException.class)
public ResponseEntity<ApiErrorResponse> handleSecurity(SecurityException ex) {
    return ResponseEntity.status(ex.httpStatus()).body(new ApiErrorResponse(ex.getMessage()));
}
스타일 평가 매우 성숙. 7개 보안 sub-class 가 분기 없이 동일 핸들러로 처리. 새 보안 예외 추가 시 sub-class 의 httpStatus() 만 정의하면 자동 흡수. 4 권위 출처 모두 이 패턴을 권장한다 (특히 AIP-193 의 "표준화된 단일 응답 모델"[4]).

3.2 @ResponseStatus 어노테이션 자동 흡수 — Spring primitive 적극 활용

도메인 예외 sub-class 에 @ResponseStatus 만 붙이면 advice 의 fallback 핸들러가 AnnotationUtils.findAnnotation 으로 그것을 읽어 자동 매핑한다 : 

@ResponseStatus(HttpStatus.CONFLICT)
public class ChildLifecycleBlockedByParentException extends ConflictException { ... }

@ResponseStatus(HttpStatus.BAD_REQUEST)
public class TypedNameMismatchException extends DomainException { ... }

@ResponseStatus(HttpStatus.SERVICE_UNAVAILABLE)
public class PurgeIoFailedException extends DomainException { ... }

그리고 ApiExceptionHandler.handleDomain 의 fallback :

@ExceptionHandler(DomainException.class)
public ResponseEntity<ApiErrorResponse> handleDomain(DomainException ex) {
    ResponseStatus rs = AnnotationUtils.findAnnotation(ex.getClass(), ResponseStatus.class);
    if (rs != null) {
        return ResponseEntity.status(rs.value()).body(new ApiErrorResponse(ex.getMessage()));
    }
    // fallback : 500
    return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body(new ApiErrorResponse(ex.getMessage()));
}
스타일 평가 Spring primitive 의 모범 활용. 신규 도메인 예외 추가 시 sub-class + 어노테이션 한 줄로 끝. advice 코드 0줄 추가. CLAUDE.md 원칙 ("framework 가 분기를 떠맡도록") 그대로다. Spring 공식 문서[1] 도 이 어노테이션을 권장한다.

3.3 도메인 super-class 계층 — 의미 단위로 분류

DomainException (abstract)
├── NotFoundException     → 404
├── ConflictException     → 409
│   ├── FieldBoundConflictException   (abstract, fieldName 보유)
│   │   └── // DuplicateISOContentException, DuplicateFilenameException 등 sub-class
│   ├── NudgeRequiredException       (abstract, payload 보유)
│   └── // ChildLifecycleBlockedByParent, IllegalOSMetadataState 등
├── FieldBoundBadRequestException  (abstract)
└── // 그 외 SoftDeleteRequiresIntent, PathCorrectionFailed, ...

SecurityException (abstract, 별도 계층 — DomainException 흡수 차단)
├── // 7 sub-class 가 httpStatus() 다형성

도메인 예외 100+ 클래스가 의미 단위 super-class 로 묶여 있어 advice 가 super-class 한 줄에 핸들러 1개로 다형성 흡수 가능. 새 예외 추가 시 적절한 super 선택 + sub-class 작성으로 끝난다.

스타일 평가 Google AIP-193 의 "표준 에러 코드 적은 수 + 자원 많은 수" 원칙[4] 과 일치. "다양한 종류의 NotFound" 가 아니라 단일 NotFoundException + 메시지로 분기. 권장 흐름 그대로.

3.4 보안 예외 계층 분리 — 사고 회피의 학습

SecurityException 이 DomainException 하위가 아닌 별도 계층인 이유가 코드 주석에 명시되어 있다 :

"컨트롤러의 catch (DomainException) 가 보안 예외를 흡수해 500 으로 새는 사고 (S3.3 silent-500 회귀) 를 차단하기 위함이다." — global/security/exception/SecurityException.java 주석

이 한 줄은 CLAUDE.md 의 분기문 확장 회피 원칙이 단순 미학이 아닌 실제 사고로부터 학습된 결정 임을 보여준다. 본 보고서의 매핑은 이 학습을 보존해야 한다.

3.5 frontend 의 글로벌 핸들링 — backend 와 동일한 흡수 패턴

global/error-modal.js 는 window.AsyncSubmitResult 라는 글로벌 polymorphic dispatch 를 둔다 : 

if (!window.AsyncSubmitResult) {
    window.AsyncSubmitResult = {
        onSuccess: () => window.location.reload(),
        onRejected: (form, status, payload) => show({ message: payload?.message, status }),
        onNetworkError: () => show({ message: '서버와 통신할 수 없어요.', status: 0 })
    };
}

페이지별 override (예: trash-action) 가 있으면 그것을 보존, 없으면 글로벌 modal. backend 의 advice 패턴과 정확히 같은 "default + specialize" 모델.

스타일 평가 backend/frontend 가 동일 철학 공유. 신규 페이지 추가 시 별도 override 없이 글로벌 흡수 — 분기문 확장 0줄. Microsoft API Guidelines 의 "centralized common error handling logic"[5] 권고와 일치.

4. 출처 ① — Spring 공식이 권장하는 단일 advice + dual produces 패턴

Spring Framework 공식 reference 의 "Exceptions" 페이지[1] 가 동일 예외 타입의 응답 형식 분기에 대해 명시한 패턴 :

"In addition to exception types, @ExceptionHandler methods can also declare producible media types. This allows to refine error responses depending on the media types requested by HTTP clients, typically in the Accept HTTP request header." — Spring Framework Reference, Media Type Mapping[1]

공식 문서가 보여주는 정형 코드 :

@ExceptionHandler(produces = "application/json")
public ResponseEntity<ErrorMessage> handleJson(IllegalArgumentException exc) { ... }

@ExceptionHandler(produces = "text/html")
public String handleHtml(IllegalArgumentException exc, Model model) { ... }

같은 advice 안의 두 메서드. Spring 공식은 이를 "중복 정의가 아니다" 라고 분명히 명시한다. ExceptionHandlerExceptionResolver 가 Accept 헤더와 매칭되는 메서드를 자동 선택.

왜 우리의 두 advice 분리보다 안전한가

비교 항목두 개 advice (현재)단일 advice + dual produces (Spring 권장)
매칭 순위 @Order 정수 비교 (overflow 위험) produces vs Accept 호환성 한 가지로 결정 (@Order 불필요)
Hybrid controller annotations 속성으로 좁히면 깨짐 (v2 사고) 스코프 좁힐 필요 자체가 없음
같은 예외에 신규 케이스 추가 두 advice 양쪽에 핸들러 추가 한 advice 의 produces 다른 메서드 두 개 추가

cause vs root 매칭 규칙

"A root exception match is generally preferred to a cause exception match. (...) This means a cause match on a higher-priority @ControllerAdvice bean is preferred to any match (for example, root) on a lower-priority @ControllerAdvice bean." — Spring Framework Reference, Exception Mapping[1]

이 문장에서 advice 간 우선순위가 진짜 의미를 갖는 케이스는 같은 예외를 root 와 cause 양쪽에서 매칭하려 할 때 뿐임을 알 수 있다. 우리 코드처럼 produces 만 다른 advice 두 개는 단일 advice 의 produces 다른 메서드 두 개로 합쳐도 의미 손실 없음.

5. 출처 ② — RFC 9457 Problem Details

응답 본문의 표준은 IETF 가 2023년 발행한 RFC 9457 "Problem Details for HTTP APIs"[3]. RFC 7807 후속 + 호환.

"This document defines a 'problem detail' to carry machine-readable details of errors in HTTP response content to avoid the need to define new error response formats for HTTP APIs." — RFC 9457, §1[3]

표준 필드 5종 + extension members

필드역할예시
type문제 유형 URIhttps://api.example/errors/iso-conflict
title사람이 읽는 짧은 제목"ISO 등록 충돌"
statusHTTP 상태 사본409
detail이번 발생의 구체 메시지"동일 경로의 deprecated ISO 가 있어요."
instance이번 발생 URI (trace 등)/management/os/1/iso/upload
"Problem type definitions MAY extend the problem details object with additional members that are specific to that problem type." — RFC 9457, §3.2[3]

즉 우리가 현재 별도 응답 타입 (NudgeRequiredResponse, DeleteRejectResponse) 으로 쓰는 도메인별 필드 (nudgeId, token, ttlSec) 는 ProblemDetail 의 extension 으로 흡수 가능.

Media Type 자동 분리

"When serialized in a JSON document, that format is identified with the 'application/problem+json' media type." — RFC 9457, §3[3]

Spring 6+ 의 ProblemDetail 클래스 (Spring Framework Reference Error Responses[2]) 는 응답 시 자동으로 이 content-type 으로 직렬화한다. frontend 가 response.headers.get('content-type') 만 봐도 표준 에러 응답임을 인식 가능.

설계 가이드 — 과도한 type URI 자제

"Truly generic problems are usually better expressed as plain status codes. For example, a 'write access disallowed' problem is probably unnecessary, since a 403 Forbidden status code in response to a PUT request is self-explanatory." — RFC 9457, §4.2[3]

모든 예외에 type URI 를 만들지 말고, 진짜 도메인 의미가 있는 것만 부여하라는 균형감각. 즉 우리의 NotFoundException 처럼 의미가 단일한 것은 type 생략 (about:blank), NudgeRequiredException 같은 도메인 특화는 type URI 부여.

6. 출처 ③ — Google AIP-193 (단일 일관 모델 + 머신 가독 식별자)

Google API Improvement Proposals (AIP) 193번 — "Errors"[4]. Google 사내 모든 API 가 따르는 강제 가이드라인.

"Services MUST return a google.rpc.Status message when an API error occurs." — AIP-193, Guidance[4]
"All error responses MUST include an ErrorInfo within details." — AIP-193, ErrorInfo[4]

핵심 통찰 :

"If the RPC has always returned ErrorInfo with machine-readable information, the content of Status.message MAY change over time." — AIP-193[4]

"메시지 텍스트는 변경 가능, 머신 가독 식별자는 고정" 분리. 우리 ApiErrorResponse 가 message 한 줄로 결정되는 것이 문제 — 한국어 메시지를 다듬으면 frontend 분기 (예: 텍스트 매칭) 가 깨질 수 있다.

"APIs SHOULD NOT support partial errors. Partial errors add significant complexity for users." — AIP-193[4]

이 권고는 R1-4-1 의 케이스 B (OS 만 복구 후 ISO 단독 복구 silent fail) 와 정확히 맞닿아 있다. AIP-193 관점에서 "OS 만 복구" 부분 액션 자체가 위험 — 전체 cascade saga 의 단일 실패 모델 권장.

7. 출처 ④ — Microsoft Azure API Guidelines

Microsoft 가 Azure / Graph API 에 강제하는 가이드라인[5]. RFC 9457 보다 한 단계 더 구체적인 응답 스키마. 

{
  "error": {
    "code": "notFound",            // camelCase, HTTP status description 과 1:1
    "message": "리소스 X를 찾을 수 없어요.",   // 개발자용, 최종 사용자 노출 부적합
    "target": "isoPath",             // 오류 대상 (필드명 등)
    "details": [...],
    "innererror": { ... }            // 더 깊은 원인 (계층 가능)
  }
}
"For transient errors that may be retried, services SHOULD include a Retry-After HTTP header indicating the minimum number of seconds that clients SHOULD wait before attempting the operation again." — Microsoft Azure REST API Guidelines[5]

우리 OptimisticLockingFailureException 처리에 그대로 Retry-After 헤더 추가 가능.

8. 우리 스타일에 권고를 흡수 — 4가지 변경의 모양

§3 의 스타일 진단과 §4~§7 의 4 출처 권고를 결합하면, 본 프로젝트에 의미 있는 변경은 정확히 4가지다. 각 변경은 §3 의 우수한 4 측면을 깨지 않는다.

즉시 채택

변경 1 — advice 통합 : WebExceptionHandler → ApiExceptionHandler 흡수

현 코드
// WebExceptionHandler.java (HTML 응답 전용)
@ControllerAdvice
@Order(HIGHEST_PRECEDENCE)
class WebExceptionHandler {
    @ExceptionHandler(NotFoundException.class,
                     produces = TEXT_HTML_VALUE)
    String handleNotFound(...)  // view "error"
}

// ApiExceptionHandler.java (JSON 응답 전용)
@RestControllerAdvice
@Order(HIGHEST_PRECEDENCE)
class ApiExceptionHandler {
    @ExceptionHandler(NotFoundException.class,
                     produces = APPLICATION_JSON_VALUE)
    ResponseEntity... handleNotFound(...)
}
→
매핑 후
// GlobalExceptionHandler.java — 단일 advice
@ControllerAdvice    // @RestControllerAdvice 아님
class GlobalExceptionHandler {

    @ExceptionHandler(value = NotFoundException.class,
                     produces = TEXT_HTML_VALUE)
    String handleNotFoundHtml(...)

    @ExceptionHandler(value = NotFoundException.class,
                     produces = APPLICATION_JSON_VALUE)
    @ResponseBody
    ResponseEntity... handleNotFoundJson(...)
}

스타일 보존 검증

  • §3.1 보안 예외 다형성 — 그대로. handleSecurity(SecurityException) 한 메서드가 7 sub-class 다형 흡수. produces 미명시 (XHR-only 흐름이므로 JSON).
  • §3.2 @ResponseStatus — 그대로. handleDomain 의 findAnnotation 흐름 유지.
  • §3.3 도메인 super-class 계층 — 그대로. NotFound/Conflict/FieldBound 분류는 그대로 유지하고 각 super 에 produces 짝만 추가.
  • §3.4 보안 계층 분리 — 그대로. SecurityException 은 여전히 DomainException 하위 아님.

왜 v3 hotfix 보다 더 안전한가

  • @Order 산술 overflow 함정 영구 제거 (advice 1개 → @Order 불필요)
  • 같은 advice 안의 produces 다중은 Spring 이 ambiguity 를 명시 거부 → silent 회귀 차단
  • Hybrid controller 의 모든 메서드가 동일 advice 의 매처로 잡힘 → v2 사고 원천 차단
중기 채택 (Adapter)

변경 2 — ApiErrorResponse 를 ProblemDetail 어댑터로 점진 전환

현 ApiErrorResponse 를 즉시 폐기하지 않는다 — 본 프로젝트 전반의 응답이 모두 이 record 에 묶여 있고 frontend 가 payload.message 를 기대한다. 대신 겸용 :

현 응답
{
  "message": "동일 경로 ISO 가 있어요.",
  "fieldErrors": [
    { "field": "isoPath",
      "message": "..." }
  ]
}
// Content-Type:
// application/json
→
RFC 9457 흡수 후
{
  "type": "https://sp.local/errors/iso-conflict",
  "title": "ISO 등록 충돌",
  "status": 409,
  "detail": "동일 경로 ISO 가 있어요.",    // = message
  "instance": "/management/os/1/iso/upload",
  "errorCode": "ISO_PATH_DUPLICATE",         // AIP-193 [4]
  "fieldErrors": [...]                          // 호환 보존
}
// Content-Type:
// application/problem+json

스타일 보존 검증

  • frontend 의 payload.message 호환을 위해 detail 외에 message 도 동봉 (deprecation 기간 둠)
  • fieldErrors 는 RFC 9457 extension members 로 그대로 유지 — 본 record 에 박혀 있던 도메인 의미를 잃지 않음
  • frontend 의 FormError.renderResponse 가 fieldErrors 를 본 것은 변경 불필요

Spring 6+ 의 ProblemDetail 활용 — Spring primitive 우선 (§3.2 와 동일 철학)

@ExceptionHandler(value = ConflictException.class, produces = APPLICATION_JSON_VALUE)
@ResponseBody
public ProblemDetail handleConflictJson(ConflictException ex) {
    ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.CONFLICT, ex.getMessage());
    pd.setTitle("리소스 충돌");
    pd.setProperty("errorCode", ex.errorCode());   // §변경 3 의 enum
    pd.setProperty("message", ex.getMessage());    // frontend 호환
    return pd;   // Content-Type 자동 application/problem+json [3]
}

Spring ProblemDetail 은 framework primitive — CLAUDE.md 원칙 "framework 측이 분기를 떠맡도록" 그대로다.

장기 채택 (점진)

변경 3 — errorCode 를 도메인 예외 super-class 메서드로 (다형성 유지)

AIP-193 의 머신 가독 식별자 권고[4] 를 적용하되, §3.1 SecurityException 의 httpStatus() 다형성 패턴을 그대로 흉내낸다. 별도 enum class 도입이 아닌 super-class 의 abstract method 추가.

옛 패턴 (현 코드)
public abstract class DomainException
        extends RuntimeException {
    // errorCode 없음
    // status 는 @ResponseStatus 어노테이션
}
→
매핑 후 (§3.1 패턴 흉내)
public abstract class DomainException
        extends RuntimeException {

    // default: 풀네임 기반 자동 생성 — sub-class 가 override 가능
    public String errorCode() {
        return getClass().getSimpleName()
            .replaceAll("Exception$", "");
        // "DuplicateISOContentException" → "DuplicateISOContent"
    }
}

왜 enum class 가 아니고 메서드인가

본 프로젝트의 도메인 예외는 이미 100+ sub-class. 별도 enum 으로 옮기면 새 예외 추가 시 두 곳 (예외 클래스 + enum) 을 동기화 해야 한다. CLAUDE.md 의 "분기문 / 매핑 테이블 확장 회피" 원칙에 어긋난다. 예외 클래스 자체가 errorCode 의 source of truth 가 되도록 default 메서드로 둔다.

도메인 의미가 다르면 sub-class 가 override 가능 :

public class DuplicateISOContentException extends FieldBoundConflictException {
    @Override
    public String errorCode() { return "ISO_DUPLICATE_CONTENT"; }   // 더 명료한 이름
}

frontend 활용

onRejected: (form, status, payload) => {
    if (payload?.errorCode === 'DELETE_INTENT_EXPIRED') {
        // 특별 처리 — 새로고침 후 modal 재호출 등
        location.reload();
        return;
    }
    // 일반 케이스 : 전역 modal
    show({ message: payload?.message ?? payload?.detail, status });
}

한국어 메시지 변경 자유 + frontend 분기는 errorCode 로 안정.

선택적 채택

변경 4 — transient 실패에 Retry-After 헤더 추가

Microsoft 권고[5] 를 우리 OptimisticLockingFailureException, PurgeIoFailedException, PurgeRetryExhaustedException 에 적용. 1줄 변경 : 

@ExceptionHandler(OptimisticLockingFailureException.class)
public ResponseEntity<ApiErrorResponse> handleOptimisticLock(OptimisticLockingFailureException ex) {
    return ResponseEntity.status(HttpStatus.CONFLICT)
            .header(HttpHeaders.RETRY_AFTER, "2")   // + 1줄
            .body(new ApiErrorResponse("동시 수정 충돌, 잠시 후 다시 시도."));
}

frontend error-modal.js 에 retry 카운트다운 표시 가능 (선택).

9. 단계별 로드맵 — 즉시 / 중기 / 장기

단계변경출처 근거회귀 위험예상 작업량
즉시 변경 1 (advice 통합) Spring 공식[1] 낮음 — 테스트 623 으로 즉시 포착 2~3 시간
중기 변경 2 (ProblemDetail adapter) RFC 9457[3], Spring[2] 중간 — frontend 동시 변경 필요. message 호환 유지로 점진 가능 1 슬라이스 (CP1~CP5)
장기 변경 3 (errorCode 다형성) AIP-193[4] 낮음 — default 메서드라 sub-class 점진 override 1~2 슬라이스 점진
선택 변경 4 (Retry-After) MS Azure[5] 없음 — 1줄 추가 30 분

9.1 추천 : 변경 1 → 변경 4 → 변경 3 → 변경 2 순서

변경 2 (ProblemDetail) 가 마지막인 이유 : frontend 동시 변경 위험이 가장 큼. 변경 1 (advice 통합) 으로 라우팅 안정화 → 변경 4 (Retry-After) 로 헤더 활용 사례 누적 → 변경 3 (errorCode) 로 frontend 분기 안정화 → 마지막에 본문 형식 표준화 (변경 2). 매 단계가 다음 단계의 회귀 안전망을 만든다.

한 슬라이스 안에 변경 1 + 4 를 묶고, 별도 슬라이스에 변경 3, 또 별도 슬라이스에 변경 2 가 자연스러울 것이다.

10. 자주 빠지는 함정 — 본 작업 중 실제 사례

10.1 @Order(HIGHEST_PRECEDENCE ± N) 산술 overflow

Ordered.HIGHEST_PRECEDENCE = Integer.MIN_VALUE. MIN_VALUE - 100 → underflow → wraparound → 양수 영역. 의도와 정반대 우선순위가 된다.

회피 : @Order 값에 HIGHEST_PRECEDENCE +/- 산술 자체를 금지. 다중 advice 우선순위는 명시적 작은 양수 (@Order(1), @Order(2)) 사용. 가장 안전한 건 §8 변경 1 처럼 advice 를 합쳐 우선순위 자체를 제거.

10.2 annotations = RestController.class 의 hybrid controller 함정

@ControllerAdvice(annotations = RestController.class) 로 advice 스코프를 좁히면 @Controller 인데 일부 메서드만 @ResponseBody 인 hybrid controller (예: OSImageController.isoProvisions) 가 advice 대상에서 빠진다. 결과 : JSON 기대 XHR 이 generic 500 으로 새거나 빈 응답.

회피 : annotations 속성으로 스코프를 좁히지 말고 모든 controller 를 단일 advice 가 담당 + produces 분기.

10.3 MockMvc 기본 Accept: */*

mockMvc.perform(post(...)) 만 호출하면 Accept 미지정 → */* → 양 advice 모두 매처 호환 → 등록 순서 의존성. 우리의 R1-4-1 v2' 시도가 여기서 28건 회귀.

회피 : 통합 테스트에서 실제 XHR 흐름 모사 시 .accept(MediaType.APPLICATION_JSON) 명시. SSR form 흐름 모사 시 .accept(MediaType.TEXT_HTML, MediaType.ALL). §8 변경 1 의 단일 advice 로 이 함정 자체 제거 가능.

10.4 frontend fetch 의 Accept 기본값

fetch 기본 Accept 가 */* → advice 매칭 우선순위 영향. 우리 hotfix v3 의 confirm-modal-base.js 가 명시적 Accept: application/json 보내도록 한 것이 옳았다.

회피 : XHR 호출은 항상 headers: {'Accept': 'application/json'} 명시. 브라우저 form submit 은 자동으로 text/html,...,*/*;q=0.8 보내므로 추가 처리 불필요.

10.5 "raw JSON 화면 노출" 의 진짜 원인 분류

  1. SSR form submit 이 application/json 응답 받음 — advice 매처 분리 실패 (§10.1, §10.2)
  2. XHR fetch 가 페이지로 navigate 됨 — form 의 onsubmit 가 preventDefault 안 함
  3. error-modal 로딩 실패 — defer 순서 문제, CSS 미로드

진단 시 가장 먼저 봐야 할 것은 DevTools → Network → 해당 요청 → Response Headers → Content-Type. JSON 인데 화면에 표시됐다면 ①, HTML 인데 분기 깨졌다면 ②, JSON 인데 modal 안 떴다면 ③.
