1. 요약 — 무엇을 고쳤나

2026-07-19 전수 점검이 ISO 등록과 휴지통 이동에서 세 가지 결함을 잡아냈다. 셋 모두 정상 브라우저 흐름에서는 드러나지 않는다 — 화면 JavaScript 가 디렉토리 선택 시 파일명을 이어붙여 완전한 경로를 만들어 보내고, Thymeleaf 체크박스가 hidden 마커 _allowCreateDirectory 를 항상 동봉하기 때문이다. 따라서 발동 경로는 curl 같은 direct POST(F-4·F-5)와 휴지통 목록의 표기(F-7)다. 정상 사용자가 즉시 체감하는 결함은 아니지만, 프로젝트가 가장 경계해 온 두 가지 사고 패턴 — 입력 실수가 서버 내부 오류(500)로 위장되는 것, 실패가 실패로 보이지 않는 silent 흡수 — 에 정확히 맞닿아 있어 안전망 관점에서 방어를 심었다.

HF4-3 은 이 셋을 한 슬라이스로 묶어 처리했다. 요지는 다음과 같다.

2. 유스케이스 전후 (F-4 · F-5 · F-7)

세 결함이 실제 조작에서 어떻게 나타났고 무엇이 달라졌는지를 상황 중심으로 서술한다. 각 케이스의 코드 위치는 §6 Critical Files 와 대응한다.

F-4 · F-4b ISO 경로에 디렉토리를 주고 파일을 첨부한 direct POST

상황. 운영자가 화면을 거치지 않고(스크립트·재시도 자동화 등) POST /management/os/{id}/iso 에 isoPath 값으로 이미 존재하는 디렉토리(예: /opt/iso/rocky9)를 넣고 ISO 파일을 함께 첨부했다. 정상 화면에서는 디렉토리를 고르면 JavaScript 가 파일명을 이어붙여 완전한 경로를 만들어 주므로 이 상태로 서버에 도달하지 않지만, direct POST 는 그 보정을 건너뛴다.

고치기 전

등록 준비(prepare)에는 "대상이 디렉토리인가" 가드가 없었다. 중복 파일 검사 조건이 Files.exists(target) && !Files.isDirectory(target) 라 디렉토리는 그대로 통과했고, 이어지는 파일 저장(storeUploadedFile)의 Files.copy(in, target) 가 "대상이 디렉토리라 쓸 수 없음" 으로 IOException 을 냈다.

그 IOException 은 ISOFileStorageException(@ResponseStatus 없음)으로 감싸져 예외 advice 의 fallback 을 타 HTTP 500 이 됐다. 사용자에겐 자기 입력 실수인지 서버 장애인지 구분할 단서가 없다.

isoPath=디렉토리 → copy IOException → ISOFileStorageException → advice fallback 500

고친 뒤

IsoRegistrationService.prepare 가 allowlist 검증(assertWritablePath) 직후 if (Files.isDirectory(target)) throw new IsoPathIsDirectoryException(resolvedPath) 로 선판정한다. 신규 예외는 FieldBoundBadRequestException(@ResponseStatus 400) 하위이고 자기 필드명 isoPath 를 들고 온다.

화면 폼(SSR). IsoUploadController.addIso 가 FieldBoundBadRequestException(추상 family)을 catch 해 rejectValue(ex.fieldName(), …) + 응답 상태 400 + 폼 재렌더로 흡수한다. 재렌더된 폼의 isoPath 슬롯에 "경로가 디렉토리입니다. 저장할 파일명까지 포함한 경로를 입력하세요 : <경로>" 가 뜬다.

XHR 채널. /iso/upload 는 같은 prepare 가드를 타므로, 기존 ApiExceptionHandler.handleFieldBoundBadRequest 가 advice 수정 없이 400 JSON + fieldErrors[0].field="isoPath" 로 변환한다.

isoPath=디렉토리 → prepare 선판정 → IsoPathIsDirectoryException(isoPath) → 400 (SSR 필드오류 / XHR fieldErrors)

F-4b — intent 이중 방어. 업로드 전 사전 핸드셰이크 IsoUploadIntentService.validateFilesystem 도 같은 exists && !isDirectory 조건이라 디렉토리에 토큰을 발급했다. 그러면 클라이언트가 대용량 바이트를 다 전송한 뒤에야 위 500 을 만난다. 고친 뒤에는 검증 첫머리에 if (Files.isDirectory(target)) throw new IsoPathIsDirectoryException(resolvedPath) 를 넣어 핸드셰이크 시점에 400 으로 거절한다 — 무의미한 전송 자체가 시작되지 않는다. intent 의 토큰·nudge·hash 단계 계약은 그대로이며, 검증 규칙 한 건만 추가됐다.

F-5 checkbox 마커가 빠진 direct POST 의 silent 200

상황. Spring 은 체크박스가 꺼진 상태를 전달하려고 hidden 필드 _allowCreateDirectory 를 함께 보내는 관례를 쓴다. direct POST 가 이 마커를 빠뜨리면 요청 객체(record)의 primitive boolean allowCreateDirectory 에 바인딩할 값이 없어 바인딩이 실패한다.

고치기 전

바인딩 실패 → bindingResult.hasErrors() → iso-new 재렌더까지는 정상 동작이다. 그런데 템플릿의 서버 렌더 오류 슬롯은 isoPath 하나뿐이라(th:errors="*{isoPath}") allowCreateDirectory 필드 오류는 표시될 자리가 없었다.

결과는 HTTP 200 으로 입력값만 보존된 폼이 다시 그려지고, 화면 어디에도 오류가 없다. 등록은 안 됐는데 화면은 멀쩡하다 — 실패가 실패로 보이지 않는 가장 조용한 유형이다.

마커 부재 → boolean 바인딩 실패 → hasErrors 재렌더 → 슬롯 없음 = 무표시 200

고친 뒤

iso-new.html 폼 상단에 서버 렌더 글로벌 오류 배너를 넣었다. #fields.detailedErrors() 를 { global or fieldName != 'isoPath' } 로 필터한 목록 — 즉 전용 슬롯이 없는 오류만 — 배너에 나열한다.

배너 헤드라인은 "입력값을 처리하지 못한 항목이 있습니다." 이고, 각 항목은 global 오류면 메시지, 필드 오류면 "필드명 — 메시지" 형태다. 기존 알럿 CSS(n-alert n-alert-danger)를 재사용하며 인라인 스타일은 쓰지 않았다.

isoPath 오류는 여전히 기존 인라인 슬롯이 담당하므로 배너와 이중 표시되지 않는다. 정상 검증 흐름(경로 공백 등)의 화면은 그대로다. 앞으로 슬롯 없는 새 필드가 추가돼도 자동으로 배너에 잡혀 silent 재발이 원천 봉쇄된다.

마커 부재 → 바인딩 실패 → 재렌더 200 + 배너("입력값을 처리하지 못한 항목이 있습니다")

F-7 휴지통 이동 리네임이 디렉토리 이름의 점을 확장자로 오분리

상황. 자원이 휴지통으로 이동할 때 충돌 방지를 위해 이름 뒤에 timestamp + UUID8 접미사를 붙인다. 파일 자원은 확장자를 보존하려고 마지막 점을 기준으로 base/ext 를 분리한다. 문제는 이름에 점이 있는 디렉토리 자원(예: BMC 펌웨어 버전 디렉토리 v1.10, v2.3)에도 같은 분리가 적용된 것이다.

고치기 전

TrashService.generateTrashedFileName 이 original.lastIndexOf('.') 로 무조건 base/ext 를 나눴다 — 대상이 파일인지 디렉토리인지 보지 않았다.

그래서 디렉토리 v1.10 이 v1_<ts>_<hash>.10 처럼 되어 ".10" 이 확장자인 양 접미사 뒤로 떨어져 나갔다. 복원은 원경로 기준이라 기능 영향은 없지만, 휴지통을 직접 열어보는 운영자에겐 버전 표기가 깨져 보였다.

"v1.10" (디렉토리) → lastIndexOf('.') 무조건 분리 → "v1_…_<hash>.10"

고친 뒤

접미사("_" + ts + "_" + uuid8)를 한 번만 계산해 두고, if (Files.isDirectory(resourcePath)) 면 확장자 분리 없이 전체 이름 + 접미사를 반환한다. 파일은 기존대로 확장자를 보존하는 분기를 탄다.

판정 시점이 moveToTrash 의 실제 이동 직전이라 원본이 파일시스템에 실재한다 — 그래서 이름을 추측하는 대신 실체로 파일/디렉토리를 판정할 수 있다. 호출부·메서드 시그니처는 바뀌지 않았다.

"v1.10" (디렉토리) → Files.isDirectory 판정 → "v1.10_<ts>_<hash>" (점 미분리)

3. 설계 결정

plan(CP1)에서 복수 대안을 비교해 고른 결정 중, 검수에서 평가받은 네 축을 중심으로 정리한다. 탈락안과 그 사유까지 함께 남긴다.

D1. 필드 직결 400 을 다형 예외로 — catch 가 sub-class 별로 자라지 않게

신규 IsoPathIsDirectoryException 을 기존 추상 계층 FieldBoundBadRequestException(fieldName() 보유, @ResponseStatus 400) 아래에 두고, 자기 필드명 isoPath 를 생성자에서 상위로 넘기게 했다. 덕분에

  • XHR 채널은 기존 ApiExceptionHandler.handleFieldBoundBadRequest 가 계층 흡수로 자동 매핑한다 — advice 에 새 분기를 추가하지 않았다.
  • 화면 폼(SSR)은 addIso 가 추상 family 를 한 번 catch 해 rejectValue(ex.fieldName(), …) 로 흡수한다. IsoLifecycleController.updateIso 의 rejectValue("isoPath", …) 선례와 동형이다.

이 형태의 핵심은 예외가 자기 필드명을 들고 오는 것이다. 앞으로 다른 필드에 직결되는 400 예외가 생겨도 컨트롤러 catch 는 ex.fieldName() 을 그대로 쓰므로 분기가 늘지 않는다 — CLAUDE.md 의 "조건분기 legacy 확장 금지 → 다형성" 원칙에 정확히 부합한다. 탈락: (b) 기존 InvalidIsoPathException(409) 재사용 — 입력 형식 오류에 409 는 의미 오염이고 fieldName 이 없어 필드 표시 불가. (c) 저장 IOException 메시지만 개선(500 유지) — 요구가 400 이고, 파일 저장을 시도한 뒤의 사후 판정이라 원인 추정이 필요.

D2. 이중 방어 — prepare 가드 + intent 핸드셰이크 가드

같은 IsoPathIsDirectoryException 을 두 지점에서 던진다. prepare 가드는 실제 등록 요청의 안전망이고, validateFilesystem 가드는 그 앞단 핸드셰이크에서 무의미한 바이트 전송 자체를 차단한다. 판정식이 동일하므로(둘 다 Files.isDirectory(target)) SSOT 가 갈라지지 않는다. XHR upload 경로와 컨트롤러가 검증을 각자 하지 않고 service 한 곳에서 판정하고, 표현만 채널별(폼 재렌더 / JSON)로 나뉜다. 탈락: 컨트롤러가 prepare 호출 전에 직접 Files.isDirectory 검사 — 검증 로직이 두 층으로 분산돼 XHR upload 경로와 SSOT 가 갈라짐.

D3. F-5 배너 대상 한정 — checkbox 마커 패턴 전수 확인

배너를 어디까지 적용할지 판단하려고, "record + primitive boolean + hidden _marker" 패턴을 쓰는 폼을 전수 확인했다. 결론은 silent 200 이 성립하는 곳은 iso-new 하나다.

폼마커 패턴silent 200 가능?배너 적용
management/os/iso-new.html있음예 — 유일하게 SSR fallback POST(addIso)가 hasErrors 시 폼을 재렌더적용
management/bios/bios-new.html있음아니오 — 등록 POST 가 전부 @ResponseBody XHR, 바인딩 오류는 400 JSON 으로 이미 표면화미적용
management/bmc/bmc-new.html있음아니오 — 동일(XHR 전용, 400 JSON)미적용
management/subprogram/subprogram-new.html있음아니오 — 동일(XHR 전용, 400 JSON)미적용

그 외 체크박스는 목록 필터(GET)와 provisioning 세팅 폼(JSON @RequestBody, 마커 개념 없음이며 수정 금지 영역)이라 이 패턴이 아니다. silent 200 은 "SSR 재렌더 + 표시 슬롯 부재" 의 교집합에서만 성립하고, 그 교집합이 iso-new 하나이므로 배너도 거기에만 넣었다. 배너 구현은 #fields.detailedErrors() 를 슬롯 있는 필드(isoPath) 제외로 필터하는 방식을 골랐다. 탈락: (b) 전체 오류 요약 배너 — isoPath 오류 시 배너+인라인 이중 표시로 정상 화면이 바뀜. (c) allowCreateDirectory 전용 슬롯 추가 — 다음에 슬롯 없는 필드가 또 생기면 같은 silent 사고 재발. 제외 목록('isoPath')은 인라인 슬롯과 같은 템플릿 파일 안에 있어, 슬롯을 추가·삭제하는 수정과 같은 diff 에서 함께 보이는 SSOT-인접 위치다.

D4. F-7 은 확장자 추정이 아닌 파일시스템 실체 판정

"마지막 세그먼트가 숫자면 확장자가 아니다" 같은 이름 휴리스틱은 v2.iso 라는 디렉토리, file.10 이라는 파일 앞에서 오판정한다. 리네임 판정 시점(moveToTrash 의 이동 직전)에는 원본이 실재하므로 Files.isDirectory 로 실체를 직접 보는 것이 유일하게 정확하다. 접미사 조립을 한 번만 계산해 두 분기가 공유하도록 해 중복도 없앴다. 존재하지 않는 경로(ghost)는 isDirectory=false 로 기존(파일) 규칙을 타지만, 어차피 직후 이동이 실패하므로 관측 가능한 행동 변화가 없다. 탈락: (b) 호출자가 boolean isDirectory 를 인자로 전달 — 호출자가 1곳뿐인데 무명 boolean 인자로 시그니처만 오염.

5. 잔여 · 후속
  • 바인딩 기본 메시지 한글화(후속 후보). F-5 배너가 표면화하는 바인딩 실패의 기본 메시지는 Spring 이 만드는 영문(typeMismatch — "Failed to convert value …")일 수 있다. 배너의 목적은 한국어 헤드라인 + 필드명/원문 나열로 표면화하는 것이며, 메시지 자체의 한글화는 프로젝트에 messages.properties 가 아직 없어 MessageSource 도입이 선행돼야 한다. 이는 전 폼 공통 개선이라 HF4-3 범위 밖의 별도 슬라이스로 남긴다.
  • 기존 휴지통의 이미 오분리된 이름은 소급 정정하지 않는다. F-7 수정은 이동 시점의 이름에만 영향을 준다. 이미 v1_…_<hash>.10 형태로 휴지통에 들어간 항목은 복원이 원경로 기준이라 정상 동작하므로, 데이터 수술 없이 그대로 둔다.
  • 건드리지 않은 것. 예외 advice 2종, IsoPathResolver, ISOCreateRequest, bios/bmc/subprogram 폼·컨트롤러(§3 D3 미적용 결론), 클라이언트 스크립트(iso-new.js), 업로드 intent 프로토콜(요청/응답 계약), 스키마·CLAUDE.md·Notion 구조, execution·provisioning 영역 — 모두 무변경.
  • 전수 점검의 다른 결함. 이번 슬라이스는 direct POST/표기 한정인 F-4·F-5·F-7 만 다뤘다. 정상 사용자 경로에서 즉시 체감되는 휴지통 연장 묶음(F-1 비라우팅 500 · F-2 모달 라벨 오배선 · F-3 오류 응답 협상)은 우선순위 P1 로 별도 처리 대상이다.
