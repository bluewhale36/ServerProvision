1. 한 줄 요약과 호출 진입점 3 가지

ISO 등록은 2 단계 트랜잭션 으로 표현된다. 단계 A (prepareIsoRegistration) 는 파일을 디스크에 안전히 자리잡게 하고, 단계 B (finalizePreparedIsoRegistration) 는 디스크에서 hash 를 계산해 모든 충돌 검사 후 DB row + 마커를 발급 한다.

호출 진입점은 3 가지 :

진입점 호출 형태 사용 흐름
addISO(...) prepare + finalize 동기 호출 wrapper SYNC 옛 form submit + 즉시 응답 (작은 ISO)
prepareIsoRegistration(3-arg) clientHash 없는 호출 A intent 미경유 backward-compat 경로
prepareIsoRegistration(4-arg) clientHash 포함 호출 A + ASYNC MK2 WAVE 3 — intent 후 BackgroundJob 흐름

본 보고서는 두 메서드의 본질 흐름과 분리 사유를 중심으로 다룬다.

2. 두 메서드의 책임 매트릭스

단계 A prepare @Transactional

FS 사전 처리 — 디스크에 자리잡기까지.

  • 부모 OSMetadata 활성 확인
  • path resolve (IsoPathResolver) — / 로 끝나면 업로드 파일명 결합
  • 부모 디렉토리 확보 (allowCreateDirectory)
  • S3 보안 — PathPolicyService.assertWritablePath
  • DB 사전 충돌 — 같은 (osMetadataId, isoPath) 활성 row
  • FS sidecar 충돌 — .provision.json 마커 이미 있으면 reject
  • 파일 업로드 시 디스크 스트리밍 저장 + POSIX 0644 권한 강제 (S3.1)
  • 업로드 없을 시 기존 디스크 파일 정규성 검증
  • DB row 는 생성하지 않음
  • PreparedIsoRegistration record 반환 (다음 단계로의 컨텍스트)

단계 B finalize @Transactional

Hash 계산 → 충돌 분기 → 영속화 → 마커 → 후속 자동 작업.

  • S3 — PathPolicyService 재검증 (이중 가드)
  • 부모 재조회
  • SHA-256(디스크 파일) → manifestHash
  • MK2 WAVE 3 — clientHash 와 manifestHash 비교 (mismatch 시 파일 정리 + throw)
  • BackgroundJob stage = CHECK_DUPLICATE
  • path 중복 재검사 (race 방어)
  • 활성 ISO 의 hash 충돌 → 파일 정리 + DuplicateISOContentException
  • dormant (soft-deleted ∪ deprecated) hash 충돌 → NudgeSession 발급 + throw, 디스크 파일은 그대로
  • Ghost (DB-only soft-deleted, FS 부재) 후보는 사전 필터링 — 사용자 흐름 차단 방지
  • BackgroundJob stage = PERSIST_METADATA
  • isoRepository.save — ISO row + checksum + manifestHash 영속화
  • finalizeMarker — sidecar .provision.json + HMAC 서명 발급

5. 컨텍스트 record — PreparedIsoRegistration

prepare 와 finalize 사이의 컨텍스트 운반체. record (불변).

public record PreparedIsoRegistration(
    Long osMetadataId,    // 부모 OSMetadata id
    String resolvedPath,  // 정규화된 ISO 절대 경로 (이미 디스크에 파일 존재)
    String description,   // 사용자 입력 메모
    String originalName,  // 업로드 파일명 (logging / 후속 표시용)
    boolean uploadedFile, // true 면 디스크 파일이 본 등록 흐름에서 만들어졌음 (실패 시 정리 대상)
    String clientHash     // MK2 WAVE 3 — intent 단계에서 client 가 미리 보낸 hash. null 가능
) {}

본 record 의 직렬화 / 영속화 형태는 별도 없다 — JVM 메모리만 거친다. prepare 가 호출자 (controller 또는 background job runner) 에게 반환 → 호출자가 finalize 인자로 그대로 넘김.

MK2 WAVE 3 의 BackgroundJob 흐름에서는 IsoRegistrationRunner 가 record 를 그대로 보관했다가 finalize 호출에 사용.

7. 두 단계로 분리한 4 가지 이유

① BackgroundJob 단계 추적 — 대용량 파일의 비동기 진행 표시

ISO 는 보통 수 GB. 한 트랜잭션 / 한 HTTP 응답 안에 모두 처리하면 응답 대기 + DB 트랜잭션 점유 시간 모두 수 분 단위로 늘어남.

Why — prepare 끝나는 시점에 호출자가 즉시 202 + jobId 반환 가능. finalize 는 비동기 runner 가 처리하면서 CHECK_DUPLICATE → PERSIST_METADATA → POST_CREATION stage 를 사용자에게 polling 으로 노출.

② MK2 WAVE 3 — client hash 의 end-to-end 검증

client 가 업로드 시작 전 파일 SHA-256 을 계산해 intent 로 전송. 그 값이 prepare → finalize 까지 record 로 전달돼 server 가 디스크에서 재계산한 hash 와 비교.

Why — 한 메서드 안에서는 "intent 시 받은 client hash" 를 보관할 자리가 없다. prepare 와 finalize 가 분리되어야 그 사이 컨텍스트를 record 로 운반할 수 있다. 전송 중 corruption 이나 의도적 변조를 finalize 시점에 발견하면 파일 + sidecar 정리 후 throw.

③ nudge 흐름의 깔끔한 confirm/replace

finalize 가 dormant (soft-deleted ∪ deprecated) 자원과 hash 충돌 발견 시 IsoNudgeRequiredException throw. 이 시점에 :

  • DB 에는 아직 새 ISO row 없음 (B10 단계 미진입)
  • 디스크 파일은 prepare 결과로 자리잡혀 있음
  • 마커 sidecar 도 아직 없음

사용자가 modal 에서 confirm → 별도 endpoint 가 다시 finalize 호출 → 같은 디스크 파일 재사용, 다시 업로드 받지 않음.

Why — 단일 메서드면 nudge throw 시 사용자에게 "재업로드 하세요" 라는 회귀가 강제됨. 두 단계라야 "디스크는 그대로, DB 만 사용자 결정 대기" 라는 nudge 의 의미가 깔끔하게 표현된다.

④ 트랜잭션 점유 시간 최소화

각 메서드에 @Transactional 별도 — 두 짧은 트랜잭션. 사이 시간 동안 OSMetadata / ISORepository 의 write lock 이 풀려 있다.

Why — 한 트랜잭션이면 업로드 시간 (분 단위) 동안 lock 점유 → 다른 사용자의 list / detail 조회까지 영향. 두 짧은 트랜잭션으로 쪼개면 hash 계산이 가장 오래 걸리는데 그건 read-only 흐름이라 lock 부담 작음.

10. 남은 의문 / 후속

  • R1-5 (가칭) — OSMetadataService 의 ISO 관련 메서드 (addISO / prepareIsoRegistration / finalizePreparedIsoRegistration / cleanupIsoArtifacts / restoreISO / purgeIso) 를 별도 service 파일 (예: IsoLifecycleService 또는 IsoRegistrationService) 로 분리하는 게 자연스럽다 — 현재 OSMetadataService (960 줄) 의 가장 큰 책임이 여전히 ISO. R1-3 의 lifecycle 분리와 같은 패턴.
  • nudge 흐름의 disk 파일 TTL — 사용자가 confirm modal 을 5 분 안에 안 누르면 disk 파일은 어떻게? 현재 NudgeRegistry 의 5 분 TTL pruner 가 임시 파일까지 정리하는지 확인 필요. 정책 미명세 시 ghost 파일 잔존 위험.
  • B6 path 재검사의 race window — prepare ~ finalize 사이 race 방어 의도지만 그 사이 동시에 다른 사용자가 같은 (osId, path) 로 finalize 가 모두 완료된다면 B6 가 잡지만, 한쪽이 prepare 중이고 다른쪽이 prepare 직후일 경우 race condition 잔존 가능. lock 도입 검토 자리.
  • BackgroundJob 의 stage 추적이 finalize 안에서 직접 호출 — Job 의 lifecycle 책임을 service 가 직접 가져가는 모양. @TransactionalEventListener 또는 AOP 로 끌어올릴 자리. S5-13 (취소된 슬라이스) 의 정신과 정렬.
