§1. 결론 요약
[정정 — 2026-06-28, HF-5 CP1 재검증] 본문이 board restore 409 를 "raw JSON"으로 적은 것은 부정확하다. native form POST 는 Accept: text/html 을 보내고, DuplicateBoardModelException → FieldBoundConflictException → ConflictException → DomainException 이 WebExceptionHandler.handleDomain(produces=text/html)에 흡수돼 error.html(스타일된 전체 페이지 에러)로 렌더된다 — raw JSON 덤프가 아니다. 5월의 raw JSON 증상은 R1-4-1 v3 + R2-3 의 dual-advice produces 라우팅이 이미 해소했다. 구조적 결론(board 만 data-async-submit 누락 → 형제 5개와 렌더 비일관, 1줄 fix)은 유효하나, 심각도는 "raw JSON 파손" → "전체페이지 에러 이탈 vs inline 모달 비일관"(경미)로 하향한다. 아래 "raw JSON" 표현은 모두 이 정정으로 읽는다.
물음답핵심
왜 R2-2-2/3 가 생겼나R2-2-1 audit 의 부산물"버튼 보이는데 누르면 예외" 패턴을 확산하다가, 부모-자식(세로) 외에 restore 중복키(가로=cross-row 유일성)라는 같은 부류의 잔존 모순을 발견 → scope 분리
원래 무엇이 문제였나restore 409 가 raw JSON 노출복구 버튼을 누르면 Duplicate*Exception(409)이 full-page 로 떠 JSON 이 화면에 덤프되던 UX 파손 (사용자 기억과 일치)
지금도 그런가거의 해소됨6개 restore 표면 중 5개(OS·ISO·BIOS·BMC·Subprogram)는 data-async-submit → ErrorModal 로 이미 완화. 비일관(전체페이지 error.html 이탈)은 BoardModel management-list 1곳만 잔존
슬라이스 구현이 필요한가불요5도메인 UI capability(per-row dup-key 질의 + toResponse static→instance 리팩터)는 over-engineering. 잔존 결함은 board restore 폼에 data-async-submit 한 줄 추가로 닫힘
한 줄 결론 — R2-2-2/3 을 촉발한 본질(raw JSON UX 파손)은 그 사이 도입된 async-submit 인프라 + R2-6 ErrorModal 로 5/6 표면에서 이미 사라졌다. 남은 단 하나의 진짜 구멍은 BoardModel 의 management-list restore 폼이 data-async-submit 을 빠뜨린 비대칭(누락 추정)이며, 형제 5폼과 동일하게 마커 한 줄을 붙이면 ErrorModal 로 수렴한다. "복구 버튼을 렌더 시점에 미리 disable+tooltip" 하는 R2-2-3 안 A 는 fix 가 아니라 polish 이고, 5도메인 질의·리팩터 비용 대비 가치가 낮아 보류 유지/DROP 을 권고한다.
§2. 등장 배경 — 어쩌다 생겼나

R2-2 가족은 모두 "버튼은 보이는데 누르면 예외" UX 를 없애는 작업이다. 그 안에서 두 개의 성격이 다른 모순을 다룬다.

단계다루는 모순관계 축상태
R2-2부모 비활성/삭제/deprecated 인데 자식 활성화·복구·undeprecate 버튼 클릭 → 409세로(부모-자식)완료 ISO/BIOS/BMC
R2-2-1위 패턴을 프로그램 전반 확산 → audit 결과 Subprogram 단일 도메인 수렴세로(부모-자식)완료 Subprogram
R2-2-2soft-deleted 자원 복구 시 같은 유일키의 활성 자원이 있으면 Duplicate*Exception 409가로(cross-row 유일성)보류
R2-2-3R2-2-2 의 안 A(UI 차단)를 구체화 — restore 버튼 렌더 시점 disable+tooltip가로(cross-row 유일성)보류

분리 경위

R2-2-1 의 18-에이전트 audit 가 "UI 1차 차단을 어디까지 확산할까"를 전수 조사하던 중, 부모-자식 lifecycle 말고 restore 경로에서 똑같은 "클릭해야 비로소 거절당하는" 패턴을 추가로 발견했다. 시나리오:

1. 드라이버 'a' 등록            → 활성
2. 'a' 삭제                    → 휴지통 (isDeleted=true)
3. 같은 키 'a' 재등록           → 활성 ('a' 가 다시 생김 — 삭제본은 유일키 점유 안 함)
4. 휴지통의 옛 'a' [복구] 클릭  → DuplicateSubprogramVersionException (409)

이건 부모-자식(세로)이 아니라 같은 테이블 안 두 행의 유일키 충돌(가로)이라 R2-2-1 에 섞으면 scope 가 흐려진다. 그래서 별도 슬라이스로 떼어낸 것이 R2-2-2 이고, 그 해법 중 "UI 차단" 안만 따로 구체화한 것이 R2-2-3 이다.

audit 가 같이 확인한 사실 — 이 충돌은 Subprogram 만의 특수상황이 아니라 5 도메인(BoardModel·OSMetadata·BIOS·BMC·Subprogram)이 restore 시점에 동일하게 Duplicate*Exception 을 throw 한다(ISO 만 content-hash 식별이라 키 충돌 개념 자체가 없음). 따라서 Subprogram 만 UI 차단하면 "5 중 1만 차단"의 새 비대칭이 생긴다 → "전 도메인 일괄" 또는 "미적용" 둘 중 하나여야 한다는 제약이 R2-2-2 의 출발점이었다.
§3. 논의된 두 갈래 · 보류 결정

R2-2-2 페이지는 "예외를 어떻게 없앨까"를 두 갈래로 적어두고 진입 시 택일하기로 했다.

안 A — restore 시점 UI 1차 차단 (= R2-2-3 으로 구체화)

  • 각 도메인 list 응답에 restoreBlockedByActiveDuplicate capability 적재 → 복구 버튼 disable + tooltip("같은 이름·버전의 활성 자원이 이미 있어 복구할 수 없어요").
  • 서버 가드(Duplicate*Exception throw)는 안전망 유지, capability 와 동일 질의 SSOT 공유(hasActiveDuplicateKey() 류 추출).
  • 비용 — deleted 행마다 활성-동일키 per-row 질의(N+1 회피 위해 batch 필요) + 일부 도메인 toResponse static→instance 전환.

안 B — 등록 시점에 충돌을 원천 차단

  • 충돌은 등록 때 "그래도 진행"으로 의도적으로 만든 것 → 등록 nudge 의 replace 가 휴지통 자원을 purge 하면 애초에 restore 충돌이 안 생긴다. 또는 "그래도 진행" 시 "기존 삭제본은 복구 불가"를 명시.
  • restore 자체는 그대로 두고 모순 발생을 입구에서 막는 접근.

보류 결정 (2026-05-31, 사용자)

근거 3가지가 Notion 본문에 기록돼 있다 — ① restore 가 이미 async(data-async-submit)라 409 가 full-page 가 아닌 alert/modal 로 떠 이미 완화 ② per-row 중복키 질의 + toResponse static→instance 리팩터 비용 ③ Subprogram 만 차단하면 새 비대칭. → 재개 시점 미정으로 보류.
본 보고서는 이 ①의 "이미 완화" 전제가 현재 코드에서 정말 성립하는지를 표면별로 실측해 검증한다(§4).
🎬 restore 중복키 시뮬레이터 — 도메인 × 표면에 따라 무엇이 렌더되나

도메인과 "어디서 복구를 눌렀는지(표면)"를 골라 복구를 시도하면, 그 409 가 error.html 전체페이지 이탈인지 inline ErrorModal 인지 실제 코드 기준으로 시뮬레이션한다. 이게 "이미 완화됐다"가 어디까지 참인지를 보여준다.

도메인 복구 표면

판정 기준: 복구 폼이 data-async-submit 을 가지면 fetch → AsyncSubmitResult.onRejected → ErrorModal(inline 스타일 모달). 없으면 form.requestSubmit() → native 내비게이션 → 409 가 WebExceptionHandler → error.html 전체페이지로 이탈. ISO 는 dup-key 가드 자체가 없어 충돌 불가.

§5. 충돌 도달 가능성 — 정상 흐름에서 진짜 일어나나

"필요한가"를 따지려면 이 충돌이 실제 사용자 흐름에서 닿는지를 봐야 한다. 도메인별로 충돌을 만들려면 "삭제본이 있는데 같은 키의 활성본을 새로 만들 수 있어야" 한다.

BoardModel — nudge 없이 도달 (가장 노출 큼)

  • 등록 가드 BoardModelMetadataService.java:111 는 existsByVendorAndModelNameAndIsDeletedFalse = 활성만 검사. 삭제본(isDeleted=true)은 카운트 안 됨 → 같은 (vendor, modelName) 활성본을 그냥 재등록 가능.
  • BoardModel 은 파일 업로드/nudge 가 없는 단순 폼이라 "그래도 진행" 마찰조차 없다 → 충돌 생성이 가장 쉽다.
  • 그렇게 만든 활성본이 있는 채로 휴지통의 옛 board 를 복구 → BoardModelLifecycleService:124 가 409. 그리고 그게 error.html 전체페이지로 이탈한다(§4). board 는 도달 용이성·렌더 비일관 두 축에서 모두 최악.

업로드 자원(OS·BIOS·BMC·Subprogram) — nudge 경유, 렌더는 ErrorModal

  • 같은 키의 활성본을 새로 만들려면 등록 시 충돌 nudge 에서 "그래도 진행"을 택해야 한다(의도적). 도달은 가능하나 한 단계 마찰이 있다.
  • 도달하더라도 restore 409 는 async → ErrorModal(명확한 메시지)로 끝난다 — raw JSON 아님.
  • 안 B(nudge replace=purge 로 입구 차단)는 이 업로드 도메인에 한정된 접근이며, BoardModel 에는 nudge 자체가 없어 적용 불가. 즉 안 B 만으로는 가장 심한 board 케이스를 못 닫는다.
함의 — 충돌은 board 에서 마찰 없이 + 전체페이지 이탈로 닿고, 업로드 자원에서는 nudge 마찰 + inline ErrorModal 로 닿는다. 그러므로 "전 도메인 일괄 UI capability"보다, board 의 전체페이지 이탈만 inline ErrorModal 로 끌어올리는 국소 수리가 비용 대비 효과의 임계점이다.
§6. 판정 — R2-2-2/3 구현이 필요한가

R2-2-2/3 (5도메인 UI pre-disable capability) 불요 — 보류 유지/DROP

  • 촉발 원인이 거의 소멸 — raw JSON UX 파손은 6표면 중 5표면에서 async+ErrorModal 로 이미 해결. 남은 board 1곳은 슬라이스가 아니라 1줄로 닫힘.
  • 잔여 가치는 polish 수준 — async 화 이후 restore 409 는 "버튼 누르면 명확한 모달". R2-2-3 이 추가하는 건 "버튼을 아예 못 누르게(disable+tooltip)"인데, 명확한 모달이 이미 이유를 알려주므로 한계효용이 작다.
  • 비용은 작지 않음 — deleted 행마다 per-row 활성-동일키 질의(N+1 회피 batch 설계) + 5도메인 toResponse static→instance 리팩터. CLAUDE.md 의 over-abstraction/churn 경계에 정확히 걸린다.
  • 안 B(nudge replace=purge)도 단독으론 불완전 — board 엔 nudge 가 없어 적용 불가. 가장 심한 케이스를 못 막는 해법은 채택 근거가 약하다.

잔존 결함 1건 국소 hotfix 권고

BoardModel 관리목록 restore = native POST → error.html 전체페이지 이탈. management/board/list.html:193 의 복구 폼에 형제 5폼과 동일하게 data-async-submit="" 한 줄을 추가하면, 그 409 가 inline ErrorModal 로 수렴한다. cascade 복구(BoardModelLifecycleService:131, 자식 동일키 충돌 시 전체 롤백 → 409)도 같은 폼이라 함께 구제된다. 이것이 이 묶음에서 실제로 갚아야 할 유일한 빚이다.

적대적 점검 (반대 입장 검토)

반론응답
"버튼 누르고 모달 뜨는 것보다 disable 이 더 좋은 UX 아닌가"맞다(국소적으로). 그러나 그 이득은 polish 이고, R2-2-3 의 5도메인 질의·리팩터 비용을 정당화하지 못한다. 부모-자식 차단(R2-2)은 부모 상태가 list 응답에 이미 있어 공짜였지만, dup-key 는 별도 per-row 질의를 새로 사야 한다 — 성격이 다르다.
"board 만 async 화하면 또 다른 비대칭 아닌가"반대다. 지금이 비대칭(5 async / 1 native)이고, board async 화가 대칭을 회복한다. 6표면 전부 ErrorModal 로 일관.
"async 라도 결국 클릭 후 거절 — R2-2 원칙(UI 1차 차단) 위반 아닌가"R2-2 원칙은 "정상 UX 흐름의 논리 모순을 UI 가 사전 차단"이다. dup-key restore 충돌은 삭제본+동일키 활성본이 공존하는 비정상에 가까운 상태에서만 나며, 빈도가 낮다. 명확한 모달로 사후 안내하는 것이 안전망으로 충분하고, 사전 차단은 비용이 이득을 넘는다.
