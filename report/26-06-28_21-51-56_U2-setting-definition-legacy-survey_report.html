한눈에 — 무엇이고, 어디에 있나

세팅 정의서는 "프로비저닝되는 서버를 어떤 단계로, 각 단계를 어떻게 구성할 것인가" 를 담은 재사용 가능한 명세(템플릿)다. 레거시에는 이 개념이 풍부하게 구현돼 있었다 — JSON 단일 컬럼 영속, Jackson 다형 단계 4종, 2단계 resolver/builder dispatch(16+ 클래스), 아코디언 기반 CRUD 화면. 그러나 현재 renew/main 브랜치에는 세팅 정의서 구현이 전무하다. 이번 조사는 그 "풍부했던 옛 형태" 를 정확히 떠내, U2 가 무엇을 계승·정제·축소할지 판단할 토대를 만든다.

01

세팅 정의서란 무엇인가

물리 서버 한 대를 프로비저닝한다는 것은 단일 동작이 아니라 순서가 정해진 여러 공정 단계의 연쇄다 — 펌웨어 업데이트 → 펌웨어 설정 → OS 설치 → OS 후처리. 세팅 정의서는 이 연쇄를 사용자가 미리 한 번 정의해 저장해 두는 자원이다. 핵심 가치는 "서버 1대를 직접 손으로 설정" 이 아니라 "설정 절차를 템플릿으로 만들어 여러 서버에 일관되게 재적용" 하는 데 있다.

레거시에서 이 자원의 이름은 ServerSetting("세팅 주문서")이었다. 한 주문서가 ① 식별 명칭(name) + ② 순서 있는 단계 목록(process) + ③ 진행 상태(status) 를 보유했고, PXE 부팅 흐름에서 서버 노드(ServerNode)가 이 주문서를 FK 로 참조해 currentStepIndex 와 함께 "지금 어떤 단계를 실행할 차례인가" 를 결정했다. 즉 정의서는 프로비저닝 엔진이 읽어 들이는 실행 대본이었다.

02

영속 — JSON 단일 컬럼 전략

레거시 영속 모델의 가장 두드러진 선택은 단계 목록 전체를 정규화 테이블이 아니라 JSON 문자열 한 컬럼에 통째로 저장한 점이다. 단계 타입마다 필드 구조가 천차만별(보드 ID 3개 vs 파티션·유저·패키지 트리)이라, 관계형 정규화로 풀면 테이블이 폭증하므로 다형 객체 그래프를 JSON 으로 직렬화하는 길을 택했다. 

// server_setting 테이블 — 단 4개 의미 컬럼
ServerSetting (BaseTimeEntity)
  ├ id      : Long (IDENTITY)
  ├ name    : String        // 주문서 식별 명칭
  ├ process : JSON          // ← SettingProcess 그래프 전체를 직렬화
  └ status  : SettingStatus // EnumType.STRING

// 직렬화 경로
SettingProcess(record)  ──@Convert──▶  SettingProcessConverter  ──▶  "process" 컬럼(JSON 문자열)
  └ processList: List<AbstractSettingProcess>  @NotEmpty

SettingProcessConverter 는 JPA AttributeConverter(autoApply=true)로, Jackson 3 런타임(tools.jackson.databind.ObjectMapper)을 Spring 컨텍스트에서 주입받아 직렬화/역직렬화한다. 엔티티는 update(name, settingProcess) 만 노출하고 dirty-checking 으로 UPDATE 가 발행되며, status 는 수정 메서드가 건드리지 않는다.

주목할 디테일 — 읽기 경로의 레거시 판별자 마이그레이션

converter 의 읽기 경로(convertToEntityAttribute)에는 스키마 진화 보정 로직이 들어 있다. 과거 OS 설치 판별자로 버전 없는 "ROCKY_LINUX" 단일 값을 썼는데, 현재는 ROCKY_LINUX_8/9/10 로 세분화됐다. 그래서 DB 에서 옛 JSON 을 읽을 때 트리를 순회하며 osMetadata.osVersion 접두사("8."/"9."/"10.")로 새 판별자를 유추해 승격하고, 파싱 실패 시 ROCKY_LINUX_9 로 폴백+경고 로그를 남긴다. 쓰기 경로는 이미 신 판별자라 보정 불필요.

시사점

JSON 컬럼 전략은 스키마 단순(테이블 1개) ↔ 단계 단위 질의 불가 + 판별자 진화를 코드로 보정해야 함 의 트레이드오프를 안는다. 이는 U1 에서 hardware_spec/software_spec 을 JSON 컬럼으로 채택한 선례와 같은 결의 선택이다.

03

다형 단계 모델 (4종)

JSON 으로 저장되는 각 단계는 AbstractSettingProcess 추상 클래스의 구현체다. Jackson @JsonTypeInfo(property="type") + @JsonSubTypes 로 판별자 기반 다형 직렬화를 구현하고, Comparable 로 SettingProcessStep.order 기준 정렬을 보장한다(서비스에서 .sorted()).

type 판별자클래스order보유 데이터 / 검증구현도
BASIC_UPDATESBasicUpdate1BoardModel·BIOS·BMC DTO. 생성자에서 BIOS/BMC 의 compatibleModel() 이 선택 보드와 일치하는지 검증.구현
BASIC_SETTINGBasicSetting2없음. 필드 0개 스텁 — "BIOS 설정 단계가 포함된다" 는 사실만 기록.스텁
OS_INSTALLATIONOSInstallation4OSMetadataDTO + 도메인 OSInstallation(파티션·유저·루트PW·타임존·환경·패키지). 생성자에서 isCompatible(osName,osVersion) 검증.구현
OS_SETTINGOSSetting5OSMetadataDTO + 도메인 OSSetting(SELinux·서비스·패키지). isCompatible 검증. RHEL 계열만.RHEL만

order 3 = BASIC_TEST("기본 테스트")로 예약·주석 처리(미구현). enum 에 빈 슬롯을 남겨 둔 형태.

SettingProcessStep enum 은 (order, description) 쌍을 갖는다 — BASIC_UPDATE(1) · BASIC_SETTING(2) · OS_INSTALLATION(4) · OS_SETTING(5). SettingStatus 는 PENDING → IN_PROGRESS → COMPLETED / PARTIAL_SUCCESS / FAILED / CANCELED 6종이나, PXE 흐름과의 상태 전이 연동은 미구현이었다(생성 시 PENDING 고정).

도메인 OS 설치/후처리 스펙(domain.os.model.installation/setting)은 별도 계층에 풍부하게 존재한다 — OSInstallation · LinuxInstallation · RHELBasedInstallation · CentOS7/RockyLinux8/9/10/UbuntuInstallation · Partition · User · RootPassword · Timezone · Environment, 후처리 측 OSSetting · RHELBasedSetting · CentOS7/RockyLinux8/9/10Setting · ServiceDirective. 정의서 모델은 이 도메인 스펙을 래핑해 메타데이터와의 호환성만 한 번 더 검증하는 application 계층 어댑터였다.

04

2단계 다형 dispatch (요청 → 도메인 모델)

프론트가 보낸 다형 요청(AbstractProcessRequest)을 도메인 모델로 변환하는 일을 두 겹의 Strategy 패턴이 맡았다. 분기문(switch) 증식 대신 List<SPI> 주입 + supports() first-match 로 dispatch하는, 현 프로젝트의 다형성 철학과 같은 구조다. 

// Tier 1 — 단계 타입별 dispatch
SettingService
  └ List<SettingProcessResolver> resolvers   // Spring 이 @Component 전부 주입
       resolvers.stream().filter(r -> r.supports(req)).findFirst()
         ├ BasicUpdateResolver    // board/BIOS/BMC 조회 + 호환성
         ├ BasicSettingResolver   // new BasicSetting() 스텁 반환
         ├ OSInstallationResolver ──┐
         └ OSSettingResolver      ──┤
                                    ▼
// Tier 2 — OS 종류·버전별 builder 선택
   List<OSInstallationBuilder> builders
     builders.stream().filter(b -> b.supports(req, osMetadata)).findFirst()
       supports() = (req instanceof RHEL/UbuntuRequest)
                  && osMetadata.osName == ROCKY_LINUX/CENTOS/UBUNTU
                  && osVersion.startsWith("8."/"9."/"10."/"22.04")
         ├ AbstractRHELInstallationBuilder (template method)
         │    ├ CentOS7Builder · RockyLinux8/9/10Builder
         │    └ 환경·패키지그룹 조회 + 교차검증 → buildRHELInstallation() hook
         └ Ubuntu2204Builder (RHEL 계층 미상속, hostname·packages 직접)

핵심 설계 답: "OS 선택 → 구체 설치 빌더" 매핑은 registry/switch 가 아니라, 각 빌더의 supports(요청타입, OSName, osVersion 접두사) 술어로 결정됐다. 새 OS 계열은 빌더 @Component 하나를 추가하면 자동 편입된다. 요청 측도 2단계 다형이다 — AbstractProcessRequest(type 판별자) 아래 OSInstallationRequest 가 다시 osFamily(RHEL_BASED/DEBIAN_BASED) 2차 판별자로 갈린다.

규모 주의

이 2단계 dispatch 는 resolver 4 + builder SPI 2 + 추상 3 + 구체 빌더 6~7 = 16+ 클래스로 비대하다. 확장성은 좋으나, 현재 renew/main 에 포팅된 OS 도메인 모델이 0 인 상태에서 이를 전부 옮기는 것은 큰 비용 — §08 의 핵심 긴장점이다.

05

검증 3중 모델

레거시는 입력 검증을 세 겹으로 쌓았고, 각 겹의 실패가 사용자에게 다르게 표현됐다.

겹수단표현 / 결과
① 형식 검증Bean Validation (@Valid)필드 누락·패턴 위반 → 400. 경로 포맷 processList[i].partitions.
② 강성 검증Resolver 내부 → FieldValidationException존재하지 않는 보드/OS, 호환성 불일치, SELinux 모드, 패키지명 셸 인젝션 방어 등. 예외에 processList[i]. 인덱스 prefix 부착해 ①과 경로 포맷 일치.
③ 비차단 경고OSRepoValidationService패키지/서비스 이름을 OS 저장소 인덱스와 대조 → ValidationWarning{field,value,message} 목록. 저장 차단 안 함 — 프론트가 모달로 안내 후 사용자가 "이해하고 진행".

도메인 계층은 DomainValidationException 의 reason enum 만 던지고, resolver 가 reason → 필드명 매핑 switch(exhaustive)를 단독 소유해 FieldValidationException(field, msg) 로 재포장했다. 새 reason 추가 시 컴파일러 exhaustiveness 경고가 매핑 누락을 잡아주는 구조다. 비차단 경고(③)는 "차단할 정도는 아니나 사용자가 알면 좋은" 정보(저장소에 없는 패키지명 등)를 예외가 아닌 데이터로 흘려보내는, UX 친화적 분리였다.

06

CRUD UX 의 형태

컨트롤러는 /pxe/v1/setting 의 혼합형(@Controller + @ResponseBody)이었다. SSR 페이지 4종과 XHR API 를 함께 제공했다.

엔드포인트종류역할
GET /settingSSR목록(생성일 내림차순). 단계 배지·상태 배지.
GET /setting/{id}SSR상세(읽기 전용). PENDING 일 때만 수정 버튼 노출.
GET /setting/newSSR작성 폼. 단계 enum·보드·OS 메타·파일시스템 선택지 주입.
GET /setting/{id}/editSSR수정 폼. PENDING 아니면 redirect 차단. 기존값 buildInitialJson 으로 pre-fill(비밀번호·타임존 제외).
POST /setting/api/newXHR생성. 201 + Location /setting/{id}.
PUT /setting/api/{id}XHR수정. findByIdForUpdate 비관적 락 + PENDING 재확인.
POST /setting/api/validateXHR사전 검증 — 저장 없이 경고만 반환(③).
GET /setting/api/default-partitionsXHROS 별 권장 파티션 프리셋.

폼의 동작 — 동적 다형 입력

  • 아코디언 + 단계 활성화 체크박스 — 단계마다 아코디언 아이템 1개. step-enable-chk 체크 시 해당 본문이 opacity:.4/pointer-events:none → 활성으로 전환. 체크된 단계만 processList 에 실림. 최소 1개 강제(클라이언트).
  • OS family fragment 동적 스왑 — rhel-based/debian-based fragment 가 둘 다 포함돼 있고, OS 버전 선택 시 data-os-family 로 해당 pane 만 hidden=false. Rocky 10 전용 allowSshRoot 같은 버전 특화 박스는 data-os-version-prefix 로 한 번 더 토글.
  • 동적 테이블 — 파티션·유저·추가패키지·서비스를 행 추가/삭제로 구성. "기본 파티션 자동 생성" 버튼이 default-partitions 호출.
  • 사전 검증 모달 — 저장 전 /api/validate 호출, warnings 있으면 모달로 "취소하고 수정 / 이해하고 진행". 비차단(③).
  • 수정 시 비밀번호 보존 — 폼은 비밀번호를 pre-fill 하지 않고, 빈 채 제출하면 keepExistingPassword:true 를 보낸다. 서비스가 patchKeepExistingPasswords 로 기존 저장값을 주입해 resolver/도메인이 항상 유효한 비밀번호를 받게 했다.
UX 요지

화면의 본질은 "가변 개수의 다형 단계를, 각 단계마다 다른 폼으로, 한 페이지에서 조립" 하는 것. JS 가 폴리모픽 JSON payload 를 만들어 단일 @RequestBody 로 전송하는 구조였다.

07

현재 브랜치 출발점과 간극

조사의 마지막 축은 "이 옛 형태가 현재 renew/main 어디에 서 있나" 다. 결론은 두 가지다.

A세팅 정의서 구현은 현재 브랜치에 전무하다

SettingDefinition · ServerSetting · SettingProcess 클래스 검색 결과 0건이다. 현재 provisioning/ 패키지는 전부 BIOS attribute 편집·BMC password 도구 + GuestServerController 로, 세팅 정의서와는 별개 관심사다. CLAUDE.md 의 "provisioning/setting(SettingDefinition + SettingProcess 다형)" 은 계획만 있고 코드는 미착수 상태다. 도메인 OS 모델(domain.os.model.installation/setting)도 renew/main 으로 아직 포팅되지 않았다.

B단, 실행 측 단계 모델은 이미 정제된 형태로 존재한다

U1 이 만든 execution 의 ProvisioningPhase(7) + ProvisioningPhaseStep(13)은 레거시 SettingProcessStep(5)의 확장·정제판이다. 둘의 대응:

레거시 SettingProcessStep현재 execution Phase / Step
BASIC_UPDATE (order 1)FIRMWARE_UPDATING → {BIOS_UPDATING, BMC_UPDATING}
BASIC_SETTING (order 2)FIRMWARE_SETTING → {BIOS_SETTING, BMC_SETTING}
(BASIC_TEST 예약, order 3)TESTING
OS_INSTALLATION (order 4)OS_INSTALLING
OS_SETTING (order 5)OS_SETTING
신규 (없음)BOOTSTRAPPING, DIAGNOSE_LINUX (자동 단계)
이미 있는 절반 — 실행(execution)
"서버가 어떤 단계를 거치는가" 의 어휘(Phase/Step enum)는 U1 에서 정의됨. 진행 추적 엔티티(ProvisioningProgress·SetupStep)도 존재.
비어 있는 절반 — 정의(setting)
"사용자가 각 단계를 어떻게 구성할지 입력하는 명세" 가 없음. 정의서 ↔ 게스트 서버 연결도 U1 D5 에서 nullable 자리로만 예약, 미구현.

즉 U2 는 백지에서 시작하는 게 아니라, "실행 어휘는 이미 선 execution 위에, 그 어휘를 사용자가 채워 넣는 정의(setting) 레이어를 얹는" 작업이다. 옛 SettingProcessStep 을 그대로 부활시키면 현재 ProvisioningPhase/Step 과 단계 모델이 이중화(drift)되므로, 이 연결을 어떻게 하나의 SSOT 로 묶을지가 설계의 첫 분기점이 된다.

08

CP1 에서 풀 설계 긴장점 (예고)

아래는 조사에서 드러난, plan(CP1)에서 대안 비교 + 적대적 비판으로 정식 결정해야 할 지점들이다. 여기서는 평가하지 않고, 무엇이 쟁점인지만 못박는다.

#긴장점무엇을 결정해야 하나
T1단계 모델 SSOT 통일레거시 SettingProcessStep 을 부활시키지 말고 현재 ProvisioningPhase/Step 을 재사용해야 drift 없음. 단 정의서가 다루는 건 "사용자 구성 입력이 필요한 단계" 의 부분집합(BOOTSTRAPPING/DIAGNOSE 는 자동 단계라 입력 없음) — 어떻게 부분집합을 표현할지.
T2JSON 단일 컬럼 vs 정규화레거시 JSON 한 컬럼 전략 계승 여부. U1 의 hardware_spec/software_spec JSON 선례와 일관되나, 단계 단위 질의·무결성을 포기. 단계만 정규화하는 절충도 후보.
T32-tier builder 16+ 클래스의 무게현재 OS 도메인 모델이 0 포팅. 전부 옮길지 vs MVP(식별·골격 + OS 1계열 또는 OS 단계 자체를 후속 슬라이스로 이연)로 좁혀 "갈라지는 시점에 분리" 원칙을 지킬지.
T4정의서 ↔ 게스트 서버 연결(척추)U1 이 spine 을 N:1 nullable 로 예약만 함(D5). U2 에서 실제 연결을 켤지, 또 다른 슬라이스로 미룰지. purpose-fit 검토가 "자동화의 척추" 로 지목한 지점.
T5스텁 단계 범위레거시 BasicSetting(BIOS 설정)·OS_SETTING 등 미완·부분 구현 단계를 U2 에서 어디까지 실제 구현할지(메모리상 Phase 5 BIOS Setting 과제와도 연동).
T6패키지 배치 / 명명U1 이 게스트 서버를 execution/ 에 둠. 정의서는 provisioning/setting(CLAUDE.md 안) vs 다른 위치 — 영역 경계 재확인 필요.
