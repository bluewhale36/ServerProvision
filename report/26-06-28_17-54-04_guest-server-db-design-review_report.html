검토 요약 · 판정

결론부터 적는다. 데이터 모델링의 방향(정체성·인벤토리·네트워크·진행상태 분리, VO 타입화)은 타당하며, 가장 중요한 등록 경로는 Notion U1 명세와 글자 단위로 일치한다. 다만 양산(실사용) 진입 전에 반드시 고쳐야 할 구조적 결함 3건이 있고, 설계 보고서 v2 가 그린 목표 모델과 실제 구현 사이에는 스코프 차이로 설명되는 간극과, 스코프로는 설명 안 되는 누락이 섞여 있다.

잘 된 점
반드시 먼저 고칠 3건

F1 (치명) — SetupStep 이 @OneToOne 으로 묶여 서버당 단계 이력이 한 행만 저장된다. "단계 실행이력" 의 본질(서버당 다수 행)과 정면 충돌하며, 자기 리포지토리의 List 조회 메서드와도 모순된다. → @ManyToOne.
F2 (높음) — ProvisioningProgress 가 @LastModifiedDate 를 쓰면서 감사 리스너(@EntityListeners)를 달지 않아 last_transition_at 이 채워지지 않는데 nullable=false → 최초 INSERT 가 깨진다.
F3 (높음) — GuestServer.modelName 컬럼은 어디에서도 기록되지 않는 고아 필드다(항상 null). 같은 화면에 노출되는 GuestServerCustom.productModelName 과 의미가 겹친다.

스코프에 대한 핵심 인식

Notion "2단계 · 사용 페이지 (a)" 는 이 작업을 "골격(skeleton)" 으로 명시하고 범위를 Guest 서버 · 상세 · NIC 바인딩 · 단계 실행이력 · 진행상태 + 공정 단계 열거형 5종으로 한정한다. 따라서 보고서 v2 의 9개 테이블 중 bmc_binding · bmc_credential_ref · location_assignment · transition_log 4종이 미구현인 것은 결함이 아니라 의도된 이연이다. 반면 같은 스코프 안에 있으면서도 누락·오류인 항목(F1~F3 등)은 별개로 다룬다.

01

프로젝트 맥락 · 구현 현황

프로그램이 하려는 일

Provisioning Server 는 생산 공정에 투입되는 물리 서버(=게스트 서버)를 PXE 부팅으로 받아, BIOS/BMC 업데이트 → BIOS 세팅 → BMC 세팅 → OS 설치 → OS 세팅 의 공정을 자동화한다(Notion KPI 초안). 서버마다 하드웨어가 달라 "이 서버에는 어떤 생산 자료가 필요한가" 를 자동 식별해야 하므로, 게스트 서버를 DB 에 정확히 데이터화하는 것이 본 작업(2단계 a)의 목적이다.

게스트 서버는 두 패키지에 걸쳐 산다

이 점이 구조 파악의 출발점이다. 같은 "게스트 서버" 애그리거트가 물리적으로 두 패키지에 나뉘어 있다.

패키지책임주요 산출물
execution 영속 모델 + 부팅 등록(쓰기) GuestServer · GuestServerDetail · GuestServerCustom · HostNicBinding · ProvisioningProgress · SetupStep · 6 Repository · MacAddressVO/IpAddressVO(+Converter) · 5 Enum · ServerRegistrationService · ExecutionRestController
provisioning 운영자 조회·인라인 수정(읽기/UI) GuestServerController · GuestServerQueryService · GuestServerCommandService · GuestServer{Summary,Detail}Response · UpdateGuestServerRequest · server-list/detail.html

provisioning 의 서비스가 execution 의 리포지토리를 직접 주입해 쓴다. 기능별(feature-first) 패키징 원칙과 CLAUDE.md("server 는 provisioning, execution 은 예약") 사이에서 경계가 모호해진 상태다(→ F7).

등록 흐름 — Notion U1 과의 대조

iPXE 가 boot.ipxe 를 해석하면 아래 체인 URL 로 게스트 서버 정보를 보고한다. 이는 Notion U1 페이지에 적힌 명세 그대로다. 

# Notion U1 명세 (= 실제 구현된 컨트롤러 매핑)
chain http://localhost:7779/api/pxe/v1/boot
        ?macAddress=${mac} &ipAddress=${ip} &systemUUID=${uuid}
        &vendor=${manufacturer} &boardModel=${product}
# 예: ${manufacturer}=Giga Computing, ${product}=MS73-HB1-000

ServerRegistrationService.initialRegistry() 는 이 한 요청으로 guest_server(id, system_uuid), guest_server_detail(vendor, board_model, discovery_stage=IPXE_REGISTERED), host_nic_binding(host_mac, lan_ip, ip_source=DHCP, is_primary=true) 를 적재한다 — U1 의 적재 명세와 정확히 일치한다(추가로 빈 guest_server_custom 1행 생성). 등록 경로의 충실도는 본 구현에서 가장 잘 된 부분이다(→ §05).

진행 단계 모델 (열거형)

ProvisioningPhase(7) = BOOTSTRAPPING · DIAGNOSE_LINUX · FIRMWARE_UPDATING · FIRMWARE_SETTING · OS_INSTALLING · OS_SETTING · TESTING. 각 Phase 아래 ProvisioningPhaseStep(12)이 enum 상수마다 소속 Phase 를 들고 있다(예: NETWORK_ALLOCATING → BOOTSTRAPPING). 공정 5단계 + 부팅/진단/테스트가 열거형으로 잘 골격화되어 있다.

02

스코프 정합 — 보고서 ↔ Notion ↔ 구현

설계 문서는 두 단계로 진화했다. v1(8 테이블, guest_server 에 신원 집중) → v2(9 테이블, 신원/인벤토리 분리 + NIC 1:N + JSON 스펙). 실제 구현은 v2 의 방향을 따르되 Notion 2단계(a) 의 골격 스코프(5종)로 범위를 좁혔다. 아래 표가 세 축을 한눈에 맞춘 것이다.

보고서 자체에 대한 권고

보고서 v2 는 목표 설계로서는 잘 쓰였다(분리 근거·낙관적 락 위치·JSON vs 컬럼 트레이드오프가 설득력 있다). 다만 지금 시점에 읽으면 "9 테이블이 다 있는 것처럼" 읽혀 구현과 어긋난다. v2 보고서에 현재 스코프(골격 5종) vs 이연(4종) 표식과, 구현이 보고서를 벗어난 3가지 결정 (board_model FK화 · guest_server_custom 신설 · IPv4 전용)을 changelog 로 반영하면 단독으로도 신뢰할 수 있는 문서가 된다.

03

엔티티별 검토

GuestServer — 가벼운 신원

id · systemUUID · name · modelName · memo + 감사필드. v2 의 "정체성/인벤토리 분리" 를 충실히 따랐다. 등록 시 name/modelName 을 모르므로 보고서의 NN 제약을 풀어 nullable 로 둔 것은 올바른 보정이다 (보고서가 내부 모순이었고, 구현이 더 정확하다). name 은 unique=true + 인라인 수정으로 채운다. 다만 modelName 은 어떤 코드 경로에서도 set 되지 않는 고아 컬럼이다(→ F3).

GuestServerDetail — 하드웨어·소프트웨어 인벤토리

vendor · boardModel(FK) · boardSerial · discoveryStage · @Version. 두 가지가 눈에 띈다. ① 보고서가 자유 문자열로 둔 board_model 을 관리 카탈로그 BoardModel 로의 FK 로 격상한 것은 설계 개선이다 — 등록 시 미등록 모델을 BoardModelNotFoundException 으로 거절해 검증된 인벤토리와 묶인다. ② 그러나 BoardModel 이 이미 vendor 를 보유하므로 detail.vendor 는 파생 가능한 중복이다(→ F5). 또한 페이지 목적("하드웨어 스펙 자동 식별")의 핵심인 hardware_spec/software_spec 이 빠져 있다(→ F6).

GuestServerCustom — 사내 식별자 슬롯 보고서 외

productModelName(7자) · productSerialNumber(20자). "진단 리눅스에서 ipmitool 로 하드웨어에 기록할 운영자 지정 모델/시리얼" 이라는 개념은 합리적이다 — 하드웨어가 보고한 값(detail.boardSerial)과 운영자가 부여해 새겨 넣을 값을 분리하는 것은 의미가 있다. 다만 (a) 두 보고서·Notion 어디에도 없어 문서화가 필요하고(I2), (b) productModelName 이 GuestServer.modelName 과 의미가 겹친다(F3). 둘 중 하나로 단일화해야 §중복 금지 원칙에 맞는다.

HostNicBinding — 호스트 NIC (1:N)

@ManyToOne 으로 1:N 을 올바로 구현했고 is_primary · bond_group 도 반영했다. MAC/IP 를 VO 로 받아 정규화·검증한다(우수). "서버당 primary 1개" 는 보고서대로 DB 가 아닌 어플리케이션이 보장하는 설계지만, 아직 그 보장 로직(기존 primary unset 트랜잭션)은 없다 — 단일 NIC 등록만 하는 현 골격에서는 문제되지 않으나 NIC 추가 기능 도입 시 필요하다. 감사필드 비상속은 F9.

ProvisioningProgress — 큰 단계 진행 상태 (1:1)

@OneToOne 1:1 은 올바르다. 그러나 (a) @LastModifiedDate 가 감사 리스너 없이 선언돼 동작하지 않으며 nullable=false 와 결합해 INSERT 를 깨고(F2), (b) 보고서가 명시한 @Version 낙관적 락이 빠졌다(F4). 진행 상태는 "여러 주체가 자주 갱신" 한다는 보고서 전제상 락이 가장 필요한 테이블이다.

SetupStep — 세부 단계 체크포인트

역할(Phase 내 개별 작업의 성공/실패/스킵 + 시각 기록)은 잘 잡혔다. 그러나 관계가 @OneToOne(guestServer) 으로 잘못 매핑돼 서버당 한 행만 들어간다(F1, 치명). "단계 실행이력" 의 정의(서버당 12 step × 다수 시도)와 자기 리포지토리의 List 반환 조회와 모두 모순된다. 부차적으로 phaseCode 는 stepCode.getPhaseType() 으로 파생 가능해 드리프트 여지가 있다(F10).

04

발견 사항 (심각도별)

버튼으로 심각도를 필터링할 수 있다. 카드를 클릭하면 근거·위치·수정안이 펼쳐진다.

F1치명SetupStep 이 @OneToOne — 서버당 단계 이력이 1행으로 잘린다▶

SetupStep.java:25

SetupStep.guestServer 가 @OneToOne 으로 매핑돼 있어 Hibernate 가 guest_server_id 에 UNIQUE 제약을 건다. 결과적으로 한 서버는 단 하나의 setup_step 행만 가질 수 있다. 그러나 ProvisioningPhaseStep 은 12개이고, 각 step 은 PENDING→RUNNING→ SUCCEEDED/FAILED/SKIPPED 로 여러 번 기록될 수 있다. Notion 도 이 테이블을 "단계 실행이력" (복수)으로 규정한다.

결정적으로, 자기 자신의 리포지토리가 이미 다수 행을 전제로 짜여 있다:

현재 (모순)
@OneToOne ... private GuestServer guestServer;
→ 서버당 1 step 만 INSERT 가능
기대
findAllByServerIdOrderByStartedAt → List<SetupStep>
→ 서버당 N step 을 시각순 조회
수정안

@OneToOne → @ManyToOne(fetch = LAZY, optional = false). 두 번째 step 을 저장하는 통합 테스트를 함께 추가해 회귀를 잡는다(현재 테스트 0건이라 이 결함이 드러나지 않았다).

F2높음ProvisioningProgress 의 @LastModifiedDate 가 동작하지 않아 INSERT 가 깨진다▶

ProvisioningProgress.java:8, 36-38

이 엔티티는 @LastModifiedDate 를 쓰지만 BaseTimeEntity 를 상속하지도, @EntityListeners(AuditingEntityListener.class) 를 선언하지도 않는다. @EnableJpaAuditing 은 전역으로 켜져 있어도, 감사 콜백은 엔티티에 리스너가 부착된 경우에만 동작한다(본 프로젝트는 BaseTimeEntity 의 @EntityListeners 로 부착). 따라서 lastTransitionAt 은 자동으로 채워지지 않는다.

그런데 컬럼은 @Column(name="last_transition_at", nullable=false) 다. 진행 상태 행을 처음 저장하는 순간 값이 null → NOT NULL 위반으로 INSERT 가 실패한다(또는 DB 설정에 따라 조용히 null).

수정안

세 갈래 중 택1 — ① ProvisioningProgress extends BaseTimeEntity (createdAt/updatedAt 일관성도 동시 확보, F9 해소), ② 클래스에 @EntityListeners(AuditingEntityListener.class) 직접 부착, ③ 감사에 의존하지 말고 전이 서비스에서 lastTransitionAt 을 명시적으로 set. ①을 권한다.

F3높음GuestServer.modelName 은 기록되지 않는 고아 컬럼 — productModelName 과 의미 중복▶

GuestServer.java:27-28, 36-39 · GuestServerCustom.java:32-33

전체 코드를 검색해도 GuestServer.modelName 에 값을 넣는 경로가 없다. 등록 빌더는 id + systemUUID 만 채우고, 인라인 수정 updateBasicInfo(name, memo) 는 name/memo 만 건드린다. 즉 이 컬럼은 항상 null인데, 상세 응답 (GuestServerDetailResponse.modelName)으로 버젓이 노출된다.

동시에 GuestServerCustom.productModelName(인라인 수정 가능, 같은 상세 화면에 노출)이 사실상 동일한 "사내 모델명" 을 담당한다. 길이도 128 vs 7 로 엇갈려 혼동을 키운다. 이는 §중복 금지 원칙(불가침)에 어긋난다.

수정안

의미를 하나로 정한다. "사내 모델명" 을 guest_server_custom 의 책임으로 본다면 GuestServer.modelName 컬럼과 응답 필드를 제거한다. 반대로 guest_server 에 남길 거라면 인라인 수정 폼이 이 필드를 쓰도록 배선하고 productModelName 과의 역할 차이를 문서화한다. (실서버 보고값이 아니라 운영자 부여값이므로 custom 쪽 단일화가 자연스럽다.)

F4중간ProvisioningProgress 에 @Version 낙관적 락 누락▶

ProvisioningProgress.java

보고서 v1·v2 모두 진행 상태 테이블에 version 을 두는 것을 명시한다("여러 주체가 갱신하므로 낙관적 락이 충돌 감지에 유용"). 구현 엔티티에는 @Version 이 없다. 공정이 비동기로 phase 를 전이시키는 Stage 4 진입 시 동시 갱신이 그대로 통과해 마지막 쓰기가 이긴다(lost update).

수정안

@Version private Long version; 추가. 흥미롭게도 거의 불변인 GuestServerDetail 에는 @Version 이 있는데 정작 잦게 갱신되는 progress 에는 없어 위치가 뒤바뀐 인상이다 — 보고서 의도대로 정렬.

F5중간vendor 중복 — detail.vendor 와 boardModel.vendor▶

GuestServerDetail.java:31-38 · BoardModel.java:32-34

GuestServerDetail 은 BoardModel 로의 FK 를 들면서 vendor 를 또 저장한다. 그런데 BoardModel 이 이미 vendor 를 보유한다 — vendor 는 보드 모델의 속성이므로 detail.getBoardModel().getVendor() 로 항상 파생된다. 따로 저장하면 detail.vendor=GIGABYTE, boardModel.vendor=ASUS 같은 모순이 가능해진다(드리프트 위험).

아이러니하게도 v2 보고서가 bmc_binding.vendor 를 제거한 이유가 바로 "vendor 중복 제거" 였는데, board_model 을 FK 로 격상하면서 그 중복이 detail↔boardModel 로 자리만 옮겨 재등장했다.

수정안

detail.vendor 를 제거하고 boardModel.getVendor() 로 파생하는 것이 정석. 조회 편의(목록 N+1)가 걱정이면 보고서 Q&A 가 제안한 대로 캐시하거나, 지금처럼 fetch join 으로 boardModel 을 함께 적재하면 된다(이미 그렇게 하고 있다). 유지한다면 "보드 교체 시 vendor 변동을 스냅샷으로 보존" 같은 명시적 사유를 문서화.

F6중간hardware_spec / software_spec(JSON) 미구현 — 페이지 목적의 핵심▶

GuestServerDetail.java

v2 보고서는 §03/§04 전체를 할애해 hardware_spec(CPU·메모리·디스크·NIC)와 software_spec(BIOS/BMC/옵션 펌웨어 버전) JSON 컬럼을 설계했다. 구현 엔티티에는 두 컬럼이 없다. 이는 단순 누락이라기보다, Notion 2단계(a) 의 명시 목적 "서버마다 하드웨어 스펙이 달라 필요한 생산 자료를 자동 식별" 의 데이터 근거가 아직 비어 있다는 뜻이다 — 무엇으로 "식별" 할지가 모델에 없다.

수정안

골격 단계의 의도된 이연일 수 있으나, 그렇다면 보고서/Notion 에 "JSON 스펙은 진단 리눅스 수집 단계와 함께 도입" 으로 명시하라. 도입 시에는 보고서 미결③(JSON 스키마 확정)대로 sealed record 와 함께 키 구조를 고정하고, DB 는 JSON_VALID CHECK 만 건다. tools.jackson 직렬화로 record↔JSON 매핑.

F7중간게스트 서버 애그리거트가 execution / provisioning 두 패키지로 분열▶

GuestServerQueryService.java:9-14 (provisioning → execution repo 직접 주입)

엔티티·리포지토리는 execution 에, 조회·수정 서비스 + 컨트롤러 + DTO + 템플릿은 provisioning 에 있다. provisioning 서비스가 execution 의 리포지토리 6종을 직접 주입한다. feature-first 패키징에서 한 애그리거트의 영속/응용 레이어가 두 기능 패키지로 갈라진 형태다. CLAUDE.md 는 "server 는 provisioning 영역, execution 은 예약(Stage 4 진입 시 결정)" 이라 적어 둔 상태라 소유권이 모호하다.

수정안

애그리거트의 단일 거처를 정한다. (a) 게스트 서버 전체를 provisioning.server 로 모으거나, (b) execution 을 게스트 서버 영속 도메인으로 확정하고 provisioning 은 그 응용 서비스만 호출(리포지토리 직접 접근 금지, execution 이 공개 서비스 API 제공)하도록 경계를 명문화. 어느 쪽이든 CLAUDE.md 의 패키지 구조 절을 갱신해야 한다.

F8중간테스트 0건 — 불가침 테스트 규율 미충족▶

src/test/.../execution → 없음

등록 서비스·조회/수정 서비스·컨트롤러에 대한 단위 테스트도, 사용자 액션 기반 통합 테스트(@WebMvcTest + MockMvc)도 전무하다. CLAUDE.md 의 "테스트 규율(불가침)" 은 Step 8 에서 두 레이어를 모두 요구한다. 실제로 F1(@OneToOne) 같은 결함이 통합 테스트 부재로 드러나지 않았다 — 두 번째 step 저장 시나리오가 있었다면 즉시 잡혔을 결함이다.

수정안

골격 단계임을 감안하더라도, CP4 상응 시점 전 최소: 등록 멱등성(이미 등록된 systemUUID 재부팅) · 미등록 boardModel 거절(404/도메인 예외) · 잘못된 MAC/IP/UUID 형식 거절 · 인라인 수정의 name/serial 중복 거절 · 상세 조회 매핑 + SetupStep 다수 행 저장/조회(F1 회귀 가드)를 추가한다.

F9낮음created_at / updated_at 적용이 엔티티마다 들쭉날쭉▶

HostNicBinding · ProvisioningProgress · SetupStep 은 BaseTimeEntity 미상속

GuestServer/Detail/Custom 은 BaseTimeEntity 를 상속해 감사필드를 갖지만, HostNicBinding(boundedAt 만) · ProvisioningProgress(inert 한 lastTransitionAt 만) · SetupStep(started/finished 만) 은 상속하지 않는다. v2 보고서 원칙5 는 "모든 본체 테이블에 created_at/updated_at" 을 표준으로 못박았다.

수정안

세 엔티티도 BaseTimeEntity 를 상속(ProvisioningProgress 는 F2 와 함께 해소). 단 setup_step 처럼 started/finished 가 별도 의미를 가지면 created_at 과 공존시키되 의미를 주석으로 구분.

F10낮음SetupStep.phaseCode 는 stepCode 로부터 파생 가능 (드리프트 여지)▶

SetupStep.java:31-36 · ProvisioningPhaseStep.java:48

ProvisioningPhaseStep 는 상수마다 소속 phaseType 을 들고 있다 (예: BIOS_UPDATING.getPhaseType() == FIRMWARE_UPDATING). 그런데 SetupStep 은 phaseCode 를 별도로 저장한다. phaseCode ≠ stepCode.getPhaseType() 인 행이 만들어질 수 있다.

수정안

조회 편의를 위한 비정규화면 유지하되 저장 시 phaseCode = stepCode.getPhaseType() 를 강제(빌더/도메인 메서드 단일 진입점). 아니면 컬럼을 빼고 파생.

F11낮음부팅 등록 입력 검증 부재 — 잘못된 파라미터가 500 으로 샌다▶

ExecutionRestController.java:19 · ServerRegistrationService.java:47

BootIPXEInfoRequest 에 검증 애너테이션이 없고, 서비스가 UUID.fromString(req.systemUUID()) 를 바로 호출한다. iPXE 가 ${uuid}/${mac} 를 비우거나 깨진 값으로 보내면 IllegalArgumentException → 500 으로 샌다(VO 가드도 IllegalArgumentException). PXE 클라이언트는 신뢰 대상이지만, 5종 파라미터 누락/형식 오류에 400 + 명확한 메시지를 주는 편이 진단에 낫다.

수정안

요청 record 에 형식 검증(또는 진입부 가드)을 두고, 형식 오류는 @ControllerAdvice 단일 진입점에서 400 으로 변환(컨트롤러 try/catch 증식 금지 원칙 준수).

F12낮음IpAddressVO IPv4 전용 + lan_ip VARCHAR(15) — 의도 확인 필요▶

IpAddressVO.java · HostNicBinding.java:42-44

VO 가 IPv4 만 허용하고 컬럼은 lan_ip VARCHAR(15) 다(보고서는 IPv6 수용용 VARCHAR(45) lan1_ip). "PXE/DHCP 는 IPv4 환경" 이라는 근거가 주석에 있어 의도된 결정으로 보이지만, 이는 향후 IPv6/BMC 관리망 표현을 영구히 막는다. 컬럼명도 lan_ip 로 spec(lan1_ip)과 다르다.

수정안

의도가 확정이면 보고서를 IPv4 전용으로 정정(미결 사항에서 닫기). bmc_binding(관리망 IP) 도입 시 동일 VO 를 재사용할지 여기서 결정해 두면 좋다.

F13낮음용어 드리프트 — CLAUDE.md 의 ProvisioningProgress(VO)/NodeStepExecution 과 불일치▶

CLAUDE.md §핵심 도메인 모델 (Provisioning)

CLAUDE.md 는 ProvisioningProgress 를 "Stage 4 의 VO, ProvisioningProgressService 가 NodeStepExecution 이력으로 계산해 반환" 으로, 단계 이력은 NodeStepExecution 으로 예고했다. 구현은 ProvisioningProgress 를 영속 엔티티로, 단계 이력을 SetupStep 으로 명명했다. 기능상 문제는 아니나 같은 이름이 다른 것을 가리켜 추적을 어렵게 한다.

수정안

이름 체계를 한쪽으로 정렬하고 CLAUDE.md 의 Stage 4 도메인 모델 절을 실제 구현명(SetupStep / 엔티티형 ProvisioningProgress)으로 갱신하거나, 구현명을 문서 용어에 맞춘다.

I1정보보고서 v2 의 4개 테이블 미구현은 결함이 아니라 의도된 이연▶

bmc_binding · bmc_credential_ref · location_assignment · provisioning_transition_log 는 Notion 2단계(a) 골격 스코프(5종) 밖이다. 미구현이 정상이다. 단 보고서 v2 가 이를 표식하지 않아 "다 있는 듯" 읽히므로, 보고서에 in-scope/deferred 구분을 넣을 것(→ §02 권고).

I2정보guest_server_custom 은 보고서 외 신규 — 문서화 + 중복 해소 필요▶

두 보고서·Notion 어디에도 없는 테이블이다. "하드웨어가 보고한 식별자(detail) vs 운영자가 부여해 ipmitool 로 새길 식별자(custom)" 분리 개념 자체는 타당하다. (a) 설계 문서에 추가하고, (b) GuestServer.modelName 과의 의미 중복을 단일화하라(F3).

I3정보/boot 가 빈 200 반환 — 다음 부팅 스크립트 미반환(7단계 이연)▶

ExecutionRestController.initialBoot() 는 ResponseEntity.ok(null) 을 돌려준다. iPXE 체인 관점에서 "다음에 부팅할 진단 리눅스 스크립트" 를 돌려주지 않는다. Notion 상 부팅 오케스트레이션은 7단계 이연이므로 현 스코프에서는 정상. 7단계 진입 시 진단 리눅스로의 chain 응답이 채워져야 함을 기록해 둔다.

05

잘 된 점

유지·계승할 설계

비판이 길어졌지만, 골격으로서 이 구현은 여러 면에서 모범적이다.

  • 등록 경로의 명세 충실도 — guest_server / detail / host_nic_binding 적재가 Notion U1 의 필드·상수(ip_source=DHCP, is_primary=true, discovery_stage=IPXE_REGISTERED)와 정확히 일치.
  • PXE 계약 일치 — GET /api/pxe/v1/boot + 5 쿼리 파라미터가 iPXE chain URL 과 동일. (CLAUDE.md 의 /pxe/v1/entry/boot 표기는 구버전 — Notion/구현이 최신 권위.)
  • 재부팅 멱등성 — existsBySystemUUID 로 매 부팅 중복 등록을 사전 차단.
  • Value Object 설계 — MacAddressVO/IpAddressVO 가 검증+정규화(구분자/대소문자 흡수), AttributeConverter 가 DB→VO 로드 시점에도 재검증. Primitive Obsession 금지 원칙 준수.
  • board_model 의 FK 격상 — 보고서의 자유 문자열을 관리 카탈로그 BoardModel FK 로 올려, 미등록 모델을 BoardModelNotFoundException 으로 거절. 검증된 인벤토리와 결합(설계 개선).
  • 정체성/인벤토리 분리 — slim guest_server + guest_server_detail 로 희소 NULL 광폭 테이블을 회피(v2 원칙 충실).
  • DTO 의 타입 보존 — 응답이 원시 문자열이 아니라 VO/Enum/UUID 를 그대로 전달.
  • N+1 회피 + CQRS 적 분리 — 목록 조회가 detail/primary NIC/progress 를 각 1쿼리로 일괄 적재, 읽기(Query)·쓰기(Command) 서비스 분리, @Transactional(readOnly=true) 적용.
  • UUID v7 PK — 시간 정렬 가능한 식별자로 클러스터링 이점.
  • 인라인 수정 검증 흐름 — name/serial 유니크 충돌을 서비스 boolean 질의 → BindingResult 필드 에러로 인라인 표시(예외 남발 대신 UI 차단/검증 — 프로젝트 예외 사용 원칙과 정합).
