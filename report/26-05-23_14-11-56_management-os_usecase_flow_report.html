13. 비대화 · 중복 · 순환 진단

사용자가 보고한 3 가지 (중복 / 클래스 수 과다 / 순환 참조 부담) 의 실증 근거. 각 진단에 증거 (코드 위치) + 리팩토링 제언.
D1 · HIGH

Nudge 도메인 3-way 중복 — proceed / replace / cancel 패턴이 3 서비스에 각각 박혀 있음

UC-25 ~ UC-33 의 9 endpoint 가 3 service (OsNudgeService, OSImageNudgeService, OsNudgeService.intent*) 에 분산. 각 서비스는 같은 4 단계 (require session + 도메인 검증 + 핵심 행위 + consumeSession) 를 별도 코드로 구현.

OsNudgeService.proceed:51 / replace:64 / cancel:87 ← OS_ISO content payload
OSImageNudgeService.proceed:34 / replace:43 / cancel:61 ← OS_IMAGE meta payload
OsNudgeService.proceedIntent:101 / replaceIntent:114 / cancelIntent:149 ← intent path/hash payload
공통 헬퍼 : requireXxxSession + consumeSession — 두 service 에 거의 동일 구현
제언 — NudgeOrchestrator 단일 service + NudgePayloadHandler 다형성 (3 sealed type : ContentNudgePayload / IntentMetaNudgePayload — 이미 존재. handler 가 도메인 종속 로직 캡슐화). CLAUDE.md 의 "신규 분기 추가 시 다형성/strategy 우선" 원칙과 부합. 9 endpoint 의 controller 도 1 endpoint 묶음으로 축소 가능.
D2 · HIGH PARTIAL FIX

OSImageService 가 hub-class — controller 측면은 해결됨 (2026-05-23), service 측 hub 잔존

2026-05-23 분리 작업으로 controller 측 hub (40 endpoint + 10 의존성을 가진 단일 OSImageController) 는 7 controller (각 의존 ≤ 3) 로 분할 완료. 그러나 service 측 hub (OSImageService 자체) 는 잔존. 의존성 매트릭스 (§3) 에서 OSImageService 의 호출 입출 차수가 여전히 압도적. finalizePreparedIsoRegistration (1126 줄짜리 파일에서 §502~584) 안에서 path 검증 / hash 계산 / 활성 충돌 / nudge 후보 / 영속화 / 마커 발급 / post-creation strategy trigger 7 책임을 한 메서드가 수행. 사용자가 "기능 추가 시 순환 참조 신경 써야 한다" 고 한 근본 원인의 절반은 여전 진행 중.
