HEADLINE

R7 후 "전역 @Lazy 제거"는 단정 불가 = 조건부다. R7 은 cycle #1(verifier ↔ scanner, ObjectProvider 가 막던 REAL 순환)만 정적으로 해소하고, cycle #2(SoftDeleteIntentService ↔ 도메인 lifecycle)는 "scanner 경유 → lifecycle 직결"로 이전할 뿐 — @Lazy 가 그대로 필요하다. 따라서 fat-Service 3분할(R4-3/R5-3/R6-3)은 typed-name inline 유지로 진행하되, ObjectProvider 도입 3단계(R4-4/R5-4/R6-2)는 ping-pong churn 으로 폐기하고, capstone 은 checkPrecondition 분리를 선행 전제로 하향해야 한다.

3 계획 수정을 만든 2대 통찰

통찰 ① — ObjectProvider 도입↔제거 PING-PONG (R4-4 / R5-4 / R6-2 폐기)

"BMC 로컬 무방비 순환"은 현재 코드에 실재하지 않는다.

BIOS/BMC/Subprogram 스캐너는 지금 도메인 fat-service 를 eager 직접주입(BoardBiosMarkableScanner:31 등, ObjectProvider 0개)인데도 부팅된다. 이 service 들이 TypedNameVerifier·MarkableScanner 에 eager 로 닿지 않기 때문이다 — 유일한 잠복 back-edge(service→SoftDeleteIntentService→@Lazy List<MarkableScanner>)는 SoftDeleteIntentService:61 의 @Lazy 가 4 도메인 공통으로 전역 차단한다.

도메인현재 스캐너→serviceNotion 계획비대칭
Board (R3-3 / R7-3)애초부터 lazy (ObjectProvider)R3-3 = 타입 narrow → R7-3 = 제거정당 (narrow→remove)
BIOS/BMC/Subprog (R4-4 / R5-4 / R6-2)현재 eager 직접주입R4-4 = eager→lazy 신규 도입 → R7-4 = 제거왕복 churn (도입→즉시제거)

Notion 이 R4-4 를 Board 의 R3-3 과 같은 부류로 취급한 게 오류다. R4-3 이 typed-name 을 inline 유지하면 보호할 순환 자체가 안 생겨 ObjectProvider 도입이 통째로 불요해진다. fb094f8 의 cascade @Lazy speculative 제거와 동형 안티패턴.

통찰 ② — R7 은 순환을 제거하지 않고 "이전"한다 (capstone 과대주장)

cycle #2 는 scanner 를 떼도 LifecycleService.softDelete → checkPrecondition 으로 다시 닫힌다.

R7 capstone 은 "scanner→service edge 를 없애면 SoftDeleteIntentService:61 의 @Lazy 도 불요 → 전역 @Lazy 제거"라고 주장한다. 적대 추적 결과 거짓: scanner 의 trash 메서드를 *LifecycleService 로 옮기면(R7), R4-3 후 softDelete(=checkPrecondition 호출자)도 그 LifecycleService 에 있으므로 BiosLifecycleService → SoftDeleteIntentService 변이 새로 생긴다. 그리고 SoftDeleteIntentService → List<TrashOperator>(=BiosLifecycleService) 가 되돌이 변이라 순환이 lifecycle 직결로 닫힌다. 실재 선례: IsoLifecycleService:57+115.

부차 결함 2건 — (a) capstone 이 List<MarkableTrashOperator> 주입을 명시했으나 SoftDeleteIntentService 의 실제 scanner 사용처(:177 applyForcedClear)는 GHOST 연산(MarkableGhostOperator)이라 trash 축 이전으로 해소 안 됨. (b) SoftDeleteIntentService:60 의 두 번째 @Lazy(PathReconciliationService)는 capstone scope 밖 — "전역" 주장이면 별도 처리 필요.

4 의존 그래프 — 현재 vs R7 후 (탭으로 순환 해소 vs 이전 비교)

cycle #1 — verifier ↔ scanner (REAL, ObjectProvider 가 차단)

TypedNameVerifierImpl ─주입 List<MarkableScanner>→ OSMetadataMarkableScanner ─ObjectProvider(lazy)→ OSMetadataLifecycleService ─주입 TypedNameVerifier→ TypedNameVerifierImpl ▲ 여기 lazy 가 순환 차단 (제거 시 부팅실패 = REAL)

동형: Board(BoardModelMarkableScanner→BoardModelLifecycleService), ISO. ← OS/Board/Iso 스캐너만 ObjectProvider 보유.

cycle #2 — SoftDeleteIntentService ↔ 도메인 service (전역 @Lazy 가 차단)

SoftDeleteIntentService ─@Lazy List<MarkableScanner>→ scanner.applyForcedClear/restore/purge → BiosService ─주입 SoftDeleteIntentService + :372 checkPrecondition→ SoftDeleteIntentService ▲ SoftDeleteIntentService:61 @Lazy 가 4 도메인 공통 전역 차단

BIOS/BMC/Subprogram 스캐너가 eager 직접주입인데도 부팅되는 이유 = 이 @Lazy.

현재 2개 순환이 @Lazy/ObjectProvider 로 가려져 있음. R4-4/R5-4/R6-2 가 겨냥한 "무방비"는 cycle #2 의 전역 @Lazy 가 이미 막고 있어 로컬 도입이 불요.

cycle #1 — 정적 해소 ✓ (R7-2/R7-3, ObjectProvider 진짜 불요)

*LifecycleService implements MarkableTrashOperator → scanner 의 restore/purge 변 제거 TypedNameVerifier → inventory-scanner.findActiveMarkableById/findTrashedById (read-only, service 무의존) ▲ scanner→service back-edge 소멸 → 순환 미폐합 → ObjectProvider 불요 (REAL 해소)

cycle #2 — 이전됨 (해소 아님) · @Lazy 여전히 필요

SoftDeleteIntentService ─List<TrashOperator>→ BiosLifecycleService ─(softDelete→checkPrecondition)→ SoftDeleteIntentService ▲ scanner 를 떼도 lifecycle 직결로 다시 닫힘 (IsoLifecycleService:57+115 선례)
@Lazy 제거 조건(iii) : (a) checkPrecondition 을 LifecycleService 밖(별도 PreconditionService)으로 분리 + (b) ghost 연산을 GhostOperator 로 유지 + (c) 두 번째 @Lazy(PathReconciliation) 별도 해소 — 셋 다 선행해야 SDIS:61 의 @Lazy 1건만 제거 가능. capstone 원안은 이를 누락.
그래도 R7 은 가치 있음 : cycle #1 의 REAL ObjectProvider 3종(OS/Board/Iso)을 진짜로 없애고, God-SPI(19시그니처·4관심사 ISP 위반)를 분리한다. "전역 @Lazy 제거"만 과대주장이고 나머지는 정당.

6 Top 리스크

  • capstone 과대주장 — R7-6 원안대로 "전역 @Lazy 제거"를 단정하면 R7 후에도 닫혀있는 cycle #2 를 만나 BeanCurrentlyInCreation 부팅실패 또는 @Lazy 재도입 churn. checkPrecondition 분리 선행 없이는 목표 미달.
  • ping-pong 자초 — R4-4/R5-4/R6-2 를 독립 선행하면 보호할 순환이 없는데 ObjectProvider 를 넣고 R7-4/5/6 이 즉시 제거. fb094f8 speculative @Lazy 제거와 동형 = refactor 부적격.
  • typed-name verifier 위임 유혹 — R4-3/R5-3/R6-3 에서 R3 의 TypedNameVerifier SSOT 매력에 끌려 verifier 를 주입하면 BIOS/BMC/Subprogram 에 순환을 새로 닫음. 제거할 복붙 helper 가 없는데(displayName().equals() 단 1줄) 순환만 자초.
  • R7-1 seam 오설계 — Notion 4관심사 분할은 ISP 만 풀고 read/operation 경계와 어긋남. findTrashedById(read)를 cycle-bearing restore/purgeFromTrash 와 같은 TrashOperator 에 묶으면 verifier/PurgeExecutor 가 순환 변을 떠안아 cycle-break 무력화.
  • 부분 중복 잔존 — R5-3/R6-3 에서 Registration 분리만 하고 *MarkerWriter 추출(marker 4-step 3중복)을 빠뜨리면 복붙이 새 클래스로 이사만 가는 churn. R5-2b 에서 displayMessage 만 빼고 applyStatus 4중복을 남기면 중복 금지 절반만 충족.
