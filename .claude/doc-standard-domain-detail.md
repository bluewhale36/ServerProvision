# 도메인 상세 문서 표준

Spring Security 공식 레퍼런스 전수 조사(2026-07-19, Opus 4.8 7-agent workflow)에서 도출한 도메인별 상세 개발자 문서의 작성 표준이다. DOC-1-1부터 모든 상세 문서(DOC-1-x)가 이 표준을 따른다. 문체 규칙은 docs/README.md 작성 규칙이 우선한다.

## 문서 1편의 절 구성

앞은 지도와 기본 동작으로 얇게 열고, 뒤로 갈수록 깊어진다. 독자는 앞 절만 읽어도 도메인을 쓰기 시작할 수 있고, 확장 코드를 짤 때만 뒤로 내려간다.

1. 문서 좌표: 상위 개관(architecture.md)과 이웃 문서를 링크하는 한두 문장.
2. 개요와 기본 동작: 설정 없이 프레임워크가 이미 해 주는 것을 커스터마이징보다 먼저.
3. 유스케이스 메뉴: "~하려면" 목표 문장 리스트로 하위 절 앵커 링크.
4. 개념과 배경: 이 설계가 왜 존재하는지 문제에서 해결로 잇는 서사. 코드 최소화.
5. 컴포넌트 지도: mermaid classDiagram 한 장(이름과 관계만) + 관계를 동사 산문으로 다시.
6. 주요 흐름: 흐름 1개당 1절(아래 흐름 절 표준).
7. 컴포넌트 상세: 컴포넌트 1개당 1절(아래 컴포넌트 절 표준). 분량으로 중요도를 표현한다.
8. 확장과 커스터마이징(아래 확장 절 표준).
9. 신규 항목 추가 시 수정 지점: 순서 있는 체크리스트, 각 항목에 책임 클래스 병기.
10. 구식 컴포넌트(있을 때만): 본문에 흩뿌리지 않고 격리 절에 대체재와 함께.
11. 실패와 문제 해결: 실제 예외 문자열 그대로 + 원인 + 해결.
12. 고려사항과 함정: 위험한 API 바로 옆 인라인 경고 위주.
13. 테스트: 무엇을 왜 고정하는지 프로덕션 코드와 결속해 서술.
14. 더 읽기.

## 컴포넌트 절 내부 구조

1. 정의 첫 문장 = 정의이자 그래프 좌표. 여는 문장이 타입을 정의하며 이웃 컴포넌트를 지목한다. SPI 구현체는 "X는 Y를 구현해 ~를 담당한다"로 못박는다.
2. 관계는 동사 산문으로: "에 포함된다", "에 위임한다", "로 감싼다", "를 구현한다", "의 가장 흔한 구현이다". 다이어그램에만 의존하지 않는다.
3. 핵심 인터페이스는 두 불릿 리스트로 해부: 역할과 용도 / 구성요소와 필드(각 항목에 대표 타입과 도메인 의미).
4. 코드는 최소 스니펫(5~7줄, 주변은 `// ...`). 전체 클래스는 독자가 그 클래스를 직접 쓰는 것이 산출물일 때만.
5. 절차 코드에는 (1)(2)(3) 콜아웃 + 아래 번호 리스트에서 각 줄의 WHY(오용, 이유)까지.
6. 보조 타입은 정의 한 문장으로 끝낸다.

## 흐름 절 내부 구조

1. 흐름이 언제 시작되고 무엇을 이루는지 한 문장. 실제 실행되는 흐름에만 번호 단계를 쓰고, 설정 주제는 조건 산문("~하면 ~한다")으로 쓴다.
2. mermaid sequenceDiagram 한 장. 화살표 번호와 본문 단계 번호를 1:1 연동한다. 시그니처와 필드는 넣지 않는다.
3. 각 단계는 현재형 능동태 한 절(정확한 클래스명 + 효과나 예외). 부수효과는 번호 아래 하위 불릿으로 남김없이.
4. 전달 값의 해상도: 타입 이름 + 생성 위치 + 서브클래스 분기까지. 직렬화된 필드값은 보이지 않는다.
5. 엔진 내부는 "의사코드" 라벨을 붙인 요지 코드로.
6. 매칭이나 정렬 같은 추상 규칙은 실제 입력값을 넣은 사례로 한 번 굴려 시연한다.
7. 실패 경로는 던지는 예외, 잡는 핸들러, 최종 HTTP 상태코드까지 인과를 단계로 편다.

## 확장 절 내부 구조

1. 확장점 역할 문장 평문 먼저.
2. 표준 문형: "X가 Y를 구현한다고 할 때, 빈으로 등록하면 기본 대신 사용된다" + 적용 조건절.
3. 단일 메서드 인터페이스면 람다 공급 가능을 안내.
4. 확장점이 여럿이면 자유도 계단(간단하지만 제약 → 최대 자유도)으로 배열하고 계단마다 트레이드오프 재요약.
5. 구현체 코드와 등록 코드를 한 쌍으로(추상 서술로 끝내지 않는다).
6. 선행 빈이 필요하면 "먼저 ~, 그다음 ~"으로 순서를 못박는다.

## 형식 규칙

- classDiagram은 아키텍처 지도 절에 한 장, 이름과 관계만. sequenceDiagram은 런타임 흐름에만.
- 참고는 `> 참고:`, 함정은 `> 주의:` 인용 블록으로, 위험한 지점 바로 옆에 인라인으로 둔다.
- 같은 개념을 두 곳에서 재정의하지 않는다. 최초 정의 절로 앵커 링크한다. 공유 개념(marker, lifecycle 등)의 SSOT는 `docs/dev/global.md`다.
- 표는 비교와 의사결정에만. 정의 나열은 불릿 리스트로.
- 시그니처 수록 경계: 콜아웃 스니펫 수준까지만. 필드와 메서드 전체 목록은 코드가 SSOT.
