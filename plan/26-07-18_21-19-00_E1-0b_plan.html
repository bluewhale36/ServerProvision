§1. 현재 상태 (진단)

E1-0a 가 서버 내부 규약(신호 컬럼 · 전이 invariant · 원장 적재 · 개시 버튼)을 완성했다. 그러나 게스트 쪽에서 보면 아무것도 변하지 않았다:

자산있음 (E1-0a 까지)비어 있음 (이 슬라이스가 채움)
GET /api/pxe/v1/boot멱등 등록 + 유보분 2행 적재응답이 여전히 빈 200 — 게스트는 등록만 되고 다음 할 일을 받지 못한다. 개시 버튼을 눌러도 게스트 동작이 바뀌지 않는다(E1-0a 비목표로 명기했던 그 공백)
전이 도메인 메서드advanceTo/markFailed/markCompleted + invariant 완비호출자 0 — 게스트 사실 신호(체크인 · 보고)를 받는 API 가 없어 신호가 발생할 수 없다
SetupStep 원장단발 기록(recordInstant) + append-only 규약게스트 실행 step 의 RUNNING 열림/닫힘 규약 미구현 (DEC-3 의 나머지 절반)
예외 채널JSON(ApiExceptionHandler) / HTML(WebExceptionHandler) 이원화iPXE 채널 없음 — 지금 /boot 에서 예외가 나면 JSON 이 iPXE 로 새어 나가 파싱 불능 → 부팅 실패 루프(1차 문서가 지적한 레거시 결함 그대로)
게스트 신원—없음 — 보고 API 를 열면 무인증 사칭 보고로 타 서버 상태를 오염시킬 수 있다(DEC-5 토큰 확정, 미구현)
하네스boot-register.sh (등록만)체크인 · 보고 시나리오 부재 — CP5 리허설 불가
핵심 진단 — E1-0a 의 "정직 기록" 그대로: 신호는 도출 가능하나 실트리거가 없다. 이 슬라이스가 게스트↔서버 프로토콜 v1(부팅 dispatch + 체크인 + 보고)을 열면, BOOTSTRAPPING→DIAGNOSE_LINUX 전이와 FAILED 실패 신호가 처음으로 실제 발생한다 — 단 실게스트가 아니라 모의 게스트(curl)로. 진단 리눅스 실부팅 자산은 E1-1 이다.
§2. 요구사항 (확정) + 비목표
  1. /boot text/plain 전환 + dispatch 매트릭스 v1(DEC-2 · D3) — 응답이 iPXE 스크립트가 된다. 판정 입력 = (등록 × 회수 × failed × completed × started × 커서 × 실행기 등록 여부) — 1차 문서 §6-4 매트릭스의 E1-0b 소유분 7행(§4).
  2. phase 실행기 SPI 골격(DEC-6) — 최소 폭 단일 인터페이스(ProvisioningPhaseExecutor: 담당 phase + 부팅 스크립트 기여) + 기동 시 수집 registry. 이번 슬라이스 구현체 0개 — 커서가 미등록 phase 에 도달하면 명시 HOLD 대기 스크립트(silent 통과 금지). 첫 구현체는 E1-1(DIAGNOSE_LINUX).
  3. 게스트 토큰(DEC-5) — 등록 시 발급 · guest_server 저장(GuestToken Value Object), 부팅 스크립트의 커널 인자로 게스트에 전달, 에이전트 API 전 요청에 요구. 기존 등록 서버(토큰 없음)는 다음 /boot 재진입 시 발급(멱등 가드 확장).
  4. 체크인 API — 진단 리눅스 에이전트의 첫 신호. 효과: 첫 체크인 = advanceTo(DIAGNOSE_LINUX)(DEC-2 실배선, 수신 트랜잭션 내 즉시). 응답은 지시 골격(JSON) — 지시 내용 채움은 E1-2.
  5. 보고 API (RUNNING 원장 규약, DEC-3 완결) — 시작 보고 = RUNNING 행 생성(응답에 step 행 식별자) / 종료 보고 = 그 식별자에 바인딩된 close(SUCCEEDED/FAILED/SKIPPED + statusMeta). 중복 종료 보고 = no-op(멱등). FAILED 종료 보고 수신 = progress.markFailed 즉시 — 실패 신호의 첫 실트리거.
  6. PXE 부팅 채널 한정 advice(DEC-5) — /boot 계열 컨트롤러 한정 @RestControllerAdvice 가 낙관적 락 충돌 포함 전 예외를 HTTP 200 + 재시도 대기 스크립트로 변환(§8 Q3). 에이전트 API(JSON)는 기존 ApiExceptionHandler 그대로.
  7. 하네스 확장 — diagnose-cycle.sh: 부팅→토큰 추출→체크인→step 시작/종료 보고→(성공/실패 변형) 시퀀스. CP5 절차의 본체.
비 목표 :
  • 진단 리눅스 실부팅 — Alpine 아티팩트 서빙 · 체인로드 스크립트 · DIAGNOSTIC_BOOTING step 은 E1-1 (첫 실행기 구현체). 이번 dispatch 는 DIAGNOSE_LINUX 도달 시 HOLD 를 준다.
  • 수집 계약(hardwareSpec 파싱 · GuestServerDetail 갱신 · DIAGNOSTIC_ENRICHED) · phase 완주 판정(PhaseSequence 소비 · completedAt 실기록) · 운영자 수동 FAILED/재시도 — E1-2.
  • completed 서버의 로컬 부팅 폴스루(exit)는 스크립트만 준비 — sanboot/exit 실효성은 T2 검증 항목(E4 시점 실검).
  • 할당(U3) 판정 — 대기 사유에 "할당" 축은 U3 이후 dispatch 에 합류.
🎬 미리보기 — /boot dispatch + 프로토콜 시뮬레이터

이 데모는 E1-0b 의 두 축을 재현한다. 왼쪽 상태를 바꾸면 아래 검은 화면에 /boot 가 실제로 반환할 iPXE 스크립트가 갱신된다(dispatch 매트릭스 = simDispatch() — 서버 판정과 1:1). 게스트 신호 버튼은 체크인 · 보고 API 의 효과(전이 · RUNNING 원장 · 실패 신호 · 멱등)를 재현한다. 토큰이 틀리면 404 로 거절되는 것도 체험할 수 있다.

이 순서가 곧 우선순위다 — GuestServerStatus.derive 진리표(회수 > 실패 > 종단 > 개시)와 정렬을 맞춰 "배지가 말하는 것 = 게스트가 받는 것" 이 되게 한다. 신규 phase 의 행 추가는 매트릭스 분기 증식이 아니라 실행기 빈 등록(6→7 자동 전환)이다 — 조건분기 확장 금지의 이행.

phase 실행기 SPI — 최소 폭 단일 인터페이스 (DEC-6)

// execution/engine — MarkableScanner 선례와 동형: phase 판별자 빈들을 registry 가 기동 시 수집.
public interface ProvisioningPhaseExecutor {
    ProvisioningPhase phase();
    String bootScript(GuestServer server, ProvisioningProgress progress);   // 부팅 스크립트 기여
}
// PhaseExecutorRegistry: Map<ProvisioningPhase, executor> — 중복 판별자 = 기동 실패(fail-fast),
// 미등록 phase 조회 = Optional.empty (dispatch 6번 HOLD — 명시 동작). 전체 enum 커버 강제 없음.
// 보고 처리 메서드는 두 번째 실물(E1-2 수집 소비)이 나타날 때 추가 — 미리 분리 금지.

    public boolean matches(String presented) { … } // 에이전트 API 인증 대조의 단일 지점

§8. 예상 부산물 / 주의 / 결정 사항

잔존 책임 · 주의 (정직 기록)

  • 실게스트는 여전히 부팅 못 한다 — dispatch 가 진단 리눅스 체인로드(7번 행)를 주려면 E1-1 실행기가 필요. 이 슬라이스의 검증 상한 = T0(계약) + T1(하네스). 실 iPXE 가 대기 스크립트를 올바르게 해석하는지는 E1-1 의 T2 에서 처음 확인된다 — sleep/chain 명령의 iPXE 빌드별 가용성이 유일한 외부 미지수(E1-R V7 과 함께 검증).
  • 체크인 → 전이는 열리지만 완주는 안 열린다 — DIAGNOSE_LINUX 진입 후 완주 판정(PhaseSequence 소비 · completedAt)은 E1-2. 하네스로 completed 상태를 만들려면 이번엔 DB 직접 조작이 필요함을 CP5 안내에 명기.
  • 중복 open 은 허용 — 에이전트 재전송(응답 유실)으로 RUNNING 행이 중복될 수 있으나 "현재 상태 = stepCode 별 최신 행"(DEC-3) 규약이 흡수한다. close 만 멱등이면 원장 정합이 유지된다 — open 멱등 키 도입은 실측 후(과잉 방지).
  • guest_token UNIQUE — 충돌 확률은 무시 가능하나 유니크 제약으로 안전망(발급 재시도 없음 — 충돌 시 등록 트랜잭션 실패 = 재부팅 재시도).
