1. 현재 상태 요약

1.1 선행 완료

  • S5-2-1 CP4 완료 — confirm-action.html + JS 의 3 actionKey (soft-delete / deprecate / restore) 도입. 단순 restore 가 5 도메인 모두에 modal 적용된 상태.
  • S5-2-2 CP4 완료 — 'purge' actionKey + typed-name input + 5 + Board 신설 purge endpoint + 12 통합 테스트.
  • CP5 (브라우저 E2E) 는 S5-2 묶음 종료 시점에 일괄 — 본 S5-2-3 까지 합쳐서 사용자가 동시 테스트.

1.2 본 sub-slice 진입 전제

  • OS / Board 의 restore 는 현재 단순 'restore' actionKey (S5-2-1) — cascade 옵션 없음.
  • OS soft-delete 시 하위 ISO 들이 자동 동반 soft-delete 됨 (현재 service 정책). restore 시 하위이 자동 복구되지 않으면 부모만 살아남고 하위은 휴지통에 남는 비대칭 상태.
  • Board 도 동일 — soft-delete 시 하위 BIOS / BMC 동반, restore 시 하위 자동 복구 안 됨.
  • ISO 의 OSEnvironment / OSPackageGroup 은 ISO soft-delete 시 DB hard-delete 됨 → cascade 대상 외 (복구 자체 불가).

1.3 사용자 동기

부모 (OS / Board) restore 시점에 하위 자원도 함께 복구할지 사용자가 modal 에서 라디오로 선택. 묵시적 동반 복구 / 무복구 양자택일이 아니라 명시 선택으로 운영 의도 명확화.

2. 페이지 요구사항 (Notion 확정)

자원 복구

자원을 복구할까요? 하위 자원도 함께 복구할지 선택해주세요.

사용자 흐름

  1. 클릭 — soft-deleted 상태 부모 자원의 "복구" 버튼 클릭 → submit 가로채기
  2. modal 표시 — title + 메시지 + 라디오 2 (cascade=true 가 default) + 하위 자원 카운트 안내
  3. 라디오 선택 — 변경 시 hidden cascade input 값 갱신. 시각적으로는 선택된 항목만 강조 (S5-2-1 의 disabled 단순화 정책 동일 — 별도 hint 문구 없음)
  4. 제출 — 확인 클릭 → form submit → POST .../restore?cascade=true 또는 ?cascade=false → service.restore(id, cascade)
  5. 결과 — 성공 시 302 redirect. cascade=true 이면 하위까지 함께 복구. 활성 자원과 (vendor, modelName) 등 충돌 시 409 (트랜잭션 전체 롤백)

백엔드 의사코드 (Service 측, OS 예시)

@Transactional
public RestoreResponse restore(Long id, boolean cascade) {
    OSImage image = osImageRepository.findByIdAndIsDeletedTrue(id)
        .orElseThrow(() -> new IllegalOSImageStateException(...));

    // 활성 (osName, osVersion) 충돌 검사 — 기존 정책
    if (osImageRepository.existsByOsNameAndOsVersionAndIsDeletedFalse(
            image.getOsName(), image.getOsVersion())) {
        throw new DuplicateOSImageException(...);
    }
    image.restore();

    int restoredChildren = 0;
    if (cascade) {
        // 하위 ISO 중 isDeleted=true 만 일괄 복구
        for (ISO iso : image.getIsos()) {
            if (iso.isDeleted()) {
                // ISO path 활성 충돌 검사 — 충돌 시 throw → @Transactional 전체 롤백
                requireNoPathConflict(iso);
                iso.restore();
                restoredChildren++;
            }
        }
    }
    return new RestoreResponse(restoredChildren);
}
partial-rollback : cascade 진행 중 하위 일부 복구 후 다른 하위이 충돌하면 @Transactional 이 전체 롤백 — 부모 / 모든 하위 모두 복구 안 된 상태로 되돌아감. cascadedChildren 응답은 트랜잭션 커밋 성공 시의 하위 복구 건수.
OSEnvironment / OSPackageGroup : ISO soft-delete 시 DB hard-delete 됨 → cascade 대상 외. 사용자는 ISO 복구 후 별도로 다시 추출해야 함. modal 메시지에 그 안내 포함.

8. 예상 부산물 · 주의
partial-rollback 정책 : cascade 진행 중 하위 일부 복구 → 다른 하위이 충돌 → @Transactional 전체 롤백. 응답의 cascadedChildren 은 트랜잭션 커밋 성공 시의 하위 복구 건수. 사용자는 페이지 reload 후 휴지통에 그대로 남은 것 확인 가능.

주의 항목

  • OSEnvironment / OSPackageGroup 의 cascade 제외 — ISO soft-delete 시 DB hard-delete 됐으므로 복구 자체 불가능. modal 메시지에 "설치 환경 / 패키지 그룹은 별도로 다시 추출해야 합니다" 안내 포함.
  • BIOS / BMC / Subprogram / ISO 의 restore — leaf 이므로 cascade 옵션 미적용. S5-2-1 의 단순 'restore' actionKey 유지.
  • Subprogram cascade 미포함 — Notion 명시에 메인보드 하위으로 BIOS / BMC 만. Subprogram (Driver / Utility) 은 board scope 일 수도 공용일 수도 있어 정책 모호 → 본 슬라이스 범위 외.
  • nudge REPLACE / TTL 자동 복구 흐름은 없음 — restore 는 사용자 명시 액션에만. S5-2-4 와 무관.
  • CLAUDE.md 수정 — 없음. (Step 1 산출물 규약은 S5-2-2 시점에 이미 docx→html 전환 명시됨.)
