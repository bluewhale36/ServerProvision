§1. 현재 상태 (진단)

E 로드맵 토론(discussion 1~3차, 2026-07-12 종결)이 엔진의 모든 골격 결정을 확정했고, 이 슬라이스는 그 결정 중 서버 내부 규약(원장 · 전이 · 신호)을 처음 코드로 만든다. U1 이 남긴 것과 빈 것:

자산있음비어 있음 (이 슬라이스가 채움)
ProvisioningProgresscurrentPhase 커서 SSOT + @Version 낙관적 락 + lastTransitionAt(명시 set 설계) + 등록 seed전이 · 신호 도메인 메서드 0개 — 커서를 움직일 수 있는 코드가 없다. 개시 · 실패 · 종단 개념 자체가 스키마에 없다
SetupStepappend-only 이력 테이블 + phase() 파생 — "행 적재는 엔진 책임" 으로 U1 이 명시 유보적재자 0 — 테이블이 항상 비어 있고, U1 이 실제 수행한 NETWORK_ALLOCATING · INIT_PERSISTING 도 기록되지 않는다
GuestServerStatus.derive2-인자 순수 함수 + REGISTERED/PROVISIONING/DECOMMISSIONED 도달FAILED · PROVISIONED 도달 불가 — 도출 입력(실패 · 종단 신호)이 존재하지 않는다 (Javadoc 이 이 슬라이스로 유보)
server-detail.html상태 배지 5종 마크업 + step 이력 테이블 렌더 골격 완비 — 행만 있으면 표시된다개시 액션 없음, 실패 · 종단 시각 표시 없음. 기본 정보 힌트에 보류된 각인(DEC-11) 문구가 잔존
핵심 진단 — 어휘(enum) · 저장 골격 · UI 렌더는 전부 준비돼 있으나 그 어휘를 움직이는 주체가 0 이다. 이 슬라이스가 끝나면: 등록 즉시 이력 2행이 보이고, 개시 버튼이 상태를 바꾸고, FAILED · PROVISIONED 가 (신호만 실리면) 도출되는 상태가 된다. 신호를 실제로 싣는 주체(게스트 보고)는 E1-0b · E1-2 가 배선한다.
§2. 요구사항 (확정) + 비목표
  1. 신호 4컬럼 — provisioning_progress 에 started_at(개시, DEC-26) / failed_at + failed_step_code(DEC-4) / completed_at(DEC-25) 추가. 전부 nullable, 도메인 메서드로만 기록.
  2. 전이 도메인 메서드 — start / advanceTo / markFailed / markCompleted + 판정 질의(isStarted 등). invariant: phase 역행 금지(선언 순 비교) · failedAt ↔ completedAt 상호배타 · 실패 · 종단 후 전이 금지. lastTransitionAt 은 전이 시 명시 set(U1 §D6 계승).
  3. SetupStepRecorder(engine 패키지 신설) — 단발 기록 API. 등록 트랜잭션이 U1 유보분 NETWORK_ALLOCATING · INIT_PERSISTING 2행(SUCCEEDED)을 실적재 — 빈 이력 테이블이 처음 채워지는 CP5 가시물. 등록 멱등 구조(systemUUID 존재 시 조기 반환)라 중복 적재 없음.
  4. PhaseSequence(engine 신설) — 다음 phase 결정 순수 함수. 규칙(DEC-8 · 25): ProvisioningPhase 선언 순 위에서 BOOTSTRAPPING→DIAGNOSE_LINUX 는 무조건, 그 이후는 "할당 정의서가 보유한 phase" 필터, 보유 마지막 phase 완주 = 종단(empty). 소비자(실제 판정 호출)는 E1-0b · E1-2 — 이번엔 함수 + 규칙 고정 테스트만.
  5. 개시 버튼(DEC-26) — 상세 페이지 "프로비저닝 진행" 섹션. POST /provisioning/server/{id}/start. UI 1차 차단: 이미 개시 → 버튼 대신 개시 시각 표시 / 회수됨 → 미노출. 서버 가드(안전망): 재개시 · 회수 서버 개시 = 409.
  6. GuestServerStatus.derive 확장 — 진리표: 회수 > 실패 > 완료 > (개시 여부 → PROVISIONING/REGISTERED). 기존 "currentPhase==BOOTSTRAPPING → REGISTERED" 분기는 startedAt 기준으로 대체된다(§4).
  7. 상세 UI 신호 표시 — 개시 · 실패(step 포함) · 완료 시각 행 + FAILED/PROVISIONED 배지 실도달.
  8. DEC-11 동반 수정 — 기본 정보 힌트의 "진단 리눅스에서 ipmitool 로 각인" 문구 제거, ProvisioningPhaseStep.IPMI_SETTING Javadoc 에 보류 명기.
비 목표 :
  • /boot 응답 변경 없음 — 빈 200 유지. text/plain 전환 · 커서 dispatch · 대기 스크립트는 E1-0b. (개시 버튼이 게스트 동작을 실제로 바꾸는 것은 E1-0b 부터 — 이번 슬라이스의 개시는 신호 기록 + 상태 표시까지다.)
  • 체크인 · 보고 API, RUNNING 열림/닫힘 원장 규약, 게스트 토큰 — E1-0b.
  • 실패 신호의 실제 발생자(게스트 실패 보고) · 운영자 수동 FAILED 전환 · 재시도 액션(failedAt 해제) — E1-2. 이번 슬라이스에서 FAILED 는 도메인 메서드 + 진리표 테스트로만 검증된다(실 트리거 없음 — 정직하게 명기).
  • "보유 phase 집합" 의 실제 공급(할당 스냅샷) — U3. PhaseSequence 는 집합을 파라미터로 받는 순수 함수로 격리.
