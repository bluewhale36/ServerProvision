1. 현재 상태 — cascade 미적용 영역

1.1 부모-자식 관계 (이번 범위)

  • OS 버전 (OSImage) ↔ ISO 1:N
  • 메인보드 모델 (BoardModel) ↔ BIOS / BMC 1:N (두 자식 타입)

1.2 6 lifecycle action × 2 부모의 현재 cascade 적용 매트릭스

ActionOS → ISOBoard → BIOS/BMC현재 상태 / 문제
toggle (활성↔비활성) 미적용 미적용 부모만 토글. 자식 동기화 안 됨 → 부모 비활성인데 자식 활성 상태 가능 (논리적 모순)
deprecate / undeprecate 미적용 미적용 부모만 deprecate. 자식 활성 유지 → 동일 모순
soft-delete 자동 자동 이미 cascade 적용 (S5-2-3-1 [commit 64955ae] 에서 trash 이동까지 정합)
restore 사용자 옵션 사용자 옵션 S5-2-3 의 cascade 라디오 도입. 사용자가 true/false 선택
purge 미검증 검증 필요 OS 버전 hard-delete 시 자식 ISO 도 cascade 되는지 코드 점검 필요. Board 는 현재 자식 잔존 시 거절 정책 — 정합?

1.3 자식 단독 lifecycle 시 부모 상태 검증 부재

  • 부모가 deleted 인데 자식만 restore 시도 — 거절 / 안내 없음 (사용자 명시 사건)
  • 부모가 deprecated 인데 자식만 undeprecate 시도 — 거절 / 안내 없음
  • 부모가 disabled 인데 자식만 enable 시도 — 거절 / 안내 없음
2. 페이지 요구사항 (사용자 명시)
  1. OS 버전을 비활성화 하면 그 아래 모든 ISO 도 비활성화.
  2. 이는 deprecate 도 마찬가지.
  3. 그리고 이러한 '하위 자원' 은 Board 와 BIOS, BMC 의 관계에도 적용됨.

여기에 더해서 다음과 같은 문제점도 있어:

  1. OS ISO 가 있는 OS 버전을 soft-delete 한 다음 ISO 만 restore 할 수 없음 (논리적 충돌)
  2. 마찬가지로, OS 버전이 purge 되었을 경우 그에 대한 OS ISO 도 같이 purge 되어야 함 (논리적 연결)
  3. 따라서 휴지통 UI 에서 이를 검증해서 적절히 사용자에게 안내해야 함.

cascade 시나리오 핵심

  1. 부모 비활성 클릭 → 자식 모두 비활성. 부모 활성화 시 자식도 모두 활성. (강제 cascade)
  2. 부모 Deprecated 클릭 → 자식 모두 Deprecated. (강제 cascade)
  3. 부모 삭제 클릭 → 자식 모두 삭제 (이미 적용됨).
  4. 부모 복구 (자식 함께) → 부모 + soft-deleted 자식 일괄 복구. (S5-2-3 옵션)
  5. 부모 복구 (부모만) → 부모만 active 로. 자식은 trash 잔존.
  6. 자식 ISO 단독 복구 → 부모가 deleted 면 거절 + UI 안내 "부모 OS 버전부터 복구해주세요."
  7. 자식 ISO 단독 활성화 → 부모가 비활성이면 거절.
  8. 자식 ISO 단독 deprecate 해제 → 부모가 deprecated 면 거절.

신규

8. 예상 부산물 · 주의
다형성 응집 (CLAUDE.md §조건 분기문) : cascade 로직은 service 의 단일 메서드에 응집. 부모 service 가 자식 service 의 lifecycle 메서드 위임 호출 (이미 S5-2-3 의 BoardModelService 의 BiosService / BmcService @Lazy 주입 패턴 그대로 활용). 신규 분기 0.
자식 단독 가드 위치 : 자식 service 의 lifecycle 메서드 본체에서 부모 상태 lookup → 부합 시 ChildLifecycleBlockedByParentException throw. 자식의 entity 가드와 별개 (entity 는 자기 상태만 검증, 부모 관계는 service 책임).

주의 항목

  • cascade 강제 정책의 backward compat 영향 — 기존 행위 변경. 한 번 deploy 후 OS 비활성화 누른 운영자 입장에선 ISO 도 비활성 — 의도된 변화이지만 회귀 테스트 필요.
  • OSImageService.purgeImage 의 ISO cascade — 현재 JPA `@OneToMany(mappedBy="osImage")` 에 cascade 없음. ISO row 가 자동 삭제 안 될 가능성 — Step 8 시점에 코드 점검 + 필요 시 명시 cascade 로직 추가.
  • UI 측 안내 — 자식 단독 restore 거절 시 ApiErrorResponse 의 부모 자원 정보를 modal 에 표시. confirm-modal-base.js 또는 별도 reject modal 확장 검토.
  • 이번 슬라이스 범위 외 — Subprogram 은 boardScope nullable (공용 가능) 라 부모-자식 관계 모호. 본 슬라이스 제외 — Driver/Utility 의 cascade 정책은 후속 검토.
  • CLAUDE.md 수정사항 — 없음.
