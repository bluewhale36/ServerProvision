0. 재설계 동기 (v1 → v2)

0.1 사용자 의도 (2026-05-23)

S5-11 에서 모든 파일에 대해 동일한 정책으로 진입점을 탐지하고 있는데, 만일 gigabyte 에서 cap 파일을 필요에 의해 추가하는 등 변수가 생길 경우를 대비해서 제조사별로 진입점 탐지 로직을 다르게 가져가는 로직을 아예 지정해둘 수 있으면 좋겠어. 단순 if-else 분기 및 단순한 조건 상의 분기는 자제해야 하므로 클래스 단위로 만들어서 분기를 형성하는게 좋겠지?

0.2 v1 의 한계 (Strategy 단일 정책)

v1 의 우선순위는 모든 vendor 에 동일하게 적용 :

1. override
2. 트리 단일 파일
3. 루트 *.cap 1 개 (ASUS 의도)
4. f.nsh
5. flash.nsh
6. 단일 *.nsh

GIGABYTE 가 운영 도구 / 부속 자료 / 디버그용 .cap 등을 BIOS 번들에 동봉할 가능성 — 이때 v1 은 .cap 을 우선 채택해 GIGABYTE 의 의도된 진입점 (f.nsh) 무시 위험.

0.3 v2 의 해결

  • Vendor 별 Strategy 분리 — 각 vendor 의 정책이 코드 위치 (클래스) 로 명확
  • 새 vendor 가 추가되면 strategy 구현체 하나 더 등록 — 호출부 / 다른 vendor 영향 0
  • 각 strategy 의 우선순위 / 폴백 로직이 한 파일에 자기 완결 — 정책 변경 시 영향 범위 명확
  • CLAUDE.md "조건 분기문 무분별 확장 금지" 원칙 일관 — S5-7 의 PostCreationTaskStrategy 와 동일 패턴

public String detect(Vendor vendor, Path treeRoot, String override) {

4.4 AsusEntrypointStrategy

@Component
@RequiredArgsConstructor
public class AsusEntrypointStrategy implements EntrypointDetectionStrategy {
    private final EntrypointPolicyService entrypointPolicyService;
    private final FileSystemSecurityProperties fsProps;

    public boolean supports(Vendor vendor) { return vendor == Vendor.ASUS; }

    public String detect(Path treeRoot, String override) {
        // 1) override
        String resolved = validateOverrideAndResolve(...);
        if (resolved != null) return resolved;
        // 2) 트리 단일 파일
        List<Path> allFiles = listRegularFilesExcludingMarker(treeRoot, fsProps.maxDepth());
        if (allFiles.size() == 1) return toRelative(treeRoot, allFiles.get(0));
        if (allFiles.isEmpty()) throw new EntrypointNotFoundException("...");
        // 3) 루트 *.cap 1 개
        List<Path> rootCap = listRootCapFiles(treeRoot);
        if (rootCap.size() == 1) return toRelative(treeRoot, rootCap.get(0));
        // 4) fallback : .cap 0 + .nsh 1 개 → 그것
        List<Path> rootNsh = listRootNshFiles(treeRoot);
        if (rootCap.isEmpty() && rootNsh.size() == 1) return toRelative(treeRoot, rootNsh.get(0));
        // 5) ambiguous / notFound
        ...
    }
}

4.7 BundleEntrypointDetector (dispatcher, 시그니처 변경)

@Service
@RequiredArgsConstructor
public class BundleEntrypointDetector {
    private final List<EntrypointDetectionStrategy> strategies;

    /**
     * Vendor 매칭 strategy 에 위임. 매칭되는 strategy 가 없으면 신규 vendor 가
     * enum 에 추가됐는데 구현체가 없는 코드 누락 상황 — EntrypointStrategyMissingException.
     */
    public String detect(Vendor vendor, Path treeRoot, String override) {
        return strategies.stream()
                .filter(s -> s.supports(vendor))
                .findFirst()
                .orElseThrow(() -> new EntrypointStrategyMissingException(vendor))
                .detect(treeRoot, override);
    }
}
