§2. 요구사항 · 설계 결정 (D1~D6, 대안 비교)

D1. DriftResolutionMode { NONE, MANUAL, AUTO } — autoApplicable boolean 교체 채택

global/marker/DriftResolutionMode 신설, DriftKind 의 4번째 필드를 mode 로 교체. 파생 접근자 isAutoApplicable()(=AUTO)·isManuallyResolvable()(≠NONE)로 서버 가드·템플릿 호출면을 보존. 배정은 §4 표.
비채택: boolean 2개 — manual=false·auto=true 불법 조합이 타입으로 표현 가능(S6 plan D4 확정 사항 재기록).

D2. DriftResolution 전략 bean + Map 디스패치 채택

maintenance/reconciliation/service/resolution/ 패키지에 인터페이스(supportedKind() + resolve(Drift, MarkableScanner))와 구현 2종 — PathDriftResolution(→applyDriftedPath(newPath)) / GhostDbRowClearResolution(→applyGhostClear). 서비스는 List<DriftResolution> 주입 → Map<DriftKind, DriftResolution> 조립(생성자에서 1회). 동작 등가 이전만 — 신규 kind 없음, 스캔 자동 적용도 같은 bean 을 호출(현행처럼 보고서에서 drift 를 제거하지 않음 — 기록 보존 동작 유지, 제거는 수동 apply() 전용).
비채택(S6 plan D3 재기록): enum method-per-constant(DI 불가·방향 역전) / resolutionType 필드 + exhaustive switch(분기 잔존 차선). resolve 시그니처에 scanner 를 넘기는 이유 — 두 구현 모두 도메인 SPI 호출이 전부라 bean 이 scanner 목록을 자체 보유할 필요가 없고, 서비스가 이미 scannerFor/scannersByType 조회를 갖고 있다.

D3. map-miss 널가드는 forced 우회 블록 밖 채택

디스패치 직전 resolutions.get(kind) == null → notApplicable 409 를 무조건 배치 — forced 는 mode 가드만 우회하는 것이지 "해결 로직이 없는 kind" 까지 통과시키지 않는다(persistAndForcedApply 가 kind 제약 없는 public — 비판 검증 MAJOR 반영). 스캔 자동 적용 루프에서는 miss 시 조용히 skip(AUTO 인데 해결 미구현인 kind 는 탐지도 미구현이라 실도달 없음 — 도달 시에도 스캔을 죽이지 않는 원칙).

D4. 설정 통합 — reconciliation.auto-apply.kinds CSV + 마스터 개명 resolution-enabled 채택

① auto-apply-path-drift/auto-apply-ghost-row 2키 삭제 → reconciliation.auto-apply.kinds=(CSV, default 빈 = 현행 default 와 동작 동일). 파싱은 기존 extraRootsCsv 선례대로 String 필드 + 스캔 시 helper 파싱 — 오타 등 무효 kind 명은 IllegalArgumentException 으로 시끄럽게 실패시켜 job 실패 UI(R9-1)로 표면화한다(침묵 무시는 설정이 안 먹는 이유를 영원히 숨김). ② 마스터 reconciliation.auto-apply → reconciliation.resolution-enabled, 서비스 isAutoApplyEnabled() → isResolutionEnabled(), 컨트롤러 뷰모델 autoApplyEnabled → resolutionEnabled, tooltip "시스템 해결 옵션이 꺼져 있어요 (reconciliation.resolution-enabled). 켠 뒤 다시 시도하세요."
비채택: @Value 로 Set<DriftKind> 직접 바인딩 — 빈 문자열 default 의 빈 컬렉션 변환이 ConversionService 구현 상세에 기대는 데다, 무효 값 실패 시점이 부팅으로 당겨져 단위 테스트 하니스(필드 직주입)와 어긋난다. String+helper 가 같은 클래스의 기존 관례.

D5. 예외 클래스 개명 — DriftResolutionNotAllowedException 채택

참조가 service 1 + 테스트 2파일뿐이라 개명 비용 최소. factory 문구 갱신: notApplicable(kind) "…종류는 시스템이 해결할 수 없습니다. {recommendedAction}" / globalOff() "시스템 해결 옵션(reconciliation.resolution-enabled)이 꺼져 있어 거절되었습니다…". ConflictException 상속(409) 불변 — advice 는 상위 타입 매핑이라 무영향.
비채택: 클래스명 유지 + 문구만 갱신 — D4 개명의 취지(어휘 정합)가 예외 이름에서 끊긴다. 참조 3파일은 grep 치환 1회 비용.

D6. 템플릿 — 버튼 조건 manuallyResolvable + 배지 mode 3단 + 버튼 라벨 "적용" 채택

C3 배지: AUTO=n-badge-orange(현행) / MANUAL=n-badge-yellow(기존 토큰) / NONE=n-badge-gray(현행) — "회색 배지 + 활성 버튼" 시각 모순 해소(비판 검증 MINOR). C2 miller tag 는 on/off 2단 자산이라 manuallyResolvable 기준 on/off 로 축약(3단은 상세 C3 가 담당 — 주석 명기). apply 폼 th:if → manuallyResolvable. 버튼 라벨 "자동 적용" → "적용" — R9-3 modal("드리프트 적용"/"{자원} 을(를) 적용할까요?")과 어휘 일치, MANUAL kind 에 "자동" 표기가 오도이기 때문.
DESIGN.md: n-badge-yellow 는 기존 토큰이라 신규 클래스 0 — 수정 불요(S6 plan §8 조건 분기의 "기존 토큰 내" 케이스).

🎬 디스패치·게이트 시뮬레이터 — kind × forced × 설정 조합의 서버 응답 재현

kind 를 고르고 게이트(마스터/kinds CSV)와 경로(수동 apply / forced / 스캔 자동)를 바꾸면 §3 가드 순서 그대로 판정한다. map(해결 bean 등록 현황)은 본 슬라이스 완료 시점 기준 — PATH_DRIFT·GHOST_DB_ROW 만 등록.

kind 마스터(resolution-enabled) auto-apply.kinds

검증 포인트: ① MISSING(NONE)은 수동에서 notApplicable 409, forced 에서도 map-miss 로 409(널가드가 우회 밖) ② MANUAL 2종은 버튼은 뜨지만 bean 미등록이라 409 — 단 탐지도 미구현이라 실화면에 등장 자체가 없음(S6-2-2/2-3 에서 bean 과 탐지가 함께 옴) ③ 스캔 자동은 mode==AUTO 이고 kinds 에 포함될 때만 ④ 마스터 OFF 는 수동·자동 모두 차단(forced 만 통과).

§8. 부산물 · 주의
  • 설정 breaking 일괄(본 슬라이스 1회 종결) — 삭제: reconciliation.auto-apply-path-drift/auto-apply-ghost-row. 개명: reconciliation.auto-apply→resolution-enabled. 신설: reconciliation.auto-apply.kinds=(빈 default). 모든 키가 default 운영이라 실효 영향 0 — 단 사용자 로컬/운영 설정 파일에 구 키가 남아 있으면 조용히 무시됨(report 재명기). properties 주석도 함께 정비.
  • MANUAL 2종의 한 슬라이스 공백 — mode 는 MANUAL 인데 해결 bean·탐지가 아직 없음. 탐지 미구현이라 화면 등장 자체가 불가능해 무해하나, direct POST 로 해당 kind drift 를 만들 수 없는 것까지 확인(탐지가 유일한 생성 경로). S6-2-2/2-3 에서 bean+탐지가 함께 도착하며 해소.
  • 스캔 자동 적용의 report 기록 동작 불변 — 자동 해결돼도 drift 는 보고서에 남는다(현행 동일, 기록 보존). 자동 적용 여부의 화면 표기는 S6-2 범위 밖(필요 시 S6-3 검토).
  • CLAUDE.md 수정 없음. DESIGN.md 수정 없음(n-badge-yellow 기존 토큰).
