§2. 요구사항 · 설계 결정 (D1~D5, 대안 비교)

D1. 전용 modal 공급 — ConfirmModalType 상수 2개 + fragment (enum 다형성 확장) 채택

DRIFT_APPLY("드리프트 적용"/확인 버튼 "적용")·DRIFT_DISMISS("보고 닫기"/확인 버튼 "보고 닫기") 상수를 추가한다. SOFT_DELETE 선례처럼 자원 lookup 없음 — 문구는 JS 가 form 의 data 속성(R9-2 가 이미 라벨·권장조치를 채움)에서 주입. fragment 는 신규 fragments/maintenance/reconciliation-modals.html 에 2개. 이 방식은 enum abstract method 다형성이라 분기 추가가 아니라 OCP 확장이고, drift 의 (resourceType, resourceId) 가 실재하므로 기존 endpoint 시그니처와도 그대로 맞는다(사전 조사의 "lazy 파이프라인 불합치" 우려는 SOFT_DELETE 계열이 자원 lookup 을 안 한다는 실측으로 해소).
비채택: ⓐ 정적 modal 마크업을 페이지에 사전 렌더 — lazy-load 로 통일된 S5-6-2 방향과 역행. ⓑ 기존 restore/soft-delete modal 에 문구 override 계속 — 제목("자원 삭제")이 fragment 에 박혀 있어 근본 해소 불가.

D2. 바인딩은 reconciliation list.js 페이지 로컬 채택

ConfirmModal.bindFormSubmit('data-confirm-drift-apply', …) 등록을 이 페이지 JS 에서 수행 — 전역 confirm-modals.html lazyShell 스크립트 목록을 건드리지 않아 전 management 페이지 blast radius 0 (사전 조사 실행성 렌즈 지적 반영).

D3. async 전환 + 성공 피드백 = sessionStorage 토스트 (R9-1 패턴 재사용) 채택

두 폼에 data-async-submit 부여. 거절(409/404/500)은 전역 AsyncSubmitResult(error-modal.js 가 전 페이지에 제공)의 ErrorModal 경로. 성공은 페이지 로컬로 window.AsyncSubmitResult 를 기존 핸들러 위임 + onSuccess 만 교체(trash-action.js 선례) — sessionStorage 토스트("드리프트를 적용했습니다"/"보고를 닫았습니다") 기록 후 reload. R9-1 의 TOAST_KEY 소비 로직을 그대로 재사용(읽는 즉시 삭제).
백엔드는 PRG(RedirectView+flash) 유지 — submitAsync 가 redirect 를 따라가 resp.redirected=true 로 onSuccess 판정하므로 계약이 이미 맞고, JS 불능 환경의 native submit fallback 도 보존된다. flash 문구만 kind 중립으로 정정("드리프트 적용 완료"/"드리프트 보고를 닫았습니다"). 비채택: 204 JSON 화 — fallback 상실 + 기존 redirect 단언 테스트 churn 대비 이득이 fetch 1회 절약뿐.

D4. native confirm 2곳 → ConfirmModal.openLazy + 신규 SCAN_CONFIRM… 대신 정적 generic modal 1개 채택

Deep 스캔·재발급 confirm 은 drift 항목이 아니라 페이지 액션이라 (resourceType, resourceId) 개념이 없다 — lazy endpoint 시그니처에 억지 dummy 를 넣는 것보다, reconciliation-modals.html 에 genericConfirmModal('reconConfirm') 정적 fragment 1개를 두고 ConfirmModal.open(prefix, opts)(S5-2 base API — title/message/버튼을 호출 시점에 주입) 로 여는 쪽이 정직하다. 문구는 기존 confirm 내용을 계승하되 구현 용어(manifestHash) 를 "파일 내용 해시"로 순화(전면 용어 재편은 R9-4).
비채택: dummy resourceType 로 lazy 신규 타입 — endpoint 의미 오염. 전역 토스트/모달 자산 개조 — scope 밖.

D5. 버튼 개명 — "무시" → "보고 닫기" 채택

실동작(보고서에서 해당 행 제거, 자원 무변경, 재보고 가능)과 일치하는 명칭. dismiss modal 문구: "'{label}' 드리프트 보고를 닫을까요? 보고 목록에서만 제외됩니다. 파일과 자원은 변경되지 않으며 다음 점검에서 다시 보고될 수 있어요."

🎬 modal·피드백 시뮬레이터 — 액션별 확인창과 결과를 체험
액션 서버 응답

Before 와 비교: 적용 확인창은 "복구할까요?"(restore 차용), 보고 닫기는 제목 "자원 삭제"+"삭제할까요?"(soft-delete 차용), 409 는 raw 응답 페이지로 이탈, Deep/재발급은 브라우저 native confirm 이었다.

§3. URL / 데이터 흐름 
[적용]  submit 가로채기(list.js 로컬 bindFormSubmit) → GET /ui/confirm-modal/DRIFT_APPLY?resourceType=..&resourceId=..

[보고 닫기]  동일 흐름, DRIFT_DISMISS + 전용 문구
[Deep 스캔/재발급]  ConfirmModal.open('reconConfirm', {title, message, confirmLabel…}) → 기존 fetch 트리거(R9-1) 그대로
[native fallback]  JS 불능 시 폼 native submit → PRG redirect + flash("드리프트 적용 완료") — 경로 보존
