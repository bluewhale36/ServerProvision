§1. 현재 상태 — R8 정의 대비 코드 실측 (R8-1 reconcile)

§2. 요구사항 · 설계 대안 비교 (D1~D4)

요구사항

  • 4 BrowseController 를 단일 DirectoryBrowseController 로 흡수하고 구 컨트롤러 삭제.
  • includeFiles 를 서버 하드코딩에서 요청 param 으로 이동 — 표시 정책 결정권을 호출처(frontend)로.
  • frontend 6 호출처의 data-browse-url 정합 + 사용자 체감 동작 완전 보존.

D1 — 통합 URL 채택: 중립 /management/browse

대안내용판정 · 사유
A (채택)중립 리터럴 /management/browsebrowse 는 도메인 무관 유틸 → URL 도 도메인 탈색이 의미 정합. 충돌 실측 0 — management 하위 컨트롤러는 전부 리터럴 prefix(os/bios/bmc/board/subprogram)이고 bare /management path-var 매핑 없음, /management/browse 미사용. 보안 설정에 URL 패턴 기반 매처 없음(회귀 불가).
B (기각)path-var /management/{domain}/browse도메인 결합을 URL 에 계속 유지 + {domain} 값 검증 분기 필요(무의미한 분기 신설). 유지 이득 0.
C (기각)/api/browse 등 신규 최상위기존 /management/* 관례 이탈. 관리자 도구라는 소속 정보가 URL 에서 사라짐.

D3 — 신규 컨트롤러 위치·형태 채택: management/common/filesystem/controller/DirectoryBrowseController

  • 패키지 — DirectoryBrowseService/DTO/예외가 사는 management/common/filesystem 아래 controller/ 신설(feature-first 정합: browse 기능의 풀스택이 한 feature 폴더에). 기각: global/ 승급 — global 은 영역 무관 인프라인데 browse 는 management 영역 관리자 도구(URL 도 /management/*)라 소속 불일치.
  • 형태 — 기존 4개와 동일한 @Controller + 메서드 @ResponseBody, 반환 ResponseEntity<?>, 이름은 R8 정의(Notion SSOT) 그대로 DirectoryBrowseController. 기각: @RestController 전환 — 네이밍 규약상 *RestController 접미사가 따라와 Notion 정의명과 어긋나고, advice 매칭은 Accept 기반이라 형태 전환 이득 0.

D4 — 구 컨트롤러 처리 채택: 4개 즉시 삭제 (한 줄 껍데기 잔존 없음)

  • R8 ③이 "R8-2 보류 시 껍데기 잔존 허용"이라 했으나 본 슬라이스가 바로 그 잔존을 제거하는 단계 — 이행기 이중 URL(신구 병행) 불필요. frontend 호출처가 전부 data-browse-url 템플릿 속성 주입이라 템플릿 6곳 교체로 참조가 원자적으로 이동하고, 외부(비 frontend) 소비자는 없다.
  • 기각: 구 URL 을 redirect/alias 로 유지 — 소비자가 자사 frontend 뿐이라 하위호환 비용을 질 이유 없음(YAGNI).
🎬 호출처 6곳 이행 시뮬레이터 — 통합 전후 요청/표시 비교

호출처와 통합 전/후를 골라 "경로 브라우저 열기"를 누르면, 실제 코드 기준으로 wire 요청 · 서버 includeFiles 해석 · 클라 필터 · 최종 표시를 시뮬레이션한다. 6곳 모두 표시 결과가 보존되는지 직접 확인.
