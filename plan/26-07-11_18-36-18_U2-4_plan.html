§1. 지금 사용자가 겪는 상황 (유스케이스 진단)

정의서의 OS 단계는 U2-1~U2-3 을 거치며 대부분 완성됐다 — 계열 선택, 환경·패키지 그룹 제약, 파티션 규칙, 접근성 규칙까지. 그러나 마지막 네 가지 상황에서 여전히 어색함이 남는다:

상황지금 일어나는 일
① Rocky Linux 정의서에서 "기본 파티션 자동 생성" 클릭 Ubuntu 와 똑같은 ext4 구성이 생성된다 — RHEL 계열 설치기의 관례 파일시스템은 xfs 인데, 사용자가 매번 손으로 바꾸거나 관례와 다른 구성으로 설치하게 된다.
② Ubuntu hostname 에 My_Server! 입력 그대로 저장된다 — 실행 시점(autoinstall)에야 잘못된 hostname 으로 실패한다. 패키지 이름·RHEL 등록 키도 마찬가지로 형식 검증이 없어 스크립트 주입 표면이 남아 있다.
③ ISO 를 아직 안 올린 OS 로 설치 단계 작성 아무 안내 없이 저장된다 — 펌웨어 부재(노란 경고)와 달리 ISO 부재는 침묵. 그런데 OS 설치는 펌웨어와 달리 건너뛸 수 없는 단계라 실행이 통째로 실패한다.
④ 코드 주석의 "U2-4 가 채운다" 약속들 5곳의 이연 마커 중 selinux 값·환경/그룹 실존은 이미 해소됐는데(U2-1 Layer A·U2-2-3) 주석은 여전히 미래형 — 문서 drift.
즉 U2-4 는 큰 구조 변화가 아니라 OS 흐름의 마감 손질이다 — 검증 그릇(계열 검사기 SPI·Layer A·ExecutionWarning 관용구)은 전부 준비되어 있고, 이번엔 그 안을 채운다. 엔티티·스키마 무변.
§2. 요구사항 (확정) + 비목표
  1. [개정] 파티션 프리셋 분기 없음 — 사용자 확정: /boot·/ 등은 계열 무관 ext4 고정(현행 프리셋 유지). 이연 주석만 "분기하지 않기로 확정"으로 현행화.
  2. 입력 문자셋 마감(Layer A) — hostname(RFC 1123), 패키지명(RHEL additionalPackages·Ubuntu packages), RHEL 등록 키 — Kickstart/autoinstall 주입 방지. §4 D2.
  3. [개정·격상] ISO 선택 계약 — OS 버전 선택 후 ISO 도 선택한다(OS_INSTALLATION 베이스에 isoId 필수). 사용 가능한 ISO 가 없는 OS 는 select 에 렌더하지 않고(옵션 제외), backend 는 ISO 실존·OS 소속·enabled 를 검증(1단 검사기 공통 — forging 차단 포함). deprecated ISO 는 기존 관용구(modal+뱃지). 저장 후 ISO 소실/비활성 레이스는 상세 "실행 불가" 경고로 안내. §4 D3'.
  4. 이연 마커 정리 — 이미 해소된 약속(selinux·환경/그룹)의 주석 현행화 + 이번에 채우는 곳 반영.
비 목표 : Debian 계열 OS 후처리 단계(§4 Q1 — 보류 권고) · Windows 계열(확장 절차 문서화로 충분 — U2-3-1 D6) · 실행 엔진(Stage 4) · 파티션 프리셋의 디스크 다중 구성 편집기.
🎬 시뮬레이터 — 계열 프리셋 · 입력 검증 · ISO 경고를 직접 체험

OS 계열을 전환하고, 실제 값을 입력해 보라 — 이 슬라이스가 끝나면 폼이 정확히 이렇게 동작한다 (검증식은 서버 Layer A 와 동일한 정규식을 그대로 실행).

설치할 OS : 이 OS 의 ISO :
① "기본 파티션 자동 생성" 결과 — 계열 관례 파일시스템 (D1)
② 입력 검증 (D2 — 서버와 같은 정규식)
체험 포인트 — ① 계열 전환 시 파일시스템만 바뀌고 크기 구성은 유지된다(노란 강조가 분기 지점). ② hostname 에 My_Server!, 패키지에 vim; rm -rf / 같은 값을 넣어보면 저장 전에 거절된다. ③ ISO 토글을 끄면 경고가 뜬다 — 펌웨어의 노란 "건너뜀" 과 달리 붉은 기운의 "실행 실패" 경고(D3 의미론 구분).
§4. 설계 결정 + 대안 비교

D1 — 파티션 프리셋: 크기 공통·파일시스템만 계열 분기

  • 분기 축은 OSFamily(RHEL→xfs / Debian→ext4). 근거: Anaconda(RHEL 계열)의 기본 파일시스템은 xfs, Subiquity(Ubuntu)는 ext4 — 설치기 관례를 따르는 것이 "기본 생성"의 의미다. 크기 구성(1G EFI/1G boot/16G swap/grow root)은 사용자 확정값을 계열 공통으로 유지.
  • 구현 위치: 조회 계약이 이미 findDefaultPartitions(osName) 로 파라미터를 받고 있었다(무시 중) — 파라미터를 osFamily 로 교체하고 폼 JS 가 선택 옵션의 계열을 넘긴다. 계열→fs 사상은 선언적 맵(검사기 SPI 를 새로 만들 규모가 아님 — 상수 2줄, 과분리 경계).
  • /boot/efi 는 EFI 유지(계열 무관 — 파티션 규칙 SSOT 와 일치), swap 도 공통.

D2 — 문자셋 마감: 전부 Layer A(@Pattern) — 검사기 아님

필드규칙

전부 형식 검증(값 그 자체의 규칙)이므로 Layer A 소관 — 계열 검사기(Layer B)는 영속 참조·cross-field 만 담는 기존 층위 원칙 그대로. 검사기에 넣는 대안은 기각(불필요한 빈 성장 + 통합 400 경로가 fieldErrors 다건 렌더를 잃음).

D3 — ISO 부재 경고: ExecutionWarning 관용구 확장 + 실패 의미론 구분

  • 판정: OS 설치 단계의 선택 OS 에 사용 가능한(enabled·비삭제) ISO 가 0개 — 조회 시점 판정(ISO 업로드 시 자동 해소), 저장은 막지 않음(정의서=재사용 템플릿, 펌웨어 방침과 동일).
  • 의미론 구분(중요): 펌웨어 부재 = 해당 축 skip+PARTIAL_SUCCESS(선행 확정) / ISO 부재 = OS 설치 수행 불가 = 실행 실패(OS 설치는 정의서의 본체 단계라 건너뛰면 이후 단계가 무의미). 문구: "설치 이미지(ISO)가 없습니다 — 실행 시점까지 등록되지 않으면 실행이 실패합니다." 상세 뱃지도 "실행 시 건너뜀 가능"(노랑)이 아닌 "실행 불가"(빨강 계열, n-badge-red) 로 구분.
  • 폼: OS 선택 시 select 아래 경고 배너(펌웨어 경고와 같은 자리·강조 스타일, 붉은 변형) / 상세: OS_INSTALLATION 카드 head 뱃지. OS 후처리(OS_SETTING)는 ISO 불필요 — 대상 아님.
  • 데이터: OS 옵션 응답에 installable(ISO 보유) 플래그 + 상세 조립 시 동일 판정 — 판정식 한 곳(Query 서비스 헬퍼)만.

§7. 예외 계층 — 신규 0

이번 슬라이스는 전부 Bean Validation(Layer A)과 표시 메타(경고)라 신규 예외 클래스가 없다 — 400 은 기존 MethodArgumentNotValidException 다건 fieldErrors 경로, 경고는 예외가 아닌 응답 필드. 기존 예외 전부 무변.

§8. 부산물 / 주의
  • Stage 4 실행 의미론 누적 기록: 펌웨어 부재=skip+PARTIAL_SUCCESS / ISO 부재=FAILED(수행 불가) / 마커 무결성 실패=FAILED — 세 층이 Notion 에 모여 Stage 4 의 입력이 된다.
  • 기존 저장본 영향: hostname @NotBlank 승격으로 hostname 없는 구 Ubuntu 저장본은 수정 저장 시 400 — 조회·상세는 무영향(스모크 데이터는 정리된 상태라 실데이터 파장 없음).
  • U2-4 이연 마커 5곳 주석 현행화(§1 ④) — 이미 해소된 selinux(U2-1)·환경/그룹(U2-2-3) 명시.
  • U2-4 완료 = U2 전체 종료 → 다음 갈래는 U3(서버 할당) 또는 E0(Redfish 연결 준비) — §11.
  • 커밋 없음(사용자 지시 대기). S·R 트랙 미커밋 변경(워킹트리의 reconciliation 파일들)은 이 세션에서 불가침.
