1. 현재 상태 — 토글 시 selectId / selectBoardId / selectKind 가 휘발

1.1 마크업 현황 — 5 도메인 list 페이지 모두 동일 inline 핸들러

5 도메인 list 페이지 (OS / Board / BIOS / BMC / Subprogram) 의 헤더 체크박스는 모두 다음 inline 핸들러를 사용 :

<label class="n-checkbox">
    <input type="checkbox"
           th:checked="${includeDeleted}"
           onchange="window.location.href = this.checked
                       ? '?includeDeleted=true'
                       : window.location.pathname">
    삭제된 항목 포함
</label>

1.2 결과 — 토글 후 querystring 통째로 휘발

this.checked ? '?includeDeleted=true' : window.location.pathname 는 현재 URL 의 다른 쿼리 파라미터 (selectId, selectBoardId, selectKind 등) 를 모두 버린다. miller column 의 선택 상태는 모두 querystring 기반이므로, 토글 직후 사용자가 보던 자원 선택이 사라진다.

1.4 사용자 동기 (Notion S5-4 원문)

Notion : "'삭제된 항목' 체크 시 기존의 miller 선택이 남아있도록 수정."

1.5 왜 inline 으로 풀린 채로 두는가

  • 5 페이지 각각에 같은 핸들러가 복붙되어 있어 — 도메인별 list.js 에 흩어두면 중복 5 곳 발생 (CLAUDE.md 의 코드 중복 절대 지양 원칙 위배)
  • BIOS / BMC 의 selectBoardId, Subprogram 의 selectKind 처리가 도메인별로 미묘하게 다르나, 본질은 "현재 URL 의 querystring 에서 includeDeleted 만 토글" 로 단일화 가능
2. 요구사항
  1. '삭제된 항목 포함' 체크박스 토글 시 사용자가 선택한 miller 항목 (selectId / selectBoardId / selectKind) 이 유지된 채로 reload
  2. 5 도메인 (OS / Board / BIOS / BMC / Subprogram) 에 동일하게 적용 — 도메인별 분기 없이 동일 JS 1 곳
  3. Subprogram 페이지의 2 체크박스 (Driver / Utility 섹션) 모두 동일 동작
  4. Server-side 변경 0 — 백엔드는 이미 모든 쿼리 파라미터를 받고 있으므로 frontend 만 손봐도 충분
  5. 새 JS 는 일반 모듈 — 다른 페이지 ("내 자원만" / "활성만" 같은 향후 querystring 토글) 가 재사용할 수 있게 generic 한 토글 빌더
  6. CLAUDE.md 의 "중복된 코드 절대 지양" 준수 — inline 핸들러 5 곳 → 일반 JS 모듈 1 곳으로 흡수

3. URL · 데이터 흐름

3.1 새 endpoint — 없음

Server-side 는 이미 모든 쿼리 파라미터를 받음. 변경 표면은 frontend 만.

3.2 URL 합성 알고리즘 (JS)

// static/global/include-deleted-toggle.js
(function () {
    'use strict';

    /**
     * 체크박스의 onchange 핸들러. data-include-deleted-toggle 속성을 가진
     * checkbox 를 자동 발견해 attach.
     *
     * 동작 :
     *   1. 현재 location 의 querystring 을 URLSearchParams 로 파싱
     *   2. checkbox 체크 여부에 따라 includeDeleted 파라미터를 set / delete
     *   3. 다른 파라미터 (selectId / selectBoardId / selectKind 등) 는 그대로 보존
     *   4. window.location.search 로 redirect
     */
    function buildNextUrl(checked) {
        const params = new URLSearchParams(window.location.search);
        if (checked) params.set('includeDeleted', 'true');
        else params.delete('includeDeleted');
        const q = params.toString();
        return window.location.pathname + (q ? '?' + q : '');
    }

    function attachAll() {
        const checkboxes = document.querySelectorAll(
            'input[type="checkbox"][data-include-deleted-toggle]'
        );
        checkboxes.forEach(cb => {
            cb.addEventListener('change', () => {
                window.location.href = buildNextUrl(cb.checked);
            });
        });
    }

    if (document.readyState === 'loading') {
        document.addEventListener('DOMContentLoaded', attachAll);
    } else {
        attachAll();
    }

    // unit test / 다른 모듈 재사용용 export
    window.IncludeDeletedToggle = { buildNextUrl: buildNextUrl };
})();

7. 예외 / 회귀

예외 신설

없음 — frontend 전용 슬라이스.

회귀 위험

  • 5 도메인 list.html 의 다른 inline 핸들러는 무관 — 본 슬라이스는 data-include-deleted-toggle 마커 붙은 checkbox 만 처리
  • Subprogram 의 2 개 체크박스 가 동시에 attach — 두 섹션이 같은 querystring 토글 키 (includeDeleted) 공유. 토글 시 두 섹션 모두 같은 모드로 reload (단일 진실 — 의도된 동작)
  • JS 모듈이 DOMContentLoaded 후 attach 되므로 SSR 첫 페인트에서 inline 핸들러 부재는 사용자 영향 없음 (체크박스 표시는 정상)

JS 로딩 실패 시 fallback

주의 include-deleted-toggle.js 로딩 실패 시 체크박스는 단순 토글만 하고 reload 안 함. 사용자가 "체크해도 변화 없음" 인지 가능. inline onchange 보다 안전한 fallback — 잘못된 URL 로 redirect 되는 회귀 없음. JS 실패는 Network 탭에서 즉시 확인 가능.
8. 부산물 · 주의

스코프 경계

  • 유사 패턴의 다른 querystring 토글 (예 : "내 자원만" / "활성만") — 본 슬라이스 외. 단 IncludeDeletedToggle.buildNextUrl() 은 충분히 generic 해서 향후 동일 모듈 확장 가능 (data-querystring-toggle="key" 같은 일반화 후보)
  • 휴지통 페이지 (/maintenance/trash) 는 항상 휴지통만 보이는 페이지라 토글 자체가 없음 — 본 슬라이스 영향 없음
  • Provisioning 영역 (U1 setting / U2 server) 에는 아직 '삭제된 항목' 토글 없음 — 향후 도입 시 동일 마커 패턴으로 흡수

  • 5 도메인 list.html 모두 동일 패턴으로 변경 — 단일 슬라이스 내 일괄 처리. 동반 수정 없음

CLAUDE.md 수정 사항

  • 없음 — 기존 "코드 중복 절대 지양" 원칙의 자연스러운 이행

향후 일반화 가능성

재사용 data-include-deleted-toggle 마커 + URLSearchParams 기반 querystring 보존 패턴은 다른 토글 (예 : "오늘만 / 최근 7일" 같은 시간 필터, "내 자원만" 같은 owner 필터) 에도 동일 적용 가능. 본 슬라이스가 만든 일반 모듈이 자연스럽게 확산되도록 함수명 / 마커명을 specific 하게 유지 (include-deleted) — 다른 토글이 도입될 시 별 마커로 분리하는 게 명료성 측면에서 더 좋음.

대안 검토 — 왜 일반 querystring 토글 모듈로 미리 일반화하지 않는가

  • 현재 토글은 includeDeleted 하나뿐 — 일반화 시점이 아직 아님 (over-abstraction 회피)
  • 두번째 토글이 도입될 때 자연스럽게 일반화 가능 — data-querystring-toggle="key:value" 같은 형태로 진화. 본 슬라이스는 그 시점까지의 정직한 중간 형태
