§1. 현재 상태 (진단)

E1-2 이후 게스트는 30초 폴링·수집 보고·완주로 상태를 계속 바꾸는데, 운영자 화면은 새로고침해야만 그것을 본다:

자산있음비어 있음 (이 슬라이스가 채움)
상태 변화 발생 지점등록·전이·수집·완주·실패·개시·회수·접촉 — 전부 코드에 명시된 트랜잭션(서버는 변화 시점을 정확히 안다)화면으로 가는 통로 없음 — 변화를 아는 쪽(서버)이 모르는 쪽(브라우저)에 알릴 수단이 없다
화면 갱신PRG(자기 액션 후 리다이렉트) + 수동 F5게스트가 일으킨 변화(체크인·완주·실패)는 운영자가 우연히 새로고침해야 보인다 — UC-3 의 "서로 기다리는" 착각도 이 공백의 산물
유사 선례background-jobs.js — fetch 주기 폴링(JSON) + DOM 조립. 최근 그 폴링의 로그 노이즈를 정비했음"변화 없는데 계속 묻는" 폴링을 서버 화면까지 증설하는 것은 역방향 — push 인프라 부재
이벤트 인프라없음 — 이 프로젝트의 첫 @EventListener 는 S5-13 에서 AFTER_COMMIT 방식으로 이연해 둔 상태그 이연분의 첫 실사례 자리가 여기다
§2. 요구사항 (확정) + 비목표
  1. SSE 단일 스트림 — GET /provisioning/server/stream (text/event-stream). 구독 필터는 클라이언트(상세 페이지는 자기 id 만 반응) — 운영자 소수·이벤트 저빈도라 서버측 필터는 과잉(§8 Q3).
  2. 도메인 이벤트 — GuestServerChangedEvent(serverId) 단일 record. 상태가 변하는 application service 트랜잭션(등록·전이·수집 소비·완주·실패·개시·회수·운영자 수정·수동 실패·재시도·접촉)에서 발행.
  3. AFTER_COMMIT 구독 — @TransactionalEventListener(phase = AFTER_COMMIT) 이 커밋 확정 후에만 브라우저로 신호를 보낸다(롤백된 변화를 화면에 쏘는 사고 차단). S5-13 이 이연해 둔 첫 EventListener 의 실사례.
  4. 신호-재조회 패턴 — payload 는 서버 id 뿐. 브라우저는 신호를 받으면 같은 URL 을 다시 fetch 해 관심 영역만 교체한다(§4) — 상태 데이터의 SSOT 가 기존 조회 서비스 하나로 유지되고, SSE payload 스키마가 아예 생기지 않는다.
  5. 부분 갱신 + 편집 보존 — 상세 페이지의 읽기 전용 영역(배지·인벤토리·NIC·진행·원장·액션 버튼)만 교체하고, 인라인 수정 폼은 절대 교체하지 않는다(입력 중 유실 방지 — §8 Q5). 목록 페이지는 테이블 영역 교체.
  6. 연결 수명 관리 — emitter 레지스트리(등록·완료·타임아웃·send 실패 시 제거) + 25초 heartbeat(프록시 idle 절단 방지) + 브라우저 EventSource 의 자동 재연결에 위임(재연결 코드 직접 작성 없음).
비 목표 :
  • WebSocket/STOMP — 양방향·브로커 인프라는 단방향 push 요구에 과잉(§8 Q1 탈락 기록). 원격 콘솔류 양방향 요구가 실재할 때 재고.
  • background job 폴링의 SSE 전환 — 같은 인프라의 후속 소비처 후보로만 기록(scope 비대 방지).
  • 이벤트 종류별 세분(전이/실패/접촉 …) — 소비자가 종류를 구분할 필요가 생길 때(예: 알림음) 확장. 지금은 "변했다" 하나면 충분.
  • 대량 게스트 스로틀 — 접촉 신호는 게스트당 30초 자연 주기. 수백 대 규모의 서버측 디바운스는 실측 후(§8 잔존).

§8. 예상 부산물 / 주의 / 결정 사항

잔존 책임 · 주의 (정직 기록)

  • 접촉 신호 빈도 — 게스트당 30초 1신호. 수십 대까지는 무의미한 부하지만 수백 대 규모에선 서버측 디바운스(예: 5초 창 합침) 재고 — 실측 후(지금 최적화 금지).
  • MVC 스레드 점유 — SseEmitter 는 비동기 컨테이너 처리라 요청 스레드를 붙잡지 않는다. 다만 구독자 수 = 열린 연결 수 — 운영자 소수 전제 명기.
  • 재조회 비용 — 신호마다 페이지 1회 재조회. 300ms 디바운스로 연속 신호를 합치므로 폴링보다 항상 적거나 같다.
  • IntelliJ devtools 재시작 — SSE 연결은 앱 재시작 시 끊기고 EventSource 가 자동 재연결한다(개발 흐름 무영향).
