1. 현재 상태

1.1 현재 진입점 탐지 우선순위

BundleEntrypointDetector.detect(Path treeRoot, String override) 의 단계 :

1
override 가 있으면 EntrypointPolicyService 검증 후 그대로 반환
사용자가 진입점 명시 지정
2
트리 전체에 (marker / OS 잡파일 제외) 파일이 1 개면 그 파일
단일 파일 번들 — ASUS .CAP 만 있는 경우 cover
3
루트 수준의 f.nsh 우선 (대소문자 무시)
GIGABYTE 표준 컨벤션
4
루트 수준의 flash.nsh fallback
일부 보드 컨벤션
5
루트의 *.nsh 가 정확히 1 개면 그것
단일 .nsh 자동 채택
!
루트 *.nsh 2 개 이상 → EntrypointAmbiguousException
0 개 → EntrypointNotFoundException

2. 요구사항 (확정)

2.1 Notion 명세 (S5-11 본문)

ASUS 의 경우 .CAP 또는 .cap 의 단일 파일, GIGABYTE 의 경우 f.nsh 파일. 제조사마다 조금씩 다른 진입점 상대경로를 자동으로 탐지하는 로직을 수정. (Fujitsu 제외)

2.3 ASUS .cap 다중 시나리오

  • 루트에 .cap / .CAP 가 정확히 1 개 → 채택
  • 2 개 이상 → EntrypointAmbiguousException + 후보 리스트에 .cap 도 포함
  • 0 개 → 단계 4 (f.nsh) 로 fall-through

2.4 .cap 우선순위가 f.nsh 보다 높은 이유

  • 실 운영 케이스에서 ASUS .cap 번들에 부속 .nsh 가 동봉되는 사례가 거의 없음 (ASUS 는 UEFI 자체 펌웨어 업데이트)
  • 혹시 동봉돼도 ASUS 의 의도된 진입점은 .cap → .cap 우선이 자연스러움
  • 실제 충돌 발생 시에는 override 로 사용자가 명시 가능

2.5 추가 제약 (불가침)

  • override 동작 그대로
  • 단일 파일 (단계 2) 동작 그대로 — 트리에 .cap 1 개만 있으면 단계 2 에서 잡히는 게 정상
  • BundleFilePolicy.isIgnorable 정책 그대로 (.DS_Store / Thumbs.db 제외)
  • 대소문자 무시 (.CAP / .cap / .Cap 모두 동일 취급)
  • BIOS_BUNDLE 외 다른 자원 (BMC / Subprogram) 에는 영향 없음 — 본 detector 가 BIOS 트리에만 적용

3.3 신규 헬퍼

private List<Path> listRootCapFiles(Path treeRoot) {
    List<Path> result = new ArrayList<>();
    try (Stream<Path> children = Files.list(treeRoot)) {
        children.filter(Files::isRegularFile)
                .filter(p -> p.getFileName().toString().toLowerCase().endsWith(".cap"))
                .filter(p -> !BundleFilePolicy.isIgnorable(p))
                .forEach(result::add);
    } catch (IOException e) {
        throw new BundleExtractionException("루트 .cap 스캔 실패 : " + treeRoot, e);
    }
    return result;
}

6. Step 8 통합 테스트 시나리오

6.1 신규 시나리오 — 4 개

  1. ASUS 단일 .CAP + 부속 README — X99E-WS.CAP + README.txt → entrypoint = X99E-WS.CAP
  2. ASUS 단일 .cap + 다중 부속 — x99.cap + flash.bin + tool.exe → entrypoint = x99.cap
  3. ASUS .cap 다중 → ambiguous — foo.cap + bar.CAP + 부속 → EntrypointAmbiguousException + 후보에 .cap 포함
  4. .cap + f.nsh 둘 다 존재 시 .cap 우선 — x99.cap + f.nsh → entrypoint = x99.cap (정책 §2.4)

6.2 회귀 대상

7. 예외 / 회귀

7.1 신규 예외 — 없음

기존 EntrypointAmbiguousException / EntrypointNotFoundException 재사용. 메시지에 .cap 후보 명시 추가만.

7.2 ambiguous 후보 리스트 갱신

루트 .cap 이 2 개 이상이면 ambiguous. 후보 리스트는 .cap + .nsh 합집합으로 정렬.

7.3 .cap 정책 안전성

  • 대소문자 무시 비교 (toLowerCase().endsWith(".cap")) — Linux 의 case-sensitive FS 에서도 안정
  • BundleFilePolicy.isIgnorable 필터링 적용 — .DS_Store 등은 .cap 후보에서도 자동 제외
  • 마커 파일 (.provision.json) 은 .cap 으로 끝나지 않으므로 영향 없음
