0. 안 B (기존 사용자 제안) vs 안 C (본 계획)의 차이

0.1 사용자 명제의 검증 결과 (2026-05-21)

사용자 직전 메시지 ("미리 삽입되어 있을 경우 오작동/악용 가능성") 검증 — modal 자체가 아니라 form 의 data-typed-name 속성에 expected value 가 박혀있다는 사실 확인 :

<!-- 5 list 페이지에 공통 패턴 -->
<form data-confirm-purge=""
      th:data-typed-name="${bios.name}"   <!-- ← 페이지 진입만으로 expected 노출 -->
      th:action="@{/management/bios/{boardId}/bios/{biosId}/purge(...)}"
      method="post">
    <button>영구 삭제</button>
</form>

이 때문에 :

  • 안 B (단순 modal lazy-load) 만으로는 보안 향상 0. 페이지 진입 즉시 form 의 data-typed-name 으로 expected 알 수 있음. modal 만 lazy-load 해도 attack surface 동일.
  • 실효 있는 friction 추가는 expected value 자체를 form 에서 떼어내 modal trigger 시점에만 BE 가 응답하는 안 C 가 필요.

0.2 두 안의 구조 차이

안 B 단순 modal lazy-load

변경 표면

  • list.html 의 <div th:replace="...modal..."> 5~6 개 제거
  • 단일 modal placeholder <div id="modalLazySlot"> 추가
  • JS : form submit 가로채기 → fragment fetch → inject → 표시
  • form 의 data-typed-name 그대로 유지

보안 향상

  • ❌ expected value 가 여전히 form 에 노출 → 페이지 진입만으로 보임
  • ❌ DevTools 로 form 의 data-confirm-approved 설정 + resubmit 으로 modal UX 우회 가능 (서버는 typed-name 만 보므로 통과)

실제 이점

  • 초기 DOM 크기 감소 (30+ hidden element 제거)
  • 마크업 응집

안 C ✓ modal + expected value 모두 lazy-load

변경 표면

  • 안 B 변경 전부 +
  • list.html 의 form 에서 data-typed-name 제거, 대신 data-resource-type + data-resource-id 보유
  • JS : form submit 가로채기 → fragment fetch (BE 가 자원 lookup + expected value 계산 + modal 에 직접 박아 응답) → inject → 표시
  • typed-name 검증은 modal 내부에서 사용자 입력 vs modal 내 expected (DOM 비교) + 서버 측 재검증 (기존 그대로)

보안 향상

  • ✅ 페이지 진입만으로 expected 노출 안 됨
  • ✅ expected 알아내려면 trigger endpoint 호출 필요 → 권한 검증 + audit log + rate-limit 가능 지점 생김 (BE 가 modal trigger 를 sensitive action 로 인지 가능)
  • ⚠ DevTools 로 modal endpoint 호출 자체는 막을 수 없음 (해당 자원에 권한 있는 사용자라면 응답 받음). 그러나 friction + audit trail + rate-limit 라는 정량적 안전장치가 추가됨 — 무작위 자원명 harvest 방어
  • ✅ 기존 BE typed-name verifier 그대로 (defense in depth)

실제 이점

  • 안 B 이점 전부 +
  • UI 안전장치 (modal 경고) 가 우회 어려워짐 — modal 없이는 expected 모름
  • 각 trigger 가 BE 에서 audit 가능

1.2 typed-name expected value 의 form 노출

5 페이지의 영구삭제 (purge) form 마다 :

<form data-confirm-purge=""
      th:data-typed-name="${...}"           <!-- ← Thymeleaf 가 페이지 렌더 시점에 값 박음 -->
      th:data-resource-label="${...}"
      th:action="..."
      method="post">
    ...
</form>

각 form 의 data-typed-name 값은 페이지 진입 즉시 DOM 에 노출되며, document.querySelectorAll('[data-typed-name]') 한 번으로 페이지의 모든 typed-name 을 한꺼번에 수집 가능.

1.3 nudge / delete-intent token 의 흐름

이 두 modal 은 server response 트리거 후에만 표시 — 페이지 진입 시점에 토큰이 DOM 에 없음. 본 슬라이스의 보안 명제와는 직접 관련 없음. lazy-load 의 부수 대상으로만 처리.

1.4 BE 측 방어 (기존)

  • TypedNameVerifier interface — 자원의 displayName() 과 사용자 입력 typedName 일치 확인. TypedNameMismatchException (400) 으로 거절
  • 각 도메인 컨트롤러가 typedNameVerifier 주입 받음 (현재 5 도메인)
  • 데이터 보호는 BE 가 담당 — modal 우회 시도해도 typed-name 검증 못 넘으면 영구삭제 불가
2. 요구사항 (안 C)
  1. list.html 들에서 사전 include 된 modal 5~6 개 모두 제거
  2. 대신 단일 modal placeholder <div id="modalLazySlot"></div> 1 개만 남김
  3. 각 form 의 data-typed-name 속성 제거. 대신 data-resource-type (Enum) + data-resource-id (Long) 만 보유
  4. form submit 가로채기 시점에 JS 가 GET /ui/confirm-modal/{modalType}?resourceType=...&resourceId=... 로 modal fragment 요청
  5. 컨트롤러 : 자원 lookup + expected value (typed-name 등) 계산 → modal fragment 에 직접 박아 응답
  6. JS : 응답 HTML 을 placeholder 에 inject → 사용자 입력 → modal 내부에서 typed-name 일치 검증 (DOM 내 비교) → 일치 시 form approve flag 설정 → resubmit
  7. 서버 측 TypedNameVerifier 그대로 (defense in depth)
  8. modal 닫힘 (취소 / 확인 후) 시 placeholder 비우기

3.6 ConfirmModalType enum 다형성 — 분기 흡수 도식

CLAUDE.md 의 "조건 분기문 무분별 확장 금지" 원칙 — modalType 별 처리는 enum 의 abstract method 로 응집한다.

public enum ConfirmModalType {

    PURGE {
        @Override
        public ModelContext resolveModel(Long resourceId, ResourceType type, ConfirmModalContext ctx) {
            String expected = ctx.verifier().resolveExpectedName(type, resourceId);
            String label    = ctx.scanner(type).findById(resourceId).displayName();
            return ModelContext.of("expectedName", expected, "label", label);
        }
        @Override public String fragmentView() { return "fragments/management/confirm-purge :: modalCard"; }
    },

    NUDGE_REPLACE {
        @Override
        public ModelContext resolveModel(Long targetId, ResourceType type, ConfirmModalContext ctx) {
            // nudge session 의 targetId 자원 lookup + expected.
            String expected = ctx.verifier().resolveExpectedName(type, targetId);
            return ModelContext.of("expectedName", expected, "targetId", targetId);
        }
        @Override public String fragmentView() { return "fragments/management/nudge-modal :: replaceCard"; }
    },

    SOFT_DELETE, DEPRECATE, RESTORE, DELETE_REJECT {  // S5-6-2 묶음
        @Override
        public ModelContext resolveModel(Long resourceId, ResourceType type, ConfirmModalContext ctx) {
            // typed-name 불요 — label / cascade preview 만.
            Markable m = ctx.scanner(type).findById(resourceId);
            return ModelContext.of("label", m.displayName(), "cascade", m.cascadePreview());
        }
        @Override public String fragmentView() { return /* per-constant view */; }
    },

    TRASH_RESULT {
        @Override
        public ModelContext resolveModel(Long ignored, ResourceType ignored2, ConfirmModalContext ctx) {
            // 자원 lookup 없음. 메시지를 그대로 전시.
            return ModelContext.of("message", ctx.queryParam("message"));
        }
        @Override public String fragmentView() { return "fragments/management/trash-result-modal :: modal"; }
    };

    public abstract ModelContext resolveModel(Long resourceId, ResourceType type, ConfirmModalContext ctx);
    public abstract String fragmentView();
}

Controller 는 단순히 modalType.resolveModel(...) + modalType.fragmentView() 만 호출 — 6 modal × 변형 사이의 if/switch 분기 없음.

7. 예외 / 회귀

예외 신설

  • global/ui/exception/ModalContextNotFoundException extends NotFoundException (404) — resourceType + resourceId 로 자원 lookup 실패

회귀 위험

  • typed-name expected value 의 출처 변경 — JS 가 form 의 data 속성 대신 modal 내 expected slot 의 textContent 를 비교. 단위 / 통합 테스트에서 회귀 확인 필수
  • modal trigger 시점 latency — 페이지 진입 즉시 modal 표시가 아니라 fetch + inject. 0.1~0.3 초 지연 가능. 첫 클릭 직후 사용자 인지 미세 차이 — 무해 수준
  • placeholder DOM 충돌 — 6 페이지의 placeholder id 가 모두 modalLazySlot 으로 동일하지만 페이지당 1 개라 무관
  • 여러 form 의 modal trigger 가 빠르게 연속 클릭 — JS 가 직전 modal 을 placeholder 에서 제거하고 새 modal 을 inject. race condition 가능성 → fetch 진행 중 새 트리거 발생 시 직전 fetch 결과 폐기
BE typed-name verifier 의 위치 무변 — 본 슬라이스는 UI / friction 측 향상만 다룸. 서버 측 보호선은 그대로 유지 (defense in depth).
