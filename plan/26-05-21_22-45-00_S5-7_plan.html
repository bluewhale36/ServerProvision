🚨 CP1 사용자 결정 사항 (코드 진입 전 합의 필요)
본 슬라이스는 CP1 plan 산출까지만. Notion 본문에 "진입 전 개발자와 구체적 상의 진행 필수" 명시. 사용자 합의 없이 코드 변경 진행 안 함. 아래 4 결정 사항 확정 시 CP2 진입.

D1 자동화 트리거 시점 — create() vs addIso()

Notion 본문 예시는 OSImageService.create() 의 step 3 직후에 자동 작업을 끼우는 흐름이었다. 그러나 :

  • create() → 빈 OSImage 메타데이터만 생성. ISO 파일은 없는 상태 → comps.xml 자동 추출 불가능
  • addIso() → 실제 ISO 파일 등록. comps.xml 추출 가능 ← 의도된 트리거 시점일 가능성 높음
A. addIso() 직후 자동 트리거 (권장) ISO 등록 완료 → 동기 응답 후 background Job 으로 comps 추출 시작. 실제 데이터 추출이 의미 있는 시점.
B. create() 직후 (Notion 본문 그대로) OS 이미지 생성 즉시 작업 등록 — 그러나 ISO 없으면 noop 또는 skip 처리. 사용자 의도와 일치할 수도 (예: 향후 Ubuntu 의 "OS 메타 등록만으로 가능한 자동화" 가 있다면).
C. 두 곳 모두 — 각 작업 별 트리거 시점 분리 comps 추출 = addIso 직후 / 다른 작업 (예: Ubuntu cloud-init 템플릿 생성) = create 직후. enum / strategy 가 자체적으로 트리거 시점 명시.

D2 분기 방식 — Notion 본문의 if-then vs CLAUDE.md 다형성 원칙

Notion 본문이 보인 코드 :

if (request.osName().hasPostCreationTask()) {
    // ROCKY_LINUX / CENTOS → 환경·패키지 추출
    // UBUNTU → 특정 작업
    // ...
}

그러나 CLAUDE.md "조건 분기문 무분별 확장 절대 지양 (불가침)" 원칙과 충돌. 같은 if-else 가 신규 OS 추가 시마다 자라 silent 누락 / 회귀 진원지가 됨. 3 안 비교 :

A. OSName abstract method

  • OSName.executePostCreationTask(OSImage, Context) abstract 추가
  • 5 상수 각각 자체 구현
  • 장점 : 새 OS 추가 시 컴파일러가 강제 구현 — 누락 불가
  • 단점 : enum 안에 Service 의존 (Context 객체로 우회 가능)

B. OSFamily abstract method

  • OSFamily.executePostCreationTask abstract
  • 3 상수 (RHEL_BASED / DEBIAN_BASED / WINDOWS_BASED) 구현
  • 장점 : 같은 family 의 다중 OSName 공통 처리 자연
  • 단점 : OS 별 미세 차이 (Rocky vs CentOS) 표현 어려움 — 그러나 본 요구에선 RHEL_BASED 묶음 적절

C. PostCreationTaskStrategy 빈 (권장)

  • PostCreationTaskStrategy 인터페이스 — supports(OSFamily) + execute(OSImage, ISO)
  • Spring 빈 등록 — JobLauncher / CompsExtractionLauncher 등 의존 주입 자연
  • 장점 : 본 프로젝트 패턴 (CompsExtractorStrategy 와 일관)
  • 단점 : 빈 추가 — 상대적으로 보일러플레이트 좀 더

plan 작성자 권장 = 안 C. 본 프로젝트의 CompsExtractorStrategy 패턴 (이미 존재) 과 일관. 본 슬라이스 진입 후 자연스러운 확장 (Ubuntu / Windows 의 자동 작업 추가 시) 가능.

D3 자동 추출 실패 시 정책

RHEL_BASED ISO 등록 후 자동 comps 추출이 실패하면 :

A. ISO 등록 자체 rollback DB 트랜잭션과 background Job 결과를 묶어 둘 다 fail-fast. 안전하지만 ISO 업로드 완료 후 추출 실패만으로 자원 등록 자체가 무효화되는 강한 강제력 — 사용자 입장에서 가혹할 수 있음.
B. 등록 성공 + Job 실패 알림 (권장) ISO 등록은 성공 (영속됨). comps 추출 Job 은 실패 상태로 종료 + 알림 패널에 "환경·패키지 추출 실패" 노출. 사용자가 수동 재추출 또는 무시 가능. 본 프로젝트의 기존 background Job 사용성과 일관.
C. silent 무시 + 로그만 알림 없이 서버 로그에만 기록. 비추천 — UX 측 모호함 누적.

D4 비RHEL OS 의 자동 작업 — 본 슬라이스 범위 결정

Notion 본문 : "UBUNTU 였을 경우 특정 작업 등 각 OS 에 맞는 추가 작업 진행". 그러나 :

  • 현재 UbuntuCompsExtractor 가 archive 에 없으며 (CLAUDE.md 명시), 자동화할 의미 있는 작업 정의 불명확
  • Windows 계열은 enum 에만 존재 — UI 활성화 안 됨 (Stage 3 PXE 재도입 시)
A. RHEL_BASED 만 자동화 (권장) 본 슬라이스 = ROCKY_LINUX / CENTOS 의 comps 추출 자동화만. Ubuntu / Windows 는 strategy 인터페이스만 두고 noop 구현. 향후 자동 작업이 정의되면 별 슬라이스.
B. Ubuntu 자동 작업도 같이 정의 plan 단계에서 Ubuntu 의 자동 작업 구체화 → 해당 추출기 신규 구현. 변경 표면 확장.
요청 위 D1~D4 결정 후 plan v2 산출. CP1 승인 시 CP2 진입 (코드 작업 시작). 결정이 v3 / v3-1 등 반복되어도 OK — 본 슬라이스는 코드 변경 0 인 plan 단계.
1. 현재 상태

1.1 현재 ISO 등록 흐름

  1. 사용자 ISO 등록 폼 제출 → OSImageController.addIso(...)
  2. OSImageService.addIso(...) 가 ISO 파일 저장 + 메타데이터 영속화 + 마커 발급

2. 요구사항 (잠정 — D1~D4 확정 후 갱신)

2.1 Notion 명세

ISO 이미지 등록 이후 자연히 하게 되는 작업을 자동으로 하도록 수정. 가령 Rocky Linux, CentOS 의 경우 설치 환경/패키지 그룹 추출 작업을 자동으로 진행하도록 변경. 진입 전 개발자와 구체적 상의 진행 필수.

2.2 잠정 정책 (D1~D4 권장안 기준)

  • 트리거 : addIso() 직후 (D1 권장 A)
  • 분기 방식 : PostCreationTaskStrategy 인터페이스 + Spring 빈 (D2 권장 C)
  • 실패 정책 : ISO 등록 성공 + Job 실패 알림 (D3 권장 B)
  • 범위 : RHEL_BASED 만 자동 (D4 권장 A). Ubuntu / Windows 는 noop strategy

2.3 신규 PostCreationTaskStrategy 인터페이스 (잠정)

public interface PostCreationTaskStrategy {
    boolean supports(OSFamily family);
    /**
     * 비동기 실행 — 메서드 자체는 jobId 등록 + Runner 트리거만 하고 즉시 반환.
     * Job 실패 시 알림 패널에 노출됨.
     */
    String trigger(OSImage osImage, ISO iso);
}

@Component
public class RhelPostCreationTaskStrategy implements PostCreationTaskStrategy {
    private final CompsExtractionLauncher compsExtractionLauncher;

    public boolean supports(OSFamily family) {
        return family == OSFamily.RHEL_BASED;
    }
    public String trigger(OSImage osImage, ISO iso) {
        return compsExtractionLauncher.startExtraction(osImage, iso);
    }
}

@Component
public class NoopPostCreationTaskStrategy implements PostCreationTaskStrategy {
    public boolean supports(OSFamily family) {
        return family == OSFamily.DEBIAN_BASED || family == OSFamily.WINDOWS_BASED;
    }
    public String trigger(OSImage osImage, ISO iso) {
        return null;  // 자동 작업 없음
    }
}

2.4 OSImageService.addIso 변경 (잠정)

@Transactional
public Long addIso(Long osImageId, ISOCreateRequest request, ...) {
    // ... 기존 ISO 영속화 + 마커 발급 + IsoRegistrationLauncher 호출 ...
    Long isoId = persistedIso.getId();

    // S5-7 — 자동 작업 트리거 (PostCreationTaskStrategy 패턴, OS Family 기준)
    postCreationTaskStrategies.stream()
        .filter(s -> s.supports(osImage.getOsName().getFamily()))
        .findFirst()
        .ifPresent(s -> s.trigger(osImage, persistedIso));

    return isoId;
}

여기서 postCreationTaskStrategies 는 List<PostCreationTaskStrategy> 로 Spring 이 자동 주입.

3. URL / 데이터 흐름 (잠정)

3.1 URL 변경

없음. 기존 POST /management/os/{osId}/iso 의 응답에 변화 없음. 자동 트리거된 Job 은 알림 패널에서 확인.

3.2 데이터 흐름 (잠정)

사용자 → POST /management/os/{osId}/iso (ISO 등록)
         ↓
       OSImageService.addIso()
         · 메타데이터 영속
         · 마커 발급
         · IsoRegistrationLauncher.start() → BackgroundJob ISO_REGISTRATION 등록
         · S5-7 신규 : PostCreationTaskStrategy.trigger() → CompsExtractionLauncher.start()
                                                          → BackgroundJob COMPS_EXTRACTION 등록
         ↓
       응답 (jobId of ISO_REGISTRATION)
         ↓
       FE 알림 패널에 두 Job 모두 노출

3.3 두 Job 의 직렬화

ISO_REGISTRATION 이 끝나기 전에 COMPS_EXTRACTION 이 시작되면 경합 발생 가능 — ISO 파일이 아직 hash 계산 / 중복 검사 중일 수 있음. 결정 사항 :

  • A. 두 Job 병렬 (현재 잠정안) : COMPS_EXTRACTION 의 PREPARE_ISO 단계가 파일 존재 + 읽기 가능성 확인하므로 ISO_REGISTRATION 의 hash 계산과 병렬 가능
  • B. ISO_REGISTRATION 완료 후 COMPS_EXTRACTION 트리거 : bgjob:completed 이벤트 또는 launcher 안에서 동기 대기. 더 안전하지만 추가 인프라 필요
D5 추가 결정 사항 두 Job 직렬화 정책. CP1 사용자 확인 추가 항목.

7.1 신규 예외

7.2 strategy 실패 격리

OSImageService.addIso 내부에서 strategy.trigger() 호출이 RuntimeException 던질 경우 — 트랜잭션 rollback 막기 위해 try-catch + 로그 + 알림 토스트. ISO 등록 자체는 영속됨.

try {
    postCreationTaskStrategies.stream()
        .filter(s -> s.supports(osImage.getOsName().getFamily()))
        .findFirst()
        .ifPresent(s -> s.trigger(osImage, persistedIso));
} catch (RuntimeException e) {
    log.error("[S5-7] 자동 작업 트리거 실패 (ISO 등록은 영속됨). osImage={}, iso={}",
              osImage.getId(), persistedIso.getId(), e);
    // 별도 BackgroundJob 도 등록 가능 — "자동 작업 트리거 실패" 알림
}
