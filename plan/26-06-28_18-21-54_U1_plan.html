§2. 요구사항 + 비목표

확정 요구사항

  1. 데이터 모델 통합·정합화 — 3중 1:1(guest_server/detail/custom)을 2테이블로 통합, vendor 파생화, setup_step 1:N, provisioning_progress 감사/락 정비, 파생·고아 필드 제거, 전 본체 감사필드 표준화.
  2. 운영 상태 도출 — 서버의 운영 상태(등록됨/프로비저닝중/완료/실패/회수됨)를 저장 컬럼이 아니라 (progress + 회수 마커)에서 도출하는 뷰모델 enum 으로 노출. 목록/상세가 한눈에 상태를 보여준다.
  3. 등록 진입점 마감 — iPXE 부팅 등록이 guest_server + detail + host_nic_binding(primary) + provisioning_progress(seed) 를 한 트랜잭션에 적재(빈 custom 생성 제거). 입력 형식 오류는 advice 단일 진입점에서 400.
  4. 조회 + 인라인 수정 CRUD — 목록/상세 페이지, 상세에서 운영자 식별 4필드(name/model_name/serial_number/memo) 인라인 수정 + 유니크(name/serial) 충돌 BindingResult 인라인 표시. 서버 회수(decommission) 액션.
  5. 척추 예약 — 서버 ↔ 세팅 정의서 연결의 자리와 카디널리티를 설계로 확정하고 후속 슬라이스로 명시 위임(dead 컬럼은 추가 안 함, §D5).
비 목표 (out of scope → 어느 슬라이스로 가는가)
  • 세팅 정의서 도메인 + 서버↔정의서 할당 로직 → 후속 setting 슬라이스. U1 은 연결 자리만 설계 확정(D5).
  • 프로비저닝 엔진(단계 전이, setup_step 행 적재, phase_meta 기록) → Stage 4. U1 은 setup_step 의 모양만 바로잡고 적재는 안 한다.
  • BMC 접속(bmc_binding/credential_ref) · hardware/software_spec 적재(컬럼은 U1 에 추가하되 수집·적재는 이연 — §D8 수정) · location_assignment · transition_log → 각 후속 슬라이스. phase enum 에 FIRMWARE/BMC 가 있으나 수행 데이터는 그 슬라이스에서.
  • 명세 충실도 / PXE 계약 재검토 — 본 슬라이스 검토 범위 밖(사용자 지시). 진입점은 데이터 적재 측면만 정비.

§4. 도메인 모델 (재설계 + 결정 D1~D11)

대안을 복수 비교하고 채택안 + 비채택 사유를 함께 적는다(CP1 불가침). 필드 길이 등 상세는 코드가 SSOT.

D1 · 3중 1:1 통합 채택 B: custom→guest_server 흡수, detail 유지
✗ A 3테이블 유지 + custom/detail lazy 생성 — 2칸짜리 테이블을 위해 lazy 분기 추가. 복잡도만 늘고 이득 없음.
✓ B custom 을 guest_server 로 흡수(model_name·serial_number 컬럼화), detail 은 분리 유지.
✗ C detail 까지 전부 guest_server 로 통합(1테이블) — detail 의 @Version 격리(운영자 인라인 수정 vs 진단 다회 갱신의 false-conflict 회피)와 hw/sw JSON 성장 여지를 잃음.

근거 — custom 은 운영자 부여 식별자라 name/memo 와 같은 인라인 폼에서 함께 수정된다(같은 write 시점·주체). version·JSON 도 없다 → 분리 정당성 0, 흡수가 맞다. 반면 detail 은 하드웨어가 보고한 사실(운영자 부여와 개념적으로 다름) + 다회 갱신 주체(진단/검증) → @Version + 향후 JSON 으로 분리 정당. "쪼개 놓고 한꺼번에 생성" 모순은 custom 에서만 발생했으므로 custom 흡수로 해소.

D2 · vendor 채택: 컬럼 제거 → boardModel 에서 도출
✓ detail.vendor 삭제, detail.boardModel.getVendor() 로 도출(Response 에 적재).
✗ 스냅샷 보존용 유지 — 보드 교체 시 boardModel FK 자체가 바뀌므로 스냅샷 가치 낮고 드리프트(detail.vendor≠boardModel.vendor) 위험만 남음.

근거 — vendor 는 boardModel 의 속성이라 항상 파생 가능. 목록 N+1 은 이미 boardModel fetch-join 으로 해결돼 있어 도출 비용 0.

D3 · model_name 고아/중복 채택: 단일화 (custom 쪽 의미 채택)

D1 의 흡수로 자연 해소. guest_server.model_name(고아, 항상 null)을 폐기하고, custom 의 "운영자 부여 + ipmitool 각인 모델명" 의미를 guest_server.model_name 단일 컬럼이 갖는다. serial_number 도 동일하게 guest_server 로. → 한 화면 한 필드, 중복 0.

D4 · 운영 상태 채택 C: 도출 + 회수 마커만 저장
✗ A GuestServerStatus 컬럼 저장 — progress 와 중복되는 4번째 상태축. 동기화 드리프트(내가 직전 검토에서 비판한 바로 그 안티패턴).
✓ C 도출형 — 상태는 derive(progress, decommissionedAt) 로 계산. 도출 불가한 사실(운영자 회수)만 decommissioned_at(nullable) 1필드 추가.
✗ B 전부 도출(회수도 progress 로) — 회수는 progress 와 무관한 운영자 결정이라 progress 로 표현 부자연.

도출 규칙 : decommissionedAt≠null → 회수됨 · progress 실패 → 실패 · progress 완료(TESTING✓) → 완료 · progress 가 BOOTSTRAPPING 초과 진행 → 프로비저닝중(phase) · 그 외 → 등록됨. enum GuestServerStatus 는 뷰모델 전용(미영속) — Primitive Obsession 금지 준수.

D5 · 서버 ↔ 세팅 정의서 (자동화의 척추) 채택 D: 설계 예약, 컬럼 미추가
✗ A 아무 것도 안 함 — 척추를 침묵으로 누락(직전 검토가 지적한 위험).
✗ B dangling UUID assigned_setting_definition_id 컬럼 추가 — 대상 테이블 부재로 FK 미강제 + Primitive Obsession + "갈라지기 전 분리" 금지 위반.
✓ D 자리·카디널리티를 설계로 확정하고 컬럼/FK 는 SettingDefinition 슬라이스에서 실제 추가.
예약 명세(확정) — 연결은 guest_server 에 @ManyToOne(nullable) settingDefinition 로 둔다(서버는 1개의 정의서를 할당받고, 한 정의서는 여러 서버에 재사용 → N:1). 미할당=null. 이 FK 와 enforcement 는 setting 슬라이스에서 SettingDefinition 엔티티와 함께 도입한다. U1 은 dead 컬럼을 만들지 않되 이 결정을 못박아 "비워둔 채 잊는 것"을 방지한다.

근거 — CLAUDE.md "미래에 갈라질 수 있으니 미리 분리 금지 — 갈라지는 시점에 분리". 대상 엔티티가 없는 지금 컬럼만 박으면 unenforced + primitive. 그러나 척추인 만큼 설계는 지금 확정해 후속 슬라이스의 전제로 박는다.

D6 · provisioning_progress 정비 + seed 채택: BaseTimeEntity + @Version + 등록 시 seed
  • 감사 정비(F2) — BaseTimeEntity 상속(created/updated 확보 + 리스너 부착). last_transition_at 은 @LastModifiedDate 의존을 버리고 전이 시 명시 set 하는 도메인 필드로(엔진이 채움). 등록 seed 시 now.
  • 락(F4) — @Version 추가. 비동기 엔진의 다회 전이 lost-update 방지.
  • seed — 등록 트랜잭션이 current_phase=BOOTSTRAPPING 1행 생성 → 1:1 불변 유지, 상세 UI 가 phase 를 표시.
✗ 대안 seed 안 하고 엔진이 최초 생성 — 상세가 null progress 가 되고 1:0..1 의 lazy/eager 비일관 재발. seed 가 깔끔.
D7 · setup_step 1:N + 커서 SSOT 채택: @ManyToOne, phase_code 도출, progress=커서
  • 1:N(F1) — @OneToOne→@ManyToOne(optional=false). 서버당 다수 단계이력 저장 가능(엔진이 적재; U1 은 모양만).
  • phase_code 도출(Q2) — step_code(ProvisioningPhaseStep).getPhaseType() 로 항상 파생되므로 컬럼 제거, Response 에서 도출.
  • 커서 SSOT(Q3) — "현재 단계"의 권위는 provisioning_progress.current_phase. setup_step 은 append-only 실행 이력/감사. 상세 UI 는 progress 를 헤드라인, step 리스트를 이력으로.
D8 · hardware_spec/software_spec(JSON) 채택(수정): U1 에서 컬럼 추가, 적재는 진단 슬라이스
CP3 중 사용자 지시로 결정 번복 — 최초 채택안은 "설계 예약 · U1 미추가"(아래 ✗)였으나, 사용자가 지금 컬럼을 생성하도록 지시. 엔티티를 어차피 재형하는 시점에 nullable JSON 2 컬럼을 함께 두어 후속 ALTER 한 번을 절약한다.
✓ 채택 guest_server_detail 에 hardwareSpec / softwareSpec 를 String + columnDefinition="json"(nullable)로 추가. 저장 형식은 기존 phaseMeta/statusMeta 관례와 동일. 적재·수집·앱측 sealed record 타입 매핑·DTO/화면 노출은 진단 리눅스 수집 슬라이스가 담당(등록 시 항상 null). JSON_VALID CHECK 도 그 슬라이스에서.
✗ 최초안 설계 예약 + U1 미추가(dead 컬럼 회피, "갈라지는 시점에 분리") — 사용자 지시로 폐기.
D9 · 감사필드 표준화 채택: 전 본체 BaseTimeEntity

HostNicBinding·ProvisioningProgress·SetupStep 도 BaseTimeEntity 상속. 도메인 시각(boundedAt/startedAt/finishedAt/lastTransitionAt)은 created/updated 와 의미가 달라 공존(주석으로 구분). GuestServerCustom 은 폐지(D1).

D10 · 등록 진입점 정비 채택: 검증 + advice, custom 생성 제거, progress seed

입력 형식 검증(UUID/MAC/IP)을 두고, 형식 오류는 컨트롤러 try/catch 가 아니라 @ControllerAdvice/@RestControllerAdvice 단일 진입점에서 400 으로(분기 증식 금지 원칙). 등록 write-set 을 D1/D6 에 맞춰 조정(빈 custom 제거, progress seed 추가). VO 가드(MacAddressVO/IpAddressVO) 예외도 advice 로 일관 매핑.

D11 · 패키지 소유 경계 채택: execution=애그리거트 소유, provisioning=UI
✓ execution 이 게스트 서버 애그리거트(엔티티·리포지토리·등록·조회/수정 application service)를 소유. provisioning 의 GuestServerController+뷰 DTO 는 execution 의 application service 만 호출(리포지토리 직접 주입 제거).
✗ A 현행 유지(provisioning 서비스가 execution repo 직접 주입) — feature 경계 침범 지속.
✗ C 전부 provisioning 으로 이동 — execution 의 존재 의의(진입점/실행 영역) 약화 + Stage 4 엔진과의 응집 깨짐.

실행 — GuestServerQueryService/GuestServerCommandService 를 execution 으로 이동(자기 repo 사용). provisioning 은 컨트롤러+뷰 Response 만 보유하고 execution 서비스를 호출. 레이어 경계(Controller↔Service=Request/Response, 뷰에 엔티티 직접 노출 금지) 준수.

D12 · iPXE 보고 모델명 ↔ 카탈로그 매칭 채택: 제조사별 정규화 → exact (CP4 중 추가)
문제 — iPXE ${product} 는 Gigabyte 의 경우 카탈로그 모델명 뒤에 -000 접미사를 붙여 보고한다(ex. 카탈로그 MS03-CE0 ↔ 보고 MS03-CE0-000). 운영자는 접미사를 모르고 입력하지 않으며, contains/prefix 로 풀면 유사·파생 모델이 다중 매칭돼 자동화가 붕괴한다.
✓ A 제조사별 정규화(canonicalization) → exact 매칭. 이건 "퍼지 검색"이 아니라 "정규화" 문제 — 정규화 후 exact 면 결과는 항상 0/1건이라 모호성이 원천 제거된다. Vendor.canonicalizeReportedModel(method-per-constant: GIGABYTE 만 -000 제거, 그 외 항등). 등록은 ① 원본 exact → ② miss 면 정규화 exact → ③ 둘 다 miss 면 404(보고 원본 동봉).
✗ contains/prefix 다중 매칭 → 자동화 불가(사용자 지적). 퍼지로 불확실성을 추측해 메우는 방식.
✗ 운영자 입력 접미사를 모르므로 입력 불가.
↪ 후속(필요 시) 미관측 제조사 quirk 가 실제 등장하면 — 트리거는 "보고 원본 문자열을 담은 404 로그". 그때 해당 Vendor 규칙 1줄 추가하거나 학습형 별칭 매핑(미인식 보고를 운영자가 카탈로그에 1회 매핑)으로 승격. 지금은 알려진 quirk(Gigabyte -000)만 있어 A 로 충분("갈라지는 시점에 분리").

안전장치 — ① 원본 exact 우선이라 실제로 -000 으로 끝나는 모델이 있어도 과잉 제거 없음. ② 정규화는 보수적(끝 접미사만, 임의 절단·중간 매칭 금지). ③ 비 Gigabyte 는 항등(미관측 규약 추측 안 함). 사용자 확인 : 현재 사용 예정 보드의 Gigabyte 규약은 -000 고정.

§8. 부산물 / 주의

D-1. 스키마 마이그레이션 (ddl-auto=update 한계)

ddl-auto=update 는 컬럼/테이블을 자동 삭제하지 않는다. 따라서 다음은 수동 DDL + 선행 chore(schema) 커밋 필요: DROP TABLE guest_server_custom(데이터 이행 후), guest_server_detail.vendor·setup_step.phase_code 컬럼 제거, guest_server.model_name 의미 재정의(고아→운영자 모델명), setup_step FK 의 unique 제약 제거(1:1→1:N). 신규: decommissioned_at, progress version·created/updated, nic/step created/updated. 현 단계는 운영 데이터가 거의 없으니(엔진 미가동) 이행 비용 낮음 — custom 의 model/serial 만 guest_server 로 복사.

D-2. 데이터 이행

기존 guest_server_custom 행의 model_name/serial_number → guest_server 동명 컬럼으로 복사 후 테이블 drop. 운영 데이터 적으면 1회 UPDATE…JOIN.

D-3. 패키지 이동(D11)

GuestServerQueryService/CommandService 를 provisioning.service → execution.service 로 이동. provisioning 은 컨트롤러+뷰 Response 만. import 경로 정리. (R 캠페인의 Controller/Service 분리 관례와 정합.)

D-4. CLAUDE.md / Notion

패키지 구조 절의 "execution=예약" 을 "execution=게스트 서버 애그리거트 소유(진입점/실행)"으로 갱신 제안(D11). Notion U1 페이지 본문 4항목(scope/비목표/잔존책임/후속) 갱신은 CP1 직후 background agent 위임(불가침 동기화 규약).

D-5. 도출 vs 중복 — 자기 점검

  • 도출 채택(vendor·운영상태·phase_code) — 같은 사실을 두 곳에 저장하지 않음. SSOT 원칙 정합.
  • 분리 유지 정당(detail) — @Version 격리 + 하드웨어/운영자 개념 분리 + JSON 성장. 도메인 가로지르는 과추상 아님.
  • 예약(미추가)(settingDefinition FK) — "갈라지는 시점에 분리" 준수. (hw/sw JSON 컬럼은 D8 최초 예약 대상이었으나 사용자 지시로 U1 에 추가 — 적재만 진단 슬라이스로 이연)
