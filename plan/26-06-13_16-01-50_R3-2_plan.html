§0. 동기 — fat 단일 컨트롤러의 책임 혼재 정리

현재 BoardModelController(211 줄 / 엔드포인트 14 개)는 CRUD(메타) · lifecycle(toggle/delete/restore/purge/deprecate) · nudge confirm 세 책임군이 한 파일에 뒤섞여 있다. R1(OS) 도메인이 이미 fat 컨트롤러를 8 컨트롤러(Upload/Integrity/Browse 등)로 분할한 선례를 따라, Board 도 책임군 경계로 쪼갠다.

  • 책임 혼재 — 한 컨트롤러가 메타 CRUD · 상태 전이 · nudge 까지 떠안아, 신규 lifecycle 동작이나 nudge 변형이 추가될 때마다 같은 파일이 비대해진다.
  • 헬퍼 복붙 위험 — redirectToListWithSelect / nullToEmpty / BindingResult→ApiErrorResponse 변환은 분할 후 3~4 컨트롤러가 동시에 필요로 한다. 그대로 분할하면 같은 헬퍼를 복붙하게 되어 CLAUDE.md §중복 금지(불가침) 위반.
CLAUDE.md §조건 분기 / try-catch 복붙 회귀 경계 — "신규 컨트롤러 추가 시 같은 catch 블록을 또 복붙하는 회귀(S3.3→S3.4 사고)" 가 본 프로젝트에서 반복됐다. 본 슬라이스는 분할과 동시에 공통 헬퍼를 BoardControllerSupport(final + private ctor 정적 유틸)로 추출해 복붙 진원지를 사전 차단한다.
refactor 불변식 — URL · HTTP status · 응답 바디 · 리다이렉트 타깃 · 뷰 모두 100% 동일. 순수하게 controller 클래스 경계만 재배치한다. 행동 변화 0.

④ BoardControllerSupport — 공통 헬퍼 (복붙 제거)

public final class BoardControllerSupport {
    private BoardControllerSupport() {}   // 인스턴스화 금지

    public static String redirectToListWithSelect(Long selectId) {
        return "redirect:/management/board?selectId=" + selectId;
    }
    public static String nullToEmpty(String value) {
        return value == null ? "" : value;
    }
    // create() 인라인 BindingResult → ApiErrorResponse 변환 추출
    public static ApiErrorResponse toValidationError(BindingResult br) {
        var fields = br.getFieldErrors().stream()
            .map(fe -> new ApiErrorResponse.FieldError(
                fe.getField(),
                fe.getDefaultMessage() != null ? fe.getDefaultMessage() : "유효하지 않은 값"))
            .toList();
        return ApiErrorResponse.ofValidation(
            "입력 값이 유효하지 않습니다 (" + fields.size() + "개 필드).", fields);
    }
}
SSOT — redirect 문자열 / null 정규화 / 검증 오류 변환의 단일 소스가 BoardControllerSupport 로 모인다. 3~4 컨트롤러가 공유 → 향후 변경 시 1곳만 수정.
도메인 특이사항 (3 컨트롤러로 축약) — Board 는 메타 자원이라 OS 의 *UploadController / *IntegrityController / *BrowseController 에 대응하는 컨트롤러가 없다. Upload/Integrity/Browse 책임 자체가 도메인에 부재.
