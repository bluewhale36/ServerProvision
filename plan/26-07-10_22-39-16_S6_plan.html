§2. 설계 결정 D1~D10 (대안 비교 · 탈락 사유)

D1. 단계 분해 — S6-2 를 sub-slice 3개로, 후보 A 는 S6-3 유지 채택

S6-1(정리) → S6-2-1(해결 디스패치 다형화 기반) → S6-2-2(SOFTDEL 계열 탐지+해결) → S6-2-3(TRASH 계열 탐지+해결) → R9-6(후보 C) → S6-3(후보 A, 하위 분해는 S6-3 plan 에서). 결정 문서가 허용한 "계열별 최상위 분리(S6-2=SOFTDEL 총 / S6-3=TRASH 총 / S6-4=A)"를 검토했으나 비채택 — 탈락 사유: ① 두 계열이 공유하는 디스패치 다형화 기반(S6-2-1)의 귀속이 애매해진다 ② 실측 결과 탐지 자체가 entity 별 Files.exists 검사로 가벼워 "방대" 조건에 해당하지 않는다 ③ "후보 A = S6-3" 라는 결정 문서의 기본 지정을 유지하는 쪽이 이력 추적이 단순하다.

D2. 탐지 설계 — findTrashed() 전수 분류 패스 + D20 지점 분류 전환, 신규 walk 0 채택

기존 (5.5) GHOST 패스를 soft-deleted 전수 분류 패스로 확장·흡수한다: scanner 별 findTrashed()(6 도메인 기구현)로 deleted entity 를 얻고, entity 의 trashedPath(T)·resourcePath(O) 두 경로에 대한 Files.exists 비트로 §4 매트릭스를 판정한다. 메타 자원 2종(OS_IMAGE/BOARD_MODEL)은 패스 진입 전 supportedType().isMetadata() 로 명시 제외(비판 검증 CRITICAL — getResourcePath()=null 이라 NPE/GHOST 오탐, TrashController:158-166 가드 선례 재사용). TRASH_MARKER_STALE 은 trash 내 자원 옆 마커를 markerService.resolveMarkerFile(T, layout) 존재 검사로 직접 확인. SOFTDEL_ESCAPE_TO_OTHER 만 기존 디스크 마커 맵을 쓴다 — D20 제외 지점(:423)의 침묵 continue 를 분류로 교체하며, 이를 위해 인벤토리 단계에서 Map<MarkerKey, Markable> deletedByKey 를 함께 구성해 발견 마커의 원 DB.path 대조에 쓴다(현행 Set<Long> 만으로는 TO_ORIGINAL/TO_OTHER 구분 불가).
비채택: ⓐ trash root 를 walk 범위에 추가 — 마커 파일 기준 발견은 우회 경로이고, entity 가 이미 정확한 T 경로를 알고 있어 직접 검사가 정확·저렴. walk 제외(:534)는 그대로 유지. ⓑ MarkableInventory 에 trashed 경로 노출 메서드 신설 — TrashController:158 의 instanceof LifecycleEntity 패턴 매칭 선례가 이미 있고, 6 도메인 entity 전부 LifecycleEntity 상속이라 SPI 오염 없이 동일 접근 가능(비상속 구현이 등장하면 그때 승격 — "갈라지는 시점에 분리").

D3. 해결 디스패치 다형화 — kind 별 DriftResolution 전략 bean (S6-2-1) 채택

apply() 의 GHOST if-else(:674-680)와 스캔 자동 적용 if-else(:475-481)를 전략으로 치환한다: DriftResolution 인터페이스(supportedKind() + resolve(Drift, MarkableScanner)) + kind 별 구현 bean, Map<DriftKind, DriftResolution> 주입(Spring 이 List→Map 조립). 신규 kind = 새 bean 클래스 1개 — 서비스 본문 분기 증가 0(OCP). 미지원 kind 는 map miss → DriftAutoApplyNotAllowedException.notApplicable 로 거절하되, 이 널가드는 forced 우회 블록 밖(디스패치 지점)에 무조건 배치한다 — forced 경로는 mode 가드만 우회하는 것이지 "해결 로직이 존재하지 않는 kind" 까지 통과시키는 게 아니다(persistAndForcedApply 가 kind 제약 없는 public 이라는 비판 검증 MAJOR 반영). S6-2-1 은 기존 2종(PATH_DRIFT→applyDriftedPath / GHOST_DB_ROW→applyGhostClear)의 동작 등가 이전만 수행(신규 kind 없음) — 회귀 면을 기반과 신규로 분리.
비채택: ⓐ DriftKind enum method-per-constant — 해결 로직은 scanner/trashService/markerService 의존이 필요한 행위인데 enum 은 DI 불가, global/marker 의 enum 이 maintenance 도메인 로직을 소유하는 방향 역전도 발생. ⓑ DriftKind 에 resolutionType 순수 데이터 필드 + 서비스 switch — 분기가 잔존하나 exhaustive switch 로 누락은 컴파일 차단되는 차선. bean 6개(각 15줄 내외)가 과하지 않아 ⓐⓑ 대비 전략 bean 이 CLAUDE.md "Spring primitive 로 framework 가 분기를 떠맡게" 에 정합.

D4. 해결 방식 3단 — autoApplicable(boolean) → DriftResolutionMode { NONE, MANUAL, AUTO } 채택

"버튼 노출"과 "스캔 중 무인 자동 적용 자격"을 한 boolean 이 겸하던 것을 단일 enum 필드로 교체: NONE=시스템 해결 없음(안내만) / MANUAL=사용자 확인 후 시스템이 해결(버튼 O, 무인 자동 X) / AUTO=MANUAL + 스캔 중 무인 적용 자격. 파생 접근자 isAutoApplicable()(=AUTO)·isManuallyResolvable()(=NONE 아님)를 두어 서버 가드·템플릿의 기존 호출면 diff 를 최소화한다. 신규 배정: SOFTDEL_ESCAPE_TO_OTHER·TRASH_LOST = MANUAL(MK3 원정의 "액션 미제공"에서 상향 — 결정 문서의 "시스템이 해결" 목표 반영, 단 무인 자동은 이중 의도 명제 존중으로 불허). 배지 색도 3단으로 — 현행 autoApplicable ? orange : gray 2색이면 MANUAL 종류가 "회색 배지 + 활성 버튼"이라는 시각 모순이 생긴다(비판 검증 MINOR). 구체 클래스는 S6-2-1 slice plan 에서 DESIGN.md 토큰 내로 확정.
비채택: boolean 2개(manuallyResolvable+autoApplicable) — manual=false·auto=true 불법 조합이 표현 가능(Primitive Obsession 금지 위반). 3단 enum 은 불법 상태를 타입으로 차단.

D5. 스캔 자동 적용 설정 통합 — reconciliation.auto-apply.kinds CSV 단일 키 채택

auto-apply-path-drift/auto-apply-ghost-row 개별 boolean 키를 Set<DriftKind> CSV 한 키로 통합(default 빈 = 현행 default 두 키 모두 false 와 동일 동작). 스캔 자동 적용 조건 = kind.mode==AUTO && kinds.contains(kind) && isAutoApplyEnabled() — AUTO kind 가 늘어도 설정 구조 불변. 기존 두 키는 제거(breaking) — §8 명기.
비채택: kind 별 boolean 키 유지·증식 — AUTO 4종이면 키 4개+분기 4개, legacy 확장 금지 위반.
마스터 스위치 의미·명칭 정비(비판 검증 MAJOR 반영): 마스터의 실의미는 "시스템이 FS/DB 를 건드리는 해결 전면 차단"(MANUAL 버튼 포함 — R9-2 D4 의 차단 범위 유지)인데, MANUAL 개념이 생기는 순간 reconciliation.auto-apply 라는 이름과 "자동 적용이 꺼져 있어요" 문구가 사용자 혼란(R9 본 목적)을 재생산한다. mode 3단을 신설하는 지금이 개명의 유일한 자연 시점 — reconciliation.resolution-enabled(default true)로 개명하고 globalOff() 문구·버튼 tooltip 을 "시스템 해결 옵션이 꺼져 있어요" 계열로 함께 갱신한다. SSOT 메서드(isAutoApplyEnabled()→isResolutionEnabled())와 뷰모델 플래그도 동반 개명. 설정 breaking 은 kinds 통합과 같은 슬라이스(S6-2-1)에 묶어 1회로 끝낸다.
비채택: ⓐ 마스터가 AUTO(무인)만 지배하고 MANUAL 버튼은 항상 활성 — "점검 전용(읽기만) 모드"를 원하는 운영자의 전면 차단 수단이 사라지고, R9-2 가 확정한 차단 범위의 회귀. ⓑ 이름 유지 + 문구만 손질 — 설정 키와 화면 문구가 서로 다른 어휘를 쓰는 어긋남이 남는다.

D6. SOFTDEL_ESCAPE_TO_ORIGINAL 해결 = scanner.restoreFromTrash(id) 재사용, AUTO 유지 채택

정합 방향은 MK3 원정의대로 FS 가 진실, DB 가 따라감(=복원): 운영자가 파일을 원위치로 되돌린 행위 자체가 복원 의도의 표현이라 의도가 단일화된다. 해결 경로는 기존 MarkableTrashOperator.restoreFromTrash 를 그대로 호출 — healOrThrowBeforeRestore 의 (원O·trashX) self-heal 분기와 trashed_path=null 단순 복원 분기(TrashLifecycleService:127-142)가 두 발생 경로를 모두 이미 소화한다. 신규 FS 로직 0.
비채택: 재격리 방향(FS 를 DB 에 맞춰 도로 trash 로) — 운영자의 외부 행위를 시스템이 되돌리는 의도 전쟁이 되고, 되돌려도 운영자가 다시 꺼내는 luta 만 반복된다.

D7. SOFTDEL_ESCAPE_TO_OTHER 해결 = 재격리(MANUAL) 채택

발견 위치의 자원(+동반 마커)을 trash 로 회수하고 markTrashed 갱신. TO_ORIGINAL 과 정합 방향이 반대인 근거: 원위치 복귀는 복원 의도로 읽을 수 있으나, 임의 위치 출현은 백업 복사·실수 등 사고 가능성이 커 DB(deleted=운영자 최종 의사표시)를 진실로 본다. 사용자 확인(MANUAL) 후 시스템이 실행 — "위치·의도를 직접 확인하세요"로 끝나던 것에서 실행 가능한 해결로 상향. FS 이동은 TrashService 재사용, entity 갱신 경로(도메인 service 경유 여부)는 S6-2-2 plan 에서 확정.
S6-2-2 plan 필수 결정 2건(비판 검증 MAJOR 반영): ① 재격리는 FS mv + DB 갱신의 신규 이중쓰기이므로 HF-1 과 동일한 same-transaction 역보상(mv 후 어느 단계든 실패 시 원위치 복귀 — softDeleteToTrash:94-99/compensateRestore 선례)을 필수 제약으로 못박는다. ② 기존 trashed_path 가 아직 유효(trash 파일 존재)한 상태에서 TO_OTHER 를 재격리하면 markTrashed 덮어쓰기로 구 trash 파일이 참조를 잃고 영구 고아가 된다(trash 는 walk 제외라 재발견 불가) — 구 파일 동반 정리 또는 재격리 거절 중 택일을 결정 항목으로 명시.
비채택: ⓐ AUTO 승격 — lifecycle+path 이중 의도 명제(MK3) 위반. ⓑ 발견 위치를 새 정위치로 수용(restore+path 갱신) — 삭제 의사표시를 뒤집는 방향이 기본이 될 수 없음. 필요 시 운영자는 복원 후 이동이라는 기존 경로가 있다.

D8. TRASH_LOST 해결 = row 정리(MANUAL) + PurgeLog 기록 채택

trash 파일이 외부에서 사라진 dead row 를 사용자 확인 후 hard-delete — GHOST_DB_ROW 와 종착은 같으나 파괴적(이력 종료) 이고 외부 개입 정황이라 무인 자동은 불허(MANUAL). 정리 시 PurgeLog(S5-2-4 단일 감사 로그, PurgeOrigin 신규 값) 기록을 남겨 "언제 무엇이 왜 사라졌나"를 추적 가능하게 한다 — 기록 상세(origin 값·details 필드)는 S6-2-3 plan 에서 확정.
비채택: AUTO — 복구 불능 확정을 시스템이 무인 단행하는 것은 "안전한 정리"(GHOST — 애초에 기록만 남은 row)와 달리 외부 원인 규명 기회를 소거.

D9. quick scan 본체 존재 검사 추가 (4a 보강, S6-1 포함) 채택

4a 에서 마커가 있어도 본체(resourcePath)가 없으면 quick scan 에서 즉시 MISSING 으로 보고(현재는 deep 주기까지 최대 24h 침묵). 비용 = active 자원당 Files.exists 1회. 이 보강이 있어야 "파일명 단독 변경" 시나리오가 deep 대기 없이 드러나 D10(RENAMED 정리)의 커버리지 논거가 완성된다.
비채택: 같은 디렉토리에서 이름만 바뀐 파일을 해시로 탐색해 자동 매칭(RESOURCE_RENAMED 본구현) — deep 해시 재계산 없이는 매칭 근거가 없고(12GB ISO 해시 수 분), 마커·파일 동시 리네임은 이미 PATH_DRIFT(:381 키 매칭이 파일명 무관)로 잡힌다. 남는 케이스는 MISSING 안내로 충분.

D10. S6-1 정리 — RESOURCE_RENAMED · RESOURCE_RENAMED_ORPHAN 상수 삭제 정리

v2 실측대로 두 상수의 실질 커버리지가 기존 kind 에 흡수돼 있어(동시 리네임→PATH_DRIFT, 그 외→MISSING) 삭제한다. 탐지 로직이 한 번도 생성한 적 없는 값이라 영속 데이터 이력 충돌 0(안전). MISSING.recommendedAction 에 리네임 복원 안내 한 줄 보강. 파급: DriftKindTest 의 AUTO 집합 단언, PathReconciliationService:677 주석, DriftKind javadoc 재편(MK3 신규 섹션 정리 — "미구현" 딱지가 S6-2 로 해소되는 것까지 반영은 각 sub-slice 에서).

🎬 soft-deleted 분류 시뮬레이터 — FS/DB 상태를 눌러 바꾸면 서버 판정·해결이 재현된다

soft-delete 된 자원 1개(OS_ISO #42)의 DB 2비트 + FS 4비트를 직접 토글 → §4 매트릭스의 서버 분류가 즉시 갱신된다. [해결 실행]을 누르면 해당 resolution 이 상태를 실제로 전이시킨다(복원/재격리/row 정리/마커 정리). MANUAL kind 는 확인 modal, 전역 OFF 시 disabled+tooltip 까지 재현.

전역 자동 적용(reconciliation.auto-apply) 무인 자동 적용 kinds(auto-apply.kinds)

검증 포인트: ① trashed_path 有 + trash 파일 소실 + 원위치 출현 = "삭제 자원 복귀"(AUTO) — 해결하면 row 가 활성으로 복원되고 drift 소멸 ② 아무 흔적 없음 = "휴지통 자원 소실"(MANUAL) — 해결 = row 정리 + PurgeLog ③ trash 자원 옆 마커 = "잔여 마커 정리"(AUTO) — 원위치 점유 여부와 무관하게 독립 평가(둘 다 켜면 drift 2건 동시 보고) ④ active 트리 다른 위치 마커 = "위치 이탈"(MANUAL, 재격리) — 원위치 파일과 동시 성립 시 모호 상태로 detail 병기·restore 게이트 최종 판정 ⑤ 원위치·trash 양쪽 존재(점유, 마커 신호 없음)는 의도적으로 drift 아님 — 동일 경로 재사용(새 자원 업로드)이 정상 워크플로이기 때문(§8). 메타 자원 2종은 이 시뮬레이터 자체의 대상이 아니다(D2 제외).

§7. 예외 계층
  • 신규 예외 클래스 0 (예정) — 해결 실패는 기존 계층 재사용: 재격리 mv 실패 = TrashMoveFailedException, 마커 IO = MarkerWriteFailedException, 복원 충돌 = RestorePathOccupiedException/RestoreTrashLostException(TO_ORIGINAL resolve 중 상태가 또 바뀐 stale 경합 — advice 가 이미 409 매핑).
  • DriftAutoApplyNotAllowedException 가드 기준이 autoApplicable → manuallyResolvable 로 이동. notApplicable·globalOff 두 factory 문구 모두 mode 어휘로 갱신(D5 마스터 개명과 동반 — "자동 적용" → "시스템 해결"). 409 계층 불변.
  • 스캔 중 무인 자동 적용 실패는 현행대로 WARN 후 계속(:482-487) — 보고서 생성 자체를 죽이지 않는 원칙 유지.
  • 각 sub-slice 에서 신규 예외가 필요해지면 해당 slice plan 의 §7 에서 확정 + 트리거 통합 테스트 동반(불가침).
§8. 부산물 · 주의
  • DDL 1건 + 권한 제약 — drift.kind VARCHAR(32) 확장(S6-2-2). claude_code 는 ALTER 불가(실측 ERROR 1142) — sql/ 스크립트 산출 후 사용자 적용(R9-5 선례). 현재 application.properties 가 사용자 변경으로 ddl-auto=update 상태라면 기동만으로 반영되나, validate 복귀 권장은 유효(스키마 안정화 커밋 2b7f397 의 취지).
  • 설정 breaking (S6-2-1 에 일괄) — ① reconciliation.auto-apply-path-drift/auto-apply-ghost-row 키 제거 → reconciliation.auto-apply.kinds CSV ② 마스터 reconciliation.auto-apply → reconciliation.resolution-enabled 개명(D5, MANUAL 도입에 따른 어휘 정합). 모든 키가 default 값으로 운영 중이라 실효 영향 0이나, 설정 파일에 구 키가 남아 있으면 무시됨을 report 에 명기.
  • findTrashed() 전수 패스 비용 가정 — quick scan(기본 1h 주기)마다 4 파일 도메인의 deleted row 전체 로드. 본 시스템의 자원 규모(도메인당 수십~수백)에서 무시 가능. 정렬 없는 전용 조회 메서드 신설은 비채택 — 호출 1곳을 위한 SPI 증식(over-abstraction), 규모가 문제 되는 시점에 분리.
  • 원위치 점유(원O·trashO)는 의도적으로 drift 로 보고하지 않는다 — 같은 경로에 새 자원을 올리는 것이 정상 워크플로라 오탐이 됨. restore 시점의 RestorePathOccupiedException 이 SSOT. 이 결정을 소스 주석에도 남긴다.
  • trash 자원의 내용 검증(deep hash)은 비목표 — trash 는 복원 시점에 manifestHash 검증이 있고, 상시 검증은 스캔 비용 대비 이득이 없다. 필요해지면 S6-3(A-4) 와 함께 재검토.
  • R9-6 의 한계 유지 — 재발급 중 "entity 갱신만 실패" 유형은 디스크 마커가 유효해 후속 스캔에 안 잡힌다(v2 §2 명기, 사용자 인지 완료). 후속 스캔은 마커 파일 계열 실패의 표면화 수단.
  • v2 보고서 비용 추정 정정 — B-2 구현 노선 "M~L (SPI 확장 + 6 도메인 파급)" → 실측상 탐지 SPI 신설 0(findTrashed 재사용), 파급은 재격리 entity 갱신 경로 1건 내외. 종합 M 이하.
  • forced-apply saga 경로(apply(id, true)·persistAndForcedApply) 불변 — resolution 디스패치로 치환하되 forced 는 mode 가드만 우회(동작 동일). 회귀 테스트로 고정.
  • CLAUDE.md 수정 없음(현황 서술은 Notion SSOT). DESIGN.md — 배지 3단 색(D4)이 기존 토큰 밖 신규 클래스를 요구하면 S6-2-1 에서 §4 표기 추가(R9 선례), 기존 토큰 내면 수정 없음.
  • 검증 이력 — 본 plan 은 작성 직후 적대적 비판 검증 1회(CRITICAL 1 · MAJOR 5 · MINOR 3)를 수행해 전부 반영한 개정본: 메타 자원 NPE/GHOST 오탐(D2 가드), 매트릭스 우선순위 은폐 2건(STALE 독립·모호 상태), forced 널가드(D3), 재격리 보상·구 trash 고아(D7), 마스터 설정명 의미 불일치(D5 개명), 배지 3단 색(D4), deletedByKey 인덱스(§3), findTrashed 비용 가정(§8).
