3.0 신규 인터페이스 LifecycleService (사용자 요구 2026-05-26 반영)

// global/lifecycle/LifecycleService.java
public interface LifecycleService {
    void toggleEnabled(Long id);
    void softDelete(Long id);

    /** default — restore(id, false) 위임. leaf / parent 공통. */
    default void restore(Long id) { restore(id, false); }
    /** parent / leaf 모두 이 한 메서드만 구현. leaf 는 cascade 와 무관하게 자식 0 복구. */
    RestoreResponse restore(Long id, boolean cascade);

    void deprecate(Long id);
    void undeprecate(Long id);
    void purge(Long id);
    void purgeWithTypedNameCheck(Long id, String typedName);
}

도메인 prefix 제거 (deprecateImage → deprecate, purgeImage → purge) 로 다른 자원 도메인이 같은 메서드명으로 구현 가능. Spring DI 는 구체 타입 (OSMetadataLifecycleService) 으로 주입 — 도메인 controller / scanner 가 자기 자원의 service 만 의존하므로 충돌 없음. 향후 cross-domain 일괄 처리 시 List<LifecycleService> 또는 Map<ResourceType, LifecycleService> 주입으로 확장 가능.

restore default 결정 (사용자 결정 2026-05-26 — 옵션 1) — cascade 시그니처를 abstract, 단순 시그니처를 default 로 두어 모든 구현체가 한 메서드만 구현 한다. leaf 자원 (ISO 등 자식 없음) 이 cascade=true 를 받아도 "자식 0 복구" 가 자연 무모순 결과 — silent 무시 위험 없음. OSMetadata 측 restore(Long id) 5 줄 위임 본체도 default 가 흡수해 제거됨.

§8. 부산물 / 주의 / 결정 사항

D-1. helper 의 중복 사본

requireActiveImage / requireDeletedImage 같은 single-row 조회 helper 가 두 service 양쪽에서 필요. 본 plan 의 채택 :

  • 안 채택 (D-1A) — 각 service 안에 private static 또는 private 사본. 도메인 추상이 약한 helper 라 over-abstraction 회피
  • (대안) — OSMetadataFinder 같은 공통 service 분리. 의존선 1 개 증가 + abstraction 1 단 추가. 본 슬라이스 범위 밖

D-2. purgeImage 의 위치

purge 는 lifecycle 의 "종착점" — enabled / deprecated / soft-deleted 와 다른 의미축 (row 자체 hard delete). 그러나 사용자 흐름상 softDelete → purge 가 연속되므로 같이 두는 게 자연스럽다. 본 plan 은 같이 이동.

D-3. purgeOSMetadataForNudge 의 잔류

이 메서드는 nudge replace 가 호출하는 purge — 다른 진입경로. nudge 의존이 강해 lifecycle service 로 이동하면 nudge 추적이 끊긴다. 본 plan 은 OSMetadataService 잔류. 향후 nudge 영역 정비 슬라이스에서 처리.

D-4. CP 침범 정책
