1. 현재 상태 — 길이 계약 전수 대조표 (실측 근거)

발단 (F-6 · O-7)

2026-07-19 샌드박스 전수 점검에서 두 결함이 잡혔다. F-6: ISOCreateRequest.description 에 @Size 가 없어 1,600자 설명이 Bean Validation 을 통과했고, background 등록 잡의 INSERT 단계에서 MariaDB 1406 (Data too long — iso.description VARCHAR(1024)) 으로 실패해 업로드 완주 후에야 DB_CONSTRAINT 격리로 흘렀다 (orphan saga 자체는 정상 동작 — 안전망이 받았을 뿐 앞단 400 필드 오류가 적정한 UX). O-7: 500 JSON 오류 바디에 trace 필드로 전체 스택트레이스가 노출됐다 (devtools 동봉 환경).

전수 대조표 — Request DTO 문자열 필드 vs 엔티티 @Column(length)

management 5 도메인 + provisioning setting 의 Request 문자열 필드 약 40개를 전수 대조한 결과. bios/bmc/subprogram/biossetting 의 Create·Update 계열은 이미 "@Size(max) = 엔티티 length + 한국어 메시지" 관례를 지키고 있고, OS·ISO·Board 3 도메인과 upload-intent 4종, setting name 만 관례에서 이탈해 있다 — 계약 설계 누락이 아니라 관례 확립(bios 계열 R 캠페인) 이전에 작성된 DTO 들이 정렬을 못 받은 것이다.

오류 표시 경로 이원화 (실측)

  • create 폼 = XHR JSON — 컨트롤러가 BindingResult 오류를 400 + ApiErrorResponse.fieldErrors[] 로 반환, static/global/form-error.js 가 data-error-field 요소에 .has-error+.field-error-message 주입 (form-validation.css 관례). description 은 전 create 템플릿에 data-error-field 가 이미 있어 슬롯 동작함.
  • edit 폼 6종 = 진짜 SSR 재렌더 — 컨트롤러가 hasErrors() 시 뷰 이름을 그대로 반환. 이 경로의 유일한 표시 수단은 <div class="n-error" th:errors> 인데, 6개 edit 템플릿 모두 description 에 n-error 슬롯이 없다 (name·version·isoPath 는 있음 — 같은 폼 안 자기모순). bios/bmc/subprogram edit 는 @Size 가 이미 있어 지금도 1025자 입력 시 오류가 무표시로 삼켜진 채 재렌더되는 재현 가능한 UX 버그.
  • 예외 케이스 — ISO 등록 form POST(IsoUploadController POST /{osId}/iso)는 create 지만 SSR 재렌더(iso-new 뷰 반환) — iso-new 의 description 도 n-error 슬롯 필요.

O-7 근본 원인 (jar 레벨 실측)

Spring Boot 4.0.3 에서 spring-boot-autoconfigure-4.0.3.jar!/META-INF/spring-devtools.properties 가 defaults.spring.web.error.include-stacktrace=always(+ include-message/include-binding-errors=always) 를 선언하고, devtools 의 DevToolsSettings.getPropertyDefaults() → DevToolsPropertyDefaultsPostProcessor 가 이를 최하순위 property source 로 주입한다. 본 프로젝트는 developmentOnly 'spring-boot-devtools' 를 쓰므로 개발/샌드박스 실행에서 기본값 never 가 always 로 상향돼 있었다. 응답 바디에서만 노출될 뿐 서버 로그의 스택은 이 설정과 무관하다.

키 함정 — 구 키 server.error.include-stacktrace 는 Boot 4.0.0 부터 구성 메타데이터 deprecation level=error (spring-boot-web-server 4.0.3 메타데이터 실측: replacement=spring.web.error.include-stacktrace) — 더 이상 바인딩되지 않는 죽은 키다. 구 키로 넣으면 devtools 가 올리는 신 키를 이기지 못해 O-7 이 고쳐지지 않는다.
2. 요구사항
  1. 앞단 400 필드 오류 (F-6) — §1 대조표의 불일치 필드 전부에 @Size(max = 엔티티 length) + 한국어 메시지("…N자 이하로 입력해주세요." — bios 계열 기존 관례 문구)를 추가한다. 업로드/등록 잡이 돌기 전에 제출 즉시 필드 오류로 거절한다.
  2. BMC legacy 255 정합 — targetDirectory 가 legacy file_path VARCHAR(255) NOT NULL 에도 미러링되므로 BMC 3 Request 의 상한을 effective 제약 255 로 정렬한다 (스키마 변경 금지 — 코드가 DB 에 맞춘다).
  3. SSR 오류 슬롯 보강 — SSR 재렌더가 실제 발생하는 7개 템플릿(edit 6종 + iso-new)의 description 에 기존 관례 그대로 n-error(th:errors) 슬롯을 추가한다. 인라인 스타일 금지, 각 템플릿의 인접 필드 마크업 복제.
  4. 오류 바디 스택 차단 (O-7) — application.properties 에 spring.web.error.include-stacktrace=never 명시 + WHY 주석(devtools 상향 근거·구 키 함정 포함). 로그 스택은 유지, 응답 바디만 차단.
  5. 테스트 — DTO Validator 단위(1024/64/255 경계) + @WebMvcTest 통합(ISO·OS·Board 대표 1,025자 → 400 fieldErrors / SSR 재렌더 필드오류) — §6.

4. 설계 결정 — 대안 비교 (채택 + 탈락 사유)

D1. F-6 해법 — 길이 검증을 어디에 둘 것인가

안내용판정
(a) 채택Request DTO 에 표준 @Size(max = 엔티티 length) + 한국어 메시지. bios/bmc/subprogram/biossetting 이 이미 확립한 관례에 OS·ISO·Board·intent·setting 을 정렬.신규 어휘 0 — 저장소 관례 그대로. BindingResult 자동매핑(Spring primitive)이 분기를 떠맡아 컨트롤러 무변경. 불일치가 "관례 미정렬" 이므로 정렬이 정답.
(b) 탈락커스텀 Bean Validation 어노테이션 또는 길이 상수 공유 클래스(LengthContract.DESCRIPTION)로 DTO·엔티티 양쪽 참조 (length SSOT 중앙화)이론상 SSOT 응집이 좋아 보이나, 이미 정렬된 bios 계열 5 도메인 전부를 함께 갈아야 반쪽이 안 된다 — 핫픽스 범위를 넘는 churn. 길이 계약은 스키마 고정 후 사실상 불변이라 drift 재발 압력도 낮다. 후속 리팩토링 후보로 §11 에 기록.
(c) 탈락DB 컬럼 확대 (VARCHAR(1024)→TEXT)스키마 변경 금지 (이 슬라이스는 코드가 DB 에 맞춘다) + 1,600자 설명 허용은 UX 요구가 아니다.
(d) 탈락Service 단 truncate/trimsilent 데이터 훼손 — 사용자가 입력 유실을 인지 못 한다.

D2. BMC targetDirectory — legacy file_path(255) 이중 매핑

안내용판정
(a) 채택BMC 3 Request(BmcCreateRequest·BmcRegisterExistingRequest 1024→255 하향, BmcUploadIntentRequest 255 신설)의 상한을 effective DB 제약 255 로 정렬 + WHY 주석. bios/subprogram(1024)과의 비대칭은 legacy 컬럼 제거 시 해소 — §8·§11 기재.기존 @Size(1024) 는 검증상한이 DB 제약보다 큰 유일 케이스 — F-6 동형 사고(검증 통과 후 INSERT 실패)의 잠복처. 256자+ 경로는 실사용 희귀라 UX 손실 미미.
(b) 탈락legacy file_path 컬럼 제거/확대스키마 변경 금지. legacy 컬럼 정리는 별도 마이그레이션 슬라이스 소관.
(c) 탈락현행 유지(1024)F-6 을 고치면서 같은 구조의 잠복 결함을 알고도 남기는 자기모순.

D3. O-7 — 어느 키로 고정하는가

안내용판정
(a) 채택spring.web.error.include-stacktrace=never (Boot 4 정식 키). application.properties 는 devtools 의 최하순위 property source 를 항상 이긴다.devtools 가 상향하는 키와 동일 키를 고정 — 어떤 실행 모드에서도 차단. 근거 주석에 jar 실측(§1) 명기.
(b) 탈락server.error.include-stacktrace=never (발견 보고서 원문 키)Boot 4.0 부터 deprecation level=error — 바인딩되지 않는 죽은 키라 devtools 의 신 키 상향을 못 막는다. 지시 취지(응답 바디 스택 고정 차단)를 신 키로 이행.
(c) 탈락spring.devtools.add-properties=false 로 devtools 기본값 전체 비활성템플릿 캐시 비활성 등 개발에 유익한 기본값까지 일괄 상실 — 과잉 차단.
(d) 탈락include-message / include-binding-errors 까지 never 고정발견은 trace 뿐. message 는 개발 진단에 유용하고 도메인 오류 응답은 커스텀 advice(ApiErrorResponse) 경로라 BasicErrorController 속성과 무관 — 범위 밖 (§8 비목표).

D4. SSR 오류 슬롯 — 추가 범위와 방식

채택: SSR 재렌더가 실제 발생하는 뷰(컨트롤러가 hasErrors() 에서 뷰 이름 반환 — edit 6종 + iso-new)의 description 에만 n-error 슬롯 추가. 각 템플릿의 인접 필드(name/version/isoPath) 마크업을 그대로 복제 — 신규 CSS 0. 탈락 1: 전 템플릿 일괄 추가 — 순수 XHR create 뷰의 th:errors 는 도달 불가 죽은 마크업(초기 GET 은 항상 무오류). 탈락 2: 오류 슬롯 fragment 공통화 — 템플릿마다 label·hint 구조가 달라 파라미터 폭발, 도메인 가로질러 의미를 잃는 분리(over-abstraction 경계).

D5. intent filename 과 경로 합성 엣지

채택: IsoUploadIntentRequest.filename 에 @Size(max=255) — 파일시스템 파일명 한계(ext4/APFS 255)와 일치. 잔여 엣지 인정: rawPath(≤1024) + filename(≤255) 합성이 1024 를 넘는 교차 필드 케이스는 필드 단위 @Size 로 못 잡는다 — 교차 필드 커스텀 검증기는 핫픽스 과설계로 탈락, 기존 DB_CONSTRAINT 격리 안전망(F-6 사고에서 정상 동작 실증)이 최후방을 유지한다. §8 기재.

8. 부산물 / 주의
  • scope 경계 — setting Process 하위 문자열(파티션·계정 등)은 JSON blob 저장이라 컬럼 length 계약이 없어 비목표. 길이 상한 정책(예: password 최대치)은 도메인 결정이 필요한 별건 — §11 후속 기록. include-message/include-binding-errors 고정도 비목표(D3-d).
  • BMC 255 하향은 행동 변경 — 기존엔 256~1024자 targetDirectory 가 검증을 통과(했다가 INSERT 실패)했다. 이제 앞단 400. bios/subprogram(1024)과의 비대칭은 legacy file_path 컬럼 제거 마이그레이션 시 1024 복원으로 해소 — WHY 주석에 명기.
  • 교차 필드 합성 엣지 잔존 — intent 의 rawPath+filename 합성 1024 초과는 필드 @Size 로 못 잡음(D5). DB_CONSTRAINT 격리 안전망 유지가 설계 의도.
  • 병렬 세션 충돌 없음 — 수정 파일은 management/·provisioning/setting/·resources/ 한정. execution/·provisioning/server* 무접촉. CLAUDE.md/AGENTS.md 무수정. 커밋 없음 (워크트리 상태로 보고).
  • intent 요청의 기존 bare @Size(BiosUploadIntentRequest version 등)는 메시지 없이도 동작 중 — 이번 추가분은 한국어 메시지를 포함하되 기존 필드는 건드리지 않는다 (핫픽스 최소 diff).
  • devtools 상향의 다른 두 키(include-message/include-binding-errors=always)는 개발 진단용으로 존치 — 운영 프로파일(LOG L4)에서 별도 점검 후보.
