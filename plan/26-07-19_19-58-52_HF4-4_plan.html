1. 현재 상태 (실측 근거)

O-3 — 스캔 이력 배지가 '미해결 잔수'로만 동적 감소한다

자원 무결성 점검(/maintenance/reconciliation)의 C1 스캔 이력 배지와 상단 요약줄("마지막 점검 … 드리프트 N건")은 모두 DriftReportResponse.driftCount 를 표시하는데, 이 값의 원천은 DriftReport.getDriftCount() = drifts.size(), 즉 지금 남아 있는 자식 행 수다. 그런데 드리프트를 [적용]/[보고 닫기] 하면 DriftReport.removeDrift() 가 자식 행을 분리하고 orphanRemoval=true 가 DB 행을 물리 삭제한다. 결과: 과거 보고서의 배지도 함께 줄어들어, "그 스캔에서 몇 건이 탐지됐었는가"라는 역사적 사실이 화면에서 사라진다. 전부 해결하면 과거 보고서가 전건 "0 건"(초록)이 되어 이력으로서의 의미를 잃는다.

표시 위치현재 코드동작
C1 스캔 이력 배지list.html — th:text="${report.driftCount} + ' 건'"해결할 때마다 4→3→…→0 감소
상단 요약줄list.html — "드리프트 ${latestReport.driftCount}건"동일하게 감소
REST /latest·/historyDriftReportResponse.driftCount외부 API 도 잔수만 노출

참고: 스캔 완료 토스트(list.js 의 job metadata driftCount)는 스캔 완료 시점 값이라 사실상 탐지치와 같다 — 문제는 그 값이 어디에도 영속 보존되지 않는다는 점이다. 선례 R9-5 는 같은 구조의 문제(자원 purge 후 보고서에서 실명 소실)를 drift.display_name 스냅샷 컬럼으로 해결했다.

O-1 — 스캔 범위의 특성이 화면 어디에도 안내되지 않는다

PathReconciliationService.computeScanRoots()(D19) 는 스캔 루트를 active 자원 path.parent 의 union + reconciliation.scan.extra-roots 설정으로 동적 산출한다. 따라서 자원을 이 범위 밖(예: 별도 백업 폴더)으로 통째 옮기면 (resourceType, resourceId) 매칭 마커를 걸을 수 없어 '경로 이동(PATH_DRIFT)'이 아니라 '자원 소실(MISSING)' 로 보고된다. 이 동작 자체는 설계 의도이나, 운영자가 알 수 있는 곳이 코드 Javadoc 뿐이다 — 화면 상단 설명("등록된 자원 정보(DB)와 …")에도, MISSING 카드의 권장 조치(DriftKind.MISSING.recommendedAction)에도 extra-roots 언급이 없다.

2. 요구사항
  1. 탐지 건수 스냅샷 (O-3) — 보고서 생성 시점에 확정되는 "탐지 N건"을 drift_report 에 영속 보존한다. 이후 해결로 자식 행이 삭제돼도 탐지치는 불변.
  2. 병기 표시 — C1 배지·상단 요약줄을 "탐지 N · 미해결 M" 병기로 바꾼다. 미해결 산출은 기존 drifts.size() 로직 그대로 재사용. 색 판정(초록/회색)도 기존처럼 미해결 > 0 기준 유지 — "탐지 4 · 미해결 0" 은 처리 완료의 초록.
  3. 구행 호환 — 스냅샷 도입 이전 행(컬럼 default 0)은 화면에서 미해결 수로 대체 표기한다. 구행 데이터 마이그레이션(backfill)은 비목표 — sql 주석으로 명시하고 FIFO retention 으로 자연 해소.
  4. 스캔 범위 안내 (O-1) — 화면 상단 설명 영역에 ① 스캔 범위 = 활성 자원 폴더 + extra-roots ② 범위 밖 이동은 '자원 소실'로 보고 ③ extra-roots 로 보강 가능, 세 사실을 안내한다.
  5. MISSING 권장 조치 보강 — '자원 소실' 카드의 권장 조치에 "스캔 범위 밖(백업 폴더 등)으로 옮긴 경우라면 reconciliation.scan.extra-roots 에 해당 경로를 등록하면 '경로 이동'으로 탐지된다" 취지 한 문장을 추가한다. 문구 SSOT 는 기존대로 DriftKind enum(R9-2 문구 표).
🎬 라이브 데모 — 스캔 이력 배지 AS-IS / TO-BE 병렬 체험 + 안내 문구 미리보기

스캔을 실행해 드리프트 4건을 탐지한 뒤 하나씩 해결해보라. 왼쪽(AS-IS)은 현재 결함 — 배지가 4→3→…→0 으로 줄며 탐지 이력이 소실된다. 오른쪽(TO-BE)은 본 plan 적용 후 — 탐지 4 는 스냅샷으로 고정되고 미해결만 감소한다. 판정식은 서버와 동일: 탐지 = addDrift 시점 누적(불변) / 미해결 = drifts.size()(잔존).

① 화면 상단 설명 배너 — 추가 줄은 초록 배경
등록된 자원 정보(DB)와 디스크의 실제 파일·마커가 서로 일치하는지 점검하고, 불일치(드리프트)를 정합합니다.
점검은 활성 자원이 등록된 폴더들과 reconciliation.scan.extra-roots 설정에 지정한 폴더만 검색합니다. 자원을 이 범위 밖(예: 별도 백업 폴더)으로 옮기면 파일을 찾을 수 없어 '경로 이동'이 아닌 '자원 소실'로 보고됩니다 — 그런 위치를 쓴다면 reconciliation.scan.extra-roots 에 해당 경로를 추가해 점검 범위를 넓힐 수 있습니다.
② '자원 소실(MISSING)' 카드의 권장 조치 — 추가 문장은 초록 배경
권장 조치 : 백업·이동·이름 변경 여부를 확인해 파일을 원래 경로·이름으로 복원한 뒤 다시 점검하세요. 시스템이 자동 복구할 수 없습니다. 점검 범위 밖(백업 폴더 등)으로 옮긴 경우라면 reconciliation.scan.extra-roots 설정에 해당 경로를 등록하면 다음 점검부터 '경로 이동'으로 탐지됩니다.
데모의 구행 재현 버튼은 스냅샷 컬럼 도입 이전에 저장된 보고서(detected_drift_count = 0, 미해결 3건 잔존)를 이력에 끼워 넣는다. TO-BE 쪽에서 이 행은 "탐지 3 · 미해결 3" 으로 — 즉 탐지 자리를 미해결 수로 대체 표기됨을 확인할 수 있다 (D3 fallback 규칙).

4. 도메인 모델 — 설계 결정 (대안 비교 포함)

D1. 탐지 건수를 어떻게 보존하나

안내용판정
(a) 채택drift_report.detected_drift_count 스냅샷 컬럼. 보고서 생성(자식 추가) 시점에 확정, 이후 불변.탐지치는 "보고서 생성 시점에 확정되는 값"이라는 도메인 사실과 일치. 선례 R9-5(drift.display_name 스냅샷)와 동일 패턴 — 컬럼 1개·ALTER 1회로 끝난다.
(b) 탈락drift 자식 행 집계 쿼리로 탐지치 복원불가 — [적용]/[보고 닫기]가 orphanRemoval 로 자식 행을 물리 삭제하므로 집계할 원본이 남지 않는다. 삭제를 soft-flag(status 컬럼)로 바꾸면 가능하지만 "Drift 는 불변, 해소 = 행 제거"라는 기존 설계와 resolution/recheck 계층 전반을 갈아엎는 큰 churn — 표시 수치 1개가 동기로는 과하며 HF4-5 와도 정면 충돌.
(c) 탈락스캔 이벤트 로그 테이블(append-only) 신설수치 1개를 위해 신규 테이블·엔티티·FIFO 연동 정리 정책까지 신설 — over-engineering. 이벤트 로그가 필요해지는 시점(감사 추적 요구)에 도입해도 늦지 않다.

D2. 스냅샷 기록 시점 — 어디서 기록하나

안내용판정
(i) 탈락performScan 저장부에서 빌더로 .detectedDriftCount(drifts.size()) 명시 전달저장 호출부가 2곳(performScan / persistAndForcedApply)이라 각각 기억해야 하고, 새 저장 경로가 생기면 누락이 침묵으로 지나간다. Service 파일 수정 범위도 커져 HF4-5 충돌면 확대.
(ii) 채택DriftReport.addDrift() 내부에서 detectedDriftCount++. removeDrift() 는 감소시키지 않음."탐지 수 = 이 보고서에 추가된 적 있는 drift 총수"라는 정의가 엔티티 안에서 SSOT 로 성립 — 호출부는 아무것도 기억할 필요 없다. saga 임시 보고서(persistAndForcedApply)도 자동으로 탐지 1 이 기록된다. JPA 로딩은 컬렉션 필드 직주입이라 addDrift() 를 타지 않는다 — 재로딩 시 증가 부작용 없음.

D3. 구행 호환 — 스냅샷 없는 행의 표기

안내용판정
(A) 탈락NULL 허용 컬럼 — null = 구행 (R9-5 displayName 방식)표기 결과는 같지만 Integer 박싱 + null 분기를 도처에 만든다. displayName 은 "null 이면 다른 정보(TYPE #id)로 대체"라 null 구분이 필수였지만, 여기는 아래 (B)처럼 0 이 자연스러운 경계라 null 이 주는 정보가 없다.
(B) 채택INT NOT NULL DEFAULT 0 — 구행은 0. 표시 규칙: 스냅샷이 0 이면 미해결 수로 대체.신행에서 스냅샷 0 은 "탐지 0건" — 이때 미해결도 반드시 0(자식이 애초에 없고 줄기만 한다)이라 대체 표기가 항상 참이다. 즉 0-경계의 의미 충돌이 구조적으로 없다. 원시 int 유지.

대체 규칙의 SSOT 는 DriftReport.getDetectedDriftCountForDisplay() 1곳 — 템플릿 표시 2곳(C1 배지·요약줄)이 각자 fallback 을 복붙하면 drift 가 생기므로, toResponse 가 이 메서드 결과를 DriftReportResponse.detectedDriftCount 에 담아 내려보내고 템플릿은 그대로 출력만 한다.

D4. O-1 안내 문구의 위치

안내용판정
(a) 채택① 상단 설명 배너에 안내 줄 추가(기존 Bootstrap text-muted 클래스 — 인라인 스타일 금지 준수) ② DriftKind.MISSING.recommendedAction 문자열에 한 문장 추가②는 R9-2 가 확립한 "사용자 노출 문구는 DriftKind enum 이 SSOT" 원칙 그대로 — 카드·확인 모달·추후 가이드 문서가 전부 한 곳을 따른다. enum 상수 신설이 아니라 기존 상수의 문구 수정이므로 HF4-5 금지 조항(DriftKind 신설 금지)과 무관.
(b) 탈락템플릿 C3 카드에 조건부 하드코딩 문구MISSING 문구의 두 번째 소유자를 만든다 — R9-2 이전의 "템플릿 하드코딩 ×3곳" 사고 재발 경로.
(c) 탈락별도 도움말 페이지/모달 신설안내 세 문장을 위해 URL·뷰·라우팅 신설은 과함. 필요 시점에 확장.

  • 매핑 fallback — latestReport() 응답: 신행(탐지 2·미해결 1) → detectedDriftCount 2 / 구행(스냅샷 0·잔존 3) → 3.
  • saga 임시 보고서 — persistAndForcedApply 경유 보고서도 탐지 1 자동 기록 (addDrift 내부화의 검증).
