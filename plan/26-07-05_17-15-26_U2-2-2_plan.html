§2. 요구사항 (확정 — 사용자 지시 3건 + 설계 승계) + 비목표
  1. 상세 화면 — 변경 속성 한눈에: 저장 flat 값을 registry 와 재조인해 메뉴 경로(MenuPath) 그룹으로 렌더. 각 행 = DisplayName 라벨(사용자 지시 승계) + 저장값(ENUMERATION 은 옵션 표시명 valueDisplayName 병기) + 기본값 대비(default → 저장값) + AttributeName(mono 보조). registry 에서 사라진 저장 속성은 stale 경고 행, 카탈로그 소실 보드는 raw flat + 경고 배너(degraded)로.
  2. Redfish Request Body 프리뷰: 상세 화면에 실행 시 전송될 요청을 표시 — PATCH /redfish/v1/Systems/Self/Bios/SD + {"Attributes":{…}}(pretty JSON, 타입 보존) + 저장 속성 중 resetRequired 존재 시 POST …/ComputerSystem.Reset {"ResetType":"ForceRestart"} 안내. 조립은 보존된 BiosRedfishPayloadAssembler.assembleAttributes() + 상수 재사용(중복 조립 금지).
  3. 수정: 상세의 '수정' 버튼 → 생성과 동일한 편집기 화면(bios-setting-editor 재사용, edit 모드) + 저장값 pre-fill → PUT /{id}. name(자기 제외 중복 409)·description·attributes 재검증(create 루프 재사용), boardKey 는 계약 미포함(불변).
  4. 삭제: 상세·목록에서 confirm-modal 후 DELETE /{id} → 204 → 목록. hard delete — 사용중 가드는 참조가 생기는 U2-2-3 에서(설계 확정).
  5. 목록 행 링크 활성화 → 상세.
  6. initial-settings(md) 메커니즘 제거 (사용자 확정 — md 는 임시물, 기본값 SSOT 는 registry/XML): BiosInitialSettingsParser 삭제 · BiosResourceProperties.Board.initialSettings 컴포넌트 제거 · BiosSetupLoader.readInitialSettings 제거 · BiosSetupMenu.currentValues 필드와 BiosSetupDataParser.parse 파라미터 제거 · BiosWidgetRowResponse.resolveBaseline 의 currentValue 분기 제거(baseline = registry DefaultValue 단일화) · BiosPageResponse 의 currentValue 조회 제거. 전부 소비 사슬이 명확한 기계적 제거 — "보드 실제 현재값" 개념은 Stage 4 의 Redfish 라이브 GET 으로 재도입 후보(§8).
비 목표 : 사용중 삭제 차단 SPI·guest 연동(→ U2-2-3) · Redfish 실호출(→ Stage 4) · MS74-HB0 파일 이슈(별도 결정) · boardKey 변경 기능(불변 설계).
🎬 미리보기 — flat 저장 ↔ 재조인 상세 ↔ Redfish body 라이브 데모

이 slice 의 본질은 "flat 저장값 하나가 세 얼굴로 렌더된다" 는 것이다: ① 재조인 상세(메뉴 경로 그룹 + DisplayName + 기본값 대비) ② Redfish Request Body(실행 시 전송분 — 저장 JSON 과 구조 동형) ③ 수정 편집기의 pre-fill(위젯 선택값만 저장값으로, diff 기준선은 불변). 아래에서 저장값을 바꾸고 stale 시나리오를 토글하면 세 뷰가 동시에 갱신된다 — 판정·조립 로직은 서버(registry 재조인 + assembler)와 1:1.

저장값 편집 :

① 상세 — 재조인 그룹 렌더 (어떤 속성이 어떻게 바뀌는지 한눈에)

② Redfish Request Body (실행 시 전송분 — BiosRedfishPayloadAssembler 조립과 동형)

③ 수정 진입 :
저장값 토글로 상세·Redfish body 가 함께 바뀌는 것을 확인하세요. stale 토글은 BIOS 카탈로그 개정 시나리오입니다.
드리프트 0 — 그룹 경로 = registry MenuPath(SSOT), 라벨 = DisplayName, Redfish body = 저장 flat 값의 무변환 투영(assembleAttributes 재사용), 수정 pre-fill 은 위젯 선택값만 저장값으로 세팅하고 diff 기준선(data-default)은 생성 때와 동일 — 사용자가 기본값으로 되돌리면 PUT diff 에서 자연 탈락한다.

구현 수준 결정

#결정근거 / 대안
D1상세 그룹 = registry MenuPath 문자열 기준(문서 순서 보존)페이지 트리 워킹 대안 기각 — MenuPath 가 registry 에 실재하는 경로 SSOT 이고, 상세는 편집기와 달리 트리 내비게이션이 불필요(평면 그룹이 "한눈에" 요구에 부합).
D2ENUMERATION 저장값은 valueDisplayName 으로 표시(원시 ValueName 은 mono 보조)DisplayName 라벨 지시의 값-측 일관 적용. 기본값도 동일 규칙.
D3Redfish 프리뷰는 조회 시점 조립(저장 안 함) + assembleAttributes/상수 재사용body 를 저장하면 값과 이중 진실. 프리뷰 = 저장 flat 의 무변환 투영이므로 조립 비용 무시 가능. 별도 프리뷰 조립기 신설 기각(중복).
D4resetRequired = 저장 속성들의 registry 재조인 OR — stale 속성은 판정 불가로 제외하고 경고에 병기실행 의미(재부팅 필요)의 표시용 근사 — 최종 판정은 Stage 4. 상세에만 표시, 저장 안 함.
D5edit pre-fill = BiosWidgetRowResponse.storedValue 신필드(위젯 선택값만 세팅), data-default(diff 기준선)는 생성과 동일저장값을 baseline 으로 쓰는 대안 기각 — 그러면 "저장값에서 안 바꾼 속성"이 diff 에서 빠져 PUT 시 기존 값이 유실된다. 기준선 불변이라 되돌리기(기본값 복귀)도 자연스럽게 diff 탈락.
D6PUT 검증 = create 의 validateAndCoerce 그대로 재사용 + existsByNameAndIdNot 중복 검사같은 서비스 내 private 재사용 — 분리 추상 불요.
D7catalogMissing 플래그가 상세의 degraded 렌더 + edit 버튼 disabled + 서버 404 의 단일 판정UI 차단 조건과 서버 가드 조건의 SSOT 공유(불가침) — Query 가 로더 실패(BiosBoardNotFound)를 흡수해 플래그화하고, editorView 는 같은 로더 호출이 404 로 남는다(안전망).
D8renderMenu 삭제PoC 폐지 후 소비자 0 — 죽은 코드. listBoards 는 소비자 1 유지(Stage 4 재소비 예정 — 공급자 계층 churn 회피).
D9initial-settings(md) 사슬 전체 제거 — baseline = registry DefaultValue 단일화사용자 확정(임시물 은퇴). 제거 후 currentValues 의 쓰기 지점이 0 이 되므로 dormant 유지 대안은 죽은 코드 존치 — 기각. MS74-HB0/기타 보드의 편집기·diff 의미론이 "공장 기본값 대비"로 전 보드 균일해진다.
적대적 자기비판 — ① stale 속성을 Redfish body 에 포함하는 게 맞나? 포함이 정직하다: 저장분은 실행 시 그대로 전송될 값이고, 임의 제외는 silent drop(경고 병기가 올바른 처리; 제거는 사용자가 수정 화면에서 재저장으로 수행 — -1 확정 의미론). ② DetailResponse 가 비대하지 않나? 상세 한 화면의 전속 뷰모델 — 재사용 목적이 없어 분해 이득 0. ③ degraded 상세에서 Redfish 프리뷰는? 표시한다 — body 는 registry 없이도 저장 flat 만으로 조립 가능(그룹 렌더만 불가). ④ PUT 에서 values 전체 교체 의미론 — diff 수집이 전체 기준선 대비라 부분 patch 아님(전체 교체가 계약) — JS 가 저장값 포함 전 변경분을 재수집하므로 유실 없음(D5). ⑤ 순환? 신규 의존 없음(biossetting → provisioning.domain/service 단방향 유지).

§8. 부산물 / 주의 / 결정 사항

의도된 잔존 / 임시 비대칭 (+ 해소 시점)

  • hard delete 에 사용중 가드 없음 — 참조가 존재하지 않는 기간(guest 연동 전). U2-2-3 에서 SPI 가드 + 삭제 버튼 disabled 도입.
  • stale 속성은 표시·경고까지만 — 실행 시 처리(전송 여부 최종 판정)는 Stage 4. 재저장 시 자연 탈락(-1 확정).
  • "보드 실제 현재값" 표시 기능의 공백 — md 메커니즘 은퇴로 전 보드가 공장 기본값 baseline. 실물 보드 현재값은 Stage 4 의 Redfish 라이브 GET …/Bios 연동이 자연스러운 재도입 지점(임시 파일 자산 없이).

주의
