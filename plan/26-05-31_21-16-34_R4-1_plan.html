§1. 현재 상태 (진단)

진입 전제

  • R2-2 / R2-2-1 완료 — 부모 상태로 자식 액션을 차단하는 read-time 가드(childEnableBlockReason / blocksChildUndeprecate / blocksChildRestore)가 (1) 서버 throw + (2) 뷰 disabled 플래그의 SSOT 로 정착.
  • R3-1 완료(green) — board-scoped Subprogram 이 BoardModel cascade 묶음(softDelete / restore / deprecate / undeprecate)에 합류. 부모 cascade 가 3 자식 도메인(BIOS / BMC / Subprogram)을 모두 다룬다.
  • 본 슬라이스는 R2-2 의 "차단(restrict)"이 아니라 "역전파(reversal)" 측 결함을 다룬다 — 부모를 되돌릴 때 자식을 어디까지 되돌리나.

핵심 결함 (workflow 전수 인벤토리로 확정)

부모 undeprecate cascade 가 자식의 "출처"를 보지 않는다.
현재 BoardModelService.undeprecate(L294–307) / OSMetadataLifecycleService.undeprecate(L144–152) 의 자식 forEach 필터는 (!isDeleted && isDeprecated) — 값만 본다. 그래서 부모를 undeprecate 하면 운영자가 손수 deprecate 해 둔 자식까지 무차별 환원된다. "값(is_deprecated=true)은 같지만, 그게 부모 cascade 로 깔린 것인지 운영자가 직접 깐 것인지" 를 시스템이 기록하지 않는 것이 결함의 진원.

§2. 요구사항 (확정) + 비목표

확정 (사용자 최종 결정)

  1. 값 = boolean 유지. is_enabled / is_deprecated / is_deleted 컬럼 그대로. 값 패킹(A안) 거부 — JPQL 에 비트 연산 & 부재로 6 repository derived query 가 전부 native 로 강제되는 회귀를 피한다.
  2. 출처 = 단일 bitmask 컬럼 cascade_origin. 차원별 1비트, 0 = 자동(AUTO, 부모 추종, 기본) / 1 = 직접(EXPLICIT, 운영자 관리). LifecycleEntity(@MappedSuperclass)에 추가 → 6 자식 테이블 일괄(ddl-auto=update 1회).
  3. 최소 적용 차원 = enabled(bit0) + deprecated(bit1). deleted(bit2)는 예약(미사용) — trash / TTL / ghost 물리부작용 + softDelete/restore 가 child-service / trash 위임이라 cascade↔manual 컨텍스트 신호가 없음(R4-2 로 이연).
  4. '자동' 체크박스로 출처를 UI 에서 입력 (사용자 신규 결정). 상세표 '상태' 행의 활성화/비활성화 버튼 왼편에 [✓ 자동] 체크박스 + 오른쪽에 ? 아이콘(hover 설명). 기본 체크(=자동/AUTO). 자식에서 활성화/비활성화 버튼을 누르면 체크가 자동 해제 → 직접(EXPLICIT).
  5. deprecated UI 이동. 기존 deprecate 버튼을 액션줄에서 떼어 상세표 '상태' 바로 밑 신규 행으로 옮기고, 버튼 옆에 [✓ 자동] 체크박스(+?)를 둔다.
  6. 방향성 일관화 — enabled 를 양방향으로 정렬. 부모 disable 시 전 자식 강제 disable(부모 우선), 부모 enable 시 자동 ✓ 자식만 re-enable + 직접 disabled 자식 보존(자동 우선). deprecate↔undeprecate 와 같은 우선순위 비대칭(무게/부담 역전 해소, Windows ACL "remove inherited copies" 와 1:1).
  7. CLAUDE.md 정합 — 분기 줄 추가 대신 메서드-per-의도. entity 에 cascade 전이(deprecateCascade, 값만)와 운영자 전이(deprecateByOperator, 값+markExplicit)를 별도 메서드로 분리 → 호출자가 메서드 선택으로 의도 표현, origin 스탬프 누락(드리프트)을 컴파일·리뷰 단에서 차단.

체크 상태 ⟺ cascade_origin 비트 : 자동 ✓ = 비트 clear(0) = AUTO(기본) 자동 ☐ = 비트 set(1) = 직접(EXPLICIT). default 0 = 자동 ✓ (요구 4 "기본 체크"). "inherit/explicit 를 백엔드로 어떻게 가져오나" 문제를 체크박스 상태 = 비트값으로 직결해 해소.

우선순위 비대칭(요구 2·3) : 자동 ☐(직접) 자식이라도 부모가 deprecate/disable/delete 되면 강제로 함께 내려간다(DOWN: 부모 우선). 반대로 부모가 풀릴(undeprecate/enable/restore) 때는 자동 ✓ 자식만 따라 올라오고, 자동 ☐ 자식은 보존된다(UP: 자동 우선). 자동 플래그는 DOWN 에서 보존되어 UP 에서 존중됨.
비 목표
  • deleted 차원 provenance — bit2 예약만. 적용은 child-service / trash 위임에 컨텍스트 신호 주입이 선행돼야 함 → R4-2.
  • scanner(reconciliation) restore 의 출처 정책 — system-driven 3rd 컨텍스트. INHERITED/EXPLICIT/system 처리 방침 확정 → R4-2.
  • effective-read(A안, read-time 계산) 거부 — 출처는 영속 컬럼으로만. derived query 무변경 보장.
  • 부모 자원(BoardModel / OS 이미지) 과 공용 Subprogram(부모 없음) — '자동' 체크박스 미노출(추종할 부모가 없어 자동/직접 구분이 무의미). 부모 자신의 deprecate 버튼은 '상태' 밑으로 이동하되 체크박스 없음.
마이그레이션 불필요 — pre-deployment. 신규 컬럼 default 0 = 전 자식 자동(✓) 으로 시작(요구 4 "기본 체크"와 정합; 공용/부모는 추종 대상 없어 무의미).

§4. 도메인 모델 (SSOT)

신규 타입 (D6 Primitive Obsession 금지 준수 — 출처는 도메인 개념)

타입종류책임
LifecycleDimensionenumENABLED(bit0) · DEPRECATED(bit1) · DELETED(bit2, 예약). 각 상수가 비트 마스크 보유
CascadeOriginVO (record, byte 래핑)비트 set(1) = 직접(EXPLICIT) / clear(0) = 자동(AUTO, 기본). isAuto(dim) · isExplicit(dim) · markExplicit(dim) · markAuto(dim) · empty()(전 차원 자동). 불변
CascadeOriginConverter@ConverterCascadeOrigin ↔ Byte(TINYINT) JPA 변환 (컬럼 cascade_origin, default 0=자동)
우선순위 비대칭 (사용자 확정) — 같은 부모-자식 cascade 라도 방향에 따라 우선순위가 뒤집힌다 :
  • DOWN 제약 (부모 → disabled / deprecated / deleted) : 부모 > 자동. 자동 OFF(직접) 자식도 강제로 함께 내려간다(값 강제). 자동 플래그는 보존.
  • UP 허용 (부모 → enabled / undeprecated / restored) : 자동 > 부모. 자동 ON 자식만 환원. 자동 OFF(직접) 자식은 보존.
⟹ "막을 땐 부모가 절대적, 풀 땐 자식의 자동 의사를 존중". R2-2 read-time 가드(자식 enable/undeprecate/restore 차단)가 DOWN 제약의 또 다른 표현과 정합.

LifecycleEntity 변경 (@MappedSuperclass → 6 엔티티 일괄)

@Column(name = "cascade_origin")
@Convert(converter = CascadeOriginConverter.class)
private CascadeOrigin cascadeOrigin = CascadeOrigin.empty();   // default 0 = 전 차원 자동(✓)
// 비트 = '운영자 직접 관리 플래그' : 1=직접(EXPLICIT, 자동 ☐) / 0=자동(AUTO, 자동 ✓, 부모 추종). 기본 자동.
//   DOWN 제약은 이 플래그를 무시하고 값을 강제(플래그 보존). UP 허용만 이 플래그를 존중.

// (A) cascade : 값만 바꾸고 자동 플래그는 보존 — 부모가 자식을 강제/환원
public void deprecateCascade()   { /* guard */ this.isDeprecated = true;  this.deprecatedAt = now(); }   // DOWN: 자동 무관 강제
public void undeprecateCascade() { /* guard */ this.isDeprecated = false; this.deprecatedAt = null;  }   // UP: 자동 ON 자식만(호출 전 filter)
public void disableCascade()     { this.isEnabled = false; }   // DOWN
public void enableCascade()      { this.isEnabled = true;  }   // UP: 자동 ON 자식만

// (B) 운영자 직접 : 값 + 자동 해제(직접 마킹) — 한 메서드로 묶어 스탬프 누락(드리프트) 차단
public void deprecateByOperator()   { /* guard */ this.isDeprecated = true;  this.deprecatedAt = now();
                                      this.cascadeOrigin = cascadeOrigin.markExplicit(DEPRECATED); }
public void undeprecateByOperator() { /* guard */ this.isDeprecated = false; this.deprecatedAt = null;
                                      this.cascadeOrigin = cascadeOrigin.markExplicit(DEPRECATED); }
public void toggleEnabledByOperator(){ this.isEnabled = !this.isEnabled;
                                      this.cascadeOrigin = cascadeOrigin.markExplicit(ENABLED); }

// (C) '자동' 재체크 : 자동 ON 으로 되돌리고 값을 부모값으로 snap
public void reinherit(LifecycleDimension dim, boolean parentValue) {
    this.cascadeOrigin = cascadeOrigin.markAuto(dim); /* + dim 에 따라 값 = parentValue */ }

                                      // 자동 ON 자식만 환원 · 직접(EXPLICIT) 자식은 보존
manual    (child service)  resource.deprecateByOperator() / undeprecateByOperator() / toggleEnabledByOperator()  // 자동 해제(☐)
재추종    (cascade-origin) resource.reinherit(dim, 부모값)                          // 자동 ✓ + 값 snap
cascade 메서드(A)와 운영자 메서드(B)를 entity 에서 분리 — 같은 전이라도 "부모 강제(플래그 보존)" 와 "운영자 직접(플래그 set)" 은 다른 메서드다. 호출자가 메서드 선택으로 의도를 표현 → origin 스탬프 누락(드리프트) 컴파일·리뷰 단에서 차단(CLAUDE.md §조건분기 — 인자 분기 대신 메서드-per-의도). 어느 자식을 환원할지 선별은 부모 service 의 filter(=SSOT). DOWN 은 origin 을 건드리지 않는다(부모가 강제해도 자식의 직접 플래그는 보존) → UP 에서 그 플래그가 존중됨.

상태 + 출처 전이 (deprecated 차원, deprecated 예시)

자식 초기 부모 deprecate (DOWN) 부모 undeprecate (UP) 최종 자동✓ 활성(C) ─▶ Deprecated, 자동✓ ─▶ 활성, 자동✓(환원) 자동✓ 왕복 직접☐ 활성(A) ─▶ Deprecated, 자동☐(강제) ─▶ 보존(자동☐) → Deprecated 직접 관리(운영자가 풀어야) 직접☐ Deprecated(B) ─▶ 이미 dep, 그대로 ─▶ 보존(자동☐) → Deprecated 직접 관리 운영자 deprecate ─▶ 값 set + 자동 해제 ☐(직접) '자동' 재체크 ─▶ 자동 ✓ + 값 = 부모값 snap

§8. 예상 부산물 / 주의 / 결정 사항
⚠ 알려진 한계 — 단순 비트 모델로 미해결 (2026-05-31 사용자 판정, CP1 PARKED)
값=boolean(1개) + 차원별 1비트(자동/직접) provenance 만으로는 아래 force-down-while-explicit 케이스를 풀 수 없다 :
  1. BIOS A = 자동 ☐(직접) + 활성 (운영자가 직접 관리, 활성 유지 의도).
  2. 부모 deprecate (DOWN, 부모>자동) → A 강제 deprecate. A = {값: deprecated, origin: 직접}.
  3. 부모 undeprecate (UP, 자동>부모) → 자동 ON 자식만 환원 → A(직접) 환원 안 됨 → A 무조건 deprecated 잔존.
근본 원인 : 2단계 후 A 의 {deprecated, 직접} 가 운영자가 직접 deprecate 한 자식 B 의 {deprecated, 직접} 와 구별 불가 → UP 이 "A 는 회복, B 는 보존"을 판별 못 함. 단일 boolean 이 A 의 cascade 이전 본래 의도값(활성) 을 기억 못 하기 때문. (본래 의도값 · 부모강제 여부 · 추종모드 3 정보를 1 boolean + 1 bit 로 표현 불가.)
⟹ 본 plan 의 단순 bitmask 설계는 보류. 사용자가 개선 아이디어를 제시 예정. 🎬 데모에서 A 가 부모 해제 후에도 Deprecated 로 남는 것이 바로 이 미해결 케이스의 시연.
