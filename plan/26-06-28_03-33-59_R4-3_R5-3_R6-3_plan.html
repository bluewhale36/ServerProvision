§1. 현재 상태 · 목표

R3-2~R6-5 리팩토링 캠페인은 OS(R1/R3)의 Controller/Service 분리를 leaf 도메인으로 확장한다. Board(R3-3)는 이미 분리 완료(BoardModelMetadataService / BoardModelLifecycleService / BoardModelMarkableScanner / BoardModelNudteService). 남은 fat 은 board-scoped leaf 3종이다.

목표 — 세 fat *Service 를 OS/Iso·Board 와 동형으로 5분할: *LifecycleService(상태 전이 8 + forging 가드) · *RegistrationService(등록 3경로 + 복붙 dedup) · *MarkerWriter(marker 4-step) · *IntegrityService(무결성 검증, IsoIntegrityService 선례) · 잔류 *Service(read+update). 복붙·marker 중복을 실제로 제거하되(churn 금지), R7 이 막 제거한 service→verifier→scanner→service 순환을 재생성하지 않는다.
전제 충족 — R7-1(scanner 4-sealed SPI) · R7-2(TypedNameGuard static) · R7-3(ObjectProvider 0) 완료. 3 fat service 의 typed-name 은 이미 TypedNameGuard.verify(static), 생성자 deps 에 scanner/verifier/ObjectProvider 0건. 따라서 본 분해는 순환 토대가 깨끗한 상태에서 시작한다.
§2. 설계 대안 (A 채택 / B · C 비채택)

대안 비교

안접근trade-off판정
(A)
선례 미러		 도메인당 독립 5분할: *LifecycleService(global LifecycleService impl) + *RegistrationService + *MarkerWriter + *IntegrityService(IsoIntegrityService 선례) + 잔류 *Service(read+update). OS/Iso 와 동형. 변경량 가장 큼(도메인당 신규 3클래스 + scanner/cascade re-point). 잔존중복 0. 책임축 1클래스=1책임으로 가장 명료. BIOS/BMC 는 *LifecycleController 가 이미 분리돼 service 와 1:1 정렬되는 이점 실재. 채택
(B)
최소 분해		 MarkerWriter + 내부 private helper 로 복붙만 제거, lifecycle/registration 은 fat service 잔류(BIOS R4-2 가 한 그대로). 변경량 최소·re-point 0 으로 안전하나 fat(~690줄)이 잔존. 캠페인 목적(fat 해소) 미달 — read+lifecycle+registration+verify 4책임 혼재. *LifecycleController 가 이미 분리됐는데 service 만 monolith 면 controller↔service 비대칭 영구화. 비채택
(C)
공통 추상		 3 도메인의 marker 4-step·등록 골격·lifecycle 을 AbstractLeaf* base/template-method 로 통합. 표면상 중복 최대 제거지만, marker attribute(Subprogram buildMarkerAttributes scope 분기 vs BIOS/BMC flat) · 등록 가드(단일축 vs 이중 scope) · entity 타입이 도메인별로 갈려 base 가 generic/hook 으로 비대. 한 도메인 변경이 타 도메인에 파급(over-coupling). 비채택

비채택 사유 (인과)

  • (B) 탈락 — 이 캠페인의 목적은 marker 복붙 제거가 아니라 fat-service 해소다. (B)는 복붙만 없애고 4책임 혼재를 남긴다. BIOS R4-2 가 lifecycle 을 남긴 것은 "그 시점 미완"이지 최적 종착이 아니며, OS/Iso 가 full split 한 사실이 더 성숙한 종착을 보여준다.
  • (C) 탈락 — CLAUDE.md §중복 금지의 단서("분리하려는 추상이 도메인을 가로질러 의미를 잃게 하면 분리하지 않는다 — over-abstraction 도 유지보수 비용")에 정면 위배. SubprogramService:474-486 buildMarkerAttributes 의 kind/scope/boardId 분기, BMC 의 boardModelIdMirror builder 처럼 attribute 조립이 도메인별로 실제로 갈라져 있다. BiosMarkerWriter 가 이미 per-domain @Component 인 사실이 "이 축은 per-domain 이 옳다"는 선결 판정.

공유 추상 판정 (sharedAbstractionDecision)

per-domain 으로 둔다. 복붙 제거는 도메인 내부 private helper(issueHashNudge / persistBundle) 차원에서만 하고, 도메인 간 공통 클래스로는 올리지 않는다. BIOS↔BMC 의 marker attribute 가 거의 동일해 "둘만 공유" 유혹이 있으나, 3-way 일관성 + BiosMarkerWriter per-domain 선례가 per-domain 을 지지한다. ResourceType enum 1개만 다른 거의 동일한 *MarkerWriter 3개를 두는 비용은, entity 타입(BoardBIOS/BoardBMC/Subprogram)이 달라 generic 화가 타입 파라미터를 강제하는 점에서 수용 가능한 잔여다 — "지금 비슷하니 묶자"를 거부하고 "갈라지는 시점(이미 갈라짐)에 분리"한 정상 상태.
🎬 라이브 — 분해 탐색 + 순환 가드 시뮬레이터

도메인을 고르고 typed-name 전략을 바꿔 보라. verifier 빈 주입을 고르면 R7 이 막 제거한 순환이 되살아나는 것을(bootRun 실패 예측) 직접 확인할 수 있다. 채택안은 static 가드 유지.

도메인 보기 typed-name 전략

분해 후 의존: registration→lifecycle / cascade→lifecycle / scanner→lifecycle 모두 부모·외부 → leaf-lifecycle 단방향. lifecycle 은 scanner/registration 을 역참조하지 않는다(IsoLifecycleService↔IsoMarkableScanner 선례 동일).

§4. 핵심 결정 (D1~D6, 비채택 병기)
#결정채택 / 비채택 사유
D1분해 형태(A) 선례 미러 5분할(Lifecycle/Registration/Marker/Integrity/잔류). 비채택 (B) 최소분해 = fat 잔존으로 캠페인 목적 미달, (C) 공통추상 = over-abstraction(§2).
D2BIOS/BMC lifecycle 1-arg 재성형채택: 1-arg + LifecycleService 구현 + assertBelongsToBoard(childId, boardId) 별도 forging 가드. IsoLifecycleService.assertBelongsToOs(isoId, osId)(L68) 를 controller 가 lifecycle 직전 호출하는 선례 정확 미러 — 부모는 bios.getBoardModel() 내부 resolve. 비채택: 2-arg 구체타입 유지(인터페이스 포기) — controller↔service 비대칭 영구화 + 다형 일관성 상실. (현재 2-arg toggleEnabled(boardId,biosId) 는 LifecycleService 1-arg 와 모순이라 둘 중 하나를 반드시 확정해야 컴파일.)
D3*MarkerWriter per-domainper-domain 3개(BIOS 기존 + BMC/Subprogram 신규). 공통 base 비채택 — attribute 조립이 도메인별로 갈림(§2 sharedAbstractionDecision).
D4typed-name / 순환TypedNameGuard static 유지, verifier 빈 미주입. *LifecycleService/*RegistrationService 어디에도 TypedNameVerifier·scanner·ObjectProvider 주입 금지 → service→verifier→scanner→service 재생성 차단(R7 산출 보존). scanner ObjectProvider 도입도 금지(보호할 순환 없음).
D5진행 순서R4-3(BIOS) → R5-3(BMC) → R6-3(Subprogram). BIOS 선두 = MarkerWriter 이미 추출돼 lifecycle/registration 분리에만 집중하는 깨끗한 레퍼런스. BIOS 가 1-arg 재성형 첫 적용처 → 그 결정이 R5-3/R6-3 템플릿을 일관시킨다. Subprogram 마지막 = 최대 fat + scope 분기 marker + 1-arg cascade 비대칭으로 변형 최대.
D6공유 가드 헬퍼잔류 *Service(update/find)와 *LifecycleService 가 함께 쓰는 requireActiveBoard/requireLive* → 복제 대신 *Guards static 추출 고려(BoardModelGuards 선례). 도메인별 판단(중복 2곳 미만이면 보류).
D7잔류 네이밍 + Integrity 추출잔류 = *Service 유지(개명 안 함) + *IntegrityService 신규 추출. 비채택: *QueryService 개명 — (1) 관리 도메인에 선례 부재: QueryService 는 provisioning GuestServerQueryService 의 CQRS read-only(↔GuestServerCommandService write) 쌍에만 존재. (2) 잔류가 update(쓰기)를 보유해 "Query"는 misnomer이며, BIOS 엔 단일 Command 짝이 없다(쓰기가 Lifecycle/Registration 으로 분산). (3) OSMetadataService(잔류 hub, create/update 보유)가 *Service 유지 선례. *IntegrityService 는 IsoIntegrityService 선례대로 findStatus/verify/verifyAndRecord 전담 → 잔류는 read+update 만 남아 더 얇아진다.
D2 가 본 CP1 의 가장 중요한 결정. adversarial critique 가 "implements LifecycleService + 2-arg 유지 = 컴파일 불가"를 잡았고, Iso 선례(assertBelongsToOs + 1-arg)가 해소책을 확정했다. 이 결정을 R4-3 에서 확정해야 R5-3/R6-3 이 드리프트 없이 복제된다. Subprogram 은 이미 1-arg 라 무관 — 3 도메인을 같은 "implements" 템플릿으로 뭉뚱그리지 않고 BIOS/BMC 만 재성형으로 분리 기술.

§6. 예외 계층 · 부산물 · 주의 · CLAUDE.md

예외 계층

신규 0. 전부 재사용 — TypedNameMismatchException(가드) · Illegal{Bios,Bmc,Subprogram}StateException · {...}NotFoundException(forging) · ChildLifecycleBlockedByParentException · *NudgeRequiredException · Duplicate*. 메서드 이동만, 예외 의미 불변.

부산물 / 주의

  • controller TypedNameVerifier 유지. *NudgeController 는 id→entity 조회가 필요해 빈 그대로(서비스 분리와 무관).
  • 전역 @Lazy 유지. SoftDeleteIntentService 의 @Lazy 는 본 분해 경로 미통과 — 건드리지 않음.
  • BIOS persistBundle helper 이득 과대평가 금지. writeSignedMarker 가 이미 1줄이라 3~5줄 절감 수준 — 그래도 복붙 자체는 제거(churn 아님).
  • Subprogram controller 분리 보류. *LifecycleController 부재(monolithic) — service-side 분리만 본 슬라이스, controller 정렬은 선택적 후속.
  • cascade @Order(Bios=10/Bmc=20/Sub=30) — re-point 대상은 @Order 가 아니라 클래스명+호출라인으로 적시(혼동 방지).
