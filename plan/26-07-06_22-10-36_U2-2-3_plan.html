§1. 현재 상태 — 연결의 전제가 전부 준비됨
전제상태
BIOS 세팅 템플릿 (U2-2-1/2)실영속 + board_model FK — guest 보드 selector 와 같은 boardModelId 축(FK 전환의 예고된 효용). 삭제는 현재 무조건 hard delete — 사용중 차단 없음.
guest 정의서 (U2-3)행 기반 영속(setting_process) — 조인 테이블이 스키마 재작업 없이 얹히는 구조(D1 의 예고). BasicSettingRequest 는 필드 0 스텁.
검증 자리 (U2-3-1)BasicSettingReferenceInspector no-op 선생성 — "U2-2-3 이 로직 주입" 이라고 명기된 그 자리.
복수 템플릿 규칙사용자 기확정: 정의서당 BIOS 템플릿은 보드 SPECIFIED ⇒ 1개(그 보드의 것), AUTO ⇒ N개(보드 중복 금지). 실행 시 감지 보드와 일치하는 1개 적용·불일치 skip.
§2. 요구사항 (확정) + 비목표
  1. 계약 실체화 — BasicSettingRequest.biosSettingTemplateIds: List<Long>(@NotEmpty — 단계를 추가했다면 템플릿 1+; 목록 내 중복 금지 @AssertTrue).
  2. 복수 템플릿 규칙 3층 검증 — Layer A(개수: SPECIFIED⇒1) + inspector(실존 404 / 템플릿 간 보드 중복 400 / SPECIFIED 보드 일치 400) — §4 D2·D3.
  3. 조인 테이블 파생 — setting_process_bios_template(process CASCADE / template RESTRICT) — payload 가 SSOT, 조인은 질의·무결성 전용 파생(U2-3 D1 논리의 확장, 기록됨).
  4. 사용중 삭제 차단 — 템플릿이 정의서에서 사용 중이면: UI 삭제 버튼 disabled+tooltip(1차) + 서버 409(안전망) + DB RESTRICT(레이스) — 3층 모두 같은 SSOT(BiosSettingTemplateUsageChecker).
  5. 폼/상세 — BASIC_SETTING 카드에 템플릿 체크 패널(보드 뱃지 표기, 보드 selector 연동 필터), 상세 카드에 템플릿 이름 표시(references 사상 확장), edit pre-fill.
비 목표 : 실행 시 적용/skip 로직(Stage 4 — 계약은 의도만 운반) · BASIC_SETTING 제조사별 다형 분기(U2-3-1 확정 — 차이가 실재로 확인될 때) · 템플릿 lifecycle(deprecated 등 — BIOS 템플릿은 lifecycle 비대상 설계 유지) · U2-4(OS 3차 검증)."
🎬 미리보기 — 보드 모드 × 템플릿 선택 규칙 + 삭제 차단 라이브 데모

보드 selector 모드를 바꾸고 템플릿을 클릭해 UI 1차 차단 규칙(SPECIFIED: 타보드 비활성+1개 제한 / AUTO: 보드당 1개)을 체험한다. 하단은 이 선택이 만드는 실행 시 의미(감지 보드 → 적용/skip)와 삭제 차단 판정이다.

보드 selector :
체험 포인트 — ① SPECIFIED 로 바꾸면 타보드 템플릿이 즉시 비활성 + 규칙 위반 선택은 자동 해제. ② AUTO 에서 같은 보드 2개째는 클릭이 차단된다(보드당 1개). ③ 선택된 템플릿은 목록 화면에서 삭제 버튼이 잠긴다 — 서버 409·DB RESTRICT 와 같은 판정(exists 쿼리).

Step 9 — 수동 DDL

CREATE TABLE setting_process_bios_template (
    setting_process_id        BIGINT NOT NULL,
    bios_setting_template_id  BIGINT NOT NULL,
    PRIMARY KEY (setting_process_id, bios_setting_template_id),
    CONSTRAINT fk_spbt_process  FOREIGN KEY (setting_process_id)
        REFERENCES setting_process (id) ON DELETE CASCADE,          -- 정의서/단계 삭제 시 참조 정리
    CONSTRAINT fk_spbt_template FOREIGN KEY (bios_setting_template_id)
        REFERENCES bios_setting_template (id)                        -- RESTRICT(기본): 사용중 템플릿 삭제 차단(레이스 안전망)
);
보드 purge 연쇄(기록 재확인): BoardModel purge → 템플릿 CASCADE 삭제 시도 → 사용 중이면 이 RESTRICT 가 연쇄를 중단시켜 보드 purge 도 실패(409) — silent 참조 소실 방지가 의도된 동작. management 보드 purge 화면의 안내 문구 정비는 §8 잔존.
§4. 도메인 모델 + 설계 대안 (SSOT)

D1 — 조인 테이블 파생 매핑: @ElementCollection 채택

① @ElementCollection ✓ 채택

  • SettingProcess 에 Set<Long> templateRefs(@CollectionTable) — 별도 엔티티 없이 파생 행 관리
  • 쓰기 시 payload 에서 단방향 생성(같은 트랜잭션 — process_type 파생 컬럼과 동일 지위), 읽기 재조립 미관여
  • D4 전체 교체(orphanRemoval) 시 컬렉션도 함께 소멸·재생성 — 관리 코드 0

② 조인 엔티티(SettingProcessBiosTemplate)

  • 행에 추가 속성(적용 순서 등)이 생기면 필요 — 지금은 순수 참조뿐이라 기반 코드만 늘림. 속성이 생기는 시점에 승격(갈라지는 시점). 기각

③ 파생 없이 JSON 역참조

  • U2-3 D1 에서 이미 기각한 안의 재림 — in-use 판정이 JSON_SEARCH, FK 무결성 상실. 기각

payload → 파생 추출의 다형화: 엔티티가 instanceof BasicSettingRequest 를 하지 않도록, AbstractProcessRequest 에 @JsonIgnore referencedBiosSettingTemplateIds()(기본 빈 목록)를 두고 BasicSettingRequest 만 override — wire 에 나가지 않는 파생 전용 다형 accessor(processType() 관용구).

D3 — inspector 시그니처에 검증 컨텍스트 도입 (SPI 1회 확장)

public record ProcessValidationContext(List<AbstractProcessRequest> processList) {
    public Optional<BasicUpdateRequest> firmwareStep() { ... }   // 형제 단계 조회 헬퍼
}
// SPI: void validateReferences(AbstractProcessRequest process, ProcessValidationContext context)
  • "SPECIFIED 보드 ↔ 템플릿 보드 일치"는 영속 조회(템플릿의 boardModelId)와 형제 단계(BASIC_UPDATE 의 selector)가 동시에 필요 — Layer A 도, 컨텍스트 없는 inspector 도 담을 수 없다. 컨텍스트 파라미터가 유일하게 층위 원칙을 깨지 않는 자리다.
  • 대안 비교: Command 에 별도 cross-validator 단계(기각 — 사다리의 재림, 지식이 inspector 밖으로 분산) / BASIC_SETTING payload 에 보드 모드 중복 저장(기각 — SSOT 위반, drift). 기존 inspector 7파일은 파라미터 추가만 받고 무시(기계적 변경) — U2-4 계열 검증도 이 컨텍스트를 쓸 수 있다.
  • describeDeprecatedReferences 는 컨텍스트 불요 — 시그니처 유지(비대칭은 필요의 반영).

D4 — 사용중 판정 SSOT: BiosSettingTemplateUsageChecker

// biossetting 도메인이 인터페이스 소유(자기 삭제 규칙의 SPI) — setting 이 구현 제공
public interface BiosSettingTemplateUsageChecker { boolean isInUse(Long templateId); }
// setting 측 구현: 조인 테이블 exists 네이티브 쿼리 (@Query — @ElementCollection 은 파생 쿼리 불가)
  • 의존 방향: setting → biossetting(계약 참조) 은 이미 존재. 역방향(biossetting 이 사용처를 앎)은 인터페이스만 — MarkableScanner SPI 관용구. 순환 없음(구현은 setting_process 조인 테이블만 조회).
  • UI(목록·상세 inUse 플래그)와 서버 409, DB RESTRICT 가 전부 이 판정 하나에서 나온다.

D5 — UI 선택 규칙 (패키지 그룹 패널 관용구 재사용)

  • BASIC_SETTING 카드: 템플릿 체크 패널(이름 + 보드 뱃지). 보드 selector 상태와 연동: SPECIFIED → 타보드 항목 비활성 + 1개 제한(두 번째 체크 시 기존 해제 or 차단 — 기존 해제(라디오 의미론) 채택: 1개 제한에서 교체가 잦은 UX), AUTO → 같은 보드 두 번째 항목 비활성.
  • 보드 selector 변경 시 재평가(규칙 위반 선택 자동 해제 + 안내), 카드 추가/제거·pre-fill 훅 — OS 동기화(syncOsSelectionLock)와 같은 배선 패턴.
  • 실행 의미 안내 힌트: "자동 감지 시 실행 시점에 감지된 보드와 일치하는 템플릿 1개가 적용되고 나머지는 건너뜁니다."

§8. 부산물 / 주의
  • 실행 의미론은 기록만: AUTO+N 의 적용/skip 판정("지연된 단언")은 Stage 4 — 계약·힌트 문구가 의도를 운반.
  • 보드 purge 연쇄 안내: RESTRICT 로 "사용중 템플릿 보유 보드 purge 실패(409)" 경로가 생김 — management purge 화면 문구 정비는 별도 잔존(§3 callout).
  • 편집 중 템플릿 삭제 레이스: pre-fill 된 템플릿이 사라진 경우 — 옵션 부재 항목은 pre-fill 시 안내 후 선택 해제(펌웨어 fallback 관용구), 최종은 inspector 404.
  • BASIC_SETTING 제조사 다형 미형성(U2-3-1 확정 유지). SPI 컨텍스트 도입으로 U2-3-1 plan D2 의 시그니처가 1회 개정됨 — U2-3-1 페이지에 기록 위임(CP4 보고 시).
  • 커밋 없음 — 사용자 지시 대기(누적).
