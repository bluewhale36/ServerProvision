2. 요구사항 (확정)

2.1 Notion 명세 (S5-9 본문)

숨김 파일이나 폴더를 회색에 가깝게 표시하여 시각적 차이를 만드는 작업.

2.2 채택안 비교 — A (FE-only) vs B (BE hidden 필드)

안 A FE-only — name prefix 검사

  • FE 가 e.name.startsWith('.') 으로 hidden 판별
  • BE 변경 0
  • 단점 : macOS 의 chflags hidden 또는 Windows 의 attrib +h 로 부여된 OS-level hidden flag 인식 못함 (S5-10 호환성 불완전)

안 B ✓ BE 가 hidden: boolean 노출

  • DirectoryListingResponse.Entry 에 hidden: boolean 1 필드 추가
  • BE 가 name.startsWith('.') || Files.isHidden(p) 결합 판별
  • FE 는 e.hidden 직접 사용
  • S5-10 의 OS-level hidden flag 부여와 자연 연계 — macOS / Windows 도 cover
  • 회귀 테스트 단순 — Entry record 시그니처 검증 1 줄 추가

2.3 시각 디자인 결정

  • color : var(--n-text-muted) — 현재 사이즈 텍스트와 같은 muted 톤
  • opacity : 0.55 — 너무 옅으면 글자가 안 보이고 너무 진하면 분리감 약함. 0.55 가 시각 분리 강도 vs 가독성 절충점
  • 아이콘 : 아이콘에도 opacity 적용 — entry 전체 li 에 opacity 부여
  • highlight 와 충돌 : hidden + highlight (예: .dvd.iso ?) 인 경우 거의 없으나, 발생 시 hidden 우선 (회색 처리)
  • 클릭 가능 : hidden entry 도 클릭 가능 — 사용자가 의도적 접근 시 막지 않음
  • hover 효과 : hidden 도 hover 시 background 변경 — 클릭 가능성 유지

2.4 추가 제약 (불가침)

  • DirectoryListingResponse 의 path / parentPath / truncated 필드 그대로
  • BE 의 entry 정렬 순서 그대로 (이름 알파벳)
  • filter 기능 없음 — hidden 도 응답 / 렌더 모두 포함 (단지 시각 차별만)
  • "숨김 항목 토글" 같은 사용자 컨트롤 추가 안 함 (본 슬라이스 외)

4. 도메인 모델

4.1 수정 — DirectoryListingResponse.Entry

public record Entry(
    String type,
    String name,
    Long size,
    boolean hidden    // ← S5-9 신규
) {
    public static Entry directory(String name, boolean hidden) {
        return new Entry("DIR", name, null, hidden);
    }
    public static Entry file(String name, long size, boolean hidden) {
        return new Entry("FILE", name, size, hidden);
    }
}

4.2 service 변경

// DirectoryBrowseService.browse() 내부
for (Path p : sorted) {
    if (entries.size() >= maxEntries) { truncated = true; break; }
    String name = p.getFileName().toString();
    // S5-9 — dot-file 관례 OR OS-level hidden flag (Windows/macOS) 통합 판별
    boolean hidden = name.startsWith(".") || isOsHidden(p);
    if (Files.isDirectory(p)) {
        entries.add(DirectoryListingResponse.Entry.directory(name, hidden));
    } else if (request.includeFiles()) {
        long size = -1L;
        try { size = Files.size(p); } catch (IOException ignore) {}
        entries.add(DirectoryListingResponse.Entry.file(name, size, hidden));
    }
}

private boolean isOsHidden(Path p) {
    try {
        return Files.isHidden(p);
    } catch (IOException e) {
        return false;
    }
}

7. 예외 / 회귀

7.1 신규 예외 — 없음

본 슬라이스는 응답 필드 +1 만. 예외 계층 변동 없음.

7.2 Files.isHidden 실패 시 fallback

IOException 발생 시 hidden=false 로 처리. 안전한 default — 사용자가 모든 항목을 일반 색상으로 본 뒤 직접 선택 가능. fail-loud 보다 fail-quiet 가 본 UI 의 의도와 부합.

7.3 응답 JSON 호환성

새 필드 hidden 추가는 기존 클라이언트 (path-browser.js) 가 무시 가능 — Jackson 의 unknown property 무시 정책 + JS 의 optional field 접근. 회귀 없음.

8. 부산물 · 주의

8.1 스코프 경계

  • "숨김 항목 토글" UI — 본 슬라이스 외 (별 슬라이스 후보)
  • hidden 필터링 (응답에서 제거) — 본 슬라이스 외. 사용자가 의도적 접근 가능성 보존
  • BE 의 hidden 판별 알고리즘 확장 (예: __ 으로 시작 등) — 본 슬라이스 외

8.2 S5-10 와의 연계

S5-10 에서 *.provision.json 에 OS-level hidden 속성을 부여하면, 본 슬라이스의 BE 판별 (name.startsWith('.') || Files.isHidden(p)) 가 양쪽 모두 cover. S5-9 → S5-10 순서 진행으로 S5-9 의 시각 효과가 S5-10 적용 후에도 일관 적용.

8.3 inline style 사용

path-browser.js 가 이미 li.style 의 inline 패턴을 사용하므로 본 슬라이스에서도 동일 패턴 유지. CSS 클래스 분리는 path-browser 의 향후 전체 refactor 슬라이스에서 처리 (본 슬라이스 외).

주의 CLAUDE.md 의 "인라인 스타일 금지" 원칙과 충돌 가능성. 그러나 path-browser.js 의 기존 패턴 (li.style.padding / cursor / background 등 inline 다수) 과 일관성을 위해 본 슬라이스에서는 inline 유지. 별 슬라이스에서 path-browser 전체 CSS 화 권장.
