§1. 현재 상태 (진단)

E1-0b 가 게스트↔서버 프로토콜 v1(dispatch 매트릭스 · 토큰 · 체크인 · 보고)을 열었다. 그러나 실행기 구현체가 0개라, 개시된 서버는 지금 전부 dispatch 6행 HOLD("phase DIAGNOSE_LINUX not implemented yet")를 받는다. 게스트는 아직 진단 리눅스로 부팅하지 못한다.

자산있음 (E1-0b 까지)비어 있음 (이 슬라이스가 채움)
dispatch 7행 (실행기 위임)SPI(ProvisioningPhaseExecutor) + registry + HOLD 폴백구현체 0 — DiagnoseLinuxExecutor 가 첫 실물
진단 리눅스E1-R 조사로 방식 확정(공식 netboot + apkovl + 부분 apk 저장소 + agent HTTP 다운로드)실물 파일이 하나도 없다 — 아티팩트 · apkovl · agent.sh · 조립 스크립트 전부 미제작
자산 서빙pxe.server.base-url 설정 키만 존재(빈 default)서빙 경로 없음 — 게스트가 커널/모듈/오버레이를 받을 HTTP 엔드포인트 부재
토큰 전달발급 · 저장 · 헤더 인증 완비게스트로 가는 운반로 없음 — 커널 인자 동봉은 체인로드 스크립트(이번) 몫
검증 환경T0(계약) + T1(하네스 curl)T2 없음 — 가상 게스트로 실부팅을 확인할 QEMU 랩 부재. E1-R 잔여 검증 V2·V4·V5·V6·V7·V9 미수행
물리 식별(DEC-33)—게스트 화면에 자기 식별 정보를 표시하는 배너 없음(UC-5)
핵심 진단 — 이 슬라이스는 코드 반, 실물 자산 반이다. 유스케이스 토론(Q-A)에서 "E1-1 먼저 + 인프라 실물 자산은 E1-1 안에서 제작" 으로 확정했다. 코드(실행기 · 서빙)는 얇고, 무게 중심은 자산 조립 파이프라인과 QEMU 랩이다 — E 로드맵 전체에서 처음으로 "게스트가 진짜 리눅스로 살아난다".
§2. 요구사항 (확정) + 비목표
  1. DiagnoseLinuxExecutor — 첫 ProvisioningPhaseExecutor 구현체(DIAGNOSE_LINUX). bootScript() 가 Alpine netboot 체인로드 iPXE 스크립트를 조립한다: kernel/initrd/modloop/apkovl/alpine_repo 절대 URL(pxe.server.base-url 기반) + provision_token=(E1-0b 예고분) + provision_base=(에이전트 콜백 주소) + EFI 용 initrd= 중복 명기(E1-R V 사실) + 로드 실패 시 재시도 폴백.
  2. 진단 리눅스 실물 자산 제작(E1-R 확정안 = 관리 자산 6종) — ① vmlinuz-lts ② initramfs-lts ③ modloop-lts(공식 netboot 그대로) ④ diag.apkovl.tar.gz(firstboot OpenRC + /etc/apk/world + 부분 저장소 공개키) ⑤ 부분 apk 저장소(dmidecode · ipmitool + 의존성, 자체 서명) ⑥ agent.sh(체크인 + 콘솔 식별 배너 DEC-33). 대용량 산출물은 git 비추적 — 조립 스크립트만 git 추적(재현 가능성이 자산).
  3. 앱 자산 서빙 — /api/pxe/v1/assets/** 를 pxe.assets.root 디렉토리에 매핑(Spring resource handler — §8 Q1). pxe.assets.root 미설정 환경에서는 실행기 빈이 생성되지 않아 기존 HOLD 동작이 그대로 유지된다(§8 Q2).
  4. agent.sh v1 (E1-1 임무 한정) — /proc/cmdline 에서 토큰 · base URL 파싱 → modprobe ipmi_devintf ipmi_si(E1-2 준비, 실패 무해) → 체크인(전이 신호) → DIAGNOSTIC_BOOTING step open/close 보고(부팅 마일스톤 원장) → 콘솔 배너 출력(서버명 · systemUUID 끝 12자 · IP — UC-5) → WAIT 폴링 루프. 수집 · 보고 본체는 E1-2.
  5. DIAGNOSTIC_BOOTING step 상수 추가 — "진단 리눅스가 실제로 부팅됐다" 는 원장 마일스톤(E1-0b plan §11 예고분). ProvisioningPhaseStep 에 DIAGNOSE_LINUX 소속으로 추가.
  6. 체크인 응답에 서버명 동봉 — AgentCheckinResponse 에 serverName 추가(배너의 유일한 서버측 정보 — §8 Q3).
  7. QEMU 랩(T2) git 추적 자산 — scripts/pxe-lab/: QEMU 내장 iPXE ROM + user-mode 네트워킹 내장 DHCP/TFTP 로 실 dhcpd 없이 부팅 체인 재현(§4). macOS 는 TCG 부팅 스모크까지(E1-R 포인트 2 확정). E1-R 잔여 검증 V2 · V4 · V5 · V6 · V7 · V9 를 이 랩에서 수행.
  8. docs/T3-checklist.md 생성(DEC-35 · Q-C 확정) — V3 · V8 + Redfish 전원 제어 · BootSourceOverride · IndicatorLED + E2 flash · E3 실 BMC 초기 적립.
비 목표 :
  • 수집 · 적재 — dmidecode 파싱 · GuestServerDetail 갱신 · 완주 판정(completedAt) · 운영자 수동 FAILED/재시도는 E1-2. agent.sh 의 수집 함수도 E1-2 에서 추가.
  • 접촉 시각 기록(DEC-32) — E1-2 로 배정(§8 Q4): 무보고 실패 대응(DEC-4 경과 표시)과 같은 관찰성 축이므로 그쪽에서 한 번에. 이번 슬라이스는 엔티티 · DDL 변경 0.
  • dhcpd · tftp 실배포 자산(dhcpd.conf · tftp 의 ipxe.efi 배치) — E1-I. 이번 랩은 QEMU 내장 DHCP/TFTP 로 대체해 실 인프라 없이 검증한다. 외부 httpd 서빙 전환 여지도 E1-I 에서 판단.
  • 회차 모델(DEC-29) · 재투입(DEC-30) · 중단(DEC-31) — 별도 슬라이스(로드맵 위치 U3 전후 권고, 토론 §5).
  • RAID CLI 동봉(OPEN-1) — E4 확정 전 조사. 진단 리눅스 자산 구조(부분 저장소)가 추후 패키지 추가를 수용하므로 지금 결정 불요.
🎬 미리보기 — 진단 리눅스 부팅 체인 시뮬레이터

이 데모는 E1-1 이후의 게스트 한살이를 재현한다. 「전원 인가」를 누른 뒤 「다음 →」로 부팅을 진행시키면 검은 콘솔에 게스트 화면(iPXE → Alpine → 배너)이 흐른다. 왼쪽 토글로 설정 · 상태 · 장애를 바꿔가며 dispatch 판정(6행 HOLD ↔ 7행 체인로드)과 로드 실패 폴백, 배너(UC-5)를 체험한다. 판정 로직은 서버 BootScriptDispatcher + DiagnoseLinuxExecutor 와 1:1 이다.

서버 설정 · 상태 :
게스트 :

# /boot 가 반환한 스크립트 (마지막 호출분)
개시 토글을 끈 채로 시작해보면 개시 대기(5행)에 머무는 것도 볼 수 있다.
드리프트 0 — simDispatch() = BootScriptDispatcher 판정 순서(회수·실패·종단 축은 E1-0b plan 시뮬레이터가 이미 다뤄 여기선 생략), 체인로드 스크립트 = DiagnoseLinuxExecutor.bootScript() 조립 결과, 부팅 진행 = Alpine diskless 순서(kernel→initramfs→modloop→apkovl→apk→agent), 배너 = agent.sh 출력 규격(DEC-33). 장애 주입 시 iPXE 가 재시도 폴백으로 떨어지는 것까지 서버 설계 그대로다.

§4. 도메인 모델 + 자산 (SSOT)

DiagnoseLinuxExecutor — 첫 SPI 구현체

// execution/engine — 활성 조건: pxe.assets.root 설정 시에만 빈 생성(§8 Q2).
// 미설정 환경 = registry 에 없음 = dispatch 6행 HOLD (기존 명시 동작 그대로 — 분기 추가 0).
@Component
@ConditionalOnProperty("pxe.assets.root")
public class DiagnoseLinuxExecutor implements ProvisioningPhaseExecutor {
    // pxe.server.base-url 이 비어 있으면 기동 fail-fast (@PostConstruct) —
    // 자산은 켰는데 콜백 주소가 없는 반쪽 설정을 조용히 허용하지 않는다.
    public ProvisioningPhase phase() { return ProvisioningPhase.DIAGNOSE_LINUX; }
    public String bootScript(GuestServer server, ProvisioningProgress progress, String rebootQuery) { … }
}

스크립트 조립은 실행기 소유의 text block(E1-0b Q4 채택안 답습) — IpxeScripts 에 넣지 않는다. 공용 유틸에 phase 별 스크립트가 쌓이면 그것이 곧 조건분기 증식이다. 공용 5종(대기 계열)은 그대로 IpxeScripts, phase 전용 스크립트는 그 phase 실행기가 갖는다.

#!ipxe
echo [provision] chainloading diagnose linux...
kernel {base}/api/pxe/v1/assets/vmlinuz-lts ip=dhcp modules=loop,squashfs console=tty0 \
  alpine_repo={base}/api/pxe/v1/assets/repo/main \
  modloop={base}/api/pxe/v1/assets/modloop-lts \
  apkovl={base}/api/pxe/v1/assets/diag.apkovl.tar.gz \
  provision_token={token} provision_base={base} initrd=initramfs-lts || goto failed
initrd {base}/api/pxe/v1/assets/initramfs-lts || goto failed
boot || goto failed
:failed
echo [provision] chainload failed. retrying...
sleep 30
chain /api/pxe/v1/boot?{rebootQuery}
  • || goto failed — 자산 404 · 네트워크 단절 시 iPXE 가 죽지 않고 기존 대기 루프와 같은 폴백으로 복귀(§7). UC-4 류 "network boot 는 됐는데 자산이 죽은" 창을 관찰 가능한 재시도로 바꾼다.
  • initrd=initramfs-lts 커널 인자 중복 명기 — EFI 부팅 필수(E1-R §1 확인 사실).
  • 절대 URL 인 이유: modloop= 등은 iPXE 가 아니라 Alpine init 이 소비하므로 스크립트 URL 기준 상대 해석이 없다. 유일한 주소 원천 = pxe.server.base-url(기존 키 재사용 — 신규 키는 pxe.assets.root 하나뿐).

§8. 예상 부산물 / 주의 / 결정 사항

잔존 책임 · 주의 (정직 기록)

  • 부팅까지만 열린다 — 진단 리눅스가 떠서 체크인 · DIAGNOSTIC_BOOTING 까지 기록되지만, 수집 · 완주는 여전히 비어 있다(E1-2). 게스트는 배너를 띄운 채 WAIT 폴링에 머문다 — 이것이 이 슬라이스의 정상 종착이다.
  • 실기(T3) 미검증 잔존 — QEMU 로 못 보는 것: V3(실보드 ipmi 모듈) · V8(dmidecode placeholder) + 실 NIC 드라이버 · 실 iPXE 펌웨어 거동. 전부 docs/T3-checklist.md 에 적립(Q-C 확정 이행).
  • 자산 버전 고정 — build-assets.sh 에 Alpine 버전(v3.22)을 변수로 박는다. dl-cdn 이 구버전을 내리면 다운로드가 깨질 수 있음 — 조립 산출물을 로컬 보존하는 이유이기도 하다(재조립은 인터넷 필요, 서빙은 불요).
  • Docker 전제 — 부분 apk 저장소 조립(②단계)은 alpine 컨테이너가 필요하다(macOS 에 apk 툴체인 없음). Docker 부재 환경은 ①·③만 수행하고 ②는 스킵 가능하게 스크립트를 단계 분리한다.
  • 모의 하네스와 실게스트의 공존 — diagnose-cycle.sh(T1)는 계속 유효(프로토콜 회귀용). QEMU 게스트(T2)는 같은 서버에 실 HTTP 로 붙는다 — 하네스 전용 규약을 새로 만들지 않는다.
