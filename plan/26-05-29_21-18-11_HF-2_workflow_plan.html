1. 배경 — 관찰된 모순

증상 — OS 버전 9.5 가 비활성화(disabled) 상태인데, 그 하위 ISO 27(Rocky 9.5 dvd iso)이 활성화(enabled). 부모가 꺼져 있는데 자식이 켜져 있는 lifecycle 논리 모순. (다른 OS 버전 8.10 / 10.1 은 활성.)

직전 HF-1 작업에서 ISO 27 을 self-heal restore 한 직후 발견. "이전에 부모-자식 가드로 다뤘던 문제 같은데 어쩌다 재발했는가" 가 핵심 질문. 이미 코드에는 부모-자식 enabled 가드가 존재하므로(아래 §3), 그 가드를 우회한 경로가 있다는 의미.

2. 답해야 할 4 질문 (사용자 요구)

Q1
어떤 유스케이스에서 발생하는가 — 어떤 사용자 액션 순서가 이 모순 상태를 만드는가 (재현 시나리오 확정)
Q2
정확히 어느 지점에서 문제가 되었는가 — 어느 메서드·라인이 부모-자식 enabled 정합을 깨뜨리는가 (file:line)
Q3
어느 지점을 수정해야 하는가 — 근본 수정 위치 + 설계 (임시 처방 아닌)
Q4
다른 자원 도메인에서 중복 발생할 수 있는가 — BoardModel→BIOS/BMC/Driver, Subprogram 등 동형 부모-자식 구조의 같은 결함 여부

3. 초기 가설 (스카우트 결과 — Workflow 가 검증·확정)

workflow_plan 작성 전 최소 스카우트로 확보한 코드 사실 2개 + 그로부터의 가설. 아직 미확정 — Workflow 의 적대적 검증 대상.

확보한 코드 사실

사실 ① — LifecycleEntity.restore() (line 108-113) 은 is_deleted = false 로만 복원하고 is_enabled 는 건드리지 않는다 (soft-delete 시점의 enabled 값 보존). softDelete() 도 is_enabled 무변경. 
public void restore() {
    if (!this.isDeleted) { ... }
    this.isDeleted = false;   // is_enabled 는 그대로 — soft-delete 전 값 보존
}
사실 ② — OSMetadataLifecycleService.toggleEnabled 는 OS disable 시 자식 ISO 를 cascade 하나 !iso.isDeleted() 필터로 soft-deleted 자식은 cascade 에서 제외한다. 또 ISO restore 의 부모 가드 (IsoLifecycleService.restore) 는 parent.isDeleted() 만 확인하고 parent.isEnabled() 는 확인하지 않는다. 
parent.getIsos().stream()
    .filter(iso -> !iso.isDeleted() && !iso.isDeprecated())   // ← soft-deleted 자식 제외
    .filter(iso -> iso.isEnabled() != target)
    .forEach(ISO::toggleEnabled);

가설 (두 갈래 — 어느 쪽이 실제 경로인지 Workflow 가 확정)

가설 A cascade 제외 + restore 보존
OS disable 시점에 ISO 27 이 이미 soft-deleted(trash) 였다면 → !iso.isDeleted() 필터로 cascade disable 제외 → ISO 27 의 is_enabled=true 보존. 이후 ISO 27 restore → is_deleted=false 로만 복원, is_enabled=true 유지 → 부모 disabled · 자식 enabled 모순.
가설 B restore 부모 enabled 가드 누락
IsoLifecycleService.restore 가 부모 isDeleted 만 가드하고 isEnabled 는 확인 안 함 → 부모가 disabled 인 상태에서 자식 단독 restore 가 enabled 자식을 되살림 → 모순. (HF-1 self-heal 경로도 동일하게 부모 enabled 미확인.)
핵심 의문 — OS 9.5 는 어떻게 disabled 가 됐는가? (a) 사용자가 OS 를 toggle disable / (b) 삭제→복구 과정에서 disabled 로 / (c) 다른 경로. 이 시점·순서가 Q1(유스케이스) 의 핵심이며 가설 A/B 중 어느 쪽인지를 가른다.

4. Workflow 설계 (phase 구조)

4 질문(Q1~Q4)을 4 phase 에 매핑한 pipeline. 각 phase 는 코드 Read/Grep 기반 사실 확인 + 적대적 검증.

PHASE 1 · INVESTIGATE

유스케이스 재구성 + 정확한 지점 (Q1 · Q2) — 병렬 4 agent

  • A. enabled cascade 경로 — OS/부모 toggle disable 시 자식 cascade 의 정확한 조건(soft-deleted/deprecated 제외 필터)이 어떤 상태의 자식을 누락시키는지. OSMetadataLifecycleService.toggleEnabled + deprecate/undeprecate cascade 전수.
  • B. restore 의 enabled 처리 — LifecycleEntity.restore() + restoreFromTrash self-heal + IsoLifecycleService.restore 부모 가드. is_enabled 보존이 부모 상태와 어떻게 어긋나는지.
  • C. OS 9.5 가 disabled 가 된 경로 — soft-delete→restore 흐름에서 OS 의 is_enabled 변화 추적. 삭제/복구/토글 중 어디서 disabled 가 됐는지 재구성.
  • D. 부모-자식 enabled invariant 의 현 가드 지도 — 기존에 "이미 다뤘다"는 가드(IsoLifecycleService.toggleEnabled 의 enable 시 부모 가드 등)가 무엇을 커버하고 무엇을 빠뜨리는지 전수.

schema: { area, mechanism, trigger_state, evidence(file:line), confidence }

PHASE 2 · VERIFY

가설 A/B 적대적 검증 — 각 진단을 코드로 반증 시도

  • 가설 A(cascade 제외) 와 가설 B(restore 부모 가드 누락) 중 실제 재현 경로를 코드로 확정. 둘 다일 수도.
  • 재현 시나리오를 단계별 entity 상태(is_enabled/is_deleted)로 추적해 모순 발생 라인 pinpoint.
  • "이전에 다뤘다"는 가드가 정확히 어떤 케이스만 막고 본 케이스를 왜 못 막았는지 (가드의 구멍).

schema: { hypothesis, confirmed, actual_path, gap_in_existing_guard, evidence }

PHASE 3 · DESIGN

근본 수정 설계 (Q3) — 후보 + judge

수정 후보 (Workflow 가 평가·선택):

  • 후보 1 — restore 시 부모 enabled 정합: 자식 restore 시 부모가 disabled 면 자식도 disabled 로 복원 (또는 enable 차단). 부모 상태에 자식 맞춤.
  • 후보 2 — cascade 필터 보정: OS toggle disable 의 cascade 가 soft-deleted 자식도 포함하거나, restore 시점에 부모 상태 재적용.
  • 후보 3 — enabled invariant 를 entity/도메인 불변식으로 승격: 부모-자식 enabled 정합을 한 곳에서 강제 (CLAUDE.md 다형성/framework primitive). 분기 확장 회피.
  • 후보 4 — 조회 시점 정합 (derived): 자식 effective-enabled 를 부모 enabled AND 자식 enabled 로 파생 표현 (저장 상태가 아닌 계산).

judge 축: correctness(모순 근절) / 단순성(CLAUDE.md) / 횡단 적용성 / 기존 가드와의 일관성. afterCommit 류 과한 인프라 배제.

schema: { approach, fix_point, invariant_enforcement, cross_domain, tradeoffs, claude_md }

PHASE 4 · CROSS-DOMAIN

다른 도메인 횡단 확인 (Q4)

  • BoardModel→BIOS/BMC/DriverPackage, Subprogram 등 부모-자식 1:N 도메인 전수. 동일한 "부모 disable cascade + 자식 restore enabled 보존" 결함 여부.
  • 각 도메인의 toggleEnabled cascade 필터 + restore 부모 가드 비교 (HF-1 의 self-invocation 처럼 도메인별 편차 가능).
  • 채택 수정이 공통 인프라(LifecycleEntity / LifecycleService / restoreFromTrash) 1곳으로 4 도메인 동시 해결 가능한지.

schema: { domain, has_same_defect, cascade_filter, restore_guard, evidence }

Workflow 산출 (종합)

Phase 1~4 종합 → 4 질문 각각의 확정 답 + 채택 수정 설계 + CP 단위 구현 계획 + 횡단 영향. (HF-1 처럼 구현은 별도 CP 로, 본 Workflow 는 진단+설계까지.)
