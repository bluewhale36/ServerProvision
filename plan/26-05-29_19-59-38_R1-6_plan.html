§2. 슬라이스 요구사항 + 비목표

본 슬라이스는 OSMetadata 부모 도메인의 nudge 영속화 응집. 10-에이전트 5관점+4후보 토론 합의 — C1 (nudge 영속화) 만 분리, C2 / C3 / C4 는 KEEP (C3 는 C1 의 공유 절단면이라 동반 처리).

  1. 책임 단방향 정렬 (C1) — completePendingOSMetadataFromNudge / purgeOSMetadataForNudge / buildOSMetadataNudgePayload / persistNewOSMetadata 4 메서드를 OSMetadataNudgeService 로 흡수. nudge 서비스가 부모 서비스를 역호출하던 비대칭 → nudge 서비스가 OSMetadataRepository + NudgeRegistry 직접 주입으로 self-contained.
  2. nudgeRegistry 의존 제거 — OSMetadataService 에서 buildOSMetadataNudgePayload 전용 의존을 완전히 끊는다.
  3. create() 재배선 (C3 공유 절단면) — create() happy-path (line 142) 와 nudge proceed (line 203) 두 곳이 persistNewOSMetadata 를 공유한다. create() 의 정상 등록도 흡수 service 의 persist 를 위임하도록 재배선해 persist 로직 중복(불가침) 0 을 보장한다. create() 의 nudge 발화도 흡수 service 의 payload 빌더를 위임.
  4. create() 의 3분기 invariant 판단은 잔류 — line 127~142 의 LifecycleStage 3분기 (ACTIVE 단독 → Duplicate / DEPRECATED|SOFT_DELETED → Nudge / row 0 → save) 는 "등록 명령의 본질" 이므로 분해하지 않는다. create() 는 "invariant 판단 → 발화 위임 / 정상 등록 위임" 으로 얇아질 뿐.
  5. 외부 endpoint 동일 — Controller URL / HTTP method / response / nudge modal 흐름 100% 동일.
  6. stale javadoc 청산 — C1 이동으로 해소되는 잔류 표기 정정 (§8 D-5).

3.2 create() 가 흡수 service 로 정방향 위임 (C3 공유 절단면)

OSMetadataService.create() 의 3분기 invariant 판단은 잔류하되, 발화·persist 는 위임한다. OSMetadataService 가 osMetadataNudgeService 를 주입 (부모 → nudge 정방향, cycle 없음 — nudge 서비스는 osMetadataService 를 더 이상 모름).

// OSMetadataService (R1-6 후) — invariant 판단만, 발화·persist 위임
@Transactional
public Long create(OSMetadataCreateRequest request) {
    Map<LifecycleStage, List<OSMetadata>> byStage = ... ;   // R1-1 1 round-trip (잔류)
    boolean hasActive = ...; boolean hasDeprecated = ...; boolean hasSoftDeleted = ...;

    if (hasActive && !hasDeprecated && !hasSoftDeleted)
        throw new DuplicateOSMetadataException(request.osName(), request.osVersion());   // 잔류 (invariant)
    if (hasDeprecated || hasSoftDeleted) {
        List<OSMetadata> candidates = ... ;                  // 후보 합성 (잔류)
        throw new OSMetadataNudgeRequiredException(
                osMetadataNudgeService.buildNudgePayload(request, candidates));   // 발화 위임
    }
    return osMetadataNudgeService.persistNew(
            request.osName(), request.osVersion(), request.description());        // 정상 등록 위임

위임 방향 검증 — R1-6 후 의존은 OSMetadataService → OSMetadataNudgeService (정방향) 단 한 줄. 반대로 OSMetadataNudgeService → OSMetadataService 역호출은 0 (Repository 직접). 따라서 cycle 없음. 만약 create() 의 위임이 부담스러우면 대안 — persistNew/buildNudgePayload 를 흡수 service 의 public 으로 두고 create() 가 호출 (본 안). 이는 ISO R1-4-2 가 OSNudgeService → IsoRegistrationService.completePendingFromNudge 를 단방향으로 만든 것과 동형이되, OSMetadata 는 부모가 자식 nudge 서비스를 호출하는 형태라 방향이 한 단계 더 명시적.
