1. 현재 상태 (실측 근거)

2026-07-19 전수 점검에서 ISO 등록·휴지통 이동의 세 가지 결함이 확인됐다. 셋 모두 정상 브라우저 흐름에서는 노출되지 않는다 — 화면 JavaScript 가 경로를 보정하고(디렉토리 선택 시 파일명 자동 추가), Thymeleaf th:field 체크박스가 hidden 마커 _allowCreateDirectory 를 항상 동봉하기 때문이다. 즉 발동 경로는 curl 등 direct POST 와 (F-7 은) 휴지통 목록의 표기다.

#층현재 동작원인 (코드 실측 — HEAD 61dc383)
F-4서버POST /management/os/{id}/iso 에 isoPath 를 기존 디렉토리로 주고 파일을 첨부하면 500IsoRegistrationService.prepare 에 "대상이 디렉토리" 가드가 없다. DuplicateFilenameException 조건이 Files.exists(target) && !Files.isDirectory(target) 라 디렉토리는 그냥 통과 → storeUploadedFile 의 Files.copy(in, target) 가 IOException(디렉토리에 복사 불가) → ISOFileStorageException(@ResponseStatus 없음) → advice fallback 500
F-4b서버(intent)XHR(XMLHttpRequest) 사전 핸드셰이크 upload-intent 도 디렉토리 경로에 토큰을 발급 — 바이트 전송이 끝난 뒤에야 위 500 을 만난다IsoUploadIntentService.validateFilesystem 도 같은 exists && !isDirectory 조건이라 디렉토리를 통과시킨다. intent 의 존재 이유(전송 전 거절)가 이 케이스에서 무력
F-5서버+뷰hidden 마커 _allowCreateDirectory 없이 POST 하면 오류 무표시 silent 200record 생성자 바인딩에서 primitive boolean allowCreateDirectory 값 부재 → bindingResult.hasErrors() → iso-new 재렌더. 그런데 템플릿의 서버 렌더 오류 슬롯은 isoPath 하나뿐(th:errors="*{isoPath}") — allowCreateDirectory 필드 오류는 어디에도 표시되지 않는다
F-7표기휴지통 이동 시 디렉토리 이름의 마지막 점을 확장자로 취급 — 실측: "v1.10" → "v1_260719-064732-312_26f378f3.10", "v2.3" → "v2_…_e2ea7c67.3"global/trash/TrashService.generateTrashedFileName(109행) 이 original.lastIndexOf('.') 로 무조건 base/ext 분리 — 대상이 파일인지 디렉토리인지 보지 않는다. 기능 영향은 없음(복원 정상), 운영자 혼란만 발생

보조 사실: 필드 직결 400 예외 계층 FieldBoundBadRequestException(abstract, fieldName() 보유, @ResponseStatus(400))이 이미 있고, XHR 채널은 ApiExceptionHandler.handleFieldBoundBadRequest 가 400 JSON + fieldErrors 로 변환한다. SSR(Server-Side Rendering — Thymeleaf 폼 흐름)에서 필드 오류를 폼 재렌더로 흡수하는 선례는 IsoLifecycleController.updateIso 의 bindingResult.rejectValue("isoPath", …) catch 다.

2. 요구사항
  1. F-4 — 입력성 오류는 400 — prepare 단계에서 resolved 경로가 기존 디렉토리면 isoPath 필드에 바인딩되는 400 예외로 거절한다. 메시지: "경로가 디렉토리입니다. 저장할 파일명까지 포함한 경로를 입력하세요." SSR 폼 재렌더 시 isoPath 슬롯에 오류가 보이고 응답 상태는 400. XHR(upload / upload-intent)은 기존 advice 로 400 JSON + fieldErrors.
  2. F-4b — intent 도 동일 방어 — 핸드셰이크 시점에 디렉토리 경로를 거절해 무의미한 바이트 전송을 차단한다(확인 결과 필요: §1 F-4b).
  3. F-5 — silent 실패 금지 — iso-new.html 폼 상단에 서버 렌더 글로벌 오류 배너를 추가한다. 바인딩 오류 중 화면에 전용 슬롯이 없는 것(global 오류 + isoPath 외 필드)을 Thymeleaf #fields 로 표시. 기존 알럿 CSS 클래스(n-alert n-alert-danger) 재사용, 인라인 스타일 금지.
  4. F-5 전수 확인 — 같은 checkbox 마커 패턴(record + primitive boolean + hidden _marker)을 쓰는 다른 폼의 적용 여부를 확인해 기록한다(§4 D3 에 결론).
  5. F-7 — 디렉토리 리네임 — 휴지통 이동 리네임에서 대상이 디렉토리면 확장자 분리 없이 이름_<ts>_<hash> 전체 접미사. 파일은 기존 동작(확장자 보존) 유지.
  6. 테스트 — 단위(prepare 디렉토리 거절 · 리네임 파일/디렉토리) + 통합 @WebMvcTest(① 디렉토리 POST → 400 + isoPath 필드 메시지 ② 마커 누락 POST → 200 재렌더 + 배너 문구) + ./gradlew test 전체 green.
금지 경계 — 커밋 금지 · execution/·provisioning/ 수정 금지 · CLAUDE.md/AGENTS.md 수정 금지 · 업로드 intent 프로토콜(핸드셰이크 계약) 자체 변경 금지. 본 슬라이스는 intent 의 검증 규칙 1건 추가일 뿐 요청/응답 계약은 그대로다.

3. URL / 데이터 흐름 
[F-4 · SSR]  POST /management/os/{osId}/iso   (multipart form, IsoUploadController.addIso)
   bindingResult.hasErrors() → 재렌더 (기존 유지 — F-5 배너가 이제 슬롯 없는 오류를 표시)
   isoRegistrationService.prepare(osId, request, file)
     ├─ resolvedPath = IsoPathResolver.resolve(…)                    // 기존
     ├─ ensureParentDirectory / assertWritablePath                   // 기존
     ├─ if (Files.isDirectory(target))                               // ★ 신설 가드
     │      throw IsoPathIsDirectoryException(resolvedPath)          //   FieldBoundBadRequestException(400, field=isoPath)
     └─ … (중복/사이드카/저장 — 기존)
   controller catch (FieldBoundBadRequestException ex)               // ★ 신설 — updateIso rejectValue 선례
     → bindingResult.rejectValue(ex.fieldName(), "fieldBound", ex.getMessage())
     → response.setStatus(400) → iso-new 재렌더 (isoPath 슬롯에 오류 표시)

[F-4 · XHR]  POST /management/os/{osId}/iso/upload  → 같은 prepare 가드
   → ApiExceptionHandler.handleFieldBoundBadRequest → 400 JSON {message, fieldErrors:[{field:"isoPath"}]}  // advice 무변경

[F-4b · intent]  POST /management/os/{osId}/iso/upload-intent
   IsoUploadIntentService.issue → validateFilesystem
     ├─ if (Files.isDirectory(target)) throw IsoPathIsDirectoryException  // ★ 신설 (같은 예외 재사용)
     └─ 기존 DuplicateFilename / DirectoryMissing / 디스크 공간 검사
   → 400 JSON (같은 advice) — 바이트 전송 전 거절 (intent 계약 자체는 무변경)

[F-5 · 뷰]  templates/management/os/iso-new.html — 폼 상단
   서버 렌더 배너 (기존 JS 배너 슬롯과 별개, 같은 n-alert n-alert-danger 클래스)
   #fields.detailedErrors() 중 { global || fieldName != 'isoPath' } 만 나열
   → 마커 누락 재렌더(200)에서 배너 표시. isoPath 오류는 기존 인라인 슬롯이 담당(중복 표시 없음)

[F-7]  TrashService.moveToTrash → generateTrashedFileName(resourcePath)
   if (Files.isDirectory(resourcePath)) return name + "_" + ts + "_" + uuid8   // ★ 확장자 분리 없음
   else 기존 base/ext 분리 유지
4. 도메인 모델 — 설계 결정 (대안 비교 포함)

D1. F-4 예외의 형태와 위치

안내용판정
(a) 채택신규 IsoPathIsDirectoryException extends FieldBoundBadRequestException(management/os/exception, fieldName="isoPath"). prepare 의 assertWritablePath 직후 Files.isDirectory(target) 가드 1곳 + intent validateFilesystem 1곳에서 던진다.XHR 은 기존 handleFieldBoundBadRequest 가 자동 매핑(advice 무변경 — hierarchy 흡수), SSR 은 fieldName 다형으로 rejectValue. 파일 첨부/미첨부(in-place 등록) 양쪽 모두 이른 시점에 명확한 사유로 거절 — 미첨부 시 기존의 오해성 메시지("ISO 파일이 해당 경로에 존재하지 않습니다" 409)도 이 케이스에선 대체된다.
(b) 탈락기존 InvalidIsoPathException(ConflictException, 409) 재사용입력 형식 오류인데 409(충돌) 는 의미 오염. fieldName 이 없어 SSR 폼 슬롯/XHR fieldErrors 에 바인딩 불가 — 요구(필드 오류 표시) 미충족.
(c) 탈락storeUploadedFile 의 IOException 메시지만 개선(500 유지)과업 요구가 400. 게다가 파일 저장을 시도한 뒤의 사후 판정이라 원인(디렉토리)을 IOException 메시지 파싱으로 추정해야 한다 — 사전 판정이 정확하고 싸다.

D2. SSR 재렌더 처리 방식

안내용판정
(a) 채택addIso 가 FieldBoundBadRequestException(추상 family) 를 catch → rejectValue(ex.fieldName(), …) + HttpServletResponse 400 + 재렌더. 재렌더 조립 3곳(기존 hasErrors / 신설 catch)은 private helper 로 추출(중복 금지).updateIso 의 기존 선례와 동형. sub-class 별 분기 없음 — 예외가 자기 필드명을 들고 오는 다형 매핑이라 향후 다른 필드 직결 예외가 생겨도 catch 는 안 자란다. 400 상태는 direct POST 전용 경로라 브라우저 UX 영향 없음.
(b) 탈락advice(WebExceptionHandler)에서 처리 — error.html 400 페이지지금도 되는 동작(handleDomain 이 @ResponseStatus 400 매핑)이지만 "폼 재렌더 시 isoPath 필드 오류 표시" 요구 미충족 — advice 는 원 폼의 Model/BindingResult 컨텍스트가 없어 재렌더가 불가(updateIso 주석의 framework 한계와 동일).
(c) 탈락prepare 호출 전에 controller 가 직접 Files.isDirectory 검사검증 로직이 controller/service 두 층으로 분산 — XHR upload 경로(같은 prepare 사용)와 SSOT 가 갈라진다. 판정은 service 1곳, 표현만 채널별.

D3. F-5 배너 범위 — checkbox 마커 패턴 전수 확인 결과

D4. F-5 배너 구현

안내용판정
(a) 채택#fields.detailedErrors() 를 { global || fieldName != 'isoPath' } 로 필터한 목록만 배너에 나열(필드명 + 메시지). 배너 고정 헤드라인 "입력값을 처리하지 못한 항목이 있습니다." + 기존 n-alert n-alert-danger 재사용.슬롯이 있는 isoPath 오류는 인라인만 표시(중복 없음) — 정상 검증 UX 불변. 제외 목록('isoPath')은 슬롯 자체와 같은 템플릿 파일 안에 있어 SSOT-인접(슬롯을 추가/삭제하는 수정과 같은 diff 에서 보인다). 향후 슬롯 없는 새 필드는 자동으로 배너에 잡힌다 — silent 재발 불가.
(b) 탈락#fields.allErrors() 전체 요약 배너isoPath 오류 시 배너+인라인 이중 표시 — 정상 흐름(경로 공백 등)의 기존 화면이 바뀐다. 핫픽스가 정상 UX 를 건드릴 이유 없음.
(c) 탈락allowCreateDirectory 전용 오류 슬롯 추가필드 하나를 지목하는 방식 — 다음에 슬롯 없는 필드가 추가되면 같은 silent 사고 재발(분기 legacy 확장과 같은 냄새).

D5. F-7 리네임 분기

안내용판정
(a) 채택generateTrashedFileName 내부에서 Files.isDirectory(resourcePath) 로 분기 — 디렉토리는 전체 이름 + _ts_uuid8, 파일은 기존 분리 유지. suffix 조립은 한 번만 계산해 두 분기가 공유(중복 금지).호출 시점(moveToTrash 의 mv 직전)에 원본이 실재하므로 파일시스템 실체 기반 판정이 정확. 호출부/시그니처 무변경.
(b) 탈락moveToTrash 가 boolean isDirectory 를 인자로 전달호출자가 1곳뿐인데 시그니처만 오염(무명 boolean 인자 — CLAUDE.md 가독성 신호). 판정 재료(경로)가 이미 메서드 안에 있다.
(c) 탈락이름 휴리스틱("마지막 세그먼트가 숫자면 확장자 아님")"v2.iso" 라는 디렉토리·"file.10" 이라는 파일이 얼마든지 가능 — 이름으로는 오판정. 실체 판정이 유일하게 정확.

Edge: 존재하지 않는 경로(ghost)는 isDirectory=false → 기존(파일) 규칙으로 이름을 만들지만 어차피 직후 moveWithRetry 가 실패하므로 행동 변화 없음.

8. 부산물 / 주의
  • scope 경계 — bios/bmc/subprogram 의 등록 경로에도 이론상 "디렉토리 경로" 유사 입력이 가능하지만 업로드 모델이 다르고(번들/디렉토리 등록이 정상 기능) 이번 발견(F-4)은 ISO 한정 — 타 도메인 가드는 만들지 않는다. updateIso 의 재렌더 200(400 아님)도 기존 동작 유지 — 이번 요구 밖.
  • intent 프로토콜 불변 — upload-intent 의 요청/응답 계약(토큰·nudge·hash 단계)은 그대로. validateFilesystem 안 검증 규칙 1건 추가일 뿐이다.
  • F-5 배너의 표시 문언 — allowCreateDirectory 바인딩 실패의 기본 메시지는 Spring 이 만드는 영문(typeMismatch)일 수 있다. 배너는 한국어 헤드라인 + 필드명/원문 나열로 표면화가 목적 — 메시지 한글화(MessageSource 도입)는 프로젝트에 messages.properties 자체가 없어 본 핫픽스 범위 밖(후속 후보, §11).
  • TrashService 리네임 변경은 이동 시점 이름에만 영향 — 기존 휴지통에 이미 잘못 리네임된 항목은 소급 정정하지 않는다(복원 정상이므로 데이터 수술 불요).
  • 동반 수정 없음 — CLAUDE.md/AGENTS.md/스키마/Notion 구조 무변경. execution/·provisioning/ 미접촉.
