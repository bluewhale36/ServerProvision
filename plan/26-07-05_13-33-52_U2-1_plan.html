§2. 요구사항 (확정) + 비목표
  1. 화면 이식 — 목록/작성/상세/수정 4 페이지 + OS 계열 fragment 4 장을 /provisioning/setting 아래 renew 디자인(n-* 클래스, 기존 CSS 재사용 + per-page setting.css)으로 재작성. 인라인 스타일 금지. navbar placeholder 활성화(activePage='setting').
  2. Request 계약 이식 — 2단 다형(type → osFamily) @JsonTypeInfo/@JsonSubTypes/@JsonCreator 구조 유지(Jackson 3 backward-compat annotation). Bean Validation(@Valid 재귀) 포함. 판별자 BASIC_UPDATES → BASIC_UPDATE 정정(renew DB 기존 데이터 0 → 호환 부담 없음).
  3. Response 계약 정의 — SettingSummaryResponse/SettingDetailResponse(형제 GuestServer 명명 관례) + SettingSaveResponse(id/name/status) + 폼 선택지 SettingBoardOptionResponse/SettingOSOptionResponse + PartitionPresetResponse.
  4. Controller 골격 — SSR SettingController(GET 4) + XHR SettingRestController(POST 생성 201+Location · PUT 수정 200 · GET default-partitions 200). 모든 응답은 스텁 데이터.
  5. Service 시그니처 + 스텁 — SettingQueryService/SettingCommandService 인터페이스 확정 + 임시 InMemorySettingStub 단일 구현(두 인터페이스 동시 구현, 더미 정의서 2건 시드) — U2-2 가 JPA 구현으로 대체.
  6. 계약측 타입 정착 — SettingProcessType(4 상수, 표시명+판별자 문자열 보유) 신설 + FileSystem·SizeUnit·ServiceAction enum 을 provisioning/setting/enums/ 로, ServiceDirective → ServiceDirectiveRequest record 로 이식(Primitive Obsession 금지 유지, domain 누수 정정).
  7. 예외/advice — SettingNotFoundException(404, NotFoundException 하위) 신설 + ApiExceptionHandler 에 HttpMessageNotReadableException → 400 매핑 보강(알 수 없는 판별자 대응). 각각 통합 테스트 필수.
  8. Step 8 테스트 — @WebMvcTest 2 파일: 다형 역직렬화 성공(4 type × osFamily) / 400(필드·판별자) / 404 / SSR 뷰 렌더.
비 목표 :
  • 엔티티·Repository·영속·스키마 — → U2-2. 이 슬라이스는 Step 6/7/9 가 N/A.
  • ProvisioningPhase/ProvisioningPhaseStep 과의 단계 모델 정합 — → U2-2 (저장 형태가 정해질 때 함께).

  • Builder/Resolver 2-tier(16+ 클래스) — → execution 도메인 편입 슬라이스(U3/Stage 4 국면). 소비자 없는 추출 금지.

🎬 미리보기 — 정의서 작성 폼 계약 라이브 데모

U2-1 이 굳히는 것은 "폼 → 다형 JSON → @Valid 판정"이라는 계약이다. 아래에서 단계 토글·OS 계열·정의서 명칭을 바꾸면 ① 전송될 JSON 이 판별자(type/osFamily)와 함께 라이브 조립되고 ② 제출 버튼의 차단(disabled+tooltip)이 갱신된다. 데모 판정 = SettingSaveRequest Bean Validation 과 동일 로직(같은 SSOT, 드리프트 0) — U2-1 은 서버 상태 가드가 없는 계약 슬라이스이므로, 여기서 재현하는 판정이 곧 이 슬라이스 서버 판정의 전부다.

적대적 자기비판 — ① 스텁 서비스는 speculative 아닌가? 아니다 — Step 5(Service 시그니처)의 산출물이며, 컨트롤러-인라인 더미(대안)는 U2-2 에서 컨트롤러 재작성 churn 을 만든다. 인터페이스를 지금 굳히면 U2-2 는 구현 교체만 한다. ② SettingStatus 를 엔티티 없이 도입하는 게 순서 위반 아닌가? — 목록/상세 화면이 상태 배지를 렌더하므로 계약의 일부다. 단 값 집합(레거시 6종)의 확정은 U2-2 CP1 에서 재심사한다고 §8 에 못박는다. ③ fragment 를 4장 다 이식하면 U2-3 범위 침범 아닌가? — 화면 골격(마크업)은 U2-1 의 "화면 이식" 범위, 그 폼이 실제로 저장·검증되게 하는 것이 U2-3. 골격 없이는 U2-3 이 화면+행동을 동시에 만들게 되어 분할 취지(계약 먼저)가 무너진다. ④ 숨은 순환은? — setting → management 참조 0(더미), setting → execution 참조 0(정합은 U2-2). 신규 패키지는 global 예외 계층에만 의존. 순환 소지 없음.
