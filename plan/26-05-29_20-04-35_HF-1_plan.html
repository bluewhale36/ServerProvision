§1. 현재 상태 요약 — 두 버그의 file:line 진단

본 슬라이스는 16-에이전트 Workflow 적대적 검증으로 근본 원인이 확정된 cross-cutting hotfix다. TrashLifecycleService.restoreFromTrash 단일 인프라에 공존하는 두 버그를 1곳 수정 + 예외 1줄 + restore form 마커로 4 도메인 (ISO·BIOS·BMC·Subprogram) 동시 해결한다. 아래 진단은 모두 코드 직접 Read 로 재확인됨.

버그 1 — FS-DB 무결성 깨짐 (ghost 양산)

  • TrashLifecycleService.restoreFromTrash 의 trashed_path != null 분기 (L123-152) 가 비가역 FS 작업을 DB 전이보다 먼저, try/catch·보상 없이 실행 :
    • L137 trashService.moveBack(trashedPath, originalPath) — trash → 원위치 mv (FS)
    • L147-148 computeSignature + markerService.write(...) — 마커 재발급 (FS)
    • L149 entity.reissueMarker(...) / L151 entity.restore() / L152 entity.clearTrashed() — DB 전이
  • 전체가 caller @Transactional 안 (IsoLifecycleService:144 메서드 어노테이션 → L163 restoreFromTrash 호출).
  • 실패 시나리오 : moveBack 성공 후 어느 단계든 RuntimeException (가장 유력 = MarkerWriteFailedException, ProvisionMarkerService:111-112, extends DomainException = RuntimeException) →
    • FS 는 자원이 원위치로 이미 복구됨 (트랜잭션 관할 밖, 롤백 불가)
    • DB 는 @Transactional 롤백 → is_deleted=true + trashed_path=stale 유효경로 로 되돌아감
    • 결과 = FS(원위치 존재) ↔ DB(삭제+trash경로 유효) 비대칭 고정
결정적 비대칭 — trashed_path == null 분기 (L114-120) 는 이미 !Files.exists(resourcePath) 사전검사로 "FS 존재 → DB 만 정합" self-heal 을 보유한다 (L118-119 restore() + clearTrashed()). 그러나 trashed_path != null 분기 (L123-152) 에는 그 사전검사가 없다. 이 비대칭이 두 버그 공통의 핵심 진원지다.
정정 — 이 stale row 는 기존 ghost 처리(MK3-1)가 못 잡는다. GhostEvaluator.isGhost 4조건은 trashed_path=null 을 포함하는데, 본 row 는 trashed_path 가 stale 하지만 유효 경로다 → ghost 4조건 불만족. MK3-1 의 GHOST_DB_ROW 흐름이 손대지 못하는 "dangling stale row" 라는 신규 결함 카테고리.

softDeleteToTrash (L60-92) 도 동형 — moveToTrash(L77) 후 markTrashed(L85) 전 예외 시 비대칭 (단 narrow window).

버그 2 — RestoreTrashLost raw 에러 (재시도 시 표면화)

  • 버그 1 로 stale row 가 고정된 뒤 사용자가 재시도 → trash 가 이미 비었으므로 L126 !Files.exists(trashedPath) → L127 throw new RestoreTrashLostException(...) (현 extends NotFoundException = 404).
  • 중대 정정 — raw JSON 이 아니다. native nav 로 렌더된 error.html 본문에 도메인 메시지가 그대로 노출된다 :
    • OS list 의 ISO restore form 에 data-async-submit 마커 부재 (list.html:382-391) — trash list 엔 존재 (trash/list.html:126).
    • → form.requestSubmit() 가 native POST (Accept: text/html) 로 나감
    • → WebExceptionHandler.handleNotFound (:58-63, produces=TEXT_HTML) → return "error"
    • → error.html:26 th:text="${message}" 가 도메인 메시지를 무가공으로 출력
R1-4-1 hotfix v3 가 못 막은 이유 — v3 는 content-type routing (Web/Api advice 의 @Order·produces 분리) 만 손댔다. (a) self-heal 부재 (b) message 무가공 (c) OS restore form 의 async-submit 비대칭 — 셋 다 안 건드림. 또한 RestoreTrashLost 만 404 인데 형제 3개 (RestorePathOccupied / RestoreTargetUnreachable / GhostRowRestoreNotAllowed) 는 모두 409 (extends ConflictException) — 계층 불일치.

§2. 슬라이스 요구사항 + 비목표

본 hotfix 는 judge 권장 후보3 hybrid (사용자 승인 완료, afterCommit 명시 배제) 를 채택한다. 4 요구사항 :

  1. ① 멱등 사전게이트 (최우선) — restoreFromTrash 의 trashed_path != null 분기 진입 직후, FS 2비트 (Files.exists(originalPath), Files.exists(trashedPath)) 로 4상태를 분류하는 private heal 메서드 도입. 이 게이트가 partial-restore 잔여를 self-heal 하면서 버그 2 의 발화 자체를 소멸시킨다.
  2. ② FS-first + 즉시 역보상 — moveBack(L137) + computeSignature/write(L147-148) 를 트랜잭션 본문 try 로 감싸고, catch 에서 1단계 역보상 후 원예외 재던짐. @Transactional 롤백과 FS 보상이 동방향 (둘 다 trash) 이라 dangling stale row 가 0 이 된다. afterCommit 미사용.
  3. ③ softDeleteToTrash 대칭 보강 — moveToTrash(L77) ↔ markTrashed(L85) 사이 역보상.
  4. ④ 버그 2 봉쇄 — RestoreTrashLostException superclass 404→409 (형제 일관성, advice polymorphic 흡수 = 신규 핸들러 0) + 메시지에 동선 추가 + OS/ISO/BIOS/BMC/Subprogram restore form 에 data-async-submit 마커 (공통 fragment 추출) + AsyncSubmitResult 친화 핸들러 로드.
스코프 한 줄 정의 — "공통 trash 인프라의 restore 흐름을 멱등 self-heal 사전게이트 + FS-first 역보상 으로 무결하게 만들고, RestoreTrashLost 의 404→409 + restore form async-submit 마커로 raw 에러를 봉쇄하여, 1곳 수정으로 4 도메인을 동시 치유하는 hotfix".

  • 예외 advice 통합 / errorCode 다형성 / ProblemDetail adapter — R2 묶음 (R1-4-1 hotfix v3 후속). 본 슬라이스 영향 0.

🎬 미리보기 — FS 2비트 4상태 매트릭스 + FS-first 흐름 before/after

본 hotfix 의 핵심 도메인 의미는 "원위치(O) · 휴지통(T) 두 FS 비트의 4상태" 다. 멱등 사전게이트가 이 4상태를 어떻게 분기하는지 먼저 보여주고, FS-first 역보상의 트랜잭션 정합을 before/after 로 비교한다.

P-1. FS 2비트 4상태 매트릭스 — restoreFromTrash 사전게이트의 핵심

(원 O · trash X) — partial-restore 잔여
exists(original)=true   exists(trashed)=false

버그 1 이 남긴 dangling 상태. self-heal : moveBack / write skip, DB 만 restore() + clearTrashed() 후 return. 예외 미발생 = 버그 2 소멸. (A-2 : 토스트 미노출 2xx)

(원 X · trash O) — 정상 복원
exists(original)=false   exists(trashed)=true

정상 경로. moveBack → write → DB 전이 진행 (단, ② FS-first try/catch 역보상으로 감싸짐).

(원 X · trash X) — 진짜 분실
exists(original)=false   exists(trashed)=false

자원이 외부에서 완전히 사라짐. RestoreTrashLostException → 409 (A-1). 메시지에 "휴지통 정리 / reconciliation GHOST_DB_ROW drift apply" 동선 추가.

(원 O · trash O) — 경로 점유
exists(original)=true   exists(trashed)=true

원위치에 동명 파일 점유 + trash 도 살아있음. 기존 RestorePathOccupiedException (409) 유지. heal 미적용.

(원 O) 분기 보강 — (원 O · trash X) 와 (원 O · trash O) 의 heal/판정 직전, 동명파일 오인 방어를 위해 manifestHash 선택 검증을 둔다. 원위치 파일이 실제로 그 자원인지(해시 일치) 확인하여 외부 무관 파일을 자원으로 오인해 DB 만 정합시키는 사고를 차단.

P-2. FS-first + 역보상 — 트랜잭션 정합 before / after

HF-1 전 — FS 비가역 + DB 롤백 = dangling

@Transactional (caller)
 ├─ moveBack(trash → 원위치)      FS (비가역)
 ├─ markerService.write(...)      FS 실패!
 │     └─ throw MarkerWriteFailed
 ├─ entity.reissueMarker(...)     (도달 X)
 ├─ entity.restore()             (도달 X)
 └─ entity.clearTrashed()        (도달 X)

결과:
  FS  = 자원 원위치에 존재 (롤백 불가)
  DB  = @Transactional 롤백
        is_deleted=true + trashed_path=stale
빨강 = FS 비가역 작업. DB 는 롤백되는데 FS 는 못 돌아옴 → dangling stale row 고정. 재시도 시 버그 2 (trash 빈 상태 → RestoreTrashLost) 표면화.

HF-1 후 — 사전게이트 + 동방향 역보상

@Transactional (caller)
 ├─ heal 사전게이트 (FS 2비트 분류)
 │    └─ (원O·trashX) → DB만 정합 return (self-heal)
 │    └─ (원X·trashX) → 409
 │    └─ (원O·trashO) → 409 (점유)
 │    └─ (원X·trashO) → 아래 진행
 ├─ try {
 │    moveBack(trash → 원위치)
 │    markerService.write(...)   실패!
 │  } catch {
 │    deleteIfExists(마커)
 │    moveBackReverse(원위치 → trash)  역보상
 │    throw 원예외
 │  }
 ├─ entity.reissueMarker / restore / clearTrashed

결과:
  FS  = 자원 trash 로 역복귀 (보상)
  DB  = @Transactional 롤백 (trash 상태 유지)
  → FS·DB 동방향 = dangling 0
초록 = HF-1 신규. catch 의 역보상이 FS 를 trash 로 되돌려 @Transactional 롤백과 동방향으로 만든다. 역보상은 TrashService.moveBackReverse (기존 moveWithRetry 3회 재사용) 로 캡슐화.
핵심 통찰 — 두 메커니즘의 분담 — ① 멱등 사전게이트는 이미 발생한 dangling 을 self-heal 하며 버그 2 를 소멸시키고, ② FS-first 역보상은 새로운 dangling 생성을 0 으로 만든다. 두 메커니즘이 함께 들어가야 "과거 잔여 치유 + 미래 발생 차단" 이 모두 충족된다. 어느 하나만으로는 불완전.
§3. 분리 후 책임 + 시그니처 (heal / moveBackReverse)

본 hotfix 는 새 service 신설이 아니라 기존 인프라 내부의 책임 정렬이다. 분기문 줄을 늘리는 대신 (CLAUDE.md §조건 분기문 무분별 확장 지양), 멱등 게이트를 단일 private 메서드로 추출하고 역보상을 TrashService 메서드로 캡슐화한다.

3.1 신규 private heal 메서드 — FS 4상태 게이트 (TrashLifecycleService)

// restoreFromTrash 의 trashed_path != null 분기 진입 직후 호출.
// return true  → caller 가 추가 작업 없이 종료 (self-heal 또는 예외 throw 가 내부에서 처리됨)
// return false → 정상 복원 흐름 (moveBack ...) 진행
private <T extends LifecycleEntity & Markable> boolean healOrThrowBeforeRestore(
        T entity, Path originalPath, Path trashedPath) {
    boolean originalExists = Files.exists(originalPath);
    boolean trashedExists  = Files.exists(trashedPath);

    if (originalExists && !trashedExists) {        // (원O·trashX) partial-restore 잔여
        assertSameResourceOrThrow(entity, originalPath);  // manifestHash 선택 검증
        entity.restore();
        entity.clearTrashed();                     // DB만 정합 (self-heal, A-2: 토스트 미노출)
        return true;
    }
    if (!originalExists && !trashedExists) {       // (원X·trashX) 진짜 분실
        throw new RestoreTrashLostException(trashedPath.toString());  // A-1: 이제 409
    }
    if (originalExists && trashedExists) {         // (원O·trashO) 경로 점유
        throw new RestorePathOccupiedException(originalPath.toString());  // 기존 409 유지
    }
    return false;                                  // (원X·trashO) 정상 복원 진행
}

이 게이트는 기존 L126 (RestoreTrashLost) / L133 (RestorePathOccupied) 의 개별 검사를 4상태 단일 분기로 응집해 흡수한다 — 기존 검사 줄을 늘리는 게 아니라 대체한다. RestoreTargetUnreachable (부모 디렉토리 부재, L129-132) 검사는 게이트 통과 후 정상 흐름 직전에 그대로 유지.

3.2 신규 TrashService.moveBackReverse — 역보상 캡슐화

/** HF-1 — FS-first 역보상. restore 본문 mv 후 후속 실패 시 원위치 자원을 trash 로 되돌린다.
 *  기존 moveWithRetry (3회 ATOMIC_MOVE → fallback) 재사용. */
public void moveBackReverse(Path originalPath, Path trashedPath) {
    moveWithRetry(originalPath, trashedPath);   // 기존 private 재사용 (신규 mv 로직 0)
    log.info("[trash] moveBackReverse (역보상) from={} to={}", originalPath, trashedPath);
}

신규 mv 로직 0 — 기존 moveWithRetry(TrashService:100-119) 를 방향만 바꿔 재사용. over-abstraction 회피.
