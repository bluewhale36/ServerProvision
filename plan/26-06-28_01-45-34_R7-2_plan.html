§1. 현재 상태 · 목표

R7-1 완료(commit 876b749) — MarkableScanner 가 4 책임으로 분리됨. R7-2 는 OS 도메인의 REAL ObjectProvider 2개(OSMetadata·ISO scanner)를 제거한다.

지금 OS scanner 의 순환 (REAL — 별도 bootRun 으로 실증된 사슬)

IsoLifecycleService → typedNameVerifierImpl → isoMarkableScanner → IsoLifecycleService ↺
(직접 주입 전환 시 BeanCurrentlyInCreationException 으로 부팅 실패 — 실측)

순환의 세 변을 풀어 쓰면 :

(A) TypedNameVerifier → scanner            scanner 가 displayName lookup 제공
(B) scanner → 도메인 service               scanner 의 복구/영구삭제가 service 역호출
(C) 도메인 service → TypedNameVerifier      LifecycleService.purgeWithTypedNameCheck 가 verify() 호출
                                            (OSMetadataLifecycleService:166 / IsoLifecycleService:227)
목표를 풀어 쓰면 — 이 순환을 끊되 MarkableScanner 의 통합(4 책임 한 SPI)을 깨지 않는다. 변 (C)를 끊는다 — OS/Iso 의 LifecycleService 가 TypedNameVerifier(빈)를 주입하는 대신 의존성 없는 static 가드를 쓰게 해, service → verifier 변을 없앤다. 그러면 scanner 는 복구/영구삭제를 그대로 들고(직접 주입, ObjectProvider 불요), MarkableScanner 는 통합 그대로다. 동작(휴지통 복구·영구삭제·재조정·검증)은 모두 불변.
§2. 설계 — 어느 변을 끊나 (대안 비교: A vs D vs E)

순환에는 끊을 수 있는 변이 둘((B) scanner→service, (C) service→verifier)이다. 어느 변을 끊느냐로 설계가 갈린다. 세 대안을 비교한다.

대안 A — 비채택
변 (B) 절단 : trash 를 scanner→LifecycleService 로 이전

MarkableTrashOperator 를 scanner 에서 떼어 LifecycleService 가 구현. 탈락 사유 : MarkableScanner 가 쪼개져 "1 도메인 = 1 SPI 구현" 통일성이 반감되고(사용자 지적), composite 재정의 + TrashController/PurgeExecutor 재배선 + BIOS/BMC/Subprogram scanner 에 MarkableTrashOperator 추가 선언까지 변경량이 크다.

대안 D — 채택
변 (C) 절단 : LifecycleService 의 typed-name 을 TypedNameVerifier(빈) → TypedNameGuard(static)로

LifecycleService.purgeWithTypedNameCheck 는 이미 엔티티를 손에 들고 있으므로(아래 §3), id→엔티티 재조회가 필요한 TypedNameVerifier 빈 대신 의존성 0 인 static 가드로 검증한다. service → verifier 변이 사라져 순환이 끊기고, scanner 는 복구/영구삭제를 그대로 들고 ObjectProvider 없이 직접 주입으로 전환된다. MarkableScanner 통합 유지, 소비자 재배선 0.

게다가 이미 절반은 이 모양 : BIOS/BMC/Subprogram 은 지금도 inline displayName().equals()(verifier 미주입)이라 scanner 가 service 를 직접 주입해도 순환·ObjectProvider 가 없다. 즉 R1-5/R3-5 가 OS/Board/Iso 만 verifier 위임으로 바꾼 것이 순환의 원인 — 대안 D 는 6 도메인을 일관된 패턴으로 모은다(중복은 TypedNameGuard 가 흡수).

대안 E — 보류
D + checkPrecondition 분리로 전역 @Lazy 까지 제거

대안 D 는 ObjectProvider 3개를 없애지만, scanner→service 변이 남아 SoftDeleteIntentService 의 전역 @Lazy 는 유지된다(아래 트레이드오프). 그 @Lazy 까지 0으로 만들려면 checkPrecondition 을 LifecycleService 밖으로 빼는 추가 분리가 필요. 보류 사유 : 그 @Lazy 는 cross-cutting helper 의 정당한 순환 차단(ObjectProvider 검증 보고서에서 "REAL"로 분류)이라 제거가 결함 해소가 아니다 — 추가 분리의 복잡도 대비 이득이 작다. "정당한 @Lazy 1개 허용" 으로 R7 capstone 목표를 완화한다.

§3. 변경 상세 (TypedNameGuard 추출 · 엔티티 재사용)

현재 코드 — 엔티티를 로딩해 버린 뒤 verifier 가 재조회 (중복)

// OSMetadataLifecycleService:165
public void purgeWithTypedNameCheck(Long id, String typedName) {
    osMetadataRepository.findByIdAndIsDeletedTrue(id)            // 로딩 → 버림(존재만 확인)
            .orElseThrow(() -> new IllegalOSMetadataStateException("…id=" + id));
    typedNameVerifier.verify(ResourceType.OS_IMAGE, id, typedName);   // 다시 id→scanner→findById (중복 조회) + service→verifier 변
    purge(id);
}

Option D — 로딩한 엔티티를 가드에 그대로 전달

// 신규 — 의존성 0 (scanner·service 무관)
public final class TypedNameGuard {
    public static void verify(Markable resource, String typedName) {
        String expected = resource.displayName();
        if (!expected.equals(typedName)) throw new TypedNameMismatchException(expected, typedName);
    }
}

// OSMetadataLifecycleService (전환 후)
public void purgeWithTypedNameCheck(Long id, String typedName) {
    OSMetadata image = osMetadataRepository.findByIdAndIsDeletedTrue(id)   // 엔티티 보존
            .orElseThrow(() -> new IllegalOSMetadataStateException("…id=" + id));
    TypedNameGuard.verify(image, typedName);   // 재조회 없음 · verifier 빈 미주입 → 변 (C) 소멸
    purge(id);
}

IsoLifecycleService:227 동일(findById 한 ISO 엔티티 재사용). 두 service 에서 private final TypedNameVerifier typedNameVerifier 필드 삭제.

scanner — ObjectProvider → 직접 주입

  • OSMetadataMarkableScanner : ObjectProvider<OSMetadataLifecycleService> → OSMetadataLifecycleService 직접 필드. restoreFromTrash/purgeFromTrash 가 getObject().restore/purge → osMetadataLifecycleService.restore/purge. trash 메서드 그대로 유지(이전 안 함).
  • IsoMarkableScanner : ObjectProvider<IsoLifecycleService> → 직접 주입. 동일.

BIOS/BMC/Subprogram inline → TypedNameGuard (공통 dedup, R7-2 에서 1회)

세 service 의 purgeWithTypedNameCheck 에 글자단위 복제된 3줄(String expected = e.displayName(); if(!expected.equals(typedName)) throw …)을 TypedNameGuard.verify(entity, typedName) 한 줄로 치환(§중복 해소). 이들은 verifier 를 원래 안 써 순환·ObjectProvider 가 없고, 본 치환도 동작 불변.

TypedNameVerifier(빈)는 남는다. nudge 교체·모달 컨트롤러(BiosNudgeController/TrashController/ConfirmModalFragmentController 등)는 엔티티가 아직 없어 id→조회가 필요하므로 빈이 맞다. "엔티티가 손에 있으면 TypedNameGuard(static), 없으면 TypedNameVerifier(빈)" 의 자연스러운 분리.

§7. 부산물 / 주의
  • MarkableScanner 통합 유지. R7-1 의 4-인터페이스 분리(ISP)는 그대로 유효 — narrow 주입(verifier→Inventory, SoftDeleteIntentService→Ghost)도 그대로. MarkableTrashOperator 는 scanner 가 계속 구현하는 살아있는 경계로 남는다(낭비 아님).
  • 전역 @Lazy 유지(정당). scanner→service 변이 남아 SoftDeleteIntentService 의 @Lazy 가 지킨다. R7 capstone 의 "전역 @Lazy 제거" 목표는 "정당한 @Lazy 1개 허용"으로 완화(§2 대안 E 보류). Notion R7-6 본문 이 방향으로 갱신 권고.
  • R1-5/R3-5 의 verifier 위임을 부분 환원. service 측 typed-name 만 가드로 되돌리고, controller 측 verifier 는 유지. R1-5/R3-5 의 dedup 의도는 TypedNameGuard 가 더 낫게 충족(중복 조회까지 제거).
  • R7-3(Board)은 TypedNameGuard 추출(본 plan)을 전제 — R7-2 와 같은 commit 묶음 또는 직후.
