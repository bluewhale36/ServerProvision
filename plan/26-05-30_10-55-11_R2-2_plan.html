§1. 현재 상태 (9-에이전트 Workflow 진단)

증상 — 부모·자식 모두 비활성에서 자식 활성화(toggle) 시도 시, SSR form POST 가 ChildLifecycleBlockedByParentException(409) 으로 거절되고 브라우저에 raw JSON 노출 (localhost/management/os/7/iso/27/toggle). 부모 삭제 시 자식 restore, 부모 deprecated 시 자식 undeprecate 도 동일.

가드 표면 — 부모 lifecycle 이 자식 단독 액션을 막는 지점이 3 도메인에 완전 대칭 :

자식 액션차단 조건 (부모)자식 precondition가드 위치 (3 도메인)
toggle (활성화)부모 DISABLED 또는 DEPRECATEDnextEnabled=true 일 때만 (disable 은 자유)IsoLifecycleService:88-101 / BiosService:370-390 / BmcService:426-445
restore부모 DELETEDnoneIsoLifecycleService:165-173 / BiosService:427-439 / BmcService:484-495
undeprecate부모 DEPRECATEDnoneIsoLifecycleService:205-214 / BiosService:468-481 / BmcService:592-605
핵심 진단 3가지 :
  1. 예외 오용 — 사용자의 정상 UX 모순을 backend 예외로 거절 → raw JSON / 500 노출 (R2 묶음 동기). 예외는 진짜 비정상 상황 전용이어야 한다.
  2. 가드 조건 3중 복붙 — !parent.isEnabled() ? "DISABLED" : parent.isDeprecated() ? "DEPRECATED" : null 삼항식이 3 service 에 99% 동일 복붙 (CLAUDE.md "복붙 3곳 즉시 추출" 위반). 게다가 ISO restore 뷰만 th:if=${os.isDeleted} 로 부모 boolean 을 직접 조합 — 서버 가드와 2원 표현 = 드리프트원. BIOS/BMC 뷰는 차단 로직이 아예 없어 무조건 form 노출 → 서버에서만 409.
  3. deleted 가드 공백 (사용자 지적) — 부모 DELETED 시 현재 restore 만 막고 enable / undeprecate 는 안 막는다. 부모가 삭제됐는데 우회 경로(direct POST / stale)로 자식을 활성화/undeprecate 할 수 있어 "삭제된 부모 밑 active 자식" 모순이 재발 가능. 또한 휴지통 자원 자기 자신도 toggle/undeprecate 가 무가드라 우회로 변경 가능. → §4 comprehensive 보강.
§2. 요구사항 + 비목표

채택 = 예외 사용 원칙 전환 + UI 1차 차단 + 가드 조건 SSOT 단일화 (CLAUDE.md 신설 불가침 원칙).

  1. UI 1차 차단 — 모순을 부를 자식 액션 버튼을 disabled + tooltip 으로 사전 차단. 전 도메인(ISO/BIOS/BMC) × 전 가드(toggle/restore/undeprecate) = 9.
  2. 가드 조건 SSOT — 9 가드 조건식 + 3 복붙 삼항식을 LifecycleEntity 의 blocksChildXxx() 4 메서드로 단일화. (1) 서버 throw 조건 (2) 뷰 disabled 플래그가 같은 메서드 바디 를 거친다 → 드리프트 구조적 차단.
  3. capability 플래그 — 자식 Response(ISO/Bios/Bmc) 에 parentBlocksEnable/Restore/Undeprecate 3 boolean 을 부모 메서드로 계산해 적재. 뷰는 boolean 1개만 읽는다.
  4. comprehensive deleted 가드 (사용자 지적 반영) — ① 부모 DELETED 도 enable/undeprecate 차단 (자식 enable 은 부모가 fully-active 일 때만). ② 휴지통 자원 self-freeze (isLifecycleFrozen()) — restore/purge 외 변경 차단. 우회 경로 전면 방어. (§4)
비 목표 :
  • 서버 가드 예외 제거 — 안 함 (불가침, 사용자 결정). throw 는 안전망 — UI 가 정상 흐름을 막으므로 direct POST / 동시성 / stale 등 비정상 경로에서만 409.
  • raw JSON 노출 근본 정리 → R2-3 (advice 통합). 본 슬라이스는 UI 차단으로 노출 빈도를 0 에 수렴시키는 1차 방어까지.
  • Driver / Subprogram — 비대상 (LifecycleEntity 메서드 공용이라 후속 동형 확장 비용 0).
  • 부모 DTO lifecycle boolean 직접 노출 / 부모 로드 경로 통합 — 불필요 (자식 Response capability 우회, FK 구조 상이). BIOS/BMC 인라인 style 제거 — 스코프 밖.

§4. 도메인 모델 (SSOT)

LifecycleEntity 에 가드 메서드 4개 추가 (별도 Policy 클래스 / 부모별 중복 금지)

// global/entity/LifecycleEntity.java — "이 자원이 부모일 때 자식의 해당 액션을 막는가" (DELETED comprehensive)
public String childEnableBlockReason() {   // null = 차단 안 함. toggle 가드 + 예외 parentState 공용
    return isDeleted ? "DELETED" : !isEnabled ? "DISABLED" : isDeprecated ? "DEPRECATED" : null;
}
public boolean blocksChildEnable()      { return childEnableBlockReason() != null; }   // 부모가 fully-active 아니면 (¬삭제·¬비활성·¬deprecated)
public boolean blocksChildRestore()     { return isDeleted; }                          // 부모 삭제면 복구 불가
public boolean blocksChildUndeprecate() { return isDeprecated || isDeleted; }          // 부모 deprecated/삭제면 해제 불가

위치 근거 (information expert) — 가드 조건이 부모 자신의 lifecycle boolean 3필드만 읽고 자식 종류와 무관하게 동일 의미. OSMetadata/BoardModel 모두 이미 LifecycleEntity 상속 → 한 곳 작성으로 양 도메인 자동 커버. currentStage() default 메서드 선례와 동형.

comprehensive 보강 (사용자 지적 반영) — 두 갈래의 deleted 우회 차단 :
  • 부모 DELETED → 자식 enable/undeprecate 도 차단 : 기존 설계는 부모 삭제 시 restore 만 막았다. 부모가 삭제됐으면 자식은 휴지통에 있어야 하는데, 우회 경로(direct POST / stale)로 자식 enable/undeprecate 가 가능했다. childEnableBlockReason 에 DELETED 추가 + blocksChildUndeprecate 에 isDeleted 추가 → "자식 enable 은 부모가 fully-active 일 때만".
  • 자기-삭제 동결 (CP2 확인 — 이미 enforce) : 휴지통 자원 자체의 toggle/deprecate/undeprecate 는 각 service 의 requireLive* 헬퍼가 이미 거부(deleted 자원에 Illegal*StateException)하고, 뷰에서도 삭제 자원은 toggle/undeprecate 버튼이 아예 안 뜨고 restore/purge 만 노출된다 → 별도 isLifecycleFrozen() 불필요(미도입). 따라서 신규 보강은 부모-DELETED SSOT 가드뿐.

자식 Response 에 capability 3 필드 추가

ISOResponse / BiosResponse / BmcResponse (record) 에 boolean parentBlocksEnable, parentBlocksRestore, parentBlocksUndeprecate. 부모 핸들이 엔티티 그래프에 이미 존재 :

  • BiosService.toResponse(BoardBIOS e) → e.getBoardModel() (repo 조회 0, 시그니처 확장 불필요)
  • BmcService.toResponse(BoardBMC e) → e.getBoardModel()
  • ISOResponse.from(ISO e) → e.getOsMetadata(). inProgress/withExtractionInProgress 는 부모 없음 → 3 플래그 false 고정

§7. 예외 계층

변경 0. ChildLifecycleBlockedByParentException 및 throw 자체 제거 안 함 (안전망 유지).

  • SSOT 환원은 가드 조건식 + 상태라벨 만 대상. throw / ResourceType / requestedAction / displayName 인자는 도메인별로 달라 그대로 (예외 합성은 이미 composeMessage 흡수).
  • 정상 UX 에서 throw 미도달 (UI 가 막음) → raw JSON / 500 노출 0. 비정상 경로에서만 409 안전망 작동.
  • self-freeze 가드 — 신규 예외 0. 휴지통 자원 자체의 toggle/undeprecate 우회 호출은 기존 IllegalLifecycleTransitionException (LifecycleEntity softDelete/restore/deprecate 가드가 쓰던 것) 재사용. 부모 DELETED 의 enable/undeprecate 차단은 ChildLifecycleBlockedByParentException(parentState="DELETED") 로 기존 예외 그대로.
