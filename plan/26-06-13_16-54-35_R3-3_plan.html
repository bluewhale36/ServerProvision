§0. 동기 — R1(OS) service 분할의 Board 미러 + R3-2 후속

R1 묶음에서 OS 도메인은 fat service 를 os/service/metadata/OSMetadataService(CRUD·조회) + OSMetadataLifecycleService(상태전이·cascade·purge, implements LifecycleService) + nudge 본체로 책임 분리했다. Board 도메인은 같은 작업을 아직 받지 못해 BoardModelService 한 클래스가 458줄로 CRUD·조회·상태전이·cascade·purge·nudge 를 모두 떠안고 있다.

  • 단일 fat service — 한 클래스에 7개 책임군이 혼재. CLAUDE.md §중복·가독성 불가침(100줄 초과 메서드·책임 혼재 신호)과 긴장. LifecycleService 인터페이스 Javadoc 이 "BIOS / BMC / Subprogram / Board / ISO 도 같은 인터페이스 구현" 을 예고하나 Board 만 미구현.
  • scanner→service ObjectProvider — BoardModelMarkableScanner 가 ObjectProvider<BoardModelService> 로 fat service 전체를 lazy 참조(restore/purge/findDeletedChildLabels 3개만 실제 사용). 의존 표면이 458줄 service 전체로 넓고, scanner→service 잠재 사이클의 한 변.

R3-2 가 컨트롤러를 Metadata / Lifecycle / Nudge 3분할로 끝냈으므로, R3-3 은 그 대칭으로 service 계층을 같은 3 책임군으로 분리하고 scanner 를 분리된 service 로 re-point 한다(행동 변화 0). 자식 컨트롤러(bios/bmc/subprogram)의 boardModelService.findById/findAllGrouped cross-feature read 는 끊지 않고 BoardModelMetadataService 로 재배선한다.

R 시리즈 정합 — R3-1(Subprogram cascade 합류) → R3-2(컨트롤러 3분할) → R3-3(service 3분할) → R3-4(자식 cascade 다형화). 본 슬라이스는 R1(OS) 이 이미 검증한 분할 형태를 Board 에 그대로 미러.

§2. 요구사항 (확정) + 비목표
  1. Metadata / Lifecycle 분리 — board/service/metadata/ 아래 BoardModelMetadataService(CRUD·조회) + BoardModelLifecycleService(상태전이·cascade·purge, implements LifecycleService). R1(OS) 형태 미러.
  2. Nudge 본체 이전 — nudge 4 메서드를 BoardModelNudgeService 로 이전. (nudge 컨트롤러는 R3-2 에서 이미 분리됨 → service 본체만 합류.)
  3. scanner re-point — BoardModelMarkableScanner 가 BoardModelService 대신 분리된 service 를 참조. 메커니즘(ObjectProvider 유지 vs 직접주입)은 D1 결정 종속(§8).
  4. cross-feature read 보존(불가침) — 자식 컨트롤러(bios/bmc/subprogram)의 boardModelService.findById/findAllGrouped 호출을 BoardModelMetadataService 로 재배선(끊지 않음).
  5. R3-2 컨트롤러 재배선 — 분할된 3 컨트롤러(Metadata/Lifecycle/Nudge)의 service 주입을 각 책임 service 로 교체.
  6. 행동 보존(불가침) — URL·HTTP status·cascade 동작·nudge 흐름·typed-name 가드 전부 동일. 기존 컨트롤러/서비스 테스트가 회귀망.
cross-feature read 보존 (불가침) — bios/bmc/subprogram 컨트롤러가 board 메타를 읽는 호출은 도메인 경계를 넘는 정당한 read 다. R3-3 은 이 호출의 대상 빈만 BoardModelMetadataService 로 바꾸고, 호출 자체를 제거하거나 이벤트로 우회하지 않는다.
비 목표 (다음 슬라이스 / D1 종속)
  • ObjectProvider 메커니즘 자체의 제거 — D1 권고(Option B)대로면 R3-3 은 타입 narrowing 까지만(ObjectProvider<BoardModelLifecycleService>), 실제 제거 + edge 소멸은 R7-3(Board) 가 SPI(4-sealed) 로 일괄. Option A 채택 시에만 R3-3 이 직접주입으로 제거 + bootRun 사이클 검증 Step 추가. → §8 D1.
  • 자식 cascade 다형화 — softDelete/restore 가 bios/bmc/subprogram service 를 형제 @Lazy 로 직접 호출하는 구조의 다형화는 R3-4.
  • 전 도메인 MarkableScanner SPI 4-sealed 분리(MarkableInventory/DriftApplier/TrashOperator/GhostOperator) — R7 전담. 본 슬라이스는 Board scanner 의 service 대상만 재배선.

② BoardModelLifecycleService (신규 · implements LifecycleService)

@Service @Transactional(readOnly = true)
public class BoardModelLifecycleService implements LifecycleService {
    // LifecycleService 계약 구현 (R1 OSMetadataLifecycleService 미러)
    void toggleEnabled(Long id);
    void softDelete(Long id);
    // default restore(Long) → restore(id, false) 는 인터페이스 default 그대로 사용
    RestoreResponse restore(Long id, boolean cascade);
    void deprecate(Long id);  void undeprecate(Long id);
    void purge(Long id);  void purgeWithTypedNameCheck(Long id, String typedName);
    // Lifecycle 고유
    List<String> findDeletedChildLabels(Long boardId);   // scanner preview 진입점
    private void cascadeRecompute(Long id);               // bios/bmc/subprogram recomputeEffective
    private void purge(BoardModel board);                 // 공통 hard-delete 본체
}

§8. 예상 부산물 / 주의 / ★ 결정 D1
★ 결정 D1 (이 plan 의 핵심) — scanner→service ObjectProvider 제거를 R3-3 에서 할지 R7 에서 할지

배경 — R3-3 의 사용자 원제 = "Service Class ObjectProvider 사용 패턴 개선 / ObjectProvider 지양의 의존성 단방향 개선". 그러나 신규 R7 단계가 전 자원 도메인 ObjectProvider 제거(MarkableScanner 4-sealed SPI 분리)의 최종 전담이다. BoardModelService 가 R3-3 에서 분리·삭제되므로 scanner re-point 는 어느 경우든 필요하나, ObjectProvider 메커니즘 자체를 R3-3 에서 없앨지 R7 로 미룰지가 결정 사안.

옵션R3-3 에서 하는 일장점단점 / 리스크
A
(R3-3 에서 제거)		 scanner ObjectProvider<BoardModelService> → 직접주입 BoardModelLifecycleService. ObjectProvider 즉시 제거. R3-3 원제(ObjectProvider 제거) 완전 충족. 의존표면 최소화. 직접주입이 사이클(SoftDeleteIntentService→List<MarkableScanner>→scanner→Lifecycle→…)을 재형성하지 않는지 bootRun 실측 필요(분석상 안전 주장은 speculative). R7-3(Board)에서 scanner 를 또 변경(직접주입→implements MarkableTrashOperator) = scanner 2회 터치 + 사이클 검증 2회.
B
(R7 에서 제거)
권장		 Service 분리 + scanner 를 ObjectProvider<BoardModelLifecycleService>로 re-point(타입만 narrowing, lazy 메커니즘 유지). ObjectProvider 제거를 전담 단계(R7)에 응집 → 전 5 도메인 동형 처리. scanner 메커니즘 1회 터치, 사이클 검증 1회. R3-3 은 저위험 "분리 + 타입 narrowing". R3-3 원제 "ObjectProvider 제거"는 narrowing 까지만 — 완전 제거는 R7-3 가 완성(원제 의미를 R7 이 이행).

권고 : Option B. 근거 — (1) R7 이 ObjectProvider 제거 전담이라는 현 로드맵과 정합, (2) scanner 더블 터치·더블 사이클검증 회피, (3) R3-3 은 Service 분리에 집중해 리스크 격리. 단, R3-3 의 정체성(제목)을 사용자가 직접 적었으므로 최종 채택은 CP1 사용자 결정 사안으로 남긴다 — 본 plan 은 B 를 default 권고로 제시하되 A/B 선택을 사용자에게 위임.
