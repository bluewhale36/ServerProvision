진입 / 이력

  • parked : plan/26-05-31_21-16-34_R4-1_plan.html (단일 boolean + cascade_origin 1bit 안) — force-down-while-explicit 미해결로 폐기.
  • 재조사 : report/26-06-01_18-45-25_R4-state-preservation-redesign_report.html — 권위 5종(Windows ACL · OverlayFS · K8s · CSS · PostgreSQL GUC) 수렴 → Option A 채택. 본 plan 이 그 구현안.
1bit 안이 막힌 케이스 (force-down-while-explicit) — 자식 A(직접 관리·활성)가 부모 deprecate 로 강제 deprecated 된 뒤, 부모 undeprecate 시 A 가 영구 deprecated 로 갇힘. 단일 boolean 이 A 의 본래 의도값(활성) 을 기억 못 해 "운영자가 직접 deprecate 한 자식"과 구별 불가.
해법 (5종 수렴) — own(본래 의도)을 effective(현재값)와 분리 저장. cascade 는 own 불가침, effective 만 재계산. 복원 = "제약 제거 시 손대지 않은 own 이 자동으로 effective 로 재계산". A.own=활성 ≠ B.own=deprecated 로 즉시 구별 → A 복원, B 보존.
§2. 요구사항 (확정) + 비목표

확정 (사용자 결정)

  1. 값 표현 재정의 — 자식에 own_enabled · own_deprecated(운영자 본래 의도) 신규. 기존 is_enabled · is_deprecated 는 effective(현재 적용값) 캐시로 의미 재정의.
  2. effective 식 (차원 독립) — is_deprecated = own_deprecated OR 부모.is_deprecated · is_enabled = own_enabled AND 부모.is_enabled. 제약(부모)은 OR/AND 로 항상 강제, own 은 보존. 두 차원은 간섭하지 않는다 — 부모 deprecate 는 자식 enabled 를 끄지 않음(deprecated ≠ disabled).
  3. cascade = 재계산 — 부모 상태 변경 시 자식 effective 만 재계산(own 불변). 기존 undeprecate 의 "deprecated 자식 전량 환원" 결함이 구조적으로 소멸(필터 자체가 사라짐).
  4. force-down-while-explicit 해결 — A(own_dep=false): 부모 deprecate→effective dep, 부모 undeprecate→활성 복원. B(own_dep=true): 항상 deprecated 보존.
  5. '자동' 체크박스 제거 — own_* 가 자동/직접 구분을 흡수. 운영자는 기존 활성/Deprecated 버튼으로 own 을 설정하고, 부모 제약은 자동 반영. R2-2 가드(버튼 disable + tooltip)는 그대로 유지(masking 액션 사전 차단).
  6. 신규 자식 own default = 부모 추종(inherit) — own_enabled=true, own_deprecated=false(중립값). 등록 직후 effective 가 부모 상태와 일치하고, 부모를 따라 내려가고 올라온다.
  7. derived query 무변경 — is_enabled/is_deprecated 가 effective(materialized)라 수십 개 Spring Data 파생 쿼리가 그대로 effective 를 읽음(=원하던 값). 회귀 0.
비 목표
  • deleted 차원 provenance — 본 슬라이스 enabled+deprecated 만. deleted 는 trash/TTL/ghost 물리부작용으로 기존 soft-delete cascade 유지 → R4-2.
  • source enum 명시 저장(report Option B) — 2단계 위계라 effective≠own 으로 "부모 강제중" 판별 가능 → 불요. 다단계/감사 요구 시 R4-2.
  • '자동' 체크박스 / cascade-origin 엔드포인트 — Option A 에서 불필요 → 신설 안 함.
  • 마이그레이션 — 미배포라 0(신규 컬럼 default + 1회 effective 재계산).
🎬 미리보기 — own/effective 라이브 데모

메인보드(부모) + BIOS 3(자식). 각 자식은 own(본래 의도 · 보라) 과 effective(현재값 · 파랑) 를 따로 가진다. 부모/자식 버튼을 눌러 ① B 를 직접 Deprecated → ② 부모 Deprecated(A·B·C 모두 강제) → ③ 부모 해제 순서로 확인 : own=활성인 A·C 만 복원, own=deprecated 인 B 는 보존. (데모 판정 = 서버 recomputeEffective() 식 1:1 재현)

부모 :
메인보드Gigabyte MS03-CE0 활성
own(보라) = 운영자 본래 의도, effective(파랑) = 부모 제약 반영 현재값. cascade 는 own 을 안 건드립니다.

BEFORE — undeprecate 결함

// 부모 undeprecate
parent.undeprecate();
children.filter(c -> c.isDeprecated())   // 값만
  .forEach(c -> c.undeprecate());        // 운영자 수동
                                         // deprecate 자식도 환원
// + force-down 시 own 소실 → A 영구 갇힘

AFTER — recomputeEffective

// 부모 어떤 전이든 → 자식 재계산
children.forEach(LifecycleEntity::recomputeEffective);
// recomputeEffective():
//   is_deprecated = own_deprecated OR 부모.is_deprecated
//   is_enabled    = own_enabled AND 부모.is_enabled   (차원 독립)
// own 불변 → 필터 없음, 결함 소멸
드리프트 0 — 데모 판정 = 서버 recomputeEffective() 단일 식. 필터/분기 없음.

§4. 도메인 모델 (SSOT)

LifecycleEntity (@MappedSuperclass) 변경 — 6 엔티티 일괄

@Column(name="own_enabled")    private boolean ownEnabled = true;     // 운영자 의도 (신규)
@Column(name="own_deprecated") private boolean ownDeprecated = false;  // 운영자 의도 (신규)
@Column(name="is_enabled")     private boolean isEnabled;    // effective 캐시 (의미 재정의)
@Column(name="is_deprecated")  private boolean isDeprecated; // effective 캐시 (의미 재정의)
// is_deleted : 변경 없음 (R4-2)

// 자식이 부모를 참조 (사용자 아이디어) — 루트/공용은 null
protected abstract LifecycleEntity parentLifecycle();

// effective 재계산 — own ⊕ 부모. final = 도메인 단일 로직(중복/파생 0)
public final void recomputeEffective() {
    LifecycleEntity p = parentLifecycle();
    boolean pDep = (p != null) && p.isDeprecated();   // 부모 effective
    boolean pEn  = (p == null) || p.isEnabled();
    this.isDeprecated = ownDeprecated || pDep;                 // OR  : 부모 deprecate → 자식 deprecate
    this.isEnabled    = ownEnabled && pEn;                     // AND : 부모 disable → 자식 disable (deprecated 무관, 차원 독립)
}

// 운영자 직접 (자식 액션) — own 변경 후 재계산
public final void operatorSetEnabled(boolean v)    { this.ownEnabled = v; recomputeEffective(); }
public final void operatorSetDeprecated(boolean v) { this.ownDeprecated = v; recomputeEffective(); }

// R2-2 가드 (유지) — 부모 effective 로 자식 액션 차단. 서버 throw + 뷰 disabled SSOT.
public final boolean parentBlocksEnable()      { /* 기존 그대로 */ }
public final boolean blocksChildUndeprecate()  { /* 기존 그대로 */ }
public final boolean blocksChildRestore()      { /* 기존 그대로 */ }
parentLifecycle() 구현 : BoardBIOS→boardModel · BoardBMC→boardModel · ISO→osMetadata · Subprogram→boardModel(공용=null) · BoardModel/OSMetadata→null(루트). 루트는 부모 없음 → effective = own.

cascade 책임 경계

자식 직접 (child service)   resource.operatorSetEnabled/Deprecated(v)   // own + 재계산
부모 cascade (부모 service)  부모.operatorSet*(v) [부모 effective 갱신]
                            → children.forEach(recomputeEffective)     // own 불변, 부모 참조
재계산 순서                  부모 먼저 → 자식 (자식이 부모의 새 effective 를 읽으므로)
materialized 캐시 동기화 규율 — effective 가 컬럼이므로 부모/own 이 바뀌는 모든 지점에서 recomputeEffective() 호출이 보장돼야 함(누락 시 stale). workflow 인벤토리(wf_50ca1724-953)가 전 write-site 를 식별 → Step 8 에서 그 지점들이 모두 재계산을 호출하는지 검증. (현행 cascade 도 이미 전 자식을 write 하므로 추가 부담 미미.)

상태 + own/effective 전이 (deprecated 예시)

자식 own_deprecated 부모 deprecate 부모 undeprecate 결과 A (직접·활성) false eff = F OR T = dep eff = F OR F = 활성 복원 ✓ B (직접 deprecate) true eff = T OR T = dep eff = T OR F = dep 보존 ✓ C (추종·활성) false(default) eff = F OR T = dep eff = F OR F = 활성 복원 ✓ // own 은 어느 단계에서도 안 바뀜. effective 만 재계산.
