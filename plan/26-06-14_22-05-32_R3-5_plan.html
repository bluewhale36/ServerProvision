§0. 동기 — 이미 존재하는 SSOT 자산을 Board 가 우회·복붙 중

R3-3(service 3분할) → R3-4(자식 cascade 다형화)로 BoardModelLifecycleService 는 상태전이·cascade 책임이 깔끔히 응집됐다. 그러나 그 안에 이미 framework/인프라에 존재하는 단일 소스를 안 쓰고 우회·복붙한 잔재 2가지가 남아 있다 — R3-5 가 이 둘을 제거해 Board lifecycle service 의 마지막 중복을 청산한다.

  1. 중복 restore(Long) override — LifecycleService 인터페이스가 이미 default void restore(Long id){ restore(id,false); } 를 제공(L45-47)하는데, BoardModelLifecycleService 가 동일 본문을 다시 명시 선언(L104-110). 그 자리에 주석으로 "R3-5 에서 default 흡수로 제거 예정" 이 직접 적혀 있다 — 예고된 청산.
  2. inline typed-name 비교 복붙 — purgeWithTypedNameCheck 가 board.displayName().equals(typedName) 비교 + throw TypedNameMismatchException 를 메서드 안에 직접 박았다(L156-164). 그런데 이 검증식의 단일 소스인 global/trash/service/TypedNameVerifier(+Impl)가 이미 존재하고 BoardModel scanner 도 SPI 등록(BOARD_MODEL)이 끝나 있다 — purge 경로만 이 SSOT 를 안 쓰고 우회 중.
R 시리즈 정합 — R3-2(컨트롤러 3분할) → R3-3(service 3분할) → R3-4(cascade 다형화) → R3-5(중복 청산: default 흡수 + typed-name SSOT). 본 plan 의 CP1 가치 = "무엇을 지우고, 무엇을 이미 있는 자산으로 대체하는가" 의 신구 코드 대조 — §3·🎬 데모가 추상 설명이 아니라 실제 삭제/위임을 전수로 보인다.
이 슬라이스의 핵심 함정 (CP1 에서 반드시 합의) — "restore 오버로드 제거" 는 순진하게 보면 "LifecycleService 전 구현체에서 일괄 제거" 로 읽힌다. 그러나 구현체 3개(Board / ISO / OSMetadata)는 제거 가능성이 서로 다르다 — ISO 는 default 의 self-invocation 이 @Transactional 프록시를 우회(HF-1 버그)하기 때문에 override 가 트랜잭션 경계 진입점으로 필요해 제거 불가다. R3-5 는 Board 만 안전하게 제거한다(§1·D1).
§1. 현재 상태 (restore 3구현체 비대칭 + typed-name 복붙 위치)

선행 완료 / 진입 전제

  • 완료 R3-4 — BoardModelLifecycleService 의 자식 cascade 가 List<BoardScopedChildLifecycle> 다형 순회로 응집. 본체 필드 = boardModelRepository + children 2개. R3-5 는 이 깔끔해진 본체 위에서 마지막 중복 2건을 청산.
  • 기존 자산 LifecycleService.restore(Long) default 이미 존재(L45-47) — restore(id,false) 위임. 진짜 구현 대상은 abstract restore(Long,boolean)(L57).
  • 기존 자산 global/trash/service/TypedNameVerifier(인터페이스) + internal/TypedNameVerifierImpl(@Component, List<MarkableScanner> 주입) 이미 존재. verify(ResourceType, Long, String) 가 Markable.displayName() 다형성 기준으로 검증 — typed-name 검증의 SSOT. 현재는 nudge 경로만 사용.
  • 기존 자산 BoardModelMarkableScanner.supportedType()=BOARD_MODEL 로 SPI 등록 완료 → TypedNameVerifierImpl 의 scannerByType 맵에 자동 포함. R3-5 신규 배선 불필요.

§2. 요구사항 (확정) + 비목표
  1. 중복 restore(Long) override 삭제 — BoardModelLifecycleService L104-110 제거. 인터페이스 default restore(Long) 가 restore(id,false) 를 위임. Board 의 restore 외부 호출자는 전부 2-arg 이므로 default 진입점 미호출 → HF-1 불가(§1·D1).
  2. purgeWithTypedNameCheck typed-name 검증을 공통 verifier 로 위임 — inline displayName().equals() + throw TypedNameMismatchException 을 typedNameVerifier.verify(ResourceType.BOARD_MODEL, id, typedName) 1줄로 대체. soft-deleted 선행 가드(findByIdAndIsDeletedTrue)와 purge(board) 는 그대로 유지.
  3. TypedNameVerifier 주입 — BoardModelLifecycleService 에 private final TypedNameVerifier typedNameVerifier; 추가(@RequiredArgsConstructor). cross-package 의존(board → global/trash)이 생기지만 순환은 scanner 의 ObjectProvider 가 차단(§8 D3, CP4 bootRun 실측).
  4. 동작·HTTP status 100% 보존(불가침) — purge typed-name 불일치 시 던지는 예외 타입이 TypedNameMismatchException(400) 으로 동일(verifier 가 같은 예외 throw). soft-deleted 아님 = IllegalBoardModelStateException(409) 동일. restore 동작 동일. 기존 통합/단위 테스트가 회귀망.
검증 의미 동치 보존 (불가침) — 기존 inline 은 이미 fetch 한 soft-deleted board.displayName() 와 비교. verifier 는 findActiveMarkableById(id).or(findTrashedById(id)) 로 lookup → 휴지통 board 를 찾아 displayName() 비교. soft-deleted board 는 trashed lookup 으로 동일 자원·동일 합성식에 도달 → 판정 결과 동일. 단, 선행 findByIdAndIsDeletedTrue 가드는 "반드시 soft-deleted" 전제를 유지하므로 그대로 둔다(verifier 는 active 도 허용하므로 가드를 대체하지 않음).
비 목표 (다음 슬라이스 / 타 도메인)
  • ISO / OSMetadata restore override — ISO 는 HF-1 로 보존 필수, OSMetadata 는 이미 흡수 완료. 둘 다 OS 도메인 = R3 범위 밖. R3-5 는 건드리지 않는다.
  • BIOS / BMC / Subprogram purge typed-name SSOT 전환 — 각 R4-x / R5-x / R6-x 에서 동일 패턴으로. OS helper 복붙은 OS 정리 후속, trash SPI canon 은 R7.
  • purgeWithTypedNameCheck 를 인터페이스에서 제거 / controller 로 끌어올림 — 검증 위치를 service→controller 로 옮기는 재배치는 blast radius 가 ISO/OSMetadata 까지 번지므로 비목표(§8 D3 Option B 기각).
  • scanner ObjectProvider 제거 — R7-3 전담.

② purgeWithTypedNameCheck — inline 비교 → verifier 위임

BEFORE (L154-165 — inline 복붙)

@Override
@Transactional
public void purgeWithTypedNameCheck(Long id, String typedName) {
    BoardModel board = boardModelRepository
        .findByIdAndIsDeletedTrue(id)
        .orElseThrow(() -> new IllegalBoardModelStateException(
            "soft-deleted 상태가 아니어서 영구 삭제할 수 "
            + "없습니다. id=" + id));
    String expected = board.displayName();
    if (!expected.equals(typedName)) {      // ← 복붙
        throw new TypedNameMismatchException(expected, typedName);
    }
    purge(board);
}

AFTER (verifier 위임 — SSOT)
