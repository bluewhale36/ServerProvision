§0. 한눈 요약 — 무엇을, 왜, 어디에 찍는가

현행 로깅의 통증은 "무엇을 찍느냐"보다 ① 요청 상관관계(requestId) 부재 · ② 사용자 액션 실패·전이 성공의 무로깅 · ③ 구조화 컨텍스트 미방출 · ④ 로그 설정 전무의 복합이다. 본 계획은 survey 의 추상 권고(L1~L4)를 전 영역 169개 actionable 로그 포인트로 구체화한다 — 각 포인트마다 위치·레벨·eventCode·즉시 적용 가능한 메시지 템플릿·마스킹을 확정했다. 추가 의존성 0(Spring Boot 4 네이티브 구조적 로깅 + 필터 1개 + advice/서비스 로깅 보강).

§2. 로깅 규약 — canonical 형식 · eventCode 택소노미 · 레벨 결정트리 · MDC

① canonical 메시지 형식 (불가침)

log.<level>("[<eventCode>] resource={TYPE}#{id} <key=value ...> outcome=<result>", args..., [ex])
  • eventCode = <area>.<operation> (예 bios.upload.complete, lifecycle.softDelete, guard.parentInactive).
  • resource={TYPE}#{id} 가 항상 선두. 자원이 없으면 가장 식별력 높은 키(board=/nudgeId=/file=)로 시작.
  • 모든 컨텍스트는 영문 key=value — grep/파싱 가능. 한국어 문장 금지.
  • requestId 는 메시지 본문이 아니라 logback %X{requestId} 패턴으로 자동 주입.
  • ERROR 는 반드시 마지막 인자로 throwable 전달 (log.error("...", ex)). e.getMessage() 만 = 스택 소실 금지.

  • L4 prod 프로파일 ECS JSON + AsyncAppender. %replace 마스킹: password=*** / token 미출력
  • 마스킹 회귀 — SubprogramNudgeService token · BmcPasswordService userName · HMAC secret · markerSignature 가 로그/SSR 어디에도 평문 없음(테스트 고정)
