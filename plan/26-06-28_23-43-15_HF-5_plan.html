§1. 현재 상태 · 목표 (report 정정 반영)
report 정정 — R2-2-2/3 조사 report 는 board restore 409 를 "raw JSON 노출"로 적었으나, CP1 적대 검증에서 틀린 단정으로 확인됐다. native form POST 는 Accept: text/html 을 보내고, DuplicateBoardModelException → FieldBoundConflictException → ConflictException → DomainException 이 WebExceptionHandler.handleDomain(produces=text/html)에 흡수돼 error.html(스타일된 전체 페이지 에러)로 렌더된다. raw JSON 아님. 5월의 raw JSON 증상은 R1-4-1 v3 + R2-3 의 dual-advice produces 라우팅이 이미 해소했다.

실제 결함 — 렌더 비일관 (경미하나 실재)

BoardModel restore (현재)형제 5개 (OS·ISO·BIOS·BMC·Subprogram)
폼 마커data-async-submit 없음 (board/list.html:193, grep -c=0)있음 (os:199 / bios / bmc / subprogram / iso)
제출form.requestSubmit() → native 내비게이션submitAsync(fetch)
409 시전체 페이지 이탈 → error.html (목록에서 벗어남, "이전 페이지로" 눌러 복귀)inline ErrorModal (목록 페이지 유지)

충돌 시나리오(정상 흐름 도달 가능): board 'Dell R740' 삭제 → 같은 (vendor, modelName) 재등록(삭제본은 유일키 점유 안 함, nudge 없음) → 휴지통 옛 board [복구] → BoardModelLifecycleService:124 가 DuplicateBoardModelException(409). cascade 자식복구 충돌(:131)도 같은 폼.

목표 — board restore 폼을 형제 5개와 동일하게 data-async-submit 으로 맞춰, 409 가 전체 페이지 이탈 대신 inline ErrorModal 로 뜨게 한다. 6개 restore 표면의 렌더 동작을 단일화(대칭 회복). 1줄 변경, 백엔드/advice/정책 무변경.

왜 안전한가 — 4가지 실측 확인

위험확인 결과
board 페이지가 async 스택을 로드하나?예. error-modal.js = layout.html:56 commonHead(board 사용). confirm-modal-base.js+confirm-restore.js = confirm-modals :: lazyShell(board list.html:239 include). ErrorModal/submitAsync/approveAndSubmit 모두 가용.
성공 경로가 깨지나?아니오. board restore 는 redirect:/management/board?select…(302), OS restore 도 동일(302, flash 없음). submitAsync 가 redirect:'follow' 로 302 추종 → resp.redirected → onSuccess → reload. OS 가 이미 async+cascade 로 동일 동작.
cascade(자식 동반복구) 옵션이 유실되나?아니오. confirm-restore.js 가 writeHiddenCascade 로 hidden input 채움 → submitAsync 의 new FormData(form) 이 포함 전송. native·async 둘 다 동일 전달.
409 가 ErrorModal 로 가나?예. async 면 fetch(Accept: application/json…) → ApiExceptionHandler.handleFieldBoundConflict(:158, produces=json) → 409 JSON → onRejected → ErrorModal.show({message,status}). (native 의 text/html → error.html 경로와 분리.)
핵심 — 이 변경은 board restore 의 제출 채널을 native(text/html)에서 XHR(application/json)로 바꾸는 효과다. 그 결과 409 가 WebExceptionHandler→error.html(전체페이지) 대신 ApiExceptionHandler→JSON→ErrorModal(inline)로 라우팅된다. advice 코드는 한 줄도 안 바뀌고, 이미 존재하는 두 채널 중 어디로 갈지가 바뀔 뿐.
🎬 라이브 — restore 409 : 전체페이지 이탈 vs inline 모달
상태 휴지통/목록의 옛 'Dell R740' [복구] 클릭 → 활성 동일키 충돌(409)
localhost/management/board?includeDeleted=true

수정 전: 복구 클릭 → 주소가 /restore 로 바뀌며 전체 페이지 에러(error.html) 로 이탈("이전 페이지로" 눌러야 복귀). 수정 후: 페이지 그대로 + inline ErrorModal 로 사유 안내.
