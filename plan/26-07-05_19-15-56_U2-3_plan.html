이 slice 의 본질은 "다형 processList 가 DB 에 어떤 모양으로 앉는가" 다(분해 report Q1 의 최종 답). 아래에서 단계를 토글하면 채택안(D1 — 행 기반)의 setting_definition + setting_process 두 테이블에 행이 어떻게 분해·적재되는지, 그리고 U2-2-3 이 쓸 in-use 질의가 왜 공짜로 생기는지(BASIC_SETTING 행의 template FK exists)를 체험한다. payload 는 U2-1 계약(wire flat + 판별자)의 무변환 직렬화 — 계약=저장 SSOT.

단계 구성 :
단계 토글로 행 분해가 바뀌는 것을 확인하세요. 타입당 1행(UNIQUE) — 순서는 enum 선언 순으로 재조립됩니다.

§3. URL / 데이터 흐름

URL 표면 무변 — 스텁이 실구현으로 교체될 뿐 U2-1 의 8 엔드포인트 계약 그대로. 달라지는 것은 이면의 데이터 흐름:

[생성]  POST /provisioning/setting → 다형 역직렬화·@Valid → selector id 실존 가드(management 조회)
        → SettingDefinition 1행 + SettingProcess N행(process_type UNIQUE, payload_json) INSERT → 201
[조회]  목록 = definition + 단계 타입 요약(process_type 컬럼 질의) · 상세/수정 pre-fill = payload 재조립(SettingProcessType 선언 순)
[수정]  PUT /{id} → 전체 교체(definition 갱신 + process 행 delete-insert — 단계 순서·구성이 자유 변형되므로 diff 병합보다 단순·정확)
[선택지] GET /new·/{id}/edit → management repository 실조회(BoardModel→BIOS/BMC, OSMetadata→환경/패키지그룹)
[삭제]  본 slice 도입 안 함 — U2-1 표면에 없었음(정의서 삭제는 서버 할당 참조가 생기는 U3 와 함께 §11)

§4. 도메인 모델 (SSOT) + 설계 대안

D1 — 저장 구조 (분해 report Q1 의 최종 답): 행 기반 채택

① definition + process 행 ✓ 채택

  • CLAUDE.md 아키텍처 명세 그대로("SettingDefinition + SettingProcess 다형")
  • U2-2-3 의 기반: BASIC_SETTING 행에 template FK 컬럼 추가 + in-use exists 쿼리가 스키마 재작업 없이 얹힌다
  • 단계 타입 요약(목록 배지)이 JSON 파싱 없이 process_type 질의로
  • payload = Request 직렬화 원문(판별자 포함) — 계약=저장 SSOT, 변환 계층 0

② 단일 process JSON 컬럼 (레거시)

  • 테이블 1개 단순성
  • 탈락 — U2-2-3 의 in-use 판정이 JSON 역참조(LIKE/JSON_SEARCH)가 되어 FK 무결성·인덱스 상실. 목록 요약도 매번 역직렬화. 지금 단순 이득이 두 슬라이스 뒤 재작업으로 돌아옴

③ 저장 전용 도메인 모델 (레거시 resolve 방식)

  • Request→도메인 변환 후 저장
  • 탈락 — 해석기(Builder/Resolver)가 execution 이연 상태라 소비자 없는 계층 + Request 와 필드 중복(중복 금지). 실행 해석은 Stage 4 가 payload 원문에서 수행

process_type 컬럼과 payload 내 판별자의 중복 허용 근거: 전자는 질의·인덱스용 파생(쓰기 시 processType() accessor 에서 단방향 생성), 후자가 SSOT. 파생 컬럼은 읽기 경로에서 payload 재조립에 관여하지 않으므로 drift 가 정합성 사고로 이어질 표면이 없다.

D2 — SettingStatus 재심사 (Q1 · 사용자 결정)

권고: 상태 제거 + 수정 자유

  • 정의서 = 재사용 템플릿 — 실행 상태(IN_PROGRESS/COMPLETED/FAILED…)는 서버 개별 실행의 속성이고 이미 execution 의 ProvisioningProgress/SetupStep 이 소유. 정의서에 실행 상태를 두면 "여러 서버가 같은 정의서" 순간 거짓이 된다
  • BIOS 템플릿과 철학 통일: 수정 자유(실행 시점 반영) + 삭제는 참조 차단(U3 서버 할당 시) — PENDING 가드(409)·상태 전이 전부 불요
  • 파급: SettingStatus enum·Response 의 status 필드·목록 상태 배지 제거(계약 축소 — 스텁 시절이라 저비용)

대안: 레거시 6종 유지 + PENDING 가드

  • Notion 본문의 기존 서술
  • 난점 — 상태를 전이시킬 주체가 없다(실행이 정의서를 소유하지 않음). CANCELED/PARTIAL_SUCCESS 는 영원히 미도달 상태로 남아 죽은 enum. PENDING 가드는 "수정 자유" 템플릿 철학과 충돌

D5선택지 실조회 — setting 서비스가 management repository 직접 주입biossetting FK 전환 선례. 조회 전용(쓰기 없음). soft-delete 제외 + enabled 우선 정렬은 구현 판단.
D6selector 실존 가드 = management 예외 재사용(BoardModelNotFoundException 등 404)UI select 가 정상 흐름 차단 — direct POST 안전망. AUTO/LATEST 는 id 없으므로 가드 불발(자연).
D7sort 컬럼 없음 + UNIQUE(definition_id, process_type) + 요청-로컬 @AssertTrue(타입 중복 금지 400)사용자 지적 수용 — 실행 순서는 타입 내장(execution 소유), 표시·재조립은 enum 선언 순으로 결정적. wire 배열 순서는 도메인 정보가 아님. UNIQUE 가 U2-1 폼의 "타입당 1개" UI 규칙에 DB 안전망을 부여(SSOT). 완화는 추후 비파괴.

적대적 자기비판 — ① ProcessPayloadConverter 가 Boot ObjectMapper 를 주입받을 수 있나? JPA Converter 는 Hibernate 가 생성하면 DI 불가 — Spring 관리 Converter 는 Boot 가 지원(레거시 선례 실증: SettingProcessConverter 가 ObjectMapper 주입 AttributeConverter 였음). 실패 시 대안: 서비스 계층 직렬화(Converter 는 String 통과). CP2 에서 실증 후 확정. ② 수정 delete-insert 가 U2-2-3 의 in-use FK 와 충돌? — 행 교체 시 FK 대상(템플릿)이 잠깐 사라졌다 재삽입되는 것은 무해(같은 트랜잭션). ③ OS 단계 payload 저장 허용이 U2-4 침범? — 저장 왕복은 U2-1 계약이 이미 보장, U2-4 는 검증·폼 완성. 막으면 퇴행. ④ status 제거 시 Response 계약 파괴? — 스텁 시절 필드라 소비자는 자사 화면뿐. 배지 제거로 목록이 단순해짐.

#질문권고
Q1SettingStatus 처리 — 제거(상태 없음+수정 자유) vs 레거시 6종 유지+PENDING 가드제거 권고(§4 D2) — 정의서=템플릿, 실행 상태는 execution 소유. BIOS 템플릿 철학 통일.

의도된 잔존 (+ 해소 시점)

  • OS 단계는 저장만 되고 3차 검증·폼 완성 없음 → U2-4.
  • BASIC_SETTING payload 는 여전히 빈 스텁({"type":"BASIC_SETTING"}) → U2-2-3 에서 biosSettingTemplateIds 실체화 + 조인 테이블 setting_process_bios_template(process CASCADE/template RESTRICT — 복수 참조라 단일 컬럼 불가) 추가, payload=SSOT·조인=파생 인덱스(§3 DDL 주석 예고).
  • 정의서 삭제 표면 없음(U2-1 부터) → 서버 할당 참조가 생기는 U3 에서 참조 차단과 함께.
