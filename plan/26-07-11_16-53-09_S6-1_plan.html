§2. 요구사항 · 설계 결정 (D1~D4, 대안 비교)

D1. 상수 2종 완전 삭제 채택

RESOURCE_RENAMED·RESOURCE_RENAMED_ORPHAN 을 enum 에서 제거하고 javadoc 의 "MK3 신규" 섹션을 실상(S6-2 가 구현할 4종만 미구현 잔존)에 맞춰 재편한다. 12종 → 10종.
비채택: ⓐ @Deprecated 잔존 — 탐지도 영속 이력도 없는 상수는 호환성 부담이 0이라 deprecation 단계가 무의미한 의식(over-abstraction 의 시간 축 버전). ⓑ 미사용 방치 — R9-2 가 문구까지 부여해 둔 상태라 "언젠가 구현될 것"이라는 오독을 계속 생산(v2 보고서가 지적한 혼란의 원천).

D2. 4a 본체 존재 검사 — 서명 검증 통과 후, deep 해시 전 채택

마커가 정상(파싱·서명 통과)인데 본체가 없으면 quick scan 에서 즉시 MISSING(detail: "마커는 있으나 본체 파일 부재 — 파일명 변경 또는 삭제 가능성"). 비용 = active 자원당 Files.exists 계열 1회.
검사 위치를 서명 검증 뒤에 두는 이유: 마커 파싱 실패·서명 불일치는 변조 의심(보안 신호)이라 자원 부재(운영 신호)보다 먼저 노출되어야 한다 — 둘 다 해당하면 SIGNATURE_INVALID 가 뜨고, 마커를 정상화한 다음 스캔에서 MISSING 이 드러나는 순차 발견이 자연스럽다. deep 분기의 recomputed.isEmpty() → MISSING 도 같은 순서(서명 뒤)라 선례 정합.
비채택: matched 직후(서명 검증 전) 본체 검사 — 변조된 마커 + 본체 부재 조합에서 MISSING 이 SIGNATURE_INVALID 를 가려 보안 신호가 침묵.

D3. 본체 술어는 4b 와 동일 layout-aware — 공용 helper 로 추출 채택

SIDECAR = Files.isRegularFile(path) / IN_TREE = Files.isDirectory(path). 4b(:388/:396)에 같은 술어가 이미 2벌 있으므로 private helper 1개(resourceBodyExists(Path, MarkerLayout))로 추출해 4a·4b 세 사용처가 공유한다(중복 금지 불가침).
참고: IN_TREE 는 마커(디렉토리/.provision.json)가 트리 내부라 "마커 존재 = 디렉토리 존재"가 거의 자명 — 검사의 실효는 SIDECAR(ISO 등 단일 파일 자원)에 있다. 그래도 layout 분기 없이 helper 한 벌로 균일 처리하는 쪽이 저렴하다.
비채택: 단순 Files.exists — SIDECAR 본체 자리에 디렉토리가 있는 비정상까지 정상 통과시켜 4b 술어와 판정이 갈라진다.

D4. MISSING 권장 조치 문구 보강 채택

recommendedAction: "백업·이동 여부를 확인해…" → "백업·이동·이름 변경 여부를 확인해 파일을 원래 경로·이름으로 복원한 뒤 다시 점검하세요. 시스템이 자동 복구할 수 없습니다." — RENAMED 정리 후 리네임 사용자가 받을 안내를 MISSING 이 승계한다. label/description 은 불변(계약 유지).

SIDECAR 자원(OS_ISO #42, DB.path=/opt/provisioning/iso/rocky.iso)의 FS 상태를 토글하면 현행 코드와 S6-1 적용 후의 스캔 판정이 나란히 재현된다. 정리 논거(동시 리네임=PATH_DRIFT 불변)와 보강 효과(마커 잔존=quick MISSING 조기화)를 직접 확인.

스캔 종류 시나리오 프리셋

검증 포인트: ① "파일+마커 동시 리네임" — 현행·S6-1 모두 PATH_DRIFT(적용 버튼 제공): RESOURCE_RENAMED 없이도 동작 동일, 정리 안전 ② "파일명만 변경" — 현행 quick 은 이상 없음(침묵), deep 에서야 MISSING. S6-1 은 quick 부터 MISSING + 보강 문구 ③ 변조 마커 + 본체 부재를 함께 켜면 SIGNATURE_INVALID 가 우선(D2 순서).

§8. 부산물 · 주의
  • 보고 증가 가능성(의도된 동작 변화) — 지금까지 deep 에서만 드러나던 "마커 有·본체 無" 자원이 quick(기본 1h) 마다 MISSING 으로 보고된다. 침묵이 버그였고 보고가 정상이므로 회귀가 아니라 개선이나, CP5 에서 기존 환경에 이런 자원이 있으면 새 drift 가 나타날 수 있음을 사용자에게 예고.
  • RENAMED 참조 잔존처는 이력 문서뿐 — plan/report html·R9-2 plan 의 JS 데이터는 시점 기록이라 수정하지 않는다. 소스(main+test) 기준 잔존 0 이 완료 조건.
  • S6 단계 plan §4 의 "AUTO = {…RESOURCE_RENAMED…} 삭제 예정" 서술이 본 슬라이스로 실현 — 단계 plan 도 이력이라 불변.
  • deep 분기의 recomputed.isEmpty() → MISSING 은 유지 — 4a 본체 검사를 통과했는데 해시 재계산이 실패하는 IO 오류 케이스가 잔존하기 때문(detail 문구로 구분).
