본 슬라이스는 ISO 무결성 검증 도메인 응집. R1-4-1 의 IsoLifecycleService / R1-4-2 의 IsoRegistrationService 와 동형 패턴이지만 무결성은 ISO 만의 고유 흐름 + Markable 추상화의 일부라 별도 service.

  4. IsoJobController 의 호출처 갱신 — integrityStatus 의 osMetadataService.findIntegrityStatus → isoIntegrityService.findStatus
  5. 외부 endpoint 동일 — Controller URL (/management/os/{osId}/iso/{isoId}/verify · /integrity-status) / HTTP method / response shape / Job 이벤트 100% 동일
  6. 의존 그래프 단방향 — IsoIntegrityService → ISORepository + OSMetadataRepository + ProvisionMarkerService. OSMetadataService 와 무의존. IsoVerificationLauncher 의 OSMetadataService 의존은 본 슬라이스에서 제거 (단, OSMetadataService 의 다른 흐름이 Launcher 에서 호출되지 않는 한)

🎬 미리보기 — IntegrityStatus 5종 도식 + dependency 그래프 before/after

본 슬라이스가 응집하는 도메인 의미를 IntegrityStatus 5종 으로 먼저 보여주고, 의존 그래프 비대칭을 비교.

5-1. IntegrityStatus 5종 — 본 service 의 핵심 도메인 의미

ORIGINAL

marker 존재 + 서명 유효 + manifestHash 일치. 원본 보존 상태. 뱃지 = 초록.

MARKER_MISSING

sidecar .provision.json 파일 자체가 없음. 재발급 필요. 뱃지 = 노랑.

SIGNATURE_INVALID

marker 존재하나 HMAC 서명 재계산 불일치. 이식/변조 의심. 뱃지 = 주황.

TAMPERED

서명 유효 + 파일 부재 OR manifestHash 불일치. 외부 수정. 뱃지 = 빨강.

NOT_VERIFIED

아직 검증하지 않은 초기 상태. 목록 뷰 기본값. 뱃지 = 회색.

verifyIntegrity 의 단계별 판정 — 위 5종 enum 으로 도메인 의미가 표현됨. 본 service 의 핵심 책임은 "디스크 + 마커 + DB 의 3-way 비교 결과를 enum 으로 환원". 분기문은 enum 매칭만 — 새 상태가 추가되면 enum + applyStatus 만 확장 (CLAUDE.md "조건 분기문 legacy 무분별 확장 절대 지양" 부합).

3.4 verify 메서드 본문 (그대로 이동, 변경 0)

public IntegrityStatus verify(Long osMetadataId, Long isoId) {
    requireActiveImage(osMetadataId);
    ISO iso = isoRepository.findByIdAndOsMetadata_Id(isoId, osMetadataId)
            .orElseThrow(() -> new ISONotFoundException(osMetadataId, isoId));
    if (iso.isDeleted()) {
        throw new IllegalOSMetadataStateException("삭제된 ISO 는 검증할 수 없습니다. isoId=" + isoId);
    }
    Path target = Path.of(iso.getIsoPath());
    MarkerContent marker;
    try {
        marker = markerService.read(target, MarkerLayout.SIDECAR);
    } catch (MarkerMissingException e) {
        return IntegrityStatus.MARKER_MISSING;
    }
    if (!markerService.verifySignature(marker)) return IntegrityStatus.SIGNATURE_INVALID;
    if (!Files.isRegularFile(target))            return IntegrityStatus.TAMPERED;
    String recomputed = computeFileSha256(target);
    if (!markerService.verifyManifestHash(marker, recomputed)) return IntegrityStatus.TAMPERED;
    return IntegrityStatus.ORIGINAL;
}

D-1. updateISO 처리 결정 — 사용자 결정 필요 (3 옵션)

OSMetadataService.updateISO 는 등록 (creation) 도 무결성 (integrity) 도 아닌 "ISO 후속 수정" 책임. 본 R1-4-3 의 검토 결과 세 옵션 :

옵션이동 위치근거비용 / 우려
A 권장 IsoIntegrityService (updateIsoPath) isoPath 변경은 marker 재발급이 필요할 수 있음 (sidecar 가 옛 경로에 묶여 있음). 무결성 service 가 마커와 함께 책임 응집 — "경로 바뀌면 마커도 같이 다뤄야 한다" 는 멘탈 모델. service 책임이 "검증" + "편집" 두 축으로 확장 — 책임 응집 단일성 약화. 단, 마커가 공통 인프라라 자연스러움.
B IsoLifecycleService (update) 편집은 lifecycle 의 한 단계 (등록 → 편집 → 삭제). LifecycleService interface 의 추가 default 메서드 후보. 4 도메인 (OS/BIOS/BMC/Subprogram) 동형 적용 가능. 현 LifecycleService interface 는 toggle / softDelete / restore 등 상태 전이만 표준화 — update 는 도메인별 DTO 가 달라 추상화 자연성 떨어짐. R2 의 cross-domain 추상화 시점에 재검토.
C OSMetadataService 잔류 R2 후속 슬라이스로 미룸. 본 슬라이스는 무결성만 응집해 scope 명확화. R1-4 묶음 종료 후에도 OSMetadataService 에 ISO 책임 1 가 잔존 — "ISO 책임 100% 분리" 의도와 어긋남.
Claude 권장 — 옵션 A. 이유 : (1) updateISO 의 PathPolicyService 의존이 무결성의 sidecar 경로 정합과 직접 연결, (2) 미래에 isoPath 변경 시 marker 재발급 자동 트리거가 필요해질 가능성 — IsoIntegrityService 안에 두면 자연스럽게 연결, (3) "ISO 책임 100% 분리" 의도 충족 (R1-4 묶음 종료 의미).

사용자가 옵션 B 또는 C 를 선택하면 §4 매트릭스 / §6 시나리오 (#18) / §10 Critical Files 표를 재산정.

D-2. computeFileSha256 helper 사본 정책 (R1-3 / R1-4-1 / R1-4-2 패턴 그대로)

D-4. reissueMarker 신설 여부 — 신설 안 함 (Notion highlight 명시)

Notion 페이지 highlight 의 결정 사항 : "마커 발급 (finalizeMarker) — R1-4-2 의 등록 흐름 종착점이라 R1-4-2 잔류 (reissueMarker 는 본 슬라이스 — 재발급은 무결성 회복의 일부)". 본 plan 작성 시점 코드 확인 결과 :

  • 현 OSMetadataService 에 service-level reissueMarker(Long, Long) 메서드 부재.
  • ISO.reissueMarker(String manifestHash, String markerSignature) 는 entity 의 Markable 인터페이스 구현 — entity 도메인 메서드 (이동 대상 아님).
  • 일괄 마커 재발급은 PathReconciliationService 가 별도 도메인에서 관할.

결정 — 본 슬라이스에서 신규 reissueMarker service 메서드를 신설하지 않는다. 이유 : (1) 현 코드에 호출처가 없어 dead code 가 됨, (2) 일괄 재발급은 reconciliation 도메인에서 처리 중이라 책임 중복, (3) 무결성 회복 (MARKER_MISSING / SIGNATURE_INVALID 상태에서의 재발급) 흐름이 UI 에 노출되지 않은 시점. 미래에 "재발급" 버튼이 UI 에 노출되면 별도 슬라이스에서 신설.
