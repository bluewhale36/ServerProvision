§2. 페이지 요구사항 (확정 규칙)
  1. displayMessage → enum 흡수. 4× statusMessage 를 IntegrityStatus.getDisplayMessage() 단일 소스로. 순수 string 이라 enum 자기 책임(도메인 설명)에 정합 — 외부 의존 0.
  2. applyStatus → 공통 reporter 추출 (enum 오염 회피). Job stage 전이(backgroundJobService · IntegrityVerificationStage)는 인프라 부수효과라 enum 에 넣지 않는다. 신규 @Component IntegrityJobReporter 가 흡수, 4 launcher 가 위임.
  3. badgeClass 정합. IntegrityStatusView.badgeClass 는 이미 단일 소스(common/view) — 중복 대상 아님. presentation(CSS) 책임이라 View 에 유지. enum 의 displayMessage(도메인) ↔ View 의 badgeClass(표현) 레이어 분리 확인 + 둘 다 5 상수 exhaustive 임을 회귀로 고정.
  4. 동작 100% 보존 (불가침). 4 launcher 의 verify 결과(Job 성공/실패 · stage 전이 · 메시지 텍스트 · badge)는 글자단위 동일하게 유지. 순수 구조 refactor.
  5. Launcher/Runner 분리 안 함. 원안 R5-2a 폐기 — 4 launcher 의 register+runAsync 단일클래스 구조 유지.
🎬 라이브 미리보기 — status 를 골라 message/badge/Job전이 + Open/Closed 비용을 직접 확인

§3. 데이터 흐름 (URL 변경 없음 — 내부 refactor)

R5-2 는 신규 엔드포인트/페이지가 없다. 각 도메인의 기존 verify 엔드포인트(예 POST /management/bios/.../verify)가 그대로 유지되며, 그 뒤 @Async 후처리 경로의 중복만 정리한다.

현재 (도메인마다 동일 사본)
@Async runAsync(jobId, ...) {
  status = service.verifyAndRecord(...)
  applyStatus(jobId, status)   // ← 4벌 복붙
}
private applyStatus(jobId, status){ switch(status){...} } // 4벌
private statusMessage(status){ switch(status){...} }       // 4벌
개편 후 (단일 소스 위임)
@Async runAsync(jobId, ...) {
  status = service.verifyAndRecord(...)
  integrityJobReporter.report(jobId, status) // 위임 1줄
}
// statusMessage 삭제 → IntegrityStatus.getDisplayMessage()
// applyStatus 삭제 → IntegrityJobReporter (공통 @Component)

§4. 도메인 모델 + 결정 (D1~D3)

IntegrityJobReporter (신규 @Component, global/job)

@Component
@RequiredArgsConstructor
public class IntegrityJobReporter {
    private final BackgroundJobService backgroundJobService;
    public void report(String jobId, IntegrityStatus status) {
        switch (status) {  // ← 4벌 → 1벌 (Job 전이 SSOT)
            case MARKER_MISSING, SIGNATURE_INVALID -> backgroundJobService.fail(jobId, status.getDisplayMessage());
            case TAMPERED -> { ... fail(status.getDisplayMessage()); }
            case ORIGINAL -> { ... complete(jobId); }
            case NOT_VERIFIED -> backgroundJobService.fail(jobId, "검증 결과를 받지 못했습니다.");
        }
    }
}

위치 : global/job/ (또는 global/job/service/) — BackgroundJobService·IntegrityVerificationStage 의존이 둘 다 job 이고 입력만 global/marker.IntegrityStatus. import: IntegrityStatus 는 이동 후 global/marker 에서.

§8. 예상 부산물 / 주의
  • applyStatus 를 enum 에 넣지 않는다 (불가침). Job stage 전이는 인프라 부수효과 — 도메인 enum 에 넣으면 IntegrityStatus 가 BackgroundJobService·IntegrityVerificationStage 에 의존하게 되어 오염. displayMessage(순수 데이터)만 enum.
  • IntegrityStatus → global/marker 이동 포함 (사용자 요청, D4). 4 도메인 공용 마커 검증 결과가 bios/vo 에 있던 smell 해소. 이동 대상은 enum 1개뿐 — View(presentation)·Stage(job)·Response(dto)·Reporter(job)는 각자 레이어 유지. import churn = 15 참조 파일의 경로 한 줄씩(로직 변경 0, IDE move-refactor). R5-2 가 이미 enum + 4 launcher 를 편집하므로 동반 비용 최소.
  • (범위 밖, 향후 후보) verifyIntegrity 자체의 도메인별 중복. BiosService.verifyIntegrity / IsoIntegrityService / BmcService / SubprogramService 가 모두 "마커 read → verifySignature → verifyManifestHash → IntegrityStatus 매핑"을 반복. IntegrityStatus 가 global/marker 로 가면 ProvisionMarkerService.verify(Markable) → IntegrityStatus 단일화 여지가 생긴다 — 단 blast radius 큼, R5-2 밖(후속 슬라이스).
  • badgeClass 는 그대로. 이미 단일 소스라 손대지 않음. "정합"은 displayMessage/badgeClass 가 각각 exhaustive 단일 소스임을 확인하는 것까지.
  • 동작 불변 보증. reporter 의 switch 는 현행 applyStatus 와 호출 시퀀스가 글자단위 동일해야 함 — CP4 단위 테스트가 InOrder 로 고정.
  • CLAUDE.md 수정 없음. 인벤토리 코드/아키텍처 변동 없는 내부 refactor.
