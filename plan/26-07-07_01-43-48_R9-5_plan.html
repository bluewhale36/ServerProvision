§2. 요구사항 · 설계 결정 (D1~D5)

D1. 스냅샷 컬럼 — 이름은 display_name (PurgeLog 선례와 통일) 채택

스캔 시점에 Markable.displayName() 을 기록. 자원이 나중에 purge 되어도 이름이 남고, 읽기 경로에 도메인 횡단 조회가 없다.
비채택: ⓐ 렌더 시 실시간 resolve — purge 후 소실 + 페이지네이션 N+1 + append-only 설계(Drift.java:13-16)와 불일치. ⓑ 컬럼명 resourceName — 동일 개념에 두 용어(PurgeLog 는 display_name).

D2. nullable + 백필 없음 — 템플릿 fallback 채택

기존 행은 null → 화면은 종전 표기("TYPE #id")로 fallback. drift 는 보고서 FIFO retention(기본 100건)의 소량 append-only 라 다음 스캔부터 자연 해소.
비채택: NOT NULL + 백필 — 기존 행의 실명을 재구성하려면 자원 조회가 필요한데 purge 된 자원은 불가. null 허용이 정직.

D3. 기록 지점 3곳 — buildDrift 중앙 + ORPHAN fallback 채택

ⓐ buildDrift(Markable …) 에 .displayName(resource.displayName()) 1줄 — 4a/4b/4c/ghost/scanForResource 경로 전부 커버. ⓑ ORPHAN(DB 매칭 없음 — Markable 부재, builder 직생성)은 마커가 가리키는 본체 파일명을 fallback 실명으로("rocky-9.6.iso"). ⓒ persistAndForcedApply 의 drift 는 scanForResource→buildDrift 산출물이라 자동 포함.

D4. 표시 — C2 주표기 실명, C3 에 '자원 이름' 행 채택

C2: 실명(null 이면 "TYPE #id") + 부라벨 resourceType. C3: '자원 이름' 행 추가(null 이면 "—(이전 점검 기록)"). REST 응답 DriftResponse 에 displayName 필드 추가(계약 확장 — 기존 필드 불변).

D5. 스키마 절차 — DDL 스크립트 산출 + 사용자 적용 + CLAUDE.md Step 9 현행화 채택

ALTER 권한 부재(실측)로 적용은 사용자 몫: sql/R9-5_drift_display_name.sql 1줄 스크립트 산출. ddl-auto=validate 라 적용 전에는 앱이 기동되지 않는다 — CP5 가이드·요약 메시지·Notion 에 최상단 명기(§8). CLAUDE.md Step 9 문구("ddl-auto=update")를 validate 절차로 갱신.

§4. 도메인 모델

Drift + displayName(String, @Column(name="display_name", length=255) nullable, 스냅샷 — 수정 없음 원칙 유지). DDL:

ALTER TABLE drift ADD COLUMN display_name VARCHAR(255) NULL AFTER resource_id;

length 255 = PurgeLog.display_name 과 동일. 스냅샷이므로 자원 개명 시 과거 보고서는 기록 시점 이름 유지(감사 로그 의미론 — PurgeLog 와 동일).
