1. 현재 상태 (실측 근거)

정상 등록 자원의 디렉토리를 — in-tree .provision.json 마커째 — 스캔 범위 안 다른 위치에 복사해 두면, 원본이 정상인 동안 스캔이 아무 신호도 내지 않는다. 침묵의 원인은 PathReconciliationService 의 두 지점이 겹쳐서다:

#지점동작 (코드 실측)
침묵1collectDiskMarkers디스크 마커 수집이 Map<MarkerKey, MarkerHit> + putIfAbsent — 같은 (resourceType, resourceId) 의 두 번째 이후 발견은 수집 단계에서 이미 버려진다 ("첫 발견 우선" 주석 존재). 어느 쪽이 남는지는 walk 순서에 달렸다.
침묵24a 분기 + (5) ORPHAN 패스원본(DB 경로)이 정상이면 4a 가 matchedMarkers.add(key) 후 continue. (5) ORPHAN 판정은 matchedMarkers 포함 key 를 건너뛰므로, 설령 사본 hit 가 살아남았어도 ORPHAN 으로도 보고되지 않는다.

이 침묵이 위험한 이유 (O-2 시나리오): 이후 원본이 유실되면 4b 분기가 그 사본을 '경로 이동됨(PATH_DRIFT)' 으로 분류하고, [적용] 은 applyDriftedPath 로 DB 경로를 사본 위치로 갱신한다. 사본이 몇 달 전 복사해 둔 낡은 버전이어도 (resourceType, resourceId) 마커 키만 맞으면 통과 — 방치된 낡은 사본이 슬그머니 정본 자리를 차지한다. 사용자가 이 상황을 인지할 기회는 원본이 살아 있는 동안에만 있는데, 그 시기에 시스템이 침묵한다.

관련 자산 현황: DriftKind 는 R9-2 에서 사용자 문구(label/description/recommendedAction)와 해결 방식(mode 3단)을 enum 필드로 SSOT 화했고, 해결은 DriftResolution 전략 bean 디스패치(1 bean = 1 kind)다. 사용자 입력을 동반하는 해결은 HASH_MISMATCH 의 HashAcceptService(전용 endpoint + typed-name 파라미터, mode=NONE — 표준 [적용] 경로 밖) 선례가 있다. drift 엔티티는 old_path/new_path 2컬럼을 보유한다(PATH_DRIFT 가 사용).

2. 요구사항 (사용자 결정 ①~④)
  1. 표면화 — 원본 정상 + 동일 (resourceType, resourceId) 마커가 DB 경로 밖에서 발견되면 별도 드리프트 상태 '자원 중복 존재' (RESOURCE_DUPLICATED) 로 보고한다. 이전 경로 = DB 기록 경로(원본), 새 경로 = 복제본 경로. displayName 스냅샷 등 기존 드리프트 생성 관례 준수.
  2. 해결 = 택일 — 모달에서 둘 중 하나를 선택해 나머지를 파일시스템에서 삭제한다.
  3. 모달 기본 선택 = DB 기록 경로(원본).
  4. 복제본 선택 시 — 원본 삭제 + DB 기록 경로를 복제본 경로로 갱신하는 작업까지 함께 진행한다.
  5. (파생 — CLAUDE.md 불가침) 신규 상수는 DriftKind 다형 체계에 합류 — controller/service 에 kind if/switch 를 늘리지 않는다. 자동 적용 대상 아님(autoApplicable=false 가 SSOT 로 표현될 것).

4. 도메인 모델 — 설계 결정 (대안 비교 포함)

D1. 탐지 우선순위 — 판정 순서를 어디에 두나

안내용판정
(a) 채택기존 4a→4b 판정 순서 무변경. 중복 판정은 4a 안에서, 원본이 완전 정상일 때만 발동한다 — "완전 정상" = 마커 존재 + 파싱 성공 + HMAC 서명 유효 + 본체 존재 (4a 의 기존 검사 4개를 모두 통과한 지점). 원본 소실 시(4b) 는 기존 PATH_DRIFT/MISSING 분류가 그대로 유효하다.사용자 결정 원문("원본이 정상인 동안")과 일치. 원본에 자체 드리프트(서명 불일치·본체 부재)가 있으면 그 신호가 우선 — 보안/소실 신호가 중복 신호에 묻히지 않는다. 원본 회복 후 다음 점검이 중복을 보고한다.
(b) 탈락원본 상태 무관하게 사본을 항상 RESOURCE_DUPLICATED 로 보고원본 소실 시 같은 사본이 PATH_DRIFT 와 RESOURCE_DUPLICATED 로 이중 보고 — 어느 카드로 해결해야 하는지 모호하고, PATH_DRIFT 의 "이동" 서사(원본이 없어야 이동)와 충돌.
(c) 탈락사본 발견 시 PATH_DRIFT 를 아예 대체(중복 kind 로 통합)PATH_DRIFT 는 자동 적용 가능(AUTO)한 확립된 흐름 — 통합하면 mv 한 자원의 자동 정합이라는 기존 가치가 사라진다. 두 상황은 원본 생존 여부로 의미가 다르다.

Deep 스캔과의 관계: HASH_MISMATCH(내용 변경)와 RESOURCE_DUPLICATED 는 독립 신호로 동시 보고될 수 있다(TRASH_MARKER_STALE "독립 신호" 선례). 중복 검사는 본체 존재 확인 직후·deep 해시 검사와 무관한 위치에 둔다.

D2. 동일 자원 복제본 3개 이상 — 행 단위 vs N지선다

안내용판정
(a) 채택복제본 경로당 drift 1행 (oldPath=원본, newPath=그 사본). 모달은 항상 2지선다(원본 vs 그 사본). 한 행에서 사본을 승격하면 남은 행들은 oldPath 가 현재 DB 경로와 달라져 stale — 해결 시도 시 stale 가드가 409 로 거절하고 다음 점검이 새 기준으로 재분류한다.drift 의 old_path/new_path 2컬럼 재사용 원칙(스키마 무변경) 충족. 카드 1장 = 결정 1개로 UX 단순. Drift 불변 설계(수정 없음)와도 정합 — N지선다는 해결마다 카드 내용 갱신이 필요해진다.
(b) 탈락자원당 drift 1행 + 모달 N지선다 (사본 목록 전체 나열)N 경로 저장이 2컬럼으로 불가 — detail 텍스트에 구겨 넣으면 해결 파라미터의 신뢰 소스가 자유 텍스트가 된다(파싱 취약). 스키마 신설(자식 테이블/JSON) 은 "기존 drift 컬럼 재사용" 원칙 위배 + HF4-4(보고서 계층) 충돌 위험. 하나 삭제 후 카드 갱신 문제도 Drift 불변 설계와 충돌.

D3. 삭제 방식 — 직접 삭제 vs 휴지통/격리 경유

안내용판정
(a) 채택모달 확인 경유 직접 삭제 (IN_TREE 는 트리 재귀 삭제, SIDECAR 는 본체+마커 2파일 삭제). [AUDIT] 로그로 삭제 경로·생존 경로·drift id 를 남긴다.사용자 결정 원문이 "파일시스템에서 삭제". 삭제 대상은 자기 DB 행이 없는 잉여 사본(또는 승격으로 참조를 잃은 구 원본)이라 복원 개념 자체가 없다.
(b) 탈락휴지통(.soft-deleted) 경유휴지통 lifecycle 은 soft-delete 된 DB 행과 결합돼 있다(trashedPath 기록·TTL 자동 정리·복원 흐름이 전부 엔티티 기준). 사본은 대응하는 DB 행이 없어 휴지통에 넣는 순간 휴지통 화면에 보이지 않고 TTL 정리도 안 되는 휴지통 안의 새 유형 orphan 이 된다 — 정합화 도구가 부정합을 생산하는 모순.
(c) 탈락ORPHAN 해결처럼 격리 구역(trashRoot/orphan/) 이동ORPHAN 격리는 "주인 없는 파일" 의 비파괴 보존이 목적. 여기는 주인이 명확하고 정본이 살아서 남는다 — 보존 가치가 없는 사본이 격리 구역에 무한 누적되며, 무엇보다 사용자 결정(삭제)과 불일치.

D4. 무인 자동 적용(auto-apply) 제외 근거 — mode 선택

안내용판정
(a) 채택DriftResolutionMode.NONE + 전용 endpoint (HASH_MISMATCH 선례 그대로). DriftResolution bean 미등록 → 표준 [적용]·forced apply·스캔 무인 적용 모두 구조적으로 도달 불가.① 해결이 택일 입력(survivor)을 요구 — 무인 실행은 입력 자체가 없다 ② 파일 삭제라는 비가역 파괴 동작 ③ 이중 의도 검증 명제(MK3): 어느 쪽이 정본인지는 운영자 의도라 시스템이 단일화할 수 없다. mode=NONE 이면 isAutoApplicable()==false 가 enum SSOT 로 자동 성립 — 별도 플래그 불요.
(b) 탈락mode=MANUAL + DriftResolution 인터페이스에 파라미터 추가표준 [적용] POST 는 파라미터가 없다 — 인터페이스 시그니처를 바꾸면 기존 8개 해결 bean 전부가 쓰지 않는 파라미터를 받는 churn. HASH_MISMATCH 가 이미 "사용자 입력 동반 해결은 전용 계약" 선례를 확립했다.

D5. 생존측 마커 재발급 필요 여부 — 실측 판정

결론: 재발급 불필요. 근거를 실측으로 확인했다:

  • MarkerContent 의 서명 대상 필드는 (resourceType, resourceId, attributes, createdAt, manifestHash) — attributes 에 절대 경로가 없다 (BIOS: boardId/version/상대경로 entrypointRelativePath, ISO: osMetadataId/originalFilename, SUBPROGRAM 동형). 마커는 위치 독립적이다.
  • 사본 마커는 원본 마커의 byte-동일 복사라 서명이 그대로 유효하고, manifestHash 는 내용 기반(트리 canonical/파일 SHA-256)이라 경로와 무관하다.
  • 선례 일치: PATH_DRIFT 해결(PathDriftResolution)도 파일은 건드리지 않고 applyDriftedPath 로 엔티티 경로만 갱신한다 — 이동한 자원의 마커를 재발급하지 않는 것이 확립된 계약.

단, 재발급 불요가 "무검증 승격" 은 아니다. 복제본 유지(승격) 직전 가드 4종을 둔다: ① 사본 마커 HMAC 서명 유효 ② 사본 마커 manifestHash == 엔티티 기록 manifestHash — 원본이 재업로드된 뒤 남은 구버전 사본의 승격(O-2 위험의 본질)을 차단한다. 내용이 다른 사본을 정본화하려면 재업로드 또는 HASH_MISMATCH 수용 흐름이 올바른 경로다 ③ 사본 본체 존재 + 원본/사본이 포함 관계가 아님(사본이 원본 트리 내부에 있으면 승격 후 원본 삭제가 사본까지 지운다 — 거절) ④ 사본 실제 바이트의 manifestHash 재계산 == 엔티티 기록 manifestHash (검수 반려로 추가 — 아래 callout).

검수 반려 이력 (2026-07-19, main 샌드박스 실측). 위 가드 ②는 사본 마커에 기록된 지문만 대조했다. 그래서 마커는 정상 최신본을 byte-동일 복사(서명 유효 + 기록 manifestHash 가 엔티티 기록값과 일치)해 두고 페이로드 바이트만 변조한 사본이 ①②③을 모두 통과했다. 재현: 정본 트리 복제 → 페이로드만 변조(마커는 원본 복사) → RESOURCE_DUPLICATED 탐지 → survivor=DUPLICATE 승격 → 302 성공으로 변조 파일이 정본으로 승격되고 원본 삭제(기대는 409).

수정. 승격 갈래 마지막에 가드 ④를 추가한다 — 사본 실제 바이트의 지문을 재계산해 엔티티 기록 지문과 대조하고, 불일치(또는 재계산 불능)면 DriftResolutionNotAllowedException.duplicateNotPromotable(...) 409. 재계산은 정밀 점검(deep scan)이 쓰는 MarkableScanner.recomputeManifestHash(Markable) 를 그대로 재사용한다(HashAcceptService·PathReconciliationService deep 경로와 동일 — 해시 규칙 중복 구현 0). SPI 시그니처는 getResourcePath() 기준으로 계산하므로, 경로만 사본으로 바꿔 끼운 읽기 전용 위임 뷰(DuplicatePathView implements Markable, 상태 변이 메서드 전부 즉시 실패)를 넘겨 6 scanner 구현 동반 수정 없이 재사용한다. 가드 ④는 저비용 가드(①②③)를 모두 통과한 맨 마지막에 둔다.

keepOriginal(원본 유지) 갈래는 재계산하지 않는다 — 어차피 삭제할 사본의 내용 진위는 결과에 영향이 없어 검증 비용이 무의미하다. 동기 수행: 승격은 운영자 확인을 거친 드문 비가역 액션이고 대용량 ISO 재계산이 수십 초 걸릴 수 있음을 감수한다(빈도가 늘면 HASH_MISMATCH 선례의 BackgroundJob 비동기 전환이 후속 후보 — D6 갱신 참조).

D6. 해결 전달 방식 — 동기 vs 비동기

채택: 동기 @Transactional 서비스(DuplicateResolveService) + 즉시 응답. 탈락: BackgroundJob 비동기 — 완료 이벤트 배선·중복 시작 차단 등 비용만 늘고 이득이 없다. survivor 파라미터는 Primitive Obsession 금지에 따라 DuplicateSurvivor enum(ORIGINAL/DUPLICATE)으로 타입화 — Spring 변환 실패가 곧 400.

검수 반려 반영 (2026-07-19). 초안은 "본 해결은 지문 재계산이 없고 파일 삭제만 빠르므로 동기로 충분" 이라 적었으나, D5 가드 ④ 추가로 승격 갈래는 이제 사본 실제 바이트의 지문을 재계산한다 — 대용량 ISO 는 수십 초가 걸릴 수 있다. 그럼에도 동기를 유지하는 이유: 승격은 운영자 확인을 거친 드문 비가역 액션이라 비동기의 완료 이벤트·중복 시작 차단 배선이 오히려 과한 비용이다. 빈도가 늘면 HASH_MISMATCH 선례의 BackgroundJob 비동기 전환이 후속 후보(코드 주석에 명기). 원본 유지 갈래는 재계산이 없어 종전대로 즉시.

D7. 트랜잭션 경계·파일 삭제 실패 시의 상태

순서가 안전장치다 (HashAcceptService 적대적 검증 선례 준수): DB 변이(승격 시 경로 갱신 + 카드 제거)를 먼저 실행하고 flush 해 낙관적 잠금 충돌을 파일 조작 전에 표면화한다. 파일 삭제는 마지막 — 실패(IOException → IllegalStateException) 시 트랜잭션이 롤백되어 드리프트 카드 유지 + (승격이었다면) DB 경로 원복 + 오류 응답. 잔여 위험은 "삭제가 도중에 끊긴 부분 잔재" 한 가지로 줄고, 그 잔재는 다음 점검이 다시 보고한다(마커가 남았으면 RESOURCE_DUPLICATED 재보고, 마커부터 지워졌으면 마커 없는 파일 — §8 알려진 한계). 역순(파일 먼저)이면 삭제는 됐는데 DB 커밋이 실패해 카드가 남고, 그 카드는 영원히 stale 이다.

D8. 경로 2필드의 사용자 라벨 — enum 다형

채택: DriftKind 에 oldPathLabel()/newPathLabel() 메서드(기본 "이전 경로"/"새 경로")를 추가하고 RESOURCE_DUPLICATED 상수만 "원본 경로 (DB 기록)"/"복제본 경로" 로 재정의(enum method-per-constant — CLAUDE.md 권장 다형). 템플릿 dt 가 이 메서드를 참조한다. 탈락1: 템플릿에 kind 비교 분기 — 사용자 문구의 SSOT 를 enum 에 모은 R9-2 원칙 역행. 탈락2: 전 상수 생성자 인자 2개 추가 — 10개 상수 전부 수정하는 churn 대비 이득 없음(재정의 1곳이 명확).

8. 부산물 / 주의
  • 병렬 슬라이스 경계 (HF4-4) — drift_report 의 보고서 저장·건수 표시 계층은 접촉하지 않는다. 본 슬라이스의 접점은 분류 로직(drifts 리스트에 행 추가)과 해결·상세 UI 뿐 — DriftReport 엔티티/리포지토리/카운트 표시 무변경.
  • 알려진 한계 1 (의도) — 마커만 복사된 사본(본체 없는 .provision.json) 은 원본 정상 시 계속 침묵한다. "복제본" 의 의미(실체 있는 사본)를 지키고 모달 택일의 전제(양쪽 다 실물)를 깨지 않기 위한 스코프 결정. 원본 소실 시엔 기존 4b 가 MISSING("마커만 이동 가능성") 으로 이미 잡는다. 표면화가 필요해지면 후속 백로그(§11).
  • 알려진 한계 2 (기존 유지) — ORPHAN·ESCAPE 분류는 여전히 key 당 첫 hit 만 사용한다(다중 orphan 사본 중 1곳만 보고). collectDiskMarkers 의 List 화로 수집은 되지만 소비를 바꾸는 것은 본 슬라이스 범위 밖 — 행동 변화를 중복 탐지에 한정해 회귀 면적을 최소화한다.
  • scan root 중첩 dedupe — 중첩된 scan root 가 같은 마커를 두 번 walk 하는 경우가 있어(부모/자식 root 동시 등록), hit 목록은 정규화 경로 기준으로 dedupe 한다. 종전엔 putIfAbsent 가 우연히 막아주던 것을 명시 가드로 승격.
  • 단일 자원 스캔(scanForResource) 무변경 — softDelete reject saga 전용 경로는 "DB 위치 마커 부재" 에서 출발하므로 중복(원본 정상) 상황과 교집합이 없다.
  • recheckable=false — [다시 점검] 축은 부여하지 않는다(PATH_DRIFT 도 false). 사본을 수동으로 지운 사용자는 [보고 닫기] 또는 다음 점검으로 해소. 재확인 지원은 §11 후보.
  • CLAUDE.md/AGENTS.md 수정 없음 · 커밋 금지 준수 · execution/·provisioning/ 무접촉.
