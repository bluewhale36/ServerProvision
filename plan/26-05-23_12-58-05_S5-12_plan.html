0.1 사용자 액션 직후 OS 상세 패널

사용자가 ISO 등록 폼 제출 → foreground 파일 업로드 → 응답 받고 OS 목록 페이지로 redirect. 이 시점에 background job 은 hash 계산 / 중복 검사 / 메타 저장 진행 중. DB 영속 전이라 새 ISO 가 아직 목록에 안 보임.

방금 등록한 minimal.iso 가 안 보임

After — 등록중 placeholder

/opt/iso/rocky/9/minimal.iso 등록 중

0.2 placeholder 의 시각 특성

  • opacity 0.55 + 회색 배경 — 정상 ISO 와 명확 분리
  • "등록 중" 배지 — blue 톤 (chunk progress bar 의 RUNNING 색상과 일관)
  • 액션 버튼 (토글 / 삭제 / 무결성 검증 등) 모두 비활성화 — placeholder 는 DB 영속 전이라 의미 없음
  • details 가 클릭되어 펼쳐져도 안에는 "등록 중" 안내만 표시
  • BackgroundJob ISO_REGISTRATION 완료 시 bgjob:completed 이벤트 → 페이지 자동 reload → placeholder 가 정상 ISO 로 교체
핵심 가치 사용자 액션 직후 "내가 등록한 자원이 시스템에 들어왔다" 시각 확인. background job 알림 패널 + 목록 placeholder 의 이중 확인 → 등록 흐름의 투명성.
1. 현재 상태

1.1 ISO 등록 흐름 (S5-7 적용 후)

  1. 사용자 ISO 등록 폼 제출 → POST /management/os/{osId}/iso
  2. OSImageController.addISO() → OSImageService.prepareIsoRegistration() → ISO 파일 저장 + sidecar 마커 생성 (foreground)
  3. IsoRegistrationLauncher.startRegistration(prepared) → ISO_REGISTRATION background job 등록 + async runner 시작
  4. 응답 즉시 반환 → OS 목록 페이지로 redirect
  5. background job 진행 :
    • COMPUTE_HASH — 해시 계산
    • CHECK_DUPLICATE — 중복 검사
    • PERSIST_METADATA — DB 영속 (이 시점에 ISO 가 목록에 추가됨)
  6. S5-7 추가 : RHEL_BASED 면 COMPS_EXTRACTION job 도 동시 트리거

1.2 UX 문제

  • Step 4 (redirect) 시점에 DB 의 ISO 목록은 새 ISO 미반영 — Step 5.PERSIST_METADATA 전이라 안 보임
  • 알림 패널을 펼쳐야 등록 진행 확인 가능 — 사용자가 알림 패널을 안 보면 "등록 안 됐나?" 혼란
  • 새로고침 자주 → background job 부담 가중 / 사용자 행동 비효율

2. 요구사항 (확정)

2.1 Notion 명세 (S5-12 본문)

OS ISO 등록 시 파일 업로드 직후 hash 계산 등의 이후 로직이 background job 으로 빠지면서 목록 화면으로 돌아오는데, 이 때 해당 OS 의 ISO 목록에 반영되어 있지 않으므로 사용자 혼란 야기 가능성 존재. 목록에 '(등록중)' 등으로 비활성화 된 요소를 미리 채워 넣어서 이를 개선.

2.2 채택 정책

  • 출처 : BackgroundJobService.snapshot() 의 활성 ISO_REGISTRATION Job 들
  • 매칭 : metadata.osId == 현재 OSImage.id
  • 합성 시점 : OSImageService 의 ISO 목록 조립 시점 (findById / findAllGrouped)
  • 중복 회피 : DB 영속 직후 race 케이스 — 이미 같은 isoPath 의 ISO 가 base 에 있으면 placeholder 추가 안 함
  • UX : opacity 0.55 + 회색 + "등록 중" 배지 + 액션 버튼 모두 비활성화 / 부재
  • 자동 reload : 기존 bgjob:completed 이벤트 (background-jobs.js) 활용 — type 이 ISO_REGISTRATION 인 경우 페이지 reload

2.3 추가 제약 (불가침)

  • DB 변경 0 — ISO entity 그대로
  • BackgroundJob 의 metadata 구조 그대로 — 이미 osId 보유
  • 실패 (FAILED job) 시 placeholder 제거 — snapshot() 의 isActive() 필터로 자연 처리
  • 새로고침 / 페이지 이동 후에도 등록중 표시 유지 — JVM 메모리에 background job 남아있는 동안 일관

2.4 LifecycleStage placeholder 처리

ISOResponse 의 state 필드는 LifecycleStage enum. placeholder 는 아직 DB 영속 전이므로 일반 lifecycle 어휘로 표현 어색. 두 옵션 :

  • A : state = ACTIVE (영속 후 의도된 상태) + inProgress=true 로 시각 분리. 채택
  • B : LifecycleStage 에 REGISTERING 신규 상수 추가 — 영향 표면 큼, 본 슬라이스 외
3. 안 비교 (A · B · C)

안 A ✓ in-memory BackgroundJob 합성

  • OSImageService 가 ISO 목록 조립 시 BackgroundJobService.snapshot() 합성
  • DB 변경 0
  • JVM 메모리 영속과 일관 (background job 자체가 in-memory)
  • JVM 재시작 시 placeholder 사라짐 — 재시작 자체로 진행 중이던 job 도 사라짐 → 일관
  • 변경 표면 최소

안 B ISO entity 에 status 컬럼

  • ISO.status enum (PENDING / ACTIVE / FAILED) 추가
  • 등록 즉시 PENDING 으로 DB 영속, PERSIST_METADATA stage 에서 ACTIVE 전환

  • 단점 : 스키마 변경 + 마이그레이션 + ISO 라이프사이클 영향 큼
  • FAILED placeholder 의 정리 (TTL) 추가 인프라 필요

안 C 클라이언트 측 localStorage

  • 사용자 브라우저에 등록 시작 기록 + 폴링
  • 장점 : BE 변경 0
  • 단점 : 다른 브라우저 / 탭 / 사용자 / 새 디바이스에서 안 보임
  • 실패 / 완료 동기화 복잡
  • 탈락

채택 = A

  • BackgroundJob 이 이미 in-memory 라 placeholder 도 같은 수명 cycle
  • 변경 표면 최소 (ISOResponse +1 필드 + OSImageService 합성 헬퍼 + list.html 시각)

5.1 ISOResponse 변경

public record ISOResponse(
    Long id,                  // placeholder 시 null
    String isoPath,
    String description,
    boolean isEnabled,
    boolean isDeleted,
    boolean isDeprecated,
    LifecycleStage state,
    boolean extracted,
    List<String> providedEnvironmentCodes,
    int providedPackageGroupCount,
    IntegrityStatus integrityStatus,
    boolean inProgress         // ← S5-12 신규
) {
    public static ISOResponse from(ISO entity) { /* inProgress=false */ }

    /**
     * S5-12 — 진행 중 ISO_REGISTRATION job 의 placeholder. DB 영속 전이라 id 는 null.
     */
    public static ISOResponse inProgress(String isoPath) {
        return new ISOResponse(
            null, isoPath, null,
            false, false, false,
            LifecycleStage.ACTIVE,    // 영속 후 의도된 상태 (시각은 inProgress 가 우선)
            false,
            List.of(),
            0,
            IntegrityStatus.NOT_VERIFIED,
            true
        );
    }
}

5.2 OSImageService.assembleIsoResponses 신규 헬퍼

private List<ISOResponse> assembleIsoResponses(OSImage image, boolean includeDeleted) {
    List<ISOResponse> base = visibleISOs(image.getIsos(), includeDeleted).stream()
            .map(ISOResponse::from)
            .collect(Collectors.toCollection(ArrayList::new));

    Set<String> existingPaths = base.stream()
            .map(ISOResponse::isoPath)
            .collect(Collectors.toSet());

    // S5-12 — 진행 중 ISO_REGISTRATION job 합성. metadata.osId 매칭, 중복 회피.
    String osIdString = String.valueOf(image.getId());
    backgroundJobService.snapshot().stream()
            .filter(j -> j.getType() == JobType.ISO_REGISTRATION)
            .filter(j -> j.getStatus().isActive())
            .filter(j -> osIdString.equals(j.getMetadata().get("osId")))
            .filter(j -> !existingPaths.contains(j.getSubtitle()))
            .forEach(j -> base.add(ISOResponse.inProgress(j.getSubtitle())));

    return base;
}

6.1 background-jobs.js 의 자동 reload

이미 background-jobs.js 가 bgjob:completed 이벤트를 document 에 dispatch. 각 페이지에서 type 별로 reload 가능. OS list 페이지에서 이벤트 listener 가 있는지 확인 필요.

// os-list.js 에 이런 listener 가 있다면 그대로 활용
document.addEventListener('bgjob:completed', (e) => {
    if (e.detail.type === 'ISO_REGISTRATION' || e.detail.type === 'COMPS_EXTRACTION') {
        // reload 또는 partial refresh
        window.location.reload();
    }
});

이미 다른 슬라이스에서 동일 패턴 사용 중. 필요 시 본 슬라이스에서 보강.

8. 예외 / 회귀

8.1 신규 예외 — 없음

본 슬라이스는 응답 합성 로직만. 예외 계층 변동 0.

8.2 race condition 회피

  • BackgroundJob 의 PERSIST_METADATA stage 완료 → ISO entity 영속 → snapshot() 의 isActive() false → placeholder 제거. 그러나 :
    • 그 사이 사용자 요청이 들어오면 base 에 DB 영속된 ISO + snapshot 의 active job 둘 다 존재 가능 (race)
    • 중복 회피 : existingPaths.contains(j.getSubtitle()) 체크로 같은 isoPath 인 placeholder 합성 안 함
  • BackgroundJob 의 COMPLETED → COMPLETED 의 keep-after-terminal-ms 동안 snapshot 에 남음. 그러나 isActive() false 이므로 placeholder 안 됨

8.3 다중 동시 등록

한 OS 에 동시에 여러 ISO 등록 (예: dvd.iso + minimal.iso) → 모두 placeholder 추가. 정렬은 별도 처리 안 함 (snapshot() 의 시간 역순).

9. 부산물 · 주의

9.1 스코프 경계

  • BIOS / BMC / Subprogram 의 동일 패턴 적용 — 본 슬라이스 외 (별 슬라이스 후보 시 동일 헬퍼 재사용)
  • placeholder 정렬 / 그룹화 / 진행률 % 표시 — 본 슬라이스 외
  • 실패한 job 의 retry 버튼 — 본 슬라이스 외

9.2 BIOS / BMC / Subprogram 확장 후보

BIOS / BMC / Subprogram 도 동일 background job 패턴 (BIOS_UPLOAD 등 없지만 verification 등) 이 있을 수 있음. 본 슬라이스 패턴 (BackgroundJobService.snapshot() 합성) 을 그대로 재사용 가능 — 별 슬라이스에서 적용.

9.3 자동 reload 정책

ISO_REGISTRATION 의 PERSIST_METADATA 완료 시 page reload. 그러나 :

  • 사용자가 다른 작업 중일 수 있음 (입력 / 스크롤 등) → reload 가 방해 가능성
  • 대안 : reload 대신 partial AJAX refresh — 본 슬라이스 외 향후 보강
  • 현재는 reload 가 단순 + 일관 — S5-12 채택
