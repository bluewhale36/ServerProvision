4 단계 사용자 흐름

  1. 클릭 — list 의 "영구 삭제" 버튼 클릭 → submit 가로채기 (capture phase, stopImmediatePropagation)
  2. modal 표시 — title, 자원명 hint, input slot 표시. 확인 버튼은 비활성 상태로 시작
  3. 입력 검증 — input 의 모든 keystroke 마다 value === target 비교. 일치 시 확인 버튼 활성, 불일치 시 비활성. 별도 시각 피드백 없음 (버튼 disabled 상태가 곧 시그널)
  4. 제출 — 확인 클릭 → hidden typedName input 에 값 복사 → form.requestSubmit() → 백엔드 재검증 → 통과 시 hard-delete, 실패 시 400

백엔드 의사코드 (Service 측, OS 예시)

@Transactional
public void purgeWithTypedNameCheck(Long id, String typedName) {
    OSImage os = repo.findByIdIncludeDeleted(id)
        .orElseThrow(() -> new OSImageNotFoundException(id));

    // typed-name 합성 — modal 메시지의 자원명과 동일 식
    String expected = os.getOsName().getDisplayName() + " " + os.getOsVersion();

    if (!expected.equals(typedName)) {
        throw new TypedNameMismatchException(expected, typedName);
    }

    purge(id);  // 기존 service-internal hard-delete 위임
}
이중 검증의 의의 : frontend 검증은 UX (잘못된 submit 사전 차단) 용. 백엔드는 frontend 우회 / 직접 POST 공격을 대비해 다시 한번 검증. 두 검증은 typed-name 합성식이 정확히 같아야 함 — 본 슬라이스의 핵심 invariant.

8. 예상 부산물 · 주의
불가침 §조건분기문 legacy 무분별 확장 절대 지양
typed-name 검증은 service 내부 1지점 (단순 if (!expected.equals(typed)) throw, 도메인 invariant 검증의 정당한 imperative). 5 도메인을 단일 추상화로 묶지 않음 — typed-name 합성 로직이 도메인별로 다름 (over-abstraction 회피).

주의 항목

  • BIOS 의 legacy data-typed-confirm="purge" + bios-list.js 의 window.prompt() 핸들러는 본 sub-slice CP4 시점에 일괄 제거.
  • nudge REPLACE 흐름 보전 — *NudgeService.purge 가 호출하는 service-internal hard-delete 는 typed-name 우회. purgeWithTypedNameCheck 는 그 메서드를 wrapping (재진입 위험 없음).
  • TTL 자동 hard-delete 도 typed-name 우회 — TrashTtlWorker (현재 식별 + 로그만 구현) 가 향후 호출할 hard-delete 도 사용자 입력 부재이므로 purge(id) 직접 호출 경로. nudge REPLACE 와 동일 정책. 본 S5-2-2 범위 외 — S5-2-4 (nudge / TTL 만료 자동 hard-delete 정책 지정) 로 분리. retry / 실패 알림 / 백업 정책은 S5-2-4 에서 결정.
  • Board / Subprogram 백엔드 / 프론트엔드 신설 — Subprogram 은 백엔드 있고 프론트엔드만 신설. Board 는 양쪽 다 신설.
  • S5-2-1 의 JS handleSubmit 와 호환 — 본 sub-slice 가 추가하는 'purge' actionKey 는 ACTION_CONFIG 에 entry 추가 + input slot 분기 추가. 기존 3 actionKey 흐름 무영향.
  • typed-name 일치성 보장 — S5-2-1 의 modal 메시지 식별자 (Service 외부) 와 본 sub-slice 의 백엔드 typed-name 합성 (Service 내부) 가 정확히 일치해야 함. 불일치 시 사용자 입력 검증 실패. 단일 helper 로 통합 검토 항목 (over-abstraction 가능성, CP2 시점에 사용자와 합의).
  • CLAUDE.md 수정 — Step 1 plan 산출물 규약을 docx → html 로 전환한 사용자 결정 반영 필요 (별도 commit 후보).
