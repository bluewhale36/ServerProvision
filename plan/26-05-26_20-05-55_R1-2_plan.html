§2. 페이지 요구사항

본 슬라이스의 핵심 : OSMetadata 의 osVersion 을 osName 과 동일하게 "수정 불가" 도메인 정책으로 격상. 이 정책 변경이 update 로직의 모든 분기·검사를 자연스럽게 소멸시킨다.

  1. 도메인 정책 — OSMetadata 의 osVersion 은 수정 불가. 변경 필요 시 soft delete 후 재등록 (osName 과 동일). DB 컬럼은 그대로 유지, 변경 채널만 차단.
  2. UpdateRequest 단순화 — OSMetadataUpdateRequest 는 description 1 필드만 가진다. osVersion 제거.
  3. Service.update() 단순화 — entity 조회 + image.update(description) 만. 충돌 검사 완전 소멸.
  4. Entity.update() 시그니처 좁힘 — OSMetadata.update(String description). osVersion 인자 제거.
  5. 수정 폼 UX — edit.html 에서 osVersion 을 osName 과 동일하게 readonly 표시. 옆에 작은 안내 ("변경하려면 삭제 후 재등록").
  6. 회귀 0 — description-only 수정 흐름은 옛 동작과 동치. 외부 endpoint URL / HTTP method / response 코드 모두 동일.
비 목표 (out of scope) — existsByOsNameAndOsVersionAndIsDeletedFalse 의 다른 3 호출처 (line 236 / 330 / restore-from-nudge 흐름). 본 슬라이스 종료 시점에 호출처 3 곳 남음. R1-3 후속.
UX 비교 분석 — 정책 변경의 사용자 영향
옛 흐름에서 사용자가 "오타 수정" 또는 "버전 표기 정정 (예: 9.6 → 9.06)" 시 update 를 사용했을 가능성이 있다. 새 정책에선 그 경로가 막힘 → soft delete 후 재등록 안내가 사용자 부담일 수 있음. 완화책 (안 적용) :
  • edit 화면의 osVersion 옆에 안내 문구 ("버전을 변경하려면 자원을 삭제 후 다시 생성해주세요.")
  • tooltip — disabled input 에 hover 시 도메인 정책 설명 (S5-3-2 의 .n-btn-tooltip-wrap 패턴 재사용)

§3. 데이터 흐름 (before vs after)

3.1 옛 코드 (현행)

// OSMetadataUpdateRequest.java
public record OSMetadataUpdateRequest(
    @NotBlank String osVersion,
    String description
) {}

// OSMetadataService.java
@Transactional
public void update(Long id, OSMetadataUpdateRequest request) {
    OSMetadata image = requireActiveImage(id);
    if (
            !image.getOsVersion().equals(request.osVersion()) &&
            osMetadataRepository.existsByOsNameAndOsVersionAndIsDeletedFalse(
                    image.getOsName(), request.osVersion())
    ) {
        throw new DuplicateOSMetadataException(image.getOsName(), request.osVersion());
    }
    image.update(request.osVersion(), request.description());
}

// OSMetadata.java (entity)
public void update(String osVersion, String description) {
    this.osVersion = osVersion;
    this.description = description;
}

3.2 새 코드 (제안)

// OSMetadataUpdateRequest.java
public record OSMetadataUpdateRequest(
    String description
) {}

// OSMetadataService.java
@Transactional
public void update(Long id, OSMetadataUpdateRequest request) {
    OSMetadata image = requireActiveImage(id);
    image.update(request.description());
}

// OSMetadata.java (entity)
public void update(String description) {
    this.description = description;
}

3.4 form 측 행위

  • edit.html 에서 osVersion input 을 readonly label 로 (osNameLabel 패턴과 동일)
  • controller editForm() 에서 osMetadataForm 의 osVersion 초기값 더 이상 안 채움 (UpdateRequest 에 필드 없음). 화면 표시는 별도 model 속성 osVersionLabel (또는 image.osVersion 직접 노출)
  • 사용자가 의도적으로 osVersion 값을 form 에 끼워 넣어 POST 해도, Spring 의 @ModelAttribute binding 에 해당 setter 가 없으므로 **무시** (silent ignore). 보안적으로 안전

§8. 부산물 / 주의 / 결정 사항

D-1. 정책 변경의 사용자 영향 — 완화 안내

edit.html 의 osVersion readonly label 옆에 1 줄 안내 표시 : "버전을 변경하려면 자원을 삭제 후 다시 생성해주세요.". CSS 클래스는 .n-help 또는 fragment 의 기존 안내 스타일 재사용.

D-2. Spring binding 동작 — silent ignore 의존 안전성

@ModelAttribute + record 바인딩에서 record 컴포넌트에 없는 form 파라미터는 자동으로 무시된다. 따라서 사용자가 직접 HTML 수정 / curl 로 osVersion=… 을 POST 해도 서버 측에서 silent ignore. 이를 명시적으로 거절하려면 별도 @InitBinder + denylist 가 필요하지만, 본 슬라이스에서는 silent ignore 그대로 채택 (사용자 친화적 + Spring 기본 동작과 정합). 관련 테스트 시나리오 §6 #3 으로 검증.

D-3. findFirstByOsNameAndOsVersionAndIsDeletedTrue 의 거취

R1-1 시점에 unused 로 식별. 본 슬라이스 범위 밖이지만 호출처가 여전히 0 이면 R1-3 또는 후속 cleanup 슬라이스에서 일괄 제거.

D-4. CLAUDE.md 정렬

  • "조건 분기문 의 legacy 무분별 확장 절대 지양" — 분기 자체를 도메인 정책으로 흡수해 소멸시키는 방식
  • "확장성 고려" — version 변경 흐름이 필요해지면 별도 endpoint (예: PUT /…/{id}/version) 를 명시적으로 추가 가능. 묵시적 채널이 아닌 명시적 채널
  • "보안 고려" — silent ignore 가 의도하지 않은 변경을 차단
