1. 현재 상태 (실측 근거)

2026-07-19 샌드박스 전수 점검에서 휴지통 "+30일 연장" 한 클릭이 세 결함을 연쇄로 노출했다.

만료 계산의 현재 SSOT 분포: 화면은 TrashController.toResponse 의 trashedAt.plus(ttl), 자동 영구삭제는 TrashTtlWorker 의 findTrashedBefore(now - ttl) — 두 곳이 같은 식을 각자 계산한다. 가산 도입 시 이 식이 3곳 이상으로 늘 수 있어 SSOT 응집이 함께 필요하다.

2. 요구사항
  1. UI 1차 차단 (불가침 원칙 복원) — 연장 미지원 자원 종류(현재: OS_IMAGE, BOARD_MODEL)의 행에서 연장 버튼을 disabled + tooltip 으로 차단한다. 차단 조건의 SSOT 는 서버 가드와 동일한 판정 하나를 공유한다.
  2. 서버 가드 안전망 — direct POST 로 미지원 자원 연장 요청 시 도메인 예외 → advice 가 409 로 변환 (JSON/HTML 협상 포함). 500 금지.
  3. 모달 라벨 정상화 — 연장 전용 확인 모달: 제목 "보존기간 연장", 확정 버튼 "연장", 본문 "{자원} 의 보존기간을 +N일 연장할까요?".
  4. 오류 응답 협상 — XHR 실패 시 서버는 JSON 오류를 응답하고, 클라이언트는 비 JSON 응답이 와도 원문을 노출하지 않는다 (일반화 메시지).
  5. 가산 전환 (O-4) — 연장 1회 = 만료일 +N일 누적. trashed_at(이동 시각) 불변. N = 운영 설정 TTL 일수(현재 30) — 버튼 라벨도 설정값 연동.
🎬 라이브 데모 — 연장 UX AS-IS / TO-BE 체험

4. 도메인 모델 — 설계 결정 (대안 비교 포함)

D1. 연장 지원 여부의 SSOT

안내용판정
(a) 채택MarkableTrashOperator 에 default boolean supportsTrashTtlExtension() { return false; } 추가, extendTrashTtl 을 override 한 파일 자원 4 스캐너만 true 로 override. UI 뷰모델 플래그와 서버 가드가 이 판정 하나를 함께 호출.차단 조건 SSOT 공유 불가침 원칙 그대로. 기존 SPI 확장이라 신규 타입 추가 시 자동 안전(default false).
(b) 탈락TrashTtlExtensionService 에 지원 타입 Set 상수SPI 밖에 두 번째 SSOT — 스캐너 구현과 Set 이 drift 하면 silent 사고 재발.
(c) 탈락메타 자원도 연장 구현해 결함 자체를 소멸메타 연장의 cascade 의미(자식 파일 자원 TTL 도 함께 연장?)가 미확정 — 지금 정하면 과설계. 후속 논의 항목으로 §11 에 기록.

D2. 가산 저장 방식

안내용판정
(a) 채택LifecycleEntity(@MappedSuperclass) 에 ttlExtensionDays int default 0 추가. 만료 = trashedAt + TTL + extensionDays. 복원·purge 시 0 초기화.이동 시각 불변(표시 정확) + 전역 TTL 설정 변경이 기존 행에도 동적 반영 유지. 라이프사이클 6 테이블 ALTER 1회.
(b) 탈락현행처럼 trashed_at 을 앞으로 밀기(재기준/shift)사용자 결정(가산)과 불일치 + "휴지통 이동 시각" 표시 왜곡 실측.
(c) 탈락trash_expires_at 절대값 저장운영 설정 TTL 변경이 기존 행에 반영 안 됨 — 현행 동적 의미 상실.

D3. 연장 step 일수

채택: TrashPolicy.getTtl() 일수 (운영 설정 ttl_days, 현재 30) — 버튼 라벨 "+{step}일 연장" 을 뷰모델로 동적화. 탈락: 고정 30 — 운영 설정을 45일로 바꾸면 라벨·동작이 설정과 drift.

D4. 자동 영구삭제(worker)의 만료 판정

채택: 기존 findTrashedBefore(now - ttl) 후보 조회 유지 + entity.trashExpiresAt(ttl).isBefore(now) Java 재검증 (연장분 제외). 탈락: SQL 날짜 연산으로 쿼리 한 방 — 4개 repository 에 방언 의존 date 연산 중복, JPQL 이식성 저하. 후보 초과조회는 소량(만료 임박 행)이라 비용 무시 가능.

D5. 오류 응답 협상 (F-3)

채택: 서버 — 신설 예외가 ConflictException 하위라 기존 이중 advice(HIGHEST + Accept 협상)가 자동으로 JSON/HTML 변환. 클라이언트 — submitAsync 의 비 JSON 분기가 바디 원문을 message 로 쓰지 않도록 수정(일반화 메시지, 원문은 console 로만). 탈락: 클라이언트만 고치고 서버 500 방치 — 로그에 unhandled ERROR 잔존, 모니터링 오염.

D6. 연장 확인 모달 (F-2)

채택: ConfirmModalType.EXTEND_TTL 상수 신설(lookup 불요 — SOFT_DELETE 패턴) + confirm-extend.html fragment + confirm-extend.js(마커 data-confirm-extend). 탈락: deprecate 모달에 라벨 파라미터 추가 — S5-2 모달 자산의 "actionKey 별 책임 분리" 위반, deprecate 라벨의 두 번째 소유자를 만든다.

6. Step 8 테스트 시나리오

단위 (JUnit + Mockito)

  • TrashTtlExtensionService — 지원 타입: 위임 호출 + step 일수 전달 확인 / 미지원 타입: TtlExtensionUnsupportedException throw / 미등록 타입: 기존 IllegalArgumentException 유지.

  • TrashTtlWorker — 후보 중 연장으로 만료 미도래 행이 purge 대상에서 제외되는 재검증 필터.

사용자 액션 통합 (@WebMvcTest TrashController + @MockitoBean 서비스단)

  • 2xx — 파일 자원 extend POST → 302 redirect /maintenance/trash.
  • 409 — 메타 자원 extend POST(신설 예외 실제 트리거): Accept json → JSON body message 필드 검증 / Accept html → error 뷰. 새 예외 클래스 동반 시나리오 규율 충족.
  • 400 — resourceType 경로변수 오타(enum 변환 실패).
  • 404 — (해당 없음 — extend 는 id lookup 을 스캐너가 수행. 스캐너 NotFound 시나리오는 단위에서 커버; 통합 404 는 기존 restore/purge 스위트가 보유)
  • 모달 fragment — GET /ui/confirm-modal/EXTEND_TTL → 200 + fragment 마크업(제목 '보존기간 연장').
