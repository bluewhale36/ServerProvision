§2. 페이지 요구사항

본 슬라이스는 파일 분리 + interface 구현. R1-3 의 OSMetadataLifecycleService 패턴 직접 재사용.

  1. 책임 분할 — ISO 의 lifecycle 메서드 8 개를 IsoLifecycleService 로 응집
  2. LifecycleService implements — R1-3 의 interface 를 ISO 도 구현. ISO 는 leaf 자원 (자식 없음) 이므로 default restore(Long) 자동 활용
  3. 메서드명 표준화 — toggleIsoEnabled → toggleEnabled, softDeleteISO → softDelete, restoreISO → restore, deprecateIso → deprecate, undeprecateIso → undeprecate, purgeIso → purge, purgeIsoWithTypedNameCheck → purgeWithTypedNameCheck
  4. 시그니처 정렬 — LifecycleService 의 단일 id 시그니처 채택 (옵션 A) — controller 는 URL 의 osMetadataId 와 entity 의 부모 id 일치 검증 가드 추가
  5. 외부 endpoint 동일 — Controller URL / HTTP method / response 100% 동일
  6. cascade 정책 유지 — R1-3 OSMetadataLifecycleService 의 자식 ISO cascade 가 이 service 의 메서드를 호출하므로 호환 보존
비 목표 (out of scope) :
  • ISO 등록 흐름 (addISO / prepare / finalize / completePendingIsoFromNudge) → R1-4-2
  • ISO 무결성 검증 (verifyIntegrity / reissueMarker) → R1-4-3
  • ISO 의 helper (cleanupIsoArtifacts / requireLiveISO 등) — purgeIso 가 호출하므로 사본 또는 위임 결정 (§8 D-1)
  • findISO / findIntegrityStatus 조회 메서드 — OSMetadata 응답 조립의 일부라 잔류
🎬 미리보기 — LifecycleService interface 의 두 구현체

R1-3 에서 도입한 interface 가 R1-4-1 에서 두 번째 구현체를 갖게 된다. parent (OSMetadata) 와 leaf (ISO) 가 같은 메서드 시그니처를 공유.

LifecycleService (interface)
  • toggleEnabled(id)
  • softDelete(id)
  • restore(id) default
  • restore(id, cascade) abstract
  • deprecate(id)
  • undeprecate(id)
  • purge(id)
  • purgeWithTypedNameCheck(id, name)
←
OSMetadataLifecycleService (parent)
  • cascade=true → 자식 ISO 동반 복구
  • 자식 lifecycle 가드 가능
  • R1-3 완료
/
IsoLifecycleService (leaf, 신규)
  • cascade=true 도 자식 0 → none() 자연 반환
  • 부모 OSMetadata 의 lifecycle 가드 (parent ChildLifecycleBlocked)
  • R1-4-1 본 슬라이스
핵심 통찰 — R1-3 의 default restore(Long) 가 정확히 이 leaf 자원의 위해 설계됨. ISO 는 cascade 시그니처 하나만 구현하고 자식 0 일 때 자연 무모순으로 동작. interface 의 의도가 코드로 검증되는 자리.
§3. 분리 후 책임 + 시그니처 정렬

3.1 IsoLifecycleService 골격

@Slf4j
@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)
public class IsoLifecycleService implements LifecycleService {

    private final ISORepository isoRepository;
    private final OSMetadataRepository osMetadataRepository;   // 부모 lifecycle 가드용
    private final TrashLifecycleService trashLifecycleService;
    private final SoftDeleteIntentService softDeleteIntentService;
    private final ProvisionMarkerService markerService;        // sidecar 정리 (purge)

    @Override @Transactional public void toggleEnabled(Long isoId) { ... }
    @Override @Transactional public void softDelete(Long isoId) { ... }
    // restore(Long) 은 interface default 가 처리 — 자동
    @Override @Transactional public RestoreResponse restore(Long isoId, boolean cascade) { ... }
    @Override @Transactional public void deprecate(Long isoId) { ... }
    @Override @Transactional public void undeprecate(Long isoId) { ... }
    @Override @Transactional public void purge(Long isoId) { ... }
    @Override @Transactional public void purgeWithTypedNameCheck(Long isoId, String typedName) { ... }

    // R1-4-1 — 자체 사본 (R1-3 D-1 안 A 정책 — over-abstraction 회피)
    private ISO requireLiveIso(Long isoId) { ... }
    private ISO requireDeletedIso(Long isoId) { ... }
    private void cleanupIsoArtifacts(ISO iso) { ... }
    private static void verifyTypedNameOrThrow(Markable resource, String typedName) { ... }

    // softDelete intent saga 분기 (DCM3-2.3~2.5) — softDeleteISOWithIntent 도 같이 이동
    @Transactional public void softDeleteWithIntent(Long isoId, DeleteAction action) { ... }
}

3.2 시그니처 정렬 결정 — 옵션 A (단일 id, 확정)

ISO 의 옛 시그니처는 (Long osMetadataId, Long isoId) 2-arg. interface 의 단일 id 와 정렬하기 위해 isoId 단일 인자 채택.

  • service 는 ISO entity 의 osMetadata reference 로 부모 lookup → 모든 lifecycle 가드 (toggle / restore / undeprecate) 그대로 동작
  • URL forging 가드 (URL osMetadataId ↔ entity 부모 id 일치) 는 IsoLifecycleService 의 가드 메서드 assertBelongsToOs(isoId, expectedParentId) 가 책임. controller 가 lifecycle 호출 직전에 호출 (§8 D-2)
  • 외부 endpoint URL 은 그대로 유지 (/management/os/{osId}/iso/{isoId}/...)
// 옛
osMetadataService.toggleIsoEnabled(osId, isoId)
osMetadataService.softDeleteISO(osId, isoId)
osMetadataService.restoreISO(osId, isoId)
osMetadataService.deprecateIso(osId, isoId)
osMetadataService.undeprecateIso(osId, isoId)
osMetadataService.purgeIso(osId, isoId)
osMetadataService.purgeIsoWithTypedNameCheck(osId, isoId, typedName)

// 새
isoLifecycleService.toggleEnabled(isoId)         // ISO entity → 부모 lookup
isoLifecycleService.softDelete(isoId)
isoLifecycleService.restore(isoId)                // default
isoLifecycleService.restore(isoId, cascade)
isoLifecycleService.deprecate(isoId)
isoLifecycleService.undeprecate(isoId)
isoLifecycleService.purge(isoId)
isoLifecycleService.purgeWithTypedNameCheck(isoId, typedName)

3.3 부모 가드 — entity 의 부모 reference 로 lookup

옛 toggleIsoEnabled 같은 메서드는 requireActiveImage(osMetadataId) 로 부모 검증. 새 service 는 :

private OSMetadata loadParent(ISO iso) {
    return osMetadataRepository.findByIdAndIsDeletedFalse(iso.getOsMetadata().getId())
            .orElseThrow(() -> new OSMetadataNotFoundException(iso.getOsMetadata().getId()));
}

ChildLifecycleBlockedByParentException 발화 조건 그대로.

3.4 IsoLifecycleService → OSMetadataLifecycleService 결합 검토

R1-3 의 OSMetadataLifecycleService.restore(id, cascade=true) 가 자식 ISO 복구를 위해 옛 osMetadataService.restoreISO(...) 를 위임 호출. R1-4-1 후엔 :

// OSMetadataLifecycleService (R1-3) — 호출처 변경
- osMetadataService.restoreISO(id, iso.getId());
+ isoLifecycleService.restore(iso.getId());   // default — interface 활용

→ OSMetadataLifecycleService 의 OSMetadataService 의존을 IsoLifecycleService 로 교체. 의존 그래프 :

  • Controller → OSMetadataLifecycleService, IsoLifecycleService (2 service 모두)
  • OSMetadataLifecycleService → IsoLifecycleService (cascade restore 위임)
  • IsoLifecycleService → (없음, leaf)

단방향 (parent → leaf). cycle 없음.

§8. 부산물 / 주의 / 결정 사항

D-1. helper 사본 정책 (R1-3 패턴 그대로)

requireLiveIso / requireDeletedIso / cleanupIsoArtifacts / verifyTypedNameOrThrow — IsoLifecycleService 안 사본 보유. over-abstraction 회피.

특히 cleanupIsoArtifacts 는 R1-4-2 에서 IsoRegistrationService 로 이동 예정 → 그 시점에 IsoLifecycleService 의 사본을 IsoRegistrationService 의 메서드 위임으로 교체할 수도 있고, 그대로 사본 유지할 수도 있음 (R1-4-2 진입 시 결정).

D-2. 부모 일치 가드 — Service 측 책임 (사용자 결정 2026-05-26)

URL 의 {osMetadataId} 와 entity 의 iso.osMetadata.id 일치 검증을 IsoLifecycleService 가 노출하는 가드 메서드 로 처리. controller 는 endpoint 진입 시 가드 + lifecycle 두 호출.

// IsoLifecycleService 의 public 가드 메서드
public void assertBelongsToOs(Long isoId, Long expectedParentId) {
    ISO iso = requireLiveIso(isoId);
    if (!iso.getOsMetadata().getId().equals(expectedParentId)) {
        throw new ISONotFoundException(expectedParentId, isoId);
    }
}

// controller endpoint
@PostMapping("/{osId}/iso/{isoId}/toggle")
public String toggleIso(@PathVariable Long osId, @PathVariable Long isoId, ...) {
    isoLifecycleService.assertBelongsToOs(isoId, osId);
    isoLifecycleService.toggleEnabled(isoId);
    return ...;
}

장점 — 가드 로직이 service 안에 응집. controller 는 단순 dispatcher. LifecycleService interface 의 단일 id 시그니처는 그대로 유지 (interface 정합성 보존).

D-3. OSMetadataLifecycleService 의 cascade 호출 — interface 활용

cascade restore 위임 시 :

// 옛 (R1-3)
osMetadataService.restoreISO(id, iso.getId());

// 새 (R1-4-1)
isoLifecycleService.restore(iso.getId());   // interface default 활용 — cascade=false

의존 그래프 단방향 정렬 — parent → leaf. R1-4-1 의 가장 중요한 의존 정리.
