§1. 현재 상태 (18-에이전트 Workflow audit 진단)

출발 — "R2-2 의 UI 1차 차단 패턴을 lifecycle 가드 외 프로그램 전반으로 확산" 이 목표. BindingResult 검증은 제외 (사용자 지시).

audit 결론 (9 도메인 + 95 예외 catalog 전수 · adversarial 검증) — "프로그램 전반 확산" 의 실질 후보는 단 한 도메인, Subprogram(Driver/Utility) 에 수렴한다. 부모-자식 lifecycle UX 모순은 형제 ISO/BIOS/BMC 3 도메인에서 이미 R2-2 완비됐고, 동일 위계인 Subprogram 만 부모 가드가 통째로 누락되어 silent 성공한다.

§2. 요구사항 + 비목표

채택 = Subprogram 을 형제(ISO/BIOS/BMC)와 동일 R2-2 수준으로 끌어올려 비대칭 해소. 신규 클래스 / 예외 / 스키마 0 — 형제가 이미 쓰는 SSOT·예외·패턴 재사용.

G1 — 부모(BoardModel) lifecycle → 자식 promote 차단 (핵심)

  1. 서버 가드 3 — toggleEnabled(enable 방향만) / undeprecate / restore 에 sp.getBoardModel() 부모 가드 추가. BMC(BmcService:432/486/598)와 동형. SSOT = BoardModel.childEnableBlockReason() / blocksChildUndeprecate() / blocksChildRestore() (LifecycleEntity 기존 메서드, 변경 0).
  2. capability 3 — SubprogramResponse 에 parentBlocksEnable / parentBlocksUndeprecate / parentBlocksRestore. 부모는 엔티티 그래프(sp.getBoardModel())로 도달 — repo 질의 0. common-scope(boardModel=null) null-guard 필수.
  3. UI 차단 — miller.html 의 toggle/undeprecate/restore form 을 BIOS list.html 동형으로 <span class="n-btn-tooltip-wrap" data-tooltip>+disabled ↔ 정상 form 분기.

G2 — restore 활성 중복키 차단 (선택, 우선순위 낮음)

restore 시 같은 (kind,scope,name,version) 활성 자원이 있으면 DuplicateSubprogramVersionException(409). 신규 hasActiveDuplicateKey() 로 서버 가드 + capability(restoreBlockedByActiveDuplicate) SSOT 공유 → 버튼 disable. 단 restore form 이 이미 data-async-submit 보유라 409 가 full-page 가 아닌 async alert 로 떠 이미 완화돼 있고, per-row 질의 + toResponse static→instance 리팩터 비용이 따른다 → CP1 결정 (§8 Q2/Q3).

비 목표 :
  • BindingResult / FieldBound* 폼 입력 검증 — 제외 (사용자 명시). 등록/수정 경로의 DuplicateSubprogramVersion · SubprogramPathConflict · TypedNameMismatch 불변.
  • 형제 ISO/BIOS/BMC 무수정. 서버 가드 throw 제거 안 함 (안전망 유지, 불가침).
  • 부모 도메인(OSMetadata/BoardModel) self-restore 중복키 확산 — 보류 (per-row 질의 미보유 + race, Q3).
  • advice 통합 / raw JSON 근본 정리 → R2-3. nudge/intent/FS 예외 무변경. self-state 모순(이미 deprecated 인데 deprecate)은 이미 조건부 렌더로 차단됨(형제 동일 정책 유지, Q1).

public void toggleEnabled(Long id){
  Subprogram sp = requireLive(id);
  BoardModel parent = sp.getBoardModel();   // 그래프 도달(repo 0)
  if (parent != null && !sp.isEnabled()) {   // common-scope null skip
    String r = parent.childEnableBlockReason();   // LifecycleEntity SSOT
    if (r != null) throw new ChildLifecycleBlockedByParentException(
      ResourceType.BOARD_MODEL, parent.getId(), r,
      ResourceType.SUBPROGRAM, id, "enable", parent.displayName());
  }
  sp.toggleEnabled();
}

§4. 도메인 모델 (SSOT)

1) LifecycleEntity — 변경 0 (재사용)

부모 가드 SSOT 4 메서드(childEnableBlockReason / blocksChildEnable / blocksChildRestore / blocksChildUndeprecate)는 R2-2 에서 이미 LifecycleEntity 에 작성됨. BoardModel 이 이를 상속하므로 Subprogram 가드는 새 도메인 메서드 0 — 형제와 같은 바디 호출.

2) SubprogramService — 부모 가드 3 추가 (BMC 동형)

// undeprecate — BmcService:598 동형
public void undeprecate(Long id){
  Subprogram sp = findLive(id);
  BoardModel parent = sp.getBoardModel();
  if (parent != null && parent.blocksChildUndeprecate())     // DEPRECATED || DELETED
    throw new ChildLifecycleBlockedByParentException(
      ResourceType.BOARD_MODEL, parent.getId(),
      parent.isDeleted() ? "DELETED" : "DEPRECATED",
      ResourceType.SUBPROGRAM, id, "undeprecate", parent.displayName());
  sp.undeprecate();                                            // self-state 가드(!isDeprecated) 안전망 유지
}
// restore — BmcService:486 동형. 진입 직후 parent.blocksChildRestore()(=isDeleted) 검사 후
//           기존 중복키 가드(G2 대상)는 그 뒤에 유지.
common-scope null-guard (Subprogram 고유) — BMC/BIOS 는 부모가 항상 존재하지만 Subprogram 은 boardModel=null(공용) 가능. 모든 가드·capability 는 parent != null 선검사 → 공용 자원은 가드 미적용(capability=false), 정상 동작 회귀 0.
