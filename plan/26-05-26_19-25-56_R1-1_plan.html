§3. 데이터 흐름 (before vs after)

3.1 옛 코드 (현행)

@Transactional
public Long create(OSMetadataCreateRequest req) {
    // (#1) 활성 + 미삭제 자원 존재? — cheap EXISTS
    if (osMetadataRepository.existsByOsNameAndOsVersionAndIsDeletedFalse(req.osName(), req.osVersion())) {
        // (#2) 그 중 deprecated 자원 — 활성 단순 충돌 vs 회수 후보 분기용
        List<OSMetadata> activeDeprecated =
                osMetadataRepository.findAllByOsNameAndOsVersionAndIsDeprecatedTrueAndIsDeletedFalse(...);
        if (activeDeprecated.isEmpty()) throw new DuplicateOSMetadataException(...);
    }
    // (#3 + #4) soft-deleted + active-deprecated  ← #4 는 #2 와 동일 쿼리 (재조회)
    List<OSMetadata> candidates = collectMetaNudgeCandidates(req.osName(), req.osVersion());
    if (!candidates.isEmpty()) throw new OSMetadataNudgeRequiredException(...);
    // (#5) INSERT
    return persistNewOSMetadata(req.osName(), req.osVersion(), req.description());
}

3.2 새 코드 (제안)

@Transactional
public Long create(OSMetadataCreateRequest req) {
    // (#1) 동일 키 row 전부 1 회에 — soft-deleted 포함
    Map<LifecycleStage, List<OSMetadata>> byStage = osMetadataRepository
            .findAllByOsNameAndOsVersion(req.osName(), req.osVersion()).stream()
            .collect(Collectors.groupingBy(
                    m -> LifecycleStage.of(m.isDeprecated(), m.isDeleted()),
                    () -> new EnumMap<>(LifecycleStage.class),
                    Collectors.toList()));

    boolean hasActive       = byStage.containsKey(LifecycleStage.ACTIVE);
    boolean hasNudgeTarget  = byStage.containsKey(LifecycleStage.DEPRECATED)
                           || byStage.containsKey(LifecycleStage.SOFT_DELETED);

    if (hasActive && !hasNudgeTarget) {
        // 활성 정상 자원 단독 충돌 — fail-fast
        throw new DuplicateOSMetadataException(req.osName(), req.osVersion());
    }
    if (hasNudgeTarget) {
        // 회수 후보 (DEPRECATED ∪ SOFT_DELETED) — nudge 분기
        List<OSMetadata> candidates = new ArrayList<>();
        candidates.addAll(byStage.getOrDefault(LifecycleStage.DEPRECATED,   List.of()));
        candidates.addAll(byStage.getOrDefault(LifecycleStage.SOFT_DELETED, List.of()));
        throw new OSMetadataNudgeRequiredException(buildOSMetadataNudgePayload(req, candidates));
    }
    // (#2) INSERT
    return persistNewOSMetadata(req.osName(), req.osVersion(), req.description());
}

3.3 분기 정합성 증명 (옛 ↔ 새 동치성)

입력 상태옛 결과새 결과일치?
키 row 0 개save (정상)hasActive=F, hasNudge=F → save✓
ACTIVE 만 1+ 개exists(T)+actDep[]=∅ → DuplicatehasActive=T, hasNudge=F → Duplicate✓
DEPRECATED 만 1+ 개 (is_del=F)exists(T)+actDep≠∅ → 통과 → candidates≠∅ → NudgehasActive=F, hasNudge=T → Nudge✓
SOFT_DELETED 만 1+ 개exists(F) → candidates≠∅ → NudgehasActive=F, hasNudge=T → Nudge✓
ACTIVE + DEPRECATED 혼합exists(T)+actDep≠∅ → 통과 → candidates≠∅ → NudgehasActive=T, hasNudge=T → Nudge✓
ACTIVE + SOFT_DELETED 혼합exists(T)+actDep[]=∅ → DuplicateNudge (안 B)의도 변경
안 B 채택 (사용자 결정, 2026-05-26) — ACTIVE + SOFT_DELETED 혼합 케이스에서 옛 코드는 Duplicate 였지만, 새 코드는 SOFT_DELETED 를 회수 후보로 보고 Nudge 로 환원한다. 사용자가 같은 키로 새 등록을 시도할 때 "휴지통에 같은 자원이 있다" 는 회수 시그널을 받도록 한다.

채택 사유 (사용자 멘탈 모델 정렬) :

  • Duplicate ⇔ 순수 ACTIVE 단독 — deprecated=F AND deleted=F 인 row 가 있고, 다른 lifecycle stage 의 회수 후보가 0 인 경우만 진짜 중복
  • SOFT_DELETED 가 1+ 개라도 있다면 사용자에게 "같은 키 자원이 휴지통에 있다" 알릴 가치가 더 큼
  • 옛 동작은 implementation 이 SOFT_DELETED 분기를 ACTIVE 분기보다 뒤에 둔 침전 결과로 보임 — 의도된 결정 근거 없음
본 슬라이스는 리팩토링 + 1 줄짜리 의도 변경을 함께 다루는 형태. 변경 표면 최소 (조건문 1 개 단어 추가).

§8. 부산물 / 주의 / 결정 사항

D-1. ACTIVE + SOFT_DELETED 혼합 case (§3.3) — 안 B 채택 (2026-05-26)

Duplicate ⇔ 순수 ACTIVE 단독 으로 정의. 새 흐름의 첫 if :

if (hasActive && !hasDeprecated && !hasSoftDeleted) {
    throw new DuplicateOSMetadataException(req.osName(), req.osVersion());
}

즉 SOFT_DELETED 또는 DEPRECATED 회수 후보가 1+ 개라도 있으면 Duplicate 가 아닌 Nudge. 사용자가 같은 키 자원이 휴지통에 있다는 시그널을 받도록 한다. 옛 동작 대비 1 케이스 (ACTIVE + SOFT_DELETED) 의도 변경.

D-2. Repository 메서드 제거 시점

CP3 진입 시 옛 3 메서드의 외부 호출처 grep 으로 0 확인 후 제거. 호출처가 1 곳이라도 있으면 제거 보류 + R1-1 본 슬라이스는 새 메서드 추가만으로 완료.

D-3. collectMetaNudgeCandidates 제거

본 helper 는 create() 단독 호출이라 같이 제거. grep 으로 재확인.

D-4. @Transactional propagation

create() 의 @Transactional (REQUIRED, default) 유지. 1 쿼리든 4 쿼리든 동일 트랜잭션 안 same snapshot 보장.

D-5. EnumMap 도입

Collectors.groupingBy(..., () -> new EnumMap<>(LifecycleStage.class), Collectors.toList()) — 키가 enum 일 때 HashMap 보다 cache-friendly + iteration 순서 enum 선언 순서.
