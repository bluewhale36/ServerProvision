§1. 현재 상태 · 문제 (God-SPI ISP 위반)

R5-2 완료(commit 201ec90). R7-1 은 R7(전 도메인 순환 근본해결)의 인프라 전제다. global/marker/MarkableScanner 는 19 메서드가 4 관심사(inventory / drift / trash / ghost)를 혼재해 ISP 위반 — SPI 자체가 두 곳에서 TODO(MK3-3): SPI 분리 시 MarkableInventory/MarkableGhostOperator 로 이전 으로 자인한다(L33, L192).

왜 R7 의 토대인가 — 순환 2종

순환구조현재 차단
cycle #1TypedNameVerifier → List<MarkableScanner> → scanner → (service) → TypedNameVerifierscanner 의 ObjectProvider (OS/Board/Iso, REAL)
cycle #2SoftDeleteIntentService → @Lazy List<MarkableScanner> → scanner → (service) → SoftDeleteIntentServiceSoftDeleteIntentService 의 @Lazy (전역)
핵심 관찰 — 두 순환 모두 scanner → service 변(restoreFromTrash/purgeFromTrash 가 도메인 service.restore/purge 역호출) 때문에 닫힌다. 그런데 verifier 와 SoftDeleteIntentService 가 실제로 호출하는 건 service 무의존 read/ghost 메서드뿐(아래 §2). SPI 를 그 경계로 쪼개 호출부가 좁은 인터페이스만 주입하면, R7-2~6 이 service 변을 LifecycleService 로 옮길 때 순환이 근본 소멸한다.
§2. 요구사항 — 재검증 5점 보강 (확정)

Notion 원안(4 관심사 sealed 분리)은 ISP 만 풀고 순환 경계와 어긋났다. 재검증으로 5점 보강 확정:

  1. read / operation 2차축 포함. 4 관심사만으론 부족 — trash 라벨 안에 cycle-free read(findTrashedById)와 cycle-bearing operation(restore/purgeFromTrash)이 공존. read 는 Inventory 로, service-back-edge operation 만 TrashOperator 로 분리.
  2. 7 호출부 재배선(원안 "3곳"은 과소). 실측: PathReconciliation · SoftDeleteIntent · TrashController · TtlWorker · TtlExtensionService · TypedNameVerifier · PurgeExecutor.
  3. verifier → Inventory 응집. TypedNameVerifierImpl 과 PurgeExecutorImpl 의 read lookup(findActiveMarkableById/findTrashedById)을 순환 없는 read 인터페이스로 모은다 — 원안 재배선 목록에 verifier 자체가 부재한 게 결함.
  4. SoftDeleteIntentService → GhostOperator. SoftDeleteIntentService 가 scanner 에서 실제로 호출하는 메서드는 applyForcedClear(L177, GHOST 연산) 하나뿐이다 — 원안의 List<MarkableTrashOperator> 주입은 사용하지 않는 메서드까지 끌어오는 오분류. 실제 사용에 맞춰 List<MarkableGhostOperator> 로 좁힌다(인터페이스 분리 원칙 = ISP). 이는 순환을 푸는 장치가 아니라 ghost 의존을 명시적으로 드러내는 것 — 순환 자체는 §5(1)의 계층 정리로 풀린다.
  5. sealed 금지 → 일반 interface. 도메인 외부(management/* 4 + os/iso)가 구현하는 개방형 SPI. sealed(permits 폐쇄)면 신규 도메인 추가가 parent 수정을 강제.
動作 불변 (불가침). R7-1 은 인터페이스 분리 + 주입 타입 narrow 만. 자원 동작 · ObjectProvider · @Lazy 제거는 전혀 안 함 — 그것은 R7-2~6 / capstone 소관(§5, §11). bootRun · 전체 테스트가 동일해야 한다.
🎬 라이브 의존그래프 — 호출부를 골라 순환 차단을 직접 검증

호출부를 고르고 before(God-SPI 주입) / after(narrow 주입) 를 토글하면, 그 호출부가 어떤 sub-interface 를 주입하고 service 역변(순환)이 생기는지 / 끊기는지를 그래프로 보여준다. R7-1(narrow 분리) + R7-2~6(service 변 이동) 후 cycle #1·#2 가 어떻게 소멸하는지가 한눈에 드러난다.

§5. 순환 해소의 진짜 기제 + GhostOperator 의 provisional 취급

(1) 순환이 끊기는 이유 — Ghost 때문이 아니라 계층 정리 때문

초안에서 "SoftDeleteIntentService 가 GhostOperator 를 주입하는 것이 순환 해소의 핵심" 이라고 적었는데, 그것은 인과를 잘못 압축한 표현이었다. 정확히 풀어 쓰면 :

순환 #2(SoftDeleteIntentService → scanner → service → SoftDeleteIntentService)가 끊기는 진짜 이유는 R7-2~6 이 "복구 / 영구삭제"(MarkableTrashOperator) 같은 도메인 service 를 호출하는 작업을 scanner 에서 LifecycleService 로 옮기기 때문이다. 그 이동이 끝나면 scanner 는 Inventory / DriftApplier / GhostOperator 만 남아 전부 repository 직접 호출이 되고, service 를 전혀 의존하지 않는 빈이 된다. 그렇게 되면 scanner 를 주입하는 쪽이 좁은 인터페이스든 통합 인터페이스든 상관없이 순환이 생기지 않는다.

그래서 SoftDeleteIntentService 의 @Lazy 제거는 GhostOperator 주입 덕이 아니다. scanner 가 service 의존을 0 으로 떨군 결과(R7-2~6)로 따라오는 것이다. 즉 순환 해소는 "작업을 알맞은 계층(LifecycleService)으로 올린다" 는 정공법으로 풀리며, provisional 한 ghost 로직에 기대지 않는다.

(2) Ghost 의존은 R7 이 새로 얹는 게 아니라 이미 존재

SoftDeleteIntentService.forcedClear 는 지금도 scanner.applyForcedClear()(L177)를 호출한다 — 이것은 MK3-2 의 "강제 정리" modal 선택지의 실제 구현이다. R7-1 은 이 의존을 새로 만들지 않고, 단지 명시적 타입(GhostOperator)으로 드러낼지 / 통합 인터페이스 뒤에 숨길지 만 정한다(아래 D4).

그리고 applyForcedClear · applyGhostClear · isGhost · findGhostMarkables 는 모두 한시적 fail-safe 로 설계된 ghost 처리 로직이다(CLAUDE.md MK3-1 : "운영 안정 후 별도 정리 슬라이스 DCM3-2.9 에서 전체 삭제 가능"). Ghost 가 더 이상 생기지 않음이 확정되면 "강제 정리" 선택지와 forcedClear 도 함께 사라진다 — 둘은 같은 fail-safe 운명 공동체다.

(3) GhostOperator 는 영구 계층의 동급이 아니라 "격리 칸"

ghost 로직이 한시적이므로, MarkableGhostOperator 를 별도 인터페이스로 떼는 것은 provisional 코드에 의존을 더 얹는 게 아니라, 나중에 통째로 떼어내기 쉽게 한 곳에 격리(quarantine)하는 것이다. 인터페이스 javadoc 에 "한시적 fail-safe (MK3-1) — DCM3-2.9 정리 슬라이스에서 본 인터페이스 + 구현 + 주입처 일괄 제거" 를 명시해, 영구 계층(Inventory / DriftApplier / TrashOperator)과 동급이 아니라 지워질 임시 칸임을 코드가 정직하게 드러내게 한다.

격리가 오히려 삭제를 돕는다. 모든 ghost 의존을 하나의 인터페이스 타입으로 묶으면, 미래의 "Ghost 정리 슬라이스" 가 그 인터페이스를 지울 때 컴파일러가 의존처(SoftDeleteIntentService.forcedClear 등)를 정확히 짚어준다. 통합 인터페이스 뒤에 숨기면 그 의존이 암묵적이 되어 정리 시 누락 위험이 커진다.

(4) R7-6 capstone 전망 (정정)

재검증(cluster F)은 "SoftDeleteIntentService → MarkableTrashOperator(=LifecycleService) → checkPrecondition → SoftDeleteIntentService 로 순환이 이전되어 @Lazy 제거에 checkPrecondition 분리가 필요"로 보았다. 그러나 실측상 SoftDeleteIntentService 는 MarkableTrashOperator 를 호출하지 않는다 — applyForcedClear(GHOST, repository 직접, IsoMarkableScanner L185)만 쓴다. 따라서 R7-2~6 후 scanner 가 service 무의존이 되면 cycle #2 는 끊기고, checkPrecondition 분리 substep 없이도 전역 @Lazy 제거가 가능할 전망이다. 단 DI(Dependency Injection — 의존성 주입) 순환은 정적 분석만으로 단정하지 않고 R7-6 에서 bootRun 으로 최종 확정한다(예상 못 한 변이 남으면 그때 checkPrecondition 분리를 fallback). 두 번째 @Lazy(SoftDeleteIntentService:60 PathReconciliationService)는 별개 변으로 capstone 에서 함께 평가하며, 본 R7-1 범위 밖이다.

→ R7-1 에서 하는 것은 (a) GhostOperator 를 provisional 격리 인터페이스로 분리(javadoc 명시), (b) D4 로 SoftDeleteIntentService 가 그 인터페이스를 명시적으로 주입하게 narrow. 실제 @Lazy 제거는 R7-6.

§8. 예상 부산물 / 주의
  • ObjectProvider/@Lazy 제거 안 함 (불가침). R7-1 은 인터페이스 분리 + 주입 narrow 만. scanner 가 아직 TrashOperator(→service)를 composite 로 구현하므로 service 변이 남아 순환 차단 기제(ObjectProvider/@Lazy)는 그대로 유지. 제거는 R7-2~6/capstone.
  • D4(SoftDeleteIntentService → GhostOperator narrow)는 R7-1 에서 적용. 주입 타입만 좁히고 실제 @Lazy 제거는 R7-6. 이는 ghost 의존을 명시적으로 드러내는 ISP 정리이며, 순환 해소 장치가 아니다(순환은 §5(1) 계층 정리로 풀림). GhostOperator 인터페이스에는 "한시적 fail-safe — DCM3-2.9 에서 일괄 제거" javadoc 을 박아 provisional 임을 명시.
  • typedName-purge default 이관(D6) 은 PurgeExecutor 로 옮길지 CP2 확정 — 옮기면 TrashOperator 가 read 의존(findTrashedById) 0 으로 더 깨끗.
  • Notion R7-6 본문 갱신 후보 — §5 refinement(@Lazy 제거가 checkPrecondition 분리 없이 가능)를 R7-6 진입 시 반영. 본 R7-1 에서는 plan 기록만.
