§0. 동기 — R3-3 분할 후 잔존한 3종 복붙 5곳

R3-3 이 BoardModelService 를 Metadata / Lifecycle / Nudge 로 3분할하면서 상태전이·cascade 책임은 BoardModelLifecycleService 로 모였다. 그러나 그 cascade 내부 구조는 R3-3 의 비목표였다 — 5 메서드 각각이 BIOS / BMC / Subprogram 3 도메인을 손으로 복붙한 블록을 그대로 떠안고 있다. R3-4 는 이 3 블록을 단일 다형 순회 (children.forEach / stream) 로 접는다.

  • 3종 복붙이 5 메서드에 산재 — cascadeRecompute · softDelete · restore(id,cascade) · purge(board) · findDeletedChildLabels. 각 메서드가 biosRepository / bmcRepository / subprogramRepository 를 따로 순회하고 자식 service 를 따로 호출한다. 자식 1종이 늘면 5 메서드 전부에 블록을 추가해야 하는 구조 — CLAUDE.md §조건 분기문 legacy 무분별 확장 금지(불가침)·§중복 금지(불가침)와 정면 긴장.
  • 시그니처·라벨 비대칭이 본체에 노출 — BIOS/BMC 는 softDelete(boardId, childId) 2-arg, Subprogram 은 softDelete(childId) 1-arg. 라벨도 "BIOS: " / "BMC: " 리터럴 vs kind.getDisplayName()+": " 로 비대칭. 이 비대칭이 BoardModelLifecycleService 본체에 그대로 새어 나와 가독성을 떨어뜨린다.

해법 = 자식 도메인별 어댑터 BoardScopedChildLifecycle impl 이 자기 repo·service·시그니처·라벨 비대칭을 내부로 흡수하고, BoardModelLifecycleService 는 List<BoardScopedChildLifecycle> 를 다형 순회만 한다. Java 다형성으로 분기를 framework 측에 떠넘기는 CLAUDE.md 권장 패턴(interface / strategy)의 교과서적 적용.

R 시리즈 정합 — R3-1(Subprogram cascade 합류) → R3-2(컨트롤러 3분할) → R3-3(service 3분할) → R3-4(자식 cascade 다형화). 본 슬라이스는 이 plan 의 CP1 가치 = 신구(before/after) 코드 대조다 — §3·§4·🎬 데모가 추상 설명이 아니라 실제 코드 split-grid 로 "무엇이 어떻게 접히는지" 를 전수로 보인다.

§2. 요구사항 (확정) + 비목표
  1. SPI 인터페이스 신설 — board/service/cascade/BoardScopedChildLifecycle : recomputeEffective(Long boardId) / softDeleteActive(Long boardId) / restoreDeleted(Long boardId):int / hasAny(Long boardId):boolean / deletedLabels(Long boardId):List<String>. 5 메서드 = 5 cascade 책임에 1:1 대응.
  2. 3 어댑터 impl(@Component) — BiosBoardScopedChildLifecycle(biosRepository+biosService) / BmcBoardScopedChildLifecycle / SubprogramBoardScopedChildLifecycle(1-arg 시그니처 + kind 라벨). 각 도메인 비대칭을 impl 내부로 흡수.
  3. 본체 다형 순회 전환 — BoardModelLifecycleService 의 3 repo + 3 자식 service 직접 주입 제거 → List<BoardScopedChildLifecycle> 주입. 5 메서드를 children.forEach / children.stream()... 로 접음.
  4. 동작·순서·예외전파 보존(불가침) — softDelete 자식 동반 trash 순서, restore 충돌 시 @Transactional 전체 롤백, purge 자식 잔존 거절, findDeletedChildLabels 라벨 순서(BIOS→BMC→Subprogram) 전부 동일. 기존 BoardModelLifecycleServiceTest 16 시나리오가 회귀망.
순서 보존 (불가침) — 다형 순회는 List 주입 순서를 따른다. 기존 BIOS→BMC→Subprogram 순서가 (1) restore count·롤백 순서, (2) findDeletedChildLabels 의 UI 라벨 표시 순서에 영향 → @Order 로 BIOS→BMC→Subprogram 고정(§8 D2).
비 목표 (다음 슬라이스)

① cascadeRecompute(id) — 자식 effective 재계산

BEFORE (3 블록)

private void cascadeRecompute(Long id) {
  biosRepository.findAllByBoardModel_IdOrderByVersionDesc(id)
    .stream().filter(b -> !b.isDeleted())
    .forEach(BoardBIOS::recomputeEffective);
  bmcRepository.findAllByBoardModel_IdOrderByVersionDesc(id)
    .stream().filter(b -> !b.isDeleted())
    .forEach(BoardBMC::recomputeEffective);
  subprogramRepository.findAllByBoardModel_Id(id)
    .stream().filter(s -> !s.isDeleted())
    .forEach(Subprogram::recomputeEffective);
}

AFTER (1 순회)

private void cascadeRecompute(Long id) {
  children.forEach(c -> c.recomputeEffective(id));
}
//  각 impl.recomputeEffective(boardId) 가
//  자기 repo 순회 + !isDeleted filter + recomputeEffective 흡수.

② softDelete(id) — 자식 동반 trash 이동 (시그니처 비대칭 흡수)

BEFORE (3 블록, 2-arg/1-arg 비대칭)

public void softDelete(Long id) {
  BoardModel board = ...requireActiveBoard(id);
  biosRepository.findAllByBoardModel_IdAndIsDeletedFalseOrderByVersionDesc(id)
    .forEach(bios -> biosService.softDelete(id, bios.getId())); // 2-arg
  bmcRepository.findAllByBoardModel_IdAndIsDeletedFalseOrderByVersionDesc(id)
    .forEach(bmc -> bmcService.softDelete(id, bmc.getId()));    // 2-arg
  subprogramRepository.findAllByBoardModel_IdAndIsDeletedFalse(id)
    .forEach(sp -> subprogramService.softDelete(sp.getId()));   // 1-arg!
  board.softDelete();
  board.markTrashed(null);
}

AFTER (1 순회)

public void softDelete(Long id) {

  children.forEach(c -> c.softDeleteActive(id));
  board.softDelete();
  board.markTrashed(null);
}
//  2-arg vs 1-arg 비대칭은 각 impl.softDeleteActive(boardId)
//  내부에서 자기 service 시그니처로 변환 — 본체에서 소멸.

③ restore(id, cascade) — 자식 동반 복구 + count (롤백 순서 보존)

BEFORE (3 for-loop + 수동 count)

public RestoreResponse restore(Long id, boolean cascade) {
  BoardModel board = ...findByIdAndIsDeletedTrue(id)...;
  if (existsByVendorAndModelName...) throw new DuplicateBoardModelException(...);
  board.restore(); board.clearTrashed();
  if (!cascade) return RestoreResponse.none();
  int restored = 0;
  for (BoardBIOS bios : biosRepository.findAllByBoardModel_IdAndIsDeletedTrue(id)) {
    biosService.restore(id, bios.getId()); restored++; }     // 2-arg
  for (BoardBMC bmc : bmcRepository.findAllByBoardModel_IdAndIsDeletedTrue(id)) {
    bmcService.restore(id, bmc.getId()); restored++; }       // 2-arg
  for (Subprogram sp : subprogramRepository.findAllByBoardModel_IdAndIsDeletedTrue(id)) {
    subprogramService.restore(sp.getId()); restored++; }     // 1-arg
  return new RestoreResponse(restored);
}

AFTER (stream sum)

public RestoreResponse restore(Long id, boolean cascade) {
  BoardModel board = ...findByIdAndIsDeletedTrue(id)...;
  if (existsByVendorAndModelName...) throw new DuplicateBoardModelException(...);
  board.restore(); board.clearTrashed();
  if (!cascade) return RestoreResponse.none();
  int restored = children.stream()
      .mapToInt(c -> c.restoreDeleted(id)).sum();
  return new RestoreResponse(restored);
}
//  순회 순서 = List 순서 = @Order(BIOS→BMC→Subprogram).
//  중간 Duplicate*Exception 발생 시 @Transactional 전체 롤백 — 동일.

④ purge(board) — 자식 잔존 검사 (OR 결합)

BEFORE (3 존재검사)

private void purge(BoardModel board) {
  Long id = board.getId();
  boolean hasBios = !biosRepository.findAllByBoardModel_IdOrderByVersionDesc(id).isEmpty();
  boolean hasBmc  = !bmcRepository.findAllByBoardModel_IdOrderByVersionDesc(id).isEmpty();
  boolean hasSubprogram = !subprogramRepository.findAllByBoardModel_Id(id).isEmpty();
  if (hasBios || hasBmc || hasSubprogram) {
    throw new IllegalBoardModelStateException("자식 ... 남아 있어 ...");
  }
  boardModelRepository.delete(board);
}

AFTER (anyMatch)

private void purge(BoardModel board) {
  Long id = board.getId();
  if (children.stream().anyMatch(c -> c.hasAny(id))) {
    throw new IllegalBoardModelStateException("자식 ... 남아 있어 ...");
  }
  boardModelRepository.delete(board);
}
//  anyMatch = short-circuit OR. 메시지·예외·status 동일.

⑤ findDeletedChildLabels(id) — 휴지통 preview 라벨 (라벨 포맷 비대칭 흡수)

BEFORE (3 블록, 라벨 포맷 비대칭)

public List<String> findDeletedChildLabels(Long boardId) {
  List<String> labels = new ArrayList<>();
  biosRepository.findAllByBoardModel_IdAndIsDeletedTrue(boardId)
    .forEach(bios -> labels.add("BIOS: " + bios.getName()));
  bmcRepository.findAllByBoardModel_IdAndIsDeletedTrue(boardId)
    .forEach(bmc -> labels.add("BMC: " + bmc.getName()));
  subprogramRepository.findAllByBoardModel_IdAndIsDeletedTrue(boardId)
    .forEach(sp -> labels.add(sp.getKind().getDisplayName() + ": " + sp.getName()));
  return labels;
}

AFTER (flatMap)

public List<String> findDeletedChildLabels(Long boardId) {
  return children.stream()
      .flatMap(c -> c.deletedLabels(boardId).stream())
      .toList();
}
//  "BIOS: " / "BMC: " 리터럴 vs kind.displayName+": " 비대칭은
//  각 impl.deletedLabels 가 자기 포맷으로 생성 — 본체에서 소멸.
//  라벨 순서 = List 순서(@Order) = 기존 BIOS→BMC→Subprogram.
본체에서 사라진 것 — BiosRepository · BmcRepository · SubprogramRepository · BiosService · BmcService · SubprogramService import 6개 + BoardBIOS/BoardBMC/Subprogram 엔티티 import 3개가 BoardModelLifecycleService 에서 전부 제거. 본체는 이제 BoardScopedChildLifecycle 1 타입만 안다.
§4. 도메인 모델 — BoardScopedChildLifecycle SPI + 3 impl (before/after)

① SPI 인터페이스 (신규)

// board/service/cascade/BoardScopedChildLifecycle.java
public interface BoardScopedChildLifecycle {
    /** 부모 effective 변화 반영 — 활성 자식 effective 재계산. (cascadeRecompute) */
    void recomputeEffective(Long boardId);
    /** 활성 자식 동반 trash 이동. (softDelete) */
    void softDeleteActive(Long boardId);
    /** soft-deleted 자식 일괄 복구 — 복구 건수 반환. (restore cascade) */
    int restoreDeleted(Long boardId);
    /** 자식 잔존 여부 — purge 가드. (purge) */
    boolean hasAny(Long boardId);
    /** soft-deleted 자식 라벨 — 휴지통 preview. (findDeletedChildLabels) */
    List<String> deletedLabels(Long boardId);
}
//  5 메서드 = §3 의 5 cascade 책임에 1:1. 전부 boardId 만 받는 균질 표면.
균질 표면 = 비대칭 흡수의 핵심 — SPI 가 모두 (Long boardId) 단일 인자로 통일되어 있어, BIOS/BMC 의 2-arg·Subprogram 의 1-arg 차이는 impl 구현 내부에만 존재한다. BoardModelLifecycleService 는 이 균질 표면만 호출.

class BiosBoardScopedChildLifecycle implements BoardScopedChildLifecycle {
  private final BiosRepository biosRepository;
  @Lazy private final BiosService biosService;   // D3 — @Lazy 어댑터로 이동
  public void recomputeEffective(Long boardId) {

      .stream().filter(b -> !b.isDeleted()).forEach(BoardBIOS::recomputeEffective);
  }
  public void softDeleteActive(Long boardId) {

★ 결정 D3 — @Lazy 처리

현재 BoardModelLifecycleService 가 자식 service 를 @Lazy 로 주입하는 이유는 형제·scanner 를 거친 speculative 순환 차단. 다형화하면 이 @Lazy 가 어댑터 impl 로 이동한다(어댑터가 자식 service 를 보유).

옵션형태판단
유지 권장어댑터 impl 의 자식 service 필드에 @Lazy 유지 (behavior-preserving)기존 거동 무변화. 가장 안전. R3-4 는 다형화에만 집중.
실측 후 제거./gradlew bootRun 으로 순환 0 확인 후 @Lazy 제거의존 그래프가 어댑터 경유로 바뀌어 순환이 사라졌을 수 있음. 단 실측 전 제거는 speculative. 제거 가능 시 R3-4 내 부수 처리 허용.

권고 : 일단 @Lazy 유지(behavior-preserving), bootRun 실측에서 순환 0 이면 R3-4 내 제거 가능. 제거는 검증된 부수효과로만.
