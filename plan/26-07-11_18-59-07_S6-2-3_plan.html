S6-2-3 — 휴지통 파일이 사라진 두 상황의 탐지와 해결 CP1

S6-2 캠페인 3/3 · 2026-07-11 KST · 이 슬라이스로 드리프트 10종 전부가 실동작하게 된다

S6-2-2 가 "삭제한 파일이 밖에 살아 있는" 두 상황을 켰다면, 이번엔 반대로 "있어야 할 것이 휴지통에서 사라진" 두 상황을 켭니다. 여기에 더해 조사 중 발견한 영구삭제의 실물 잔존 공백(아래 ③)을 함께 보강하고, 유령 기록 탐지를 새 분류 체계로 통합해 휴지통 대조를 완성합니다.

🎬 드리프트 상황 시뮬레이터의 "휴지통 소실"·"잔여 마커" 버튼이 이 슬라이스의 완성 모습입니다 — 지금은 "S6-2-3 예정" 표시가 붙어 있고, 완료 시 현행화됩니다.

상황 ① — 휴지통 자원 소실 (휴지통 파일이 사라짐)

휴지통 자원 소실 내가 확인해 주면 시스템이 기록 정리 + 감사 기록

실제 상황 : 디스크 청소 스크립트·용량 정리·수동 rm 이 휴지통 폴더까지 지웠다. 복원이 불가능해진 상태.
지금은 : 완전 침묵 — 복원을 시도하는 순간에야 "휴지통 파일이 없다"는 거절로 드러난다.
앞으로는 : 매시간 점검이 발견 — "휴지통 자원 소실" 카드(노랑 — 확인 필요).
[적용]하면 : 남은 기록을 정리하되, 기존 영구삭제 파이프라인을 그대로 태워 감사 기록(purge log)에 "언제·무엇이·어떤 경로로 정리됐는지"가 자동으로 남는다 — 감사 화면(휴지통 정리 이력)에서 다른 삭제 이력과 함께 조회된다. 복구 불능 확정은 되돌릴 수 없는 판단이라 무인 자동은 불허.
거절되는 경우 : 적용 시점에 휴지통 파일이 다시 나타나 있으면(백업 복원 등) "상태가 바뀌어 처리할 수 없다"며 재점검을 안내 — 멀쩡해진 자원을 지우는 사고 차단.

상황 ② — 잔여 마커 정리 (휴지통 파일 옆 마커 찌꺼기)

잔여 마커 정리 필요 시스템이 해결 — 자동 처리도 가능

실제 상황 : 삭제 처리 때 마커 정리 단계가 디스크 오류로 실패해 경고만 남기고 넘어간 찌꺼기. 또는 위치 이탈 회수 전에 딸려 들어간 마커.
지금은 : 영구 미발견 — 휴지통은 점검 수색 범위 밖이라 볼 기회 자체가 없다.
앞으로는 : 매시간 점검이 발견 — 휴지통 기록이 있는 자원마다 그 옆 마커를 직접 들여다본다(수색 확대 없이 정확한 위치만 확인). 다른 판정과 독립 — 예컨대 위치 이탈 카드와 동시에 뜰 수 있다.
[적용]하면 : 마커 파일 하나만 지운다. 자원·기록은 안 건드리는 무해한 청소라 자동 후보.

상황 ③ — [발견된 공백] 영구삭제가 휴지통 실물을 남긴다

선행 결함 보강 이 슬라이스 조사에서 발견

무슨 문제 : 휴지통 자원을 영구삭제(직접·TTL 자동 모두)하면 기록은 지워지는데 휴지통의 실물 파일은 그대로 남는다. 삭제 절차가 원래 자리의 부산물만 정리하도록 만들어져 있어서다(삭제 시 원래 자리 경로를 보존하는 설계의 부작용). 휴지통은 점검 범위 밖이라 이 잔존 파일은 영원히 발견되지 않고 디스크만 차지한다.
왜 지금 : 상황 ①·② 가 "휴지통 정합"을 완성하는 슬라이스인데, 정작 영구삭제가 매번 새 잔존물을 만들면 밑 빠진 독이다.
보강 : 영구삭제의 단일 통로(세 진입경로가 모두 지나는 실행기)에서 성공 직후 휴지통 실물을 함께 정리 — 한 곳 수정으로 전 도메인·전 경로에 적용된다.

함께 정리되는 것 — 유령 기록 탐지의 통합

유령 기록(GHOST) 탐지를 새 휴지통 대조 패스로 흡수

지금은 유령 기록만 별도 통로(도메인별 유령 조회)로 찾는다. S6-2-2 가 만든 "삭제 자원 전수 대조" 패스가 이미 같은 정보를 다 들고 있으므로, 유령 판정도 그 안으로 통합한다 — 같은 정의(휴지통 기록도 실물도 원위치 파일도 전부 없음), 같은 결과, 통로만 하나로. 이로써 삭제 자원의 다섯 상태(정상/복귀/이탈/소실/유령)가 한 패스에서 한 번에 판정되고, 남는 옛 통로(용도가 사라진 조회 창구 1개)는 함께 제거해 반쪽 상태를 남기지 않는다.

소실 정리는 영구삭제 파이프라인 재사용 — 감사 기록을 공짜로

기록 정리를 직접 구현하지 않고 기존 영구삭제 실행기에 새 진입경로("점검 — 휴지통 소실 정리")를 하나 추가해 태운다. 감사 기록 작성·실패 처리·이력 화면 노출이 전부 기존 인프라 그대로 따라온다(중복 구현 0). 자원명 입력 확인은 요구하지 않는다 — 실물이 이미 없어 잃을 것이 없는 기록 정리라는 점에서 유령 기록 정리와 같은 등급(선례).

모든 해결은 실행 직전 상태를 재확인한다

점검 보고는 스냅샷이라 화면과 실제 사이에 시간 차가 있다. 소실 정리는 "파일이 정말 아직 없는지", 잔여 마커 정리는 "자원이 정말 아직 휴지통에 있는지"를 실행 직전에 다시 확인하고, 달라졌으면 "상태가 바뀌어 처리할 수 없다"(409)로 정중히 거절한다 — S6-2-2 검증이 남긴 "탐지를 켤 때 해결을 함께, stale 화면에 안전하게" 원칙의 적용.
