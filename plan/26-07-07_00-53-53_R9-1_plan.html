§2. 요구사항 · 설계 결정 (D1~D6, 대안 비교 포함)

요구: 사용자 불만 ① "스캔 후 화면을 새로고침 할 필요가 있다" 완결 해소 + 불만 ③ 일부(재발급 부분 실패 표면화). 신규 인프라 발명 없이 기존 자산(폴링/토스트/이벤트) 접속으로 해결한다.

D1. 완료 결과 metadata — complete(jobId, 결과 map) 오버로드 + 응답 병합 채택

BackgroundJob 에 volatile resultMetadata 필드 + complete(Map) 오버로드 추가(기존 complete() 는 빈 map 위임). BackgroundJobResponse.from 이 등록 metadata 와 병합해 기존 metadata 필드로 응답 — background-jobs.js:327 가 이미 metadata 를 이벤트 detail 에 실어주므로 전역 JS 무변경. 키 충돌 금지 규약을 javadoc 에 명시(등록 키는 도메인 식별자, 결과 키는 결과 수치 — reconciliation 은 driftCount/reissueSucceeded/reissueFailed 사용).
비채택: ⓐ 응답에 별도 resultMetadata 필드 — 전 페이지 로드 전역 JS(background-jobs.js) 수정 + 소비자(os-list.js)의 metadata 계약과 이원화. ⓑ 등록 시점 metadata 에 예측값 — 건수는 완료 시점에만 확정되므로 불가(검증 렌즈 高 지적).

D2. stage 계측 — performScan 내부 (4)분류/(6)저장 경계에 startStage 채택

dead 파라미터였던 jobId 를 활용해 (4) 직전 CLASSIFYING, (6) 직전 PERSISTING 진입을 계측. 트랜잭션 정합 우려는 성립하지 않는다: startStage 는 RUNNING 표시일 뿐이고 DONE 은 performScan 리턴(=커밋) 후 runAsync 의 complete() 가 채우며, 롤백 시 fail() 이 현재 단계를 ERROR 로 마킹하므로 화면 표시가 실제와 일치한다(비판 검증의 "커밋 전 완료 신호" 우려는 startStage=DONE 으로 오해한 것 — 절충안 불필요).
비채택: @TransactionalEventListener AFTER_COMMIT — 저장소 선례 0 + 표시 전용 신호에 과설계.

D3. 재발급 전용 stage — ReissueStage { RESIGNING("마커 재서명") } 단일 단계 채택

스캔 3단계 차용(거짓 라벨)을 해소하는 기존 결함 수정. 재발급은 순회-재서명 단일 작업이라 1단계가 실체와 일치.

D4. 완료 반영 정책 — 자기 트리거 스캔만 자동 reload 채택

이벤트동작근거
completed · PATH_RECONCILIATION · 자기 jobIdsessionStorage 에 토스트 문구 기록 → location.reload() → 로드 직후 읽고 즉시 삭제 후 토스트 표시("점검 완료 — 드리프트 N건")reload 가 곧 갱신. 토스트를 reload 너머로 전달하는 sessionStorage read-즉시-delete 는 os-list.js:176-182 선례. 삭제 누락 시 재방문마다 낡은 토스트 재노출(비판 검증 지적)
completed · PATH_RECONCILIATION · 타 jobId (주기 스캔 등)자동 reload 하지 않음 — 페이지 배너 "새 점검 보고서 도착 — [새로고침]"열람 중 예고 없는 화면 리셋 방지. bgjobToast 는 텍스트 전용(액션 버튼 미지원)이라 전역 토스트 확장 대신 페이지 로컬 배너 — 전역 자산 변경 0

failed · 두 타입 · 자기 jobId토스트(error) + 버튼 원복시작 실패(HTTP 오류)는 기존 ErrorModal 경로 유지 — 진행 중 실패만 이벤트로 수신

reload 후 같은 완료 이벤트 재발화 없음 — background-jobs.js:342-349 seedTerminalSeen 이 첫 폴링의 기존 종료 Job 을 침묵 등록(실측 확인). 이벤트 detail 의 job 식별자 필드명은 id(트리거 응답의 jobId 와 다름 — 구현 시 오기 주의, 비판 검증 지적).

D5. reload 위치 보존 — ReconciliationController 에 selectKey/selectId alias 수용 채택

Miller 드릴다운 시 os-list.js 가 URL 을 ?selectKey=..&selectId=.. 로 동기화하는데 컨트롤러가 그 이름을 안 읽어 reload 마다 위치가 초기화된다. 명시 파라미터(selectReportId/selectDriftId) 우선 + alias fallback 으로 수용 — 자동 reload UX 의 전제 결함 수정(비판 검증 발견). 비채택: os-list.js 파라미터명 변경 — management 전 페이지 공용 JS 라 blast radius 과대.

D6. 버튼 상태 — 고정 3초 타이머 제거, terminal 이벤트로 원복 채택

트리거한 jobId 의 completed/failed 수신 시 라벨 원복. 시작 토스트("자원 점검을 시작했습니다") 추가 — management 업로드와 동일한 시작 피드백. 동시 실행은 서버 AtomicBoolean + 409 가 이미 차단.
