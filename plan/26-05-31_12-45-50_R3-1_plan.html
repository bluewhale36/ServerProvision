§1. 현재 상태 (진단 — workflow wmjx7pfcz)
근본 원인 — BoardModelService 가 biosRepository / bmcRepository + @Lazy biosService / bmcService 두 도메인 의존만 갖고 subprogram* 의존이 0 이다 (필드 주석 L50-51 이 "A5 추가 시 DriverRepository 도 함께 주입… 3회 반복 확보 후 이벤트 기반 리팩터 검토" 를 예고만 하고 미이행). Subprogram(MA5)이 BIOS(MA3)/BMC(MA4) cascade 기계 완성 뒤 합류하며 관계 배선이 빠졌다. 단순 cascade 누락이 아니라 BoardModel 이 Subprogram 도메인을 아예 인지하지 못함.

§2. 요구사항 + 비목표

채택 = BoardModelService 가 Subprogram 도메인을 1급으로 인지하도록 의존 배선 + BIOS/BMC 와 동형으로 board-scoped Subprogram 을 전 lifecycle cascade · purge 잔존검사 · 확인모달 · 집계에 합류. 공용(boardModel=null) 은 boardModel.id 매칭 쿼리로 자연 제외.

  1. 의존 주입 — BoardModelService 에 subprogramRepository + @Lazy subprogramService 추가 (L50-51 미이행 주석 이행).
  2. cascade 5 전이 — disable-toggle / deprecate / undeprecate (entity-direct) + softDelete / restore (service 위임) 에 board-scoped Subprogram 루프 추가.
  3. purge 잔존검사 — hasSubprogram OR 합류 (자식 먼저 정리 강제).
  4. 집계 + 모달 — BoardModelResponse.subprogramCount + board/list.html 모달 카피(delete/deprecate/restore) 동적화 + "현재 연결" Subprogram 노출.
  5. Repository SPI — SubprogramRepository 에 board-scoped(kind 무관) + 삭제상태별 파생 쿼리 추가.
비 목표 :
  • includeDeleted/Miller 선택 보존 parity (진단 I1) — cascade 와 직교 → 별도 CH 슬라이스.
  • BIOS/BMC/Subprogram 3중 복붙 cascade 루프의 추상화 (child-service 인터페이스 / @EventListener) — over-abstraction 위험 평가 후 R3-2(후보). 본 슬라이스는 동형 루프 추가(최소-안전).
  • 모달 자식요약을 BE(ConfirmModalType)로 옮기는 아키텍처 변경 — 현 JS+data-* 패턴 유지(개수만 동적 산출).
  • R2-2-2(restore 중복키) 보류. 형제 BIOS/BMC 코드 무수정.
  • cascade 위임 시그니처 통일(2-인자) — open question(§8 Q1).
🎬 미리보기 — cascade BEFORE/AFTER 라이브

부모 메인보드 액션을 눌러보세요(비활성화 / Deprecated / 삭제 / 복구). 자식 4종이 형제 BIOS 기준으로 따라가는지 라이브로 보입니다. BEFORE 에서 비활성화/Deprecated/삭제하면 — BIOS/BMC 는 전이되는데 보드전용 Subprogram 만 잔존(⚠ 비대칭). AFTER 면 형제와 동일하게 전이. 단 활성화(enable) 는 양쪽 다 cascade 안 함(자식 개별 활성 — OS↔ISO 동형 "비활성화는 전이 · 활성화는 전이 X"), 공용 Subprogram 은 부모가 없어 항상 제외됩니다. (모드 전환 시 초기화)

§4. 도메인 모델 (SSOT)

1) SubprogramRepository — board-scoped 파생 쿼리 4 추가 (BIOS/BMC 동형, @Query 불필요)

2) BoardModelService — 의존 주입 + cascade 7 지점 (BIOS/BMC 동형)

// 필드 (L52-59 옆)
private final SubprogramRepository subprogramRepository;
@Lazy private final SubprogramService subprogramService;

// toggleEnabled (243-253 동형) — entity-direct. 비활성화만 cascade, 활성화는 early-return (OS↔ISO 동일)
if (!parent.isEnabled())   // 부모가 방금 disable 된 경우에만
    subprogramRepository.findAllByBoardModel_Id(id).stream()
        .filter(Subprogram::isEnabled).forEach(Subprogram::toggleEnabled);
// deprecate (262-267 동형) — entity-direct
subprogramRepository.findAllByBoardModel_Id(id).stream()
    .filter(s -> !s.isDeleted() && !s.isDeprecated()).forEach(Subprogram::deprecate);
// softDelete (299-302 동형) — service 위임 (board.softDelete 前)
subprogramRepository.findAllByBoardModel_IdAndIsDeletedFalse(id)
    .forEach(sp -> subprogramService.softDelete(sp.getId()));
// restore cascade (339-346 동형, board.restore 後)
for (Subprogram sp : subprogramRepository.findAllByBoardModel_IdAndIsDeletedTrue(id)) {
    subprogramService.restore(sp.getId()); restored++; }
// purge 잔존검사 (397-399) — OR 합류
boolean hasSubprogram = !subprogramRepository.findAllByBoardModel_Id(id).isEmpty();
if (hasBios || hasBmc || hasSubprogram) throw new IllegalBoardModelStateException(...);
R2-2-1 양방향 공존 — R2-2-1(자식→부모 promote 가드)과 R3-1(부모→자식 cascade)이 SubprogramService.restore/undeprecate/toggleEnabled 를 공유한다. (1) toggle/deprecate/undeprecate cascade 는 entity-direct 라 service 가드 미경유.
