§1. 현재 상태 · 목표

R2 재검증(2026-06-28, 10-agent workflow)으로 R2-3 을 "단일 advice 물리 통합 + @Order 제거"에서 "라우팅 드리프트를 구조적으로 닫기"로 재초점했다. 물리 통합 헤드라인은 제외 — live 결함을 안 닫고(v3 가 라우팅·HTML-bleed 이미 차단) R1-4-1 3차 회귀 계열(HIGHEST_PRECEDENCE 산술 / Accept */* fallthrough) + webflux 공존 blind spot 의 흉터 위험만 진다.

목표 — 예외→HTTP status 매핑을 @ResponseStatus 다형(SSOT)으로 끌어올려 분기를 줄이고, browse 컨트롤러 try/catch 안티패턴을 advice 로 승급(R4-1 이 명시적으로 R2-3 으로 이월해 둔 작업)하며, 라우팅 invariant 를 회귀 가드로 고정한다. 동작 불변 — 사용자가 보는 status·바디는 그대로다.
코드가 이미 절반 준비됨 — 양 advice 의 handleDomain 이 이미 AnnotationUtils.findAnnotation(@ResponseStatus) 로 status 를 흡수한다(ApiExceptionHandler:203 / WebExceptionHandler:110). browse 예외도 이미 도메인 계층 정합(BrowseTargetNotFoundException extends NotFoundException, BrowseTargetNotDirectoryException extends ConflictException). 본 슬라이스는 그 토대 위에 super 어노테이션을 달고 catch 를 떼는 일이다.
§2. 현 라우팅 위상 (실측)

두 advice 모두 @Order(HIGHEST_PRECEDENCE) + produces 매처로 Accept 헤더에 따라 분리:

  • ApiExceptionHandler(@RestControllerAdvice) — JSON 16핸들러. produces=json. handleDomain fallback 이 @ResponseStatus 흡수(없으면 500).
  • WebExceptionHandler(@ControllerAdvice) — HTML 5핸들러(NotFound/ChildLifecycleBlocked/Conflict/OptimisticLock/Domain fallback). produces=text/html. handleDomain 이 @ResponseStatus 4xx 흡수(없으면 500).

§3. 변경 상세 (4 항목)

① 상태 매핑 다형화 (@ResponseStatus SSOT)

// global/exception 도메인 super 에 어노테이션 부착 (additive — 현재 모두 미부착)
@ResponseStatus(HttpStatus.NOT_FOUND)   abstract class NotFoundException
@ResponseStatus(HttpStatus.CONFLICT)    abstract class ConflictException
@ResponseStatus(HttpStatus.BAD_REQUEST) abstract class FieldBoundBadRequestException

효과: 양 advice handleDomain 이 hierarchy 로 status 흡수 → FieldBoundBadRequest SSR 비대칭(latent) 소멸(@ResponseStatus(400) → Web handleDomain 이 400 매핑). ConflictException 부착으로 그 하위(FieldBoundConflict 포함)도 일관.

② 일반 핸들러 수렴 (분기 감소)

①의 결과로 plain-body 전용 핸들러 handleNotFound·handleConflict(양 advice 4개)가 handleDomain 과 중복 → 삭제·수렴. 특수 body 핸들러는 유지(FieldBoundConflict/FieldBoundBadRequest=fieldErrors, Nudge=NudgeRequiredResponse, SoftDeleteReject=DeleteRejectResponse, ChildLifecycleBlocked=6필드 로그, OptimisticLock=메시지 치환). 로그 라벨이 advice.notFound→advice.domain.mapped 로 바뀌는 것만 수용.

③ browse 4-catch → advice 승급

// InvalidBrowsePathException 만 @ResponseStatus 추가 (나머지 3종은 이미 계층 정합)
@ResponseStatus(HttpStatus.BAD_REQUEST) class InvalidBrowsePathException extends DomainException
//   BrowseTargetNotFoundException   extends NotFoundException  → 404 (자동)
//   BrowseTargetNotDirectoryException extends ConflictException → 409 (자동)
//   DirectoryBrowseIoException       extends DomainException    → 500 (handleDomain)

// Bios/OS/Bmc BrowseController.browse() : try/catch 삭제 → 한 줄 위임
return ResponseEntity.ok(directoryBrowseService.browse(new DirectoryBrowseRequest(pathParam, false)));

browse 는 @ResponseBody XHR → advice 의 JSON 핸들러가 동일 ApiErrorResponse 로 응답. 동작 100% 보존. Subprogram browse 는 이미 0-catch.

④ produces 명시 + 회귀 가드 (D4 결정 따름)

JSON-전용 4핸들러의 raw JSON bleed(latent) 대응 — D4에서 (A)produces 부착+whitelabel 수용 vs (B)가드만으로 invariant 고정 중 택. 어느 쪽이든 AdviceExceptionMappingTest 에 native-form·browse·@ResponseStatus 수렴·*/* 회귀 가드 신설은 필수(critique).

§4. 핵심 결정 (D1~D6, 비채택 병기)
#결정채택 / 비채택
D1advice 구조2-advice produces 유지(KEEP)(Boot 4 canonical). 비채택: 단일 advice 물리 통합 + @Order 제거(원안) — live 결함 0 + R1-4-1 3차 회귀(v1 underflow/v2 28건) 흉터 위험 + ExceptionLogPolicy 존재이유(2-advice drift 방지) 소멸 + webflux blind spot.
D2status 매핑@ResponseStatus(404/409/400) super 부착 → handleDomain 흡수(이미 findAnnotation). 비채택: 전용 핸들러에 status 계속 hard-code(분기 잔존).
D3핸들러 수렴plain-body handleNotFound/handleConflict 수렴(삭제), 특수 body 핸들러 유지. 비채택: 전부 유지(로그 라벨 보존하나 분기 미감소).
D4JSON-전용 4핸들러 bleed(권장 B) produces 미부착 + 회귀 가드로 invariant 고정 — latent(XHR 전용)에 Accept-매처 변(3차 회귀 계열) 위험을 안 들임. (A) produces 부착 + whitelabel 수용도 가능(구조적 close, 단 SSR 시 whitelabel fall-through + 가드 필수). 사용자 택 1 요망.
D5browse 승급InvalidBrowsePathException @ResponseStatus(400) + Bios/OS/Bmc browse try/catch 삭제. R4-1 이 R2-3 으로 명시 이월. 비채택: 유지(안티패턴 잔존).
D6회귀 가드AdviceExceptionMappingTest 신설/확장 필수: @ResponseStatus 수렴(NotFound/Conflict/FieldBoundBadRequest XHR+SSR status), browse 4 예외 status, native-form bleed/whitelabel(D4), */* pin. critique: 가드는 선택이 아니라 전제.
D4 가 유일한 미결 결정 — 나머지는 권장안 확정. D4 는 (B)가드-only(변경 최소·안전, 권장) vs (A)produces+whitelabel(구조적 close) 중 사용자 선택을 요청한다. 둘 다 회귀 가드는 필수.

§6. 예외 계층 · 부산물 · 주의

예외 계층

신규 0. 기존 super 3종(NotFound/Conflict/FieldBoundBadRequest) + InvalidBrowsePathException 에 @ResponseStatus 어노테이션만 추가. 클래스 신설·삭제 없음.

부산물 / 주의

  • 로그 라벨 변화 — 수렴된 NotFound/Conflict 가 advice.notFound→advice.domain.mapped. ExceptionLogPolicy 의 status 다형 레벨(4xx WARN)은 불변.
  • ChildLifecycleBlockedByParentException 은 이미 @ResponseStatus(409) + 전용 핸들러(6필드 로그) 보유 → 전용 핸들러 유지(로그 때문). 수렴 대상 아님.
  • R2-5 병합 아님 — nudge-expired errorCode 수리는 R2-6 에서. 본 슬라이스는 backend status 라우팅만.
  • browse 승급은 Bios/OS/Bmc 3 컨트롤러. Subprogram 은 이미 0-catch. Metadata 컨트롤러의 browse 외 catch 는 본 슬라이스 범위 밖(있다면 별도 식별).
