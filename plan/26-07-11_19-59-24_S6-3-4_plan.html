S6-3-4 — 내용 변경 수용 CP1 세부 설계 (구현 대기)

S6-3 마지막 슬라이스 · 2026-07-11 KST · CP1 승인(사용자) — 확인 창 표시는 감지 시각·정본 인정 시각 채택, 파일 수정 시각(mtime)은 위조 가능성으로 반려. 재업로드 1차 권장 원칙 반영

"내용 변경 감지" 카드에서 [현재 내용을 정본으로 수용] — 의도된 파일 교체를 사람이 선언하면, 시스템이 등록 지문을 현재 파일 기준으로 다시 계산해 갱신합니다. 잘못 누르면 실제 변조가 정상으로 굳는 가장 무거운 액션이라 자원명 입력 확인 + 감사 기록을 요구하고, 대용량 파일의 지문 재계산(수 분) 때문에 백그라운드로 실행합니다.

유보됐던 3가지 질문의 답

① 백그라운드 작업과 카드의 연결 — "연결하지 않는다"가 답

완료 시점에 서버가 카드를 제거하고, 화면은 기존 완료 알림 체계(점검·재발급과 같은 bgjob 이벤트)로 토스트를 받아 새로고침하면 끝 — 작업↔카드 매핑 테이블 같은 신규 배선이 필요 없다. 시작 시 응답으로 받은 작업 번호를 페이지가 기억(점검 버튼과 동일 패턴)하고, 완료 이벤트의 결과 값(자원 이름·성공 여부)을 토스트 문구에 쓴다. 실패 시 카드는 그대로 남아 재시도 가능(작업 실패 알림이 사유 표시).

② 자원명 입력 확인 창 — 영구삭제의 확인 창 계열을 재사용

영구삭제·nudge 가 쓰는 자원명 입력 확인 자산(typed-name modal)이 이미 검증돼 있다 — 드리프트 카드용 변형 1종을 그 계열에 추가한다(신규 발명 없음). 검증은 서버의 기존 TypedNameGuard(자원 표시명 일치) 재사용.
비채택: 표준 [적용] 폼에 입력칸 추가 — [적용]은 "누르면 그 자리에서 끝"이라는 동기 계약이라, 비동기 + 입력 검증이 섞이면 두 계약이 오염된다.

③ 점검 잠금과의 관계 — 공유하지 않는다

수용 작업 중 점검이 같은 마커를 읽어도 최악은 "일시적으로 옛 판정이 한 번 더 보고"되는 것 — 다음 점검에서 자연 소멸하는 무해한 경합이라 잠금 공유(=수 분간 점검 전면 차단)의 비용이 더 크다. 대신 같은 카드의 중복 수용은 시작 시점에 진행 중 작업 존재 여부로 거절한다.

동작 설계

단계내용
1. 버튼·확인내용 변경 카드에 표시되는 판단 재료(승인 검토 정보): 감지 시각 · 정본 인정 시각(마커가 마지막으로 서명된 때) · 등록 지문 · 감지 시점의 현재 지문(전문 — 벤더 공식 체크섬/변경 티켓과 대조용). 파일 수정 시각은 위조 가능성으로 표시하지 않음(반려 확정). 안내: "원본 재업로드가 더 안전합니다(검증 재통과). 의도된 교체가 확실할 때만 자원명을 입력해 수용하세요."
2. 시작(동기)카드 유효성·자원 존재·자원명 일치(TypedNameGuard)·중복 작업 검증 → 통과 시 백그라운드 작업 등록, 작업 번호 응답. 거절은 전부 409 + 이유.
3. 실행(비동기)내용 지문 재계산(대용량 수 분, 진행 단계 표시) → 감지 시점 지문과 대조 — 그 사이 파일이 또 바뀌었으면 실패 처리(사용자가 확인한 내용과 다른 것이 정본화되는 사고 차단 — Tripwire high-security 모드 등가) → 마커에 새 지문·새 서명 기록 + DB 갱신 → 카드 제거 → 완료(결과에 자원 이름). [AUDIT] 경고 로그.
4. 완료(화면)완료 알림 → "수용 완료 — {자원명}의 등록 지문을 현재 내용으로 갱신했습니다" 토스트 + 새로고침(카드 소멸 반영). 실패 → 사유 토스트, 카드 유지.
CP1 승인 (2026-07-11) — 아래 유보 사유는 승인으로 해소, CP4 까지 무중단 진행. 원문 보존: 구현 대기 사유(이력) — 보안상 가장 민감한 액션(변조 수용 선언)인 데다 확인 창·백그라운드 완료 배선이 이 캠페인의 다른 슬라이스와 달리 화면 자산을 새로 조합한다. 장시간 자율 세션의 끝자락에서 서두르기보다, 본 세부 설계를 검토받은 뒤 새 세션에서 착수하는 것이 안전하다고 판단(자율 위임 범위 안에서의 신중 선택). 캠페인의 나머지 전부(드리프트 10종 실동작 + 액션 3종)는 완료 상태.

검증 계획 (구현 시)

  • 자원명 불일치 / 카드 stale(자원 소멸·내용 원복) / 중복 작업 → 각 409 + 디스크 무변경
  • 수용 완료 후: 마커의 지문=현재 파일, 서명 유효, 카드 제거, 다음 정밀 점검 무보고
  • 재계산 도중 실패(IO) → 마커·DB 무변경(원자성) + 작업 실패 알림 + 카드 유지
  • DriftKindTest 배정표(HASH → MANUAL 또는 전용 축) 갱신 — 채택 형태는 구현 시 확정: 전용 엔드포인트+전용 버튼(잠정) vs resolution 프레임 편입(비동기 킥오프형) 양안을 코드 실측으로 판정
