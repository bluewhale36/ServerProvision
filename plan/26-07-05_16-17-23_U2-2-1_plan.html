🎬 미리보기 — 편집기 diff 수집 → flat JSON → 저장 판정 라이브 데모

이 slice 의 본질은 "registry 기반 편집 → 변경분(diff)만 flat JSON → 검증 → 영속" 이다. 아래 미니 편집기에서 값을 바꾸면 ① 기본값과 달라진 행만 diff 에 실리고(같으면 자동 탈락) ② 전송 JSON 이 라이브 조립되며 ③ 저장 버튼 차단 판정(name @NotBlank / 빈 diff)이 갱신된다. PASSWORD 속성은 위젯 자체가 출력되지 않음(구조적 UI 차단)과 readOnly 스킵도 재현했다 — 판정 SSOT 는 BiosAttributeType.templatable()/readOnly() 로 서버와 동일(드리프트 0).

템플릿 명칭 : 보드 : MS74-HB0 (선택 완료 — 랜딩에서 확정, 이후 불변)

SETUP003_FirmwareVersion READONLY 읽기 전용 — diff 수집 제외(기존 PoC 정책 승계)

드리프트 0 — 데모의 diff 수집(기본값 비교)·PASSWORD 미출력·빈 diff/name 차단 = 기존 bios-setup.js 수집 정책 + 신규 templatable() + BiosSettingValues 최소 1속성 invariant + @NotBlank 의 1:1 재현. JSON 의 값 타입(문자열/숫자/불리언)이 coerce 후 저장·Redfish 페이로드와 동형임도 표시된다.

구현 수준 결정 (설계 report 승계분 외 신규)

#결정근거 / 대안
D1VO 매핑 = AttributeConverter 2종 (autoApply=false)U1 선례(IpAddressConverter/MacAddressConverter) 그대로. values 컬럼은 columnDefinition="json"(U1 hardware_spec 선례). @Embeddable record 대안은 선례 이탈이라 기각.
D2요청의 attributes 는 Map<String,String>(폼 문자열) — 서버가 coerce 후 VO 화기존 PoC 계약(BiosSettingsSaveRequest)과 동일한 wire — 이관되는 JS 수집 로직 무변경. 타입화는 저장 경계에서.
D3편집기 뷰모델의 PASSWORD 제외는 뷰모델 조립 시점(Query 서비스)templatable() 단일 SSOT — 템플릿 편집기에는 위젯이 아예 없고(구조적 차단), Command 는 같은 판정으로 direct POST 를 400 거절. 구 PoC 렌더(renderMenu)는 무변경(BmcPassword 등 다른 소비자 고려 — 실제로는 소비자가 없어져 잔류 비대칭, §8).
D4목록의 "변경 속성 수" = values.size() 파생denorm 컬럼 기각(설계 report) — 상태 복제 drift 방지.
D5편집기 화면·JS 파일명 = bios-setting-editor.html/js 로 이관(git mv 후 개조)이관(move) 원칙 — diff 로 개조 내역이 추적됨. 복제 승격 기각(중복 금지).
D6navbar 'BIOS 세팅' — '세팅 정의서' 와 '서버' 사이guest 정의서 작성 전에 템플릿을 먼저 만드는 흐름을 nav 순서로 암시.
적대적 자기비판 — ① NotFound 예외 없이 목록·생성만으로 슬라이스가 성립하나? 성립한다 — 조회-by-id 표면이 없으므로 404 대상은 boardKey(기존 예외 재사용)뿐. 목록 행의 상세 링크는 -2 에서 활성화(그 전까지 비링크 텍스트 — §8 잔존). ② 검증 루프 '이동' 이 PoC save 폐지와 원자적인가? 같은 슬라이스의 같은 CP2~CP4 안에서 수행 — 두 소비자 창 0(설계 report 의 이관 원칙). ③ BiosSettingValues 최소 1속성 invariant 와 "빈 diff 400" 이 중복 아닌가? 아니다 — 같은 SSOT 다: Request 검증(@AssertTrue 또는 서비스 가드)이 VO invariant 를 호출-전 방어로 쓰고, VO 는 생성자에서 최종 강제(도메인 invariant if-throw — 정당한 imperative 가드). ④ 숨은 순환? biossetting → provisioning.domain/service(카탈로그) 단방향. setting 참조 없음(-3 에서 SPI 로 단방향 유지 예정). 순환 소지 없음.

§8. 부산물 / 주의 / 결정 사항

의도된 잔존 / 임시 비대칭 (+ 해소 시점)

  • 목록 행이 비링크 — 상세 화면이 없으므로. U2-2-2 에서 링크 활성.
  • 삭제·수정 UI 없음 — 생성만 가능. U2-2-2(PUT/DELETE) · U2-2-3(사용중 409).
  • BiosSetupService 잔류 비대칭 — save() 를 잃고 listBoards/renderMenu 만 남은 카탈로그 공급자(자체 엔드포인트 없음). 카탈로그 엔진 패키지 정비는 후속 R 후보(설계 report §04).
  • renderMenu 의 PASSWORD 미필터 — 구 PoC 렌더 경로가 사라져 실소비자는 신규 Query 서비스뿐이나, 시그니처 정리는 -2(overlay 오버로드 추가 시)에서 일괄.
  • 카탈로그는 여전히 파일 기반(application.properties boards[]) — 병합은 Stage 4 준비 슬라이스(Q3 승인).
