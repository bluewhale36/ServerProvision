1. 현재 상태 — 시스템 진입 hard-delete 경로의 결손

1.1 사용자 진입 vs 시스템 진입

S5-2-2 가 사용자 직접 hard-delete 경로에 typed-name 검증 을 도입해 우발 삭제를 방어했다. 그러나 사용자 입력이 부재한 두 가지 시스템 진입 경로는 여전히 결손 상태다.

진입 경로현재 상태결손
사용자 직접 (휴지통 / 5 list page 영구삭제 버튼) 정합 typed-name 검증 적용 (S5-2-2 + S5-2 typed-name 다형성) —
nudge REPLACE (등록 시 중복 자원 발견 → 사용자 "REPLACE" 클릭) 부분 service.purgeXxxForNudge(target) 직접 호출. typed-name 우회 (사용자 nudge confirm 클릭이 의도 증거) retry / 실패 알림 / 백업 정책 부재. I/O 실패 시 silent 500
TTL 자동 만료 (휴지통 30일 경과 자원의 cron 진입) 미구현 TrashTtlWorker.purgeExpired() 가 만료 식별 + 경고 로그만. 실제 hard-delete 미수행 전체. 정책 결정 필요

1.2 TrashTtlWorker.purgeExpired 의 현재 stub

@Scheduled(cron = "${trash.ttl.purge-cron:0 0 * * * *}")
public void purgeExpired() {
    Instant threshold = Instant.now().minus(trashPolicy.getTtl());
    for (MarkableScanner scanner : scanners) {
        List<Markable> expired = scanner.findTrashedBefore(threshold);
        if (!expired.isEmpty()) {
            log.warn("[trash:ttl] type={} expired count={} (threshold={}). 자동 hard-delete 는 S5-2-2 와 합쳐 결정 — 현재는 보고만.",
                    scanner.supportedType(), expired.size(), threshold);
        }
    }
}

코드 주석 자체가 "S5-2-2 와 합쳐 결정" 으로 위임을 명시. S5-2-2 범위에서 제외되었으므로 본 슬라이스에서 결론을 짓는다.

1.3 nudge REPLACE 의 retry / 알림 부재

6 도메인 NudgeService (OS / Bios / Bmc / BoardModel / Subprogram / OsImage) 가 동일 패턴으로 purgeXxxForNudge(target) 직접 호출. 성공/실패는 호출자 controller 의 HTTP 응답으로만 노출 — 실패 시 자원이 어떤 상태로 남는지, retry 가 가능한지 일관된 정책이 없다.

2. 페이지 요구사항 + 결정 항목

2.1 핵심 원리 (Notion 페이지에서 확정)

  • S5-2-2 의 typed-name 은 사용자 우발 삭제 방어용 — 시스템 자동 진입엔 적용 불가능
  • 시스템 진입의 보호 메커니즘은 TTL (30일) + 사전 알림 (7일 / 1일) + retry + 실패 복구

설계 원칙 모든 hard-delete 는 PurgeExecutor 단일 진입점을 통과한다. 사용자 직접 / nudge REPLACE / TTL 자동 — 3 진입점이 동일 retry / receipt / 알림 인프라를 공유. typed-name 검증만 controller / scanner 측 분기 (이미 다형성 적용).

3.2 PurgeExecutor 단일 진입점

// global/trash/PurgeExecutor.java (신규)
public class PurgeExecutor {
    public PurgeResult execute(PurgeRequest req) {
        // 1) receipt 저장 (실패 시 throw — hard-delete 안 함)
        purgeReceiptWriter.write(req);
        // 2) scanner.purgeFromTrash(id) 호출 (도메인 service 가 파일 + DB row 정리)
        try {
            attemptWithRetry(req);  // 1s/2s/4s backoff
            return PurgeResult.success(req);
        } catch (PurgeIoFailedException ex) {
            markFailed(req, ex);  // purge_failed_at 갱신
            return PurgeResult.failed(req, ex);
        }
    }
}

6. Step 8 통합 테스트 시나리오 (16 시나리오)

PurgeExecutorTest (단위)

  1. 성공 happy — receipt 저장 → scanner.purgeFromTrash 1회 호출 → PurgeResult.success
  2. retry 후 성공 — 1회 IOException → 2회째 성공. attemptCount 2, success
  3. retry 3회 후 실패 — 모두 IOException → markFailed 호출, purge_failed_at NOT NULL, PurgeResult.failed
  4. receipt 저장 실패 — receiptWriter 가 throw → hard-delete 시도 안 함 (선행조건 실패), PurgeResult.failed
  5. backoff 시간 검증 — 1s · 2s · 4s 의 sleep 호출 확인 (mockable clock)

TrashTtlWorkerTest (단위)

  1. 만료 자원 0건 — log 만 출력, PurgeExecutor 호출 0회
  2. 만료 자원 N건 — PurgeExecutor.execute(TTL 진입) N회 호출
  3. D+23 사전 알림 — JobType.TTL_NOTIFY 1회 등록 (7일 윈도우)
  4. D+29 사전 알림 격상 — JobType.TTL_NOTIFY 1회 등록 (1일 윈도우)
  5. auto-purge.enabled=false — purgeExpired() 가 식별만 + log, PurgeExecutor 호출 0회

RetryFailedControllerTest (HTTP)

  1. 성공 — purge_failed_at 보유 자원 2건 → PurgeExecutor 2회 호출, 200 응답
  2. 실패 자원 0건 — 200 응답 + count=0 메시지
  3. 인증 실패 — 비관리자 → 403

NudgeReplace 통합 (6 도메인 중 OS 만 회귀)

  1. nudge REPLACE 성공 — PurgeExecutor.execute(NUDGE 진입) 1회, 작업 조회 아이콘에 TRASH_AUTO_PURGE 카드 생성
  2. nudge REPLACE I/O 실패 — purge_failed_at 갱신, 409 응답 + 안내
  3. typed-name 검증 우회 — PurgeOrigin.NUDGE 진입은 typed-name 입력 없이 통과
7. 예외 계층

신규

  • PurgeIoFailedException extends DomainException — hard-delete 도중 IO/DB 실패. retry 대상.
  • PurgeReceiptWriteFailedException extends DomainException — receipt 저장 실패. hard-delete 차단.
  • PurgeRetryExhaustedException extends DomainException — 3회 연속 실패 시 자원 단위 격상. @ResponseStatus(SERVICE_UNAVAILABLE) 가능성 검토 (사용자 진입에선 안 만남)

재사용

  • TypedNameMismatchException (S5-2-2) — 사용자 직접 진입 검증 단계에서만 발생. PurgeExecutor 진입 전 controller / scanner 의 typed-name 검증 통과 후 호출되므로 본 슬라이스 변경 없음
  • ChildLifecycleBlockedByParentException (S5-2-3-1) — restore 측 거절. purge 와 무관
8. 예상 부산물 · 주의

스코프 경계

  • 외부 알림 채널 (이메일 / Slack) 도입 안 함 — 작업 조회 아이콘 + 서버 로그로 충분. 운영자가 외부 채널 도입 시점에 별도 슬라이스
