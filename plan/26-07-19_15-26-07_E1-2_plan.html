§1. 현재 상태 (진단)

E1-1 이후 게스트는 진단 리눅스로 부팅해 배너를 띄우고 WAIT 폴링에 머문다. 프로토콜의 나머지 절반이 비어 있다:

자산있음 (E1-1 까지)비어 있음 (이 슬라이스가 채움)
에이전트 지시AgentDirective.WAIT 골격 + "E1-2 가 COLLECT·REBOOT 추가" 예고 주석지시 내용 없음 — 게스트는 수집하라는 말을 못 듣고 영원히 대기한다
하드웨어 인벤토리GuestServerDetail.hardwareSpec/softwareSpec JSON 컬럼(U1 예약 — 항상 null) + 상세 페이지 "— (진단 단계에서 수집)" placeholder + DiscoveryStage.DIAGNOSTIC_ENRICHED 상수수집·파싱·적재 전부 미구현 — boardSerial 도 빈 채. BMC 신원(IP/MAC)의 저장 자리 자체가 없다
완주markCompleted 도메인 메서드 + PhaseSequence 종단 규약(DEC-25) 완비실호출자 0 — completedAt 이 기록될 길이 없어 어떤 서버도 PROVISIONED 에 도달 못 한다
dispatch 4행completedAt → exit(로컬 부팅 폴스루)위험 — U3 전 진단만 완주한 베어메탈은 OS 가 없어 exit = 부팅 실패 루프(로드맵 §3-E1-3 지적). 완주를 실배선하는 순간 4행 이분이 필수가 된다
실패 대응failedAt 신호 + 게스트 FAILED 보고 트리거(E1-0b)운영자 쪽 절반 없음 — 무보고 침묵(UC-4)을 실패로 전환할 수단도, FAILED 를 풀 재시도도 없다(javadoc "해제는 운영자 재시도 액션(E1-2)" 예고)
관찰성 (DEC-32)—게스트 접촉 시각 기록·표시 없음 — "개시 안 눌렀는데 서로 대기"(UC-3)·"침묵 감지"(UC-4)가 화면에 없다
핵심 진단 — E1-0a 가 만든 신호 4컬럼 중 completedAt 만 아직 실트리거가 없다. 이 슬라이스가 수집→적재→완주를 이으면 게스트 한 대가 등록부터 종단(입고 검수 대기)까지 처음으로 완주하고, 상세 페이지의 모든 placeholder 가 실값으로 바뀐다. 물리 서버 불요 — 수집 경로는 QEMU -smbios 주입으로 실보드와 동일하게 재현된다(유스케이스 토론 §1-2 합의).
§2. 요구사항 (확정) + 비목표
  1. 지시 프로토콜 확장 — AgentDirective +COLLECT·REBOOT. 체크인 응답 판정: 커서 DIAGNOSE_LINUX + 미수집(DiscoveryStage.IPXE_REGISTERED) → COLLECT / 완주(isCompleted) → REBOOT / 그 외 → WAIT.
  2. agent.sh v2 수집 임무 — COLLECT 수신 시: INFORMATION_COLLECTING open → 수집 → JSON 조립 → close(SUCCEEDED, statusMeta=수집 결과). REBOOT 수신 시 재부팅. 수집 항목(2026-07-19 사용자 확정 스펙):
    • CPU — 제조사 + 모델명만 (dmidecode -t processor). 코어 수 등 불요.
    • 메모리 — DIMM 슬롯별 1행: 슬롯 위치 · 제조사 · 용량 (dmidecode -t memory type 17 — 몇 개 꽂혔는지는 행 수가 말한다).
    • 디스크 — SSD/HDD 구분 + HDD 는 SAS/SATA 구분 + 용량 (lsblk -o NAME,SIZE,ROTA,TRAN). 한계 명기: RAID 카드 뒤 물리 디스크는 OS 에 안 보인다 — 벤더 CLI(storcli 류) 동봉이 필요한 OPEN-1 후속에서 확장, 이번엔 OS 가시 디스크(온보드 SATA·NVMe·JBOD)까지.
    • PCIe 슬롯 — 꽂힌 카드 목록 (lspci): RAID 카드(제조사+모델명) · LAN · 10G LAN(UTP / SFP+) · Fibre(16Gb / 32Gb) · NVIDIA GPU(모델명). 종류 분류는 lspci 모델명 기반 규칙 — 실보드 실측(T3)으로 보강.
    • 보드·펌웨어 — 보드 시리얼 · BIOS 버전 (dmidecode -s) + ipmitool BMC 조회(미검출 = 정상 degrade — BMC 필드 생략).
    수집 도구 추가분(pciutils · lsblk)은 부분 apk 저장소 fetch 목록에 동봉(build-assets.sh 갱신).
  3. 서버 수집 소비 — SPI 확장(DEC-6 의 "두 번째 실물" 시점) — ProvisioningPhaseExecutor 에 보고 소비 훅 추가, DiagnoseLinuxExecutor 가 구현: 관용 파싱(파싱 가능한 필드만 구조화, 원문은 원장 statusMeta 가 보존 — raw 컬럼 신설 불요) + placeholder 필터("To Be Filled By O.E.M." 류 → null 처리, boardSerial UNIQUE 충돌 회피) → GuestServerDetail 갱신(boardSerial·hardwareSpec·softwareSpec·BMC 신원 구조화 2컬럼) + DIAGNOSTIC_ENRICHED 승급 + INFORMATION_PERSISTING 단발 기록.
  4. 수집 계약 Value Object — hardwareSpec/softwareSpec record 를 vo/ 에(로드맵 확정 — spec/ 신설안 탈락), Jackson 3(tools.jackson.*) 직렬화.
  5. 완주 판정 실배선(DEC-25) — 수집 적재 성공 시 PhaseSequence.nextAfter(DIAGNOSE_LINUX, 보유집합): U3 전 보유 = 빈 집합 → empty → markCompleted. 다음 체크인이 REBOOT 지시.
  6. dispatch 4행 이분(로드맵 D3 폴백 이분) — completedAt + 커서 < OS_INSTALLING → 입고 검수 대기 스크립트(sleep+chain — U3 할당이 생기면 이 폴링이 재개 트리거) / 커서 ≥ OS_INSTALLING → 기존 exit.
  7. 접촉 관찰성(DEC-32) — guest_server.last_seen_at: /boot·에이전트 API 접촉 시 갱신(판정 입력 아님 — DEC-2 무관 관찰 로그). 상세·목록에 "접촉 중(N초 전)/무접촉(N분)" + '등록됨·접촉 중·미개시' 조합에 개시 유도 안내(UC-3).
  8. 운영자 액션 2종(DEC-4) — ① 수동 FAILED 전환: PROVISIONING 에서만(무보고 침묵 대응 — UC-4), failedStepCode=null 허용(수동 표식). ② 재시도: FAILED 에서만, clearFailed()(전진 가드의 유일한 명시 예외) → 커서 유지 재개. 펌웨어 실패(BIOS/BMC_UPDATING)는 재시도 차단(벽돌 리스크) — 차단 조건 도메인 메서드 1개를 UI disabled+tooltip 과 서버 가드가 공유(SSOT).
  9. 검증 자산 확장 — run-guest.sh 에 SMBIOS 확장 주입(type=0 BIOS 버전·type=2 보드 시리얼 — placeholder 값 주입 변형 포함), 하네스 diagnose-cycle.sh v2(COLLECT 시퀀스), T2 랩 저널에 캠페인 적재.
비 목표 :
  • IPMI 각인 — DEC-11 보류 그대로 (상수 IPMI_SETTING 유지, 소비 없음).
  • U3 할당 감지·재개 — 보유 집합은 빈 집합 상수. 대기 스크립트의 "할당 감지 후 재개" 는 U3 가 보유 공급을 배선할 때 자동 성립(폴링 루프가 이미 재개 트리거).
  • V3(실보드 ipmi 모듈)·V8(실보드 placeholder 실측) — T3 체크리스트 적립 그대로. placeholder 필터는 알려진 패턴 목록으로 시작, 실측 후 보강.
  • 자동 타임아웃 워치독 — 무접촉 자동 FAILED 는 실측 후(로드맵 D6 명시 유보). 이번엔 표시 + 수동 전환만.
  • 회차 모델(DEC-29)·재투입(DEC-30)·중단(DEC-31) — 별도 슬라이스.

§4. 도메인 모델 (SSOT)

지시 판정 — 체크인 응답 (AgentReportService, 분기 아닌 우선순위 규약)

REBOOT : progress.isCompleted()                          — 완주했으니 진단 리눅스를 떠나라
COLLECT: 커서 DIAGNOSE_LINUX && detail.discoveryStage == IPXE_REGISTERED  — 수집 임무
WAIT   : 그 외 (수집 적재 완료 후 판정 대기 등 과도 상태)
// 멱등: COLLECT 재수신(응답 유실 재체크인)은 무해 — 소비 훅이 승급을 1회만 수행(ENRICHED 재적재 no-op)

SPI 확장 — 보고 소비 훅 (DEC-6 "두 번째 실물이 나타날 때" = 지금)

public interface ProvisioningPhaseExecutor {
    ProvisioningPhase phase();
    String bootScript(GuestServer server, ProvisioningProgress progress, String rebootQuery);
    /** 이 phase 의 step 종결 보고 소비(E1-2 신설) — default no-op: 소비할 게 없는 실행기는 구현 안 한다. */
    default void onStepClosed(GuestServer server, ProvisioningProgress progress, SetupStep step) { }
}

  + retryBlocked()           — failedStepCode ∈ {BIOS_UPDATING, BMC_UPDATING} → true (벽돌 리스크, DEC-4)
                               ← UI disabled+tooltip 과 서버 가드가 이 메서드 하나를 공유 (SSOT)

§8. 예상 부산물 / 주의 / 결정 사항

잔존 책임 · 주의 (정직 기록)

  • V3·V8 은 여전히 실보드 몫 — QEMU 는 ipmi 가 없어 BMC 축은 degrade 경로만 검증된다. placeholder 필터도 패턴 목록 기반 — 실측(T3) 후 보강 전제.
  • RAID 카드 뒤 물리 디스크는 이번에 안 보인다 — OS 는 컨트롤러의 논리 볼륨만 본다. 물리 디스크 슬롯·구성 수집은 벤더 CLI 동봉(OPEN-1 — storcli 류, E4 전 조사)에서 확장. PCIe 목록에 RAID 카드 자체(제조사+모델)는 이번에 잡힌다.
  • PCIe 종류 분류(10G UTP/SFP+ · FC 16/32Gb)는 모델명 규칙 기반 — 사내 사용 카드 목록으로 시작하고 실보드 lspci 실측(T3 항목 추가)으로 보강. 미분류는 ETC + 원문 그대로 표시(수집 유실 없음).
  • 입고 검수 대기는 무한 폴링 — U3 전엔 할당이 생길 수 없어 완주 서버는 계속 30초 폴링한다(전원을 끄면 침묵 — 정상). 자동 타임아웃 없음은 의도.
  • 수동 FAILED 의 failedStepCode=null — UI 는 '운영자 전환' 으로 표시. 기존 failed(step) 스크립트 문구는 null 안전 처리.
  • last_seen 갱신 쓰기 — 게스트당 30초 1 UPDATE. 현 규모 무시 가능 — 대량화 시점에 버퍼링 재고(지금 최적화 금지).
