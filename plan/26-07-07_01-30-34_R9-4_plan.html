§2. 요구사항 · 설계 결정 (D1~D5)

D1. 명칭 단일화 — 자원 무결성 점검 (Q1 비판 검증 확정) 채택

근거: ① 감지 12종 중 "경로" 문제는 3종뿐 — 서명 불일치/내용 변조/유령 DB 행 등 과반이 경로 무관이라 "경로 점검"은 부분집합 명칭. ② "무결성"은 이 화면의 확인 문구와 management 6도메인 상세의 <dt>무결성</dt> 필드가 이미 쓰는 기존 어휘 — 신조어가 아니다. ③ (c)"정합 점검"은 "정합"이 적용(fix) 행위 어휘로 이미 쓰여 탐지/도구 분리를 흐림. 적용 지점 전수: navbar/탭/h1/JobType typeLabel/Job title("자원 무결성 점검"·"정밀 점검")/컨트롤러·템플릿 주석/교차 참조 문구(os·bios list, orphan modal).

D2. 헤더 버튼 2계층 — 일상 점검 1차, 재발급은 하단 "관리 도구" 구획 채택

헤더 = [지금 점검] [정밀 점검] 2개(+ hover tooltip: 정밀은 "파일 내용까지 검사 — 용량에 따라 수 분"). '마커 서명 재발급'은 페이지 하단 관리 도구 카드로 이동 — 무엇인지/언제 쓰는지 상시 설명문과 함께(클릭해야 보이던 confirm 의존 해소). 버튼 id(reissueBtn) 유지 → list.js 무변경. 비채택: 재발급 별도 페이지 — 1회용 도구에 네비 항목 추가는 과함(조사 §7-8).

D3. '격리 복구' → navbar 독립 메뉴 업로드 실패 복구 채택

reconciliation 헤더에서 링크 제거, navbar Maintenance 군에 독립 항목. quarantine 페이지는 navbar('quarantine') + 제목 "업로드 실패 복구"(도메인 중립 — saga 는 ResourceType 무관 설계) + "← 경로 점검" back-link 제거(navbar 로 대체). 발생 맥락(업로드 실패 modal)과 발견 맥락(navbar)이 분리되어도, 이름 자체가 "업로드 실패"를 말하므로 오독이 사라진다.

D4. 요약 배너 + 격리 대기 안내 (이 페이지 한정 count 조회) 채택

헤더 아래 도메인 설명 1문장("등록된 자원 정보(DB)와 디스크의 실제 파일·마커가 서로 일치하는지 점검하고 정합합니다.") + 최신 점검 요약("마지막 점검 {시각} · 자원 N건 · 드리프트 M건"). 격리 PENDING>0 이면 안내 배너("업로드 실패로 격리된 파일 N건이 복구를 기다립니다 → [업로드 실패 복구]"). count 는 OrphanQuarantineService.countPending() 신설(repository 파생 countByState) — reconciliation 페이지 렌더에만 얹는다. 비채택: navbar 전역 배지 — 전 페이지 렌더마다 도메인 횡단 count 쿼리(조사 실행성 렌즈 기각).

D5. DESIGN.md — §4 Distinctive Components 에 Maintenance 패턴 3종 기록 채택
