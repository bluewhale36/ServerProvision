§0. 동기 — "보류 권고" 에서 "진행" 으로, S5-13 을 위한 첫 이벤트 도입

R3-4 plan §8 과 R3-5 §11 은 R3-6(cascade 이벤트화)을 "over-abstraction 보류 권고"로 표기했다. 단순 디커플링만 보면 R3-4 의 BoardScopedChildLifecycle SPI 가 이미 의존을 충분히 응집해 두었기 때문이다. 그러나 사용자가 방향을 정했다 — 이벤트 기반 설계로 조금씩 전환하고, 특히 보류 중인 S5-13(자원 등록 후 무결성 검증 자동화)이 EventListener 를 핵심 수단으로 채택했으므로, R3-6 에서 프로젝트 최초의 이벤트 관례를 먼저 확립해 둔다.

프로젝트 현황 (조사 결과) — 코드베이스에 ApplicationEventPublisher 사용 0건, 도메인 이벤트 클래스 0건. 유일한 @EventListener 는 PathReconciliationService 의 ApplicationReadyEvent 수신뿐(Spring 생명주기). 즉 R3-6 은 프로젝트 첫 ApplicationEventPublisher 도입이다. 현행 비동기 디커플링은 전부 @Async + BackgroundJobService 직접 호출.

§1. 현재 상태 — cascade 5연산의 이벤트 적합성 + R3-4 보존 invariant

선행 / 현 구조 (R3-4)

  • 완료 R3-4 — BoardModelLifecycleService 가 List<BoardScopedChildLifecycle> children(어댑터 3종 @Order 10/20/30, @Lazy 자식 service)을 동기 다형 순회로 cascade. 5 메서드 SPI.
  • 완료 R3-5 — restore default 흡수 + typed-name verifier 위임. R3 도메인 사실상 마감, R3-6 은 그 위 마지막(선택) 슬라이스.

★ 핵심 — cascade 5연산의 이벤트 적합성 분류

이벤트(publishEvent)는 fire-and-forget(void) 이다. 호출자에게 값을 돌려줄 수 없다. 이 한 가지 사실이 R3-6 의 범위를 결정한다.

cascade 연산반환이벤트?이유
softDeleteActive(boardId)void command가능(동기만)순수 side-effect. 단 동기 같은-트랜잭션 리스너라야 원자성·순서 보존
recomputeEffective(boardId)void command가능(동기만)부모 own flip 後 자식 effective 재계산. 동기 같은-트랜잭션 필수
restoreDeleted(boardId)int return불가복구 건수를 .sum() 으로 합산해 RestoreResponse 반환 — 이벤트가 핸들러 결과를 발행자에 못 모아줌
hasAny(boardId)boolean query불가purge 선결 가드 — 그 자리에서 boolean 받아 거절 분기. fire-and-forget 불가
deletedLabels(boardId)List<String> query불가휴지통 preview read — 라벨 수집 반환 필요
결론 — 하이브리드가 유일한 정합 : 이벤트화 가능 = void command 2개(softDelete / recompute), 그것도 동기 @EventListener 한정. 반환/쿼리 3개는 SPI(BoardScopedChildLifecycle) 직접 호출로 잔류. SPI 는 5 → 3 메서드로 축소.

R3-4 가 못박은 보존 invariant (이벤트화가 깨면 안 되는 4가지)

invariant동기 @EventListener@TransactionalEventListener(AFTER_COMMIT)
(a) 자식 충돌 시 @Transactional 전체 롤백보존 같은 트랜잭션 — 리스너 예외가 발행자로 전파붕괴 부모 이미 commit → 부분 적용
(b) @Order 순회 순서(BIOS→BMC→Subprogram)리스너 @Order 로 재현 가능하나 command 2개는 자식 간 순서 무관(아래)붕괴
(c) 예외 전파보존 동기 전파붕괴 commit 후라 못 끌어옴
(d) softDelete : 자식 우선 → board.softDelete 後보존 발행 지점으로 보장(board.softDelete 前에 publish)붕괴 순서 역전
다행한 정합 — 이벤트화 후보 2개(softDelete/recompute)는 자식 간 순서가 무관하다(BIOS 먼저든 BMC 먼저든 결과 동일). 순서가 중요한 것은 restore count·labels(둘 다 SPI 잔류)다. 따라서 2 command 리스너에는 @Order 결합이 불필요 → "발행자-구독자 비결합"이라는 이벤트 본래 취지가 살아난다. 부모-자식 순서(d)는 발행 지점(board.softDelete 직전 publish)으로 자연 보존.
§2. 요구사항 (확정) + 비목표 (정직한 한계)
  1. 첫 이벤트 인프라 도입 — BoardModelLifecycleService 에 ApplicationEventPublisher 주입. 신규 이벤트 record 2종(management/board/event/). 동기 @EventListener 리스너(자식 도메인).
  2. void command 2개 이벤트화 — softDeleteActive / recomputeEffective 를 SPI 직접 호출 → 이벤트 발행으로 전환. 발행 지점은 기존 호출 지점과 동일(softDelete: board.softDelete 前 / recompute: 부모 own flip 後).
  3. SPI 축소 + 잔류 — BoardScopedChildLifecycle 에서 2 command 제거 → 3 메서드(restoreDeleted/hasAny/deletedLabels)만 남김. BoardModelLifecycleService 는 List<BoardScopedChildLifecycle>(3 SPI)를 계속 주입 + ApplicationEventPublisher 추가.
  4. invariant 100% 보존(불가침) — (a)(c) 동기 같은-트랜잭션으로, (d) 발행 지점으로 보존. 기존 BoardModelLifecycleServiceTest 회귀가 망(단, 검증 방식은 "자식 직접 호출" → "이벤트 발행" 으로 전환 — §6).
비 목표 / 정직한 한계 (불가침 — over-abstraction 경계)
  • Board→자식 컴파일타임 의존 "완전" 단절 아님 — 반환/쿼리 3개가 SPI 로 남으므로 BoardModelLifecycleService 는 List<BoardScopedChildLifecycle> 타입 의존을 유지한다. 이벤트는 2 command 의 메서드 호출 엣지만 제거. @Lazy 도 대체로 잔류(어댑터가 3 SPI 위해 자식 service 보유).
  • AFTER_COMMIT 비동기 이벤트 도입 아님 — R3-6 은 동기 cascade 전용. AFTER_COMMIT 패턴은 S5-13 의 몫(R3-6 은 publisher 관례만 선공급).
  • 다른 도메인(OS/BIOS 자체) 이벤트화 아님 — R3-6 은 Board cascade 2 연산 파일럿. 확산은 후속.
  • BackgroundJobService 의 이벤트 대체 아님 — S1 Job 허브는 그대로. 무분별한 이벤트화는 중복 — cascade 의 적합 2 연산에만 한정.
CP1 에서 반드시 합의 — 본 슬라이스의 실익은 제한적이다(5연산 중 2개, SPI·@Lazy 잔류). 1차 가치는 "Board cascade 완전 디커플링"이 아니라 ① S5-13 이 재사용할 첫 이벤트 관례 확립 + ② 적합 2 연산의 이벤트 파일럿이다. 이 가치 판단에 동의하는지가 D1 의 본질.

§3. 신구 비교 (split-grid) — softDelete / recompute / SPI 축소

① softDelete — SPI 직접 호출 → 이벤트 발행

BEFORE (R3-4/R3-5)

@Override @Transactional
public void softDelete(Long id) {
  BoardModel board =
    BoardModelGuards.requireActiveBoard(repo, id);
  children.forEach(c -> c.softDeleteActive(id)); // 직접
  board.softDelete();
  board.markTrashed(null);
}

AFTER (R3-6)

@Override @Transactional
public void softDelete(Long id) {
  BoardModel board =
    BoardModelGuards.requireActiveBoard(repo, id);
  // board.softDelete 前 발행 → 동기 리스너가
  // 같은 트랜잭션에서 자식 먼저 처리 (순서 d 보존)
  events.publishEvent(new BoardSoftDeletedEvent(id));
  board.softDelete();
  board.markTrashed(null);
}

② recompute — 3 호출부(toggle/deprecate/undeprecate) 공통

BEFORE

private void cascadeRecompute(Long id) {
  children.forEach(c -> c.recomputeEffective(id));
}
// toggle: parent.toggleEnabled(); cascadeRecompute(id);

AFTER

private void cascadeRecompute(Long id) {
  events.publishEvent(new BoardEffectiveChangedEvent(id));
}
// toggle: parent.toggleEnabled(); cascadeRecompute(id);
// (호출부 불변 — 발행 지점 = 부모 own flip 後 유지)

§6. 테스트 — 이벤트 도입에 따른 검증 구조 변화 (첫 패턴)
핵심 변화 — 이벤트화로 BoardModelLifecycleService.softDelete 는 더 이상 자식을 직접 호출하지 않는다. 따라서 service 단위 테스트는 "자식 softDelete 호출 검증" → "이벤트 발행 검증" 으로 바뀐다. 자식 cascade 동작은 리스너 단위 + 통합 테스트로 분리 검증. 이 3층 구조가 S5-13 이 차용할 선례.

① service 단위 — 발행 검증 (Mockito)

  • softDelete → verify(events).publishEvent(new BoardSoftDeletedEvent(id)) + board.softDelete 호출 + (자식 직접 호출 검증은 제거).
  • toggle/deprecate/undeprecate → BoardEffectiveChangedEvent 발행 검증.
  • restore/purge/labels → 기존 SPI mock 검증 그대로(이벤트화 안 됨).

② 리스너 단위 — 자식 cascade 동작 (Mockito)

  • BiosBoardScopedChildLifecycle.onSoftDeleted → 활성 자식 조회 후 biosService.softDelete 호출 검증(기존 softDeleteActive 검증을 리스너로 이동).

③ 통합 — 발행→구독 와이어링 + 원자 롤백 (신규 패턴)

  • @SpringBootTest + @RecordApplicationEvents — softDelete 호출 시 이벤트 1건 발행 + 3 리스너 동기 수신 확인.
  • 롤백 보존 검증(불가침) — 한 자식 리스너가 예외 throw → 전체 트랜잭션 롤백(부모 board 도 미삭제)로 invariant (a)(c) 실증.
범주 커버리지 — 이벤트 경로는 HTTP status 무변(컨트롤러 불변)이라 기존 @WebMvcTest 회귀는 그대로. 신규는 발행/리스너/롤백 시나리오.
§7. 예외 계층 — 신규 0

신규 예외 0. 동기 @EventListener 는 리스너 예외를 발행자 트랜잭션으로 그대로 전파하므로 기존 자식 service 예외(Duplicate* 등)와 롤백 의미가 불변. IllegalBoardModelStateException(purge 가드, SPI 잔류) 등도 그대로.

주의 — 이벤트의 예외 전파 : 동기 발행은 리스너 예외를 즉시 던진다(보존). 만약 향후 누가 리스너를 @Async 로 바꾸면 예외가 발행자에서 사라진다 — R3-6 리스너는 동기(@Async 금지) 임을 주석으로 못박는다.
