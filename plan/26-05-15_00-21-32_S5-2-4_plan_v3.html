3.2 PurgeExecutor (v3)

// global/trash/PurgeExecutor.java
public class PurgeExecutor {
    public PurgeResult execute(PurgeRequest req) {
        // 단일 cron tick / 1 사용자 호출 = 1 purge_log 행 (성공 or 마지막 실패)
        Markable snapshot = scanner.findTrashedById(req.resourceId()).orElseThrow();
        Exception lastFailure = null;
        for (int attempt = 1; attempt <= retryAttempts(req); attempt++) {
            try {
                scanner.purgeFromTrash(req.resourceId());
                purgeLogRepository.save(PurgeLog.success(req, snapshot));
                return PurgeResult.success(req);
            } catch (PurgeIoFailedException ex) {
                lastFailure = ex;
                if (attempt < retryAttempts(req)) {
                    Thread.sleep(backoff(attempt));  // 1s/2s/4s — TTL_AUTO 만
                }
            }
        }
        int historicalFailedCount = purgeLogRepository.countByResourceAndOutcome(
            req.resourceType(), req.resourceId(), PurgeOutcome.FAILED);
        purgeLogRepository.save(PurgeLog.failure(req, lastFailure, historicalFailedCount + 1));
        return PurgeResult.failed(req, lastFailure);
    }

    private int retryAttempts(PurgeRequest req) {
        return req.origin().retriesAllowed()
                ? trashSettings.getRetryMaxAttempts()
                : 1;
    }
}

JSON 컬럼 채택 근거 outcome=SUCCESS 의 manifestHash 와 outcome=FAILED 의 exceptionClass 는 완전 disjoint. 컬럼으로 표현하면 SUCCESS row 의 exceptionClass 컬럼이 항상 null, FAILED row 의 manifestHash 가 항상 null. JSON 으로 응집하면 outcome 별 키만 존재. 새 필드 추가도 schema migration 불요.
성능 메모 3회 초과 실패 자원 알림 등 JSON 키 기반 조회 (예: details->'$.attemptNumber' > 3) 가 MariaDB 의 JSON 인덱스 부재로 느려질 가능성. 현재 슬라이스는 단순 count() / 시간순 조회 정도라 무리 없을 것. 후속 운영에서 병목 발생 시 PostgreSQL 이관 검토 (jsonb + GIN 인덱스).
