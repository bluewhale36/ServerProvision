S6-2-2 — 삭제한 자원이 밖에서 발견되는 두 상황 CP1

S6-2 캠페인 2/3 · 2026-07-11 KST · 상위 결정: S6 단계 plan D2·D6·D7

지금의 점검은 휴지통으로 보낸 자원에 무슨 일이 생겨도 아무 말이 없습니다. 삭제한 파일이 원래 자리에 슬쩍 돌아와 있어도, 엉뚱한 폴더에 사본이 굴러다녀도 화면엔 "이상 없음"만 뜹니다. 이 슬라이스는 그중 "파일이 어딘가에 살아 있는" 두 상황을 점검이 발견하게 만들고, 발견 즉시 시스템이 해결까지 해 주도록 합니다. (파일이 아예 사라진 휴지통 상황 두 가지는 다음 슬라이스 S6-2-3 몫입니다.)

🎬 두 상황을 직접 만들어 보려면 — 드리프트 상황 시뮬레이터(report/26-07-11_17-57-19_drift-simulator_guide.html)에서 "삭제 자원 복귀"·"삭제 자원 위치 이탈" 버튼을 눌러 보세요. 발견되는 모습부터 [적용]의 결과, 회수가 거절되는 충돌 상황까지 전부 체험할 수 있습니다.

상황 1 — 삭제 자원 복귀 (휴지통 파일이 원래 자리로 돌아옴)

삭제 자원 복귀 시스템이 해결 — 자동 처리도 가능

실제 상황 : 급한 김에 휴지통 폴더에서 ISO 를 손으로 꺼내 원래 자리에 되돌려 놓았다. 또는 삭제 당시 파일이 이미 없어서 기록만 삭제됐던 자원의 파일이 나중에 원위치에 나타났다.
지금은 : 완전 침묵 — DB 는 여전히 "삭제됨"인데 파일은 사용 위치에 있는 모순이 아무 데도 표시되지 않는다. 나중에 그 자원을 복원하려 할 때에야 얽힌 상태가 드러난다.
앞으로는 : 매시간 점검이 발견 — "휴지통에 있어야 할 자원이 원래 위치로 돌아와 있습니다" 카드가 뜬다.
[적용]하면 : 파일을 되돌려 놓았다는 건 다시 쓰겠다는 의도이므로, 시스템이 정식 복원 절차를 대신 밟아 준다 — 삭제 해제 + 휴지통 기록 정리 + 마커(자원 신분증) 재발급. 관리 화면 휴지통에서 직접 복원한 것과 완전히 같은 결과.

상황 2 — 삭제 자원 위치 이탈 (삭제한 파일이 엉뚱한 곳에서 발견)

삭제 자원 위치 이탈 내가 확인해 주면 시스템이 해결

실제 상황 : 삭제 처리한 자원의 파일이 마커까지 달고 전혀 다른 폴더에서 발견됐다 — 백업에서 잘못 복사했거나, 누가 휴지통에서 꺼내 엉뚱한 곳에 두었다.
지금은 : 완전 침묵 — 점검이 그 마커를 보고도 "삭제된 자원 것이네" 하고 조용히 넘어간다.
앞으로는 : 매시간 점검이 발견 — 발견 위치와 함께 카드가 뜬다. 원래 자리에도 파일이 있는 애매한 경우면 "원위치에도 파일 존재"까지 함께 알려준다.
[적용]하면 : 복귀와 달리 의도를 알 수 없는 상황이라 자동으로 움직이지 않는다. 내가 [적용]으로 확인해 주면 그때 발견된 파일을 휴지통으로 도로 회수하고 기록을 맞춘다. 다시 쓸 파일이었다면 회수 대신 휴지통 화면에서 정식 복원을 하면 된다.
회수가 거절되는 경우 : 휴지통에 이 자원의 사본이 아직 살아 있으면 회수를 진행하지 않고 안내한다 — 그대로 회수하면 기존 사본이 아무도 못 찾는 파일로 남기 때문. 휴지통 사본을 먼저 확인·정리한 뒤 다시 점검하면 된다.

설계 결정 — 왜 이렇게 하는가

복귀는 "자동 가능", 이탈은 "확인 필요"로 나눈 이유

원래 자리로 되돌려 놓는 행동은 해석이 하나뿐이다(다시 쓰겠다). 반면 엉뚱한 위치의 사본은 실수 복사일 수도, 백업 꼬임일 수도 있어 해석이 여럿이다 — 해석이 하나로 모이는 것만 자동 후보로 삼는다는 원칙(삭제 여부와 파일 위치가 동시에 얽힌 상황은 자동 제외)을 그대로 따른다.

복귀의 해결은 기존 복원 절차를 그대로 재사용한다

휴지통 복원 기능에는 이미 "파일이 먼저 원위치에 돌아와 있으면 기록만 맞춰 주는" 자가 치유 단계가 들어 있다. 점검의 [적용]이 그 절차를 그대로 호출하므로 복원 코드가 두 벌이 되지 않고, 복원이 원래 갖고 있던 안전장치(예: 상위 자원이 삭제된 상태면 하위 자원 복원 거절)도 자동으로 계승된다 — 시스템이 안전장치를 우회해 복원해 버리는 사고가 원천 차단된다.

회수 중 문제가 생기면 되돌린다

회수는 "파일 이동 + 기록 갱신" 두 걸음이다. 이동까지 해 놓고 기록 갱신이 실패하면 파일을 원래 발견 위치로 되돌려, 화면에 보이는 것과 실제 디스크가 어긋난 채 남지 않게 한다 (복원·삭제 기능이 이미 쓰고 있는 되돌림 방식과 동일).

점검 기록 저장 공간을 한 글자 늘려야 한다 (DB 변경 1건)

점검 결과를 저장하는 칸이 지금은 24자까지인데, "삭제 자원 복귀"의 내부 이름이 26자라 이 변경 없이는 발견 순간 저장이 실패한다. 한 줄짜리 DB 변경 스크립트(sql/S6-2-2_drift_kind_widen.sql)를 만들어 두며, 서버를 새로 켜기 전에 사용자가 직접 적용해야 한다(이 계정엔 DB 구조 변경 권한이 없음 — R9-5 때와 같은 절차). 적용 전에 켜면 서버가 불일치를 감지하고 기동을 멈추므로 데이터가 깨질 일은 없다.
