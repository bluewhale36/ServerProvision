§1. 현재 상태 (8-에이전트 Workflow 적대적 검증 진단)

증상 — OS 버전 9.5 = 비활성화(disabled), 하위 ISO 27 = 활성화(enabled). 부모가 꺼져 있는데 자식이 켜진 lifecycle 논리 모순.

Q1 — 유스케이스 (순서 의존 3단계, 가설 A∧B 곱) :

  1. 자식 ISO soft-delete → is_deleted=true, is_enabled=true 보존
  2. (자식이 trash 에 있는 상태에서) 부모 OS disable toggle → cascade 가 !iso.isDeleted() 필터로 soft-deleted 자식 제외 → 자식 is_enabled=true 그대로 (모순 씨앗 = Leg A)
  3. 자식 restore → parent.isDeleted() 만 가드(통과), is_enabled=true 보존 → 부모 disabled · 자식 enabled 표면화 = Leg B
핵심 순서 : 부모 disable 이 자식 soft-delete 후. HF-1 self-heal override 가 비로소 DB flush 시켜 stale enabled 표면화.

Q2 — 정확한 지점 (두 레그의 곱) :

레그file:line결함
A (stale 생성, ★ v2 수정 대상)OSMetadataLifecycleService:58.filter(!iso.isDeleted() && !iso.isDeprecated()) — soft-deleted 자식 cascade 제외 → 부모 disable 이 trash 자식에 전파 안 됨
B (surface)IsoLifecycleService:165if (parent.isDeleted()) — parent.isEnabled() 미확인. v2 에선 Leg A 차단으로 stale 자체가 안 생겨 미수정
v2 전환 — 초기 진단의 judge 권장(후보1 = Leg B restore 종속)을 사용자 지적으로 재검토. "부모 disable 시 DB 의 모든 자식을 동기화하면 되는 문제 아니냐" → Leg A(cascade 필터) 를 고치면 stale 씨앗 자체가 안 생겨 restore 가드 불필요. 더 단순. (§8 후보 전환 근거)
§2. 요구사항 + 비목표 (후보2 — cascade 비대칭)

채택 = 후보2 : toggleEnabled cascade 를 비대칭으로 수정. 부모 disable 시 soft-deleted 자식까지 동기화해 stale 씨앗(Leg A) 자체를 차단.

  1. disable cascade — 전체 자식 : 부모 disable 시 모든 자식(trash 포함, deprecated 무관) 중 enabled 인 것을 disable → 복구 시 자동 정합.
  2. enable — cascade 없음 : 부모 enable 은 자식에 cascade 하지 않는다. 부모 ceiling 만 상승하고 개별 자식 활성화는 운영자 몫 (부모 활성화가 운영자가 개별 비활성한 자식을 임의 부활시키지 않도록). 비대칭 (E2E 후 사용자 정정).
  3. 3 도메인 동시 : OS/ISO(cascade 1) + BoardModel/BIOS·BMC(cascade 2) = 총 3 cascade.
비 목표 :
  • restore 부모 종속 가드(후보1) 미채택 — cascade 수정만으로 stale 씨앗 차단 → 모순 근절. restore 3곳 무수정 (사용자 결정 : cascade 수정만).
  • deprecate cascade 필터(!iso.isDeleted()) 유지 — LifecycleEntity.deprecate() 가 soft-deleted 에 IllegalDeprecationStateException 가드 보유 → soft-deleted 자식 deprecate 는 예외. 필터 필수 (§8 D-2).
  • enabled invariant 도메인 불변식 승격 — 도메인 4+ 늘 때 후속(§11).
  • Driver/Subprogram lifecycle service 미구현 — 향후 동형.
  • LifecycleEntity 미수정.
🎬 미리보기 — cascade 비대칭 before/after + 상태전이
[T1] ISO 27 soft-delete → ISO 27 {enabled, DELETED} [T2] OS 9.5 disable toggle BEFORE: cascade .filter(!iso.isDeleted()) ✗ ISO 27 제외 → ISO 27 {enabled, DELETED} ★ stale AFTER : disable cascade 전체 ✓ ISO 27 포함 → ISO 27 {disabled, DELETED} ★ 정합 [T3] ISO 27 restore → is_enabled 보존 BEFORE: {ENABLED, active} ← 모순 AFTER : {disabled, active} ← 정합 (restore 무수정으로도 자동 해결)

BEFORE — 대칭 필터 (현재)

parent.toggleEnabled();
boolean target = parent.isEnabled();
parent.getIsos().stream()
    .filter(iso -> !iso.isDeleted()    // ← trash 자식 제외
               && !iso.isDeprecated())
    .filter(iso -> iso.isEnabled() != target)
    .forEach(ISO::toggleEnabled);
// disable 시 trash 자식 누락 → stale 씨앗

AFTER — 비대칭 (disable 전체 / enable cascade 없음)

parent.toggleEnabled();
if (parent.isEnabled()) {
    return;   // enable: cascade 없음 (자식 개별 활성은 운영자 몫)
}
// disable: enabled 자식 전부(trash/deprecated 포함) 비활성화
parent.getIsos().stream()
        .filter(ISO::isEnabled)
        .forEach(ISO::toggleEnabled);
// disable 시 trash 자식도 동기화 → stale 0
핵심 — early-return 으로 비대칭 달성. enable 은 자식 cascade 없이 부모 ceiling 만 상승, disable 은 enabled 자식 전부(trash 포함) 비활성화. invariant "자식 enabled ≤ 부모 enabled" 는 disable cascade 만으로 충족(자식을 끌어내림); enable 은 ceiling 만 올려 위반 불가. Leg A 차단으로 stale 씨앗 0 → restore 무수정으로도 모순 근절.
§3. 수정 책임 + 시그니처 (3 cascade 비대칭)

3 cascade (OS→ISO 1, BoardModel→BIOS 1, BoardModel→BMC 1) 에 동형 비대칭 적용.

// OSMetadataLifecycleService.toggleEnabled (자식 ISO)
parent.toggleEnabled();
if (parent.isEnabled()) return;                  // enable: cascade 없음 (자식 개별 활성은 운영자 몫)
parent.getIsos().stream()                         // disable: enabled 자식 전부(trash/deprecated 포함)
        .filter(ISO::isEnabled).forEach(ISO::toggleEnabled);

// BoardModelService.toggleEnabled (자식 BIOS + BMC — 2 cascade 동형)
if (parent.isEnabled()) return;                  // enable: cascade 없음
biosRepository.findAllByBoardModel_IdOrderByVersionDesc(id).stream()
        .filter(BoardBIOS::isEnabled).forEach(BoardBIOS::toggleEnabled);
// bmc 동형
중복 회피 (§8 D-3) — 3 cascade 가 동형이나 자식 타입(ISO/BoardBIOS/BoardBMC)·조회 방식이 다름. 지금은 각 service 사본(R1-4 helper 사본 정책 동형). 도메인 4+ 늘 때 §11 공통 헬퍼 승격.

§7. 예외 계층

변경 0. cascade 동기화는 상태 전이라 신규 예외 없음.

  • LifecycleEntity.toggleEnabled() 는 가드 없음 → soft-deleted 자식 toggle 해도 예외 0 (후보2 가능의 핵심).
  • LifecycleEntity.deprecate() 는 soft-deleted 가드 보유 → deprecate cascade 는 !isDeleted() 필터 유지 (toggle 과 분리).
§8. 부산물 / 결정 (후보 전환 근거)

D-1. 후보1(restore 종속) → 후보2(cascade 비대칭) 전환 근거

초기 Workflow judge 는 후보2 를 "soft-deleted 자원에 toggle 강제 시 entity 가드/trash 의미 손상" 으로 기각했다. 그러나 코드 재검증 :

  • LifecycleEntity.toggleEnabled() (94-96) — 트랜지션 가드 없음 (this.isEnabled = !this.isEnabled). soft-deleted 자식 toggle 가능.
  • LifecycleEntity.deprecate() (118-) — soft-deleted 가드 있음.

→ judge 가 deprecate 의 제약을 toggleEnabled 에 잘못 일반화. toggleEnabled cascade 필터는 entity 가드와 무관하므로 제거 가능. 사용자 직관(후보2)이 옳고 더 단순 — Leg A 차단으로 restore 가드(후보1) 불필요.

D-2. disable / enable 비대칭 (사용자 결정)

disable : enabled 자식 전부(trash / deprecated 포함) 비활성화 — 부모 disabled 하에 enabled 자식 0 보장. enable : cascade 없음 — 부모 ceiling 만 상승, 자식 활성화는 운영자 개별 액션. invariant "자식 enabled ≤ 부모 enabled" 는 disable cascade 만으로 충족(자식을 끌어내림); enable 은 ceiling 상승이라 위반 불가. E2E 후 사용자 정정 : 초기 v2 의 "enable active 자식만" → "enable cascade 없음" (부모 활성화가 운영자가 개별 비활성한 자식을 임의 부활시키면 안 됨).

D-3. 종속 로직 사본 (지금) vs 승격 (후속)

3 cascade 동형이나 자식 타입·조회 상이. 각 service 사본(over-abstraction 회피). 도메인 4+ 늘 때 §11.

D-4. CLAUDE.md 정합

  • "분기문 무분별 확장 금지" — if(target) 한 줄은 도메인 의미(disable=전체/enable=active)의 정당한 단일 분기. 책임 늘리는 분기 누적 아님.
  • "중복/over-abstraction 금지" — 각 service 사본, 승격은 4+ 에서.

D-5. 모순 잔존 자원 (선택)

이미 DB 에 부모 disabled·자식 enabled 로 남은 자원(ISO 27 등)은 본 수정 후 부모를 disable→(자동 cascade) 또는 자식을 개별 disable 로 정합. 일괄 마이그레이션은 비목표.

D-6. N+1 / cascade 쓰기 성능 (3-렌즈 Workflow 적대 검증 + 사용자 결정)

사용자 우려 "stream+forEach 가 N+1 을 일으키지 않나" → SQL 추적 / 숨은 lazy-load 사냥 / bulk 대안 3 독립 렌즈 만장일치 :

  • N+1 SELECT 없음 — 필터 술어(isDeleted/isDeprecated/isEnabled)와 toggleEnabled() 는 동일 row scalar 컬럼만 접근. 루프 내 per-element lazy 재조회(=N+1 트리거) 부재. SELECT 고정 : OS = 부모1 + getIsos() LAZY1 = 2, Board = 부모1 + bios1 + bmc1 = 3. 자식 수와 무관.
  • 발생하는 건 dirty checking 의 자식당 UPDATE N개 — "N+1 SELECT 문제" 아니라 "N행을 실제로 끄는 필연 쓰기". bulk 로 접어도 쓸 row 수는 동일.
  • bulk JPQL(UPDATE ... WHERE id IN) 기각 — @LastModifiedDate updated_at auditing 깨짐(JPQL 수동 set 필요) + 영속성 컨텍스트 stale(clearAutomatically 필요) + LifecycleEntity 도메인 메서드/guard 우회 + 6 cascade 혼재(또는 repository 메서드 3중 폭증). cardinality 한 자릿수에서 실익 없음.

채택 : spring.jpa.properties.hibernate.jdbc.batch_size=20 + order_updates=true 전역 설정. N개 UPDATE 를 단일 JDBC 배치 왕복으로 묶어 사용자 목표("N개 따로 발행 회피")를 충족하면서 도메인 메서드·auditing·guard 전부 보존. @GeneratedValue(IDENTITY) 라 INSERT 배칭은 불가하나 UPDATE/DELETE 는 적용. 모든 write 경로 전역 수혜. HF-2 본 cascade 코드(D-2)는 무변경 — config 1회 추가뿐.
