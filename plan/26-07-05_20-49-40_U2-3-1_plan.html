§1. 현재 상태 — instanceof 사다리 2곳 복제

U2-3 본편 + lifecycle 유효성 + 환경-그룹 정합이 순차 반영되면서, "단계 타입 X 가 어떤 management 자원을 참조하고 어떻게 검증되는가"라는 지식이 두 서비스의 instanceof 사다리에 복제된 상태다.

위치분기 내용
JpaSettingCommandService.validateReferences (쓰기 가드) BasicUpdate(보드 실존·enabled / 펌웨어 소속·enabled) / OSInstallation(OS 실존·enabled) + RHEL 중첩 분기(환경-그룹 정합) / OSSetting(OS 실존·enabled) — 3분기 + 중첩 1단
JpaSettingQueryService.collectDeprecatedUsages (조회 표시) 같은 3분기 복제 — 같은 참조를 deprecated 관점으로만 다시 순회
왜 지금인가 — 분기 수 자체(3)보다 기울기가 문제다: 이번 세션에서만 2→3분기+중첩 1단이 됐고, 직후의 U2-4(OS 3차 검증 — 파티션 규칙·계정·계열별 값)는 이 사다리 안에서 자랄 수밖에 없다. CLAUDE.md 불가침 "분기문은 도메인 의미가 늘 때마다 같이 자란다"의 전형이며, "갈라지는 시점에 분리한다"의 그 시점이 도래했다 (U2-3 §8 에 "2건 초과 시 strategy 승격 검토"로 예고했던 임계 초과).
§2. 요구사항 (확정) + 비목표
  1. SPI 도입 — ProcessReferenceInspector(단계 타입별 참조 검사기) + 타입별 구현 빈 3개. 서비스 2곳의 사다리 소멸.
  2. 두 관심사 응집 — 쓰기 가드(validateReferences)와 조회 표시(describeDeprecatedReferences)를 타입당 한 inspector 에.
  3. 동작 무변 — HTTP 계약·예외·상태코드·메시지 전부 동일. 기존 통합 테스트 무수정 그린이 증명.
  4. OS 2단 계열 다형(사용자 지시) — OSInstallation/OSSetting inspector 내부의 계열 처리도 instanceof 가 아닌 OSFamily 키 2단 SPI 로: RHEL/Debian 지금 등록, Windows 는 확장 절차만 문서화(§4 D6).
  5. BASIC_SETTING no-op inspector 선생성(사용자 지시) — 바로 다음 단계(U2-2-3)가 로직을 주입할 자리. 전 타입 등록이 되므로 dispatch 미스 = fail-fast 로 강화(§4 D4 개정).
  6. U2-4 의 그릇 — 계열별 검증이 각 계열 inspector 내부에서만 자라도록 확장점 명시(구현은 비목표).
비 목표 : U2-4 OS 3차 검증 내용 자체(파티션/계정/selinux 값 검증) · BASIC_SETTING 의 실로직(U2-2-3 관계 연결 후) 및 제조사별 다형 분기(차이가 실재로 확인될 때 — 사용자 확정: 지금 형성하지 않음) · Bean Validation(@Valid 1차) 재배치 — Layer A(어노테이션)는 그대로 두고 영속 참조가 필요한 Layer B 만 다형화한다.
🎬 미리보기 — dispatch 라이브 데모 (사다리 vs SPI)

processList 를 토글하면 각 단계가 어느 inspector 로 라우팅되고 무엇이 검증되는지, 그리고 "U2-4 확장"·"U2-2-3 확장" 토글로 미래 요구가 들어올 때 두 방식(사다리 vs SPI)의 변경 파장이 어떻게 다른지 체험한다.

정의서 단계 구성 :
미래 확장 시나리오 :
processList (wire 순)
→
Map<SettingProcessType, ProcessReferenceInspector> dispatch
체험 포인트 — ① RHEL 의 환경-그룹 정합이 OSInstallationReferenceInspector 내부의 계열 처리로 들어가 서비스에서 중첩 분기가 사라진다. ② 확장 토글을 켜도 서비스/기존 inspector 는 무변 — 변경이 해당 inspector 1개(또는 신규 빈 1개)에 국소화된다. 사다리 방식이었다면 두 서비스의 분기 본문이 함께 자란다.
§3. 데이터 흐름 — URL 무변, 내부 위임만 변경 
[쓰기]  POST/PUT → @Valid(Layer A, 무변) → CommandService
          기존 : for(process) { if instanceof … }               ← 사다리
          변경 : for(process) inspectorOf(process.processType()).validateReferences(process)
[조회]  GET /{id} → QueryService.findDetail
          기존 : collectDeprecatedUsages 의 instanceof 사다리     ← 복제 사다리
          변경 : for(process) inspectorOf(...).describeDeprecatedReferences(process) → DeprecatedUsageResponse
[빈 구성] Spring 이 List<ProcessReferenceInspector> 주입 → 서비스(또는 registry)가
          Map<SettingProcessType, Inspector> 로 색인 (중복 target 은 기동 실패 = 결함 조기 노출)

HTTP 표면·DB 스키마·계약 record 전부 무변 — 순수 내부 구조 이동이다.

§4. 설계 + 대안 비교 (SSOT)

D1 — dispatch 방식: SettingProcessType 키 맵 채택

① target() + Map dispatch ✓ 채택

  • inspector 가 SettingProcessType target() 선언 → 서비스가 주입 List 를 Map 으로 색인
  • ProcessRequestDeserializer 등록 맵과 동일 관용구 — 판별자(processType)로 라우팅하는 기존 패턴의 서버측 대칭
  • 타입당 정확히 1개 보장: 중복 키 = 기동 실패(조기 결함 노출)

② supports() 순회

  • 범용적이나 이 도메인에선 라우팅 키(processType)가 이미 1급 값 — 순회 판정은 정보를 버리는 우회
  • 중복 매칭·무매칭이 런타임까지 숨음. 기각

③ Request 다형 메서드

  • AbstractProcessRequest.validate(...) — wire 계약 객체(DTO)가 repository 를 알게 됨
  • 레이어 경계 위반(계약↔영속 결합) + 직렬화 대상 오염. 기각

D2 — 인터페이스 형태 (두 관심사 응집)

public interface ProcessReferenceInspector {
    SettingProcessType target();                                        // dispatch 키 (판별자와 동일 축)
    void validateReferences(AbstractProcessRequest process);            // 쓰기 가드 — 404/409/400 throw
    List<String> describeDeprecatedReferences(AbstractProcessRequest p); // 조회 표시 — deprecated 자원명
}
  • ISP 로 쪼개지 않는 근거: 소비자(Command/Query)는 다르지만 지식("이 타입이 참조하는 자원과 그 해석")은 하나다. 인터페이스를 소비자별로 나누면 구현 클래스도 갈라져 같은 참조 순회 코드가 다시 두 곳에 생긴다 — 도메인을 가로지르는 과분리(불가침의 over-abstraction 경계).
  • 인자는 AbstractProcessRequest + 구현 첫 줄 1회 캐스팅. generic 기반 추상 클래스(TypedInspector<T>)는 구현 3개 규모에서 기반 코드만 늘림 — 구현이 5개를 넘고 캐스팅 실수가 실제 발생하면 그때 도입(갈라지는 시점 원칙).

D3 — 1단 구현 빈 4개와 다형 깊이 (사용자 지시로 타입별 차등)

Inspector (@Component)내부 다형근거와 이동 로직
BasicUpdateReferenceInspectorflat사용자 확정 — 단순 자원 업데이트라 계열 축이 없다. Command 의 validateFirmwareSelectors + Query 의 collectFirmwareDeprecations 이동. 다형 분기 도입하지 않음.
BasicSettingReferenceInspectorno-op 선생성사용자 확정 — 바로 다음 단계(U2-2-3)가 템플릿 관계 연결 후 로직을 주입할 자리를 지금 만든다. 제조사별 차이는 미지이므로 그 다형 분기는 형성하지 않음(차이가 실재로 확인될 때).
OSInstallationReferenceInspectorOSFamily 2단 SPI공통(osMetadataId 실존·enabled·deprecated 서술)만 직접 수행하고 계열 고유 참조는 OSInstallationFamilyInspector 맵으로 위임 — §D6 상세.

D4 (개정) — 전 타입 등록 + dispatch 미스 = fail-fast

  • BASIC_SETTING no-op 선생성으로 4 타입 전부가 맵에 등록된다 → "맵 미스 = 참조 없는 타입" 의 관용을 버리고 ProcessReferenceInspectors.inspectorFor(type) 미스 시 IllegalStateException(설정 결함의 조기 노출)으로 강화한다. 신규 단계 타입이 deserializer 에는 등록되고 inspector 를 빠뜨리는 실수가 기동·첫 요청에서 즉시 드러난다.
  • 2단(계열) 맵은 반대로 미스 = 계열 고유 참조 없음(통과) 를 유지 — wire 에 존재하는 계열(RHEL/DEBIAN)은 모두 등록하므로 실제 미스는 발생하지 않지만, 공통 검증이 이미 수행된 뒤라 의미론이 안전하다. (전 계열 등록이 유지되는지는 계열 추가 절차(D6)의 체크 항목.)

D6 (신설) — OS 계열 2단 SPI 상세 (RHEL/Debian 지금, Windows 확장 절차)

// 2단 SPI — wire 의 2단 판별자(osFamily())와 같은 축으로 dispatch (deserializer OS_SUBTYPES 의 서버측 대칭)
public interface OSInstallationFamilyInspector {
    OSFamily family();
    void validateReferences(OSInstallationRequest request);              // 계열 고유 참조만
    List<String> describeDeprecatedReferences(OSInstallationRequest request);
}
public interface OSSettingFamilyInspector {   // 요청 타입이 달라 별도 인터페이스 (제네릭 통합은 기반 코드 대비 이득 없음 — 과추상 경계)
    OSFamily family();
    void validateReferences(OSSettingRequest request);
    List<String> describeDeprecatedReferences(OSSettingRequest request);
}
계열 구현 빈책임 (현재 이동분 / U2-4 성장분)
RHELInstallationFamilyInspector현재: 환경-그룹 정합(comps.xml 관계, InvalidEnvironmentSelectionException) 이동 / U2-4: kickstart 파티션 규칙·계정·registrationKey 검증
DebianInstallationFamilyInspector현재: 계열 고유 참조 0(명시적 빈으로 완전성 확보) / U2-4: preseed 파티션 규칙·hostname·패키지 검증
RHELOSSettingFamilyInspector현재: 0 / U2-4: selinux 값·서비스 액션 검증
  • 공통/계열 책임 분담 원칙: 베이스 계층(OSInstallationRequest)의 필드는 1단 inspector 가, 계열 계층(RHEL 중간층/하위)의 필드는 계열 빈이 검증한다 — U2-1 의 얇은 베이스+계열 중간층 계약 구조와 1:1 대응이라 "어느 검증이 어디에 사는가"가 계약 구조에서 기계적으로 결정된다.
  • Windows 확장 절차(문서화만 — 구현은 해당 슬라이스): ① setting OSFamily 에 WINDOWS 실체화(U2-1 plan v2 §0 예약) ② ProcessRequestDeserializer OS_SUBTYPES 에 하위 Request 등록 ③ WindowsInstallationRequest(베이스 직속 — Linux 중간층 미경유) 작성 ④ WindowsInstallationFamilyInspector(+OSSetting 대응) 빈 추가. ①~④ 전 과정에서 기존 inspector·서비스 코드 무변 — 본 슬라이스가 보장하는 확장 계약.

D5 — 배치·의존

provisioning/setting/service/reference/ (신규 — 검사기 군집) ├── ProcessReferenceInspector.java (1단 SPI — SettingProcessType dispatch) ├── ProcessReferenceInspectors.java (1급 컬렉션 — 전 타입 등록 검증 + inspectorFor fail-fast) ├── BasicUpdateReferenceInspector.java (flat — Board/Bios/Bmc repository 주입) ├── BasicSettingReferenceInspector.java (no-op — U2-2-3 로직 주입 자리) └── os/ ├── OSInstallationReferenceInspector.java (공통 osMetadata + OSFamily 맵 위임) ├── OSInstallationFamilyInspector.java (2단 SPI) ├── RHELInstallationFamilyInspector.java (환경-그룹 정합 이동 — OSEnvironmentRepository 주입) ├── DebianInstallationFamilyInspector.java (현재 0 — U2-4 성장 지점) ├── OSSettingReferenceInspector.java (공통 + OSFamily 맵 위임) ├── OSSettingFamilyInspector.java (2단 SPI) └── RHELOSSettingFamilyInspector.java (현재 0 — U2-4 성장 지점) Jpa*Service 는 ProcessReferenceInspectors 만 주입(management repository 의존 제거) — 트랜잭션은 기존 서비스 경계 그대로(inspector 는 그 안에서 호출되는 무상태 빈)
적대적 자기비판 — ① 순환 위험? inspector 는 management repository 만 주입 — R7 금지 목록(TypedNameVerifier/scanner/ObjectProvider)과 무관, 순환 표면 없음. ② Query 의 deprecated 수집은 "존재+deprecated" 만 보고 Command 는 "부재를 throw" — 한 인터페이스가 어색하지 않나? 두 메서드의 계약이 다른 것이지(가드 vs 서술) 지식은 같다 — 같은 참조를 두 번 정의하는 현 상태보다 어색함이 적다. ③ Map 색인을 서비스마다 반복? 두 서비스가 같은 색인을 만들면 그 자체가 중복 — 색인을 ProcessReferenceInspectors(1급 컬렉션 빈, deserializer 의 registry 관용구)로 한 번만 구성해 양쪽에 주입한다. ④ 이 SPI 가 U2-4 의 "값 검증"(참조 아닌 파티션 규칙 등)까지 담아야 하나? — 담는다. 인터페이스 이름의 Reference 는 1차 책임을 말할 뿐, 단계 타입별 서버 검증의 단일 진입점이 되는 것이 목적(메서드 추가가 아니라 기존 validateReferences 내부가 자란다). 이름을 ProcessValidator 로 넓힐지 U2-4 진입 시 재평가(§8 기록).
