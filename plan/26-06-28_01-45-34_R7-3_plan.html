§1. 현재 상태 · 목표

R7-3 은 R7-2 와 대칭이다 — 도메인이 Board 하나라는 것과, TypedNameGuard 추출을 R7-2 가 이미 했다는 것만 다르다. R7-2 와 같은 commit 묶음 또는 직후 진행.

지금 Board scanner 의 순환 (REAL)

TypedNameVerifier → BoardModelMarkableScanner
   → ObjectProvider<BoardModelLifecycleService> (lazy)
   → BoardModelLifecycleService.purgeWithTypedNameCheck → typedNameVerifier.verify() (L157)
   → TypedNameVerifier ↺

주석 BoardModelMarkableScanner:86 이 "ObjectProvider 제거는 R7-3" 으로 명시. 변 (C) service → verifier 를 끊으면 ObjectProvider 가 불필요해진다.

목표 — BoardModelLifecycleService.purgeWithTypedNameCheck 가 typedNameVerifier.verify() 대신 손에 든 엔티티로 TypedNameGuard.verify(board, typedName) 를 쓰게 한다. service → verifier 변이 사라져 순환이 끊기고 ObjectProvider 가 제거된다. scanner 의 복구/영구삭제/cascade preview 는 그대로(MarkableScanner 통합 유지). 동작(보드 복구·cascade·영구삭제·자식 이름 미리보기)은 불변.
§2. 설계 (R7-2 와 동일 — 변 (C) 절단)

R7-2 §2 의 대안 비교(A vs D vs E)가 그대로 적용된다. 요지만 :

  • 채택 — 변 (C) service→verifier 절단(Option D) : MarkableScanner 통합 유지 + 변경 최소 + 6 도메인 일관.
  • 비채택 — 변 (B) scanner→service 절단(A) : MarkableScanner 가 쪼개지고 소비자 재배선 필요(사용자 우려).
  • 전역 @Lazy : scanner→service 변이 남아 SoftDeleteIntentService 의 @Lazy 는 유지(정당한 cross-cutting 차단).
Board 의 추가 이점 — Board 는 cascade(보드 복구 시 자식 BIOS/BMC 동반)가 본질인데, 그 로직은 BoardModelLifecycleService 가 이미 보유(BoardScopedChildLifecycle 다형 순회). typed-name 만 가드로 바꾸면 되고, cascade·findDeletedChildLabels(L145)는 그대로라 변경 표면이 작다.

판정은 §3 설계 재현 — bootRun 결과 예측.

§3. 변경 상세

BoardModelLifecycleService — verifier → TypedNameGuard (엔티티 재사용)

// 전환 전 (L157)
boardModelRepository.findByIdAndIsDeletedTrue(id).orElseThrow(...);   // 로딩 → 버림
typedNameVerifier.verify(ResourceType.BOARD_MODEL, id, typedName);     // 재조회 + 변 (C)
purge(id);

// 전환 후
BoardModel board = boardModelRepository.findByIdAndIsDeletedTrue(id).orElseThrow(...);  // 엔티티 보존
TypedNameGuard.verify(board, typedName);    // 재조회 없음, verifier 빈 미주입 → 변 (C) 소멸
purge(id);

private final TypedNameVerifier typedNameVerifier 필드 삭제. (정확한 라인은 CP2 에서 메서드 본문 확인 후 확정 — 패턴은 OS 와 동일.)

BoardModelMarkableScanner — ObjectProvider → 직접 주입

  • ObjectProvider<BoardModelLifecycleService> 필드/생성자 인자(L25, L29) → BoardModelLifecycleService 직접 주입.
  • restoreFromTrash/purgeFromTrash/findDeletedChildLabels 가 getObject().restore/purge/findDeletedChildLabels → boardModelLifecycleService.… 직접 호출. trash 메서드 그대로 유지(이전 안 함).
TypedNameGuard 는 R7-2 가 추출. R7-3 은 그 헬퍼를 호출만 한다(신규 파일 없음). R7-2 와 같은 commit 묶음으로 진행하면 추출↔사용이 한 번에 맞물린다.
