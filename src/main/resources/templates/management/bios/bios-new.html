대상 메인보드 :
사람이 식별하기 쉬운 레이블.
같은 메인보드 내에서 버전은 유일해야 합니다.
번들 등록 방식에 따라 의미가 달라집니다 — 업로드 모드에서는 "전개 대상 (비어있어야 함)", 기존 디렉토리 등록 모드에서는 "이미 콘텐츠가 차 있는 디렉토리". 경로를 직접 타자하거나 "탐색" 버튼으로 서버 디렉토리를 골라 채울 수 있습니다.
경로가 존재하지 않을 경우 자동으로 생성합니다.
단일 폴더 하나만 선택합니다. 개별 파일 여러 개를 흩뿌려 올리는 방식은 거절됩니다. 선택한 폴더 자체 는 대상 디렉토리에 만들어지지 않고, 그 안의 내용물만 대상 디렉토리로 전개됩니다 (폴더 이름 prefix 자동 제거). Gigabyte 서버보드 BIOS (RBU / SPI_UPD / f.nsh 등 다중 파일) 에 적합.
zip 내용이 파일들을 직접 담고 있으면 그대로 전개. zip 이 단일 폴더로 한번 감싼 형태이면 그 폴더는 제거되고 내용물만 전개됩니다. zip 파일 자체는 전개 후 폐기.
파일 하나가 BIOS 전체를 담고 있는 경우 (예: ASUS 의 .cap/.CAP). 해당 파일이 대상 디렉토리에 저장되고, 서버가 그 파일 자체를 진입점(대상 파일) 으로 등록합니다. Fujitsu 처럼 iRMC 웹 API 기반 업데이트는 파일 업로드 모델로 표현 불가해 지원되지 않습니다.
대상 디렉토리 에 이미 BIOS 번들 파일들이 존재한다는 전제로 등록합니다. 업로드 바이트 전송 없이 서버가 트리를 직접 스캔해 manifest · 진입점 · marker 를 생성합니다.
  • 경로가 존재하고 디렉토리여야 합니다.
  • 이미 다른 등록의 .provision.json 마커가 있는 디렉토리는 거절됩니다.
  • "디렉토리 생성 허용" 옵션은 무시됩니다 (디렉토리는 이미 있어야 함).
비워두면 서버가 트리 루트에서 f.nsh → flash.nsh → 단일 *.nsh 순으로 자동 탐지합니다. 진입점은 실행 파일 자체만 지정합니다. 실행 시 파라미터 등은 향후 Execution 도메인에서 관장합니다.
